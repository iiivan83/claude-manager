# Claude Manager

## Что это за проект

Telegram-бот, который служит пультом управления Claude Code с телефона. Пользователь пишет сообщение в Telegram — бот запускает Claude Code на компьютере, получает ответ и пересылает обратно.

Бот работает **локально** на компьютере пользователя (не на сервере). Общается с Claude Code CLI через протокол stream-json — отправляет сообщения через stdin, читает ответы из stdout.

**Ключевые возможности:**
- Отправка сообщений в Claude Code и получение ответов
- Управление сессиями: создание, переключение, мониторинг
- Наблюдение за сессиями в реальном времени (watcher)
- Отправка фотографий и файлов (Claude сам читает файл)
- Дневная нумерация сессий (#1, #2, #3...)

## Глобальные референсы

Всё, что повторяется между скиллами — стандарты, шаблоны, правила — лежит в глобальной папке `~/.claude/references/`, чтобы не дублировать внутри каждого скилла. Скиллы читают эти файлы при запуске, а не хранят копии у себя.

Ключевые референсы:
- `~/.claude/references/writing-style-guide.md` — единый стиль письма для всех текстов (отчёты, спеки, документация)
- `~/.claude/references/skill-testing-standard.md` — стандарт тестирования скиллов
- `~/.claude/references/cli-testing-patterns.md` — паттерны тестирования скиллов через CLI-подпроцесс (изоляция, таймауты, отладка)
- `~/.claude/references/document-naming-and-placement.md` — правила именования и размещения документов
- `~/.claude/references/agent-document-triggers.md` — когда агенты должны создавать документы
- `~/.claude/references/evals-schema.md` — единая JSON-схема формата evals.json для тестовых наборов
- `~/.claude/references/pipeline-skill-checklist.md` — чеклист для проверки пайплайн-скиллов

## Скиллы

Скиллы живут в `.claude/skills/`. Каждый — папка с `SKILL.md` и (опционально) агентами, скриптами, справочными материалами. Правила для скиллов — в `.claude/skills/AGENTS.md`.

Актуальный список скиллов с описаниями — в `dev/docs/skill-index.md`.

## Технологии

- **Telegram:** python-telegram-bot >=21.10
- **Конфигурация:** python-dotenv >=1.1.0
- **Протокол с Claude Code:** stream-json (потоковый JSON через stdin/stdout)
- **ОС:** macOS (fcntl для файловой блокировки)
- **Тесты:** pytest >=8.0, pytest-asyncio >=0.24, pytest-mock >=3.14

## Архитектурные принципы

### Слоёная архитектура

Код организован по слоям с чёткими границами ответственности:

- **Транспортный слой** (bot.py) — приём сообщений из Telegram, отправка ответов. Знает о Telegram API, не знает как работает Claude.
- **Слой бизнес-логики** (session_manager, daily_session_registry) — управление сессиями, нумерация. Не знает ни про Telegram, ни про процессы.
- **Слой инфраструктуры** (process_manager, claude_runner) — запуск процессов, чтение stdout, протокол stream-json. Не знает про Telegram.
- **Утилиты** (message_splitter, session_reader) — вспомогательные функции без состояния.
- **Мониторинг** (session_watcher) — фоновый цикл опроса файлов сессий, координация с обработчиком через pause/resume.

**Правило зависимостей:** верхний слой может вызывать нижний, но не наоборот. Инфраструктура не импортирует бизнес-логику.

### Асинхронность

Весь код — async/await на asyncio. Блокирующие операции (чтение stdout, работа с файлами) выносятся в потоки через `asyncio.to_thread()`. Никаких `time.sleep()` — только `asyncio.sleep()`.

### Состояние в памяти

Состояние хранится в словарях на уровне модулей: session_manager (привязки chat_id → session_id), daily_session_registry (дневные номера), process_manager (процессы, флаги занятости), session_watcher (счётчики обработанных сообщений). Персистентность — через JSON-файлы (sessions.json, daily_sessions.json). При потере состояния в памяти — автовосстановление из файлов при перезапуске.

### Защита персистентных данных от затирания

Модули с файловым хранилищем (daily_session_registry, session_manager) отслеживают флаг `_loaded_from_disk`. При запуске модуль пытается загрузить файл с диска (с повторными попытками при ошибке). Функция записи на диск проверяет этот флаг и отказывается записывать, если состояние не было загружено — чтобы пустые данные не перезаписали валидный файл. Если загрузка не удалась после всех попыток — бот сообщает пользователю в Telegram.

## Принципы разработки скиллов и пайплайнов

- **Универсальность** — эталонные скиллы не привязаны к конкретному проекту, они должны работать везде
- **Измеримость** — каждый скилл тестируется на реальных задачах, результаты фиксируются
- **Итеративность** — лучше быстро собрать, протестировать и улучшить, чем долго проектировать идеальное решение
- **Документация** — каждый шаг оставляет след: спеки, отчёты, ADR, логи

## Правила разработки

### Python стиль

- **Версия:** Python 3.13, используем все современные возможности (match/case, type hints с `|`, f-strings)
- **Type hints:** обязательны для всех публичных функций и методов. Используем встроенные типы (`list`, `dict`, `str | None`) вместо `typing.List`, `typing.Optional`
- **Docstrings:** одна строка для каждой публичной функции — что она делает. Формат: `"""Описание."""`
- **Импорты:** стандартная библиотека → сторонние пакеты → локальные модули. Разделяются пустой строкой
- **Именование:** snake_case для функций и переменных, PascalCase для классов, UPPER_SNAKE для констант

### Обработка ошибок

- Все ошибки от внешних систем (Telegram API, Claude CLI, файловая система) — ловятся и обрабатываются
- Пользователь всегда получает понятное сообщение об ошибке
- Логирование через `logging` — не через `print()`
- Уровни: `info` для штатных событий, `warning` для восстановимых проблем, `error` для сбоев

### Безопасность

- Токены и секреты — только в .env файле, никогда в коде
- Белый список пользователей (ALLOWED_USER_IDS) — проверяется при каждом запросе
- HTML-экранирование всех пользовательских данных через `html.escape()`
- Режим инструментов — полный доступ (все инструменты Claude, включая Bash)

### Тестирование

- Тесты в папке `tests/`, юнит-тесты зеркалят структуру `src/`
- Запуск: `python -m pytest tests/ -v`
- **Юнит-тесты** (`tests/test_*.py`) — по одному файлу на каждый модуль, моки для внешних зависимостей
- **Интеграционные тесты** (`tests/integration/`) — жизненный цикл сессий, путь сообщения, файлы, конкурентный доступ, координация watcher/handler
- **E2E тесты** (`tests/e2e/`) — через Telethon, реальные сообщения в Telegram
- asyncio_mode = "auto" (не нужен декоратор `@pytest.mark.asyncio` на каждом тесте)

## Важные детали для разработки

- **Протокол stream-json:** полная документация формата сообщений — в `dev/docs/claude-cli-stream-json-protocol.md`. Формат пользовательского сообщения: `{"type": "user", "message": {"role": "user", "content": "текст"}}`. НЕЛЬЗЯ использовать `user_message` — Claude CLI молча зависнет без ошибки
- Telegram ограничивает сообщения до 4096 символов — длинные ответы разбиваются с починкой HTML-тегов
- Watcher и handle_message координируются через механизм паузы (pause_session/resume_session в session_watcher), чтобы не дублировать ответы
- Session_id может измениться в любом событии от Claude — нужно обновлять ключи во всех словарях
- Глобальная файловая блокировка `~/.claude-manager.lock` через fcntl.flock() — защита от запуска двух копий бота (единый путь для LaunchAgent, watch_and_restart.sh и ручного запуска)
- При подключении к сессии через `/N` бот создаёт ОТДЕЛЬНЫЙ процесс с `--resume`, а не подключается к терминальному. Терминальная сессия не видит сообщения от бота — они работают через общий файл на диске

## Стандарты разработки скиллов и пайплайнов

### Принцип enforcement-first

Если правило критично для работы пайплайна — оно должно проверяться скриптом или валидатором, а не только текстовой инструкцией. Claude может «забыть» текстовое правило после нескольких тяжёлых этапов. Скрипт не забудет.

Примеры:
- Порядок переходов между этапами — проверять через лог (orchestrator-log.json): если предыдущий этап не записан как завершённый, следующий не запускается
- Обязательные секции в файле — проверять через `structural-validator.sh`
- Формат данных — проверять через JSON-схемы

### CLI-архитектура пайплайнов

Межскилловые вызовы (когда один скилл запускает другой) идут через CLI-подпроцесс (`claude -p`), а не через Skill/Agent tool. Причина: Agent tool ограничен одним уровнем вложенности — скилл, запущенный через Skill tool, не может запускать собственных агентов через Agent tool.

Стандартный набор флагов для CLI-вызова:

```bash
source ~/.claude/cli-budgets.env 2>/dev/null || true
env -u CLAUDECODE claude -p \
  --output-format text \
  --effort max \
  --dangerously-skip-permissions \
  --max-budget-usd "${BUDGET_NORMAL:-5}" \
  <<'PROMPT'
...
PROMPT
```

Бюджеты хранятся в одном месте — `~/.claude/cli-budgets.env`. Там три категории: `BUDGET_LIGHT` (session-report и лёгкие отчёты), `BUDGET_NORMAL` (стандартные агенты-исполнители), `BUDGET_HEAVY` (вложенные тяжёлые пайплайны). Правь только этот файл — нигде больше числовых литералов бюджета быть не должно. Fallback `:-5` в команде защищает на случай отсутствия файла.

Требования к CLI-готовности пайплайн-скиллов:
- Все межскилловые вызовы — через `claude -p`, не через Skill/Agent tool
- Бюджет подставляется через `$BUDGET_LIGHT`/`$BUDGET_NORMAL`/`$BUDGET_HEAVY` из `cli-budgets.env`
- Перед bash-блоком — строка `source ~/.claude/cli-budgets.env 2>/dev/null || true`
- Проверка результата после каждого CLI-вызова — не доверять слепо

### Единый контракт данных

Форматы данных (evals, логи, артефакты) стандартизированы через JSON-схемы и референсы в `~/.claude/references/`. Это предотвращает фрагментацию — когда каждый скилл изобретает свой формат, а потом скрипты агрегации не могут их обработать.

Ключевой контракт: `~/.claude/references/evals-schema.md` — единая структура тестовых наборов (evals.json) для всех скиллов.

### Валидация скиллов

Каждый пайплайн-скилл должен соответствовать `AGENTS.md` и глобальным референсам. Соответствие проверяется `structural-validator.sh` — без прохождения валидации скилл считается незавершённым.

Валидатор проверяет: frontmatter, лимит строк, наличие файлов агентов, ссылки на обязательные референсы, session-report.

### Пропорциональность тестирования

Уровень тестирования соответствует масштабу изменений:
- **Простые правки** (добавить ссылку, обновить формулировку) — визуальная верификация
- **Структурные изменения** (новый этап, изменение логики агентов) — полное сравнительное тестирование по протоколу из `~/.claude/references/skill-testing-standard.md`
- **Новый скилл** — обязательное сравнительное тестирование new_skill vs baseline
