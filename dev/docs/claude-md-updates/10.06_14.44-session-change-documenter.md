# CLAUDE.md update: структура process lifecycle после разбиения process_manager

**Дата:** 10-06-2026
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter

## Затронутая секция

Обновлены секции `Структура проекта`, `Архитектурные принципы` и `Состояние в памяти`.

## Было

`CLAUDE.md` описывал `process_manager.py` как единый владелец жизненного цикла процессов Claude:

```markdown
process_manager.py    # Жизненный цикл процессов Claude (запуск, ретраи, /stop, прогресс)
process_state.py      # In-memory state процессов CLI: процессы, busy-флаги, stop-events, alias temp→real
```

В архитектурном описании инфраструктурный слой перечислялся как `process_manager`, `process_state`, `claude_runner`, а в разделе состояния было сказано, что `process_manager` остаётся владельцем запуска, retry и `/stop`.

## Стало

`CLAUDE.md` теперь описывает `process_manager.py` как compatibility facade, а реальную process lifecycle логику раскладывает по профильным модулям:

```markdown
process_manager.py       # Compatibility facade для старых импортов process lifecycle
process_types.py         # Общие типы и результаты process lifecycle
process_events.py        # Чтение stream-json событий, progress/result и обновление session_id
process_stop.py          # /stop, стратегия остановки процессов и stop_all_processes
process_lifecycle.py     # Создание temp session_id, запуск subprocess и restart
process_retry.py         # Retry-loop, ожидание с проверкой /stop и permanent-error classification
process_send.py          # send_message dispatcher и legacy Claude send path
process_backend_send.py  # Backend-aware Claude/Codex send path
process_state.py         # In-memory state процессов CLI
```

Раздел `Состояние в памяти` теперь прямо говорит, что старые импорты идут через `process_manager`, а поведение запуска, retry, чтения событий и `/stop` разнесено по `process_lifecycle`, `process_retry`, `process_events`, `process_stop`, `process_send` и `process_backend_send`.

## Причина

После предыдущих правок `process_manager.py` был больше 1000 строк и уже считался срочным кандидатом на разбиение. Рефакторинг уменьшил файл до тонкого фасада и сохранил внешний контракт, поэтому будущие изменения в retry, `/stop` или чтении событий можно делать в маленьких файлах без повторного раздувания монолита.
