# CLAUDE.md update: горячие пути не сканируют внешнюю историю повторно

**Дата:** 30-05-2026
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter

## Затронутая секция

`CLAUDE.md` — раздел «Архитектурные принципы», новый подраздел «Горячие пути не сканируют внешнюю историю повторно».

## Было

В `CLAUDE.md` уже были правила про привязку рабочей директории к запросу и однопользовательский инвариант, но не было явного запрета на повторный обход внешней истории CLI в командах, которые блокируют ответ пользователю.

Из-за этого для Codex можно было случайно вернуть старый паттерн: каждый вызов `/pN`, reset watcher-а или pending-сбор заново открывает глобальную папку `~/.codex/sessions/YYYY/MM/DD/` и читает `session_meta` у тысяч rollout-файлов.

## Стало

Добавлен принцип:

```markdown
### Горячие пути не сканируют внешнюю историю повторно

Команды, которые блокируют ответ пользователю (`/pN`, `session_watcher.reset_state`, watcher `poll_once`, pending-сбор), не должны повторно обходить глобальную историю внешних CLI. Для Codex это особенно важно: rollout-файлы лежат в общей папке `~/.codex/sessions/YYYY/MM/DD/`, а проект определяется только через `session_meta.payload.cwd` внутри JSONL. Поэтому горячий листинг с `lookback_days` должен идти через `codex_session_index`: лёгкий in-memory индекс файлов за operational-окно. Полные scan-ы допустимы только в совместимых legacy-ветках, диагностике или нечастых UI-сценариях, где это явно обосновано.
```

## Причина

RCA по медленному переключению проектов показал, что задержка 7-17 секунд возникала не из-за Telegram и не из-за команды `/pN` как таковой. Практическая причина была в повторных full scan-ах Codex rollout-файлов на горячем пути.

Новый принцип нужен, чтобы будущие изменения не вернули тот же класс проблемы: если команда блокирует ответ Ивану, она должна использовать быстрый operational index или другой дешёвый источник, а не заново перечитывать внешнюю историю CLI.
