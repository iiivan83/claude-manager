# CLAUDE.md update: post-flight проверяет Python-процесс бота

- **Дата:** 14-05-2026
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter

## Затронутая секция

Проектный `CLAUDE.md` — раздел **Принципы эксплуатации**, правило **Принцип verify-before-and-after для рестарта сервисов**.

## Было

Правило требовало делать pre-flight и post-flight при рестарте через `launchctl kickstart -k`, но описывало post-flight слишком общо: `launchctl list | grep` и `tail` логов через 5–10 секунд.

Этого оказалось недостаточно для сервисов, запущенных через shell-обёртку: `launchctl list` показывает PID обёртки, а не гарантирует, что реальный Python-бот уже поднялся.

## Стало

Правило уточнено: для launchd-сервисов, запущенных через shell-обёртку, post-flight должен проверять и PID обёртки, и дочерний процесс реального приложения.

Для claude-manager успешный restart означает, что под `~/.local/bin/start-claude-manager.sh` появился Python-процесс `claude_manager`. Stale `last exit code = -15` не считается провалом, если Python-бот уже работает.

## Причина

14-05-2026 `restart-claude-manager.sh` ложно падал после успешного restart из-за старого `last exit code = -15`. Первый вариант проверки только по PID wrapper-а тоже был недостаточен: wrapper может быть жив во время retry-паузы, пока Python-бот ещё не запущен.

## Связанные файлы

- **`restart-claude-manager.sh`** — реализована проверка дочернего Python-процесса.
- **`tests/test_restart_claude_manager_script.py`** — добавлены регрессионные тесты post-flight.
- **`dev/docs/adr/14.05_16.12-session-change-documenter-restart-postflight-python-process.md`** — зафиксировано архитектурное решение.
