# Сессия 10-06: root-cause анализ — resume ушёл в чужую сессию + «пустой ответ»

## Коротко

Прогнали пайплайн `root-cause-analysis` (скилл глубокого поиска первопричин) по инциденту
10-06 в проекте `bloger`: после команды «стоп» продолжение разговора попало в чужую
сессию (в «куратора» вместо «форматов постов»), а параллельно ~20 минут сыпались ошибки
про пустой/отсутствующий ответ. Пайплайн нашёл первопричины обоих симптомов и записал
отчёт с чек-листом исправлений. **Кодовых изменений в этой сессии нет** — только
исследование и документ.

Первопричины:

- **Симптом 1 (сообщение ушло в чужую сессию).** Пользователь всё утро работал только
  Telegram-реплеями (ответами на конкретные сообщения бота). Reply-роутинг доставляет
  текст в сессию-источник, но **не меняет активную привязку чата** (запись «этот чат
  сейчас работает с этой сессией» в `session_manager`). Когда пользователь отправил
  обычное сообщение без реплея, оно ушло в активную привязку — а та со вчерашнего
  вечера указывала на куратора. Корень: два понятия «текущей сессии» (у пользователя —
  разговор, которого он касался последним; у бота — последняя явная привязка)
  разошлись без какой-либо обратной связи.
- **Симптом 2 (повторяющийся «пустой ответ»).** Два retry-цикла работали на одном
  общем слоте сессии и каждые ~60 секунд убивали процессы друг друга: шаг «останови
  старый процесс из слота» в `_execute_single_retry` не проверяет, чей это процесс.
  Корень — тот же, что у починенной ранее гонки stop-флага: у слота сессии не было
  понятия «владелец хода». Дополнительный слой: Claude CLI при `--resume` оборванного
  хода вставляет синтетическую пару «Continue from where you left off.» → «No response
  requested.» — за время дуэли таких пар записалось ~18 штук.

Гонка stop-флага (старый retry-цикл воскресал после «стоп») была починена в предыдущей
сессии — этот пайплайн разбирал только два оставшихся симптома.

## Решения

- **Кодовые правки не делались намеренно** — пайплайн только исследует; все исправления
  вынесены в чек-лист RCA-отчёта.
- **Owner-проверка перед остановкой процесса в ретрае** (`src/claude_manager/process_retry.py:60-62`) —
  одобрено: продолжение owner-механизма, уже введённого фиксом stop-флага.
- **Фильтр resume-placeholder в progress-пути** — одобрено с замечанием: фильтровать
  только точное совпадение строки «No response requested.», как уже делает poller.
- **Единая константа маркера** вместо 4 разъехавшихся дублей (`claude_interaction`,
  `process_types`, `claude_code_session_file_reader`, `coding_agent_session_file_poller`) +
  обязательный grep-проход по всем потребителям.
- **Закрытие расхождения «текущей сессии» — решение за пользователем**, два варианта:
  reply-роутинг обновляет активную привязку («куда ответил — там и работаю») ИЛИ
  мгновенный ack «→ #1 куратор» при приёме обычного сообщения.
- **Два новых принципа для CLAUDE.md**: «один владелец хода на сессию» (операции над
  общими слотами проверяют тождество владельца, не только ключ сессии) и «reply-роутинг
  не меняет активную привязку; любая новая фича адресации обязана явно решить вопрос
  привязки».
- **Паттерн повторных инцидентов**: три RCA-отчёта об одном модуле — сигнал для
  ретроспективы процесса; скилл `development-process-retrospective` сейчас лежит только
  в `.claude/skills/_archive/` и не срабатывает.

## Риски и ограничения

- **Не доказуемо, какую формулировку «пустого ответа» видел пользователь** — бот не
  логирует доставку сообщений. Это уже третья рекомендация добавить логирование доставки
  (08-06 RCA, отчёт 09-06, теперь этот) — так и не реализована.
- **Контракт stdout не проверен эмпирически**: попадает ли синтетическая пара
  «Continue…» → «No response requested.» в stdout-поток событий или только в файл сессии —
  неизвестно. Перед фиксом фильтра progress-пути обязательна проверка реальным запуском
  CLI (правило CLAUDE.md «контракты проверяются эмпирически»).
- **Остаточный риск дуэли**: фикс stop-флага закрыл воскрешение цикла, но
  `process_retry.py:60-62` по-прежнему убивает процесс из слота без проверки владельца.
- В рабочем дереве лежат незакоммиченные правки других сессий
  (`src/claude_manager/claude_code_session_file_reader.py`, `tests/test_claude_code_backend.py`,
  архив скиллов `_archive/`, обновления docs-index) — этот пайплайн их не трогал и ничего
  не коммитил.

## Продолжение

Точка входа — раздел «Чек-лист исправлений» в RCA-отчёте. Порядок, который напрашивается:

1. **[CODE]** Owner-проверка перед остановкой `old_process` в
   `src/claude_manager/process_retry.py:60-62` — закрывает остаточный риск дуэли циклов.
2. **[CODE]** Эмпирически проверить попадание «No response requested.» в stdout, затем
   фильтр в progress-пути (`process_events.py` или `claude_code_backend.py`) — только
   точное совпадение строки.
3. **[CODE]** Единая константа маркера + `grep -r "No response requested"` и правка всех
   потребителей.
4. **[CODE]** Логирование каждой отправки пользователю (chat_id, заголовок, is_final,
   первые 80 символов) в `telegram_sender` или `telegram_response_delivery`.
5. **[CODE, спросить пользователя]** Выбор: rebind активной привязки при reply-роутинге
   ИЛИ мгновенный ack «→ #N имя-сессии» для обычных сообщений.
6. **[DOC]** Два принципа в CLAUDE.md (владелец хода; правило активной привязки) +
   дополнить `dev/docs/claude-cli-stream-json-protocol.md` про resume-placeholder при
   оборванном `tool_use` и пару на каждый retry-resume.

   правило «3+ RCA-отчёта об одном модуле → ретроспектива» реально срабатывало.
