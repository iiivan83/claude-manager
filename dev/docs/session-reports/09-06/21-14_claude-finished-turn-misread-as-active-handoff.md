# Handoff: ответы Claude теряются — завершённый ход читается как активный

## Коротко

В Telegram-боте `claude_manager` (проект `bloger`) часть финальных ответов Claude
не доходит до пользователя, при этом Codex-ответы приходят нормально. Причина
найдена и эмпирически подтверждена, но **фикс ещё не сделан** — эта сессия только
диагностика. Следующая сессия реализует надёжное исправление по плану ниже.

Суть бага: включён режим тишины (доставляются только финальные сообщения), а
читатель JSONL-файлов Claude ошибочно считает завершённый ход активным. Из-за этого
финальный ответ помечается как «не финал», и режим тишины его выбрасывает. У Codex
этого нет, потому что Codex пишет явную отметку завершения хода, а внешние сессии
Claude Code — нет.

**Точный триггер (подтверждён пользователем и кодом).** Проблема возникает именно
когда пользователь делает Telegram-**реплей** на сообщение бота. Обычное (не-reply)
сообщение идёт прямым путём и доходит. Реплей короткозамыкает прямой путь и уходит в
`reply_route_handler`, который доставку финального ответа делегирует watcher-у, а
watcher для Claude сломан (см. ниже). Поэтому «сообщения не перестают приходить, пока
не сделаешь реплей».

## Симптом (со слов пользователя)

- Создаёт сессии через бота в проекте `bloger`, задаёт вопросы/ставит задачи.
- Часть ответов Claude видна, часть просто не приходит.
- В Codex ответы приходят нормально.
- Когда пишет вопрос и сразу видит, что Claude начал отвечать, — ответ доходит.

Разделение «доходит / не доходит» соответствует двум путям доставки, и выбор пути
зависит от того, был ли это реплей:
- **Прямой путь** (обычное сообщение, не реплей) — `telegram_input_handlers.handle_message`
  вызывает `claude_interaction.send_to_claude_and_respond`, тот отдаёт `result.text` с
  `is_final=True`, режим тишины пропускает → пользователь видит.
- **Путь реплея → watcher** (сообщение-реплей на сообщение бота) —
  `telegram_input_handlers.handle_message` отдаёт управление
  `reply_route_handler.try_handle_text_reply` и сразу выходит (строки 109-110), прямой
  путь не выполняется. Фоновая отправка `_send_routed_text_in_background` гоняет Claude,
  но при успехе **не доставляет `result.text`** — обрабатывает только ошибку
  (`if result.is_error`, строка 248). Доставка финала целиком ложится на watcher, а
  watcher для Claude помечает финал `is_final=False` из-за бага → режим тишины
  выбрасывает.

## Корневая причина (эмпирически подтверждена)

**1. Режим тишины включён.** Файл `~/.claude-manager-silence-mode` = `{"enabled": true}`.
В режиме тишины доставляются только сообщения с `is_final=True`.

**2. Читатель Claude ошибочно держит завершённый ход активным.**
В `src/claude_manager/claude_code_session_file_reader.py` активность хода считается
обратным проходом по записям: ход закрыт только если встретилась запись `result`
раньше записи `assistant`/`progress`/`queue-operation` (`BUSY_EVENT_TYPES`,
строка 27). Затронуты две функции:
- `_compute_is_turn_active_from_parsed_records` (полный snapshot-reader) — строка 373.
- `_read_cursor_count_last_record_and_turn_active` (быстрый cursor-reader) — строка 453.

**3. Контракт внешних Claude Code-сессий другой.** Внешние сессии Claude Code в
`~/.claude/projects/...` завершают ответ записью `assistant` со
`stop_reason=end_turn` и **без** записи `result`. После неё могут идти служебные
записи `last-prompt`, `ai-title`, `mode`. Обратный проход никогда не находит
`result`, упирается в `assistant` → возвращает «ход активен» навсегда.

**4. Цепочка подавления.** При `is_turn_active=True`:
- `coding_agent_session_file_poller.py:314` — `hold_final_message=True`.
- `coding_agent_session_file_poller.py:330-334` — последнее (финальное) сообщение
  исключается из доставки.
- `coding_agent_session_file_poller.py:365` — `is_final` принудительно `False`
  для всех остальных доставляемых сообщений.
- `telegram_response_delivery.py:269` / `:307` / `:345` — в режиме тишины всё
  не-финальное выбрасывается (`if not is_final and silence_mode_registry.is_enabled(): return`).

Итог: одиночный финальный ответ через watcher не доставляется вообще (он и есть
последний индекс, который придерживается), а более ранние сообщения хода — только
как «не финал», то есть в режиме тишины тоже выбрасываются.

## Почему Codex работает

`codex_session_file_reader.py:204` — `_compute_is_turn_active_for_codex` смотрит на
последнюю валидную запись и проверяет явный терминальный маркер
(`is_turn_terminal_session_record`). Codex пишет такую отметку в конце каждого хода,
поэтому завершённый ход читается как `is_turn_active=False`, финал проходит через
режим тишины. Claude такой отметки во внешних сессиях не пишет.

## Эмпирическая проверка контракта

Проверены 6 последних JSONL-файлов сессий Claude в
`~/.claude/projects/-home-ivan-claude-sandbox-bloger`:
- ни в одном нет записи `result`;
- несколько заканчиваются записью `assistant` со `stop_reason=end_turn` (завершённый
  ответ), например файлы `4bfc114b`, `9f9fef57`, `bca38223`, `d05f269c`;
- самый свежий `fa39b8f4` реально в середине хода (`stop_reason=tool_use`) — для него
  `is_turn_active=True` корректно, его трогать нельзя.

Это подтверждает: дефект бьёт именно по завершённым внешним ходам, а реально активные
ходы (tool_use в середине) должны остаться активными.

## Решения

- Триггер — реплей. Фикс читателя чинит watcher, от которого зависит путь реплея, и
  закрывает симптом целиком. Отдельно рассмотреть, не должен ли
  `_send_routed_text_in_background` доставлять `result.text` напрямую (как прямой путь),
  чтобы не зависеть от watcher-а вовсе — но тогда нужна защита от двойной доставки, если
  watcher тоже доставит этот же финал.
- Решаем **надёжно через фикс читателя**, а не полумерой `Silence off`.
- Правило именования и место handoff-отчёта соблюдены:
  `dev/docs/session-reports/09-06/`.
- Фикс затрагивает контракт с внешним инструментом (формат JSONL Claude Code),
  поэтому правка должна опираться на эмпирически подтверждённый контракт (см. выше),
  а не на догадку — это требование CLAUDE.md проекта.

## Риски и ограничения

- **Не сломать активные ходы.** Запись `assistant` со `stop_reason=tool_use` плюс
  последующий `user` с `tool_result` без `result` обязана остаться
  `is_turn_active=True` — это реально незавершённый ход с инструментами.
- **Чинить оба читателя.** snapshot-reader и cursor-reader используют разную логику
  прохода; правка только одного оставит баг в половине сценариев (watcher poll и
  reset_state идут через cursor-reader).
- **Служебные записи после финала.** `last-prompt`, `ai-title`, `mode`,
  `file-history-snapshot` после `end_turn` не должны снова делать ход активным.
- **Полумера `Silence off` неполная.** Промежуточные сообщения станут видны, но
  одиночный финал через watcher всё равно придерживается из-за той же ошибки.
- **Текущий проект и `/all`.** Даже после фикса обычный watcher смотрит только
  текущий проект (`config.WORKING_DIR`). Для живых уведомлений из `bloger`, когда
  активен другой проект, нужен режим `/all`. Старые pending-сообщения живут 3 часа.
- **Логи не доказывают Codex-кейсы.** Успешная/подавленная watcher-доставка сейчас
  не логируется, поэтому Codex-сессии #27/#33/#34 из вчерашнего RCA закрыть по логам
  нельзя без добавления логирования.
