# Сессия 31-05: handoff по event-driven индексу сессий

## Коротко

В этой сессии разобрали, почему ускорения переключения и режима `/all` всё ещё
недостаточно. Главный вывод: бот не должен каждый раз сканировать всю историю
сессий. Его лучше строить как мессенджер: держать индекс сессий, проверять только
изменившиеся файлы и использовать события файловой системы как сигнал.

Для Ивана это значит: правильное направление не просто “взять последние 30-40
сессий”, а перейти к модели “есть список свежих сессий, и он обновляется только
когда что-то изменилось”. Старые сессии не должны тормозить обычный Telegram UX.

## Рабочие файлы

- **`CLAUDE.md`** — главный документ проекта. В нём уже есть принцип, что горячие
  пользовательские пути не должны повторно сканировать внешнюю историю. Сейчас
  этот принцип нужно распространить на `/sessions` и `/all`.
- **`src/claude_manager/codex_session_index.py`** — текущий in-memory индекс Codex.
  Индекс уже умеет один раз прочитать служебные записи сессий и потом быстро
  возвращать список файлов проекта.
- **`src/claude_manager/codex_session_file_listing.py`** — модуль поиска Codex
  rollout-файлов. Rollout-файл — это JSONL-история одной Codex-сессии. В этом
  модуле сейчас остаются user-facing и legacy scan-ветки.
- **`src/claude_manager/all_projects_monitor.py`** — режим `/all`, который следит
  за сообщениями из всех проектов. Сейчас он собирает слишком широкий список
  сессий и поэтому остаётся главным кандидатом на ускорение.
- **`src/claude_manager/project_pending_delivery.py`** — pending-доставка при
  возврате в проект. Здесь уже применён mtime/cursor-first подход: если файл не
  менялся, полный snapshot не читается.
- **`dev/docs/logs/root-cause-reports/30-05_18-32_slow-project-switch.md`** —
  RCA медленного переключения проектов. Объясняет, почему повторный global scan
  Codex-истории был корнем задержки.
- **`dev/docs/logs/root-cause-reports/31-05_02-47_slow-session-list.md`** —
  RCA медленного `/sessions`. Объясняет, почему user-facing список тоже не должен
  оставаться на повторном scan-е.
- **`dev/docs/adr/30.05_19.28-session-change-documenter-codex-operational-session-index.md`**
  — ADR по уже реализованному operational index.
- **`dev/docs/adr/31.05_03.37-session-change-documenter-codex-session-list-two-day-hotfix.md`**
  — ADR по временному двухдневному hotfix для `/sessions`.

## Решения

- **Модель должна стать похожей на мессенджер.** Мессенджеры не перечитывают все
  чаты при каждом открытии. Они держат локальный индекс, получают события о новых
  сообщениях и обновляют только изменившийся чат. Для Claude Manager сессия
  равна чату, файл сессии равен истории чата, а `mtime` равен признаку “в этой
  сессии что-то поменялось”.
- **Сканирование должно быть metadata-first.** Сначала читается только “шапка”
  сессии: session id, project path и время изменения. Полный текст читается
  только когда файл реально изменился.
- **`/all` должен быть ограниченным inbox-режимом.** Режим должен следить за
  последними 30-40 активными сессиями, а не за всеми 1700+ найденными сессиями.
  Если старая сессия не попадает в активное окно, её сообщения не теряются:
  при входе в конкретный проект pending-механизм сможет их доставить.
- **Обновление списка и проверка сообщений нужно разделить.** Проверка 40
  известных файлов может идти часто, например каждые 2 секунды. Пересборка
  списка свежих сессий должна идти реже или по событию.
- **Event-driven слой желателен, но не без страховки.** На Linux можно использовать
  события файловой системы, например inotify: “файл создан” или “файл изменён”.
  Но полностью полагаться только на события нельзя: после рестарта, переполнения
  очереди или смены дня нужен редкий фоновый resync.

## Риски и ограничения

- Код в этой сессии не менялся. Это handoff-отчёт и дизайн-направление, не
  реализация.
- Время в замерах зависит от текущей нагрузки диска, но порядок величин
  подтверждён: главная проблема в алгоритме, а не в Telegram.
- Чистый “event-only” подход опасен без fallback. Файловые события могут быть
  пропущены, поэтому нужен редкий resync.
- Нельзя просто добавить крупный код в текущие большие файлы:
  - `src/claude_manager/all_projects_monitor.py` уже 633 строки;
  - `src/claude_manager/codex_session_file_listing.py` уже 499 строк;
  - `src/claude_manager/codex_session_index.py` уже 295 строк.
  Новую логику лучше выносить в отдельные модули.
- Если `/all` будет следить только за последними 30-40 сессиями, это меняет
  практический контракт режима: он становится “последние активные”, а не “вся
  история всех проектов”.

## Продолжение

1. Написать короткую спецификацию на event-driven session index.
2. Ввести общий metadata index для `/all` и `/sessions`: session id, backend,
   project path, file path, `last_modified_at`, lightweight cursor.
3. Сделать `/all` bounded: активное окно последних 30-40 сессий, а не все
   найденные сессии.
4. Разделить fast poll и discovery:
   - fast poll часто проверяет только известные активные файлы;
   - discovery редко или по событию обновляет список активных сессий.
5. Добавить Linux file-event слой через inotify/watchdog/watchfiles или другой
   небольшой механизм, совместимый с текущей Linux-only эксплуатацией.
6. Оставить редкий background resync как страховку от пропущенных событий.
7. Добавить regression tests:
   - повторный `/all` poll не делает full scan;
   - неизменившийся файл не читается целиком;
   - новая сессия попадает в индекс без ручного scan-а всех файлов;
   - cap 30-40 реально ограничивает baseline и poll;
   - fallback resync восстанавливает индекс после пропущенного события.
