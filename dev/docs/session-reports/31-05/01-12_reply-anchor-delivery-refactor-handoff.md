# Handoff: подготовка доставки ответов перед reply-якорями

## Коротко

В этой сессии проверили, что работа по reply-якорям уже началась не с самой функции, а с подготовительного рефакторинга.

Reply-якорь — это Telegram `message_id` входящего сообщения Ивана, на которое бот потом должен ответить реплаем. Правильный якорь должен привязываться не к последнему сообщению в чате вообще, а к сообщению, которое реально было принято в конкретную сессию Claude или Codex.

Текущий незакоммиченный код выносит доставку ответов из большого `bot.py` в отдельный модуль. Это важно перед reply-якорями: дальше нужно будет менять правила отправки сообщений, и безопаснее делать это в небольшом модуле доставки, а не раздувать основной Telegram-обработчик.

Проверка показала, что перенос доставки не ломает существующие сценарии: целевые тесты прошли, `172 passed`.

## Рабочие файлы

- **`CLAUDE.md`** — главный документ проекта. Он объясняет, что `claude_manager` — локальный Telegram-бот для управления Claude Code и Codex с телефона, а также описывает архитектурные границы.
- **`docs/superpowers/plans/2026-05-30-reply-anchor-v1-implementation.md`** — план реализации reply-якорей. В нём описано, что нужно хранить ключ `project_path + backend + session_id -> message_id`.
- **`dev/docs/session-reports/30-05/20-51_reply-anchor-design-handoff.md`** — предыдущий handoff по дизайну reply-якорей. Он фиксирует смысл фичи, спорные сценарии `/all`, `/stop`, переключения проектов и сессий.
- **`src/claude_manager/bot.py`** — основной Telegram-слой: принимает команды и сообщения. В текущем diff из него убрана часть форматирования и отправки ответов.
- **`src/claude_manager/telegram_response_delivery.py`** — новый модуль доставки ответов. Он форматирует заголовки, обрабатывает файловые маркеры, учитывает silence mode и отправляет части сообщений в Telegram.
- **`src/claude_manager/claude_interaction.py`** — оркестрация одного запроса в Claude/Codex. В текущем diff изменены комментарии и callback-регистрация так, чтобы доставка шла через новый модуль.
- **`tests/test_telegram_response_delivery_behavior.py`** — новые поведенческие тесты доставки. Они проверяют заголовки, пустые ответы, файловые маркеры, silence mode и callback-разрешение.
- **`tests/test_bot.py`** — существующие тесты Telegram-слоя. Они адаптированы к тому, что функции доставки теперь импортируются из нового модуля.
- **`tests/test_claude_interaction.py`** — существующие тесты оркестрации запросов. Они адаптированы к новому delivery-модулю.
- **`dev/docs/docs-index.md`** — индекс документации. В этой сессии он обновляется, чтобы новый отчёт было проще найти.

## Решения

- Текущий код считать подготовительным рефакторингом, а не реализацией reply-якорей.
- Следующий практичный шаг — сначала закрепить зелёный перенос доставки отдельным чекпоинтом или коммитом, затем добавлять reply-якоря поверх него.
- Делегирование агенту для отчёта не использовалось: текущие runtime-инструкции разрешают запуск sub-agent только по явной просьбе пользователя.
- Для будущей реализации reply-якорей остаётся актуальным ограничение по размеру файлов. Нельзя молча дописывать большие модули, потому что в проекте действует size gate.

Текущее состояние size gate:

- **`src/claude_manager/bot.py`** — 1279 строк, 18 публичных функций. Файл выше порога 500 строк и выше порога 10 публичных функций.
- **`src/claude_manager/claude_interaction.py`** — 543 строки, 11 публичных функций. Файл выше порога 500 строк и выше порога 10 публичных функций.
- **`src/claude_manager/telegram_response_delivery.py`** — 253 строки, 4 публичные функции. Файл пока ниже порога 300 строк, но до warning-порога осталось 47 строк.

## Проверки

- Проверили рабочее дерево через `git status --short`. Это важно, чтобы новая сессия увидела, какие файлы уже изменены и не перепутала текущие правки с чистым состоянием.
- Проверили общий размер diff через `git diff --stat`. На момент проверки в отслеживаемых файлах было `53 insertions` и `232 deletions`, плюс новые untracked-файлы.
- Проверили размеры и количество публичных функций через `wc -l` и `grep -Ec`. Это нужно из-за активного правила проекта: большие кодовые файлы нельзя молча раздувать.
- Попытка запустить тесты через `python -m pytest ...` не прошла, потому что в shell нет команды `python`.
- Целевые тесты успешно запущены через проектный venv:

```bash
.venv/bin/python -m pytest tests/test_telegram_response_delivery_behavior.py tests/test_bot.py tests/test_claude_interaction.py -q
```

Результат:

```text
172 passed in 1.64s
```

## Риски и ограничения

- Полный набор тестов проекта в этой сессии не запускался. Проверены только тесты, связанные с текущим переносом доставки.
- Reply-якоря ещё не реализованы: нет `reply_anchor_registry`, не протянут incoming `message_id`, не обновлён `telegram_sender` для reply-параметров.
- Большие файлы остаются проблемой. Новые изменения в `bot.py` и `claude_interaction.py` должны быть минимальными и явно отмеченными.
- Новый `telegram_response_delivery.py` уже близко к 300 строкам. Если reply-якоря потребуют большой логики доставки, лучше вынести отдельные helper-модули, а не складывать всё туда.
- В отчёт не включались секреты, токены, строки подключения или содержимое `.env`.
- В рабочем дереве есть другие untracked-документы и логи, которые не создавались в рамках этого отчёта. Их не нужно удалять или переписывать без отдельного решения.

## Продолжение

1. Принять решение по текущему refactor-чекпоинту: оставить как отдельный коммит или продолжать в том же наборе правок.
2. Если продолжаем reply-якоря, начать с TDD: создать тесты и модуль `reply_anchor_registry`.
3. Добавить поддержку optional reply-параметров в `telegram_sender`, включая fallback на обычную отправку, если Telegram отклонит reply metadata.
4. Протащить Telegram `message_id` из входящего update через `bot.py` в `claude_interaction.py`, но обновлять якорь только после принятия сообщения в Claude/Codex.
5. Подключить чтение якоря в прямой доставке ответа, watcher-сообщениях, `/all`-доставке и pending-доставке.
6. Перед каждой правкой кодовых файлов снова проверить size gate. Для `telegram_response_delivery.py` особенно важно не превысить 300 строк без явного сигнала.
7. После реализации запустить сначала целевые unit-тесты, затем полный `tests/`, а E2E через Telethon оставить отдельным шагом, если нужны реальные Telegram-реплаи.
