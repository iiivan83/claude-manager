# Сессия 31-05: handoff по простой таблице recent sessions

## Коротко

В этой сессии уточнили идею ускорения `/sessions`, `/all` и переключения между
проектами. Первую версию лучше делать проще, чем event-driven архитектуру:
достаточно одной лёгкой таблицы `recent_sessions`, где по каждому проекту хранятся
последние 30 сессий.

Для Ивана это значит: в следующей сессии нужно писать спецификацию не на большую
систему индексов, а на прагматичную v1. Список сессий и `/all` должны брать данные
из готовых заголовков, а не каждый раз заново обходить файловую историю.

## Рабочие файлы

- **`dev/docs/session-reports/31-05/14-44_event-driven-session-index-handoff.md`** —
  предыдущий handoff. Он фиксирует исходную проблему: повторное сканирование всей
  истории сессий делает `/all`, `/sessions` и переключение проектов медленными.
- **`src/claude_manager/coding_agent_backend.py`** — общий контракт backend-ов
  Claude/Codex. В нём уже есть `SessionFileInfo` с временем изменения файла
  сессии, это основа для быстрого сравнения "менялось / не менялось".
- **`src/claude_manager/codex_session_index.py`** — существующий оперативный индекс
  Codex-сессий. Он показывает, что metadata-first подход уже частично встроен в
  проект.
- **`src/claude_manager/codex_session_file_listing.py`** — текущая логика поиска
  Codex rollout-файлов. Именно отсюда нужно убирать повторный широкий scan для
  горячих пользовательских путей.
- **`src/claude_manager/all_projects_monitor.py`** — режим `/all`, который следит
  за сессиями во всех проектах. В первой версии он должен читать активные сессии
  из `recent_sessions`.
- **`src/claude_manager/telegram_session_handlers.py`** — Telegram-команда
  `/sessions`. В первой версии она тоже должна брать список из `recent_sessions`,
  чтобы список открывался быстро.
- **`src/claude_manager/project_pending_delivery.py`** — уже реализованный пример
  правильного подхода: сначала проверяется время изменения, потом лёгкий cursor,
  и только затем читается полный snapshot.

## Решения

- **Первая версия должна быть простой.** Не начинаем с отдельной сложной
  event-driven системы. Сначала делаем одну таблицу заголовков последних сессий.
- **`recent_sessions` хранит последние 30 сессий на проект.** Это не вся история,
  а рабочий список свежих сессий, которые вероятнее всего нужны Ивану в Telegram.
- **В таблице хранятся только заголовки.** Минимальный набор: backend, project path,
  session id, file path, время последнего изменения, короткий preview и позиция
  чтения.
- **`/sessions` должен читать эту таблицу.** Список сессий не должен заново
  сканировать файлы ради показа последних сессий текущего проекта.
- **`/all` должен читать эту таблицу.** Режим всех проектов должен собирать
  кандидатов из последних 30 сессий каждого проекта, а не из всей исторической
  базы файлов.
- **События файловой системы откладываются.** Inotify/watchdog/watchfiles можно
  добавить позже, если после v1 останутся задержки. Сейчас главный выигрыш должен
  дать отказ от повторного полного scan-а.

## Проверки

- Проведён код-ревью идеи по текущей архитектуре: подход реализуем, потому что
  в проекте уже есть `last_modified_at`, session cursor и Codex operational index.
- Подтверждён главный узкий участок: `/all` каждый цикл заново собирает список
  проектных сессий, а Codex bulk-listing по всем проектам всё ещё идёт широким
  scan-ом.
- Подтверждено, что `/sessions` тоже должен выиграть от `recent_sessions`, если
  preview хранится в таблице и не требует повторного чтения файлов.
- Код в этой сессии не менялся. Это handoff для следующей сессии и будущей
  спецификации.

## Риски и ограничения

- Если хранить только последние 30 сессий и полностью забывать старые, можно
  сломать восстановление состояния старых ссылок или cursor. В спецификации нужно
  решить, где и сколько хранить старые состояния.
- Если в таблице не хранить preview, `/sessions` может снова начать открывать
  файлы ради красивого списка, и часть тормоза вернётся.
- Если проектов много, схема "30 сессий на проект" может дать большой общий список
  для `/all`. В спецификации нужно предусмотреть дополнительный глобальный cap,
  например последние 40-80 активных сессий после объединения.
- Полностью event-only режим пока не нужен и опасен без fallback: события файловой
  системы могут быть пропущены при рестарте или переполнении очереди.
- Нельзя раздувать текущие большие модули. Новую логику `recent_sessions` лучше
  вынести в отдельный модуль, а в существующих файлах оставить тонкую интеграцию.

## Продолжение

1. В следующей сессии написать спецификацию v1 на `recent_sessions`.
2. В спецификации зафиксировать контракт таблицы: поля, лимит 30 сессий на проект,
   правила обновления и срок хранения старых состояний.
3. Описать, как `/sessions` читает таблицу и когда просит обновление проекта.
4. Описать, как `/all` берёт сессии из таблицы, применяет глобальный cap и не
   делает повторный scan всей истории.
5. Описать safety-refresh: редкое обновление последних сессий проекта, чтобы новые
   сессии не терялись даже без файловых событий.
6. Заложить тесты:
   - `/sessions` берёт данные из `recent_sessions`;
   - `/all` не запускает полный historical scan при каждом poll;
   - неизменившаяся сессия не читается целиком;
   - изменившаяся сессия обновляет preview и cursor;
   - лимит 30 на проект соблюдается;
   - глобальный cap для `/all` работает предсказуемо.
