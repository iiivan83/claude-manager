## Решения

- **Оркестратор — полный автомат.** Причина: пользователь хочет минимальное участие, оркестратор идёт от фазы к фазе сам, останавливается только при ошибке
- **Скиллы самостоятельные.** Причина: не зависят от существующих скиллов проекта (make-script-spec, dev-script-implement и т.д.), каждый самодостаточен
- **Вход — готовый BRD.** Причина: BRD уже существует, пайплайн начинается с декомпозиции
- **Ревью — Claude автоматически.** Причина: 4-проходное ревью (качество, безопасность, архитектура, BRD-соответствие) выполняется скиллом review-code
- **E2E без моков — Telethon.** Причина: живое тестирование через реальный Telegram-аккаунт пользователя. Telethon выбран как самая популярная Python-библиотека для user-аккаунтов
- **Юнит/интеграционные — с моками.** Причина: тесты должны быть быстрыми и детерминированными. pytest + pytest-asyncio + pytest-mock
- **LLM-верификация — сейчас, не на будущее.** Причина: пользователь настоял что ничего не откладывать. Claude проверяет ответы бота на соответствие BRD прямо в E2E
- **Универсальные имена скиллов.** Причина: убрать prefix `pipeline-` из универсальных скиллов. Только оркестратор `pipeline-run` сохраняет prefix
- **Реализованные спеки → realized/.** Причина: implement-module после реализации переносит спеку в `development/specs/realized/`
- **Даты в именах дублируемых документов.** Формат: `_ДД-ММ-ЧЧ-ММ`. Причина: несколько прогонов тестирования создают разные файлы
- **Документы тестирования → `development/docs/testing/`.** Причина: отдельная папка для всех тестовых артефактов
- **Всегда скиллы, не вручную.** Причина: для скриптов run-all-tests.sh и check-phase-gate.py добавлены скиллы-генераторы (create-test-runner, create-phase-gate-checker)

### Решения

- **Фаза 0 — валидация BRD первой.** Причина: пользователь показал реальные проблемы в проекте (сломанный pyproject.toml, противоречия «15 vs 5 сессий», отсутствующий FULL-TECHNICAL-SPEC.md) — без аудита эти баги дошли бы до кода
- **Git: коммит после каждого артефакта, пуш по подтверждению.** Причина: без коммитов нет точек отката, а git checkout упомянутый в обработке ошибок не сработает
- **Всё через агентов.** Причина: оркестратор координирует, а не делает работу сам. Параллельные задачи одного слоя — несколько Agent tool вызовов в одном сообщении
- **Интерактивные шаги.** Причина: Telethon SMS, git push, вопросы по BRD — оркестратор останавливается и спрашивает пользователя
- **Возврат спеки из realized/.** Причина: если при реализации выяснилось что спека неправильная — вернуть, исправить, закоммитить
- **Очистка тестовых данных.** Причина: живые тесты создают реальные сессии/файлы/сообщения в Telegram — нужно убирать
- **Открытые направления вместо закрытых списков.** Причина: пользователь попросил чтобы validate-brd искал ВСЕ возможные ошибки, а не только из заданного списка
