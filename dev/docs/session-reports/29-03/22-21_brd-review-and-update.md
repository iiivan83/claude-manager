# Сессия 29-03: ревью BRD и обновление по результатам

## Резюме

Полный ревью BRD (brd-user-journeys.md) с составлением списка технических вопросов и пробелов. Итеративная проверка вопросов через агента-ревьюера (2 цикла). Получены ответы пользователя на все 14 финальных вопросов. BRD, CLAUDE.md и .env.example обновлены по результатам.

## Изменённые файлы

- `development/docs/brd-user-journeys.md` — изменён — 14 правок: удалены процессы/idle timeout/очередь, добавлены обработка ошибок/восстановление/скоуп проекта/дневная нумерация
- `CLAUDE.md` — изменён — удалена эпоха, project_only→полный доступ, удалён пул процессов
- `.env.example` — изменён — удалён IDLE_TIMEOUT_SECONDS

## Решения

- **Архитектура без процессов в BRD**: BRD описывает сессии, а не процессы Claude. Процессы — деталь реализации. **Причина**: пользователь воспринимает систему как переключение между сессиями, а не управление процессами.

- **Удалён idle timeout**: процессы Claude не завершаются по таймауту. **Причина**: бот подключается к сессиям Claude, а не управляет их жизненным циклом.

- **Удалена эпоха**: механизм «эпохи» для защиты от перезаписи сессий удалён из CLAUDE.md и не будет реализован. **Причина**: пользователь решил что он не нужен.

- **Нет очереди сообщений**: сообщения передаются напрямую в текущую сессию Claude без очередей. **Причина**: пользователь явно указал — никаких очередей.

- **Обработка ошибок Claude**: при ошибке — 10 повторов с интервалом 1 минута, уведомление пользователя о каждой попытке. **Причина**: пользователь хочет знать о каждой проблеме и попытке её решить.

- **Восстановление после перезапуска**: бот читает sessions.json, восстанавливает привязки chat_id→session_id, присваивает номера новым сессиям. Пользователь возвращается в то же состояние. **Причина**: бот должен быть устойчив к перезапускам.

- **/stop остаётся**: команда /stop прерывает работу Claude в сессии (без упоминания процессов в BRD). **Причина**: пользователь явно подтвердил.

- **Режим инструментов — полный доступ**: Claude работает с полным доступом (включая Bash), а не project_only. **Причина**: пользователь решил не ограничивать.

- **ALLOWED_USER_IDS — один пользователь**: несколько ID допускаются только для одного человека с разных устройств. **Причина**: бот рассчитан на одного пользователя.

- **Watcher**: следит за сессиями текущего проекта за последние 2 дня, подхватывает сессии из терминала. **Причина**: ответы пользователя на вопросы В-2 и В-3.

- **Дневная нумерация**: сброс в полночь по локальному времени, старые номера продолжают работать но перезаписываются новыми. **Причина**: ответ пользователя.

- **Кликабельные номера сессий**: формат `/номер` в тексте сообщения — Telegram автоматически делает их кликабельными. **Причина**: самый простой и надёжный способ, пользователь одобрил.

- **Промежуточные обновления**: источник — рассуждения Claude (его мысли о задаче). **Причина**: ответ пользователя.

- **Скоуп /sessions и watcher**: только текущий проект (CLAUDE_WORKING_DIR). **Причина**: пользователь — один проект пока, расширение в будущем.

## Контекст для следующей сессии

BRD (brd-user-journeys.md) теперь актуален и содержит все ответы на технические вопросы. Основные концептуальные изменения:
- Нет понятия "процессов" в BRD — только сессии
- Нет idle timeout, нет эпохи, нет очереди сообщений
- Есть обработка ошибок с 10 повторами
- Есть восстановление состояния после перезапуска
- Полный доступ к инструментам Claude

Следующий шаг — реализация кода на основе обновлённого BRD. Файл `development/docs/brd-user-journeys.BEFORE.md` содержит предыдущую версию BRD для сравнения.
