# Сессия 29-05: диагностика повторяющегося краха сессии бота на скилле judge-prompt-iterative-tuner

## Резюме

Разобрали повторяющуюся ошибку `API Error: 400 ... thinking blocks cannot be modified`, которая возникает в проекте **bloger** при работе скилла `judge-prompt-iterative-tuner` через бот **claude_manager**. Установлена первопричина (extended thinking + `--resume` + длинный/оборванный ход с подписанными thinking-блоками = постоянная, невосстановимая ошибка) и усилитель со стороны бота (retry-цикл не отличает постоянные ошибки от временных). Сессия — чисто исследовательская, кода не меняли. Принято решение: берём **рекомендацию №2** — переделать рабочий процесс скилла на отдельный detached-процесс со статус-файлом и короткими ходами оркестратора.

## Проблемы и решения

- **Проблема**: в проекте bloger при запуске скилла `judge-prompt-iterative-tuner` (итеративный тюнер промпта LLM-судьи) бот раз за разом получает от Claude `API Error: 400 messages.N.content.M: thinking or redacted_thinking blocks in the latest assistant message cannot be modified` и спамит этим сообщением пользователя. Это уже второй случай именно с этой задачей.
- **Первопричина**: бот на каждое сообщение пользователя заново запускает `claude -p --resume <session_id>` и переигрывает всю предыдущую переписку из JSONL-файла сессии на диске. Флаг `--effort max` зашит в `claude_runner.py:77`, то есть extended thinking (внутренние «размышления» модели) включён на каждом ходу. Каждый блок размышлений несёт криптографическую подпись; правило API Anthropic — блоки размышлений в ПОСЛЕДНЕМ сообщении ассистента нужно переигрывать байт-в-байт, любое изменение даёт 400. Когда длинный, насыщенный размышлениями ход обрывается (например, убивают фоновый прогон тюнера) ИЛИ один ход накапливает десятки подписанных блоков, повторный `--resume` подаёт на вход состояние JSONL, которое стабильно воспроизводит 400. Повторный resume того же сломанного JSONL воспроизводит ошибку снова — она **постоянная**, а не временная.
- **Усилитель со стороны бота**: retry-цикл в `process_manager.py` (`MAX_RETRIES = 10`, `RETRY_INTERVAL_SECONDS = 60`) повторяет ВСЕ ошибки одинаково и не различает постоянные и временные. В итоге постоянный 400 прокручивается 10 раз за ~10 минут, заваливая пользователя одинаковыми сообщениями об ошибке.

### Доказательная база (по файлам сессий на диске)

- **Сессия `0b36079f` (та, что на скриншоте)**: один огромный агентный ход с ~19 блоками размышлений запустил фоновый прогон тюнера (Bash-таск `toolu_01Ls6u`, «Запуск полного прогона тюнера судьи на gpt-5.4 (фон)»). Дальше прилетело `<task-notification> ... <status>killed</status>` — прогон убили. После этого начался шторм ретраев: каждые 60 секунд тот же текст пользователя переотправлялся и записывался тот же `API Error: 400 messages.5.content.24`.
- **Сессия `329d5e9a`**: сломалась ПОСРЕДИ хода на `messages.33.content.1`, 51 блок размышлений, без всякого kill. Это доказывает, что сбой случается и без обрыва — на одних только длинных, насыщенных размышлениями ходах.
- **Признаки давности**: в bloger лежит handoff-документ от 27-05 про восстановление оборванной сессии бота, начинающийся с разбора того же `API Error 400 (thinking ...)`. То есть проблема известна и повторяется минимум с 27-05.

### Состояние на конец разбора

Ничего сейчас не «долбится»: ретраи исчерпаны в 03:10 (лог бота), проект затем переключился на claude_manager (доставка подавлена), живого процесса Claude по этой сессии нет. Отдельно останавливать сессию не пришлось — бот остановился сам.

## Решения

- **Решение**: берём **рекомендацию №2** — переделать рабочий процесс скилла `judge-prompt-iterative-tuner` так, чтобы долгий прогон шёл полностью отдельным detached-процессом, который сам пишет статус в файл, а оркестратор проверял этот статус короткими отдельными сообщениями («посмотри статус»). **Причина**: короткие ходы оркестратора = мало блоков размышлений в каждом ходу = почти нет шанса накопить состояние, которое ловит постоянный 400. Это бьёт по первопричине именно для этой задачи, а не маскирует симптом. Реализация — отдельно, в проекте **bloger**, где живёт скилл; там же при старте работ заводится формальный ADR с альтернативами и связанными файлами.
- **Решение**: рекомендацию №1 (починка бота: распознавать постоянный 400, не уходить в 10 ретраев, а сразу начинать свежую сессию без `--resume` либо честно сообщать «сессия сломана, начни новую») и рекомендацию №3 (пересмотреть жёстко зашитый `--effort max` для таких задач) сейчас НЕ реализуем. **Причина**: пользователь выбрал путь №2. Оба пункта зафиксированы как открытые опции на будущее — №1 особенно ценен, потому что закрывает проблему сразу для всех скиллов, а не только для тюнера.
