# Сессионный отчёт: `/restart` при активных дочерних задачах

## Коротко

Проверили состояние Telegram-бота Claude Manager после подозрительного поведения при перезапуске. Сам бот не умер: живой Python-процесс держал соединения с Telegram и продолжал вести две дочерние Codex-задачи. Но команда `/restart` запустила перезапуск через `launchd`, пока эти дочерние задачи ещё работали, и вокруг сервиса начался шумный crash loop.

Главная гипотеза: `/restart` сейчас не проверяет, есть ли активные дочерние CLI-процессы. Перед самоперезапуском нужно запрещать рестарт, если есть активные сессии или выполняющиеся дочерние задачи, и просить пользователя сначала дождаться завершения или остановить их через `/stop`.

## Рабочие файлы

- `src/claude_manager/bot.py` — обработчики Telegram-команд. В нём находится `handle_restart`, который отвечает за команду `/restart`.
- `src/claude_manager/process_manager.py` — модуль управления дочерними CLI-процессами. Он хранит `_processes`, `_busy_flags`, `_stop_events` и уже умеет проверять отдельную сессию через `has_process()` и `is_busy()`.
- `restart-claude-manager.sh` — внешний скрипт безопасного рестарта через `launchd`. Важно: он уже запрещает запуск изнутри процесса бота.
- `/Users/ivan/.local/bin/start-claude-manager.sh` — wrapper-скрипт запуска через `launchd` с retry-логикой.
- `~/Library/Logs/claude-manager.log` — основной лог бота.
- `~/Library/Logs/claude-manager.error.log` — stderr-лог wrapper-а и Python-крэшей.
- `/Users/ivan/.claude-manager.lock` — lock-файл, который держит живой бот и не даёт поднять вторую копию.

## Что наблюдалось

- Живой бот: Python-процесс `PID 1654`, запущен 12 мая 2026 около 15:04.
- У процесса есть соединения с Telegram и открытый lock-файл `/Users/ivan/.claude-manager.lock`.
- Под ботом работали два дочерних процесса:
  - `PID 12312`: `codex exec resume ...`, проект `bloger`, задача «Продолжи реализацию».
  - `PID 12389`: `codex exec resume ...`, проект `reviews-analyzer`, задача human-in-the-loop ревью спеки.
- `launchd` продолжал пытаться стартовать сервис `com.ivan.claude-manager`, но новые копии падали с ошибкой «Бот уже запущен».
- `launchctl print gui/501/com.ivan.claude-manager` показывал `state = spawn scheduled`, `runs = 9`, `last exit code = 1`.
- В `claude-manager.error.log` повторялось, что wrapper не может прочитать `.env` из-за `Operation not permitted`, затем отправляет macOS notification. Это выглядит как побочный шум при crash loop, а не как причина того, что основной бот не работает.
- Был короткий всплеск CPU у Python-процесса, но повторная проверка показала 0% CPU у бота и дочерних Node-процессов. `sample` показал в основном ожидание событий, чтение stdout дочерних процессов и разбор JSON.

## Почему это похоже на баг

`handle_restart` в `src/claude_manager/bot.py` делает следующее:

- проверяет доступ пользователя;
- отправляет сообщение «Перезапускаюсь через 2 сек...»;
- пишет chat id в marker-файл для уведомления после рестарта;
- запускает отвязанный `bash`, который через 2 секунды выполняет `launchctl kickstart -k`.

В этом пути нет проверки активных дочерних задач. Поэтому пользователь может попросить рестарт в момент, когда бот ещё держит рабочие `codex exec` subprocess-ы. После этого `launchd` пытается перезапустить сервис, а старый процесс всё ещё живёт и держит lock. Новые попытки старта закономерно получают «Бот уже запущен».

Дополнительный риск: дочерние Codex-задачи могут быть убиты или осиротеть в неподходящий момент. Даже если текущий старый бот выжил, состояние становится неоднозначным: `launchd` считает сервис падающим, логи зашумлены, а пользователь получает неочевидное поведение.

## Решения

- Считать `/restart` опасной командой, если в `process_manager` есть активные дочерние процессы.
- Перед запуском `launchctl kickstart -k` нужно проверить активные сессии.
- Если активные сессии есть, не запускать рестарт. Ответить пользователю понятным текстом: «Сейчас выполняются N задач. Дождитесь завершения или остановите активную через `/stop`, потом повторите `/restart`».
- Не использовать `stop_all_processes()` как автоматический pre-step для `/restart` без явного подтверждения пользователя. Остановка дочерних задач может потерять результат.

## Предлагаемый план фикса

1. Добавить в `process_manager.py` публичную функцию для безопасной проверки активных процессов, например `list_active_processes()` или `get_active_process_count()`.
2. Функция должна читать `_processes` под `_busy_lock`, чтобы не ловить промежуточное состояние между запуском процесса и обновлением busy-флага.
3. Вернуть достаточно данных для сообщения пользователю: количество процессов, session_id, backend, и признак busy/retry, если он нужен.
4. В `bot.py` перед отправкой сообщения «Перезапускаюсь...» вызвать эту проверку.
5. Если активные процессы есть, отправить отказ от рестарта и не писать `RESTART_MARKER_PATH`.
6. Добавить unit-тесты в `tests/test_bot.py`:
   - `/restart` при нуле активных процессов запускает текущий путь;
   - `/restart` при активных процессах не запускает `asyncio.create_subprocess_exec`;
   - при отказе не создаётся marker-файл рестарта.
7. Добавить unit-тесты в `tests/test_process_manager.py` или отдельный whitebox-тест:
   - новая функция видит процессы для Claude-only ключей;
   - новая функция видит backend-aware tuple-ключи для Codex;
   - busy-флаги корректно отражаются в результате.

## Проверки

Проверки, которые уже были сделаны в этой сессии:

- `launchctl list | rg -i 'claude|manager|bot'` — показал сервис `com.ivan.claude-manager`.
- `ps aux | rg -i 'claude_manager|claude-manager|watch_and_restart|python -m claude_manager|telegram'` — нашёл живой Python-процесс бота.
- `tail -n 160 ~/Library/Logs/claude-manager.log` — показал повторяющиеся старты, ошибку «Бот уже запущен» и активность вокруг `/restart`.
- `tail -n 200 ~/Library/Logs/claude-manager.error.log` — показал retry wrapper-а и `Operation not permitted` при чтении `.env`.
- `lsof /Users/ivan/.claude-manager.lock` — подтвердил, что lock держит живой Python-процесс.
- `pgrep -P 1654 -lf .` — подтвердил две дочерние Codex-задачи под ботом.
- `sample 1654 3 -file /tmp/claude-manager-1654.sample.txt` — показал, что бот в основном ждёт события, читает stdout дочерних процессов и разбирает JSON.

Кодовые тесты не запускались, потому что задача была документальной: зафиксировать баг и контекст для следующей сессии.

## Риски и ограничения

- Баг ещё не исправлен. Отчёт только фиксирует состояние и план.
- Нельзя вслепую убивать живой бот, пока под ним есть дочерние задачи: можно потерять результаты этих задач.
- Нельзя автоматически останавливать все дочерние задачи при `/restart` без явного подтверждения пользователя.
- Нельзя выводить содержимое `.env` в отчёты или логи. В этой сессии проверялся только факт ошибки доступа, секреты не читались и не записывались.
- Возможна отдельная проблема с TCC/provenance macOS: wrapper через `launchd` получает `Operation not permitted` при чтении `.env`. Это стоит исследовать отдельно, но главный баг текущего отчёта — отсутствие guard-а перед `/restart`.

## Продолжение

5. После зелёных тестов решить отдельно, нужен ли root-cause отчёт по `Operation not permitted` для `.env` в wrapper-е.
