# Handoff: voice в Telegram, OpenAI transcription и фикс `/all`

## Коротко

В этой сессии закрыли две связанные, но разные задачи.

Первая задача — режим `/all`, то есть режим, где бот должен показывать сообщения из всех проектов, а не только из текущего. Баг был в том, что монитор всех проектов мог брать старый список последних сессий из кэша и не обновлять его перед обходом. Из-за этого сообщения из активных сессий не доходили.

Вторая задача — подготовка распознавания голосовых сообщений. Бот научили принимать Telegram `voice` и `audio`, скачивать файл, показывать его метаданные и пока не отправлять его дальше агенту. После этого реальный Telegram voice-файл проверили через OpenAI API: исходный `Ogg/Opus` файл был принят моделью `whisper-1` без конвертации.

## Рабочие файлы

- `src/claude_manager/recent_sessions_refresh.py` — модуль, который обновляет таблицу последних сессий. В `get_global_recent_sessions()` добавлен параметр `refresh_on_hit`, чтобы режим `/all` мог обновлять даже уже заполненный кэш.
- `src/claude_manager/all_projects_monitor.py` — монитор режима `/all`, который опрашивает последние сессии по всем проектам. Теперь он вызывает глобальное обновление с `refresh_on_hit=True`.
- `src/claude_manager/telegram_file_downloader.py` — общий скачиватель файлов из Telegram. Он теперь понимает не только фото и документы, но и голосовые/аудио-сообщения, включая MIME `audio/ogg`.
- `src/claude_manager/telegram_input_handlers.py` — обработчики входящих Telegram-сообщений. Добавлен диагностический `handle_voice()`: он скачивает аудио, строит короткий отчёт с MIME, размером, длительностью, `file_id` и путём сохранения.
- `src/claude_manager/bot.py` — место, где регистрируются обработчики Telegram. Добавлена регистрация `filters.VOICE | filters.AUDIO`.
- `tests/test_recent_sessions_refresh.py` — регрессионный тест на обновление уже заполненного глобального кэша.
- `tests/test_all_projects_monitor.py` — проверка, что `/all` вызывает глобальное обновление с `refresh_on_hit=True`.
- `tests/test_telegram_audio_file_downloader.py` — новые тесты на извлечение информации из Telegram voice/audio.
- `tests/test_telegram_audio_input_handlers.py` — новые тесты на диагностический обработчик голосовых сообщений.
- `tests/test_telegram_audio_handler_registration.py` — новые тесты на экспорт и регистрацию voice/audio handler.

## Решения

- Telegram Bot API не даёт параметра, который заставил бы входящий `voice` вернуться в другом формате. `getFile` возвращает путь к файлу, а не конвертирует его.
- На тестовом сообщении Telegram прислал `voice` с MIME `audio/ogg`, размером 25 476 байт и длительностью 6 секунд. Локальная проверка файла показала `Ogg data, Opus audio, mono, 48000 Hz`.
- Сначала был сделан диагностический этап без OpenAI и без конвертации. Это было нужно, чтобы увидеть реальный формат от Telegram, а не строить интеграцию на предположениях.
- Реальный запрос в OpenAI прошёл успешно: `whisper-1` принял исходный `.ogg` и вернул текст: «Смотри, пожалуйста, ты сейчас распознаешь, что я говорю, вообще видишь эти записи или нет?»
- Официальная документация OpenAI Speech to text сейчас перечисляет поддержанные форматы `mp3`, `mp4`, `mpeg`, `mpga`, `m4a`, `wav`, `webm`, но этот конкретный `Ogg/Opus` файл API фактически принял. Документация: https://developers.openai.com/api/docs/guides/speech-to-text
- OpenAI credential для проверки был взят из проекта `bloger`, из его `.env`. Секрет в вывод и в этот отчёт не включался.

## Риски и ограничения

- Живой бот после правок не перезапускали. В этом проекте нельзя рестартовать `claude-manager` из собственного дерева процессов; безопасный путь — `/restart` в Telegram или внешний терминал.
- В проекте `claude_manager` пока нет собственного OpenAI key в `.env`. Для ручной проверки ключ взят из `bloger/.env`. Для постоянной работы бота нужно решить, где будет жить разрешённый OpenAI credential.
- Документация OpenAI и фактическое поведение API расходятся по `ogg`: документация не заявляет `ogg`, но тестовый файл принят. Это нужно учитывать как риск: при будущих изменениях API может потребоваться конвертация в `webm`, `m4a` или `wav`.
- В окружении нет `ffmpeg` и `ffprobe`. Сейчас это не блокер, потому что `.ogg` принят напрямую, но если API перестанет принимать `ogg`, конвертация потребует отдельной установки или зависимости.
- `src/claude_manager/telegram_input_handlers.py` вырос до 339 строк и пробил warning-порог 300 строк. Следующую голосовую бизнес-логику лучше вынести в отдельный модуль, чтобы обработчик входящих сообщений не расползался.
- `src/claude_manager/all_projects_monitor.py` сейчас 696 строк. Это больше 500 строк и всего 4 строки до stop-порога 700. Дальше в этот файл стоит добавлять только маленькие фиксы или начинать разбиение.
- В рабочем дереве есть незакоммиченные изменения по двум темам: `/all` и voice/OpenAI diagnostics. Новая сессия должна не потерять untracked тесты `tests/test_telegram_audio_*.py`.

## Продолжение

1. Решить, где хранить OpenAI key для `claude_manager`: перенести в локальный `.env`, читать из общего разрешённого места или добавить явную настройку пути к credential. Секреты не выводить в логи и не добавлять в отчёты.
2. Вынести голосовую логику из `telegram_input_handlers.py` в отдельный модуль, например `telegram_audio_transcription.py` или `openai_transcription.py`. Это снизит риск дальнейшего роста файла, который уже больше 300 строк.
3. Добавить OpenAI transcription как следующий этап после успешного скачивания voice/audio:
   - скачать Telegram-файл;
   - отправить исходный `.ogg` в OpenAI `whisper-1`;
   - получить текст;
   - отправить этот текст дальше в активную сессию как обычное пользовательское сообщение.
4. Добавить тесты на успешное распознавание через замоканный OpenAI-клиент и на ошибку OpenAI API. В тестах не использовать реальный ключ.
5. После реализации прогнать targeted audio tests, соседние handler/downloader tests, полный не-E2E pytest и `git diff --check`.
6. Перезапустить живой бот только безопасным способом и вручную проверить Telegram-сценарий: отправить voice, увидеть распознанный текст в активной сессии и ответ агента.
