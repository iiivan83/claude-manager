# Сессия 13-04: root-cause анализ падения обработчиков из-за send_chat_action

## Резюме

Пайплайн root-cause-analysis завершён. Найдена корневая причина: три обработчика пользовательских сообщений в `bot.py` (`handle_message`, `handle_photo`, `handle_document`) вызывают `send_chat_action(ChatAction.TYPING)` без `try/except` — при ошибке Telegram API (TimedOut, throttling) весь обработчик падает, основная логика не выполняется, пользователь не получает ответа. Побочное: `session_watcher` спамит `error.log` (~1 МБ) из-за ссылок на удалённые сессии.

## Решения

- **Корневая причина — архитектурный пробел, не баг кода.** В CLAUDE.md нет принципа разделения вызовов внешних API на критические (результат нужен для продолжения) и декоративные (индикатор "печатает...", можно пропустить). Без этого принципа агент, писавший спецификацию `bot_spec.md`, не мог знать, что `send_chat_action` нужно оборачивать — правило нигде не описано.
- **Три отдельных отклонения, одна системная причина.** (1) `send_chat_action` без защиты роняет обработчики, (2) нет глобального error handler как последнего рубежа, (3) watcher спамит логи из-за невалидированных сессий из реестра.
- **Busy flag НЕ утекает.** Проверено: `send_chat_action` падает ДО вызова `process_manager.send_message()`, флаг занятости не устанавливается.
