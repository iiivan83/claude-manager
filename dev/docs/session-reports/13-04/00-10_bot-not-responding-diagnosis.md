# Сессия 13-04: диагностика «бот не отвечает на сообщения»

## Резюме

Пользователь сообщил, что бот не отвечает на текстовые сообщения в Telegram (команды `/new`, `/56` работали). Проведена диагностика по логам и коду — найдена корневая причина и связанные проблемы. Код не менялся — только расследование.

## Проблемы и решения

### Основная проблема: TimedOut на send_chat_action роняет обработчик

- **Симптом**: бот молчит на текстовые сообщения, но команды (`/new`, `/56`) работают
- **Корневая причина**: `bot.py:887` — вызов `send_chat_action(ChatAction.TYPING)` не обёрнут в try/except. Когда Telegram API возвращает TimedOut, исключение пробрасывается наверх, `handle_message` падает целиком, `_send_to_claude_and_respond` (строка 888) не вызывается. Telegram-библиотека логирует `"No error handlers are registered"` и тихо глотает ошибку
- **Почему команды работали**: у command-хэндлеров (`/new`, `/56`) другой путь обработки, они не вызывают `send_chat_action` перед основной логикой
- **Триггер**: одновременно с пользовательскими сообщениями шёл полный прогон E2E тестов (21 тест через реальный Telegram) — нагрузка на Bot API от одного токена, вероятный throttling
- **Статус**: не починено, только диагностировано

### Побочная проблема: watcher спамит error.log

- `session_watcher` каждые 2 секунды пытается прочитать 3 JSONL-файла сессий, которых нет на диске:
  - `70ca7205-a1d5-4fb2-aaf8-dc3809fb7bfa.jsonl`
  - `669ddf67-7487-443a-9e56-e2c6f2c7ba6f.jsonl`
  - `d2c6a9c3-d1c3-432b-8b68-b3867b872b71.jsonl`
- Лог `claude-manager.error.log` — ~1 МБ, почти целиком из этого спама
- Причина: записи в `sessions.json` (реестр привязок chat_id → session_id) ссылаются на удалённые сессии
