# Сессия 13-04: диагностика спама в логах watcher

## Резюме

Диагностировали проблему: session_watcher бесконечно опрашивает удалённую сессию `dc96ec21-2ce5-426e-9d18-b130e548dd45` (#45), генерируя 32 954 WARNING в error.log (9.5 МБ). Выбран вариант исправления — запоминание «мёртвых» сессий в watcher.

## Решения

- **Решение**: вариант 1 — добавить в `session_watcher.py` множество `_missing_file_sessions`, запоминать сессии с отсутствующими `.jsonl` файлами и пропускать их в последующих циклах опроса. Множество сбрасывается при `reset_state()`. **Причина**: самый чистый подход — watcher перестаёт опрашивать удалённые сессии, не трогая логику `session_reader` и `daily_session_registry`. Второй вариант (понижение WARNING до DEBUG в `session_reader`) отклонён как основной, но может быть дополнением.

## Проблемы и решения

- **Проблема**: error.log раздут до 9.5 МБ (32 954 одинаковых WARNING). Watcher каждые 2 секунды вызывает `session_reader.get_session_messages()` для сессии #45 (`dc96ec21...`), чей `.jsonl` файл удалён с диска. Файл отсутствует — WARNING. Следующий цикл — снова WARNING. Бесконечно. **Корневая причина**: `_get_sessions_to_monitor()` (session_watcher.py:105) собирает сессии из двух источников — файлы на диске (`session_reader.get_recent_sessions`) и дневной реестр (`daily_session_registry.get_all_today_sessions`). Дневной реестр содержит сессию #45, хотя файл удалён. Watcher не имеет механизма запоминания «мёртвых» сессий — пробует снова и снова. **Решение**: пока не реализовано, выбран вариант 1 (см. секцию Решения).
- **Проблема (вторичная)**: новые сессии с временным ID `_new_*` тоже генерируют WARNING «файл не найден» в течение нескольких секунд, пока temp ID не заменится на реальный UUID через `session_id_callback`. **Решение**: это ожидаемое поведение, но `_missing_file_sessions` решит и этот случай — temp-сессия попадёт в множество, а при `update_session_id()` старый ID очистится.
