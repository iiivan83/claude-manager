# Сессия 16-06: фикс повторной доставки в режиме /all при каждом входе

## Резюме

Починили баг режима `/all` (глобальный мониторинг сообщений из всех проектов): одно и то же сообщение приезжало заново при каждом входе в `/all`, пока пользователь не откроет его внутри проекта. Теперь режим помнит собственный курсор доставки между входами и не дублирует уже показанное. Все 1203 юнит-теста проходят.

## Изменённые файлы

- **`src/claude_manager/all_projects_monitor.py`** — изменён — `_baseline_state_from_snapshot` получил параметр `previous_all_mode_state` и ветку приоритета: для сессий вне активного проекта baseline берётся из собственного прежнего курсора `/all`, а к `unread_buffer` обращается только при первом показе сессии. `_read_baseline_state` и `_build_baseline_states` пробрасывают `previous_states`; `enable_for_chat` снимает копию `_states` до её очистки.
- **`tests/test_all_projects_monitor.py`** — изменён — добавлен регресс-тест `test_reentering_all_mode_does_not_redeliver_shown_backlog` (вход → показ → выход → повторный вход → нет повторной доставки, при этом `unread_buffer` остаётся нетронутым) и хелпер `_poll_with_mtime`.
- **`dev/docs/adr/16.06_15.01-session-change-documenter-all-mode-delivery-cursor-survives-reentry.md`** — создан — ADR с решением о приоритете собственного курсора `/all` над `unread_buffer`.
- **`dev/docs/brd/brd-user-journeys.md`** — изменён — в CJM-07 добавлен пункт «Что может пойти не так» про повторную доставку при входе в `/all` и её решение.

## Решения

- **Решение**: источником правды для прогресса доставки `/all` между включениями стал собственный курсор `all_projects_monitor` (`_states`), а `unread_buffer` оставлен только как fallback для первого показа бэклога. **Причина**: `unread_buffer` обслуживает pending-доставку в реальном проекте и специально не очищается при показе в `/all`. Строить baseline из него при каждом входе означало каждый раз отматываться к позиции до сообщения. У `/all` должна быть собственная память о показанном, отдельная от памяти о непрочитанном для проекта.
- **Решение**: правка прицельно затрагивает только ветку `watcher_state is None` (сессии вне активного проекта). **Причина**: баг проявлялся именно там; ветку активного проекта (watcher-state) трогать не нужно, чтобы не сломать уже корректное поведение.

## Проблемы и решения

- **Проблема**: предыдущая правка (baseline из `unread_buffer`) корректно показывала бэклог из другого проекта, но породила повторную доставку при каждом входе в `/all`. **Решение**: добавили приоритет собственного курсора `/all` перед `unread_buffer` — оба требования (показать бэклог один раз и не дублировать при повторном входе) теперь сосуществуют.
- **Проблема**: нужно было убедиться, что новый тест реально ловит баг, а не проходит случайно. **Решение**: временно отключили новую ветку приоритета — тест упал с повторной доставкой; вернули ветку — тест прошёл.

## Контекст для следующей сессии

- Правка лежит в рабочем дереве и **ещё не применена в живом боте**. Чтобы она заработала, бота нужно перезапустить — безопаснее всего командой `/restart` из самого бота (перезапуск из его же процесса через `restart-claude-manager.sh` убивает дерево процессов, exit 137).
- В рабочем дереве остаётся много прежних незакоммиченных изменений, не относящихся к этой сессии (удаления зеркал скиллов, правки `claude_code_backend.py`, `claude_code_session_file_reader.py`, `claude_runner.py`, `reply_route_handler.py`, отчёты RCA и др.). Они появились до этой сессии — в коммит этой сессии они намеренно не включаются.
- Память `/all` и `unread_buffer` обе in-memory: после перезапуска бота оба источника пусты, повторной доставки после рестарта не будет.
