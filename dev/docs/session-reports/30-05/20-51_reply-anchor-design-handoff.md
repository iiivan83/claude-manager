# Handoff: reply-якоря для ответов Telegram-бота

## Коротко

Обсуждали, как сделать так, чтобы Telegram-бот отвечал реплаем на сообщение Ивана, которое запустило работу Claude/Codex.

Главный вывод: это реализуемо, но правильный якорь должен быть не “последнее сообщение пользователя в Telegram”, а “последнее сообщение пользователя, которое реально попало в конкретную сессию Claude/Codex”.

Выбранный рабочий контракт:

`project_path + backend + session_id -> last_accepted_telegram_message_id`

Где:

- **`project_path`** — рабочая папка проекта. Нужна, чтобы не путать одинаковые сессии между проектами.
- **`backend`** — выбранный агент: Claude или Codex.
- **`session_id`** — технический ID диалога агента.
- **`last_accepted_telegram_message_id`** — ID сообщения Ивана в Telegram, которое бот реально отправил в Claude/Codex.

## Рабочие файлы

- `CLAUDE.md` — описание проекта и архитектуры. Подтвердило, что это локальный Telegram-бот для управления Claude Code/Codex с телефона.
- `src/claude_manager/bot.py` — транспортный слой Telegram: принимает сообщения, команды `/all`, `/stop`, `/sessions`, `/pN`, `/N`, отправляет ответы в Telegram.
- `src/claude_manager/claude_interaction.py` — оркестрация одного запроса: фиксирует рабочий проект, ставит watcher на паузу, отправляет сообщение в агент, доставляет финальный ответ.
- `src/claude_manager/telegram_sender.py` — низкоуровневая отправка сообщений в Telegram с retry и fallback. Сюда надо будет добавить поддержку reply-параметров.
- `src/claude_manager/session_watcher.py` — фасад фонового watcher-а. Watcher — наблюдатель, который периодически читает файлы сессий и досылает новые ответы в Telegram.
- `src/claude_manager/coding_agent_session_file_poller.py` — основной poller watcher-а: читает файлы сессий, выбирает новые assistant-сообщения, вызывает callback доставки.
- `src/claude_manager/all_projects_monitor.py` — глобальный мониторинг `/all`, который сканирует все проекты и доставляет сообщения с префиксом проекта/сессии.
- `src/claude_manager/session_manager.py` — привязка Telegram-чата к активной сессии, переключение через `/N`, режим мониторинга `/all`.
- `src/claude_manager/process_manager.py` — жизненный цикл процесса агента, busy-флаги, retry и `/stop`.

## Решения

### 1. Reply нужен через Telegram message_id

Telegram уже поддерживает ответ “реплаем”: в текущей версии `python-telegram-bot` метод `Bot.send_message()` принимает `reply_parameters` и также совместимый `reply_to_message_id`.

Проверка локальной версии показала, что оба параметра доступны в установленном окружении проекта.

### 2. Не хранить “последнее сообщение вообще”

Глобальное “последнее сообщение пользователя” ломается в реальных сценариях:

- Иван написал запрос A.
- Агент ещё работает.
- Иван написал запрос B, но B не попал в сессию, потому что агент занят.
- Watcher увидел прогресс по A.
- Если брать последнее Telegram-сообщение вообще, прогресс A ошибочно станет reply к B.

Поэтому якорь обновляется только тогда, когда сообщение реально принято в Claude/Codex.

### 3. Watcher-сообщения должны искать якорь по источнику

Watcher знает источник сообщения: проект, backend и session_id. Поэтому он должен брать reply-якорь по этому же ключу.

Если якорь есть — watcher может отправить сообщение реплаем. Если якоря нет — отправляет обычным сообщением.

### 4. Переключение проекта не удаляет якорь

Если запрос принят в проекте A, а Иван переключился в проект B, якорь проекта A должен остаться.

Если позже из проекта A придёт watcher/pending-сообщение, оно может ответить реплаем на исходный запрос из проекта A. Это полезно: в Telegram будет видно, что сообщение относится к старому запросу из другого проекта.

### 5. `/all` не обновляет якорь, но может его использовать

Режим `/all` — это мониторинг всех проектов, а не отправка запросов в конкретную сессию.

Команды и обычный текст в `/all`, которые бот отклоняет предупреждением, не должны менять reply-якорь. Но `all_projects_monitor` может использовать существующий якорь конкретного проекта/сессии, если он есть.

### 6. `/stop` не является якорем

`/stop` — команда управления процессом, а не сообщение в Claude/Codex.

Правило:

- `/stop` не обновляет reply-якорь.
- Активный якорь текущего хода нужно закрыть или пометить отменённым.
- Следующее сообщение, которое реально принято в эту же сессию, становится новым якорем.
- Чтобы старый хвост после остановки не привязался к новому запросу, вместе с якорем желательно хранить позицию старта хода в файле сессии.

### 7. Переключение сессии не обновляет якорь

Команды `/sessions`, `/N` и `/3s12` только меняют активную сессию. Они не попадают в Claude/Codex, поэтому не должны становиться reply-якорями.

Если Иван переключился на сессию #8:

- watcher-сообщения из #8 ищут якорь #8;
- watcher-сообщения из старой #5 ищут якорь #5;
- новое пользовательское сообщение после переключения попадёт уже в #8 и обновит якорь #8.

### 8. Длинные ответы лучше реплаить только первым чанком

`message_splitter` разбивает длинные ответы на несколько Telegram-сообщений.

Рекомендация: reply ставить только на первый чанк ответа, а остальные отправлять обычными сообщениями. Иначе Telegram будет многократно показывать одну и ту же цитату и чат станет шумнее.

## Проверки

Фактические проверки в этой сессии:

- Прочитан `CLAUDE.md`, чтобы подтвердить архитектуру проекта и роль Telegram-бота.
- Просмотрены обработчики сообщений, watcher-а, `/all`, `/stop`, переключения проекта и переключения сессии.
- Проверена локальная сигнатура `telegram.Bot.send_message()` через `.venv/bin/python`: установленная библиотека поддерживает `reply_parameters` и `reply_to_message_id`.
- Код не менялся, тесты не запускались, потому что задача была дизайн/handoff, а не реализация.

## Риски и ограничения

- В файлах сессии Claude/Codex нет Telegram `message_id`. Его нельзя надёжно восстановить потом по тексту. Связь нужно сохранять в момент входящего Telegram-сообщения.
- Простая схема “последнее принятое сообщение в сессии” почти достаточна, но для `/stop` и поздних хвостов нужна дополнительная защита через позицию старта хода или turn-поколение.
- Если бот перезапустится, in-memory якорь потеряется. Нужно решить, достаточно ли этого, или нужен JSON-реестр на диске.
- Для старых сессий, сообщений из ручного CLI или внешних источников якоря может не быть. В таких случаях бот должен отправлять сообщение без reply.
- Медиа-группы требуют отдельного решения: якорем можно считать первое сообщение альбома или сообщение с caption. Это нужно явно выбрать перед реализацией.

## Продолжение

1. Написать короткую спецификацию фичи reply-якорей перед кодом.
2. Спроектировать модуль реестра якорей, например `reply_anchor_registry`, с ключом `project_path + backend + session_id`.
3. Добавить в реестр состояние “последнее принятое сообщение” и, желательно, позицию старта хода в файле сессии.
4. Расширить `telegram_sender.send_telegram_message()` поддержкой optional reply-параметров и fallback на обычную отправку.
5. Протащить `message_id` из `bot.py` в `claude_interaction.send_to_claude_and_respond()`.
6. Обновлять якорь только после того, как сообщение принято в сессию, а не при любом входящем Telegram-сообщении.
7. Использовать якорь в обычном `send_response()`, watcher-доставке, `/all`-мониторе и pending-доставке после переключения проекта.
8. Добавить тесты:
   - обычный текстовый запрос отвечает reply;
   - busy-сообщение не обновляет якорь;
   - `/all` не обновляет якорь, но watcher может его использовать;
   - `/stop` закрывает старый ход и не привязывает старые хвосты к новому запросу;
   - переключение проекта сохраняет раздельные якоря;
   - переключение сессии использует якорь выбранной сессии.
