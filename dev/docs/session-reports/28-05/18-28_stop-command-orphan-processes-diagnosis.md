# Сессия 28-05: разбор «команда /stop не останавливает пятую сессию», убийство orphan-процессов claude

## Резюме

Пользователь пожаловался, что пятая сессия в боте (`ff22b2a9-e25a-4571-82e5-c58dc65cec2b`) не останавливается командой `/stop` — бот отвечает «остановлено», но процессы продолжают работать. Диагностика показала, что под одним session_id в системе живут несколько процессов claude (583830, 583939, 584044), а бот в словаре `_processes` хранит ссылку только на последний созданный — все предыдущие становятся orphan-процессами, неуправляемыми из бота. Найден root cause в `process_manager._send_message_backend_aware` и `_restart_process` — они не останавливают существующий процесс перед записью нового в реестр. Убито 2 orphan-процесса (583939, 584044), self-PID 586387 и старая signer-сессия 3273288 оставлены живыми. Код в этой сессии не правили, только диагностировали.

## Изменённые файлы

- **dev/docs/session-reports/28-05/18-28_stop-command-orphan-processes-diagnosis.md** — создан — этот сессионный отчёт

Файлов исходного кода не правили. Бот PID 210627 продолжает работать со старой логикой — починка отложена до следующей сессии.

## Решения

- **Решение**: убить только orphan-процессы пятой сессии (PID 583939 и 584044), не трогать self-PID 586387 и старую signer-сессию 3273288. **Причина**: PID 586387 — это сам Claude, через который пользователь общается (если убить — разговор оборвётся). PID 3273288 (сессия `f1420f92` в проекте `/home/ivan/claude-sandbox/signer`, запущена 27 мая) держит `python3 -m http.server` на порту 8765 — возможно, это рабочий сервер, который пользователь активно использует.

- **Решение**: не править код процесс-менеджера в этой сессии. **Причина**: пользователь явно попросил только диагностику («посмотри, что происходит, почему так») и срочный kill orphan-процессов. Починка кода требует тестов и валидации — это отдельная сессия.

- **Решение**: не делать полный рестарт бота (вариант «kickstart»). **Причина**: рестарт убил бы и self-PID, обрывая текущий разговор. Кроме того, рестарт через `kickstart` из дочернего процесса бота — известный антипаттерн (CLAUDE.md, «Принцип запрета самоперезапуска из собственного дерева процессов»), который раньше приводил к exit 137.

## Проблемы и решения

- **Проблема**: пятая сессия имеет 3 живых процесса claude с одинаковым `--resume ff22b2a9-e25a-4571-82e5-c58dc65cec2b` (PID 583830, 583939, 584044). Бот через словарь `_processes[session_id]` помнит только последний созданный — `/stop` от пользователя останавливает один, остальные продолжают работать. **Решение**: `kill -9 583939 584044`. SIGKILL сработал мгновенно — значит проблема не в нечувствительности claude к сигналам, а в логике бота.

- **Проблема**: лог `~/Library/Logs/claude-manager.log` забит pytest-выводом с одинаковым timestamp `2026-05-28 18:08:12` (имена тестовых сессий: `abc-def-111`, `test-stop-during-wait`, `_new_test_temp_id` и т.д.) вперемешку с реальными событиями бота — выглядит как «параллельно кто-то ещё бота редактирует». **Решение**: разобрался, что один из orphan-процессов (PID 584044) запустил через Bash tool команду `python -m pytest tests/ --ignore=tests/e2e`, тесты импортировали `claude_manager.main`, тот через `_setup_logging()` подцепил RotatingFileHandler на тот же лог-файл бота. Pytest завершился сам — шум прекратился, действий не потребовалось.

- **Проблема**: лог бота лежит по macOS-пути `~/Library/Logs/claude-manager.log`, хотя в `main.py:36-38` уже стоит XDG-путь `~/.local/state/claude-manager/claude-manager.log`. **Решение**: оставить как есть. Бот PID 210627 запущен 22 мая (uptime 6 суток) со старым кодом, файл-handler уже открыт. После следующего рестарта лог переедет на новый путь автоматически — никаких действий сейчас не требуется.

### Не починено
Баг с orphan-процессами в `src/claude_manager/process_manager.py` остаётся. Любое сообщение пользователя в существующую сессию, у которой ещё есть процесс в реестре, создаст новый процесс claude без остановки старого. Воспроизводится сценарием: «отправить сообщение → получить ошибку от API → /stop → отправить новое сообщение в ту же сессию». Старый процесс становится orphan.

### Что чинить
Минимальная починка — добавить три строки в `_send_message_backend_aware` (`process_manager.py:933-948`) по аналогии с уже существующим кодом в `_execute_single_retry:521-523`:

```python
async with _busy_lock:
    existing_process = _processes.get(process_key)
if existing_process is not None and existing_process.is_running():
    await _apply_backend_stop_strategy(existing_process, effective_backend)
```

Эти строки нужно вставить ПОСЛЕ захвата `_busy_flags` и ДО вызова `_restart_process`.

Дополнительная защита — в самом `_restart_process` (`process_manager.py:669-731`) сделать ту же проверку перед записью нового процесса в `_processes[process_key]`. Это «пояс и подтяжки»: даже если новый код будет вызывать `_restart_process` напрямую, старый процесс корректно остановится.

Третий уровень защиты — после `_apply_stop_strategy` верифицировать, что процесс реально мёртв через `os.kill(pid, 0)` (бросает `ProcessLookupError`, если процесса нет). Сейчас бот доверяет тому, что `process.wait()` вернулся — но Node.js может откладывать завершение через `process.on('SIGTERM', ...)` handler, а asyncio child watcher может вернуть returncode по closed pipe ещё до того, как ОС reap'ит процесс.

### Что нужно решить отдельно

- **Подход к тестированию починки** — придумать unit-тест, который воспроизводит сценарий «два turn'а подряд для одной сессии», и проверяет, что между ними старый процесс убит.

### Root cause в коде

Главный виновник — функция `_restart_process` в `src/claude_manager/process_manager.py:669-731` и то, как её вызывает `_send_message_backend_aware` (строки 933-948).

**Что должно происходить при отправке сообщения в существующую сессию:**
1. Проверить, нет ли уже живого процесса под этим session_id
2. Если есть — остановить его (отправить сигналы и дождаться завершения)
3. Запустить новый, записать ссылку в реестр `_processes`

**Что происходит на самом деле:**
В `_send_message_backend_aware` после захвата `_busy_flags` сразу вызывается `_restart_process(...)` (строка 941). Сам `_restart_process` (строки 678-731) проверяет `stop_event` на cancellation, но **не проверяет, есть ли уже процесс в `_processes[process_key]`**. В строке 718 он просто пишет: `_processes[process_key] = claude_process`. Если там лежала ссылка на старый процесс — она перезаписывается, старый PID остаётся жить orphan'ом.

Защита от этого есть только в одном месте — в `_execute_single_retry` (строки 521-523):

```python
old_process = _processes.get(process_key)
if old_process is not None and old_process.is_running():
    await _apply_backend_stop_strategy(old_process, backend_name)
```

Эти три строки гасят старый процесс перед запуском нового. Но в основном пути `_send_message_backend_aware` (когда пользователь шлёт обычное сообщение, не retry) этой защиты нет.

### Цепочка событий, которая создаёт orphan-процессы

1. Сессия #5 активна, PID 583830 работает в retry-loop (Anthropic API вернул ошибку, бот ждёт 60 секунд до следующей попытки)
2. `/stop` от пользователя: `stop_process` ставит `stop_event.set()`, делает `_processes.pop()`, шлёт SIGTERM. PID 583830 умирает. Бот пишет «остановлено».
3. Пользователь шлёт новое сообщение в ту же сессию #5. Бот: `_busy_flags.get((ff22b2a9, claude))` = False (был pop'нут раньше) → стартует новый turn → `_restart_process` запускает PID 583939, пишет в `_processes`.
4. PID 583939 получает ошибку от Anthropic API → запускается retry-loop.
5. `/stop` от пользователя: `stop_process` снова делает `pop`, отправляет SIGTERM. SIGTERM встречает claude в момент retry-ожидания, retry-loop ловит `stop_event` через `_check_stop_requested()` и бросает `ProcessStoppedError`. Эта ошибка всплывает в `_send_message_backend_aware`, попадает в `finally` блок (строки 1006-1020).
6. **Здесь критично**: `finally` блок `_send_message_backend_aware` НЕ удаляет процесс из `_processes` — чистит только `_busy_flags` (если `process_started`) и `_stop_events`. Старый процесс остаётся в реестре.
7. Между моментами «`stop_process` сделал `pop`» и «`finally` запустился» уже мог сработать `_restart_process` из retry-loop, который вписал в `_processes` ещё один процесс (PID 584044). А PID 583939 в этот момент остался orphan'ом.
8. Каждый следующий `/stop` берёт из `_processes` только текущую (самую свежую) ссылку и убивает её. Старые orphan'ы остаются жить.

### StopStrategy для claude (для понимания, что сигналы — правильные)

В `claude_code_backend.py:63-68`:

```python
CLAUDE_CODE_STOP_STRATEGY = StopStrategy(
    steps=(
        StopSignalStep(signal.SIGTERM, float(TERMINATE_TIMEOUT_SECONDS)),  # 5 секунд
        StopSignalStep(signal.SIGKILL, 0.0),
    )
)
```

То есть сначала SIGTERM с 5-секундным окном на graceful shutdown, потом SIGKILL мгновенно. Сами сигналы корректные — мой ручной `kill -9 583939 584044` тоже сработал мгновенно. Проблема не в сигналах, а в логике реестра процессов в боте.
