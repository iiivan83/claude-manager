## Решения

- **Решение**: починить только FLOW-02, FLOW-06 оставить как есть. **Причина**: FLOW-06 прошёл со второго раза без изменений кода — это флейк тайминга Claude CLI (задача «прочитать config.py» иногда затягивается больше 90 сек), а не регрессия. Трогать тайм-аут или добавлять ретрай без явной просьбы пользователя не нужно.
- **Решение**: править не продакшен-код, а только тест. **Причина**: баг был в логике самого теста (слабый маркер синхронизации), бот работал корректно. Заголовок ответа формируется в `bot.py:119-122` в формате `#{session_number} ✅ ` для финала и `#{session_number} ⏳ ` для thinking — бот ничего не ломал.
- **Решение**: усилить маркер во всех четырёх местах FLOW-02, а не только там, где падало. **Причина**: шаги 7 и 9 используют ту же слабую подстроку `#{num_a}` / `#{num_b}` и точно так же рискуют ловить thinking-сообщение вместо финала — устранять корень, а не симптом.

## Проблемы и решения

- **Проблема**: FLOW-02 стабильно падал на шаге 5 (`/sessions` → ждёт `/{num_a}`), получая вместо этого запоздавший `#{num_b} ✅ ок`. **Причина**: `wait_for_matching_response` в `tests/e2e/test_client.py:126-162` ищет подстроку по всему накопительному буферу `_all_responses` с `checked_index = 0`. На шаге 4 ждут маркер `#{num_b}`, но такой же маркер есть и в промежуточном thinking `#{num_b} ⏳ …`. Функция мгновенно возвращает первое совпадение — тест уходит вперёд до реального финала Claude, отправляет `/sessions` раньше времени, финал прилетает уже во время ожидания ответа на `/sessions`. **Решение**: маркер усилен до `#{num} \u2705` — это однозначно совпадает только с финальным заголовком.
- **Проблема**: FLOW-06 упал на ожидании финального `✅` 90 сек, пришёл только `⏳ Пользователь просит прочитать файл config.py…`. **Решение**: не чинил. Повторный прогон без изменений прошёл за ~15 сек — подтверждённый флейк тайминга Claude CLI.

## Контекст для следующей сессии

- Все шесть FLOW-тестов зелёные. Никаких незавершённых правок.
- Продакшен-код бота не менялся. Бот (PID 89196) работает через LaunchAgent, перезапускать не требуется.
- Если FLOW-06 снова начнёт флейкать — корень в нестабильности тайминга самого Claude CLI (поле `thinking` в stream-json приходит, а финальный event иногда задерживается). Лечить либо повышением `CLAUDE_RESPONSE_TIMEOUT_SECONDS` в этом конкретном тесте, либо автоматическим ретраем. Не трогать без явной просьбы.
- `wait_for_matching_response` в `tests/e2e/test_client.py` — накопительная: смотрит весь буфер `_all_responses` с индекса 0. Любой тест, где в буфере случайно уже есть подстрока маркера (например, из предыдущего шага), рискует получить ложное совпадение. При добавлении новых E2E тестов использовать точные маркеры вида `#{num} \u2705` (финал) или `#{num} \u23f3` (thinking), а не голые `#{num}`. Это ловушка, в которую легко попасть снова.
- Формат заголовка ответа задан в `src/claude_manager/bot.py:119-122`: `f"#{session_number} {status_icon} "`, где `status_icon = "\u2705" if is_final else "\u23f3"`. Любые изменения этого формата сломают FLOW-тесты — выровнять маркеры синхронно.
- Прошлые предположения об этих тестах ("падают из-за лимита Claude API") — частично правда: раньше был лимит, сейчас сам лимит не влиял. Но под лимитом прятался реальный баг в FLOW-02, который проявился, когда лимит снялся.
