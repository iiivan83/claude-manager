## Решения

- **Решение:** Корневую причину классифицировал как архитектурную, а не как разовый баг. **Причина:** Формулировка про «crash» скопирована в 4 файла координаторов без изменений — systematic pattern. Проблема повторяется в разных частях скилла, решение не описано ни в одном документе, несколько скиллов проекта придётся чинить одной и той же заплаткой — все три признака архитектурной природы проблемы по чек-листу root-cause-analysis.

- **Решение:** Главную правку определил на уровне глобального референса `~/.claude/references/skills-agents-rules.md` (новый раздел 19), а не на уровне проектного `CLAUDE.md` или SKILL.md. **Причина:** Правило универсальное — любой пайплайн-скилл в любом проекте, использующий делегирование через Agent tool или CLI-подпроцесс, должен следовать ему. Правь один источник истины, не дублируй.

- **Решение:** В SKILL.md скилла `feature-pipeline` добавить только **ссылку** на новый раздел 19, без дублирования полной формулировки правила. **Причина:** Избежать расхождений между скиллом и референсом. Если правило потом уточнится в референсе, а в скилле останется старая копия — начнётся разнобой.

- **Решение:** Применение схемы ко всем пайплайн-скиллам проекта разбить на **две волны**: сначала пилот на `feature-pipeline`, потом остальные 5 (`pipeline-implementer`, `autofix-e2e`, `script-creator-pipeline`, `pipeline-run`, `universal-bug-fixer`). **Причина:** Делать сразу все 6 — большой объём, который может скрыть ошибки подхода. Сначала проверить на пилоте.

- **Решение:** Отклонил гипотетическую рекомендацию «парсить время сброса лимита из текста ошибки Agent tool и ждать автоматически». **Причина:** Over-engineering. Проблема не в знании точного времени, а в том, что оркестратор вообще не останавливается. Парсинг времени из текста ошибки — brittle-код, зависящий от формата сообщения Claude-модели, который может измениться. Нужно сначала остановить оркестратор и эскалировать пользователю, а не учить его самостоятельно ждать.

## Контекст для следующей сессии

**Что Claude должен знать, продолжая эту работу:**

**Симптом и корневая причина.** Скилл `feature-pipeline` при отказе трёх параллельных Opus-агентов по лимиту на фазе 1 переключился в режим «делаю сам» и выполнил все 10 фаз напрямую в контексте оркестратора. Фича сама по себе (команда `/projects` + `/pN` для переключения проектов в Telegram-боте, коммит `5b0ac7e`) работает — 459 тестов зелёные, 75 новых. Проблема не в фиче, а в том, что оркестратор нарушил архитектуру пайплайна (изоляция контекстов, параллельность). Root-cause отчёт — в `dev/docs/logs/root-cause-reports/10-04_20-21_feature-pipeline-orchestrator-self-execution.md`.

**Три самых важных факта для понимания проблемы:**

1. **Fallback-цепочка в `SKILL.md` описана только в одну сторону.** Строки 96-97: «CLI проектного скилла не работает → откатись к внутреннему агенту (fallback)». Следующего звена цепочки («внутренний агент не работает → ???») нет. Оркестратор достроил его по аналогии: «делаю сам». Это важно — правило нужно сформулировать **с явным stop/escalate как последним звеном**, иначе LLM достроит по аналогии туда, куда тебе не нужно.

2. **Во всех 4 координаторах `feature-pipeline` описан только случай `crash`.** Файлы `feature-impact-analysis.md`, `feature-implement.md`, `feature-review.md`, `feature-docs.md` в `.claude/skills/feature-pipeline/agents/`. Секция «Обработка сбоев суб-агентов» одинакова во всех. Слова `лимит`/`limit`/`quota`/`rate`/`You've hit` в скилле не встречаются вообще — категория «инструмент временно недоступен» — системный слепой пятно.

3. **У `feature-pipeline` нет `structural-validator.sh`.** Этот скрипт есть только у `pipeline-designer` и `pipeline-implementer` (меташкиллы). Даже у них 960 строк валидации не содержат ни одной проверки дисциплины оркестратора — только структурные проверки файлов, YAML, Python AST. Принцип enforcement-first (раздел 17 глобального референса) провозглашён, но не применён к самому критическому поведенческому правилу из раздела 12.3 — «дисциплина оркестратора (только управляет, не делает работу сам)».

**Артефакты инцидента, которые стоит иметь под рукой:**

- Исходный сессионный отчёт про фичу с self-report: `dev/docs/session-reports/10-04/19-51_feature-pipeline-project-switching.md` (секция «Ретроспектива процесса» строки 93-180 — там Claude честно разбирает причины выбора «делаю сам» и предлагает новое правило `feedback_agent_limits_behavior`)
- `orchestrator-log.json` инцидента: `dev/docs/logs/skills-modifications/10.04_19.07-feature-pipeline-project-switching/orchestrator-log.json` — строки 41-49 содержат запись `action: fail` с честным комментарием «Координатор (я) продолжает работу сам без делегирования». Это диагностически критичная запись, которую никакой валидатор не перехватил
- `pipeline-state.json` того же инцидента — все 10 фаз помечены `completed`, создаёт ложное впечатление штатного прогона

**С чего лучше начать следующую сессию:** с пункта 1 списка «Незавершённое» — добавление раздела 19 в глобальный референс `~/.claude/references/skills-agents-rules.md`. Это самая высокая по иерархии правка (ARCHITECTURE), от неё наследуют все остальные. Раздел 19 должен содержать явные шаги процедуры при отказе инструмента делегирования, с чётко сформулированным запретом «оркестратор не переключается в режим делаю сам без явного разрешения пользователя». Остальные правки ([DOC], [CODE], [SKILL]) становятся применением этого принципа к конкретному скиллу.

**Важное ограничение:** правило должно применяться **буквально, без интерпретаций**. Главная ошибка в инциденте была не в отсутствии правила, а в том, что правила либо не было, либо оно допускало широкую интерпретацию. Новое правило должно быть сформулировано так, чтобы у Claude не было соблазна «достроить цепочку по аналогии» — с явным stop/escalate как единственным допустимым действием.
