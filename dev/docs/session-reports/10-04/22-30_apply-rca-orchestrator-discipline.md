## Решения

- **Решение**: пропускать все пункты с разметкой `✓ делегировано →` в чек-листе отчёта. **Причина**: пользователь явно указал это в аргументах к скиллу (не трогать `~/.claude/skill-templates/` и проектные копии фабричных скиллов `feature-pipeline`, `pipeline-implementer`, `script-creator-pipeline`, `universal-bug-fixer`), и сам отчёт размечал такие пункты как уже делегированные в budgets spec или в фабрику — применять повторно значит создавать расхождение формулировок.
- **Решение**: проверить существование раздела 19 глобального референса `~/.claude/references/skills-agents-rules.md` ДО применения правок. **Причина**: все три правки добавляют ссылку на раздел 19. Если бы budgets spec ещё не был применён, раздел 19 мог не существовать, и ссылки стали бы битыми. Проверка показала: раздел 19 существует на строке 320, занимает строки 320–353, содержит полную процедуру из 5 шагов с блоками `Why` и `How to apply`.
- **Решение**: верификатор (Agent tool, Opus) не просто даёт вердикты, а сам читает целевые файлы и возвращает точные тексты для вставки с привязкой к якорным строкам. **Причина**: исполнители не тратят контекст на повторное чтение файлов и поиск места вставки. Фаза 2 становится четырьмя короткими Edit-операциями с заранее известным содержимым.
- **Решение**: разделить рекомендацию 3 (правка в двух SKILL.md) на двух отдельных агентов. **Причина**: принцип «один агент = одна задача» из процедуры скилла. Разные файлы с разными соседними абзацами — две задачи, даже если текст вставки похож.
- **Решение**: параллельный запуск четырёх агентов-исполнителей в одном сообщении. **Причина**: все четыре правки независимы (разные файлы, разные репозитории — память вообще вне git). Конфликтов не было.
- **Решение**: не использовать `git commit --amend --no-edit` из процедуры скилла. **Причина**: правило git safety требует не использовать amend без явной просьбы пользователя. Вместо этого сделан один коммит, куда включены и правки, и rename отчёта через `git mv` — получается атомарно без amend.
- **Решение**: включить в коммит пользовательские пометки `✓ делегировано` в теле отчёта. **Причина**: после `git mv` working tree версия файла с несохранёнными правками оказывается в новом расположении. Оставить их unstaged значило бы висящие правки без дома (файл-источник уже не существует). Пометки — легитимная работа пользователя, относящаяся к той же теме делегирования.
- **Решение**: использовать Skill tool напрямую для запуска session-report, а не Agent tool с инструкцией «используй /session-report». **Причина**: прямой вызов эффективнее вложенного, и `session-report` есть в списке user-invocable skills. Процедура скилла говорит «запусти агента для создания сессионного отчёта», но Skill tool — это и есть штатный механизм запуска таких скиллов, без лишнего уровня делегирования.

## Проблемы и решения

- **Проблема**: файл отчёта `10-04_20-21_feature-pipeline-orchestrator-self-execution.md` был в состоянии `modified` ещё ДО начала сессии — у него были несохранённые пользовательские правки (разметка делегирования и блок «Статус применения»). **Решение**: сделан `git diff` файла до `git mv`, чтобы понять характер правок. Убедился, что это именно разметка делегирования и шапка «Статус применения» (11 добавленных строк + 8 модификаций в рекомендациях с пометками `✓ делегировано →`). Все правки относятся к той же теме отчёта и должны попасть в архивированную версию. Сохранены через `git mv` (переносит working tree версию, а не HEAD-версию).
- **Проблема**: процедура скилла инструктирует добавлять шаблоны X.1/X.2 (запись в `.claude/skills/` из CLI-подпроцессов) в промпт агентов, правящих файлы внутри `.claude/skills/`. **Решение**: переданы инструкции «сначала попробуй Edit, если откажет — используй шаблон X.2». Все четыре агента успешно применили штатный Edit — hooks и разрешения позволили суб-агентам Agent tool писать в `.claude/skills/` напрямую, шаблоны не понадобились.
