## Решения

- **Решение**: оставить полный текст делегированных рекомендаций в отчёте, не вырезая их. **Причина**: сохраняется самодостаточность документа (читается без открытия budgets spec), сохраняется исторический контекст из раздела «Верификация решений», а `apply-root-cause-fixes` получает явный флаг «пропустить» вместо загадочной пустоты на месте вырезанных пунктов.
- **Решение**: ставить пометку `✓ делегировано → budgets spec` в 9 местах, а не только в чек-листе. **Причина**: скилл-исполнитель при применении отчёта смотрит в 3 разные секции (чек-лист для быстрого прохода, «Рекомендации по предотвращению» для формулировок, «Верификация решений» для обоснования). Одна пометка только в чек-листе позволила бы случайно применить пункт через другой раздел.
- **Решение**: добавить секцию «Связанные документы» вместо расширения «Верификации решений». **Причина**: «Верификация решений» — это аудит, сделанный в момент написания отчёта, его не стоит трогать задним числом. Новая секция — отдельный постскриптум, не ломающий архитектуру документа.
- **Решение**: в секции «Связанные документы» явно указать, что шаблон `check_orchestrator_discipline` и 4 новых значения enum `action` из budgets spec можно переиспользовать для оставшихся `[CODE]` и `[SKILL]` правок в claude_manager. **Причина**: превращает budgets spec из «параллельной работы» в полезный источник готовых артефактов, ускоряет применение оставшихся пунктов отчёта.
- **Решение**: не трогать расхождение в enum-значениях между отчётом (`quota_limit_hit`, `delegation_paused`, `user_escalation`) и budgets spec (`budget_limit_hit`, `quota_limit_hit`, `delegation_unavailable`, `subprocess_crashed`). **Причина**: пользователь просил дедуплицировать шаги, а не унифицировать терминологию. Унификация enum — отдельная задача со своим обоснованием (инструменты делегирования разные: Agent tool в claude_manager vs CLI-подпроцесс в budgets).

## Проблемы и решения

- **Проблема**: при правке чек-листа `### Изменения в архитектуру [ARCHITECTURE]` Edit tool вернул ошибку «File has been modified since read, either by the user or by a linter» — видимо после предыдущих правок сработал какой-то линтер или авто-форматирование. **Решение**: перечитал нужный блок через Read с offset/limit, убедился что содержимое совпадает с ожидаемым, повторил Edit — прошло успешно.
- **Проблема**: оба ключевых документа (root-cause отчёт 14254 токена и budgets spec 14887 токенов) превышают лимит одного Read-вызова в 10000 токенов. **Решение**: чтение по чанкам через offset/limit — по 120–200 строк за раз. Это сохранило контекст и позволило точно определить границы делегируемых пунктов.
