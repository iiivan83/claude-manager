# Сессия 30-03: root-cause analysis и исправление зависания /new

## Резюме

Расследована проблема: после `/new` ответ от новой сессии не приходит. Найдена корневая причина — Claude CLI запускался заранее (в `/new`), зависал на инициализации, и бот бесконечно ждал ответ от зависшего процесса. Исправлено: процесс теперь запускается лениво, при первом сообщении.

- `development/docs/root-cause-reports/fix-process/30-03_04-00_new-session-no-response.md` — создан — отчёт о процессе применения исправления

## Решения

- **Решение**: отложить таймаут в `read_events`, коммит `--resume` fix, обновление CLAUDE.md и скиллов. **Причина**: пользователь решил применять пошагово, сначала главное исправление.

- **Проблема**: логи бота не найдены в `~/Library/Logs/claude-manager.error.log` после 03:00. **Решение**: `lsof -p 15905` показал что stderr бота перенаправлен в task output Claude Code (`/private/tmp/claude-501/.../tasks/bcanx5kcv.output`), а не в launchctl log.
- **Проблема**: в закоммиченном коде была ещё ошибка — `create_process` передавал `_new_0001` как `--resume _new_0001`. **Решение**: исправление уже было в рабочем дереве (незакоммичено, сделано в 02:48), бот подхватил его при перезапуске в 03:00:45. Но коммит ещё не сделан (отложено).

## Сессия 30-03-2026 04:34 — Ленивый запуск не помог, найден настоящий баг: формат сообщений

### Резюме

Ленивый запуск процесса (убрали `create_process` из `/new`) не решил проблему — Claude CLI зависал точно так же. Провели ручное тестирование Claude CLI из командной строки — в режиме `--input-format stream-json` он зависает даже из терминала. Поиск по документации и GitHub issues выявил настоящую причину: **неправильный формат JSON-сообщения в stdin**. Бот отправлял `{"type": "user_message", ...}`, а Claude CLI ожидает `{"type": "user", "message": {"role": "user", ...}}`. После исправления формата Claude CLI мгновенно ответил.

### Изменённые файлы

- `src/claude_manager/claude_runner.py` — изменён — формат сообщения в `send_message()` (строка 90): `{"type": "user_message", "content": text}` → `{"type": "user", "message": {"role": "user", "content": text}}`
- `tests/test_claude_runner.py` — изменён — обновлены 3 теста (`test_send_message_writes_json_to_stdin`, `test_send_message_with_unicode`, `test_send_message_with_special_json_characters`): проверка нового формата `parsed["message"]["content"]` вместо `parsed["content"]`

### Решения

- **Решение**: изменить формат JSON-сообщения на `{"type": "user", "message": {"role": "user", "content": text}}`. **Причина**: ручной тест из командной строки подтвердил — Claude CLI с `--input-format stream-json` молча игнорирует сообщения в старом формате `user_message`. Новый формат (из официального Agent SDK) работает — Claude ответил за 2.7 секунды.

### Проблемы и решения

- **Проблема**: ленивый запуск (`eca5e5c`) не решил проблему — Claude CLI (PID 40028) снова завис при запуске из бота (210MB памяти, 0% CPU, нет сетевых соединений). **Решение**: проблема не в гонке, а глубже — Claude CLI не понимает формат сообщения и молча висит.
- **Проблема**: Claude CLI в режиме `--input-format stream-json` зависает даже при запуске из терминала (`echo '...' | claude -p --output-format stream-json --verbose --input-format stream-json`), а в обычном `-p` режиме (`echo "текст" | claude -p`) — работает. **Решение**: поиск по документации и GitHub issues. Найдено что формат `user_message` недокументирован, а SDK используют формат `{"type": "user", "message": {...}}`.
- **Проблема**: при перезапуске бота `kill PID` не срабатывал из Claude Code (нет прав на `-9`). **Решение**: пользователь убивал процесс вручную через `! kill -9 PID`.
- **Проблема**: бот, запущенный через `run_in_background`, отчитывался как завершённый, хотя процесс продолжал работать. **Причина**: фоновая задача Claude Code завершилась (stdin/stdout отвязаны), но процесс бота продолжал работать в фоне.

### Ключевые находки из исследования

Агент нашёл в интернете важную информацию о протоколе stream-json:

- **Правильный формат stdin**: `{"type": "user", "message": {"role": "user", "content": "текст"}}` (из Agent SDK Python/TypeScript)
- **Баг буферизации stdout** (GitHub issue #25670): при pipe вывод буферизуется блоками, а не построчно. На macOS нет `stdbuf -oL`.
- **Баг зависания после result** (issue #25629): после финального `result` процесс не завершается, stdout остаётся открытым.
- **Зависание на втором сообщении** (issue #3187): после первого ответа второе сообщение может вызвать зависание.
- **Документация stdin-формата отсутствует** (issue #24594): формат `--input-format stream-json` официально не задокументирован.

### Выполненные команды

- `echo '{"type":"user_message",...}' | claude -p --output-format stream-json --verbose --input-format stream-json --dangerously-skip-permissions` — тест старого формата: **зависает**
- `echo "скажи одно слово" | claude -p --dangerously-skip-permissions` — тест обычного `-p` режима: **работает** (ответ «Привет.»)
- `echo '{"type":"user","message":{"role":"user","content":"..."}}' | claude -p --output-format stream-json --verbose --input-format stream-json --dangerously-skip-permissions` — тест нового формата: **работает** (ответ за 2.7 сек, полный поток событий system→assistant→rate_limit→result)
- `python -m pytest tests/ -v -x` — 382 теста зелёные после исправления формата
