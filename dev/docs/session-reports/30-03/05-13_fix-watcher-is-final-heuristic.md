# Сессия 30-03: исправление эвристики is_final в watcher

## Резюме

Повторное расследование проблемы: промежуточные watcher-сообщения показывают ✅ вместо ⏳. Предыдущий фикс (коммит `ea3ec95`) не работал — эвристика «последнее в пачке = финальное» ошибочна при 2-секундном поллинге. Откатил старый фикс, применил новый: теперь watcher проверяет тип последнего события в JSONL-файле.

## Решения

- **Откат коммита ea3ec95 вместо патча поверх**: пользователь явно попросил откатить прошлые изменения. Revert через `git revert`, затем чистая реализация нового подхода
- **Эвристика по типу последнего события JSONL вместо «последнее в пачке»**: исследование реальных JSONL-файлов показало: (1) событий `result` в файлах нет; (2) типы: assistant, progress, user, system, file-history-snapshot, queue-operation, last-prompt; (3) между текстовыми ответами всегда есть progress/tool_use события. Проверка типа последнего события надёжно определяет «Claude ещё работает» vs «обмен завершён»
- **Ограничение принято**: если Claude закончил и файл заканчивается на assistant (нет user-события после) — последнее сообщение останется с ⏳. Это гораздо лучше текущего «всегда ✅» и не требует редактирования Telegram-сообщений

## Проблемы и решения

- **Проблема**: предыдущий фикс использовал эвристику `is_final = (index == last_index)` — при 2-секундном поллинге обычно приходит 1 сообщение за цикл, оно всегда «последнее в пачке» → всегда ✅. **Решение**: заменена на проверку типа последнего события в JSONL-файле. `ACTIVE_EVENT_TYPES = {"assistant", "progress", "queue-operation"}` — если файл заканчивается одним из них, Claude работает → ⏳
- **Проблема**: JSONL-файлы сессий НЕ содержат событий `result` (они есть только в потоковом протоколе stdout). **Решение**: использованы другие типы событий — `user`, `system`, `last-prompt` как маркеры завершения обмена
