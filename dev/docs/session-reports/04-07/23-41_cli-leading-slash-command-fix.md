# Сессия 04-07: фикс «Unknown command» для сообщений, начинающихся со слэша

## Коротко

Чинили баг: когда пользователь копировал и вставлял боту прошлое сообщение, а оно начиналось со слэша (`/`), бот отвечал «Unknown command». Оказалось, что это писал не бот, а сама программа Claude Code (утилита командной строки, с которой бот общается): текст, начинающийся со слэша, она принимает за свою внутреннюю команду и отклоняет ещё до запуска модели. Лечение — перед отправкой в утилиту бот подставляет один пробел в начало любого сообщения, которое начинается со слэша. Видимый текст не меняется, но утилита больше не видит команду и передаёт текст модели как обычный. Бота перезапустили, чтобы фикс заработал вживую, и задокументировали изменение.

## Рабочие файлы

- **src/claude_manager/process_send.py** — здесь сам фикс. Добавлена функция `_prevent_cli_slash_command_misparse` (если текст начинается с `/`, добавляет ведущий пробел) и её вызов в начале диспетчера `send_message` — единого узла, через который проходит весь текст пользователя перед отправкой в CLI.
- **tests/test_process_manager.py** — новые тесты: чистой функции (со слэша → пробел; без слэша → без изменений) и поведения `send_message` (в CLI уходит текст с ведущим пробелом для слэш-сообщений и без изменений для обычных).
- **dev/docs/claude-cli-stream-json-protocol.md** — в раздел про пользовательское сообщение добавлена заметка о контракте «ведущий `/` = slash-команда» и способе обезвреживания.
- **dev/docs/adr/04.07_23.41-session-change-documenter-cli-leading-slash-command-neutralization.md** — ADR с решением, альтернативами и обоснованием.
- **dev/docs/brd/brd-user-journeys.md** — обновлён сценарий CJM-02 (отправка текстового сообщения): вставленный текст с любым ведущим слэшем теперь уходит в Claude как обычное сообщение, и описан механизм обезвреживания.
- **dev/docs/adr/project_architecture.md** — в разделе «Отправка в Claude (stdin)» добавлена короткая заметка про нейтрализацию ведущего слэша.

## Решения

- **Лечить в едином диспетчере `process_send.send_message`, а не в транспортном слое или в runner.** Причина: через `send_message` проходят все пути (обычный текст, адресный reply, медиагруппы) и оба движка (Claude и Codex), а ретраи переиспользуют тот же уже обезвреженный текст. Одна точка правки закрывает всё.
- **Добавлять пробел, а не удалять слэш.** Причина: удаление ведущего `/` испортило бы легитимный контент — например, путь `/home/ivan/readme.md` превратился бы в `home/ivan/readme.md`. Пробел сохраняет точное видимое содержимое.
- **Пробел ставится только если текст начинается со слэша.** Обычный текст и слэши внутри строки (`посмотри /home/ivan/foo`) не трогаются.

## Риски и ограничения

- **Про Codex — предположение по аналогии.** Эмпирически проверен только Claude CLI. Для второго движка (Codex) фикс тоже применяется (диспетчер общий), и ведущий пробел ему точно не вредит; но что Codex так же отклоняет ведущий слэш — не проверялось на реальном Codex. Если появятся образцы поведения Codex — стоит подтвердить.
- **Ведущий пробел устойчив для текущей версии Claude CLI.** Проверено на установленной версии. Если Claude CLI когда-нибудь начнёт обрезать ведущие пробелы перед разбором команды — фикс придётся пересмотреть (тогда «Unknown command» вернётся, и это будет сигналом).
- В строке лога до перезапуска мелькал `RuntimeError: store locked` с трейсбеком из `mock.py` — это записи ДО рестарта, к правке отношения не имеют и в этой сессии не разбирались.
