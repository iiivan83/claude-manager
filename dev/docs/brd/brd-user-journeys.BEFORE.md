# BRD: Все пользовательские пути (CJM) Claude Manager

## О чём этот документ

Здесь описаны все сценарии того, как пользователь взаимодействует с Telegram-ботом Claude Manager. Каждый сценарий — это CJM (Customer Journey Map, то есть «карта пути пользователя»): что пользователь делает, что видит в ответ, и что может пойти не так.

Документ написан простым языком. Технические термины объясняются в скобках.

**Дата:** 29-03-2026
**Источник:** `development/specs/FULL-TECHNICAL-SPEC.md` v2.0

---

## CJM-01: Первый запуск и настройка

Пользователь хочет впервые запустить бота на своём компьютере.

### Что нужно заранее

- **Python 3.13** — язык программирования, на котором написан бот
- **Claude Code CLI** — программа Claude, которая должна быть установлена и доступна из терминала (командной строки)
- **macOS** — бот работает только на маке, потому что использует специфичный для macOS способ блокировки файлов

### Что делает пользователь

1. Скачивает (клонирует) проект на свой компьютер
2. Копирует файл `.env.example` в `.env` — это файл с настройками, который не попадает в общий доступ
3. Заполняет в `.env` два обязательных поля:
   - **TELEGRAM_BOT_TOKEN** — уникальный ключ бота, который выдаёт Telegram при создании бота через @BotFather
   - **ALLOWED_USER_IDS** — список Telegram-ID пользователей, которым разрешено пользоваться ботом (защита от чужих людей)
4. По желанию можно настроить дополнительные параметры:
   - **CLAUDE_WORKING_DIR** — папка проекта, с которой Claude будет работать по умолчанию
   - **IDLE_TIMEOUT_SECONDS** — через сколько секунд бездействия выключать процесс Claude (по умолчанию 7200 секунд, то есть 2 часа)
5. Устанавливает зависимости (дополнительные библиотеки, нужные для работы): `pip install -r requirements.txt`
6. Запускает бота: `python -m claude_manager`

### Что происходит при запуске

Бот проходит через несколько этапов:

1. **Проверка настроек** — бот читает файл `.env` и проверяет, что обязательные поля заполнены. Если чего-то не хватает — программа сразу останавливается и пишет понятное сообщение о том, что именно забыли указать

2. **Защита от двойного запуска** — бот создаёт специальный файл-замок `bot.pid`. Если другая копия бота уже работает и держит этот замок — новая копия не сможет запуститься и завершится. Это нужно, чтобы два бота не мешали друг другу

3. **Подключение к Telegram** — бот начинает слушать сообщения от Telegram. Все старые сообщения (которые пришли пока бот был выключен) пропускаются — бот реагирует только на новые

4. **Настройка подсказок** — бот устанавливает меню команд в Telegram (кнопка «/» в поле ввода), чтобы пользователь видел список доступных команд

### Что видит пользователь

Бот запущен и молча ждёт сообщений. Пользователь открывает Telegram, находит бота и может начать общение.

### Что может пойти не так

- **Забыли указать токен бота** — программа сразу завершается с сообщением «TELEGRAM_BOT_TOKEN не задан»
- **Забыли указать список разрешённых пользователей** — программа завершается с сообщением «ALLOWED_USER_IDS не задан»
- **Бот уже запущен в другом окне терминала** — новая копия не сможет захватить файл-замок и завершится. Это защита, а не ошибка
- **Настройки из .env перезаписывают системные** — если одна и та же переменная задана и в системе, и в `.env`, победит значение из `.env`

## CJM-02: Отправка текстового сообщения

Это основной сценарий — пользователь пишет сообщение в Telegram, а Claude отвечает.

### Что делает пользователь

1. Открывает чат с ботом в Telegram
2. Пишет текстовое сообщение — например, «Посмотри файл main.py и скажи что он делает»

### Что происходит внутри (полная цепочка)

**Шаг 1 — Проверка «свой или чужой»**

Бот проверяет Telegram-ID отправителя по белому списку из настроек. Если пользователя нет в списке — сообщение молча игнорируется (чужой человек даже не узнает, что бот получил его сообщение).

**Шаг 2 — Проверка текущего состояния**

Бот проверяет, есть ли у пользователя активная сессия (диалог с Claude). Тут возможны два варианта:

- **Сессия есть** — отлично, сообщение пойдёт в этот диалог
- **Сессии нет** — будет создан новый диалог автоматически


не так, если пользователь не в сессии, то нужно сообщить пользователю, что нужно войти сначала в сесисию, потом уже писать, новую просто так создавать не надо

**Шаг 3 — Передача сообщения в Claude**

Сообщение передаётся напрямую в текущую сессию Claude без ожидания — даже если Claude прямо сейчас обрабатывает предыдущее сообщение. Claude сам разберётся с порядком обработки в рамках своей сессии.

> **Открытый вопрос:** нужно проверить техническую возможность отправки сообщений в сессию, пока Claude занят предыдущим запросом. Протокол обмена может быть не через stdin — выбор конкретного механизма будет определён при реализации.

**Шаг 4 — Подготовка к отправке**

- В Telegram появляется индикатор «печатает...» — он держится всё время, пока Claude обрабатывает запрос
- Бот отправляет подтверждение, что сообщение передано в Claude, с указанием номера сессии - этот шаг делать не нужно

**Шаг 5 — Создание новой сессии (если нужно)**

Если это новый диалог — бот резервирует дневной номер (например, #3 — третий диалог за сегодня) и передаёт его Claude как имя сессии. Пользователь видит подтверждение с номером новой сессии.
Это происходить только при выполнении команды нью

**Шаг 6 — Обработка ответа от Claude**

Когда Claude отвечает:

- Бот сохраняет связку «этот Telegram-чат привязан к этой сессии Claude» — чтобы следующие сообщения шли в тот же диалог
- Привязывает дневной номер (#1, #2...) к сессии
- Конвертирует ответ из Markdown (формат, в котором отвечает Claude — со звёздочками для жирного, обратными кавычками для кода) в HTML (формат, который понимает Telegram — с тегами `<b>`, `<code>`)
- Добавляет заголовок с номером сессии и галочкой: **#3** ✅ перед текстом ответа
- Если ответ длиннее 4096 символов (ограничение Telegram) — разбивает на несколько сообщений. При этом чинит HTML-теги: если жирный текст начался в одном сообщении, но продолжается в следующем — теги корректно закрываются и открываются
- Отправляет в Telegram. Если HTML-форматирование не сработало (ошибка парсинга) — отправляет обычным текстом без форматирования

### Что видит пользователь

1. Отправляет сообщение
2. Видит подтверждение с номером сессии
3. Видит «печатает...» в чате (всё время пока Claude работает)
4. Возможно видит промежуточные обновления (что Claude сейчас делает) — они приходят не чаще раза в 30 секунд, показываются курсивом с номером сессии и значком ⏳. Промежуточные обновления показываются полностью, без обрезки
Очень важно, чтобы в них тоже был номер сессии по всем правилам, но с песочными часами


5. Получает полный ответ Claude с форматированием и номером сессии с галочкой

### Что может пойти не так

- **Пустой ответ** — Claude обработал запрос (например, отредактировал файл), но не написал текст в ответ. Пользователь видит «Claude обработал запрос, но не дал текстовый ответ»
- **Ответ «No response requested.»** — это служебный ответ Claude, он не пересылается пользователю. Вместо него — то же сообщение о пустом ответе
- **Сессия не найдена** — Claude не нашёл указанную сессию (например, файл был удалён). Бот сообщает пользователю, что сессия не найдена
- **Ответ слишком длинный** — разбивается на несколько сообщений по 4096 символов с правильным форматированием

## CJM-03: Отправка фотографии

Пользователь может отправить боту фотографию — Claude посмотрит на неё и ответит.

### Что делает пользователь

1. Отправляет фотографию в чат с ботом
2. По желанию добавляет подпись — например, «Что здесь не так?» или «Перепиши текст с этого скриншота»

### Что происходит внутри

1. **Все те же проверки**, что и при текстовом сообщении — доступ, сессия (см. CJM-02)

2. **Скачивание фото** — бот скачивает фотографию в максимальном качестве из Telegram

3. **Сохранение на диск** — фото сохраняется в папку `received_photos/` с именем вида `photo_20260328_143022_abc123.jpg` (дата, время, уникальный код)

4. **Формирование задачи для Claude** — бот не отправляет фото напрямую, а составляет текстовое сообщение для Claude:
   - **Если пользователь добавил подпись:** «Пользователь отправил фотографию с подписью: [подпись]. Файл: [путь к файлу]. Прочитай файл инструментом Read и выполни задачу из подписи»
   - **Если подписи нет:** «Пользователь отправил фотографию без подписи. Файл: [путь к файлу]. Прочитай файл и опиши, что на фотографии»

5. **Claude читает файл** — получив это сообщение, Claude сам открывает сохранённый файл своим инструментом Read (инструмент для чтения файлов) и «видит» фотографию

6. Дальше всё как с текстовым сообщением — ответ конвертируется и отправляется в Telegram

### Что видит пользователь

1. Отправляет фото (с подписью или без)
2. Видит «печатает...»
3. Получает ответ Claude — описание фото или результат выполнения задачи из подписи

### Что может пойти не так

- **Не удалось скачать фото** — Telegram мог не отдать файл. Пользователь видит сообщение об ошибке
- **Фото без подписи** — Claude сам решит, что с ним делать (обычно описывает содержимое)

## CJM-04: Создание новой сессии (/new)

Пользователь хочет начать разговор с Claude с чистого листа — создать новый диалог.

### Что делает пользователь

Отправляет команду `/new`

### Что происходит внутри

Команда `/new` — это просто создание новой сессии. Никакие процессы не убиваются, никакие связки не разрываются:

1. **Создание новой сессии** — бот резервирует новый дневной номер (например, #4) и создаёт новую сессию Claude
2. **Привязка чата к новой сессии** — теперь сообщения пользователя идут в этот новый диалог
3. **Остальные сессии продолжают работать** — если в другой сессии Claude что-то делает, ответы оттуда продолжают приходить пользователю
4. **Подтверждение** — пользователь видит сообщение о создании новой сессии с её номером

### Что видит пользователь

Подтверждение: «Создана новая сессия #4». Теперь можно писать — и это будет новый диалог. Сообщения из остальных сессий продолжают приходить.

### Что может пойти не так

- **Ничего особенного** — команда простая и надёжная

## CJM-05: Просмотр списка сессий (/sessions)

Пользователь хочет увидеть последние диалоги с Claude и выбрать, к какому подключиться.

### Что делает пользователь

Отправляет команду `/sessions`

### Что происходит внутри

1. **Чтение файлов с диска** — бот заглядывает в папку, где Claude Code хранит свои сессии (`~/.claude/projects/...`). Путь к проекту кодируется: слеши и пробелы заменяются дефисами (например, `/Users/ivan/Desktop/my project` превращается в `-Users-ivan-Desktop-my-project`)

2. **Выбор последних 15 сессий** — бот берёт 15 самых свежих сессий, отсортированных по времени

3. **Извлечение информации о каждой сессии:**
   - Уникальный идентификатор сессии
   - Время создания
   - Первое сообщение пользователя — для превью (чтобы понять, о чём был диалог)

4. **Регистрация в дневном реестре** — каждая сессия получает номер в рамках текущего дня (#1, #2, #3...). Если сессия уже была зарегистрирована — номер не меняется

5. **Очистка превью** — из первого сообщения убираются служебные XML-теги Claude Code (технический мусор, который не нужен пользователю)

6. **Формирование списка** — для каждой сессии показывается: номер + начало первого сообщения (до 120 символов). Новые сессии — сверху

7. **Отправка без HTML-форматирования** — это сделано намеренно, чтобы Telegram автоматически подсвечивал `/1`, `/2`, `/3` как кликабельные команды

### Что видит пользователь

Список вроде такого:

```
/3 Посмотри файл main.py и скажи что он делает
/2 Добавь обработку ошибок в функцию send_message
/1 Привет, расскажи про этот проект
```

Номера кликабельные — нажав на `/3`, пользователь подключится к третьей сессии и увидит подтверждение с номером сессии.

### Что может пойти не так

- **Нет сессий на диске** — пользователь увидит пустой список или сообщение «Нет сессий»
- **Сессия без первого сообщения** — покажется только номер и название

## CJM-06: Переключение на сессию (/1, /2, /3...)

Пользователь хочет подключиться к одному из предыдущих диалогов с Claude.

### Что делает пользователь

Отправляет `/3` — номер нужной сессии из списка `/sessions`

такая возможность должна быть не только из сессий, но и в люое время.


### Что происходит внутри

1. **Поиск сессии** — бот ищет в дневном реестре (списке сессий за сегодня), какая сессия соответствует номеру 3

2. **Привязка чата к сессии** — бот запоминает: «этот Telegram-чат теперь привязан к сессии #3». Теперь все сообщения пользователя идут в этот диалог

3. **Сообщения из других сессий продолжают приходить** — мониторинг всех остальных сессий не прерывается. Если в другой сессии Claude ответит — пользователь увидит полное сообщение с номером сессии в виде кликабельной ссылки (кроме текущей сессии — для неё ссылка неактивна)

4. **Подтверждение** — пользователь видит, к какой сессии подключился, с превью первого сообщения и номером сессии

### Что видит пользователь

Подтверждение подключения с номером сессии и кратким описанием. Теперь все сообщения идут в этот диалог. Сообщения из других сессий приходят с полным текстом и кликабельными номерами для быстрого переключения.

### Что может пойти не так

- **Номер не найден** — если набрать `/99`, а такой сессии сегодня не было — бот сообщит об ошибке

## CJM-07: Мониторинг всех сессий (/all)

Пользователь хочет отключиться от конкретного диалога и просто получать сообщения из всех активных сессий — как «радар».

### Что делает пользователь

Отправляет команду `/all`

### Что происходит внутри

1. Бот отвязывает чат от конкретной сессии — пользователь больше не привязан ни к какому диалогу
2. Запускается мониторинг всех сессий — каждые 2 секунды проверяются все файлы сессий в папке проекта
3. Подтверждение пользователю

### Как работает мониторинг всех сессий

- Каждые 2 секунды проверяет все файлы сессий
- Когда в какой-то сессии Claude что-то написал — пользователю приходит полное сообщение (не короткое уведомление)
- В сообщении номер сессии оформлен как кликабельная ссылка — нажав на неё, пользователь переключается на эту сессию (в поле ввода подставляется `/номер` и сразу отправляется)


Такой же режим должен быть и тогда когда пользователь в сессии за исключением номера сессии в сообщениях, где-то с ссылкой, где0то нет

### Что видит пользователь

Подтверждение: «Режим мониторинга всех сессий». Теперь пользователь видит полные сообщения из всех диалогов и может быстро переключиться на любой, нажав на номер сессии.

### Важные особенности

- **Писать сообщения нельзя** — нет активной сессии, куда их отправлять. Чтобы написать — сначала подключитесь к конкретной сессии через номер или команду
- **Из этого режима можно подключиться к сессии** — нажав на номер сессии в сообщении или набрав `/3`

## CJM-08: Остановка Claude (/stop)

Пользователь хочет прервать работу Claude — например, если задача заняла слишком много времени или была отправлена по ошибке.

### Способ 1: команда /stop

**Что делает пользователь:** отправляет `/stop`

**Что происходит:**
- Если пользователь подключён к сессии — процесс Claude для этой сессии завершается: сначала мягко (сигнал «заверши работу»), если за 5 секунд не завершился — принудительно
- Если пользователь находится в режиме `/all` (не подключён ни к какой сессии) — бот отвечает: «Команда /stop работает только внутри сессии. Подключитесь к сессии через /sessions»
- Пользователь видит подтверждение «Claude остановлен»

### Способ 2: текстовая команда

**Что делает пользователь:** пишет «стоп», «отмена», «stop» или «cancel»

**Что происходит:**
- Сообщение передаётся в Claude как обычное текстовое сообщение
- Claude сам распознаёт команду остановки и прекращает текущую работу в рамках своего процесса
- Это мягкая остановка — Claude сам решает, как корректно завершить текущую задачу

### Важная разница между способами

- **/stop** — принудительная остановка на уровне процесса, бот убивает процесс Claude
- **Текстовая команда** — мягкая остановка, сообщение просто передаётся Claude, и он сам решает что делать

## CJM-09: Защита от двойного запуска бота

Этот механизм не даёт запустить два экземпляра бота одновременно — иначе они будут мешать друг другу.

### Первый уровень: файл-замок

При запуске бот пытается «захватить» специальный файл `bot.pid`:

- **Успешно** → файл-замок принадлежит этому экземпляру, бот записывает свой номер процесса (PID) в файл и продолжает работу
- **Не удалось** → другой экземпляр уже держит замок → программа завершается

Почему именно файл-замок, а не просто проверка «работает ли процесс с таким PID»: файл-замок — это одна неделимая операция. А при проверке PID есть промежуток между «прочитать» и «записать», когда два экземпляра могут решить, что оба первые.

### Второй уровень: защита от Telegram

Даже если каким-то чудом запустились два бота — Telegram не будет с этим мириться. Когда два бота с одним токеном одновременно опрашивают Telegram, API возвращает ошибку Conflict. Бот ловит эту ошибку и завершается.

Если бот настроен на автоматический перезапуск через LaunchAgent (системная служба macOS для автозапуска программ) — перезапустится только один экземпляр.

## Карта состояний пользователя

Пользователь всегда находится в одном из трёх состояний:

### Состояние 1: «Чистый лист» (нет сессии)

См комментарии выше

- **Как сюда попасть:** первый запуск бота
- **Что можно делать:** написать сообщение → создастся новая сессия (переход в состояние 2)
- **Мониторинг:** нет
- **Куда можно перейти:**
  - Написать сообщение → состояние 2
  - `/all` → состояние 3
  - `/номер сессии` → состояние 2

### Состояние 2: «Подключён к сессии» (основное рабочее состояние)

- **Как сюда попасть:** первое сообщение, `/номер сессии`, `/new`
- **Что можно делать:** писать сообщения, отправлять фото, получать ответы Claude
- **Мониторинг:** сообщения из всех сессий приходят полностью. Для текущей сессии — номер без ссылки. Для остальных сессий — номер в виде кликабельной ссылки для быстрого переключения
- **Куда можно перейти:**
  - `/new` → создаёт новую сессию, остаётся в состоянии 2
  - `/all` → состояние 3
  - `/номер другой сессии` → состояние 2 (переключение)

### Состояние 3: «Радар» (мониторинг всех сессий)

- **Как сюда попасть:** команда `/all`
- **Что можно делать:** только смотреть сообщения и переключаться по ссылкам. Писать сообщения нельзя — нет активной сессии
- **Мониторинг:** полные сообщения из всех сессий с кликабельными номерами
- **Куда можно перейти:**
  - `/номер сессии` → состояние 2
  - `/new` → состояние 2

## Механизмы, которые работают во всех сценариях

### Проверка доступа

Перед каждым действием (команда, сообщение, фото) бот проверяет: есть ли Telegram-ID отправителя в белом списке разрешённых пользователей. Если нет — запрос молча игнорируется. В лог записывается предупреждение.

### Промежуточные обновления (прогресс)

Пока Claude думает, бот может присылать промежуточные обновления — что Claude сейчас делает. Ограничения:
- **Не чаще раза в 30 секунд** — чтобы не заспамить пользователя
- **Показываются полностью** — без обрезки
- **Формат:** `#3 ⏳ текст обновления` (номер сессии + песочные часы + текст курсивом)

### Одновременная работа из Telegram и терминала

Пользователь может работать с одной и той же сессией одновременно из Telegram (через бота) и из терминала на компьютере. Сообщения появляются и там и там — никакой блокировки нет. Это позволяет:
- Начать задачу в терминале, а следить за прогрессом через телефон
- Писать Claude и из Telegram, и из терминала — в одну и ту же сессию
- Видеть ответы Claude в обоих местах

### Сообщения из других сессий

Когда пользователь подключён к конкретной сессии (состояние 2), он продолжает получать полные сообщения из всех остальных сессий. Номера сессий в таких сообщениях оформлены как кликабельные ссылки — нажав, можно быстро переключиться. Для текущей сессии номер показывается без ссылки.

### Координация мониторинга и ответов

Когда бот сам отправляет сообщение в Claude, возникает проблема: и наблюдатель (watcher), и обработчик сообщений получат ответ — пользователь увидит его дважды.

Решение — механизм координации:
1. Перед отправкой: наблюдатель за текущей сессией координируется с обработчиком, чтобы не дублировать ответ
2. При этом наблюдатели за другими сессиями продолжают работать без паузы — сообщения из остальных сессий приходят всегда

### Обновление идентификатора сессии

Когда создаётся новая сессия, бот сначала не знает её настоящий идентификатор — использует временный (вида `_new_XXXX`). Настоящий идентификатор приходит от Claude в первом же ответе. Бот обновляет его во всех внутренних словарях — процессы, задачи чтения, блокировки.

### Отправка сообщений в Telegram

- Основной формат — HTML (жирный текст, код, блоки кода)
- Если Telegram не смог обработать HTML — бот убирает все теги и отправляет обычный текст
- Длинные ответы (> 4096 символов) разбиваются на несколько сообщений
- При разбивке чинятся HTML-теги: если `<b>` открылся в одном сообщении, но текст продолжается в следующем — тег закрывается `</b>` в конце первого и открывается `<b>` в начале второго
- Кнопки (если есть) прикрепляются только к последнему сообщению

### Жизненный цикл процесса Claude

- На каждую сессию запускается один процесс Claude — он живёт долго и обрабатывает все сообщения в этом диалоге
- Новый процесс не создаётся на каждое сообщение — переиспользуется существующий
- Каждые 60 секунд бот проверяет все процессы: мёртвые удаляются, бездействующие дольше 2 часов — завершаются
- При выключении бота все процессы Claude завершаются
