# BRD: Все пользовательские пути (CJM) Claude Manager

## О чём этот документ

Здесь описаны все сценарии того, как пользователь взаимодействует с Telegram-ботом Claude Manager. Каждый сценарий — это CJM (Customer Journey Map, то есть «карта пути пользователя»): что пользователь делает, что видит в ответ, и что может пойти не так.

Бот рассчитан на одного пользователя — управляет Claude Code на личном компьютере с телефона.

Документ написан простым языком. Технические термины объясняются в скобках.

**Дата:** 29-03-2026
**Источник:** техническая спецификация проекта (v2.0)

## CJM-01: Первый запуск и настройка

Пользователь хочет впервые запустить бота на своём компьютере.

### Что нужно заранее

- **Python 3.13** — язык программирования, на котором написан бот
- **Claude Code CLI** — программа Claude, которая должна быть установлена и доступна из терминала (командной строки)
- **Linux** — бот работает под Linux с systemd. Использует `fcntl` для файловой блокировки от двойного запуска и `systemd --user` для автозапуска

### Что делает пользователь

1. Скачивает (клонирует) проект на свой компьютер
2. Копирует файл `.env.example` в `.env` — это файл с настройками, который не попадает в общий доступ
3. Заполняет в `.env` два обязательных поля:
   - **TELEGRAM_BOT_TOKEN** — уникальный ключ бота, который выдаёт Telegram при создании бота через @BotFather
   - **ALLOWED_USER_IDS** — Telegram-ID пользователя, которому разрешено пользоваться ботом (защита от чужих людей). Можно указать несколько ID через запятую — для одного человека с разных устройств

> **Важно:** добавление ID другого человека не поддерживается и приведёт к конфликтам состояния — бот архитектурно однопользовательский. Подробности в CLAUDE.md, раздел «Однопользовательский инвариант».

4. По желанию можно настроить дополнительные параметры:
   - **CLAUDE_WORKING_DIR** — папка проекта, с которой Claude будет работать по умолчанию
   - **OPENAI_API_KEY** — ключ OpenAI API для распознавания голосовых и аудиосообщений. Без него бот продолжит работать с текстом, фото и файлами, но аудио не сможет превратить в текст для агента
   - **OPENAI_TRANSCRIPTION_MODEL** — модель распознавания речи. Если не указана, используется `whisper-1`
5. Устанавливает зависимости (дополнительные библиотеки, нужные для работы): `pip install -r requirements.txt`
6. Запускает бота: `python -m claude_manager`

### Что происходит при запуске

Бот проходит через несколько этапов:

1. **Проверка настроек** — бот читает файл `.env` и проверяет, что обязательные поля заполнены. Если чего-то не хватает — программа сразу останавливается и пишет понятное сообщение о том, что именно забыли указать

2. **Защита от двойного запуска** — бот создаёт специальный файл-замок `bot.pid`. Если другая копия бота уже работает и держит этот замок — новая копия не сможет запуститься и завершится. Это нужно, чтобы два бота не мешали друг другу

3. **Восстановление состояния** — бот читает сохранённые данные (sessions.json) и восстанавливает привязки: какой Telegram-чат был подключён к какой сессии Claude. Также сканирует сессии Claude на диске и присваивает дневные номера новым сессиям. Пользователь оказывается в том же состоянии, в котором был до перезапуска

4. **Подключение к Telegram** — бот начинает слушать сообщения от Telegram. Все старые сообщения (которые пришли пока бот был выключен) пропускаются — бот реагирует только на новые

5. **Настройка подсказок** — бот устанавливает меню команд в Telegram (кнопка «/» в поле ввода), чтобы пользователь видел список доступных команд

### Что видит пользователь

Бот запущен и молча ждёт сообщений. Пользователь открывает Telegram, находит бота и может начать общение.

### Что может пойти не так

- **Забыли указать токен бота** — программа сразу завершается с сообщением «TELEGRAM_BOT_TOKEN не задан»
- **Забыли указать список разрешённых пользователей** — программа завершается с сообщением «ALLOWED_USER_IDS не задан»
- **Не указали OpenAI API key** — бот запустится, но при отправке voice/audio ответит, что распознавание аудио не настроено. Это не мешает текстовым сообщениям, фото и файлам
- **Бот уже запущен в другом окне терминала** — новая копия не сможет захватить файл-замок и завершится. Это защита, а не ошибка
- **Настройки из .env перезаписывают системные** — если одна и та же переменная задана и в системе, и в `.env`, победит значение из `.env`

## CJM-02: Отправка текстового сообщения

Это основной сценарий — пользователь пишет сообщение в Telegram, а Claude отвечает.

### Что делает пользователь

1. Открывает чат с ботом в Telegram
2. Пишет текстовое сообщение — например, «Посмотри файл main.py и скажи что он делает»
3. Может вставить текст, скопированный из ответа бота. Если такой текст начинается с кликабельного номера сессии, например `/8 ...` или `/3s12 ...`, и после номера есть обычный текст, бот считает это новым сообщением, а не командой переключения

### Что происходит внутри (полная цепочка)

**Шаг 1 — Проверка «свой или чужой»**

Бот проверяет Telegram-ID отправителя по белому списку из настроек. Если пользователя нет в списке — сообщение молча игнорируется (чужой человек даже не узнает, что бот получил его сообщение).

**Шаг 2 — Проверка текущего состояния**

Бот проверяет, есть ли у пользователя активная сессия (диалог с Claude). Тут возможны два варианта:

- **Сессия есть** — отлично, сообщение пойдёт в этот диалог
- **Сессии нет** — бот отвечает: «Вы не подключены к сессии. Создайте новую через /new или подключитесь к существующей через /sessions». Сообщение не отправляется в Claude

**Шаг 3 — Передача сообщения в Claude**

Сообщение передаётся напрямую в текущую сессию Claude.

Если сообщение начинается с номера сессии и текста после него, например `/8 ⚡ Codex ✅ ...`, Telegram помечает начало строки как команду. Бот отдельно распознаёт этот случай: точная команда `/8` переключает сессию, а строка `/8 ...` с продолжением идёт в текущую сессию как обычное сообщение.

**Шаг 4 — Подготовка к отправке**

- В Telegram появляется индикатор «печатает...» — он держится всё время, пока Claude обрабатывает запрос

**Шаг 5 — Отправка в существующую сессию**

Сообщение передаётся в текущую активную сессию. Каждый процесс Claude запускается с флагом `--effort max` — максимальная глубина «размышлений» модели (extended thinking). Claude дольше думает перед ответом, но выдаёт более качественные результаты. Создание новой сессии с резервированием дневного номера происходит только по команде `/new` (см. CJM-04)

**Шаг 6 — Обработка ответа от Claude**

Когда Claude отвечает:

- Бот сохраняет связку «этот Telegram-чат привязан к этой сессии Claude» — чтобы следующие сообщения шли в тот же диалог
- Привязывает дневной номер (#1, #2...) к сессии
- Конвертирует ответ из Markdown (формат, в котором отвечает Claude — со звёздочками для жирного, обратными кавычками для кода) в HTML (формат, который понимает Telegram — с тегами `<b>`, `<code>`)
- Добавляет заголовок с номером сессии и галочкой: **#3** ✅ перед текстом ответа
- Если ответ длиннее 4096 символов (ограничение Telegram) — разбивает на несколько сообщений. При этом чинит HTML-теги: если жирный текст начался в одном сообщении, но продолжается в следующем — теги корректно закрываются и открываются
- Отправляет в Telegram. Если HTML-форматирование не сработало (ошибка парсинга) — отправляет обычным текстом без форматирования

### Что видит пользователь

1. Отправляет сообщение
2. Видит «печатает...» в чате (всё время пока Claude работает)
3. Возможно видит промежуточные обновления (что Claude сейчас делает) — они приходят не чаще раза в 30 секунд, показываются полностью, без обрезки. Формат: **#3 ⏳** текст обновления курсивом. Номер сессии обязателен — по тем же правилам, что и в финальных ответах (кликабельная ссылка для чужих сессий, без ссылки для текущей)
4. Получает полный ответ Claude с форматированием и номером сессии с галочкой

### Что может пойти не так

- **Пустой ответ** — Claude обработал запрос (например, отредактировал файл), но не написал текст в ответ. Пользователь видит «Claude обработал запрос, но не дал текстовый ответ»
- **Ответ «No response requested.»** — это служебный ответ Claude, он не пересылается пользователю. Вместо него — то же сообщение о пустом ответе
- **Сессия не найдена** — Claude не нашёл указанную сессию (например, файл был удалён). Бот сообщает пользователю, что сессия не найдена
- **Ответ слишком длинный** — разбивается на несколько сообщений по 4096 символов с правильным форматированием
- **Временная ошибка от Claude** (сбой API, обрыв соединения и подобное — повтор может помочь) — бот сообщает пользователю о проблеме и автоматически повторяет попытку. Всего до 10 попыток с интервалом в 1 минуту. О каждой попытке пользователь получает уведомление
- **Постоянная ошибка от Claude** (повтор бессмыслен — каждая попытка грузит то же самое состояние) — бот распознаёт такие ошибки и НЕ запускает цикл повторов, а сразу отправляет понятное сообщение. Два случая: **переполнение сессии** (история перестала помещаться в модель) → «Сессия переполнилась и больше не может принимать сообщения. Начни новую через /new»; **исчерпан лимит запросов к Claude** → «Исчерпан лимит запросов к Claude — повтор не поможет. Дождись обновления лимита и попробуй снова». Распознавание зависит от выбранного движка: для Claude эти ошибки определяются по тексту ответа, для Codex пока все ошибки считаются временными (повтор запускается, пока не появятся образцы ошибок Codex)

## CJM-03: Отправка фотографии или файла

Пользователь может отправить боту фотографию или файл — Claude посмотрит и ответит.

### Что делает пользователь

1. Отправляет фотографию или файл в чат с ботом. Способ отправки не важен — бот обработает и обычное фото (сжатое Telegram), и картинку отправленную «как файл» (без сжатия), и другие документы (PDF, текстовые файлы и т.д.)
2. По желанию добавляет подпись — например, «Что здесь не так?» или «Перепиши текст с этого скриншота»

### Что происходит внутри

1. **Все те же проверки**, что и при текстовом сообщении — доступ, сессия (см. CJM-02)

2. **Скачивание файла** — бот скачивает фотографию (в максимальном качестве) или документ из Telegram

3. **Сохранение на диск** — файл сохраняется в папку `received_files/` с именем вида `file_20260328_143022_abc123.jpg` (дата, время, уникальный код)

4. **Формирование задачи для Claude** — бот не отправляет файл напрямую, а составляет текстовое сообщение для Claude:
   - **Если пользователь добавил подпись:** «Пользователь отправил файл с подписью: [подпись]. Файл: [путь к файлу]. Прочитай файл инструментом Read и выполни задачу из подписи»
   - **Если подписи нет и это изображение:** «Пользователь отправил фотографию без подписи. Файл: [путь к файлу]. Прочитай файл и опиши, что на фотографии»
   - **Если подписи нет и это документ:** «Пользователь отправил файл без подписи. Файл: [путь к файлу]. Прочитай файл и опиши его содержимое»

5. **Claude читает файл** — получив это сообщение, Claude сам открывает сохранённый файл своим инструментом Read (инструмент для чтения файлов) и «видит» фотографию или читает документ

6. Дальше всё как с текстовым сообщением — ответ конвертируется и отправляется в Telegram

### Что видит пользователь

1. Отправляет фото или файл (с подписью или без)
2. Видит «печатает...»
3. Получает ответ Claude — описание фото, содержимое документа или результат выполнения задачи из подписи

### Что может пойти не так

- **Не удалось скачать файл** — Telegram мог не отдать файл. Пользователь видит сообщение об ошибке
- **Файл без подписи** — Claude сам решит, что делать (фото — опишет, документ — расскажет о содержимом)
- **Неподдерживаемый формат** — Claude может не суметь прочитать некоторые форматы файлов. В этом случае пользователь получит ответ от Claude с объяснением

## CJM-04: Создание новой сессии (/new)

Пользователь хочет начать разговор с Claude с чистого листа — создать новый диалог.

### Что делает пользователь

Отправляет команду `/new`

### Что происходит внутри

Команда `/new` — это просто создание новой сессии. Никакие процессы не убиваются, никакие связки не разрываются:

1. **Создание новой сессии** — бот резервирует новый дневной номер (например, #4) и создаёт новую сессию Claude
2. **Привязка чата к новой сессии** — теперь сообщения пользователя идут в этот новый диалог
3. **Остальные сессии продолжают работать** — если в другой сессии Claude что-то делает, ответы оттуда продолжают приходить пользователю
4. **Подтверждение** — пользователь видит сообщение о создании новой сессии с её номером

### Что видит пользователь

Подтверждение: «Создана новая сессия #4». Теперь можно писать — и это будет новый диалог. Сообщения из остальных сессий продолжают приходить.

### Что может пойти не так

- **Ничего особенного** — команда простая и надёжная

## CJM-05: Просмотр списка сессий (/sessions)

Пользователь хочет увидеть последние диалоги с Claude и выбрать, к какому подключиться.

### Что делает пользователь

Отправляет команду `/sessions`

### Что происходит внутри

1. **Чтение индекса последних сессий** — бот сначала читает `recent_sessions`: небольшую SQLite-таблицу с быстрыми заголовками сессий. В ней хранятся backend, путь проекта, session_id, путь к файлу сессии, время изменения и короткое превью. Полный текст диалога остаётся в исходных JSONL-файлах Claude и Codex, индекс не становится новой историей сообщений.

2. **Ограниченное обновление при пустом индексе** — если для текущего проекта в `recent_sessions` нет строк, бот делает один ограниченный refresh. Для Claude это чтение файлов проекта, для Codex — bounded-поиск по рабочему окну, а не полный обход всей истории `~/.codex/sessions/`. Если строки уже есть, пользователь сразу получает список, а обновление может пройти в фоне.

3. **Отбор пользовательских диалогов** — в список попадают только сессии, которые были начаты пользователем. Codex-сессии субагентов (рабочие и reviewer-запуски, созданные из другой сессии) распознаются по metadata в JSONL: `thread_source: subagent` или `source.subagent`. Такие строки не показываются в `/sessions`; если старая строка уже лежала в SQLite-индексе, бот помечает её скрытой при чтении.

4. **Выбор последних 15 сессий** — бот показывает 15 самых свежих пользовательских сессий, отсортированных по времени. Сам индекс хранит до 30 последних сессий на проект across all backends, чтобы у UI был небольшой запас

5. **Извлечение информации о каждой сессии:**
   - Уникальный идентификатор сессии
   - Время создания
   - Превью исходного запроса пользователя — для понимания, о чём был диалог. Если сессия началась с файла с подписью, берётся сама подпись, а не служебный текст бота с путём к файлу

6. **Регистрация в дневном реестре** — каждая сессия получает номер в рамках текущего дня (#1, #2, #3...). Если сессия уже была зарегистрирована — номер не меняется. Нумерация сбрасывается в полночь по локальному времени. Номера предыдущего дня продолжают работать, но постепенно перезаписываются новыми сессиями

7. **Очистка превью** — из исходного текста убираются служебные XML-теги Claude Code и техническая обёртка файловых задач

8. **Формирование списка** — для каждой сессии показывается: кликабельный номер, короткая иконка backend-а и очищенный заголовок исходного запроса. Название backend-а (`Claude` или `Codex`) в строке списка не дублируется. Длинные заголовки сжимаются до компактного двухстрочного бюджета Telegram: сейчас это 120 символов с `...` в конце. Если в быстром индексе остался старый заголовок с `...`, бот сначала пробует дочитать полный первый пользовательский запрос из файла сессии, а потом применяет компактный лимит

9. **Отправка без HTML-форматирования** — это сделано намеренно, чтобы Telegram автоматически подсвечивал `/1`, `/2`, `/3` как кликабельные команды

### Что видит пользователь

Список вроде такого:

```
/3 ⚡ Посмотри файл main.py и скажи что он делает
/2 🤖 Добавь обработку ошибок в функцию send_message
/1 ⚡ Привет, расскажи про этот проект
```

Номера кликабельные — нажав на `/3`, пользователь подключится к третьей сессии и увидит подтверждение с номером сессии. Иконка помогает отличить backend, но не забирает место повтором названия модели.

### Что может пойти не так

- **Нет проиндексированных сессий** — если индекс пуст и ограниченное обновление тоже ничего не нашло, пользователь увидит «Нет сессий»
- **Видны не все Codex-файлы** — это нормально, если часть файлов является субагентами. Reviewer, worker и другие spawned-сессии скрываются из пользовательского списка, чтобы не смешивать обычные диалоги с внутренней агентной работой
- **Сессия без первого сообщения** — покажется только номер и название
- **Старая сессия не видна в списке** — `recent_sessions` хранит последние 30 сессий на проект. Более старые файлы не удаляются, но могут не попадать в быстрый список `/sessions`
- **Индекс временно недоступен** — если SQLite-файл заблокирован, повреждён или refresh не смог прочитать backend, бот показывает понятную ошибку или добавляет degraded-сообщение к списку, не переходя к полному scan истории

## CJM-06: Переключение на сессию (/1, /2, /3...)

Пользователь хочет подключиться к одному из предыдущих диалогов с Claude.

### Что делает пользователь

Отправляет `/3` — номер нужной сессии в текущем проекте. Это можно сделать из списка `/sessions`, находясь в другой сессии, или даже при первом запуске бота. Из глобального режима `/all` пользователь переходит по команде вида `/2s9`, где `2` — номер проекта, а `9` — номер сессии внутри этого проекта


### Что происходит внутри

1. **Поиск сессии** — бот ищет в дневном реестре (списке сессий за сегодня), какая сессия соответствует номеру 3

2. **Привязка чата к сессии** — бот запоминает: «этот Telegram-чат теперь привязан к сессии #3». Теперь все сообщения пользователя идут в этот диалог

3. **Сообщения из других сессий текущего проекта продолжают приходить** — мониторинг остальных сессий активного проекта не прерывается. Если в другой сессии Claude ответит — пользователь увидит полное сообщение с номером сессии в виде кликабельной ссылки (кроме текущей сессии — для неё ссылка неактивна)

4. **Подтверждение** — пользователь видит, к какой сессии подключился, с превью первого сообщения и номером сессии

### Что видит пользователь

Подтверждение подключения с номером сессии и кратким описанием. Теперь все сообщения идут в этот диалог. Сообщения из других сессий текущего проекта приходят с полным текстом и кликабельными номерами для быстрого переключения.

### Что может пойти не так

- **Номер не найден** — если набрать `/99`, а такой сессии сегодня не было — бот сообщит об ошибке

## CJM-07: Мониторинг всех проектов (/all, /all_projects)

Пользователь хочет отключиться от конкретного проекта и диалога и видеть новые ответы агента из всех доступных проектов.

### Что делает пользователь

Отправляет команду `/all`, выбирает команду `/all_projects` в меню Telegram или выбирает строку `/all all` в списке `/projects`.

### Что происходит внутри

1. Бот отвязывает чат от текущей сессии — пользователь больше не привязан ни к конкретному проекту, ни к конкретному диалогу
2. Бот читает `recent_sessions` across projects and backends и собирает snapshot недавних пользовательских сессий для мониторинга. Snapshot — это зафиксированный список кандидатов на момент входа в `/all`, чтобы последующий poll не пересканировал историю. Codex-субагенты не попадают в этот snapshot по тем же правилам, что и в `/sessions`
3. Если индекс пуст, бот делает один ограниченный refresh по проектам. Если после этого кандидатов нет, режим `/all` не включается и пользователь получает сообщение, что пока нет проиндексированных недавних сессий
4. Запускается отдельный монитор `all_projects_monitor`, который каждые 2 секунды проверяет только известные файлы из snapshot по времени изменения и cursor-ам, а не пересобирает список всех сессий
5. Обычный `session_watcher` ставится на глобальную паузу, чтобы просмотр в `/all` не пометил сообщения прочитанными в исходном проекте
6. Бот отправляет подтверждение, что режим all включён и писать агенту из него нельзя

### Как работает мониторинг всех проектов

- При включении `/all` кандидаты берутся из `recent_sessions` с глобальным лимитом, а не из полного обхода истории CLI
- Каждые 2 секунды проверяются только session files, попавшие в snapshot при включении режима
- Для каждого кандидата сохраняется baseline cursor — позиция чтения на момент входа в `/all`, чтобы старые сообщения не приехали повторно как новые
- Когда в какой-то сессии появился новый ответ агента, пользователю приходит полное сообщение
- В начале сообщения стоит кликабельная команда вида `/3s12 bloger`: `3` — номер проекта, `12` — номер сессии, `bloger` — имя проекта
- Сообщение, показанное в `/all`, остаётся непрочитанным для исходного проекта. Когда пользователь входит в этот проект и сессию, оно доставляется повторно через обычную pending-доставку

### Что видит пользователь

Подтверждение: «Режим all включён: показываю сообщения из всех проектов. Писать агенту отсюда нельзя — сначала выберите проект и сессию.»

Новые сообщения приходят в формате:

```text
/3s12 bloger
текст ответа агента
```

### Важные особенности

- **Писать сообщения нельзя** — нет активного проекта и сессии, куда их отправлять. Текст, фото, документы и `/new` блокируются с подсказкой выбрать проект и сессию
- **Из этого режима можно подключиться к сессии** — нажав команду `/3s12` в сообщении all-mode или выбрав проект через `/projects`
- **Переход в уже активный проект тоже доставляет pending** — если all-mode был включён поверх текущего проекта, бот всё равно собирает непрочитанные сообщения для этого проекта

### Что может пойти не так

- **Задача «раздвоилась» на два номера** — пользователь создал сессию и видит подтверждение «Создана сессия #20». Но через пару секунд в ленте мониторинга промежуточные обновления по этой же задаче приходят под номером #21. Выглядит так, будто одна задача зарегистрировалась дважды. Причина: наблюдатель (watcher) обнаружил файл новой сессии на диске раньше, чем бот успел обновить привязку временного ID на настоящий UUID. В итоге watcher зарегистрировал сессию как новую и присвоил ей следующий дневной номер. Решение — мгновенное уведомление о смене session_id через callback в process_manager: бот узнаёт настоящий UUID сессии до того, как watcher начнёт сканирование, и «раздвоение» не происходит
- **Нет недавних кандидатов** — если `recent_sessions` пуст и bounded refresh не нашёл пользовательских сессий, пользователь увидит `Режим all недоступен: пока нет проиндексированных недавних сессий`. Если в проекте есть только субагентские Codex-файлы, они не считаются подходящими кандидатами для `/all`
- **Индекс недоступен** — если чтение индекса недавних сессий упало, бот не включает режим вслепую и отвечает `Режим all временно недоступен: не удалось прочитать индекс недавних сессий`
- **Переключение проекта из `/all` не удалось** — если пользователь нажал `/pN` или `/3s12`, но переключение проекта завершилось ошибкой, бот снова включает глобальный мониторинг и не возобновляет обычный watcher поверх него
- **Команда `/3s12` устарела** — если ссылка больше не находится во внутреннем реестре all-mode, бот пытается переключиться по номеру проекта и номеру сессии через текущий дневной реестр проекта. Если сессия не найдена, пользователь получает понятную ошибку

## CJM-08: Остановка Claude (/stop)

Пользователь хочет прервать работу Claude — например, если задача заняла слишком много времени или была отправлена по ошибке.

### Способ 1: команда /stop

**Что делает пользователь:** отправляет `/stop`

**Что происходит:**
- Если пользователь подключён к сессии — бот прерывает работу Claude в этой сессии
- Если пользователь находится в режиме `/all` (не подключён ни к какому проекту и сессии) — бот отвечает, что команда `/stop` работает только внутри сессии и сначала нужно выбрать проект и сессию
- Пользователь видит подтверждение «Claude остановлен»
- После остановки сессию можно продолжить — следующее сообщение в эту сессию снова запускает Claude. Сессия не «умирает» после `/stop`

### Способ 2: текстовая команда

**Что делает пользователь:** пишет «стоп», «отмена», «stop» или «cancel»

**Что происходит:**
- Сообщение передаётся в Claude как обычное текстовое сообщение
- Claude сам распознаёт команду остановки и прекращает текущую работу в рамках своего процесса
- Это мягкая остановка — Claude сам решает, как корректно завершить текущую задачу

### Важная разница между способами

- **/stop** — принудительная остановка, бот прерывает работу Claude немедленно
- **Текстовая команда** — мягкая остановка, сообщение просто передаётся Claude, и он сам решает что делать

## CJM-09: Защита от двойного запуска бота

Этот механизм не даёт запустить два экземпляра бота одновременно — иначе они будут мешать друг другу.

### Первый уровень: файл-замок

При запуске бот пытается «захватить» специальный файл `bot.pid`:

- **Успешно** → файл-замок принадлежит этому экземпляру, бот записывает свой номер процесса (PID) в файл и продолжает работу
- **Не удалось** → другой экземпляр уже держит замок → программа завершается

Почему именно файл-замок, а не просто проверка «работает ли процесс с таким PID»: файл-замок — это одна неделимая операция. А при проверке PID есть промежуток между «прочитать» и «записать», когда два экземпляра могут решить, что оба первые.

### Второй уровень: защита от Telegram

Даже если каким-то чудом запустились два бота — Telegram не будет с этим мириться. Когда два бота с одним токеном одновременно опрашивают Telegram, API возвращает ошибку Conflict. Бот ловит эту ошибку и завершается.

Если бот настроен на автоматический перезапуск через systemd user service — перезапустится только один экземпляр.

Штатный restart через `restart-claude-manager.sh` считается успешным только после того, как поднялся реальный Python-процесс бота. `systemctl --user is-active` сам по себе недостаточен: между «сервис активен» и «Python-бот принимает сообщения» может быть зазор в несколько секунд. Post-flight проверяет оба сигнала: `is-active` и `pgrep -f claude_manager`.

## CJM-10: Живой стриминг промежуточных обновлений через watcher

Этот сценарий проверяет критически важный механизм — наблюдатель за сессиями (session_watcher). Наблюдатель — это фоновый процесс внутри бота, который каждые 2 секунды опрашивает JSONL-файлы сессий Claude на диске и пересылает новые строки пользователю в Telegram. Без него пользователь не увидит промежуточные обновления (мысли Claude, вызовы инструментов) в реальном времени — они либо придут пачкой в конце, либо потеряются.

Сценарий покрывает класс багов «контракт с внешним CLI»: когда наш код делает предположения о том, как Claude CLI именует файлы и папки сессий на диске, а эти предположения расходятся с реальностью. Именно такой баг произошёл 10-04-2026 — функция `_encode_project_path` возвращала имя папки с подчёркиваниями, а Claude CLI создавал папку с дефисами вместо подчёркиваний. Наблюдатель искал несуществующую папку и ничего не находил, живой стриминг молча переставал работать.

### Что делает пользователь

1. Подключается к сессии (любой способ из CJM-04 или CJM-06)
2. Отправляет сообщение, которое требует от Claude заметной работы — например, «Прочитай файл `src/claude_manager/bot.py` и опиши, какие команды он обрабатывает». Важно выбрать задачу, где Claude точно будет использовать инструменты (Read, Bash, Grep) — это гарантирует, что появятся промежуточные события в JSONL-файле сессии
3. Ждёт ответ

### Что происходит внутри

**Шаг 1 — Старт Claude**

Бот запускает Claude CLI как подпроцесс с флагами `--resume <session-id>` и `--effort max` (максимальная глубина размышлений). Claude начинает работу и открывает для записи свой JSONL-файл в папке `~/.claude/projects/<закодированное-имя-проекта>/<session-id>.jsonl`.

**Шаг 2 — Claude пишет промежуточные события**

Пока Claude думает над задачей, он пишет в JSONL-файл события разных типов:
- `assistant` с `thinking` — внутренние рассуждения Claude
- `assistant` с `tool_use` — вызов инструмента (например, Read, Bash)
- `user` с `tool_result` — результат выполнения инструмента
- `assistant` с `text` — финальный текст ответа

Каждое событие — новая строка в файле. Файл растёт по мере работы Claude.

**Шаг 3 — Наблюдатель находит файл**

Параллельно session_watcher в фоне делает три вещи:
- Вычисляет путь к папке сессий: берёт путь проекта из `CLAUDE_WORKING_DIR`, прогоняет через функцию `_encode_project_path` (она должна точно воспроизводить алгоритм `sanitizePath` из Claude CLI), добавляет к `~/.claude/projects/`
- Находит в этой папке файл `<session-id>.jsonl` для текущей сессии
- Запоминает сколько строк уже прочитано

**Шаг 4 — Наблюдатель стримит новые строки**

Каждые 2 секунды наблюдатель открывает файл, читает все строки после последней прочитанной, парсит JSON, фильтрует события по типу и отправляет пользователю в Telegram промежуточные обновления. Ограничения:
- Не чаще раза в 30 секунд на одну сессию (см. раздел «Промежуточные обновления»)
- Формат: `#N ⏳` + текст курсивом
- Координация с обработчиком сообщений через pause/resume — чтобы ответ не задвоился

**Шаг 5 — Финальный ответ**

Когда Claude закончил работу, бот получает ответ через прямой stdout (а не через watcher). Этот финальный ответ уходит в Telegram с заголовком `#N ✅`.

### Что видит пользователь

1. Отправил сообщение
2. Через несколько секунд получил первое промежуточное обновление — мысль Claude или упоминание что он читает файл (формат `#3 ⏳ ...`)
3. Через 30+ секунд — возможно ещё одно промежуточное обновление
4. В конце — полный ответ Claude (формат `#3 ✅ ...`)

**Ключевой признак что всё работает:** промежуточные обновления приходят ДО финального ответа, а не одновременно с ним. Если приходит только финальный — значит наблюдатель молча сломался.

### Что должно быть в логах

В файле `~/.local/state/claude-manager/claude-manager.log` после этого сценария **не должно быть** следующих предупреждений:
- «Папка сессий не найдена» — значит `_encode_project_path` возвращает неправильное имя, наблюдатель ищет несуществующий путь
- «Файл сессии не найден» для файла текущей сессии — значит имя JSONL не совпадает с тем, что создаёт Claude CLI

Если эти предупреждения появились — это сигнал что контракт с Claude CLI сломан, нужен root-cause анализ.

### Что может пойти не так

- **Контракт имени папки сломан** — функция `_encode_project_path` не воспроизводит алгоритм `sanitizePath` из Claude CLI (например, забыли обработать точки или подчёркивания). Симптом: в логах warning «Папка сессий не найдена», живые обновления не приходят, но финальный ответ всё равно долетает через прямой stdout. Пользователь может не заметить — но для долгих задач это критично
- **Контракт имени файла сессии сломан** — Claude CLI поменял формат имени JSONL-файла. Симптом: папка найдена, но конкретный файл не виден
- **Формат stream-json изменился** — Claude CLI поменял структуру событий в JSONL. Симптом: наблюдатель падает при парсинге или не распознаёт события
- **Watcher не стартовал** — бот завёлся, но фоновый процесс наблюдателя не запустился. Симптом: промежуточные обновления вообще не приходят, только финальные ответы

### Как ловить регрессию автоматически

Юнит-тесты (`tests/test_session_reader.py`) проверяют что `_encode_project_path` возвращает дефисы для всех спецсимволов — это ловит проблему на уровне кода. Интеграционный тест (`tests/integration/test_claude_cli_contract.py`) запускает реальный Claude CLI и сверяет имя созданной папки с выводом нашей функции — это ловит расхождение контракта. E2E тест (этот CJM) ловит проблему на уровне пользовательского опыта — что промежуточные обновления реально долетают до Telegram.

## CJM-11: Переключение между проектами

Обычно у пользователя несколько проектов на компьютере — каждый в своей папке. Бот знает только один проект за раз, тот что указан в `CLAUDE_WORKING_DIR` в файле `.env`. Чтобы переключиться на другой проект, раньше нужно было закрыть бота, поправить `.env`, запустить снова. Этот сценарий решает проблему: пользователь может переключаться между проектами прямо из Telegram, без перезапуска.

Процессы Claude при переключении **не останавливаются** — они продолжают работать в фоне. Когда пользователь возвращается в проект, бот доставляет непрочитанные сообщения — ответы Claude, которые накопились за время отсутствия.

Бот ищет проекты в корневой папке, которая задаётся переменной окружения `PROJECTS_ROOT_DIR` (обязательная — без неё бот не запускается). Все подпапки этой папки — это проекты. Скрытые папки (начинающиеся с точки) и символические ссылки исключены — только обычные директории.

### Что делает пользователь

1. Открывает меню бота в Telegram через слэш или нажимает на команду `/projects`
2. Получает список всех доступных проектов в виде кликабельных команд `/p1`, `/p2`, `/p3`, ...; первой строкой доступен глобальный режим `/all all`
3. Видит маркер ● рядом с текущим активным проектом — сразу понятно, где сейчас работает бот
4. Кликает на нужный проект — команда `/pN` автоматически отправляется в чат
5. Получает подтверждение переключения

### Что происходит внутри

**Шаг 1 — Сканирование папки проектов**

Команда `/projects` вызывает функцию `scan_available_projects` в модуле `project_manager` (новый модуль для управления проектами). Она читает содержимое `PROJECTS_ROOT_DIR` и отфильтровывает всё, что не является проектом: файлы (например, `.zip` или `.DS_Store`), скрытые папки (`.git` и подобные), символические ссылки на внешние директории. Для каждой оставшейся папки определяется, совпадает ли она с текущим активным проектом (`CLAUDE_WORKING_DIR`) — совпадающая помечается флагом `is_current=True`.

**Шаг 2 — Формирование списка в Telegram**

Обработчик `handle_projects` собирает сообщение из строк вида `/all all` и `/pN <имя_папки>`. Текущий проект получает префикс ● (чёрный кружок — визуальный маркер). Если активен глобальный режим `/all`, маркер получает строка `/all all`. Сообщение отправляется как plain text (`parse_mode=None`) — это важно для того, чтобы команды оставались кликабельными в Telegram.

**Шаг 3 — Переключение проекта**

Когда пользователь кликает `/p3`, обработчик `handle_switch_project` извлекает номер (3) и находит проект в отсортированном списке. Затем вызывается `project_manager.switch_project(target_path)` — функция переключения с транзакционной семантикой:

- **Валидация пути** — проверяется, что папка существует, это действительно директория, лежит строго внутри `PROJECTS_ROOT_DIR` (защита от path traversal через `os.path.realpath` + `startswith`), и доступна на чтение
- **Проверка уже активного проекта** — если `target_path` указывает на тот же проект, что и `CLAUDE_WORKING_DIR`, возвращается результат с флагом `already_active=True`, никаких действий не выполняется
- **Сохранение снапшота watcher** — вызывается `unread_buffer.save_snapshot()`. Эта функция запоминает текущие счётчики наблюдателя (watcher) — сколько сообщений уже обработано в каждой сессии. Снапшот нужен для того, чтобы при возврате в проект бот мог определить, какие сообщения появились за время отсутствия
- **Глобальная пауза watcher** — вызывается `session_watcher.pause_all()` ДО смены `CLAUDE_WORKING_DIR`. Watcher полностью останавливается: пока пауза активна, `_poll_sessions()` (функция опроса файлов сессий) пропускает весь цикл. Это защита от гонки (race condition) — без паузы watcher мог проснуться между сменой `WORKING_DIR` и сбросом state-модулей и записать сессии из нового проекта в реестр старого
- **Процессы Claude продолжают работать** — в отличие от сброса состояния, процессы Claude не останавливаются. Если Claude выполняет задачу — он доделает её в фоне, а результат будет доставлен при возврате в проект
- **Обновление `CLAUDE_WORKING_DIR`** — меняется глобальная переменная, все последующие вызовы Claude CLI будут запускаться с новым CWD (current working directory)
- **Сброс состояния модулей** — `session_manager.reset_state` и `session_watcher.reset_state`. Эти функции очищают внутреннее состояние привязок чатов и счётчиков наблюдателя, а затем перезагружают данные для нового проекта. Дневной реестр (`daily_sessions.json`) не сбрасывается отдельным шагом: `session_watcher.reset_state` сам читает сегодняшние номера, когда строит baseline, поэтому отдельный reset только дублировал бы загрузку и очистку записей-сирот
- **Возобновление watcher** — вызывается `session_watcher.resume_all()` в finally-блоке. Watcher возобновляется даже при ошибке переключения, чтобы бот не потерял мониторинг навсегда
- **Сохранение выбора** — путь записывается в файл `~/.claude-manager-current-project` атомарно (через временный файл + rename). Это нужно для восстановления проекта при следующем запуске бота

**Шаг 4 — Подтверждение пользователю**

Пользователь видит одно из трёх сообщений:
- **Успех** — `Переключено на проект: <имя>` (если непрочитанных нет) или `Переключено на проект: <имя>\nНепрочитанных сообщений: <N>` (если за время отсутствия в проекте появились ответы Claude)
- **Уже активен** — `Уже работаю в проекте: <имя>`
- **Ошибка** — `Ошибка переключения: <причина>`

**Шаг 5 — Доставка непрочитанных сообщений**

Если пользователь возвращается в проект, где за время его отсутствия Claude успел ответить — бот автоматически доставляет эти ответы. Механизм работает так:

1. При возврате в проект бот сравнивает текущие JSONL-файлы сессий (файлы, куда Claude пишет свои ответы) с ранее сохранённым снапшотом счётчиков. Все сообщения, которые появились после снапшота, считаются «непрочитанными»
2. Сообщения старше 3 часов (TTL — time to live, время жизни) не доставляются. Это защита от лавины старых сообщений — если пользователь не заходил в проект полдня, ему не нужно получать 50 устаревших обновлений
3. Каждое непрочитанное сообщение доставляется отдельно, с обычным заголовком сессии — `#N ✅ текст ответа`. Выглядит так же, как если бы пользователь был в проекте и получил ответ в реальном времени
4. После доставки всех непрочитанных снапшот обнуляется — повторной доставки не будет

### Что видит пользователь

1. Отправил `/projects` (или кликнул на пункт меню)
2. Получил список всех проектов с кликабельными командами `/p1`, `/p2`, ... Маркер ● показывает текущий
3. Кликнул на нужный проект
4. Получил подтверждение с именем проекта. Если в проекте есть непрочитанные сообщения — видит их количество
5. Если непрочитанные были — получает каждое из них отдельным сообщением с привычным форматом `#N ✅`
6. Следующее сообщение в Telegram уйдёт в Claude Code, запущенный в новом проекте. Процессы Claude в предыдущем проекте продолжают работать в фоне — если вернуться, накопившиеся ответы будут доставлены

### Что должно быть в логах

После успешного переключения в `~/.local/state/claude-manager/claude-manager.log` должны появиться строки:
- `Мониторинг всех сессий приостановлен (глобальная пауза)` (info)
- `Сохранён снапшот watcher для проекта <старый_путь>` (info)
- `Переключение проекта выполнено: <старый_путь> -> <новый_путь>` (info)
- `Состояние session_manager сброшено и перезагружено` (info)
- `Состояние daily_session_registry сброшено и перезагружено` (info)
- `Удалено <N> записей-сирот без файлов на диске` (info) — если в реестре были «призрачные» записи
- `Состояние session_watcher сброшено для переключения проекта` (info)
- `Мониторинг всех сессий возобновлён (глобальная пауза снята)` (info)

При возврате в проект с непрочитанными:
- `Найдено <N> непрочитанных сообщений в проекте <путь>` (info)
- `Доставлено непрочитанное сообщение из сессии <session_id>` (info) — для каждого сообщения
- `Пропущено <N> сообщений старше TTL (3 часа)` (info) — если были просроченные

При ошибке валидации — warning с причиной: `Отклонено переключение на <путь>: <причина>`.

### Что может пойти не так

- **Папка проекта удалена между сканированием и кликом** — между вызовами `/projects` и `/p3` пользователь (или ОС) могут удалить папку. Тогда валидация упадёт, бот отправит `Ошибка переключения: Папка не существует: <путь>`. Состояние не меняется
- **Нет прав на чтение папки** — `os.access(target, R_OK)` возвращает False. Бот отправит `Ошибка переключения: Нет прав на чтение папки: <путь>`
- **Попытка переключиться на путь вне PROJECTS_ROOT_DIR** — через символическую ссылку или path traversal. Защита срабатывает, бот отправит `Ошибка переключения: Путь вне корневой папки проектов: <путь>`
- **Несуществующий номер `/p99`** — если пользователь ввёл номер, которого нет в списке. Бот отправит `Проект #99 не найден`. Переключение не происходит
- **Параллельные клики `/p1` и `/p2`** — если пользователь быстро кликнул два разных проекта, `asyncio.Lock` внутри `switch_project` гарантирует последовательное выполнение: сначала полное переключение на первый, потом на второй
- **Сбой во время переключения** — если `reset_state` одного из модулей упал, `switch_project` откатывает `CLAUDE_WORKING_DIR` обратно и пытается перезагрузить состояние старого проекта. Пользователь получает `Ошибка переключения: <причина>`, файл последнего проекта НЕ обновляется
- **Файл `~/.claude-manager-current-project` повреждён или содержит несуществующий путь** — при следующем запуске бота `load_last_selected_project` вернёт None, бот стартует с `CLAUDE_WORKING_DIR` из `.env`
- **Список проектов пуст** (в `PROJECTS_ROOT_DIR` нет ни одной подходящей подпапки) — бот отправит `Проекты не найдены в папке <root>`
- **`PROJECTS_ROOT_DIR` удалён после старта бота** — сканирование возвращает пустой список, сообщение как в предыдущем пункте
- **JSONL-файл сессии удалён или повреждён** — при попытке прочитать непрочитанные сообщения после возврата в проект бот может не найти файл сессии (например, пользователь вручную почистил папку). В этом случае сессия пропускается, остальные непрочитанные доставляются нормально
- **Все непрочитанные старше TTL (3 часа)** — если пользователь не заходил в проект больше 3 часов, все накопившиеся сообщения будут пропущены. Пользователь увидит `Непрочитанных сообщений: 0` и начнёт работу с чистого листа. Это намеренное поведение — старые сообщения чаще мешают, чем помогают

### Как ловить регрессию автоматически

- **Юнит-тесты `tests/test_project_manager.py`** — тесты на сканирование (фильтры, скрытые, символические ссылки, пустая папка, отметка текущего), переключение (happy path, уже активен, path traversal, несуществующий, не-папка, сохранение снапшота watcher, сброс state, сохранение в файл, откат при ошибке, сериализация параллельных вызовов), загрузка последнего проекта (нет файла, пустой, невалидный, валидный)
- **Юнит-тесты `tests/test_unread_buffer.py`** — тесты на модуль `unread_buffer` (буфер непрочитанных сообщений): сохранение и загрузка снапшота, поиск новых сообщений в JSONL-файлах после снапшота, TTL-фильтрация (сообщения старше 3 часов пропускаются), обработка повреждённых и отсутствующих файлов
- **Юнит-тесты `test_config.py::TestProjectsRootDir`** — 8 тестов на переменную окружения `PROJECTS_ROOT_DIR`
- **Юнит-тесты `TestResetState`** в каждом state-модуле (`test_session_manager`, `test_daily_session_registry`, `test_session_watcher`)
- **Юнит-тесты `test_bot.py::TestHandleProjects` и `test_bot.py::TestHandleSwitchProject`** — тесты на обработчики команд, включая доставку непрочитанных при переключении
- **Интеграционный тест `tests/integration/test_project_switching.py`** — сценарии полного цикла переключения на реальных временных директориях: проверка восстановления привязок при возврате в старый проект, no-op при переключении на текущий, фоновая работа процессов, доставка непрочитанных при возврате, корректный откат при ошибке, блокировка path traversal

## CJM-12: Доставка файлов из ответа агента

Когда Claude или Codex хочет показать пользователю файл (например, исходный код, конфигурацию, отчёт), он вставляет в текст ответа специальный маркер `[SEND_FILE:/absolute/path/to/file]` или `[SHOW_FILE:/absolute/path/to/file]`. Бот находит эти маркеры, определяет действие и отправляет файл пользователю в Telegram либо показывает содержимое текстового файла прямо в чате.

### Что делает пользователь

1. Пишет сообщение, которое подразумевает работу с файлами — например, «Покажи мне config.py» или «Сгенерируй отчёт и пришли»
2. Выбранный агент выполняет задачу и в финальном ответе вставляет маркеры `[SEND_FILE:...]` или `[SHOW_FILE:...]` для каждого файла

### Что происходит внутри

**Шаг 1 — Проверка: финальный ли ответ**

Маркеры обрабатываются только в финальных ответах (`is_final=True`). Промежуточные обновления (thinking от агента, которые показывают ход мыслей) маркеры не трогают — это защита от преждевременной отправки файлов, пока агент ещё думает.

**Шаг 2 — Парсинг маркеров**

Функции из модуля `file_sender` (утилита без состояния для работы с файлами) находят все вхождения `[SEND_FILE:путь]` и `[SHOW_FILE:путь]` в тексте с помощью регулярных выражений. Извлекаются только абсолютные пути.

**Шаг 3 — Очистка текста**

Функция `strip_file_markers()` вырезает все маркеры из текста ответа, чтобы пользователь не видел служебные конструкции. Лишние пустые строки после вырезки схлопываются.

**Шаг 4 — Отправка каждого файла**

Для каждого извлечённого пути бот определяет тип файла:

- **Текстовый файл** (`.md`, `.py`, `.json`, `.sh` и ещё ~30 расширений, а также файлы вроде `Makefile`, `.gitignore`) — бот читает содержимое, рендерит через `telegramify-markdown` (библиотека для преобразования Markdown в формат Telegram) и отправляет как сообщение с нативным форматированием (entities API). Перед содержимым идёт заголовок с иконкой скрепки и именем файла. Если файл длиннее 4096 символов — разбивается на несколько сообщений
- **Бинарный файл** (`.png`, `.pdf`, `.zip` и любые другие) — отправляется через `send_document` как вложение в чат

Ограничения по размеру: текстовые файлы до 1 МБ, бинарные до 50 МБ (лимит Telegram Bot API).

**Шаг 5 — Обычная обработка текста**

После отправки файлов очищенный текст (без маркеров) проходит обычный путь — конвертация Markdown в HTML через `message_splitter` (модуль разбивки сообщений), разбивка на части, отправка в Telegram.

### Что видит пользователь

1. Отправил сообщение в Claude
2. Видит «печатает...»
3. Получает файл (или файлы) — текстовые с форматированием прямо в чате, бинарные как вложения
4. Получает текстовый ответ агента (если он есть помимо файлов) — с обычным заголовком `#N ✅`

### Что может пойти не так

- **Файл не найден** — агент вставил маркер с путём к файлу, который не существует (удалён или переименован между генерацией ответа и отправкой). Пользователь получает сообщение `Файл не найден: /path/to/file`
- **Нет прав на чтение** — файл существует, но бот не может его прочитать. Пользователь получает `Нет доступа к файлу: /path/to/file`
- **Файл слишком большой** — текстовый больше 1 МБ или бинарный больше 50 МБ. Пользователь получает соответствующее сообщение
- **Файл не является текстовым** — расширение текстовое, но содержимое бинарное (повреждённый файл). Пользователь получает `Файл не является текстовым: /path/to/file`
- **Маркер в промежуточном обновлении** — маркер не обрабатывается, отображается как обычный текст. Это by design (намеренное поведение)

### Как ловить регрессию автоматически

- **Юнит-тесты `tests/test_file_sender.py`** — покрывают все публичные функции: парсинг маркеров, определение типа файла, чтение, рендеринг, конвертацию entities, проверку бинарных файлов
- **Юнит-тесты `tests/test_bot.py`** — тесты на интеграцию file_sender в `send_response` и `send_watcher_message`: маркеры обрабатываются при `is_final=True`, игнорируются при `is_final=False`, ошибки чтения не крашат бота
- **Юнит-тесты `tests/test_codex_backend_bot_mode_prompt.py`** — проверяют, что Codex получает Telegram-инструкцию про `[SEND_FILE:...]` и `[SHOW_FILE:...]` и сохраняет пользовательский текст в prompt
- **Интеграционный тест `tests/integration/test_message_path.py`** — полный путь сообщения с маркером от Claude до Telegram

## CJM-16: Переключение CLI-бэкенда (/agent)

Пользователь хочет выбрать, какая CLI-программа будет создавать новые сессии: Claude Code CLI или Codex CLI. CLI-программа — это локальная командная утилита, через которую бот запускает кодового агента на компьютере пользователя.

### Что делает пользователь

1. Отправляет команду `/agent`
2. Видит текущий выбранный агент и кнопки доступных вариантов
3. Нажимает кнопку нужного агента: `Claude` или `Codex`
4. Получает подтверждение переключения
5. Если хочет начать работу уже через новый агент — отправляет `/new`

### Что происходит внутри

**Шаг 1 — Открытие выбора агента**

Бот проверяет доступ пользователя, читает текущий выбранный CLI-бэкенд из `current_backend_registry` и получает список доступных бэкендов через общий backend-интерфейс. Backend-интерфейс — это общий контракт для Claude Code CLI и Codex CLI, чтобы верхний слой бота мог работать с ними одинаково.

**Шаг 2 — Показ кнопок**

Бот отправляет inline-клавиатуру с вариантами. Текущий выбранный агент отмечается галочкой, чтобы пользователь видел активный выбор.

**Шаг 3 — Сохранение выбора**

Когда пользователь нажимает кнопку, бот сохраняет выбранный backend в файл `~/.claude-manager-current-backend`. Этот выбор глобальный для всего процесса бота и применяется только к будущим новым сессиям.

**Шаг 4 — Защита существующих сессий**

Текущая и старые сессии не переносятся между Claude и Codex. Каждая сессия продолжает открываться тем CLI-бэкендом, который создал её изначально. Поэтому после переключения `/agent`:

- `/new` создаёт новую сессию через выбранный backend
- `/1`, `/2`, `/3` открывают существующую сессию через её собственный backend
- `/stop` останавливает активную сессию через её собственную стратегию остановки
- watcher продолжает следить за сессиями всех backend-ов

### Что видит пользователь

При открытии команды:

```text
Текущий агент: 🤖 Claude
```

После выбора Codex, если активная сессия есть:

```text
Теперь новые сессии будут создаваться через ⚡ Codex.
Текущая сессия #1 остаётся на 🤖 Claude.
Чтобы начать новую Codex-сессию, отправьте /new.
```

Если активной сессии нет:

```text
Теперь новые сессии будут создаваться через ⚡ Codex.
Чтобы начать новую сессию, отправьте /new.
```

Если пользователь нажал уже выбранный вариант:

```text
Уже выбран: 🤖 Claude.
```

### Что может пойти не так

- **Пользователь не в белом списке** — команда игнорируется, как остальные команды бота. В лог пишется предупреждение
- **Кнопка устарела или повреждена** — бот отвечает `Неизвестный агент`, выбор не меняется
- **Не удалось записать выбор на диск** — бот сообщает `Не удалось переключить агента: <причина>`. Старый выбор остаётся активным
- **Выбранный CLI не установлен** — `/agent` всё равно может сохранить выбор. Ошибка появится позже, когда пользователь создаст новую сессию и бот попробует запустить выбранный CLI
- **Пользователь ожидает перенос текущей сессии** — текущая сессия не меняется. Чтобы начать работу через новый backend, нужно создать новую сессию через `/new`

### Как ловить регрессию автоматически

- **Юнит-тесты `tests/test_bot.py`** — проверяют показ текущего backend-а, переключение кнопкой, обработку неизвестной callback data и то, что повторный выбор не пишет файл заново
- **Юнит-тесты `tests/test_current_backend_registry.py`** — проверяют загрузку, сохранение, откат при ошибке записи и legacy-значения
- **Интеграционные тесты backend-aware цепочки** — проверяют, что `/agent` влияет только на новые сессии, а существующие сессии продолжают работать через свой backend

## CJM-17: Ответ на старое сообщение бота в Telegram

Пользователь хочет ответить прямо на сообщение, которое бот уже прислал из конкретной сессии, и не хочет вручную переключать текущий проект или текущую сессию.

### Что делает пользователь

1. Находит в Telegram старое сообщение бота с ответом агента
2. Нажимает «Ответить» в Telegram
3. Пишет обычный текст или записывает voice/audio
4. Отправляет сообщение

### Что происходит внутри

1. Бот смотрит, является ли входящее сообщение reply на сообщение бота
2. По `message_id` сообщения бота ищет маршрут ответа — запись, восстановленную с диска в память процесса, где сохранены проект, session_id, backend и номера для ссылки; бот хранит последние 200 таких маршрутов
3. Если маршрут найден, бот помечает целевую сессию как занятую для address-reply и сразу отправляет пользователю подтверждение
4. Сам текст уходит в исходную сессию фоновой задачей через сохранённые `project_path`, `session_id` и `backend`
5. Бот не переключает текущий проект и не меняет активную сессию пользователя
6. Если пользователь находится в `/all`, обычный запрет на текстовые сообщения не применяется к такому reply: адрес уже известен из маршрута
7. Ответ агента потом приходит обычным механизмом: через `/all`-монитор, watcher текущего проекта или pending-доставку при возврате в проект

Если reply отправлен голосом или аудиофайлом, бот сначала скачивает аудио и распознаёт его в текст через OpenAI Speech to text. После этого распознанный текст проходит тот же адресный маршрут, что и обычный текстовый reply. Это работает и в `/all`: пользователь остаётся в режиме мониторинга, но голосовой ответ уходит в сессию-источник сообщения.

### Что видит пользователь

После успешных предварительных проверок бот сразу отправляет короткое подтверждение, не дожидаясь завершения агента:

```text
Передал в /3s12
```

Если пользователь уже находится в том же проекте, где лежит исходная сессия, ссылка короче:

```text
Передал в /12
```

### Что может пойти не так

- **Маршрут не найден** — сообщение было отправлено до появления персистентного reply-routing, файл маршрутов недоступен/повреждён или это было служебное сообщение. Пользователь видит: `Не понял, куда передать ответ. Нажми ссылку на нужную сессию и отправь сообщение там`
- **Целевая сессия занята** — текст не отправляется в очередь. Пользователь видит: `Не передал в /3s12: сессия занята. Подождите или /stop`
- **Второй быстрый reply в ту же цель** — пока первая фоновая отправка ещё не дошла до `process_manager`, бот всё равно считает цель занятой и отвечает busy-сообщением, чтобы не создать очередь
- **Целевой проект недоступен** — проект удалён, переименован или больше не виден в списке проектов. Пользователь видит короткую ошибку со ссылкой на цель
- **Целевая сессия недоступна** — backend больше не видит файл сессии. Пользователь видит короткую ошибку со ссылкой на цель
- **Ответ с вложением** — поддержаны текст и voice/audio после распознавания. Фото, документы, файлы и альбомы не скачиваются и не отправляются агенту; пользователь видит, что ответы с вложениями пока не работают
- **Не удалось распознать voice/audio reply** — текст не отправляется в сессию. Пользователь видит такое же сообщение об ошибке распознавания, как при обычном голосовом вводе
- **Ошибка backend после подтверждения** — если агент сообщает переполнение контекста, бот отдельным сообщением предлагает начать новую сессию через `/new`; если произошла обычная ошибка, бот показывает понятную причину вместо общего сообщения «сессия недоступна»

### Как ловить регрессию автоматически

- **Юнит-тесты `tests/test_reply_route_registry.py`** — проверяют маршруты по `chat_id + bot_message_id` и восстановление после перезагрузки реестра
- **Юнит-тесты `tests/test_reply_route_handler.py` и `tests/test_reply_route_handler_error_results.py`** — проверяют успешные reply, busy-сценарии, недоступные цели и ошибки backend
- **Юнит-тесты `tests/test_reply_route_handler_background.py`** — проверяют, что подтверждение приходит до завершения агента, а быстрый второй reply блокируется in-flight guard-ом
- **Юнит-тесты `tests/test_telegram_response_delivery_reply_routes.py`** — проверяют регистрацию маршрутов только после успешной отправки сообщения в Telegram
- **Юнит-тесты `tests/test_telegram_input_handlers.py`** — проверяют, что routed reply обрабатывается до `/all`-ограничений и не скачивает unsupported-вложения
- **Юнит-тесты `tests/test_telegram_audio_input_handlers.py`** — проверяют, что voice/audio reply распознаётся и уходит в сохранённую сессию, даже если пользователь находится в `/all`

## CJM-18: Отправка голосового или аудиосообщения

Пользователь может надиктовать сообщение голосом в Telegram. Бот распознаёт речь в текст и передаёт этот текст агенту так же, как если бы пользователь написал его руками.

### Что нужно заранее

- **Активная сессия или адресный reply** — пользователь должен быть подключён к сессии, как в CJM-02, либо отправлять voice/audio как Telegram reply на сообщение бота с сохранённым маршрутом
- **OPENAI_API_KEY** — ключ OpenAI API в `.env`, потому что распознавание делает внешний Speech to text API

### Что делает пользователь

1. Открывает чат с ботом в Telegram
2. Отправляет voice-сообщение или аудиофайл. Также может нажать «Ответить» на старое сообщение бота и записать voice/audio как адресный reply
3. Ждёт ответ агента

### Что происходит внутри

1. Бот проверяет доступ пользователя по Telegram-ID
2. Если это обычное voice/audio без reply и пользователь находится в режиме `/all` или без активной сессии, бот не скачивает аудио и просит сначала выбрать проект и сессию
3. Если это обычное voice/audio без reply и текущая сессия уже занята, бот сразу сообщает busy-состояние и не тратит сетевые запросы на скачивание
4. Бот скачивает аудиофайл из Telegram в `received_files/`
5. Бот отправляет файл в OpenAI Speech to text через endpoint `/v1/audio/transcriptions`
6. Полученный текст передаётся в активную CLI-сессию как обычное пользовательское сообщение. Если voice/audio был Telegram reply на сообщение бота, полученный текст передаётся в сессию-источник этого сообщения через address-reply route
7. Дальше работает стандартный путь ответа агента: прогресс, финальный ответ, форматирование и reply-anchor

### Что видит пользователь

1. Отправляет voice/audio
2. Видит «печатает...», пока бот распознаёт аудио и агент обрабатывает текст
3. Получает ответ агента в текущей сессии

### Что может пойти не так

- **Не настроен OpenAI API key** — пользователь видит сообщение, что нужно добавить `OPENAI_API_KEY` в `.env`
- **Не удалось скачать аудио** — Telegram не отдал файл или сеть оборвалась. Пользователь видит просьбу отправить аудиофайл ещё раз
- **OpenAI не распознал файл** — пользователь видит сообщение, что аудио не удалось распознать, и может отправить текстом
- **Формат Telegram Ogg/Opus** — реальный тест показал, что `whisper-1` принял исходный Telegram `.ogg`, но официальная документация OpenAI не заявляет `ogg` как поддержанный формат. Если API изменит поведение, может понадобиться конвертация в `webm`, `m4a` или `wav`
- **Voice/audio reply без маршрута** — если сообщение бота старое, служебное или маршрут не найден, бот не угадывает сессию и просит перейти по ссылке на нужную сессию

### Как ловить регрессию автоматически

- **Юнит-тесты `tests/test_openai_transcription.py`** — проверяют multipart-запрос к OpenAI, отсутствие ключа, некорректный ответ и HTTP-ошибки
- **Юнит-тесты `tests/test_telegram_audio_input_handlers.py`** — проверяют успешный путь voice/audio, ошибку скачивания, ошибку OpenAI и отсутствие ключа
- **Юнит-тесты `tests/test_telegram_audio_file_downloader.py`** — проверяют, что Telegram voice/audio сохраняются с правильным расширением
- **Юнит-тесты `tests/test_telegram_audio_handler_registration.py`** — проверяют регистрацию Telegram handler-а для voice/audio

## Карта состояний пользователя

Пользователь всегда находится в одном из трёх состояний:

### Состояние 1: «Чистый лист» (нет сессии)

- **Как сюда попасть:** первый запуск бота
- **Что можно делать:** нельзя писать сообщения в Claude — бот попросит сначала создать или выбрать сессию
- **Мониторинг:** нет
- **Куда можно перейти:**
  - `/new` → состояние 2
  - `/all` или `/all_projects` → состояние 3
  - `/номер сессии` → состояние 2
  - `/agent` → остаётся в состоянии 1, меняет backend для будущей `/new`
  - Написать сообщение → бот отвечает подсказкой (остаётся в состоянии 1)

### Состояние 2: «Подключён к сессии» (основное рабочее состояние)

- **Как сюда попасть:** `/номер сессии`, `/new`
- **Что можно делать:** писать сообщения, отправлять фото, документы, voice/audio, получать ответы Claude, а также отвечать текстом или voice/audio reply на старые сообщения бота из любой маршрутизируемой сессии
- **Мониторинг:** сообщения из всех сессий текущего проекта приходят полностью. Для текущей сессии — номер без ссылки. Для остальных сессий — номер в виде кликабельной ссылки для быстрого переключения
- **Куда можно перейти:**
  - `/new` → создаёт новую сессию, остаётся в состоянии 2
  - `/all` или `/all_projects` → состояние 3
  - `/номер другой сессии` → состояние 2 (переключение)
  - `/agent` → остаётся в состоянии 2, меняет backend только для будущих новых сессий
  - Текстовый или voice/audio reply на сообщение бота → остаётся в состоянии 2, но текст или распознанный текст уходит в сессию-источник сообщения

### Состояние 3: «Радар» (мониторинг всех проектов)

- **Как сюда попасть:** команда `/all` или `/all_projects`
- **Что можно делать:** смотреть сообщения, переключаться по ссылкам и отвечать текстом или voice/audio reply на маршрутизируемые сообщения бота. Обычные новые сообщения, фото, документы, voice/audio и `/new` всё ещё блокируются — нет активного проекта и сессии
- **Мониторинг:** полные сообщения из всех проектов с кликабельными командами вида `/<project_number>s<session_number>`
- **Куда можно перейти:**
  - `/pN` → состояние 2 в выбранном проекте
  - `/<project_number>s<session_number>` → состояние 2 в выбранном проекте и сессии
  - `/agent` → остаётся в состоянии 3, меняет backend только для будущих новых сессий
  - Текстовый или voice/audio reply на сообщение бота → остаётся в состоянии 3, но текст или распознанный текст уходит в сессию-источник сообщения

## Механизмы, которые работают во всех сценариях

### Проверка доступа

Перед каждым действием (команда, сообщение, фото) бот проверяет: есть ли Telegram-ID отправителя в белом списке разрешённых пользователей. Если нет — запрос молча игнорируется. В лог записывается предупреждение.

### Промежуточные обновления (прогресс)

Пока Claude думает, бот присылает промежуточные обновления — рассуждения Claude (его мысли о текущей задаче). Ограничения:
- **Не чаще раза в 30 секунд** — чтобы не заспамить пользователя
- **Показываются полностью** — без обрезки
- **Формат:** `#3 ⏳ текст обновления` (номер сессии + песочные часы + текст курсивом)

### Одновременная работа из Telegram и терминала

Пользователь может работать с одной и той же сессией одновременно из Telegram (через бота) и из терминала на компьютере. Сообщения появляются и там и там — никакой блокировки нет. Это позволяет:
- Начать задачу в терминале, а следить за прогрессом через телефон
- Писать Claude и из Telegram, и из терминала — в одну и ту же сессию
- Видеть ответы Claude в обоих местах

### Сообщения из других сессий

Когда пользователь подключён к конкретной сессии (состояние 2), он продолжает получать полные сообщения из всех остальных сессий. Номера сессий в таких сообщениях оформлены как кликабельные ссылки — нажав, можно быстро переключиться. Для текущей сессии номер показывается без ссылки.

### Адресные ответы на сообщения бота

Когда бот отправляет ответ агента в Telegram, он запоминает маршрут: какое сообщение бота относится к какому проекту, сессии и backend. Если пользователь потом отвечает текстом на это сообщение через Telegram reply, бот отправляет текст именно в сессию-источник и не меняет текущий проект пользователя. Если пользователь отвечает voice/audio, бот сначала распознаёт аудио и отправляет в сессию-источник уже распознанный текст.

Маршруты живут только в памяти процесса. После перезапуска старые сообщения Telegram не маршрутизируются автоматически: пользователь должен перейти по ссылке на нужную сессию и написать сообщение там.

Адресные reply поддерживают текст и voice/audio после распознавания. Фото, документы и альбомы не скачиваются и не отправляются агенту, чтобы бот случайно не передал файл не в ту сессию.

### Владение сессии CLI-бэкендом

Каждая сессия принадлежит тому CLI-бэкенду, который её создал. Это нужно, чтобы после переключения `/agent` бот не пытался открыть старую Claude-сессию через Codex или старую Codex-сессию через Claude.

Глобальный выбор из `/agent` используется только для новых сессий. Все действия с существующей сессией — отправка сообщения, `/номер сессии`, `/stop`, watcher и доставка непрочитанных — используют backend, сохранённый у самой сессии.

### Координация мониторинга и ответов

Когда бот сам отправляет сообщение в Claude, возникает проблема: и наблюдатель (watcher), и обработчик сообщений получат ответ — пользователь увидит его дважды.

Решение — механизм координации:
1. Перед отправкой: наблюдатель за текущей сессией координируется с обработчиком, чтобы не дублировать ответ
2. При этом наблюдатели за другими сессиями продолжают работать без паузы — сообщения из остальных сессий приходят всегда

### Защита от одновременной записи

Бот может обрабатывать несколько команд одновременно. Если два действия одновременно читают и пишут в один файл (sessions.json или daily_sessions.json) — второе может затереть изменения первого. Для предотвращения этого используется блокировка на операции с файлами — только одно действие может писать в файл в один момент.

Дополнительно, запись в файлы выполняется атомарно: сначала данные записываются во временный файл (.tmp), затем он переименовывается в основной. На Linux переименование внутри одной файловой системы — мгновенная атомарная операция (`rename(2)`). Если бот упадёт в момент записи — основной файл останется целым.

### Обновление идентификатора сессии

Когда создаётся новая сессия, бот сначала не знает её настоящий идентификатор — использует временный (вида `_new_XXXX`). Настоящий идентификатор приходит от Claude в первом же ответе. Бот обновляет его во всех внутренних словарях.

### Отправка сообщений в Telegram

- Основной формат — HTML (жирный текст, код, блоки кода)
- Если Telegram не смог обработать HTML — бот убирает все теги и отправляет обычный текст
- Длинные ответы (> 4096 символов) разбиваются на несколько сообщений
- Длина сообщения при разбивке считается после HTML-экранирования — потому что экранирование увеличивает длину (например, символ `&` превращается в `&amp;`, из 1 символа в 5)
- При разбивке чинятся HTML-теги: если `<b>` открылся в одном сообщении, но текст продолжается в следующем — тег закрывается `</b>` в конце первого и открывается `<b>` в начале второго
- При конвертации Markdown в HTML учитываются вложенные блоки кода — когда Claude объясняет как писать Markdown, в его ответе тройные обратные кавычки могут встретиться внутри блока кода
- Кнопки (если есть) прикрепляются только к последнему сообщению
- При ошибке отправки сообщения (обрыв сети, таймаут) — бот повторяет отправку 2-3 раза с паузой между попытками
- При получении ошибки RetryAfter от Telegram (превышен лимит ~20 сообщений в минуту в один чат) — бот ждёт указанное Telegram количество секунд и повторяет отправку

### Обработка ошибок Claude

- При любой ошибке от Claude (сбой API, обрыв связи, неожиданный ответ) — бот сообщает пользователю о проблеме
- Автоматически повторяет попытку до 10 раз с интервалом в 1 минуту
- О каждой попытке повтора пользователь получает уведомление
- Ошибки записываются в лог-файл — для анализа проблем, если что-то пойдёт не так

### Восстановление после перезапуска

- При запуске бот читает сохранённые данные (sessions.json) и восстанавливает привязки: какой чат был подключён к какой сессии
- Сканирует сессии Claude на диске и присваивает дневные номера новым сессиям
- Пользователь оказывается в том же состоянии, в котором был до перезапуска
- Сессии Claude сохраняются на диске и доступны после любого перезапуска бота
