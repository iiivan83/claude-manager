# ADR: запрет инструмента AskUserQuestion при запуске Claude CLI из Telegram-бота

**Дата:** 28-05-2026
**Статус:** принято
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter
**Сущность:** claude_code_backend.ClaudeCodeBackend.compose_subprocess_command_args

## Контекст

Пользователь жаловался, что Claude через бот в Telegram «откуда-то берёт ответ и идёт дальше», хотя пользователь не успел ответить. В чате оставалась только короткая фраза «Чтобы не угадывать за тебя, спрошу.», а следующим сообщением Claude писал «Похоже, ты не ответил на вопрос — иду разумным дефолтом» и продолжал работу с придуманным за пользователя выбором.

Корень — встроенный SDK-инструмент Claude Code под именем `AskUserQuestion`. Это интерактивный инструмент: в TUI Claude Code он рисует «плашку» с заранее подготовленными вариантами ответа, пользователь выбирает кнопкой. Имя инструмента подтверждено по официальной декларации SDK: `/home/ivan/.npm-global/lib/node_modules/@anthropic-ai/claude-code/sdk-tools.d.ts:588`, интерфейс `AskUserQuestionInput`.

В bot-режиме Claude CLI запускается с флагами `-p --input-format stream-json --output-format stream-json --dangerously-skip-permissions --effort max` через `claude_code_backend.compose_subprocess_command_args`. При вызове `AskUserQuestion`:

- Claude шлёт `assistant`-событие с двумя content-блоками: `text` (вступительная фраза) и `tool_use` с input `{questions: [{question, header, options[]}]}`.
- Бот читает только text-блок через `claude_code_backend.read_progress_text_from_event` — tool_use-блоки игнорируются, в Telegram не отправляются. Пользователь видит только вступительную фразу, без вопроса и вариантов.
- Claude получает от инструмента какой-то «пустой» ответ (плашку показать некому, stdout не TTY), интерпретирует это как «пользователь не ответил» и идёт «разумным дефолтом» — сам выбирает за пользователя.

Дополнительная сложность: пользователь хочет сохранить поведение `AskUserQuestion` для обычного Claude Code в терминале (где плашка работает корректно). Отключать инструмент глобально нельзя — только в bot-режиме.

## Решение

Передавать в Claude CLI при запуске из bot-режима два флага, которые отсутствовали:

- `--disallowedTools AskUserQuestion` — флаг Claude CLI, который добавляет инструмент в список запрещённых. Claude видит это в `system/init`-событии и не вызывает инструмент в течение всего turn-а. Если Claude хотел бы задать вопрос, он формулирует его обычным текстом в ответе.
- `--append-system-prompt <инструкция>` — флаг Claude CLI, который дописывает к системному промпту дополнительный текст. Используется как страховка: даже если Claude забудет про disallow-листинг (например, из-за prompt injection из внешних источников), у него явно сказано, в каком контексте он работает. Текст инструкции на английском: `You are running inside a Telegram bot, not the interactive Claude Code TUI. The user cannot see interactive pickers or option menus. If you need to ask the user a question, write the question as plain text in your response (with numbered options when appropriate) and wait for the user's next text message. Never rely on UI-based question tools.`

Оба флага добавлены только в `claude_code_backend.compose_subprocess_command_args` — функция формирует argv для подпроцесса Claude CLI, который запускает бот. Обычный Claude Code, который пользователь запускает в терминале, не проходит через этот код и продолжает работать с `AskUserQuestion` как раньше.

Константы вынесены отдельно для документирования смысла и для тестируемости:

- `DISALLOWED_TOOLS_IN_BOT_MODE = "AskUserQuestion"` — пока один инструмент в списке. Если в будущем понадобится отключить ещё какие-то интерактивные инструменты (например, `EnterPlanMode`, `ExitPlanMode`), они добавятся через пробел в эту же строку.
- `BOT_MODE_SYSTEM_PROMPT_APPENDIX` — текст инструкции.

Альтернатива, рассмотренная и отвергнутая:

- **Перехват tool_use AskUserQuestion на стороне бота через control protocol.** Backend парсит блок с `name=AskUserQuestion`, форматирует questions+options как обычный текст в Telegram («Вопрос: …. Варианты: 1. …»), ждёт ответа пользователя в чате и упаковывает ответ обратно как `tool_result` через `control_response` в stdin Claude CLI. Это значительно сложнее: требует state-машины ожидания ответа, синхронизации с уже работающими механизмами `/stop` и `handle_message`, обработки таймаутов и переключения проектов. Это полноценная фича на несколько дней работы, а пользовательская боль решается двумя CLI-флагами за часы.

## Связанные файлы

- **`src/claude_manager/claude_code_backend.py`** — добавлены две константы `DISALLOWED_TOOLS_IN_BOT_MODE` и `BOT_MODE_SYSTEM_PROMPT_APPENDIX` с комментарием, объясняющим причину их существования (плашка не работает в Telegram, Claude уходит в дефолт). В метод `compose_subprocess_command_args` добавлены четыре элемента argv: `--disallowedTools`, значение константы, `--append-system-prompt`, значение константы. Порядок размещён после `--effort max` и до `--resume`-блока — флаги передаются на каждый запуск, независимо от того, новая это сессия (`_new_*`) или восстановление существующей.
- **`tests/test_claude_code_backend.py`** — добавлены два регрессионных теста:
  - `test_compose_args_disallow_ask_user_question_tool` — проверяет что в argv присутствует `--disallowedTools`, и что значение содержит токен `AskUserQuestion` (с поддержкой будущего расширения через запятую или пробел).
  - `test_compose_args_append_system_prompt_explains_text_only_questions` — проверяет что в argv присутствует `--append-system-prompt`, и что значение упоминает слово «telegram» (без регистра). Это закрепляет контракт: системный промпт должен называть платформу, а не быть generic-инструкцией.

## Примечания

Источник имени инструмента — официальная TypeScript-декларация Claude Code SDK на текущей машине: `/home/ivan/.npm-global/lib/node_modules/@anthropic-ai/claude-code/sdk-tools.d.ts`. Поиск по проекту показал, что слово `AskUserQuestion` встречалось только в документации скиллов (`dev/docs/specs/pipeline-spec.md`, в session reports), но не в коде самого бота — обработка инструмента ранее не была предусмотрена.

Codex backend (`codex_backend.py`) не затронут: у Codex своя система инструментов, `AskUserQuestion` там отсутствует, поведение `reasoning`-блоков и `agent_message` в Codex не пересекается так, чтобы вызывать аналогичную проблему.

Проверка работоспособности через TDD цикл RED→GREEN→Revert→Restore: после отката изменений в `claude_code_backend.py` оба новых теста падают с сообщениями «Bot must pass --disallowedTools to disable interactive UI tools» и «Bot must pass --append-system-prompt to instruct Claude on text-only questions». После восстановления — оба зелёные, все 28 тестов в `test_claude_code_backend.py` проходят.
