# ADR: Operational lookback window для оперативного листинга сессий при переключении проекта

- **Дата:** 28-05-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** контракт `CodingAgentBackend.list_all_session_files_for_project`, константа `config.OPERATIONAL_SESSION_LOOKBACK_DAYS`, реализации Claude и Codex backend, вызовы из `session_watcher.SessionWatcher.reset_state` и `project_manager._collect_pending_for_backend`

## Контекст

Переключение проекта в Telegram-боте (команды `/projects` и `/pN`) занимало 9-10 секунд даже при пустом списке непрочитанных сообщений. Утренний коммит `60e710a perf(project-switch): параллельная инициализация при смене проекта` параллелил чтение JSONL-файлов через семафор N=16, но не помог — оптимизация улучшила обработку **найденных** файлов, тогда как реальное узкое место было в самом **поиске**.

Codex CLI хранит все rollout-файлы в общей папке `~/.codex/sessions/YYYY/MM/DD/` — одна папка для всех проектов пользователя. У текущего пользователя там лежит 10 157 файлов. При каждом переключении бот:

- Делал `os.walk` по всей папке (10 157 файлов)
- Для каждого открывал JSONL, читал первую `session_meta`-запись, проверял поле `payload.cwd`, чтобы понять «к какому проекту этот файл относится»
- Возвращал только подходящие — все остальные открывались зря

Эта работа повторялась **дважды** за одно переключение:

- В `session_watcher.SessionWatcher.reset_state` (пересборка курсоров наблюдателя)
- В `project_manager._collect_pending_for_backend` (сбор непрочитанных для возврата в проект)

Прямое доказательство в логах бота: между записями `Состояние session_watcher сброшено для переключения проекта` (21:44:06) и `Переключение проекта выполнено` (21:44:15) проходило 9 секунд при `непрочитанных=0`.

Альтернативные подходы, которые рассматривались и отвергнуты:

- **Глобальный кэш индекса файлов** — отдельная сущность с инвалидацией по mtime или filesystem-events. Дорого по сложности, легко рассинхронизировать, требует отдельного жизненного цикла.
- **Изменение Codex CLI**, чтобы он хранил сессии по проектам. Невозможно — мы не контролируем поведение внешнего CLI.
- **Отдельный метод** `list_recent_operational_session_files_for_project(lookback_days)` рядом с существующим. Раздувает контракт и требует isinstance-проверок в местах вызова.
- **Глобальная переменная** «текущий lookback», читаемая внутри backend'а. Нарушает изоляцию слоёв — инфраструктурный backend начинает зависеть от прикладного конфига.

## Решение

Введён паттерн **operational lookback window**, реализованный через опциональный параметр `lookback_days: int | None = None` в контракте `CodingAgentBackend.list_all_session_files_for_project`:

- `lookback_days=None` сохраняет старое поведение (полный листинг). Обратная совместимость для случаев, где нужен полный список.
- Положительное значение N ограничивает листинг N последними днями.

Семантика реализации:

- **Codex backend** — при `lookback_days=N` использует `_list_rollout_files_blocking(sessions_root, lookback_days=N, today=date.today())`, которая обходит только `~/.codex/sessions/YYYY/MM/DD/` за последние N дней. Полный `os.walk` не выполняется.
- **Claude backend** — при `lookback_days=N` фильтрует найденные файлы по `os.path.getmtime` через новый helper `_filter_paths_within_lookback_window`. Папка Claude уже изолирована по проекту, но крупные проекты (например, у пользователя в `bloger` 1252 файла) выигрывают от mtime-фильтра.

Единый источник правды для значения окна — константа `config.OPERATIONAL_SESSION_LOOKBACK_DAYS = 3`. Три дня покрывают TTL непрочитанных сообщений (`UNREAD_BUFFER_TTL_HOURS = 3`) с запасом на «возврат после выходных».

Места вызова, переключённые на lookback:

- `session_watcher.SessionWatcher.reset_state` — через `_get_sessions_to_monitor(..., lookback_days=config.OPERATIONAL_SESSION_LOOKBACK_DAYS)`.
- `project_manager._collect_pending_for_backend` — передаёт `lookback_days=config.OPERATIONAL_SESSION_LOOKBACK_DAYS` в `backend_adapter.list_all_session_files_for_project`.

Места, осознанно оставленные без lookback:

- `session_watcher.SessionWatcher.poll_once` — continuous polling должен видеть любые новые сессии независимо от возраста.
- `session_watcher.SessionWatcher.resume_session` — отдельная сессия, ищется напрямую по `session_id`.
- `all_projects_monitor` — режим `/all` сохраняет полный обзор всех сессий каждого проекта.
- `daily_session_registry._remove_orphan_entries` — отдельный путь через `session_file_exists_for_project`, не через listing.

## Заменяет

- **`dev/docs/adr/28.05_16.25-session-change-documenter-parallel-project-switch-init.md`** — частично. Утренний ADR описывал параллелизацию чтения JSONL внутри `SessionWatcher.reset_state` и `_remove_orphan_entries`. Это решение остаётся в силе, но не было достаточным: реальное узкое место — листинг файлов в глобальной папке Codex. Новый ADR не отменяет параллелизацию, а добавляет вышестоящее ограничение (lookback окно).

## Примечания

Не покрыты этим ADR — оставлены как кандидаты для следующих ADR, если измеренная латентность переключения после фикса окажется недостаточной:

- Двойной вызов `daily_session_registry.load_registry` в `_reset_all_state_modules` — один через `session_manager.reset_state` → `session_manager.load_bindings`, второй прямой через `daily_session_registry.reset_state`. Каждый вызов запускает чистку фантомов, дубликатов и сирот.
- Параллелизация `_collect_pending_for_backend` — сборка pending сейчас идёт последовательно по файлам внутри одного backend'а, аналогично проблеме, которую утром починили в `SessionWatcher.reset_state`.
- Перенос сбора pending в фоновую задачу через `asyncio.create_task`, чтобы пользователь получал подтверждение `/pN` мгновенно, а непрочитанные досылались отдельным потоком.

Эти задачи зафиксированы в TaskList текущей сессии и будут разобраны после live-замера эффекта от lookback окна.
