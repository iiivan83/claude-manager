# ADR: устранение дубля progress+final через look-ahead в обработке stream-json

**Дата:** 28-05-2026
**Статус:** принято
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter
**Сущность:** process_manager._process_events, claude_code_backend.ClaudeCodeBackend

## Контекст

Пользователь жаловался, что бот в Telegram перед каждым финальным ответом Claude шлёт второе сообщение с точно таким же текстом — одно с маркером промежуточного `⏳`, второе с финальным `✅`. Воспроизводилось каждый раз, когда Claude отвечал текстом без вызова инструментов (Bash, Read, Edit и т.д.).

Корень причины — пересечение двух источников текста ассистента в stream-json протоколе Claude CLI:

- `assistant`-событие содержит `message.content[]` с блоками типа `text`, `tool_use`, `thinking`. Текстовый блок последнего `assistant`-события за turn — это и есть финальный ответ Claude.
- `result`-событие приходит после всех `assistant`-событий и содержит поле `result` — финальный текст для пользователя. Внутри Claude CLI это поле собирается из последнего текстового блока последнего `assistant`-сообщения (фильтр в `QueryEngine.ts:1124-1131` отбрасывает только синтетические сообщения `SYNTHETIC_MESSAGES`).

То есть для типичного turn без инструментов последний `assistant.text` == `result.result`.

В `process_manager._process_events` оба источника читались независимо:

- `claude_code_backend.read_progress_text_from_event` извлекал `text`/`thinking` из `assistant`-события — текст сразу шёл через `progress_callback` как промежуточное (флаг `is_final=False`, иконка `⏳`).
- `claude_code_backend.read_assistant_text_from_event` извлекал `result.result` из `result`-события — текст накапливался в `last_assistant_text` и возвращался в `SendResult` как финальный (флаг `is_final=True`, иконка `✅`).

Throttle (`PROGRESS_THROTTLE_SECONDS = 30`) частично маскировал дубль в моках, но в реальной жизни Claude между `assistant` и `result` мог тратить много секунд (особенно при долгих ответах с tool_use), и тротл пропускал финальный `text` как progress, а потом тот же текст приходил как финал.

## Решение

Ввести look-ahead-буфер `pending_progress_text` в `process_manager._process_events`. Семантика: текст из `assistant`-события не отправляется как progress сразу — он сначала кладётся в буфер. Дальнейшая судьба буфера зависит от следующего события:

- **Следующее не-terminal событие** (`tool_result`, новое `assistant` с другим text-блоком, `thinking` и т.д.) — буфер flush-ится как progress через `progress_callback` (с учётом тротла), сбрасывается в `None`. Это значит, что текст в буфере был промежуточным шагом, а turn продолжается.
- **Terminal событие** (`result`) — сравниваем буфер с `last_assistant_text` (содержимым поля `result.result`). Если равны — буфер подавляется: финал уйдёт один раз через обычный путь `SendResult` → `claude_interaction.handle_claude_result` → `bot.send_response(is_final=True)`. Если не равны — flush как progress: это был thinking-блок (внутренние размышления Claude) или промежуточный текст, который не попал в финальный `result.result`, его пользователь должен получить.

Сравнение точное (`==`), без нормализации. Это безопасно, потому что Claude CLI выдаёт финальный `result.result` дословно из последнего текстового блока — без обрезки, переноса строк или дополнительного форматирования.

Альтернативы, рассмотренные и отвергнутые при принятии решения:

- **Вариант B — `read_progress_text_from_event` возвращает только `thinking`-блоки, никаких `text`.** Проще в реализации (правка в одной функции, не нужен буфер в `_process_events`), но обрывает доставку промежуточных текстов между шагами с инструментами. На длинных turn-ах пользователь увидел бы только финал — это ухудшение UX, и пользователь явно отверг этот вариант.
- **Вариант C — дедупликация на стороне `handle_claude_result`: если `result.text == последний отправленный progress`, не слать финал.** Самый простой, но самый хрупкий: пользователь не увидит `✅`-маркер финала, а если включён silence mode (`silence_mode_registry`) — финал подавится вместе с промежуточным и пользователь не получит ничего.

## Связанные файлы

- **`src/claude_manager/process_manager.py`** — в функцию `_process_events` добавлена переменная `pending_progress_text: str | None = None` и логика look-ahead: flush буфера на не-terminal событии (с учётом `_should_send_progress`), сравнение буфера с `last_assistant_text` на terminal-событии и решение «подавить или flush». Логика не меняет интерфейс функции, не задевает обработку `session_id_callback`, `update_session_id`, `_check_stop_requested`, retry/stop-логику.
- **`tests/test_process_manager.py`** — добавлены два регрессионных теста:
  - `test_progress_not_duplicated_when_assistant_text_equals_final_result` — собирает поток из `assistant`+`result` с одинаковым текстом, проверяет что `progress_callback` НЕ был вызван с этим текстом, а `SendResult.text` равен ожидаемому финалу.
  - `test_intermediate_assistant_text_still_sent_as_progress` — собирает поток с двумя assistant-событиями: промежуточный текст + `tool_use`, потом `tool_result`, потом финальный assistant, потом `result`. Проверяет что промежуточный текст ушёл как progress, а финальный — нет.

## Примечания

Тесты с потоками `thinking`+`result` (например, `test_send_message_with_progress`, `test_progress_throttle_allows_after_interval`, `test_progress_callback_receives_remapped_session_id`) проходят: `thinking`-текст не совпадает с `result.result`, поэтому буфер flush-ится как progress на terminal-событии.

Эмпирическая проверка через TDD цикл RED→GREEN→Revert→Restore: после отката изменений в `src/claude_manager/process_manager.py` и `claude_code_backend.py` (через `git checkout --`) новый тест `test_progress_not_duplicated_when_assistant_text_equals_final_result` падает с сообщением «Финальный текст не должен дублироваться как progress (было отправлено: [...])». После восстановления — все 94 теста в `test_process_manager.py` + `test_claude_code_backend.py` зелёные.

Источник протокола stream-json и фильтрации `SYNTHETIC_MESSAGES`: `dev/docs/claude-cli-stream-json-protocol.md` (раздел «result — финальный результат запроса» и раздел «Синтетические сообщения в JSONL-файлах сессий»).
