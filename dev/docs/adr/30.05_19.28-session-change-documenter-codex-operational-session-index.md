# ADR: Codex operational index для быстрого переключения проектов

- **Дата:** 30-05-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** `codex_session_index`, `config.OPERATIONAL_SESSION_LOOKBACK_DAYS`, pending-сбор при `/pN`

## Контекст

Переключение проекта через `/pN` стало занимать 7-17 секунд на профиле с большой историей Codex. Причина была не в Telegram и не в reset state как таковом: горячий путь несколько раз подряд сканировал глобальную папку `~/.codex/sessions/YYYY/MM/DD/`.

Codex хранит rollout-файлы не внутри проекта. Чтобы понять, относится ли файл к проекту, нужно открыть JSONL и прочитать `session_meta.payload.cwd`. При тысячах файлов это превращает обычное переключение проекта в повторный global scan внешней истории.

Lookback-only решение от 29-05-2026 уменьшило окно, но не убрало саму проблему: каждый вызов всё равно заново открывал meta-записи всех файлов в окне.

## Решение

Добавлен in-memory operational index для Codex rollout-файлов:

- `codex_session_index.py` хранит карту `project_dir -> list[SessionFileInfo]` за operational-окно.
- `codex_session_index_paths.py` держит path/date/signature helpers, чтобы основной модуль остался меньше 300 строк.
- `codex_session_file_listing.list_all_session_file_infos_for_project(..., lookback_days=N)` делегирует в индекс.
- Ветка `lookback_days=None` оставлена как legacy full scan для совместимости.
- `config.OPERATIONAL_SESSION_LOOKBACK_DAYS` увеличен до 4 дней, потому что повторные обращения в окне теперь переиспользуют индекс.

Индекс инвалидируется не только по TTL. Перед переиспользованием проверяется дешёвая подпись date-директорий: путь директории, `st_mtime_ns` и число rollout-файлов. Если подпись изменилась, индекс пересобирается. Если подпись не изменилась, для возвращаемых файлов проекта обновляется только `mtime`.

## Pending no-op

Pending-доставка при `/pN` тоже стала дешевле:

- watcher сохраняет в `SessionUnreadState` поле `last_modified_at`;
- `unread_buffer` хранит это поле вместе с cursor-ом;
- pending-сбор сначала сравнивает `SessionFileInfo.last_modified_at` с сохранённым значением;
- если файл не менялся, JSONL не читается;
- если `mtime` изменился, сначала читается лёгкий cursor;
- полный snapshot читается только когда вырос `raw_record_count`.

Ошибки отдельных session files логируются и не отменяют сбор pending для остальных файлов. Изменившиеся файлы проверяются параллельно с лимитом 16.

## Связанные файлы

- **`src/claude_manager/codex_session_index.py`** — новый operational index: строит и переиспользует карту Codex rollout-файлов по проектам за 4-дневное окно.
- **`src/claude_manager/codex_session_index_paths.py`** — вынесены path/date/signature helpers, чтобы основной индекс не разрастался.
- **`src/claude_manager/codex_session_file_listing.py`** — горячий листинг с `lookback_days` теперь делегирует в индекс, а `lookback_days=None` сохраняет legacy full scan.
- **`src/claude_manager/project_pending_delivery.py`** — pending-сбор вынесен из `project_manager.py` и стал mtime/cursor-first.
- **`src/claude_manager/project_manager.py`** — переключение проектов использует новый pending-сбор без изменения UX.
- **`src/claude_manager/coding_agent_backend.py`** — `SessionUnreadState` получил `last_modified_at`, чтобы pending мог отличать неизменившиеся файлы.
- **`src/claude_manager/session_file_polling_cursors.py`** — watcher cursor хранит `last_modified_at`.
- **`src/claude_manager/unread_buffer.py`** — unread snapshot сохраняет `last_modified_at` вместе с cursor-ом.
- **`src/claude_manager/coding_agent_session_file_poller.py`** — watcher записывает `last_modified_at` при reset, poll, resume и snapshot export.
- **`src/claude_manager/config.py`** — operational lookback увеличен с 2 до 4 дней, потому что повторный global scan убран из горячего пути.
- **`tests/test_codex_session_index.py`** — регрессии по индексу: окно, cache reuse, directory-signature rebuild, `mtime` refresh и concurrent cold callers.
- **`tests/test_project_manager_pending_optimization.py`** — регрессии по mtime/cursor-first pending и изоляции ошибки одного session file.
- **`tests/test_session_watcher_mtime.py`** — регрессии по сохранению `last_modified_at` в watcher state.

## Последствия

UX `/pN` не изменён: пользователь по-прежнему получает подтверждение переключения после pending-проверки, а не раннее подтверждение с фоновой досылкой.

Положительные последствия:

- повторные operational-листинги Codex в одном свежем окне не перечитывают `session_meta` всех rollout-файлов;
- чистый pending no-op не читает полные snapshot-ы неизменившихся файлов;
- `project_manager.py` вышел из error-зоны по размеру за счёт выноса pending-сбора в `project_pending_delivery.py`.

Ограничения:

- индекс живёт только в памяти процесса и пересобирается после рестарта;
- `/sessions` не использует operational index, потому что ему нужен preview;
- `/all` не переписан на этот индекс в этой задаче;
- `coding_agent_session_file_poller.py` остаётся больше 500 строк и требует отдельного разреза по уже зафиксированному плану.

## Проверки

- `tests/test_codex_session_index.py` проверяет группировку по проектам, 4-дневное окно, переиспользование индекса, directory-signature rebuild, refresh `mtime` и concurrent cold callers.
- `tests/test_project_manager_pending_optimization.py` проверяет mtime/cursor-first pending и локализацию ошибок одного файла.
- `tests/test_session_watcher_mtime.py` проверяет, что watcher сохраняет `last_modified_at` в reset, poll и resume.
