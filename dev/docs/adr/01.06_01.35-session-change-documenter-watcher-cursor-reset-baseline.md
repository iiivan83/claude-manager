# ADR: лёгкий cursor baseline при reset watcher

- **Дата:** 01-06-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** `session_watcher.reset_state`, project switch, backend cursor contract

## Контекст

При переключении проекта бот сбрасывает watcher-состояние: это нужно, чтобы фоновой наблюдатель начал смотреть файлы нового проекта и не доставил старые сообщения повторно.

Раньше reset мог читать полный snapshot файла сессии. Для больших Codex rollout JSONL это дорого: ради baseline нужно знать только позицию в файле, а не разбирать всю историю сообщений. Такая работа попадала в горячий путь переключения проекта и могла добавлять задержку там, где пользователь ждёт простой ответ от бота.

## Решение

`SessionWatcher.reset_state()` должен использовать lightweight cursor через backend-метод `read_session_file_cursor()`. Cursor snapshot хранит:
- количество непустых JSONL-записей;
- признак активного хода;
- `mtime` файла;
- при наличии сообщений - их количество, но полный список сообщений для baseline не обязателен.

Если backend вернул cursor без сообщений, watcher сохраняет sentinel `CURSOR_ONLY_PARSED_MESSAGE_COUNT = -1`. На следующем poll он сравнивает новые сообщения не по старому индексу списка сообщений, а по `raw_record_index > previous.raw_count`. Это защищает от повторной доставки старых сообщений после cursor-only reset.

Отдельный reset дневного реестра при переключении проекта убран из `project_manager`: watcher reset сам читает сегодняшние номера, когда строит baseline. Повторный reset дневного реестра дублировал загрузку и очистку записей.

## Связанные файлы

- **`src/claude_manager/coding_agent_session_file_poller.py`** — вводит cursor-only sentinel и доставку новых сообщений по `raw_record_index`.
- **`src/claude_manager/project_manager.py`** — перестаёт отдельно сбрасывать `daily_session_registry` при switch project.
- **`tests/test_session_watcher_reset_cursor.py`** — фиксирует, что reset использует cursor-reader и не redeliver-ит старые сообщения.
- **`tests/test_project_manager.py`** — фиксирует отсутствие лишнего reset дневного реестра.
- **`dev/docs/brd/brd-user-journeys.md`** — уточняет пользовательский сценарий переключения проекта.
- **`dev/docs/adr/project_architecture.md`** — описывает watcher reset baseline как архитектурное правило.

## Примечания

Решение не меняет формат пользовательских сообщений. Оно снижает стоимость reset и сохраняет прежнюю защиту от повторной доставки.
