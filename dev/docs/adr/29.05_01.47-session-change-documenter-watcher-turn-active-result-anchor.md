# ADR: критерий «turn активен» в session_watcher — по result-event, а не по типу последнего record

**Дата:** 29-05-2026
**Статус:** принято
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter
**Сущность:** `claude_code_session_file_reader._compute_is_turn_active_from_parsed_records`, `_read_cursor_count_last_record_and_turn_active`

## Контекст

Пользователь жаловался, что в режиме тишины (silence mode — глобальный флаг, при котором бот доставляет только финальные ответы Claude, а промежуточные `thinking`/`progress` подавляются) всё равно валится много «финальных» сообщений с галочкой ✅. На длинных запросах с инструментами (Bash, Edit, Read и т.д.) один пользовательский запрос мог родить несколько сообщений с пометкой `is_final=True`, причём настоящий ответ был только в самом конце.

Корень причины — критерий завершения turn'а в `session_watcher` (фоновый наблюдатель, который опрашивает `.jsonl`-файлы сессий Claude каждые 2 секунды и доставляет накопленные сообщения от других сессий или после возврата в проект):

- В `claude_code_session_file_reader._read_session_file_snapshot_blocking` поле `is_turn_active` (флаг «Claude ещё работает над запросом») вычислялось как `last_record.get("type") in BUSY_EVENT_TYPES`, где `BUSY_EVENT_TYPES = {"assistant", "progress", "queue-operation"}`.
- В `coding_agent_session_file_poller.SessionWatcher._check_session` финал доставлялся по правилу `is_final = (not snapshot.is_turn_active and position == len(deliverable) - 1)`.

Проблема: при вызове инструмента в файле сессии записи идут так — `assistant` (с tool_use), затем `user` (tool_result — результат работы инструмента), затем снова `assistant` или новый tool_use. После любой строки `user`-tool_result последний record имеет тип `user`, который не входит в `BUSY_EVENT_TYPES` → `is_turn_active = False` → watcher решает, что turn закрыт, и помечает последнее текстовое assistant-сообщение как финал.

Когда баг особенно бьёт по UX:
- Активная сессия после истечения watcher-паузы (`PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS = 120` секунд — защита от утечки пауз) — для длинных turn'ов watcher просыпается и шлёт промежуточные тексты как финалы.
- Фоновые сессии в обычном режиме — пользователь видел галочку ✅ на промежуточных текстах.
- В silence mode эффект максимально заметен, потому что промежуточные сообщения скрыты, и фон «реальных финалов» в виде шума бросается в глаза.

Связанный фон: ADR от 28-05 `28.05_17.02-session-change-documenter-stream-json-progress-final-deduplication.md` чинит другую часть проблемы — дубликат progress+final в `process_manager._process_events`. Этот ADR продолжает ту же линию с другой стороны: устраняет ложные финалы из `session_watcher`.

## Решение

Изменить критерий `is_turn_active` для Claude-бэкенда: вместо проверки типа единственного последнего record сканировать парсенные records с конца к началу и принимать решение по первому встреченному «значимому» record:

- Первым встретился `result` → turn закрыт, `is_turn_active = False`. Claude CLI пишет `result` только при штатном завершении turn'а (см. `dev/docs/claude-cli-stream-json-protocol.md`, раздел «result — финальный результат запроса»), поэтому это однозначный маркер.
- Первым встретился `assistant`/`progress`/`queue-operation` (любой из `BUSY_EVENT_TYPES`) → turn активен, `is_turn_active = True`. Эти типы означают «Claude печатает или генерирует».
- Ни того, ни другого до начала файла не встретилось → `is_turn_active = False` (безопасный дефолт для пустых или повреждённых файлов).

Реализация — общая функция `_compute_is_turn_active_from_parsed_records(parsed_records)` для snapshot reader'а (полный парсинг), и расширенная `_read_cursor_count_last_record_and_turn_active(file_path)` для cursor reader'а (используется в режиме `/all` для baseline без полного парсинга messages). Cursor reader делает reverse-проход по сырым строкам и парсит каждую через `json.loads` до первого result/busy-event'а — в реальных файлах это 1–2 строки с конца, потому что user-tool_result обычно идёт сразу за assistant с tool_use.

Поле `last_record` в `SessionFileSnapshot` сохраняет старую семантику — первый валидный record с конца файла. Эта семантика используется в существующих тестах и проверках (`test_claude_cli_contract.py`, `test_codex_cli_contract.py`), её ломать нельзя. `is_turn_active` теперь вычисляется отдельно, не строго из `last_record`.

Codex-бэкенд не трогался. У Codex есть собственный надёжный критерий завершения turn'а — `is_turn_terminal_session_record(record)`, который проверяет `record.payload.type in {task_complete, terminal_failure_subtypes...}`. Для Codex проблемы tool_result не существует — он использует другой протокол rollout-файлов.

Альтернативы, рассмотренные и отвергнутые:

- **Вариант B — добавить `"user"` в `BUSY_EVENT_TYPES`.** Запись `user` встречается не только как tool_result — это также свежее сообщение пользователя (новый запрос). Если посередине файла оказывается `user`-сообщение нового запроса, это сигнал, что turn закрылся и пользователь снова что-то спросил. Перепутать эти два кейса по типу нельзя без разбора `message.content[].type == "tool_result"`. Получается громоздко и хрупко.
- **Вариант C — в silence mode подавлять и финалы от watcher, кроме как от фоновых сессий.** Лечит симптом в silence mode, но в обычном режиме ложные финалы от watcher остаются.
- **Вариант D — снять защиту `PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS`.** Тогда зависший обработчик может оставить watcher на паузе навсегда, а защита введена именно из инцидента 22-04-2026 — это регрессия.

## Связанные файлы

- **`src/claude_manager/claude_code_session_file_reader.py`** — добавлена функция `_compute_is_turn_active_from_parsed_records(parsed_records)`, переписана `_read_cursor_count_last_record_and_turn_active(file_path)` взамен старой `_read_cursor_record_count_and_last_record`. Оба ридера (`_read_session_file_snapshot_blocking` и `_read_session_file_cursor_blocking`) перешли на новый критерий. Файл вырос до 512 строк — превышение проектного порога 500 строк на 12 строк зафиксировано осознанно, разбиение модуля по ответственности отложено на отдельную итерацию.
- **`tests/test_claude_code_backend.py`** — добавлены два регрессионных теста:
  - `test_read_session_file_snapshot_turn_active_after_tool_result` — пишет JSONL вида `[user, assistant(text+tool_use), user(tool_result)]` (без `result`-event'а) и проверяет, что `snapshot.is_turn_active is True`. До фикса падал с `False`.
  - `test_read_session_file_cursor_turn_active_after_tool_result` — то же для cursor reader'а (используется в `/all`-режиме).

## Примечания

Эмпирическая проверка через TDD-цикл RED → GREEN: после написания обоих тестов до фикса они падали (`is_turn_active=False` при ожидании `True`). После реализации фикса — оба теста зелёные, полный прогон `pytest tests/ -v` — 1020 passed, 4 skipped, 0 failed.

UX-эффект после фикса:
- В обычном режиме реже встречается галочка ✅ — только настоящие финалы. Промежуточные тексты Claude между шагами с инструментами теперь корректно идут как ⏳ (`is_final=False`) и в обычном режиме отображаются курсивом, а в silence mode подавляются.
- В silence mode пользователь видит только реальный финал на запрос (или вопрос Claude — у CLI это тоже `result`-event с финальным текстом).
- Сообщения от фоновых сессий: пока Claude в фоновой сессии работает — приходят как промежуточные; когда там закрывается turn (появляется `result`-event в `.jsonl`) — приходит финал ✅.

Источник протокола stream-json и описания событий: `dev/docs/claude-cli-stream-json-protocol.md` (раздел «result — финальный результат запроса»). Связанный ADR по дедупликации progress+final: `28.05_17.02-session-change-documenter-stream-json-progress-final-deduplication.md`.
