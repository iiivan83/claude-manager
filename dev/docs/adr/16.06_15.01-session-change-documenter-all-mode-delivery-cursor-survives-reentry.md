# ADR: режим /all переживает повторный вход на собственном курсоре доставки

**Дата:** 16-06-2026
**Статус:** принято
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter
**Сущность:** baseline-курсор `all_projects_monitor` при включении режима `/all`

## Контекст

Режим `/all` (глобальный мониторинг сообщений из всех проектов) держит собственные курсоры доставки в `all_projects_monitor._states` и намеренно не двигает обычный watcher-state, чтобы показанное в `/all` сообщение осталось непрочитанным для исходного проекта. Память о непрочитанном живёт в `unread_buffer` — отдельном in-memory буфере курсоров, который потребляется pending-доставкой при возврате в реальный проект.

Предыдущая правка чинила баг «в `/all` не приходят сообщения из сессий, которые накопили бэклог в другом проекте». Решение было: при включении `/all` строить baseline-курсор каждого кандидата из `unread_buffer`, если там есть снапшот. Это показывало бэклог, но породило новый баг: `unread_buffer` специально не очищается при показе в `/all` (его ждёт доставка в проекте), поэтому при каждом повторном входе в `/all` baseline снова «отматывался» к позиции до сообщения, и одно и то же сообщение приезжало заново. Пользователь видел это как «пока не открою сообщение внутри проекта, оно валится в `/all` каждый раз».

Корень в том, что `unread_buffer` обслуживает двух разных потребителей с противоречащими требованиями: pending-доставка в проекте хочет курсор ДО непоказанных сообщений, а `/all` при повторном входе не должен заново показывать уже показанное. Один и тот же снапшот не может удовлетворить обоих.

## Решение

Сделать собственный курсор доставки `all_projects_monitor` источником правды между включениями `/all`, а `unread_buffer` оставить только как fallback для первого показа бэклога.

При включении режима (`enable_for_chat`) перед очисткой `_states` снимается копия прежних курсоров (`previous_states`) и пробрасывается в построение baseline. Приоритет в `_baseline_state_from_snapshot` теперь такой:

1. **watcher-state активного проекта** — если сессия принадлежит текущему проекту, baseline берётся из обычного watcher (самый свежий показ пользователю).
2. **собственный прежний курсор `/all`** — если режим уже показывал эту сессию в прошлый раз, baseline продолжает с его позиции, поэтому повторный вход не дублирует показанное. `unread_buffer` при этом не читается и остаётся нетронутым для pending-доставки в проекте.
3. **снапшот `unread_buffer`** — только если у `/all` нет собственной памяти о сессии (первый показ): baseline строится из сохранённого при переключении проекта курсора и один раз показывает бэклог.
4. **конец файла** — если ничего из выше, baseline = текущий конец сессии (показываем только то, что появится дальше).

`unread_buffer` ни в одной ветке не потребляется режимом `/all` — он чистится только pending-доставкой при входе в реальный проект. Так одновременно живут оба требования: бэклог из другого проекта показывается один раз, повторный вход в `/all` ничего не дублирует, а вход в сам проект всё ещё досылает сообщение обычным pending-путём.

## Связанные файлы

- **`src/claude_manager/all_projects_monitor.py`** — `_baseline_state_from_snapshot` получил параметр `previous_all_mode_state` и ветку приоритета собственного курсора; `_read_baseline_state` и `_build_baseline_states` пробрасывают `previous_states`; `enable_for_chat` снимает копию `_states` до очистки.
- **`tests/test_all_projects_monitor.py`** — добавлен регресс-тест `test_reentering_all_mode_does_not_redeliver_shown_backlog` и хелпер `_poll_with_mtime`.

## Примечания

Память `/all` и `unread_buffer` обе in-memory, поэтому после перезапуска бота оба источника пусты — повторной доставки после рестарта тоже не будет, поведение согласовано. Ветка watcher-state активного проекта оставлена без изменений: правка прицельно касается только сессий вне активного проекта (`watcher_state is None`), где и проявлялся баг.
