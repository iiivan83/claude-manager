# ADR: finally-блок хода снимает паузу и watchdog наблюдателя только для владеющего хода

## Метаданные

- **Дата:** 05-07-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** `claude_interaction.send_to_claude_and_respond` (finally-блок), поле `SessionWatcherState.pause_owner_token`, методы `SessionWatcher.pause_session` / `resume_session` / `is_pause_owner`

## Контекст

Обработка одного пользовательского сообщения — «ход» (`send_to_claude_and_respond`) — на старте ставит паузу наблюдателя за файлом сессии (`session_watcher.pause_session`), запускает watchdog тишины и помечает флаг `handler_owns_final_delivery` (обработчик сам доставит финал, watcher его придерживает). В `finally` ход всё это снимал: `cancel_agent_silence_watchdog` и `resume_session`.

Проблема в том, что `finally` снимал состояние **безусловно**, не проверяя, принадлежит ли оно ещё текущему ходу. Полное ревью 04.07.2026 выделило из этого два подтверждённых бага (кластер «жизненный цикл хода»):

- **P2-20 (гонка двух ходов одной сессии).** busy-флаг снимается в `finally` инфраструктурного `process_backend_send` РАНЬШЕ, чем ход доставит финал и сгенерирует summary (окно до 45 сек, `SESSION_SUMMARY_TIMEOUT_SECONDS`). Второе сообщение той же сессии в этом окне проходит `is_busy`, ставит свою паузу и свой watchdog — а `finally` первого хода их сносит по тому же `session_id`. У второго хода дублируются промежуточные сообщения (watcher + progress) и нет watchdog-защиты.
- **P2-19 (протечка флага владения финалом).** `resume_session` снимает `handler_owns_final_delivery` только при чистом завершении хода (`not is_turn_active` и появилось новое сообщение). После `/stop` (SIGTERM) `is_turn_active` остаётся True, нового финала нет — условие не срабатывает, флаг остаётся True. Одновременно `finally` обнуляет `paused_at`, из-за чего 120-секундный safety-автоклир в поллере (`PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS`, срабатывает только при `paused_at is not None`) тоже не срабатывает. Флаг зависает навсегда; следующий внешний финал этой сессии (например, продолжение из терминала) молча придерживается и не доставляется.

Общий корень обоих — `finally` не отличает «моё состояние» от «состояния, которое уже занял следующий ход» или «состояния, которое надо принудительно освободить».

## Решение

Введён **токен владения ходом** — единый механизм, закрывающий оба бага.

1. **Токен в состоянии наблюдателя.** В `SessionWatcherState` добавлено поле `pause_owner_token: object | None`. Каждый вызов `send_to_claude_and_respond` создаёт уникальный `object()` и передаёт его в `pause_session(session_id, backend, owner_token=...)`. Второй ход, ставя свою паузу, перезаписывает токен своим. `None` означает, что паузой никто не владеет.

2. **Предикат владения.** `SessionWatcher.is_pause_owner(session_id, token)` возвращает True, если состояния сессии нет вовсе, паузой никто не владеет (`pause_owner_token is None`), либо владеет именно этот токен. False — только если паузой владеет ДРУГОЙ ход. Такой предикат позволяет `finally` безопасно чистить своё даже после сброса состояния при переключении проекта и при этом не трогать чужое.

3. **Ownership-gate в finally.** `finally` теперь снимает watchdog и паузу только внутри `if is_pause_owner(session_id, backend, turn_pause_token)`. Если паузу перехватил второй ход — `finally` первого хода не трогает ни его watchdog, ни его паузу (P2-20).

4. **Снятие владения финалом на путях прерывания.** Локальный флаг `final_answer_delivered_to_user` ставится True только в ветке реальной доставки (`handle_claude_result`). На путях `/stop`, ошибки, смены проекта и all-mode он остаётся False, и `finally` (внутри ownership-gate) вызывает `clear_handler_owns_final_delivery` — принудительно снимает зависший флаг (P2-19).

5. **Согласованность с progress и ремапом id.** `reset_watchdog_on_progress` принимает тот же токен хода и переставляет паузу им же, чтобы progress текущего хода не выглядел как перехват паузы. Токен живёт в объекте `SessionWatcherState`, который при смене `session_id` (temp→real) переносится тем же объектом через `update_session_id`, поэтому переживает ремап.

`resume_session` также принимает `owner_token`: если паузой владеет другой ход — выходит рано (не снимает чужую паузу); при штатном снятии освобождает токен (`pause_owner_token = None`).

## Примечания

- **Отвергнутая альтернатива — идентичность watchdog-таска как признак владения.** Watchdog-таск можно было бы использовать как токен: `start_agent_silence_watchdog` отменяет предыдущий таск и ставит новый. Но progress-события того же хода тоже вызывают `start_agent_silence_watchdog` и заменяют таск СВОИМ новым в пределах одного хода — то есть таск, захваченный на старте хода, устаревает уже после первого progress. Стабильного per-turn признака из таска не выходит. Токен-объект в состоянии сессии стабилен на весь ход и меняется только между ходами — поэтому выбран он.
- **Предикат намеренно возвращает True при отсутствии состояния и при `None`-токене.** «Сносить нельзя» — только при чужом токене. Это даёт безопасную очистку своего состояния даже после сброса `_states` при переключении проекта и не блокирует штатный путь одиночного хода.
- **Детальный TDD-план решения:** `dev/docs/specs/05.07_01.49-stage-3-turn-lifecycle-ownership-plan.md`. Реестр багов: `dev/docs/specs/04.07_13.12-project-wide-bug-review-findings.md` (P2-19 строка 70, P2-20 строка 71).
- **Об имени ADR.** Имя `turn-lifecycle-pause-ownership` согласовано с конвенцией существующих ADR проекта (`watcher-turn-active-result-anchor`, `process-manager-state-split`) и терминологией роудмапа («этап 3 — жизненный цикл хода»). Контекст-свободный агент-ревьюер имён отмечал доменные термины «turn»/«watcher» как непонятные без контекста — это ограничение ревьюера, а не имени: те же термины уже используются в ADR проекта и в CLAUDE.md.
