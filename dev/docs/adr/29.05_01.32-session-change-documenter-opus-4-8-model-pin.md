# ADR: фиксация точной версии модели Claude Opus 4.8 для основного процесса бота

**Дата:** 29-05-2026
**Статус:** принято
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter
**Сущность:** CLI-команда запуска `claude`, формируемая в `claude_runner._build_command_args` и `ClaudeCodeBackend.compose_subprocess_command_args`

## Контекст

До изменения бот не передавал в Claude Code CLI флаг `--model`. Модель определялась дефолтом, привязанным к подписке: на Max-плане это Opus, на Pro/Team-плане — Sonnet. Поведение бота зависело от внешних настроек подписки, которые могут меняться без нашего участия.

28-05-2026 Anthropic выпустила Claude Opus 4.8 (API id `claude-opus-4-8`, цена и контекст 1M токенов остались как у 4.7, существенно подняты бенчмарки агентного кодинга и долгих автономных задач). Алиас `opus` теперь резолвится в 4.8, но это плавающий алиас — при выпуске Opus 4.9 он автоматически переедет на новую модель, и поведение бота снова станет недетерминированным между релизами.

## Решение

Бот всегда передаёт в Claude CLI флаг `--model claude-opus-4-8` — точное имя версии модели, без алиаса.

Технические инварианты решения:

- Точное имя версии (`claude-opus-4-8`), а не алиас `opus` — поведение остаётся воспроизводимым между релизами Anthropic. При выходе 4.9 переход управляется явно правкой одной константы.
- Имя модели вынесено в единую константу `CLAUDE_OPUS_MODEL_ID` в `src/claude_manager/claude_code_backend.py`. Модуль `src/claude_manager/claude_runner.py` импортирует константу — нет дубликата литерала `"claude-opus-4-8"` в двух местах. Правило чистого кода: одна строка, повторяющаяся в 2+ местах, → именованная константа.
- Флаг `--model claude-opus-4-8` ставится в команде ДО `--resume <session_id>`. Это требование контрактного теста `test_compose_args_for_resume_session_appends_resume`, который проверяет, что `--resume` и session_id — последние два аргумента.
- Режим `--effort max` сохранён без изменений: флаг уже стоял в обоих местах сборки команды и не убирался.

Эффект на субагентов (вспомогательные мини-сессии Claude, запускаемые через инструмент Agent/Task для конкретных задач):

- В Claude Code субагенты по умолчанию используют `model: inherit` — наследуют модель родительского процесса.
- В этом проекте ни у одного агента (ни в глобальном `~/.claude/agents/`, ни в `.claude/skills/**/agents/`, ни в `~/.claude/skills/**/agents/`) поле `model:` во frontmatter не задано.
- Переменная окружения `CLAUDE_CODE_SUBAGENT_MODEL` не установлена ни в окружении бота, ни в `.env` проекта.
- Следовательно: все кастомные субагенты, запускаемые ботом, пойдут на `claude-opus-4-8` через наследование.
- Исключение: built-in утилитарные субагенты Claude Code (`Explore`, `statusline-setup`, `claude-code-guide`) имеют захардкоженную модель внутри CLI. Это известный баг Anthropic `anthropics/claude-code#31490`. Флаг `--model` родителя их не переопределяет; обходного пути из коробки нет. По умолчанию они работают на Haiku — для их задач (быстрый поиск файлов, настройка статус-строки) это адекватно по цене и скорости.

Эффект на fast mode не реализован в этой итерации:

- Anthropic пока выкатил `/fast` только как slash-команду интерактивного TUI Claude Code.
- Для headless-режима (`-p --output-format stream-json`), на котором работает бот, официального CLI-флага для fast mode нет.
- Пользователь явно попросил fast mode не трогать.

## Связанные файлы

- **`src/claude_manager/claude_code_backend.py`** — добавлена константа `CLAUDE_OPUS_MODEL_ID = "claude-opus-4-8"` (источник правды), в `compose_subprocess_command_args` добавлены аргументы `"--model", CLAUDE_OPUS_MODEL_ID`.
- **`src/claude_manager/claude_runner.py`** — добавлен импорт `from claude_manager.claude_code_backend import CLAUDE_OPUS_MODEL_ID`, в `_build_command_args` добавлены аргументы `"--model", CLAUDE_OPUS_MODEL_ID`.
- **`tests/test_claude_code_backend.py`** — добавлен тест `test_compose_args_pins_opus_4_8_model`, фиксирующий требование передачи точной версии модели через backend-адаптер.
- **`tests/test_claude_runner.py`** — добавлен тест `test_build_command_args_pins_opus_4_8_model`, фиксирующий то же требование на legacy-пути.

## Примечания

Контрактный тест `tests/integration/test_claude_cli_contract.py` запускает реальный бинарь `claude` через сгенерированную backend-адаптером команду. После изменений тест прогнан и завершился успехом (2 теста, 10.92 секунды). Это эмпирическое доказательство, что у текущей подписки есть доступ к модели `claude-opus-4-8` и CLI принимает флаг `--model claude-opus-4-8` без ошибок.

При будущем переезде на Opus 4.9 правка тривиальна: изменить литерал в константе `CLAUDE_OPUS_MODEL_ID` в одном месте (`claude_code_backend.py`), запустить контрактный тест для проверки доступности новой модели.
