# ADR: классификация ошибок backend по повторяемости — постоянные ошибки не уходят в retry-цикл

**Дата:** 29-05-2026
**Статус:** принято
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter
**Сущность:** `coding_agent_backend.PermanentErrorKind` (enum видов постоянных ошибок), `CodingAgentBackend.classify_permanent_error` (метод контракта), `ClaudeCodeBackend.classify_permanent_error` (реализация для Claude), `process_manager.SendResult.permanent_error_kind` (поле результата), `process_manager._classify_permanent_error_result` (гард перед повтором), `claude_interaction.PERMANENT_ERROR_MESSAGES` / `_build_permanent_error_message` (человекочитаемые сообщения)

## Контекст

Пользователь столкнулся с тем, что переполненная сессия #9 ~11 минут параллельно с новыми сессиями слала сообщения с ошибкой `Prompt is too long` и циклом из 10 повторов. Когда история сессии перестаёт помещаться в модель, финальное событие Claude приходит с `is_error=true` и текстом `Prompt is too long`. Повтор через `--resume` каждый раз грузит ту же самую переполненную историю, поэтому каждая из 10 попыток (`MAX_RETRIES = 10`, интервал `RETRY_INTERVAL_SECONDS = 60`) обречена падать снова.

Корень причины (зафиксирован в RCA-отчёте `dev/docs/logs/root-cause-reports/29-05_11-26_session-overflow-retry-spam.md`) — архитектурный: контракт ошибок backend (`coding_agent_backend.py`) был бинарным (`is_error: bool`, `TerminalStatus SUCCESS|FAILED`), без разделения ошибок на временные (повтор может помочь) и постоянные (повтор бессмыслен). Поэтому `_retry_loop` в `process_manager.py` слепо повторял любой `is_error`.

Это рецидив инцидента 13-04 (`dev/docs/logs/root-cause-reports/13-04_17-45_retry-loop-session-multiplication.md`): тот же модуль, тот же класс проблемы. Частичный фикс тогда закрыл одно проявление (Claude отвечает без финального события), но саму классификацию ошибок так и не добавили — поэтому новая постоянная ошибка дала точно такой же рецидив.

## Решение

Ввести классификацию ошибок по повторяемости и разместить её в контракте backend — потому что только конкретный backend знает форматы ошибок своего CLI-движка (Claude и Codex сообщают об ошибках по-разному).

Конкретные изменения:

- **`coding_agent_backend.py`** — добавлен enum `PermanentErrorKind` с двумя видами: `CONTEXT_OVERFLOW` (переполнение контекста сессии) и `USAGE_LIMIT` (исчерпан лимит запросов). В абстрактный базовый класс `CodingAgentBackend` добавлен **неабстрактный** метод `classify_permanent_error(error_text) -> PermanentErrorKind | None` с дефолтом «возвращает `None`» — то есть по умолчанию любая ошибка считается временной. Дефолт неабстрактный специально: Codex наследует безопасное поведение без собственной реализации.
- **`claude_code_backend.py`** — `ClaudeCodeBackend` переопределяет метод: распознаёт `prompt is too long` → `CONTEXT_OVERFLOW`, `hit your limit` → `USAGE_LIMIT`. Подстроки вынесены в именованные константы `CLAUDE_CONTEXT_OVERFLOW_ERROR_MARKERS` и `CLAUDE_USAGE_LIMIT_ERROR_MARKERS`, сравнение регистронезависимое. Источник подстрок — реальные инциденты из логов бота, а не догадка.
- **`process_manager.py`** — в `SendResult` добавлено поле `permanent_error_kind: PermanentErrorKind | None`. Добавлена функция `_classify_permanent_error_result`: спрашивает у backend-контракта, постоянная ли ошибка; если да — собирает финальный `SendResult` с `retries_used=0` и заполненным `permanent_error_kind`, логирует warning и возвращает его (повтор не запускается). Гард встроен во все три точки запуска повтора: `_retry_loop` (внутри цикла, если очередной повтор вернул постоянную ошибку), `_send_message_backend_aware` и legacy-путь `_execute_send`.
- **`claude_interaction.py`** — транспортный слой владеет текстом для пользователя: словарь `PERMANENT_ERROR_MESSAGES` и функция `_build_permanent_error_message`. В `handle_claude_result` при `permanent_error_kind is not None` пользователю уходит человекочитаемое сообщение вместо «повтор N/10»: для `CONTEXT_OVERFLOW` — «Сессия переполнилась… Начни новую через /new», для `USAGE_LIMIT` — «Исчерпан лимит запросов к Claude… Дождись обновления лимита». Тексты разные намеренно: совет `/new` бесполезен при исчерпанном лимите.

Распределение по слоям: классификация — в контракте backend (формат ошибок специфичен для движка); retry-механизм опирается на классификацию, а не повторяет всё слепо; формулировки для пользователя — в транспортном слое (инфраструктура не знает про Telegram).

Альтернативы, рассмотренные и отвергнутые:

- **Классификация в `process_manager`.** Отвергнута: `process_manager` backend-агностичен и не должен знать конкретные тексты ошибок отдельных CLI-движков.
- **Реализация классификации для Codex сразу.** Отложена намеренно: нет эмпирических образцов того, как Codex сообщает о переполнении/лимите, а в проекте действует правило «контракты с внешними системами проверяются эмпирически, а не по догадке». Codex наследует безопасный дефолт (все ошибки временные) до появления реальных образцов его ошибок.

## Связанные файлы

- **`src/claude_manager/coding_agent_backend.py`** — добавлен enum `PermanentErrorKind` и неабстрактный метод-дефолт `classify_permanent_error`.
- **`src/claude_manager/claude_code_backend.py`** — реализация `classify_permanent_error` для Claude и константы-маркеры ошибок.
- **`src/claude_manager/process_manager.py`** — поле `permanent_error_kind` в `SendResult`, функция `_classify_permanent_error_result`, гарды в `_retry_loop`, `_send_message_backend_aware`, `_execute_send`.
- **`src/claude_manager/claude_interaction.py`** — `PERMANENT_ERROR_MESSAGES`, `_build_permanent_error_message`, ветка в `handle_claude_result`.
- **`tests/test_claude_code_backend.py`** — тест распознавания постоянных ошибок (overflow/limit, регистронезависимость, временная ошибка → `None`).
- **`tests/test_process_manager.py`** — тесты «постоянная ошибка пропускает повторы» и «временная ошибка всё ещё повторяется».
- **`tests/test_claude_interaction.py`** — тест «постоянная ошибка переполнения шлёт подсказку про /new».

## Примечания

Проверка через TDD-цикл RED → GREEN: 4 поведенческих теста до фикса падали (overflow повторялся, сообщения про `/new` не было), после фикса — зелёные. Полный прогон `pytest tests/` — 1028 passed, 4 skipped, 0 failed.

Превышения порога размера файлов (зафиксированы осознанно, разбиение — отдельная задача): `coding_agent_backend.py` — 318 строк и `claude_code_backend.py` — 326 строк (порог предупреждения 300); `process_manager.py` — 1328 строк и `claude_interaction.py` — 543 строки (порог 500, превышение было до этой правки).

Открытые продолжения:
- Классификация постоянных ошибок для Codex — после появления реальных образцов его ошибок.
- Принцип «классификация ошибок backend по повторяемости» рекомендован к фиксации на уровне всего проекта (`architecture.md` или раздел «Архитектурные принципы» в `CLAUDE.md`) — решение о месте открыто.
- Вторичные рекомендации RCA по `/new` (останавливать зависший запрос прошлой сессии) и `/stop` (бить и по фоновым сессиям) — к обсуждению отдельно.
