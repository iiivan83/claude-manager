# ADR: persistent recent_sessions index для быстрых /sessions и /all

- **Дата:** 31-05-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** `recent_sessions_store`, `recent_sessions_refresh`, `/sessions`, `/all`

## Контекст

Claude Manager показывает пользователю последние диалоги через `/sessions` и слушает активность всех проектов через `/all`. До этой работы горячие пользовательские пути могли снова обходить историю session files внешних CLI.

Для Claude Code это обычно терпимо: сессии лежат в папке конкретного проекта. Для Codex это дорого: rollout-файлы лежат в общей папке `~/.codex/sessions/YYYY/MM/DD/`, а проект определяется только после чтения metadata из JSONL. Поэтому быстрый список сессий и `/all`-monitoring не должны зависеть от повторного полного scan-а Codex history.

Предыдущее решение с in-memory operational index ускорило часть operational-путей, но не стало постоянным user-facing индексом для `/sessions` и `/all`.

## Решение

Введён persistent индекс `recent_sessions`:

- `recent_sessions_store.py` хранит быстрые заголовки сессий в SQLite-файле `~/.local/state/claude-manager/recent_sessions.sqlite3`;
- таблица `recent_sessions` хранит только metadata и preview, а полный текст сообщений остаётся в исходных JSONL session files;
- ключ заголовка — `(project_path, backend, session_id)`;
- storage держит максимум 30 последних сессий на проект across all backends;
- cursor-состояние хранится отдельно в `session_cursor_state`, чтобы pruning заголовков не удалял данные, нужные для защиты от повторной доставки;
- SQLite-операции идут через async facade с `asyncio.to_thread`, write lock и `busy_timeout`;
- `recent_sessions_refresh.py` выполняет bounded refresh, но не превращает Telegram hot path в полный historical scan;
- `/sessions` читает `recent_sessions` как основной источник списка и показывает максимум 15 строк;
- `/all` получает capped snapshot кандидатов из `recent_sessions`, а poll-loop каждые 2 секунды проверяет только известные files по `mtime` и cursor-ам.

## Связанные файлы

- **`src/claude_manager/recent_sessions_store.py`** — новый storage boundary: schema init, upsert, query, pruning, independent cursor state и async wrapper над SQLite.
- **`src/claude_manager/recent_sessions_refresh.py`** — bounded refresh facade для project/global candidates и degraded messages.
- **`src/claude_manager/telegram_session_handlers.py`** — `/sessions` читает recent rows, регистрирует дневные номера и сохраняет plain text output для кликабельных `/1`, `/2`, `/3`.
- **`src/claude_manager/all_projects_monitor.py`** — `/all` включает мониторинг по snapshot кандидатов из recent index и не пересобирает discovery в poll-loop.
- **`tests/test_recent_sessions_store.py`** — регрессии по schema, stable sort, cap 30, cursor-state и degraded SQLite errors.
- **`tests/test_recent_sessions_refresh.py`** — регрессии по bounded refresh, preview update и forbidden-scan expectations.
- **`tests/test_all_projects_monitor.py`** — проверяет, что `/all` использует recent global query и poll работает по snapshot.
- **`tests/test_telegram_session_handlers.py`** — проверяет, что `/sessions` читает recent rows и показывает degraded messages.

## Последствия

Положительные последствия:

- `/sessions` и `/all` получают быстрый источник кандидатов без повторного полного обхода Codex history;
- полный текст сообщений не переносится в SQLite, поэтому исходные session files остаются источником истины;
- storage и refresh вынесены в отдельные модули, а Telegram handlers остаются тонкими интеграционными слоями;
- forbidden-scan тесты фиксируют не только текст ответа, но и отсутствие запрещённых вызовов discovery.

Ограничения:

- первый пустой индекс всё равно требует bounded refresh;
- старые сессии за пределами последних 30 на проект могут не попасть в быстрый `/sessions`;
- `all_projects_monitor.py` после интеграции вырос до 695 строк и стал срочным кандидатом на разбиение перед следующими функциональными добавлениями;
- `recent_sessions_store.py` создан сразу на 435 строк, поэтому следующий шаг по storage-логике лучше начинать с выделения schema/query helpers.

## Примечания

Это решение дополняет ADR `30.05_19.28-session-change-documenter-codex-operational-session-index.md`. Operational index ускоряет bounded listing в памяти процесса, а `recent_sessions` даёт постоянный user-facing слой для `/sessions` и `/all`.
