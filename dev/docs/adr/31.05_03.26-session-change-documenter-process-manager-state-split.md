# ADR: вынос состояния process_manager в process_state

- **Дата:** 31-05-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** модули `src/claude_manager/process_manager.py` и `src/claude_manager/process_state.py`

## Контекст

`src/claude_manager/process_manager.py` — инфраструктурный модуль, который запускает Claude/Codex CLI-процессы, отправляет сообщения, читает stream-json события, выполняет retry и обрабатывает `/stop`.

Перед разрезом файл вырос до 1328 строк. Это выше проектного срочного порога 1000 строк. Внутри одного файла смешались две разные ответственности:

- поведение lifecycle: запуск, отправка сообщения, retry, `/stop`, чтение событий
- in-memory state: словари процессов, busy-флаги, stop-events, alias temp session id → real session id

Главный риск был не в новом поведении, а в конкурентном состоянии. Если ошибиться с lock или alias temp→real, можно сломать `/stop`, retry или обновление session_id без заметного изменения публичного API.

## Решение

Принят первый безопасный разрез: вынести только in-memory state и state helpers в новый модуль `src/claude_manager/process_state.py`.

`process_state.py` теперь отвечает за:

- `ProcessKey` и `ManagedProcess`
- `_processes`, `_busy_flags`, `_busy_lock`, `_stop_events`, `_session_id_aliases`
- построение и разбор process key
- разрешение alias-ов session id
- удаление alias-ов
- `is_busy`, `has_process`, `update_session_id`

`process_manager.py` остаётся владельцем поведения:

- `create_process`
- `send_message`
- `_process_events`
- `_retry_loop`
- `_restart_process`
- `stop_process`
- `stop_all_processes`

Для обратной совместимости `process_manager.py` импортирует и реэкспортирует state-объекты из `process_state.py`. Поэтому старый доступ вида `process_manager._processes` продолжает указывать на тот же mutable-объект, что и `process_state._processes`.

Логгер в `process_state.py` намеренно использует имя `claude_manager.process_manager`, чтобы перенесённые helper-функции не поменяли имя источника логов. Это сохраняет поведение для возможных фильтров и привычного чтения логов.

## Связанные файлы

- **`src/claude_manager/process_state.py`** — создан, хранит state процессов и atomics вокруг alias/session-id remap
- **`src/claude_manager/process_manager.py`** — уменьшен с 1328 до 1188 строк, импортирует и реэкспортирует state-объекты
- **`tests/test_process_manager.py`** — добавлен guard-тест, который проверяет identity re-export для state-объектов
- **`docs/superpowers/specs/2026-05-31-process-manager-state-split-design.md`** — спецификация первого разреза
- **`docs/superpowers/plans/2026-05-31-process-manager-state-split-implementation.md`** — план реализации

## Примечания

`process_manager.py` после разреза всё ещё 1188 строк и остаётся срочным кандидатом на следующий разрез. В этой сессии scope намеренно не расширялся: event reader, retry loop и stop strategy не переносились, чтобы не смешивать механический state split с изменением поведения.

Проверки после реализации:

- guard-тест re-export сначала упал из-за отсутствующего `process_state.py`, затем прошёл после реализации
- минимальный process/retry/stop gate: 105 passed
- широкий gate вокруг Telegram-оркестрации: 233 passed
- полный non-E2E suite: 1097 passed, 5 skipped, 3 warnings
