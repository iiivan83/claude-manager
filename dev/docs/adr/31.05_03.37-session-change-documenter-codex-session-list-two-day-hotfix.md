# ADR: двухдневный hotfix для Codex-списка сессий

- **Дата:** 31-05-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** `/sessions`, `LOOKBACK_DAYS_FOR_SESSION_LISTING`, Codex rollout listing

## Контекст

Команда `/sessions` стала отвечать 8-10 секунд на профиле с большой историей Codex. Для пользователя это выглядит как задержка простого списка прошлых диалогов.

RCA `dev/docs/logs/root-cause-reports/31-05_02-47_slow-session-list.md` показал корень: Codex хранит rollout-файлы в общей папке `~/.codex/sessions/YYYY/MM/DD/`, а проект определяется только через `session_meta.payload.cwd` внутри JSONL. Поэтому старый 30-дневный UI-listing открывал и парсил `session_meta` у 14 106 файлов на каждый вызов `/sessions`.

Правильный долгосрочный фикс — user-facing index/cache для `/sessions`: индексировать лёгкие метаданные, а preview дочитывать только для 15 видимых сессий. Иван выбрал быстрый hotfix, чтобы сразу убрать долгую загрузку списка.

## Решение

Временно уменьшить `LOOKBACK_DAYS_FOR_SESSION_LISTING` для Codex UI-listing с 30 до 2 дней.

Практический контракт:

- `/sessions` для Codex просматривает только сегодня и вчера;
- итоговый Telegram-список по-прежнему объединяет Claude и Codex и ограничивается 15 свежими строками после merge;
- Codex-сессии старше двух дней не удаляются, но могут не появляться в `/sessions`;
- operational full scan через `list_all_session_files_for_project(..., lookback_days=None)` остаётся полным и не ограничивается двумя днями.

Это сознательный временный компромисс, а не замена index/cache-фикса.

## Связанные файлы

- **`src/claude_manager/codex_session_file_reader.py`** — значение `LOOKBACK_DAYS_FOR_SESSION_LISTING` изменено с 30 на 2.
- **`tests/test_codex_session_file_listing.py`** — добавлен тест, который фиксирует передачу двухдневного окна в user-facing Codex listing.
- **`tests/test_codex_backend.py`** — ожидания старого 30-дневного поведения обновлены под hotfix-контракт.
- **`dev/docs/brd/brd-user-journeys.md`** — CJM-05 теперь описывает, что старые Codex-сессии могут быть скрыты из `/sessions`.
- **`dev/docs/specs/realised/codex_backend_spec.md`** — технический контракт Codex backend-а обновлён с 30 дней на временные 2 дня.

## Последствия

Положительные последствия:

- `/sessions` перестаёт перечитывать 30-дневную Codex-историю на каждом вызове;
- hotfix маленький и легко откатывается при появлении user-facing индекса;
- full operational-путь остаётся доступен для внутренних сценариев.

Ограничения:

- пользователь может не увидеть в `/sessions` Codex-сессию старше двух дней;
- корневая причина не устранена: повторный global scan внешней истории остаётся, просто окно стало меньше;
- нужен отдельный index/cache-фикс для возвращения широкого окна без задержки.
