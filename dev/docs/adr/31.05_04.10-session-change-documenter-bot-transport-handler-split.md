# ADR: разрез bot.py на Telegram handler-модули

- **Дата:** 31-05-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** transport-слой Telegram-бота

## Контекст

`src/claude_manager/bot.py` — входная точка Telegram-бота. Через него `python-telegram-bot` получает команды, текст, фото и документы, а затем передаёт работу нижним модулям.

Перед разрезом файл вырос до 979 строк и 15 публичных top-level функций. Это было выше stop-порога проекта в 700 строк и выше порога 10 публичных функций, после которого модуль нужно проверять на роль god-module. В одном файле смешались разные ответственности:

- сборка `Application`
- команды `/new`, `/sessions`, `/agent`, `/stop`, `/all`, `/restart`
- текстовые сообщения, фото и документы
- lifecycle после запуска
- callbacks watchers
- режим тишины
- compatibility re-export старых имён

Риск был не в новом пользовательском поведении, а в поддерживаемости. Любая правка transport-слоя требовала читать почти тысячу строк и легко могла задеть соседний сценарий.

## Решение

Принят разрез facade + профильные handler-модули.

`bot.py` теперь остаётся facade-точкой сборки. Он создаёт `Application`, регистрирует handlers, подключает глобальный error handler, инициализирует callback-зависимости и реэкспортирует старые публичные имена для совместимости.

Реальная логика Telegram-сценариев разделена по модулям:

- **`telegram_agent_handlers.py`** — `/agent`, inline-клавиатура и callback data выбора CLI-бэкенда
- **`telegram_session_handlers.py`** — `/new`, `/sessions`, `/all`, `/stop` и переключение сессий по дневному номеру
- **`telegram_input_handlers.py`** — обычный текст, фото, документы, warning-и для monitoring mode и reply anchor для прямого ответа
- **`telegram_lifecycle_handlers.py`** — `post_init`, watcher callbacks, `/restart`, `/silence_on`, `/silence_off`

Handler-модули не импортируют `bot.py`. Если им нужен Telegram `Application` или проверка доступа, `bot.py` передаёт это через `init_callbacks`. Такой контракт уже использовался в проектных handler-ах и уменьшает риск циклических импортов.

`process_manager.py` и `process_state.py` не менялись. Это было отдельное ограничение: текущий refactor transport-слоя не должен смешиваться с lifecycle/state разрезом процессов.

## Связанные файлы

- **`src/claude_manager/bot.py`** — уменьшен с 979 до 266 строк, стал facade для сборки приложения и регистрации handlers
- **`src/claude_manager/telegram_agent_handlers.py`** — создан, владеет сценарием выбора CLI-бэкенда
- **`src/claude_manager/telegram_session_handlers.py`** — создан, владеет командами сессий и `/stop`
- **`src/claude_manager/telegram_input_handlers.py`** — создан, владеет пользовательским вводом: текстом, фото и документами
- **`src/claude_manager/telegram_lifecycle_handlers.py`** — создан, владеет lifecycle Telegram-приложения и служебными командами
- **`tests/test_bot.py`** — уменьшен до facade/registration/re-export проверок
- **`tests/test_telegram_agent_handlers.py`** — создан, проверяет agent handler-сценарии
- **`tests/test_telegram_session_handlers.py`** — создан, проверяет session handler-сценарии
- **`tests/test_telegram_input_handlers.py`** — создан, проверяет input handler-сценарии
- **`tests/test_telegram_lifecycle_handlers.py`** — создан, проверяет lifecycle handler-сценарии
- **`docs/superpowers/specs/2026-05-31-bot-transport-handler-split-design.md`** — спецификация разреза
- **`docs/superpowers/plans/2026-05-31-bot-transport-handler-split-implementation.md`** — план реализации и gates

## Примечания

Пользовательское поведение Telegram-бота не менялось: команды и сообщения регистрируются теми же handler-именами, а старые импорты через `claude_manager.bot` сохранены как compatibility re-export.

Проверки после merge в `main`:

- полный non-E2E suite: 1102 passed, 5 skipped, 3 warnings
- `compileall src/claude_manager`: ok
- `git diff --check`: ok

Size guard после разреза:

- `src/claude_manager/bot.py` — 266 строк, 1 публичная top-level функция
- новые production handler-модули ниже 300 строк
- тестовый техдолг остался: `tests/test_bot.py` 1455 строк, `tests/test_telegram_input_handlers.py` 555 строк, `tests/test_telegram_session_handlers.py` 572 строки
