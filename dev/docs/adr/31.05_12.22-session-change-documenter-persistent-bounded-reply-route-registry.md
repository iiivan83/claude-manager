# ADR: персистентный bounded reply-route registry

- **Дата:** 31-05-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** `reply_route_registry.py`, формат `reply_routes.json`, сценарий address-reply после restart

## Контекст

Claude Manager позволяет Ивану ответить Telegram reply на старое сообщение бота. Для этого бот хранит route: связь между Telegram `message_id` сообщения бота и исходными `project_path`, `session_id`, `backend` и номерами для ссылки.

Предыдущее решение хранило route только в памяти процесса. После `/restart` бот забывал старые маршруты. На практике это проявилось так: Иван ответил на сообщение бота после restart, но короткое подтверждение `Передал в ...` не пришло. Текст ушёл как обычное сообщение в активную сессию, потому что handler не нашёл route и вернул управление обычной обработке текста.

Нужно было решить две задачи:

1. Сохранять маршруты между штатными restart-ами.
2. Не допустить бесконечного роста файла маршрутов.

## Решение

`reply_route_registry.py` заменён с in-memory-only реестра на disk-backed реестр с оперативной копией в памяти.

Приняты правила:

1. Route сохраняется в `~/.local/state/claude-manager/reply_routes.json`.
2. `telegram_lifecycle_handlers.post_init()` загружает этот файл при старте бота.
3. `reply_route_registry.register_route()` сохраняет файл после регистрации нового route, если реестр был успешно загружен.
4. Реестр хранит последние 200 маршрутов. При добавлении 201-го самый старый route удаляется.
5. Если пользователь отвечает на сообщение текущего бота, но route неизвестен, handler не угадывает цель по активной сессии и показывает понятную ошибку.

Формат persisted-записи хранит минимальный route target:

- `chat_id`
- `bot_message_id`
- `project_path`
- `session_id`
- `backend`
- `session_number`
- `project_number`
- `project_name`

## Связанные файлы

- **`src/claude_manager/reply_route_registry.py`** — добавлены загрузка, сохранение, JSON-сериализация и ограничение до последних 200 маршрутов
- **`src/claude_manager/telegram_lifecycle_handlers.py`** — `post_init()` загружает persisted reply routes при старте бота
- **`src/claude_manager/reply_route_handler.py`** — unknown reply на сообщение текущего бота больше не уходит молча в активную сессию
- **`tests/test_reply_route_registry.py`** — проверяет reload persisted route и вытеснение самой старой записи при 201-м route
- **`tests/test_reply_route_handler.py`** — проверяет unknown bot reply внутри проекта
- **`tests/test_reply_route_handler_background.py`** — проверяет, что persisted route после reload сохраняет быстрое подтверждение
- **`tests/test_telegram_lifecycle_handlers.py`** — проверяет загрузку reply-route registry в `post_init()`
- **`dev/docs/brd/brd-user-journeys.md`** — обновлён CJM-17: route восстанавливается с диска, хранится последние 200 записей
- **`dev/docs/specs/31.05_04.56-reply-routing-v1-spec.md`** — обновлена спецификация v1: persistent storage больше не является нецелью
- **`dev/docs/adr/project_architecture.md`** — обновлена архитектурная схема reply-route registry

## Заменяет

Частично заменяет решение из `dev/docs/adr/31.05_11.13-session-change-documenter-telegram-reply-route-registry.md`: прежний route target и разделение `reply_route_registry`/`reply_anchor_registry` остаются в силе, но ограничение «только in-memory, после restart маршруты теряются» больше не действует.

Также уточняет ограничение из `dev/docs/adr/31.05_11.43-session-change-documenter-reply-route-background-dispatch.md`: быстрые подтверждения сохраняются, но route registry теперь переживает штатный `/restart`.

## Примечания

TTL по времени не вводился. Ограничение сделано по количеству, потому что пользователь явно выбрал окно в 200 записей: это просто, предсказуемо и достаточно для недавних Telegram reply.
