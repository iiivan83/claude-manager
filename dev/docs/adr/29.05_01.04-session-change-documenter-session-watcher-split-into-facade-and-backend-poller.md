# ADR: разбиение session_watcher на facade-модуль и per-backend poller

- **Дата:** 29-05-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** модуль `src/claude_manager/session_watcher.py` и связанные с ним новые модули в `src/claude_manager/`

## Контекст

Модуль `src/claude_manager/session_watcher.py` отвечает за фоновое наблюдение за файлами сессий CLI-агентов (Claude Code и Codex) и доставку появляющихся в них ответов ассистента пользователю в Telegram. К моменту начала работы он накопил 847 строк кода и содержал минимум пять разнородных ответственностей в одном файле:

- константы таймингов (как часто опрашивать, какой backoff применять)
- два dataclass'а состояния (`SessionWatcherState`, `MissingFileRetryState`)
- legacy-хелперы для чтения старого формата Claude JSONL (`_extract_message_text`, `_extract_assistant_messages`, `_is_empty_response`)
- хелперы для вызова callback'а с двумя возможными сигнатурами (`_callback_accepts_backend`, `_invoke_callback`, `_current_session_matches`)
- большой класс `SessionWatcher` с механикой polling'а (~470 строк)
- module-level фасадные функции (`start`, `pause_all`, `resume_all`, `pause_session`, `resume_session`, `reset_state`, `update_session_id`, `get_seen_counts_snapshot`, `is_session_paused`) и реестр watcher'ов

847 строк — это явное превышение проектных порогов: больше 300 строк — предупреждение, больше 500 — ошибка с обязательным разбиением. Кроме того, файл попадает под определение god-модуля (10+ публичных функций и классов разной направленности).

Дополнительный фактор — на модуль завязано большое количество тестов и внешних потребителей (`bot.py`, `claude_interaction.py`, `project_manager.py`, `all_projects_monitor.py`, интеграционные и юнит-тесты). Любая ломка публичного API или приватных имён `session_watcher._watchers`/`_callback`/`_get_current_session` сломала бы значимую часть тестового набора (1016 тестов на момент рефакторинга).

Часть кода (compat-хелперы для старого формата Claude JSONL — `_extract_message_text`, `_extract_assistant_messages`) фактически не использовалась в production: их единственный потребитель — три юнит-теста класса `TestLegacyExtractionHelpers`. Это «зомби»-код, который продолжал жить только за счёт собственных тестов и удорожал любую дальнейшую правку модуля.

## Решение

Принято два связанных архитектурных решения:

### 1. Разбиение на facade + 3 кирпича через реэкспорты

`session_watcher.py` превращается в тонкий facade-модуль (~194 строки), который:

- держит глобальный реестр `_watchers: dict[BackendName, SessionWatcher]`, глобальный callback и getter активной сессии (используются `start()` и фасадными функциями)
- содержит все публичные функции фасада: `start`, `pause_all`, `resume_all`, `pause_session`, `resume_session`, `reset_state`, `update_session_id`, `get_seen_counts_snapshot`, `is_session_paused`, и приватные `_get_watcher`, `_get_all_watchers`, `_poll_sessions`
- реэкспортирует все имена, на которые опираются тесты (`SessionWatcher`, `SessionWatcherState`, `_is_empty_response`) и внешние модули (`POLL_INTERVAL_SECONDS`, `ERROR_RETRY_DELAY_SECONDS`, `BackendName`)
- импортирует `config`, `coding_agent_backend`, `daily_session_registry`, `session_manager` как module-level имена — чтобы тесты `patch.object(session_watcher.X, ...)` продолжали работать

Реальный код переезжает в три новых модуля:

- `session_file_polling_intervals.py` — все числовые константы про тайминги (POLL_INTERVAL_SECONDS, ERROR_RETRY_DELAY_SECONDS, PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS, MISSING_FILE_RETRY_*, MAX_CONCURRENT_RESET_READS)
- `session_file_polling_cursors.py` — dataclass'ы состояния `SessionWatcherState` и `MissingFileRetryState`, а также type aliases `MessageCallback` и `CurrentSessionGetter`
- `coding_agent_session_file_poller.py` — класс `SessionWatcher` со всей механикой polling'а плюс приватные хелперы `_is_empty_response`, `_message_should_be_delivered`, `_callback_accepts_backend`, `_invoke_callback`, `_current_session_matches`, константа `NO_RESPONSE_MARKERS`

Параллельно полностью удаляются legacy compat-функции `_extract_message_text`, `_extract_assistant_messages` и три теста класса `TestLegacyExtractionHelpers`, которые их проверяли. Эти функции не использовались в production и поддерживали только собственные тесты.

### 2. Явное согласие на временное превышение порога 500 строк в `coding_agent_session_file_poller.py`

После рефакторинга `coding_agent_session_file_poller.py` получился 608 строк (с учётом маркер-комментария в шапке — 625), что превышает проектный порог 500. Класс `SessionWatcher` сам по себе занимает ~470 строк — это не god-модуль, а один цельный класс с одной чёткой ответственностью (наблюдение за файлами сессий одного coding-agent backend'а), но крупный.

С явного согласия владельца проекта принято решение зафиксировать текущее состояние с обязательным маркер-комментарием в шапке файла, чтобы превышение не было «тихим». Маркер описывает:

- факт превышения (608 строк против порога 500) и причину
- что согласие получено явно после рефакторинга
- план дальнейшего сокращения через два следующих шага: вынести callback-хелперы в sibling-модуль (~55 строк) и выделить backoff-логику отсутствующих файлов в отдельный класс `MissingSessionFileBackoffTracker` (~50 строк), что суммарно приведёт файл к ~500 строкам

## Связанные файлы

- **`src/claude_manager/session_watcher.py`** — переписан в тонкий facade-модуль с реэкспортами и фасадными функциями (847 → 194 строки)
- **`src/claude_manager/session_file_polling_intervals.py`** — создан, содержит числовые константы таймингов polling'а (22 строки)
- **`src/claude_manager/session_file_polling_cursors.py`** — создан, содержит dataclass'ы `SessionWatcherState`, `MissingFileRetryState` и type aliases callback-функций (30 строк)
- **`src/claude_manager/coding_agent_session_file_poller.py`** — создан, содержит класс `SessionWatcher` и приватные хелперы фильтрации/dispatch'а (625 строк, превышение порога зафиксировано в шапке файла с планом сокращения)
- **`tests/test_session_watcher.py`** — удалены импорты `_extract_message_text`, `_extract_assistant_messages` и три теста класса `TestLegacyExtractionHelpers`; класс переименован в `TestEmptyResponseDetection` (оставлен один тест `test_empty_response_markers`)

## Примечания

- Все 1016 пройденных тестов проекта остались зелёными после рефакторинга — это валидация того, что публичное API и приватные имена, на которые опираются тесты, сохранились в `session_watcher.py`. Никаких изменений в тестах кроме удаления трёх legacy-тестов не потребовалось.
- Facade-модуль импортирует модули `config`, `coding_agent_backend`, `daily_session_registry`, `session_manager` под теми же именами — это критично для существующих `patch.object(session_watcher.X, ...)` в тестах. Альтернативный путь (выделить facade в подпакет `session_watcher/`) был отвергнут как более инвазивный.
- Решение про temp-превышение в `coding_agent_session_file_poller.py` фиксируется именно в ADR, а не в комментариях кода, чтобы оставить долгоживущий след архитектурного компромисса. Маркер в шапке файла указывает на этот ADR неявно через описание плана сокращения.
