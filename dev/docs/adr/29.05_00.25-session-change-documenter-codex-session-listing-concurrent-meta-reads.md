# ADR: Параллельное чтение метаданных Codex-сессий с ограничением N=16

- **Дата:** 29-05-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** модуль `codex_session_file_listing`, константа `MAX_CONCURRENT_FILE_READS`, хелпер `_gather_optional_results_with_concurrency_limit`

## Контекст

При листинге Codex-сессий для проекта (вызовы из `SessionWatcher.reset_state`, `SessionWatcher.poll_once`, `project_manager._collect_pending_for_backend` и режима `/all`) модуль `codex_session_file_listing` выполнял три типа I/O-операций над каждым найденным файлом:

- `_read_project_meta_pair` — открыть JSONL, прочитать первую `session_meta`-запись и достать `payload.cwd`, чтобы проверить принадлежность файла к нужному проекту
- `_build_session_file_info` — построить полный объект `SessionFileInfo` с preview и активностью
- `_build_operational_session_file_info` — построить лёгкий объект `SessionFileInfo` без preview

Все четыре обхода были написаны как обычные `for`-циклы с `await` на каждой итерации. На большом профиле пользователя (десятки сессий внутри даже двухдневного lookback окна) это давало последовательное выполнение десятков-сотен I/O-операций — несмотря на то, что технически каждое чтение уже шло через `asyncio.to_thread`. Под `asyncio.gather` они могли бы выполняться одновременно, но рядом существовала и обратная опасность — если просто прогнать всё через `gather` без лимита, на больших списках сессий получим сотни одновременных file descriptors и упрёмся в системные ограничения или thread pool.

Проблема обострилась после расширения operational lookback на `poll_once` (отдельный ADR от 29.05): теперь listing вызывается каждые 2 секунды, и последовательная обработка наблюдалась как непрерывная нагрузка на горячем пути.

Альтернативы, которые рассматривались и отвергнуты:

- **`asyncio.gather` без лимита** — на больших списках поднимает сотни одновременных I/O, грозит исчерпанием file descriptors и насыщением thread pool, особенно когда listing вызывается ещё и из `/all` режима параллельно по всем проектам.
- **Полный пересмотр на отдельный thread pool / process pool** — несоразмерно сложности задачи; модуль и так живёт в asyncio.
- **Кэширование результатов meta-чтения на диске** — отдельный persistent state с отдельной инвалидацией; не лечит первое чтение.

## Решение

Введён общий хелпер `_gather_optional_results_with_concurrency_limit(awaitables)` — запускает все awaitable одновременно, но через `asyncio.Semaphore(MAX_CONCURRENT_FILE_READS)`, где `MAX_CONCURRENT_FILE_READS = 16`. Хелпер также фильтрует `None` результаты (это совпадает с предыдущим поведением — функции `_read_project_meta_pair` и `_build_*_session_file_info` возвращают `None`, если файл не относится к проекту или невалиден).

Все четыре последовательные циклы переведены на этот хелпер:

- В `_list_session_file_infos_from_paths` — чтение `meta_pairs` и построение полных `session_file_infos`
- В `_list_operational_session_file_infos_from_paths` — чтение `meta_pairs` и построение лёгких infos
- В `_build_operational_infos_by_project` — построение лёгких infos для известного множества проектов
- В `list_all_session_file_infos_by_project` — фильтрация по множеству проектов через `_read_project_meta_pair_for_known_projects`

Число 16 — компромисс между параллелизмом и системными лимитами. Это совпадает с лимитом, который уже используется в проекте для параллельной инициализации при смене проекта (см. ADR 28.05_16.25 про parallel-project-switch-init), и держит общий стиль в одном порядке величины.

## Связанные файлы

- **`src/claude_manager/codex_session_file_listing.py`** — добавлены константа `MAX_CONCURRENT_FILE_READS = 16` и хелпер `_gather_optional_results_with_concurrency_limit`; четыре последовательные `for`-цикла с `await` переведены на хелпер
- **`tests/test_codex_backend.py`** — добавлен тест `test_list_all_session_files_reads_meta_records_concurrently`: подменяет `_read_project_meta_pair` трекером, считает peak concurrency и падает, если в пике видна только одна параллельная задача (то есть код снова стал последовательным)

## Примечания

Хелпер фильтрует `None`-результаты — это сохраняет поведение «файл не относится к проекту или невалиден ⇒ просто пропускаем без ошибки». Если в будущем понадобится различать `None` как сигнал ошибки — нужен отдельный вариант хелпера.

Тест в `test_codex_backend.py` проверяет не точное значение `peak_in_flight_count == 16`, а более мягкое `peak > 1`. Это защищает от регресса вида «кто-то заменил `_gather_optional_results_with_concurrency_limit` обратно на цикл», но не привязывает тест к конкретной величине семафора — число 16 может ещё перенастраиваться без переписывания тестов.
