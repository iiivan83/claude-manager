# ADR: параллельная инициализация курсоров при переключении проекта

**Дата:** 28-05-2026
**Статус:** принято
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter
**Сущность:** session_watcher.SessionWatcher.reset_state, daily_session_registry._remove_orphan_entries

## Контекст

Переключение проекта (команда `/projects`, `/pN`) занимало 10–15 секунд на большом проекте (`claude_manager` содержит десятки .jsonl-файлов сессий за месяцы работы; пользовательская машина — 1315 файлов сессий суммарно).

Корень проблемы — последовательное чтение и парсинг файлов сессий внутри двух функций инициализации:

- `SessionWatcher.reset_state` — для каждой сессии нового проекта последовательно вызывал `backend.read_session_file_snapshot(file_path)`, который читает файл целиком и парсит все JSONL-записи. Для N=100 сессий это N последовательных I/O+CPU операций, по 30–50 мс каждая.
- `daily_session_registry._remove_orphan_entries` — для каждой записи реестра последовательно вызывал `backend.session_file_exists_for_project(...)` (это stat() на диске).

Аналогичный паттерн уже был оптимизирован в коммите `ad9f419 perf(all): speed up all projects monitoring` для функции `all_projects_monitor._build_baseline_states`: использован `asyncio.gather` с семафором `MAX_CONCURRENT_BASELINE_READS = 16`. Но оптимизация не была распространена на `reset_state` и `_remove_orphan_entries`.

История ребейза, объясняющая разрыв: фича `all-projects monitoring` была добавлена коммитом `520b42b`, откачена `cc5a054 Revert`, переписана `793b6b5`, затем оптимизирована `ad9f419`. Между этим переключение проекта получало новые медленные пути (`a99e1fb feat: фоновые сессии и буфер непрочитанных при переключении проектов`, `82a151a feat: codex support baseline` — второй бэкенд), но без сопутствующей параллелизации.

## Решение

Зафиксировать единый паттерн инициализации курсоров при переключении проекта: **параллельное чтение через `asyncio.gather` с семафором, лимит 16 одновременных I/O операций**.

Применить паттерн к двум функциям:

- `SessionWatcher.reset_state` — выделить новый приватный метод `_read_baseline_states_concurrently`, который собирает корутины чтения снапшотов для всех сессий, ограничивает параллельность через `asyncio.Semaphore(MAX_CONCURRENT_RESET_READS)` и собирает результаты через `asyncio.gather`. Лимит — константа `MAX_CONCURRENT_RESET_READS = 16` в `session_watcher`.
- `daily_session_registry._remove_orphan_entries` — разделить функцию на три этапа: `_classify_registry_entries` (синхронная классификация записей на «требует проверки на диске» и «удалить сразу»), параллельная проверка существования файлов через `asyncio.gather` с `asyncio.Semaphore(MAX_CONCURRENT_ORPHAN_CHECKS)`, `_delete_orphan_entries` (синхронное удаление по итогам). Лимит — константа `MAX_CONCURRENT_ORPHAN_CHECKS = 16` в `daily_session_registry`.

Лимит 16 выбран по аналогии с `MAX_CONCURRENT_BASELINE_READS = 16` в `all_projects_monitor`, чтобы стандарт был единым по проекту. Семафор нужен не для ускорения (диск всё равно не обслужит сотни I/O одновременно), а как защита от исчерпания файловых дескрипторов при больших проектах.

Альтернативы, которые рассматривались и отвергнуты:

- **Cursor-only чтение через `read_session_file_cursor` вместо `read_session_file_snapshot` в `reset_state`** — даёт дополнительное ускорение (не парсит все исторические сообщения, нужны только `raw_record_count` и `is_turn_active`). Отвергнуто на этом шаге: меняет shape данных (`len(snapshot.messages)` теряется), требует расширения cursor-функции. Зафиксирован как кандидат на следующий шаг оптимизации.
- **Параллелизация цикла по бэкендам в `session_watcher.reset_state` (внешний уровень)** — выигрыш ~30% сверху, но текущий выигрыш уже закрывает 10–15-секундную регрессию. Отвергнуто до подтверждения, что новый таймаут пользователя не устраивает.
- **Параллелизация `_collect_pending_messages` в `project_manager`** — не bottleneck: внутри фильтруется по `unread_buffer.restore_snapshot`, и фактическое чтение файла происходит только для 1–2 сессий с непрочитанными сообщениями.

## Связанные файлы

- **`src/claude_manager/session_watcher.py`** — добавлена константа `MAX_CONCURRENT_RESET_READS = 16`. Метод `SessionWatcher.reset_state` упрощён до сборки результата от нового приватного метода `_read_baseline_states_concurrently`, который реализует параллельное чтение снапшотов с семафором.
- **`src/claude_manager/daily_session_registry.py`** — добавлена константа `MAX_CONCURRENT_ORPHAN_CHECKS = 16`. Функция `_remove_orphan_entries` переписана на три этапа: классификация записей, параллельная проверка существования файлов через `asyncio.gather`, удаление сирот. Выделены вспомогательные функции `_classify_registry_entries` и `_delete_orphan_entries`.
- **`tests/test_session_watcher.py`** — добавлен регрессионный тест `TestResetState.test_reset_state_reads_session_snapshots_concurrently`, который отслеживает максимум одновременных in-flight чтений через счётчик `peak_in_flight_count` и падает, если параллельность отсутствует (`peak == 1`).
- **`tests/test_daily_session_registry.py`** — добавлен регрессионный тест `TestRemoveOrphanEntries.test_orphan_cleanup_checks_files_concurrently` с тем же подходом к проверке параллельности.

## Примечания

Подтверждение через микро-бенчмарк (искусственная задержка 30 мс на каждый I/O):

- 10 сессий: 31 мс параллельно против 300 мс последовательно (10× быстрее)
- 50 сессий: 122 мс против 1500 мс (12× быстрее)
- 100 сессий: 214 мс против 3000 мс (14× быстрее)

Кандидаты на следующие шаги, если фактической производительности окажется недостаточно:

- Перевести `SessionWatcher.reset_state` на `read_session_file_cursor` (требует расширения cursor-функции, чтобы возвращала и количество распарсенных сообщений)
- Параллелизовать цикл по бэкендам в `session_watcher.reset_state` (внешний уровень)
- Параллелизовать `_collect_pending_messages` в `project_manager` (внутренний цикл по файлам и/или внешний по бэкендам)

Источники истории, которые позволили восстановить регрессию: `520b42b`, `cc5a054`, `793b6b5`, `ad9f419`, `a99e1fb`, `82a151a` в `git log` ветки `codex-support-spec-implementation-cycle`.
