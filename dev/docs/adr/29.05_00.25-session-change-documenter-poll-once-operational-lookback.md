# ADR: Operational lookback window применяется и в poll_once, окно сужено до 2 дней

- **Дата:** 29-05-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** константа `config.OPERATIONAL_SESSION_LOOKBACK_DAYS`, метод `session_watcher.SessionWatcher.poll_once`, вызов `_get_sessions_to_monitor`

## Контекст

Предыдущий ADR от 28.05.2026 ввёл паттерн «operational lookback window» и применил его только в двух точках: `SessionWatcher.reset_state` (пересборка курсоров при переключении проекта) и `project_manager._collect_pending_for_backend` (сбор непрочитанных при возврате в проект). Метод `poll_once` (фоновый опрос сессий каждые 2 секунды) был осознанно оставлен без lookback — обоснование звучало так: «continuous polling должен видеть любые новые сессии независимо от возраста».

На практике это решение сломалось двумя способами.

**Первое — производительность.** `poll_once` вызывал `backend.list_all_session_files_for_project(...)` без `lookback_days`, что для Codex-бэкенда означало полный `os.walk` по `~/.codex/sessions/` (десятки тысяч файлов от всех проектов пользователя) — и так каждые 2 секунды. На большом профиле это давало непрерывную нагрузку на файловую систему и thread pool asyncio.

**Второе — cold-start flood для сессий за пределами baseline.** При переключении проекта `reset_state` строил baseline курсоров по lookback окну. Сессии за пределами окна в baseline не попадали. Когда `poll_once` потом видел эти сессии в общем сканировании, он считал их «никогда не виденными» (`last_delivered_idx = -1`) и выливал всю их историю в Telegram как новые сообщения. Это прямой регресс пользовательского сценария — пользователь после переключения проекта получал шквал старых сообщений из давно завершённых сессий.

Изначальное обоснование «poll должен видеть любые сессии» не выдержало столкновения с реальностью эксплуатации:

- В Codex-бэкенде «любая сессия» означает «любая из десятков тысяч файлов, оставшихся от других проектов за всё время работы». Без ограничения это бесконечная работа.
- Контракт baseline/курсоров требует, чтобы то же самое окно использовалось и в poll, иначе baseline и poll не согласованы по множеству наблюдаемых сессий.

Альтернативы, которые рассматривались и отвергнуты:

- **Использовать в `poll_once` другое (более широкое) окно** — даёт другой класс несогласованности с baseline и не лечит производительность горячего пути.
- **Запоминать на диске множество `session_id`, которые baseline видел в прошлый раз** — отдельный persistent state с собственной инвалидацией при переключении проекта; легко рассинхронизируется с реальностью.
- **Отказаться от operational lookback и оставить полный листинг везде** — возврат к 9-секундному переключению проекта, которое и привело к ADR от 28.05.

## Решение

Operational lookback window применяется и в `SessionWatcher.poll_once` — через тот же параметр `lookback_days=config.OPERATIONAL_SESSION_LOOKBACK_DAYS` в вызове `_get_sessions_to_monitor`. Это означает, что poll-тик каждые 2 секунды теперь сканирует только сессии последних N дней, точно так же, как baseline.

Значение константы `OPERATIONAL_SESSION_LOOKBACK_DAYS` сужено с 3 дней до 2. Обоснование: `poll_once` теперь работает на горячем пути каждые 2 секунды — окно должно быть минимально возможным без ущерба для пользовательских сценариев. `UNREAD_BUFFER_TTL_HOURS = 3 часа` (TTL непрочитанных) гораздо строже, чем окно мониторинга, поэтому уменьшение с 3 до 2 дней не влияет на пользовательский сценарий «возврат в проект — получи накопленные за время отсутствия». Сценарий «возврат после выходных, когда последняя активность была пять дней назад» уже не покрывался — все сообщения старше 3 часов отбрасываются по TTL.

Поведенческий инвариант теперь единый: baseline (`reset_state`) и горячий путь (`poll_once`) используют одно и то же окно. Никакая сессия не может попасть в poll, не пройдя через baseline — а значит, cold-start flood больше невозможен по построению.

## Связанные файлы

- **`src/claude_manager/config.py`** — константа `OPERATIONAL_SESSION_LOOKBACK_DAYS` уменьшена с 3 до 2; комментарий переписан: окно теперь применяется и в `reset_state`, и в `poll_once`, и при сборе pending-сообщений; зафиксирован риск cold-start flood в обосновании
- **`src/claude_manager/session_watcher.py`** — в `poll_once` добавлен явный параметр `lookback_days=config.OPERATIONAL_SESSION_LOOKBACK_DAYS` при вызове `_get_sessions_to_monitor`
- **`tests/test_session_watcher.py`** — добавлен тест `TestPollingThroughBackendContract.test_poll_once_requests_operational_lookback_window`: проверяет, что `poll_once` передаёт ровно `OPERATIONAL_SESSION_LOOKBACK_DAYS` в backend; docstring теста явно описывает риск cold-start flood без этого ограничения

## Заменяет

- **`dev/docs/adr/28.05_22.29-session-change-documenter-operational-session-listing-lookback.md`** — частично. Сам паттерн «operational lookback window» (опциональный параметр `lookback_days` в контракте backend'а) сохраняется. Заменяется (а) число — 3 → 2 дня и (б) карта применений — `poll_once` исключён → `poll_once` включён. Изначальное обоснование «poll должен видеть любые сессии» отвергнуто после двух выявленных регрессов: непрерывный `os.walk` по `~/.codex/sessions/` каждые 2 секунды и cold-start flood для сессий за пределами baseline.

## Примечания

После этого ADR в коде остаются места без lookback — они оставлены осознанно:

- `SessionWatcher.resume_session` — ищет одну конкретную сессию по `session_id`, не делает листинг
- `all_projects_monitor` — режим `/all` сохраняет полный обзор всех сессий каждого проекта (другой пользовательский сценарий, другая семантика)
- `daily_session_registry._remove_orphan_entries` — отдельный путь через `session_file_exists_for_project`, не через listing
