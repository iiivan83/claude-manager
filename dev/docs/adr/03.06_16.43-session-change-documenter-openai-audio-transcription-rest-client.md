# ADR: OpenAI audio transcription через прямой REST-клиент

**Дата:** 03-06-2026
**Статус:** принято
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter
**Сущность:** Telegram voice/audio transcription

## Контекст

Бот начал принимать Telegram `voice` и `audio`. Диагностический этап показал, что Telegram присылает voice как `Ogg/Opus`, а реальная проверка через OpenAI API подтвердила: модель `whisper-1` приняла исходный `.ogg` без конвертации.

Следующий шаг требовал встроить распознавание в пользовательский путь: скачать аудио, получить текст и отправить этот текст в активную CLI-сессию как обычное сообщение. При этом проект не имел прямой зависимости от OpenAI SDK, а `telegram_input_handlers.py` уже превысил 300 строк и не должен был дальше забирать сетевую бизнес-логику.

## Решение

Для распознавания аудио добавлен отдельный модуль `openai_transcription.py`. Он вызывает OpenAI Audio Transcriptions API напрямую через стандартную библиотеку Python: `urllib.request`, multipart form-data, endpoint `/v1/audio/transcriptions`.

Ключ берётся из `OPENAI_API_KEY` в локальном `.env`. Модель выбирается через `OPENAI_TRANSCRIPTION_MODEL`, а если переменная не задана, используется `whisper-1`, потому что именно она была проверена на реальном Telegram `Ogg/Opus` файле в этой сессии.

OpenAI SDK не добавляется в зависимости проекта. Для одного endpoint-а прямой REST-клиент уменьшает изменение dependency surface и сохраняет текущую простую модель установки.

## Связанные файлы

- **`src/claude_manager/openai_transcription.py`** — новый REST-клиент OpenAI transcription: multipart upload, разбор JSON-ответа, обработка конфигурационных и HTTP-ошибок
- **`src/claude_manager/telegram_input_handlers.py`** — voice/audio handler теперь скачивает файл, вызывает transcription-клиент и отправляет распознанный текст агенту
- **`.env.example`** — добавлены `OPENAI_API_KEY` и `OPENAI_TRANSCRIPTION_MODEL`
- **`tests/test_openai_transcription.py`** — тесты REST-контракта без реального OpenAI key
- **`tests/test_telegram_audio_input_handlers.py`** — тесты пользовательского voice/audio flow

## Примечания

Официальная документация OpenAI Speech to text перечисляет поддержанные форматы `mp3`, `mp4`, `mpeg`, `mpga`, `m4a`, `wav`, `webm`, но не `ogg`. Реальный API-запрос в этой сессии принял Telegram `.ogg`; это зафиксировано как практический факт, а не как гарантированный контракт документации. Если OpenAI перестанет принимать `ogg`, потребуется отдельная конвертация в поддержанный формат.
