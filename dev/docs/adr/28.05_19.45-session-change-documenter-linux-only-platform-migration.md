# ADR: миграция claude-manager с macOS на Linux как единственную целевую платформу

**Дата:** 28-05-2026
**Статус:** принято
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter
**Сущность:** платформа эксплуатации, инфраструктура запуска, обработчик `/restart`, shell-скрипты, расположение логов, обязательность переменной окружения

## Контекст

До этой работы проект claude-manager был привязан к macOS:

- Самоперезапуск бота (`/restart` в Telegram) шёл через `launchctl kickstart -k 'gui/<uid>/<label>'` — команда менеджера сервисов macOS (launchd). На Linux её нет.
- Основной лог писался в `~/Library/Logs/claude-manager.log` — это стандартное место для macOS-приложений, на Linux такой папки не существует.
- Переменная окружения `PROJECTS_ROOT_DIR` (где бот ищет проекты для команды `/projects`) имела дефолт `/Users/ivan/Desktop/claude-sandbox` — путь macOS, на Linux недействителен.
- Скрипты эксплуатации (`start-claude-manager.sh`, `restart-claude-manager.sh`, `watch_and_restart.sh`) использовали launchd, `launchctl list`, `ps -axo`, file fingerprint polling через `stat -f` — всё это macOS-зависимая инфраструктура.
- В CLAUDE.md и BRD были описаны принципы эксплуатации, привязанные к macOS: изоляция venv от TCC-защищённых директорий (Desktop, Documents, Downloads), retry-обёртка для launchd-сервисов, транзиентная ошибка `EDEADLK` (errno 11) на APFS.

Пользователь перевёл проект на Linux. Старая инфраструктура физически не запускается: `launchctl: command not found`, `~/Library/Logs/` не существует, watcher падает на `stat -f` (на Linux флаг `-f` другой).

## Решение

Заменить всю macOS-инфраструктуру на Linux-аналоги одним связанным изменением:

- **Менеджер сервисов:** systemd user services (`systemctl --user`) вместо launchd. Бот запускается через unit-файл `~/.config/systemd/user/claude-manager.service` с `Restart=always` + `RestartSec=10` — systemd сам перезапускает упавший процесс через 10 секунд, что заменяет роль самописной retry-обёртки `start-claude-manager.sh`.
- **Самоперезапуск через `/restart`:** `asyncio.create_subprocess_exec("bash", "-c", "sleep 2 && systemctl --user restart claude-manager.service", start_new_session=True, stdout=DEVNULL, stderr=DEVNULL)`. Параметр `start_new_session=True` отвязывает subprocess от cgroup сервиса, иначе systemd убил бы команду рестарта вместе с ботом.
- **Расположение логов:** XDG-стандарт `~/.local/state/claude-manager/claude-manager.log` (вместо `~/Library/Logs/`). XDG Base Directory Specification — общепринятый Linux-стандарт для пользовательского состояния.
- **PROJECTS_ROOT_DIR обязательная:** удалён хардкод-дефолт `/Users/ivan/Desktop/claude-sandbox`. Пустое значение поднимает `ConfigError` с понятным сообщением. Это устраняет mine для нового пользователя на Linux, который не задал переменную и получил бы загадочную ошибку «папка не существует».
- **Watcher для разработки:** `inotifywait -m -r -e modify,create,delete --include '.*\.py$'` из пакета `inotify-tools` (Debian/Ubuntu: `sudo apt install inotify-tools`). Заменяет старый polling через `find ... -exec stat -f ... \; | sort` каждые 2 секунды. Inotify подписывается на события ядра напрямую — мгновенно и без нагрузки.
- **Post-flight рестарта:** `systemctl --user is-active --quiet` + `pgrep -f claude_manager`. Двойная проверка нужна потому, что между «сервис активен» и «Python-процесс принимает сообщения» бывает зазор в несколько секунд (особенно при холодном старте). Диагностика при провале — `journalctl --user -u claude-manager.service` + `tail` XDG-лога.
- **`start-claude-manager.sh` удалён.** На Linux systemd сам делает retry через `Restart=always` + `RestartSec=10`. Самописная shell-обёртка с retry-логикой не нужна, и тестовый файл `tests/test_start_claude_manager_script.py` (который проверял конкретные строки удалённой обёртки) тоже удалён.

Принцип «запрета самоперезапуска из собственного дерева процессов» сохранён в неизменном виде (он не привязан к macOS — то же справедливо для systemctl: вызов из subprocess бота убивает собственное дерево cgroup, exit 137). Только формулировка обновлена: «launchd-сервис» → «сервис под supervision», «bash → kickstart» → «bash → systemctl restart».

## Связанные файлы

- **`src/claude_manager/config.py`** — удалена константа `DEFAULT_PROJECTS_ROOT`, функция `_resolve_projects_root` теперь поднимает `ConfigError` при `None`/`""`.
- **`src/claude_manager/main.py`** — `MAIN_LOG_PATH` переехал на `~/.local/state/claude-manager/claude-manager.log`; комментарии про LaunchAgent заменены на systemd/journalctl (продакшен-логика не тронута).
- **`src/claude_manager/bot.py`** — `handle_restart` переписан под `systemctl --user restart`, удалена константа `LAUNCHD_SERVICE_LABEL`, `RESTART_DELAY_BEFORE_KICKSTART_SECONDS` переименована в `RESTART_DELAY_BEFORE_SYSTEMCTL_SECONDS`, убран мёртвый `import os`.
- **`restart-claude-manager.sh`** — полностью переписан: хелперы `check_editable_install`, `service_is_running`, `print_diagnostics_on_failure` (вынесены для тестируемости через source-only mode), основной flow через `systemctl --user restart`.
- **`watch_and_restart.sh`** — полностью переписан под `inotifywait` + дебаунс 1 секунда + `systemctl --user restart`. Проверяет наличие `inotifywait` на старте с подсказкой по установке.
- **`start-claude-manager.sh`** — удалён.
- **`tests/test_bot.py`** — добавлен класс `TestHandleRestart` с двумя тестами (warning message + detach-флаги subprocess).
- **`tests/test_restart_claude_manager_script.py`** — полностью переписан под новые хелперы (6 тестов).
- **`tests/test_start_claude_manager_script.py`** — удалён.
- **`tests/integration/test_claude_cli_contract.py`** — macOS fallback `/Users/ivan/.npm-global/bin/claude` заменён на универсальный `os.path.expanduser("~/.npm-global/bin/claude")`. На Linux 2 ранее skip-нутых контрактных теста теперь реально запускаются и зелёные.
- **`tests/test_config.py`** — удалены тесты на default-поведение `PROJECTS_ROOT_DIR`, добавлены `test_none_raises_error` и `test_empty_raises_error`.
- **`.env.example`** — `PROJECTS_ROOT_DIR` перенесён в раздел «Обязательные настройки», убран macOS-default, примеры путей переведены на `/home/you/...`.
- **`CLAUDE.md`** — удалены 3 раздела про macOS-специфику (TCC-изоляция venv, TCC-изоляция скрипта запуска, retry-обёртка для launchd), переписаны принцип verify-before-and-after и принцип запрета самоперезапуска под systemctl, обновлены «Команды разработки», «Структура проекта», ОС-линия. Удалён буллет про `EDEADLK` (macOS-специфика APFS).
- **`dev/docs/bot-launch-infrastructure.md`** — полностью переписан под Linux: systemd user service, цепочка ExecStart → entry point → Python, /restart через отвязанный subprocess, диагностика через journalctl + XDG-лог.
- **`dev/docs/deployment-guide.md`** — полностью переписан под Linux (Ubuntu/Debian/Fedora): установка inotify-tools, настройка systemd user service, команды `systemctl --user enable/start/stop/status`, проверка через `journalctl`.
- **`dev/docs/brd/brd-user-journeys.md`** — убран launchd-язык в CJM-09 (защита от двойного запуска), заменены устаревшие пути `~/Library/Logs/` на `~/.local/state/claude-manager/`, описание «бот только на маке» заменено на «бот работает под Linux с systemd», поправлены примеры путей в иллюстрации алгоритма `_encode_project_path`.
- **`dev/docs/docs-index.md`** — обновлены описания `deployment-guide.md` и `bot-launch-infrastructure.md` под Linux/systemd.

## Примечания

**Что НЕ изменилось.** Архитектура кода (слоистая модель, поток данных, протокол stream-json, защита через `fcntl.flock`, координация watchers через паузу) полностью сохранена — миграция касалась только инфраструктуры эксплуатации, не бизнес-логики. Все 1012 тестов проходят (единственное падение — унаследованный baseline `test_registry_survives_reload`, существовал до начала работы).

**Совместимость со старыми установками macOS.** Ветка с этой работой (`codex-support-spec-implementation-cycle`) не сохраняет обратную совместимость с macOS-эксплуатацией. На macOS-машине после слияния этой ветки бот не запустится: `launchctl` команд больше нет, ожидаются systemd-команды. Это сознательное решение — у проекта один пользователь, и он целиком переехал на Linux.

**Источники.** Все изменения сделаны в рамках спецификации `dev/docs/specs/28.05_17.25-linux-bot-commands-cleanup-spec.md` и плана `dev/docs/specs/28.05_18.00-linux-bot-commands-cleanup-plan.md`. Финальные коммиты: `6594145`, `396052b`, `ab08cbc`, `fb4846e`, `8ebe4cd`, `70067ac` в ветке `codex-support-spec-implementation-cycle`.

## Заменяет

- **`dev/docs/adr/14.05_16.12-session-change-documenter-restart-postflight-python-process.md`** — старый ADR описывал post-flight для launchd-сервиса с shell-обёрткой (проверка PID обёртки + дочернего Python-процесса). На Linux нет обёртки, post-flight проверяет `systemctl --user is-active` + `pgrep -f claude_manager`. Принцип «двойная проверка не только активности сервиса, но и реального Python-процесса» сохранён — поменялась только реализация под другую систему управления сервисами.
- **Записи `claude-md-updates/03.05_10.58-venv-launchd-tcc-migration.md`** — описывала миграцию venv в `~/.venvs/` (вне TCC-защищённой Desktop-зоны macOS). На Linux нет TCC, venv возвращён в проект `.venv/`. Принципы изоляции venv от TCC больше не применяются — удалены из CLAUDE.md.
