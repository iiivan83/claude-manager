# ADR: sentinel-поле `parsed_message_count` разделяет raw- и parsed-курсоры доставки непрочитанных сообщений (кластер курсоров этапа 1)

- **Дата:** 04-07-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** контракт данных `SessionUnreadState`; новый модуль `jsonl_incomplete_trailing_line_dropper`; политика продвижения курсора в `all_projects_monitor`

## Контекст

Доставка непрочитанных сообщений (pending-путь при возврате в проект и путь `/all`) опирается на два принципиально разных курсора одной сессии, которые до этого решения смешивались:

- **raw-курсор** — `raw_record_count`, число непустых строк JSONL-файла сессии; сообщение считается новым, если его `raw_record_index` (1-based номер непустой строки) больше сохранённого `raw_record_count`.
- **parsed-курсор** — `last_delivered_idx`, 0-based индекс в списке уже распарсенных сообщений `snapshot.messages`.

Лёгкое чтение файла `read_session_file_cursor` историю не парсит и возвращает `messages=[]`, из-за чего `last_delivered_idx` получает значение `-1`. Это корректный sentinel только внутри поллера, но контракт `SessionUnreadState` не имел поля, которое переносило бы «parsed-историю не читали» дальше. При сохранении снапшота в `unread_buffer` sentinel терялся, и потребитель трактовал `-1` как parsed-индекс, срезая `messages[0:]` — всю распарсенную историю сессии в чат.

Полный разбор — в реестре находок `dev/docs/specs/04.07_13.12-project-wide-bug-review-findings.md` (P1-1, P1-2, P2-26) и в детальном плане `dev/docs/specs/04.07_21.27-project-wide-bug-fix-roadmap-and-stage-1-cursor-plan.md`. Три бага одного кластера с общим корнем — смешение raw/parsed курсоров:

- **P1-1** — sentinel `-1` из baseline попадал в `unread_buffer`; при входе в проект в чат вываливалась вся история сессии.
- **P1-2** — в `/all` raw-курсор продвигался мимо придержанного «turn-active» финала, и на следующем поллинге финал уже не был кандидатом (`raw_record_index` больше не превышал курсор) — финалы терялись навсегда.
- **P2-26** — `raw_record_count` считал недописанную последнюю строку файла (CLI дописывает JSONL на лету); дописанная позже запись получала уже «прочитанный» raw-индекс и не доставлялась никогда.

## Решение

Три связанных решения, реализованных строгим TDD (сначала падающий тест, затем минимальный фикс, затем полный прогон):

1. **Sentinel как часть контракта (P1-1).** В `SessionUnreadState` добавлено поле `parsed_message_count: int` с константой-меткой `CURSOR_ONLY_PARSED_MESSAGE_COUNT = -1` (default = сентинел: забытый call-site отдаёт «недоверенное» состояние и уходит в безопасную raw-ветку, а не срезает историю). Поле пробрасывается через все точки, которые строят и переносят снапшот: `coding_agent_session_file_poller`, `unread_buffer.save_snapshot`, `project_manager`, `claude_interaction`, а в `all_projects_monitor` — baseline и `_ensure_unread_snapshot`. Сбор pending (`project_pending_delivery._collect_pending_for_session_file`) ветвится: при сентинеле выбирает сообщения по raw-курсору через хелпер `_messages_after_raw_cursor` (а без raw-индексов не доставляет ничего — лучше тишина, чем лавина), иначе сохраняет прежнюю срез-семантику `messages[last_delivered_idx+1:]`.

2. **Raw-курсор не съедает придержанный финал в `/all` (P1-2).** Хелпер `_raw_cursor_excluding_held_message` останавливает raw-курсор перед последним сообщением активного хода (`min(raw_record_count, held.raw_record_index - 1)`), чтобы на следующем поллинге придержанный финал снова стал кандидатом. Сохранение unread-снапшота (`_ensure_unread_snapshot`) перенесено в начало `_deliver_project_session_delta`, ДО фильтра deliverable, и вызывается безусловно: даже когда все кандидаты придержаны и доставлять нечего, pending-путь проекта получает до-`/all` курсор. Это безопасно, потому что `_ensure_unread_snapshot` внутренним guard'ом не перезаписывает более ранний снапшот.

3. **Читатели не считают недописанную последнюю строку (P2-26).** Новый модуль `jsonl_incomplete_trailing_line_dropper` с единственной функцией `drop_incomplete_trailing_jsonl_line`: убирает последнюю строку файла, если она без завершающего перевода строки И не парсится как JSON (недописанная запись CLI). Строка с `\n` или валидный JSON без `\n` остаётся. Dropper применяется в четырёх точках — snapshot- и cursor-чтения обоих читателей (`claude_code_session_file_reader`, `codex_session_file_reader`) — до подсчёта raw-записей.

## Связанные файлы

- **`src/claude_manager/coding_agent_backend.py`** — добавлены константа `CURSOR_ONLY_PARSED_MESSAGE_COUNT` и поле `parsed_message_count` в dataclass `SessionUnreadState` (ядро контракта).
- **`src/claude_manager/coding_agent_session_file_poller.py`** — удалён локальный дубль константы, импорт из контракта; `get_seen_counts_snapshot` пробрасывает поле.
- **`src/claude_manager/unread_buffer.py`** — параметр `parsed_message_count` в `save_snapshot` и его перенос в конструктор состояния.
- **`src/claude_manager/project_manager.py`** — поле в `snapshot_kwargs` при захвате снапшотов при переключении проекта.
- **`src/claude_manager/claude_interaction.py`** — поле в вызове `save_snapshot`.
- **`src/claude_manager/project_pending_delivery.py`** — хелпер `_messages_after_raw_cursor` и ветвление cursor-only / parsed в `_collect_pending_for_session_file`.
- **`src/claude_manager/all_projects_monitor.py`** — хелпер `_raw_cursor_excluding_held_message`, переписан `_next_state_from_snapshot`, перенос сохранения снапшота в `_deliver_project_session_delta`, sentinel в `_baseline_state_from_snapshot` и `_ensure_unread_snapshot`.
- **`src/claude_manager/jsonl_incomplete_trailing_line_dropper.py`** — новый модуль (создан).
- **`src/claude_manager/claude_code_session_file_reader.py`**, **`src/claude_manager/codex_session_file_reader.py`** — применение dropper в snapshot- и cursor-чтениях.

## Примечания

- **Реализовано через субагентов** (по одному на задачу плана, строго последовательно из-за зависимостей и пересечения файлов), пять fix-коммитов `8a197e6`, `a387f38`, `62e12c8`, `20f2988`, `f9ed81e` плюс docs-коммит `fd1c0c0` с отметками об исправлении в реестре и роудмапе. Полная сюита после этапа 1: 1222 passed, 4 skipped (+19 новых тестов, каждый сначала воспроизводил баг).
- **Два расхождения плана с кодом, пойманные при реализации.** (а) В буквальном коде плана для `_deliver_project_session_delta` ранний `if not candidate_indices: return` стоял ДО сохранения снапшота — в придержанном случае кандидатов нет, и снапшот бы не сохранился (противоречие async-тесту, который сам план требует зелёным). Исправлено переносом `_ensure_unread_snapshot` в самое начало функции. (б) Номера строк для `claude_code_session_file_reader.py` в плане устарели из-за ранее влитого фикса `is_turn_active` по `stop_reason` — точки вставки dropper находились по именам функций, а не по номерам строк.
- **Sentinel как default сознателен.** `parsed_message_count` по умолчанию равен сентинелу `-1`: пропущенный при вызове параметр отдаёт «недоверенное» состояние, и pending-путь уходит в безопасную raw-ветку вместо среза всей истории. У внутреннего типа `_AllMonitorState` в `/all` default остаётся `0` — его состояния всегда строятся из snapshot-чтений, sentinel в них попадает только из baseline-веток.
- **Контракт внешней системы проверен эмпирически** (правило проекта): reproduce-тесты P2-26 реально дописывают оборванную строку в файл сессии и проверяют, что она не попала в `raw_record_count` — сначала падают (баг воспроизведён), затем лечатся.
