# ADR: точечная проверка Codex-сессий по UUIDv7 вместо полного обхода архива

**Дата:** 29-05-2026
**Статус:** принято
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter
**Сущность:** `codex_session_file_listing.session_file_exists_for_project`, быстрый lookup Codex rollout-файла по UUIDv7-времени, fallback на полный scan

## Контекст

Пользователь заметил, что переключение в проект `bloger` занимает намного больше времени, чем переключение в другие проекты. Подозрение было в большом количестве накопленных сессий и возможном решении «инициализировать только последние 30».

Проверка показала другой корень задержки. В `bloger` файл `daily_sessions.json` содержит 1408 записей, из них 246 принадлежат Codex. При переключении проекта `daily_session_registry._remove_orphan_entries` проверяет, существуют ли файлы для записей дневного реестра. Для Claude это дешёвая проверка точного пути, а для Codex старый метод `session_file_exists_for_project` каждый раз делал полный обход `~/.codex/sessions`.

На фактических данных это выглядело так:

- Полная проверка записей-сирот для `bloger` до изменения занимала 7.39 секунды за один проход.
- Та же проверка для `claude_manager` занимала 0.10 секунды.
- Та же проверка для `budgets` занимала 0.63 секунды.

Дополнительный фактор: при переключении проекта реестр загружается дважды — через `session_manager.reset_state`, а затем через `daily_session_registry.reset_state`. Поэтому дорогая Codex-проверка умножалась на два и становилась заметной пользователю.

## Решение

Для точечной проверки существования Codex-сессии выбран быстрый путь по контракту имени rollout-файла.

Codex хранит rollout-файлы в структуре:

```text
~/.codex/sessions/YYYY/MM/DD/rollout-YYYY-MM-DDTHH-MM-SS-<session_id>.jsonl
```

Практические Codex `session_id` имеют формат UUIDv7: первые 48 бит содержат timestamp создания. Поэтому `session_file_exists_for_project` теперь:

1. Декодирует время из UUIDv7.
2. Строит вероятные точные пути rollout-файла в локальном времени и UTC.
3. Если файл найден, читает только его `session_meta` и проверяет `payload.cwd`.
4. Если точный путь не найден, использует старый полный scan как fallback.

Это не удаляет сессии, не меняет дневную нумерацию и не вводит лимит «последние 30». Ограничение по количеству сессий не понадобилось, потому что реальная проблема была не в размере истории как таковой, а в повторном полном обходе глобального Codex-архива для каждой записи реестра.

## Связанные файлы

- **`src/claude_manager/codex_session_file_listing.py`** — добавлен быстрый поиск вероятного точного rollout-файла по UUIDv7-времени перед полным scan.
- **`tests/test_codex_session_file_listing.py`** — добавлен регрессионный тест, который падает, если быстрый путь снова начнёт вызывать полный `_list_all_rollout_files_blocking`.

## Заменяет

- **`dev/docs/adr/28.05_22.29-session-change-documenter-operational-session-listing-lookback.md`** — частично дополняет. Тот ADR ограничил operational listing для watcher и pending-сбора, но прямо зафиксировал, что `daily_session_registry._remove_orphan_entries` остаётся отдельным путём через `session_file_exists_for_project`. Текущий ADR закрывает именно этот оставшийся путь.

## Примечания

Проверенный эффект на локальных данных:

- `bloger`: 7.39 секунды до изменения, 0.63 секунды после изменения.
- `claude_manager`: 0.10 секунды до изменения, 0.01 секунды после изменения.
- `budgets`: 0.63 секунды до изменения, 0.05 секунды после изменения.

Оставшийся технический долг:

- `daily_session_registry.load_registry` всё ещё вызывается дважды при переключении проекта. После быстрого Codex lookup это уже не главный тормоз, но архитектурно дублирование стоит убрать отдельной задачей.
- `src/claude_manager/codex_session_file_listing.py` вырос до 494 строк. Это ниже stop-порога 500, но выше warning-порога 300; следующий рост лучше начинать с выделения lookup/indexing helpers в отдельный модуль.
