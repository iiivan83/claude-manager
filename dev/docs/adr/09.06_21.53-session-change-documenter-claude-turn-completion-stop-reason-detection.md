# ADR: завершённость turn'а внешних Claude Code-сессий определяется по stop_reason, а не только по result-event

**Дата:** 09-06-2026
**Статус:** принято
**Пайплайн:** session-change-documenter
**Агент:** session-change-documenter (standalone)
**Скилл:** session-change-documenter
**Сущность:** `claude_code_session_file_reader._compute_is_turn_active_from_parsed_records`, `_read_cursor_count_last_record_and_turn_active`, `_assistant_record_keeps_turn_active`

## Контекст

В режиме тишины (silence mode — глобальный флаг, при котором бот доставляет только финальные ответы Claude с `is_final=True`, а промежуточные подавляются) часть финальных ответов Claude не доходила до пользователя. Codex-ответы при этом приходили нормально. Точный триггер — Telegram-реплей на сообщение бота: обычное сообщение идёт прямым путём (`claude_interaction.send_to_claude_and_respond`) и доставляется, а реплей уходит в `reply_route_handler`, который доставку финала делегирует фоновому наблюдателю (`session_watcher` — опрашивает `.jsonl`-файлы сессий каждые 2 секунды).

Корень — критерий `is_turn_active` (флаг «Claude ещё работает над turn'ом») для Claude-бэкенда, введённый ADR `29.05_01.47-session-change-documenter-watcher-turn-active-result-anchor.md`. Тот ADR сделал reverse-проход по записям: `result` → turn закрыт; любой из `BUSY_EVENT_TYPES = {"assistant", "progress", "queue-operation"}` → turn активен.

Проблема в контракте внешних сессий Claude Code (запускаются не ботом, лежат в `~/.claude/projects/...`): они завершают ответ записью `assistant` со `stop_reason=end_turn` и **никогда не пишут запись `result`** (record `result` пишет только сам бот через свой stream-json-процесс). Reverse-проход упирается в `assistant`, по старому правилу безусловно возвращает `is_turn_active=True`, и turn навсегда считается активным. Дальше цепочка `coding_agent_session_file_poller` придерживает последнее сообщение (`hold_final_message`) и принудительно ставит `is_final=False`, а silence-фильтр в `telegram_response_delivery` выбрасывает всё не-финальное. Итог — финальный ответ не доставляется.

Эмпирическая проверка контракта (требование CLAUDE.md по внешним системам): 6 последних `.jsonl`-файлов сессий Claude в проекте `bloger` — во всех 0 записей `result`; несколько заканчиваются `assistant` со `stop_reason=end_turn`; самый свежий реально в середине turn'а со `stop_reason=tool_use`, для него активность должна остаться `True`.

## Решение

Сделать критерий завершённости turn'а зависящим от `stop_reason` последней значимой `assistant`-записи, а не от самого факта типа `assistant`. Reverse-проход сохраняется, но ветка `assistant` теперь решает по `stop_reason`:

- `stop_reason in {"tool_use", "pause_turn"}` → turn активен. Claude остановился, чтобы вызвать инструмент, или поставил длинный turn на паузу — продолжит тот же turn.
- `stop_reason is None` → turn активен. Сообщение ещё стримится, причина остановки не записана.
- любой явный терминальный `stop_reason` (`end_turn`, `stop_sequence`, `max_tokens`, `refusal`) → turn закрыт.

Полный критерий reverse-прохода:

- `result` → turn закрыт (для процессов бота; у внешних сессий не встречается — работает ветка `assistant`).
- `assistant` → решает `stop_reason` через `_assistant_record_keeps_turn_active`.
- `progress` / `queue-operation` (новая константа `MID_TURN_STREAM_EVENT_TYPES`) → turn активен.
- ничего значимого до начала файла → `is_turn_active=False` (безопасный дефолт).

Это закрывает два дефекта сразу: завершённый внешний turn (`end_turn` без `result`) теперь читается как закрытый, и служебные записи после финала (`last-prompt`, `ai-title`, `mode`, `file-history-snapshot`) больше не открывают turn заново, потому что проход останавливается на `assistant` и смотрит его `stop_reason`, а не на тип последней строки файла.

Фикс внесён в **оба** ридера — snapshot reader (полный парсинг, `_compute_is_turn_active_from_parsed_records`) и cursor reader (лёгкий проход для baseline в режиме `/all`, `_read_cursor_count_last_record_and_turn_active`) — иначе баг выжил бы в половине сценариев (watcher poll и reset_state идут через cursor reader). Общий предикат `_assistant_record_keeps_turn_active` гарантирует одинаковую логику в обоих местах.

`reply_route_handler` не менялся: после фикса ридера watcher корректно доставляет финал по пути реплея, поэтому прямая доставка `result.text` в `_send_routed_text_in_background` не вводилась — это исключило риск двойной доставки.

Альтернативы, рассмотренные и отвергнутые:

- **Полумера `Silence off`.** Промежуточные станут видны, но одиночный финал через watcher всё равно придерживается из-за той же ошибки чтения. Симптом, не корень.
- **Доставлять `result.text` напрямую в пути реплея.** Чинит только путь реплея, оставляет watcher сломанным для других сценариев (фоновые сессии, `/all`), плюс риск двойной доставки с watcher'ом.
- **Whitelist `{tool_use, pause_turn}` с трактовкой `None` как терминального.** Отвергнут: существующие контрактные тесты и реальные стримящиеся записи требуют, чтобы `stop_reason=None` означал «активен». Терминальность — только по явному терминальному reason.

## Связанные файлы

- **`src/claude_manager/claude_code_session_file_reader.py`** — добавлены константы `TURN_CONTINUING_STOP_REASONS` и `MID_TURN_STREAM_EVENT_TYPES`, добавлен предикат `_assistant_record_keeps_turn_active`, переписаны ветки решения в `_compute_is_turn_active_from_parsed_records` и `_read_cursor_count_last_record_and_turn_active`. `BUSY_EVENT_TYPES` оставлен без изменений — это отдельный контракт бэкенда (`claude_code_backend.event_types_meaning_cli_is_busy`), не критерий завершённости turn'а. Магические строки `"result"`/`"assistant"` заменены на существующие константы `EVENT_TYPE_RESULT`/`EVENT_TYPE_ASSISTANT`. Файл вырос до 545 строк — превышение проектного порога 500 строк зафиксировано осознанно (+33 к прошлым 512), разбиение модуля по ответственности отложено на отдельную итерацию.
- **`tests/test_claude_code_backend.py`** — добавлены 9 регрессионных тестов (snapshot и cursor reader): терминальные `stop_reason` (`end_turn`, `stop_sequence`, `max_tokens`) закрывают turn; служебные записи после `end_turn` не открывают turn заново; `tool_use` оставляет turn активным.

## Заменяет

Частично заменяет ADR `29.05_01.47-session-change-documenter-watcher-turn-active-result-anchor.md`. Сохраняется его reverse-проход и якорь `result` → turn закрыт. Заменяется правило «любой `assistant` → turn активен»: теперь `assistant` решает по `stop_reason`. Причина замены — то правило было выведено из сценария «бот-процесс с tool_use в середине», но не покрывало контракт внешних Claude Code-сессий, которые завершаются `assistant end_turn` без `result`.
