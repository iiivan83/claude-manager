# ADR: in-memory маршруты для Telegram reply на сообщения бота

- **Дата:** 31-05-2026
- **Статус:** принято
- **Пайплайн:** session-change-documenter
- **Агент:** session-change-documenter (standalone)
- **Скилл:** session-change-documenter
- **Сущность:** Telegram reply-routing, transport-слой, контракт маршрута ответа

## Контекст

Claude Manager показывает в Telegram ответы из разных сессий, проектов и CLI-бэкендов. До этого пользователь мог перейти в нужную сессию по ссылке `/N` или `/PsN`, но прямой Telegram reply на старое сообщение бота не был адресным: текст уходил в текущую активную сессию или блокировался в `/all`.

Это создавало практическую проблему. Иван видел старый ответ агента, хотел быстро уточнить его, нажимал «Ответить» в Telegram, но бот не знал, к какой проектной сессии относится выбранное сообщение. Автоматически угадывать цель по текущему проекту нельзя: сообщение могло прийти из другого проекта, другого backend-а или из режима `/all`.

## Решение

Принят отдельный in-memory реестр маршрутов reply-сообщений.

Когда бот успешно отправляет маршрутизируемое сообщение в Telegram, delivery-слой сохраняет связь:

- `chat_id`
- Telegram `message_id` отправленного ботом сообщения
- `project_path`
- `session_id`
- `backend`
- номер сессии
- номер и имя проекта, если они нужны для ссылки из `/all`

Если пользователь отвечает текстом на это сообщение, входящий handler ищет route по `chat_id + reply_to_message.message_id` и отправляет текст напрямую в сохранённые `project_path + session_id + backend`. Текущий проект и активная сессия пользователя не переключаются.

Реестр намеренно хранится только в памяти процесса. После перезапуска старые Telegram-сообщения не маршрутизируются: бот показывает понятную ошибку и просит перейти по ссылке на нужную сессию вручную. Это ограничение принято для v1, чтобы не вводить новый персистентный формат без отдельной необходимости.

Файлы, фото, документы и альбомы в адресном reply не поддержаны в v1. Handler отклоняет их до скачивания из Telegram, чтобы не передать вложение в неправильную сессию.

## Связанные файлы

- **`src/claude_manager/reply_route_registry.py`** — создан in-memory реестр route target-ов
- **`src/claude_manager/reply_route_handler.py`** — создан обработчик входящих text reply и unsupported-вложений
- **`src/claude_manager/telegram_response_delivery.py`** — регистрирует route только после успешной отправки сообщения в Telegram
- **`src/claude_manager/telegram_input_handlers.py`** — проверяет routed reply до обычных `/all`-ограничений
- **`src/claude_manager/telegram_sender.py`** — возвращает отправленное Telegram-сообщение, чтобы delivery-слой получил `message_id`
- **`src/claude_manager/claude_interaction.py`** — прокидывает исходный проект и anchor-данные в delivery-слой
- **`src/claude_manager/project_pending_delivery.py`** — сохраняет route target для pending-доставки после возврата в проект
- **`tests/test_reply_route_registry.py`** — проверяет запись, чтение и очистку route registry
- **`tests/test_reply_route_handler.py`** — проверяет успешные сценарии, `/all`, cross-project ссылки и busy-состояния
- **`tests/test_reply_route_handler_error_results.py`** — проверяет понятные ошибки backend вместо маскировки под «сессия недоступна»
- **`tests/test_telegram_response_delivery_reply_routes.py`** — проверяет регистрацию routes после отправки и для длинных chunk-ов
- **`dev/docs/specs/31.05_04.56-reply-routing-v1-spec.md`** — исходная спецификация поведения
- **`dev/docs/specs/31.05_05.02-reply-routing-v1-implementation-plan.md`** — план реализации

## Примечания

Это решение не заменяет `reply_anchor_registry`. У реестров разные направления:

- `reply_anchor_registry` отвечает, к какому сообщению Ивана привязать будущий ответ бота
- `reply_route_registry` отвечает, куда отправить новый текст Ивана, если он ответил на сообщение бота

Проверки после merge в `main`:

- полный non-E2E suite до merge: 1131 passed, 5 skipped, 3 warnings
- полный non-E2E suite после merge в `main`: 1131 passed, 5 skipped, 3 warnings

Живой manual smoke через настоящий Telegram не запускался в этой среде. Его нужно выполнить отдельно, потому что он требует реального Telegram-аккаунта и работающего бота.
