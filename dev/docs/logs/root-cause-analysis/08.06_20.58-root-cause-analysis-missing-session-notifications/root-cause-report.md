# RCA: missing Telegram notifications from active bloger sessions

## Короткий вывод

Проблема не похожа на сбой Telegram-доставки. Основная цепочка такая:

1. Включён режим тишины: файл `~/.claude-manager-silence-mode` содержит `{"enabled": true}`.
2. В режиме тишины бот отправляет только финальные сообщения.
3. Рабочие Claude-сессии проекта `bloger` остаются для бота в состоянии `active=true`.
4. Из-за `active=true` watcher считает все их сообщения нефинальными.
5. Поэтому режим тишины подавляет эти сообщения, и пользователь не видит уведомления.

Корневая причина глубже: код чтения Claude JSONL считает ход закрытым только по записи `result`, но внешние/рабочие Claude Code-сессии в `~/.claude/projects/...` часто завершают ответ записью `assistant` со `stop_reason=end_turn` и не добавляют `result`. Бот пропускает служебные записи редактора, видит последний значимый `assistant` и ошибочно считает, что Claude всё ещё пишет.

## Что проверено

**Проект и режим мониторинга**

- В 20:57 бот переключился с `/home/ivan/claude-sandbox/bloger` на `/home/ivan/claude-sandbox/claude_manager`.
- Обычный `session_watcher` после переключения смотрит только текущий проект.
- За вечер 18:00-20:59 в логе нет включения global all-project mode для чата `REDACTED_OWNER_TELEGRAM_ID`, то есть `/all` не был активен.
- При переключении в 20:57 удалено 7 просроченных unread snapshots; TTL буфера равен 3 часа.

**Состояние bloger-сессий**

Проверены Claude-сессии `bloger` #28, #29, #31, #32, #35, #36, #37, #38, #39, #40, #41, #42, #43.

- Все 13 из 13 прочитались как `active=true`.
- У 11 из 13 последняя запись содержит видимый текст, который похож на завершённый ответ.
- Примеры последних текстов:
  - #28: “Собрал фактуру по Газпрому...”
  - #31: “Готово. Собрал свежую фактуру...”
  - #41: “Остановился. Ты отклонил чтение шаблонов...”
  - #42: “Остановился. Жду, что скажешь...”

**Codex-сессии**

Проверены Codex-сессии `bloger` #27, #33, #34.

- Все 3 прочитались как `active=false`.
- Значит, конкретный дефект `active=true` не объясняет их недоставку.
- В логе они зарегистрированы, ошибок отправки watcher-сообщений рядом не найдено.
- Успешная отправка watcher-сообщений сейчас явно не логируется, поэтому по логам нельзя доказать, были ли эти сообщения реально отправлены или пользователь их не заметил в потоке.

## Техническая цепочка

**1. Фильтр режима тишины**

В `src/claude_manager/telegram_response_delivery.py` watcher-сообщение не отправляется, если `is_final=False` и включён silence mode:

```python
if not is_final and silence_mode_registry.is_enabled():
    return
```

Это ожидаемое поведение режима тишины: промежуточные сообщения не показываются.

**2. Watcher делает активные ответы нефинальными**

В `src/claude_manager/coding_agent_session_file_poller.py`:

```python
hold_final_message = snapshot.is_turn_active or previous.handler_owns_final_delivery
...
if hold_final_message and candidate_indices:
    last_idx = len(messages) - 1
    candidate_indices = [index for index in candidate_indices if index < last_idx]
...
is_final = not hold_final_message and position == len(deliverable) - 1
```

Если `snapshot.is_turn_active=True`, последний ответ удерживается, а остальные отправляются как нефинальные. При включённом silence mode эти нефинальные сообщения тоже отбрасываются.

**3. Claude reader ошибочно оставляет рабочие сессии активными**

В `src/claude_manager/claude_code_session_file_reader.py` активность определяется обратным проходом по JSONL:

```python
if record_type == "result":
    return False
if record_type in BUSY_EVENT_TYPES:
    return True
```

Это хорошо работает для потока Claude CLI, где после завершения приходит `result`. Но в проверенных рабочих файлах Claude Code в `bloger` финальный ответ выглядит иначе:

- последняя содержательная запись часто `assistant`;
- `message.stop_reason` равен `end_turn`;
- после неё могут идти служебные записи `last-prompt`, `ai-title`, `mode`;
- записи `result` нет.

Из-за этого код классифицирует завершённый ответ как активный.

## Почему это проявилось именно в работающих сессиях

Работающие сессии в `bloger`, судя по файлам, созданы/ведутся как внешние Claude Code-сессии, а не как полностью управляемый ботом CLI-turn с гарантированным `result` в stdout. Для таких файлов текущий критерий “есть `result` после последнего assistant” слишком строгий.

Пока режим тишины выключен, пользователь мог бы видеть хотя бы промежуточные watcher-сообщения. При включённом режиме тишины всё зависит от корректного `is_final`. Так как `is_final` не наступает, Telegram молчит.

## Дополнительный фактор: текущий проект и `/all`

Даже после исправления `active=true` обычный watcher не обязан смотреть все проекты сразу.

- `session_watcher` смотрит `config.WORKING_DIR`, то есть текущий выбранный проект.
- После перехода из `bloger` в `claude_manager` он перестал постоянно мониторить `bloger`.
- Для постоянного мониторинга всех проектов нужен режим `/all`.
- При возврате в проект pending delivery использует сохранённые unread snapshots, но они живут 3 часа. В 20:57 часть snapshots уже была удалена как просроченная.

Это не главная причина молчания внутри активного `bloger`, но объясняет, почему после переключения проекта старые сообщения не приехали “задним числом”.

## Рекомендации

**CODE: исправить определение завершённого Claude-turn для внешних Claude Code JSONL**

Добавить правило: если при обратном проходе найден `assistant` со `message.stop_reason == "end_turn"` и нет более позднего признака незавершённого tool-use/tool-result цикла, считать turn закрытым (`is_turn_active=False`). Это должно применяться и в полном snapshot-reader, и в cursor-reader.

Обязательно покрыть тестами:

- `assistant stop_reason=end_turn` без `result` должен давать `is_turn_active=False`;
- `assistant stop_reason=tool_use` + последующий `user tool_result` без `result` должен оставаться `is_turn_active=True`;
- служебные записи `last-prompt`, `ai-title`, `mode` после `end_turn` не должны снова делать turn активным;
- watcher при silence mode должен доставлять такой ответ как финальный.

**CODE: добавить диагностическое логирование успешной/подавленной watcher-доставки**

Сейчас по логам видно регистрацию сессии и ошибки, но не видно успешной отправки и не видно, что сообщение подавлено silence mode. Из-за этого Codex-кейсы #27/#33/#34 нельзя закрыть полностью.

Минимально полезные события:

- watcher попытался доставить сообщение;
- сообщение подавлено silence mode, потому что `is_final=False`;
- сообщение отправлено в Telegram;
- целевой chat_id выбран как owner или broadcast.

**UX/DOC: объяснить пользователю ограничение режима тишины**

Команда `/silence_on` сейчас говорит “буду присылать только финальные ответы”. Это правда, но при ошибочной классификации финала выглядит как “бот молчит”. После фикса можно дополнительно подсветить в документации: если рабочие внешние сессии не присылают ответы, проверить `/silence_off` и `/all`.

**CODE/UX: подумать о TTL pending snapshots**

Текущий TTL равен 3 часам. Для долгих рабочих сессий это может быть мало. Если ожидается сценарий “ушёл в другой проект на полдня, потом вернулся и хочу получить всё непрочитанное”, TTL нужно увеличить или сделать настраиваемым командой/конфигом.

## Быстрые обходные действия

- Для будущих сообщений из всех проектов включить `/all`.
- Если нужно увидеть промежуточные сообщения прямо сейчас, выключить режим тишины через `/silence_off`.
- Для старых Claude-сессий, которые уже застряли как `active=true`, надёжный путь — исправить reader; простое переключение проекта может не доставить финал, потому что старый код всё равно считает его нефинальным.
