# Анализ первопричины: retry loop создаёт десятки новых сессий вместо повтора в существующей

**Дата анализа:** 13-04-2026
**Проект:** Claude Manager — Telegram-бот для управления Claude Code с телефона
**Статус:** Первопричина найдена

## Симптом

Пользователь открыл бот и увидел, что дневной счётчик сессий дошёл до #140, хотя столько сессий он не создавал. При просмотре через `/132`, `/133`, `/134`, `/135` — во всех одно и то же сообщение, а ответов от Claude нет. За весь день **ни одна** из 116 сессий не получила результат.

Скриншот: сессии #132-#135 показывают одинаковый текст «В общем, беда, смотри как работает клод код, я пишу ему сообщение, он начинает работать...» — пользователь описывал проблему Claude, и это сообщение «размножилось» в 10 сессий.

## Цепочка причин

Обнаружены **три отклонения**, которые вместе создают каскадный эффект. Каждое по отдельности не вызывает катастрофы — но вместе они превращают одну ошибку Claude API в десятки призрачных сессий.

### Отклонение 1: rate limit не распознаётся как ответ (триггер)

1. **Claude не отвечает на сообщения пользователя** → потому что →
2. **Пользователь исчерпал лимит запросов к Claude API** — Claude Code CLI получает запрос, но вместо нормальной обработки возвращает текст: "You've hit your limit · resets 8pm (Asia/Yekaterinburg)" → потому что →
3. **Это сообщение приходит как обычный ответ ассистента (`type: "assistant"`), а НЕ как событие результата (`type: "result"`)** — бот ждёт событие `result` в функции `_process_events()` (`process_manager.py`:226-229), а получает `assistant` → потому что →
4. **Claude Code CLI не оборачивает rate limit в событие `result`** — он возвращает текст ограничения как текст ассистента и завершает процесс. Событие `result` (финальный маркер, который бот считает признаком успешного ответа) не отправляется.
5. **В боте нет проверки на rate limit** — `_process_events()` не анализирует текст ответа ассистента на предмет ключевых слов вроде "hit your limit", не пытается передать его пользователю как информационное сообщение.

**Результат:** Флаг `got_result` остаётся `False`, `_process_events()` возвращает `SendResult(is_error=True)`, запускается retry loop.

### Отклонение 2: retry loop создаёт новые сессии вместо повтора в существующей (усилитель)

1. **Каждый retry создаёт совершенно новую сессию Claude** → потому что →
2. **Функция `_restart_process()` (`process_manager.py`:348-352) проверяет, начинается ли `session_id` с `_new_`** — если да, запускает Claude Code CLI **без флага `--resume`**, что значит «создай новую сессию с нуля»:

```python
is_temp_session = session_id.startswith(TEMP_SESSION_PREFIX)
cli_session_id = None if is_temp_session else session_id
claude_process = await start_process(cli_session_id)  # None → без --resume
```

3. **Retry loop (`_retry_loop()`, строка 297-332) получает `session_id` как параметр и НИКОГДА не обновляет его** — передаётся исходный `_new_xxx`, и все 10 попыток используют один и тот же временный ID:

```python
async def _retry_loop(session_id, text, ...):  # session_id = "_new_xxx"
    for attempt in range(1, MAX_RETRIES + 1):
        result = await _execute_single_retry(
            session_id, text, ...  # Всегда "_new_xxx" → каждый раз без --resume
        )
```

4. **`session_id_callback` вызывается внутри `_process_events()` при каждом retry** — обновляет `session_manager._bindings` на новый реальный UUID. Но `session_id` внутри `_retry_loop` остаётся `_new_xxx`. Callback обновляет привязки «снаружи» (в session_manager и session_watcher), но сам retry loop об этом не знает.

**Результат:** Каждая из 10 попыток retry запускает Claude CLI с нуля, Claude создаёт новый `.jsonl` файл с новым UUID на диске. За один пользовательский запрос появляется до 10 новых сессий.

### Отклонение 3: retry бессмысленен для rate limit (расточитель)

1. **Retry loop ждёт 60 секунд между попытками и делает до 10 повторов** — это 10 минут бесполезного ожидания → потому что →
2. **Rate limit сбрасывается в 20:00, а не через 60 секунд** — ни одна из 10 попыток не может увенчаться успехом → потому что →
3. **Бот не различает «Claude упал» и «Claude ответил, но не так, как мы ожидали»** — оба случая одинаково попадают в retry. Для падения retry осмыслен (перезапустил → заработало). Для rate limit retry — бесполезная трата ресурсов, создающая мусорные сессии.

## Корневая причина

Корень проблемы — **в retry loop нет механизма обновления `session_id` с временного (`_new_xxx`) на реальный UUID.** Это архитектурный пробел в `process_manager.py`.

Цепочка привела к каскаду:

- `_retry_loop()` (строка 297) принимает `session_id` как параметр и передаёт его в каждый `_execute_single_retry()` без изменений
- `_execute_single_retry()` (строка 251) вызывает `_restart_process(session_id)` с тем же `_new_xxx`
- `_restart_process()` (строка 348) видит `_new_` → запускает Claude без `--resume`
- Claude создаёт новую сессию с новым UUID

При этом `_process_events()` (строка 191) внутри каждого retry корректно обнаруживает новый UUID и вызывает `session_id_callback`, но:
- Callback обновляет `session_manager._bindings` и `session_watcher` — внешние модули
- Callback НЕ обновляет `session_id` внутри `_retry_loop` — его некуда обновить, переменная локальная

**Дополнительный пробел**: бот не умеет отличать rate limit от падения Claude. Оба случая считаются ошибкой и запускают retry. Для rate limit retry не просто бесполезен — он создаёт мусорные сессии.

## Почему проблема не была предотвращена

### Уровень архитектуры (CLAUDE.md)

CLAUDE.md подробно описывает механизм `session_id_callback` (замена `_new_xxx` на реальный UUID при первом событии), но **не описывает, что должно происходить при retry**. Раздел «Единство идентификатора сессии (temp → real)» говорит только про штатный путь — первое сообщение в новой сессии. Сценарий «callback сработал, но retry loop продолжает использовать старый ID» не рассматривается.

### Уровень кода

- `_retry_loop()` не принимает mutable-контейнер для session_id и не возвращает обновлённый ID между попытками
- `_execute_single_retry()` возвращает `SendResult` с `session_id`, но `_retry_loop()` не использует это значение для обновления своей переменной
- Нет проверки текста ассистента на rate limit перед запуском retry

### Уровень тестирования

В тестах `test_process_manager.py` есть тесты retry loop, но они не покрывают сценарий «retry с временным session_id, где каждая попытка порождает новую сессию». Мок Claude CLI не создаёт реальных файлов на диске, поэтому побочный эффект (множество `.jsonl` файлов) невидим в юнит-тестах.

## Рекомендации по исправлению

### 1. Обновлять session_id в retry loop после получения реального UUID

**Файл:** `src/claude_manager/process_manager.py`, функции `_retry_loop()` и `_execute_single_retry()`

**Что сделать:** После каждого retry, если `result.session_id` отличается от текущего `session_id`, вызвать `update_session_id(old, new)` и обновить локальную переменную. Тогда следующий retry вызовет `_restart_process("real_uuid")`, который увидит НЕ `_new_` → передаст `--resume` → Claude продолжит существующую сессию.

**Зачем:** Устраняет размножение сессий при retry. Вместо 10 новых сессий все retry будут работать с одной.

### 2. Распознавать rate limit и не запускать retry

**Файл:** `src/claude_manager/process_manager.py`, функция `_process_events()`

**Что сделать:** Если процесс завершился без события `result`, но есть ответ ассистента с текстом, содержащим "hit your limit" или "resets" — считать это не ошибкой, а информационным ответом. Вернуть `SendResult(is_error=False, text=rate_limit_text)`, чтобы бот переслал текст пользователю.

**Зачем:** Пользователь сразу увидит «лимит исчерпан, сброс в 20:00» вместо 10 минут тишины с последующим «ошибка Claude».

### 3. Разделить «Claude упал» и «Claude ответил без result-события»

**Файл:** `src/claude_manager/process_manager.py`, функция `_process_events()`

**Что сделать:** Добавить третий исход: если `got_result = False`, но есть текст от ассистента — передать этот текст как ответ (возможно, с пометкой). Retry запускать только если ассистент вообще ничего не ответил (процесс упал).

**Зачем:** Обобщённое решение. Rate limit — частный случай. Claude Code CLI может возвращать и другие сообщения без `result`-события (например, предупреждения). Пользователь должен их видеть, а не ждать retry.

## Верификация решений

### Рекомендация 1: обновлять session_id в retry loop

**Стратегия:** Точечное исправление в `_retry_loop()`. После получения `result.session_id != session_id` — вызвать `update_session_id()` и присвоить `session_id = result.session_id`. Архитектурно безопасно — `update_session_id()` уже используется в `send_message()` (строка 465) для аналогичной задачи.

**Вердикт:** ОДОБРЕНО

### Рекомендация 2: распознавать rate limit

**Стратегия:** В `_process_events()` проверять текст `assistant`-события на паттерн rate limit. Не хардкодить конкретную строку — Claude Code CLI может менять формулировку. Лучше проверять: «процесс завершился без `result`, но есть текст ассистента → передать текст пользователю».

**Вердикт:** ОДОБРЕНО — но реализовать как общее решение (рекомендация 3), а не только для rate limit.

### Рекомендация 3: разделить «упал» и «ответил без result»

**Стратегия:** В `_process_events()` собирать текст из `assistant`-событий в переменную `assistant_text`. Если `got_result = False` и `assistant_text` непустой — вернуть `SendResult(is_error=False, text=assistant_text)`. Retry запускать только если `got_result = False` и `assistant_text` пустой.

**Вердикт:** ОДОБРЕНО — это обобщение рекомендации 2.
