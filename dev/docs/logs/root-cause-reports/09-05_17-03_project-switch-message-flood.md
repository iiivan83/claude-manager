# Анализ первопричины: массовая рассылка старых сообщений при переключении проекта

**Дата анализа:** 09-05-2026
**Проект:** Claude Manager
**Статус:** Первопричина найдена

## Симптом

При переключении в проект `budgets` бот отправил много старых сообщений из прошлых Claude-сессий. На скриншоте видно заголовок вида `/98`, а не `#98`. Это важно: `/98` использует `session_watcher` (фоновый наблюдатель за файлами сессий, который читает JSONL-файлы Claude и пересылает новые сообщения), а `#98` обычно ставится прямой доставкой ответа из обработчика команды.

То же повторилось при переходе в `reviews-analyzer`: проект вообще не имеет Claude-папки сессий, но в него попали сообщения из старых бюджетных файлов. Значит проблема не в самих данных проекта `reviews-analyzer`, а в состоянии фонового наблюдателя во время переключения.

## Цепочка причин

1. **Пользователь увидел поток старых сообщений** -> потому что ->
2. **`session_watcher` решил, что часть старых Claude-файлов еще не доставлялась**. `session_watcher` - это фоновый цикл, который каждые несколько секунд читает файлы сессий и сравнивает их с внутренним счётчиком уже доставленных сообщений (cursor, то есть позиция "дочитано до сюда") -> потому что ->
3. **При сбросе watcher после переключения на `budgets` были инициализированы только 483 Claude-сессии, хотя в папке Claude проекта лежит 746 JSONL-файлов**. Разница ровно 263 файла -> потому что ->
4. **Эти 263 сессии ранее были помечены как временно отсутствующие в проекте `reviews-analyzer` и попали под backoff-фильтр**. Backoff-фильтр - это задержка повторной проверки отсутствующего файла, чтобы не спамить файловую систему и логи. Во время `reset_state()` этот фильтр применился к реальным файлам уже нового проекта и не дал поставить им счётчик "уже прочитано" -> потому что ->
5. **КОРНЕВАЯ ПРИЧИНА: сброс состояния watcher использует общий метод `_get_sessions_to_monitor()`, который предназначен для обычного фонового опроса и применяет missing-file backoff. Для сброса после смены проекта это неверно: reset должен видеть все реальные файлы нового проекта и не зависеть от списка отсутствующих файлов старого проекта.**

Отдельное продолжение той же проблемы:

6. **При переходе из `budgets` в `reviews-analyzer` старый обход `budgets` еще продолжал отправлять сообщения** -> потому что ->
7. **`pause_all()` ставит глобальную паузу только для будущих циклов, но не отменяет уже запущенный `poll_once()`**. `poll_once()` - один проход watcher по списку файлов. Он уже взял бюджетные файлы и продолжал их доставлять, когда текущий проект и дневной реестр уже стали `reviews-analyzer`.

## Корневая причина

Главная ошибка находится в `src/claude_manager/session_watcher.py` (модуль фонового наблюдения за файлами сессий).

В `SessionWatcher.reset_state()` на строках 487-509 сброс вызывает `_get_sessions_to_monitor()`. Эта функция на строках 261-290 делает не только список реальных файлов, но и применяет `_should_check_missing_file_now()` к каждому session id. Это нормально для обычного мониторинга, но опасно для сброса при смене проекта.

Что произошло вживую:

- В 16:49:30 лог сказал, что у `reviews-analyzer` нет Claude-папки сессий.
- В 16:49:33 при переключении на `budgets` watcher сбросил состояние только для 483 Claude-сессий.
- На диске в Claude-папке `budgets` сейчас 746 JSONL-файлов. Разница: 263.
- В 16:50:42 при переключении обратно в `reviews-analyzer` бот удалил ровно 263 записи-сироты. Запись-сирота - это запись в дневном реестре, для которой в текущем проекте нет файла сессии.

Эти числа сходятся слишком точно, чтобы быть случайностью: 263 бюджетные сессии не получили состояние "уже прочитано", потому что были недавно помечены как отсутствующие в `reviews-analyzer`. После сброса `_missing_files` очищается, но уже поздно: эти сессии не попали в `_states`, то есть в память watcher со счётчиками. На следующем опросе код в `_check_session()` на строке 299 создаёт пустое состояние, где `last_delivered_idx = -1`, и затем строки 328-342 выбирают все старые assistant-сообщения как кандидаты на отправку.

Вторая ошибка находится в том же модуле: уже начавшийся `poll_once()` не защищён от смены проекта. Когда пользователь переключился из `budgets` в `reviews-analyzer`, часть бюджетной рассылки еще выполнялась. Поэтому `daily_session_registry.register_session()` на строках 344-349 начал записывать бюджетные session id уже в дневной реестр нового проекта. `daily_session_registry` - это модуль, который присваивает сессиям дневные номера вроде `#98`.

Это подтверждается логами:

- В 16:50:43 `reviews-analyzer` переключён успешно, Claude-сессий у него 0.
- С 16:51:00 по 16:51:19 всё равно регистрируются Claude-сессии `#267` - `#344`.
- В эти же секунды `file_sender` жалуется на файлы из `/Users/ivan/Desktop/claude-sandbox/budgets/...`, хотя текущий проект уже `reviews-analyzer`.

## Почему проблема не была предотвращена

В проекте уже есть защита от похожей гонки: `project_manager` (модуль переключения проектов) вызывает глобальную паузу watcher перед сменой `WORKING_DIR`. `WORKING_DIR` - настройка текущего проекта, от неё зависит, какие файлы сессий читать.

Но защита покрывает только новый старт опроса. Она не ждёт завершения уже идущего опроса и не отменяет его. Также она не учитывает, что данные `_missing_files` из старого проекта нельзя применять во время reset нового проекта.

### Проверенные скиллы

- `root-cause-analysis` (скилл поиска первопричины) - использован. Он требует идти от симптома к системной причине и проверять альтернативы.
- `universal-bug-fixer` (скилл исправления багов) - не запускался, потому что запрос был на анализ, а не на внесение кода.
- `feature-pipeline` (скилл полного цикла доработки) - связан исторически: функциональность переключения проекта и буфер непрочитанных реализовывались отдельными пайплайнами, но текущий дефект находится в стыке watcher reset, missing-file backoff и in-flight polling.

### Проверенные документы

- `dev/docs/specs/realised/project_manager_spec.md` (спецификация переключения проектов) - описывает паузу watcher при смене проекта, но не фиксирует требование "reset не должен использовать missing-file backoff старого проекта".
- `dev/docs/specs/realised/unread_buffer_spec.md` (спецификация буфера непрочитанных сообщений) - подтверждает, что обычная доставка непрочитанных должна идти только из сохранённых cursor-ов. В инциденте при переходе в `budgets` было `непрочитанных=0`, значит поток шёл не через этот буфер.
- `dev/docs/brd/brd-user-journeys.md` (бизнес-сценарии пользователя) - описывает, что watcher нужно ставить на паузу при переключении, но не описывает защиту от уже запущенного обхода.
- `dev/docs/logs/root-cause-reports/13-04_10-44_cross-project-session-registration.md` (старый RCA по похожей гонке между проектами) - проблема класса "сессии одного проекта попадают в реестр другого" уже встречалась, но текущая ветка причины другая: не только смена `WORKING_DIR`, а stale missing-file backoff плюс незавершённый poll.

## Рекомендации по исправлению

1. В `src/claude_manager/session_watcher.py` разделить обычный список мониторинга и список для reset.
   Для `reset_state()` нужен прямой список реальных файлов из `backend.list_all_session_files_for_project(config.WORKING_DIR)` без `_should_check_missing_file_now()` и без добавления записей из дневного реестра. Это гарантирует, что все реальные файлы нового проекта получат `last_delivered_idx = len(messages) - 1`.

2. В `SessionWatcher.reset_state()` очищать `_missing_files` до построения нового состояния или вообще не использовать его в reset.
   Это простой защитный слой: информация "файл отсутствовал в прошлом проекте" не должна влиять на новый проект.

3. Добавить поколение проекта (project generation) в `session_watcher`.
   Поколение проекта - это счётчик версии текущего проекта. При каждом переключении он увеличивается. `poll_once()` должен запомнить поколение в начале и перед каждой доставкой проверять, что поколение не изменилось. Если проект уже другой, старый обход должен остановиться без отправки.

4. Перенести `pause_all()` в `project_manager._perform_switch()` перед `_capture_unread_snapshots()` или добавить общий lock между переключением и `poll_once()`.
   `_capture_unread_snapshots()` - это сбор позиций "что уже прочитано" перед уходом из проекта. Пока он идёт, новый watcher-опрос не должен параллельно менять те же данные.

5. В `SessionWatcher._deliver_message()` ограничить широкую рассылку неизвестных сессий.
   Сейчас, если владелец сессии не найден, код отправляет сообщение всем разрешённым пользователям. Для исторических терминальных сессий это делает единичную ошибку очень шумной. Нужно как минимум не рассылать неизвестные старые сессии, если их файл старше момента активации проекта.

## Рекомендации по предотвращению

1. [ARCHITECTURE] Добавить в архитектурный документ принцип: "reset состояния после смены проекта не использует кэши, backoff-состояния и registry-записи предыдущего проекта; он строится только из фактических файлов нового проекта".
   Место: новый `architecture.md`, раздел "Project switching and watcher state". Сейчас в корне проекта такого файла нет, поэтому его стоит создать коротким документом с этим принципом.

2. [DOC] Обновить `dev/docs/specs/realised/project_manager_spec.md`: явно описать, что `session_watcher.reset_state()` обязан обходить все реальные файлы проекта без missing-file backoff, а уже запущенный watcher-опрос обязан прекращать доставку после смены проекта.

3. [CODE] Добавить unit-тест в `tests/test_session_watcher.py`: если сессия была в `_missing_files`, а потом появилась как реальный файл нового проекта, `reset_state()` всё равно должен поставить ей cursor на конец файла и не доставлять старые сообщения.

4. [CODE] Добавить integration-тест в `tests/integration/test_project_switching.py`: переключение в проект с большим числом старых Claude JSONL-файлов не вызывает callback watcher, если новых сообщений после переключения не было.

5. [CODE] Добавить тест на in-flight poll: если проект сменился во время долгой доставки, старый `poll_once()` прекращает отправку и не пишет session id старого проекта в новый `daily_sessions.json`.

## Верификация выводов

- Перепроверен скриншот: заголовок `/98` указывает на watcher-доставку, а не на обычную доставку pending-сообщений.
- Перепроверены логи вокруг 16:49-16:51: при переходе в `budgets` pending был 0, но сразу после этого начались регистрации watcher-сессий.
- Проверена Claude-папка `budgets`: в ней 746 JSONL-файлов. Лог reset показал только 483 Claude-сессии. Разница 263 совпала с числом записей-сирот, удалённых при следующем переключении.
- Проверена альтернативная гипотеза "это обычный unread buffer": она не подходит, потому что лог показал `непрочитанных=0`, а формат сообщения на скриншоте watcher-овский.
- Проверена альтернативная гипотеза "это проблема проекта reviews-analyzer": она не подходит, потому что у `reviews-analyzer` нет Claude-папки сессий, а предупреждения `file_sender` ссылались на пути из `budgets`.
- Проверена альтернативная гипотеза "session_id нестабилен между reset и poll": реальные файлы бюджетного проекта имеют session id, совпадающий с именем файла; это не объясняет массовую рассылку.

## Верификация решений

- **Рекомендация:** reset watcher должен читать все реальные файлы без missing-file backoff.
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** Это устраняет точную причину, по которой 263 файла не получили cursor.
  - **Корректировка:** Без изменений.

- **Рекомендация:** остановка in-flight poll при смене проекта через project generation.
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** Это закрывает вторую часть инцидента, когда бюджетные сообщения попали в `reviews-analyzer`.
  - **Корректировка:** Без изменений.

- **Рекомендация:** полностью отключить watcher при переключении проектов.
  - **Вердикт:** ОТКЛОНЕНО
  - **Обоснование:** Watcher нужен для фоновых терминальных сессий и доставки живого прогресса. Проблема не в самом watcher, а в неправильном reset и отсутствии проверки смены проекта внутри долгого обхода.
  - **Корректировка:** Заменено на project generation и правильный reset.
