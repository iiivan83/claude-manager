# Анализ первопричины: дублирование номеров сессий (#20 и #21 — одна и та же сессия)

**Дата анализа:** 12-04-2026
**Проект:** Claude Manager (Telegram-бот для управления Claude Code)
**Статус:** Первопричина найдена

## Симптом

Пользователь поставил задачу в сессии #20 через Telegram-бот. Агенты и задачи начали выполняться в сессии #21. При этом обе сессии — это одна и та же сессия Claude Code (один файл на диске, один UUID `c5d4c2f4-5a7d-4961-a658-684bbb4a36eb`).

Проблема системная: за 3 дня в дневном реестре `daily_sessions.json` (файл, в котором бот хранит соответствие «номер сессии → UUID») обнаружено **19 случаев дублирования** — один и тот же UUID зарегистрирован под двумя разными номерами:
- **10 апреля:** 4 случая (#19/#30, #44/#45, #50/#52, #55/#56)
- **11 апреля:** 13 случаев
- **12 апреля:** 2 случая (#20/#21, #23/#24)

## Цепочка причин

1. **Пользователь видит задачу в #20, а работу в #21** → потому что →
2. **Watcher (фоновый наблюдатель за сессиями) зарегистрировал тот же UUID как новую сессию #21** → потому что →
3. **Watcher увидел файл сессии на диске раньше, чем бот успел обновить временный ID на реальный** → потому что →
4. **КОРНЕВАЯ ПРИЧИНА: механизм паузы watcher использует временный session_id (`_new_XXXX`), а файл на диске создаётся с реальным UUID — watcher не знает, что это одна и та же сессия**

## Корневая причина

Два процесса в боте — обработчик сообщений и watcher — работают с сессиями параллельно. Между ними есть механизм координации (пауза/возобновление), но он привязан к **идентификатору сессии**. Проблема в том, что у новых сессий два разных идентификатора в разное время: временный (`_new_abc123`) и настоящий UUID (`c5d4c2f4-...`), а координация работает только по одному из них.

### Хронология гонки (race condition)

Вот что происходит по шагам:

1. **Пользователь жмёт /new** — `session_manager.create_new_session()` (файл `session_manager.py:180`) генерирует временный ID вида `_new_abc123` и регистрирует его в `daily_session_registry` как #20

2. **Пользователь отправляет сообщение** — `_send_to_claude_and_respond()` (файл `bot.py:449`) ставит watcher на паузу для `_new_abc123`:
   ```
   session_watcher.pause_session("_new_abc123")  # Watcher не будет трогать _new_abc123
   ```

3. **Бот запускает Claude CLI** — `process_manager.create_process()` (файл `process_manager.py:357`) видит, что ID временный, и запускает Claude **без** `--resume` (новая сессия). Claude CLI создаёт файл `c5d4c2f4-...jsonl` на диске

4. **[ОКНО ГОНКИ]** — Watcher каждые 2 секунды сканирует файлы на диске. Он находит `c5d4c2f4-...jsonl`. Проверяет: эта сессия на паузе? Нет — на паузе стоит `_new_abc123`, а не `c5d4c2f4-...`. Watcher вызывает `daily_session_registry.register_session("c5d4c2f4-...")`. Реестр ищет этот UUID в записях сегодняшнего дня, находит только `_new_abc123` под #20 — UUID не совпадает. Реестр регистрирует UUID как **новую сессию #21**

5. **Claude заканчивает работу** — бот получает результат с реальным UUID и обновляет запись #20: `_new_abc123` → `c5d4c2f4-...`. Переносит паузу на реальный UUID. Но уже поздно — #21 уже создан

6. **Итог:** в реестре #20 и #21 указывают на один UUID: `c5d4c2f4-5a7d-4961-a658-684bbb4a36eb`

### Ключевые файлы

- `session_watcher.py:127-203` (`_check_session`) — проверяет `_paused_sessions` по session_id, но на паузе стоит только временный ID
- `session_watcher.py:228-230` (`pause_session`) — принимает session_id как строку, не знает про ассоциацию temp↔real
- `daily_session_registry.py:92-108` (`register_session`) — честно ищет UUID среди существующих записей, но temp ID и real UUID — это разные строки
- `bot.py:449-495` (`_send_to_claude_and_respond`) — ставит паузу на temp ID в начале, обновляет на real UUID в конце, а между этими моментами — окно гонки
- `process_manager.py:190-235` (`_process_events`) — узнаёт реальный session_id из событий Claude, но уведомляет внешние модули только после завершения всего потока

### Почему это корневая причина, а не промежуточное звено

Проблема не в конкретной строке кода — каждый модуль по отдельности работает правильно. Проблема в **отсутствии концепции «эти два ID — одна сессия»** на уровне архитектуры. Механизм координации бота и watcher опирается на точное совпадение строки session_id, но у новых сессий идентификатор меняется в процессе — и ни один модуль не знает об этом до завершения обработки.

## Почему проблема не была предотвращена

### Проверенные скиллы

- `feature-pipeline` (пайплайн доработки функционала) — не содержит проверок на гонки между watcher и обработчиком. Это архитектурная проблема, а не пропуск в фиче
- `universal-bug-fixer` (пайплайн исправления багов) — мог бы поймать, если бы пользователь сформулировал симптом раньше
- `review-code` (ревью кода) — 4-проходное ревью фокусируется на качестве кода отдельных модулей, а не на межмодульных гонках в асинхронных сценариях

### Проверенные документы

- `CLAUDE.md` — описывает координацию watcher и обработчика через `pause_session/resume_session`, но не упоминает проблему с временными ID. Раздел «Состояние в памяти» описывает защиту через `asyncio.Lock`, но гонка здесь не на уровне одного словаря, а на уровне двух независимых процессов
- `dev/docs/brd/brd-user-journeys.md` — CJM-07 описывает мониторинг сессий, но не рассматривает сценарий, когда watcher видит сессию раньше, чем обработчик обновляет привязку

## Рекомендации по исправлению

### 1. Раннее уведомление о смене session_id [CODE]

Сейчас `_process_events()` в `process_manager.py` узнаёт реальный session_id из первых событий Claude, но сообщает об этом вызывающему коду только после завершения всего потока (в `SendResult`). Нужно добавить callback `on_session_id_changed`, который вызывается сразу при обнаружении нового ID — чтобы бот мог мгновенно перенести паузу и обновить реестр.

Это закроет окно гонки: watcher увидит файл на диске, но к этому моменту паузу уже перенесут на реальный UUID.

**Файлы:** `process_manager.py` (добавить параметр `session_id_callback` в `send_message` и `_process_events`), `bot.py` (передать callback, который обновляет `session_watcher`, `session_manager`, `daily_session_registry`)

### 2. Очистка дубликатов в реестре [CODE]

Добавить функцию `_remove_duplicate_entries()` в `daily_session_registry.py`, которая при загрузке реестра находит UUID, зарегистрированные под несколькими номерами, и оставляет только запись с наименьшим номером (первую регистрацию). Вызывать после `_remove_phantom_entries()` в `load_registry()`.

**Файлы:** `daily_session_registry.py`

## Рекомендации по предотвращению

### 1. [ARCHITECTURE] Добавить в CLAUDE.md принцип «единство идентификатора сессии»

Сейчас CLAUDE.md описывает, что session_id может измениться в любом событии, и что нужно обновлять ключи. Но не описывает **проблему окна**: между появлением файла на диске и обновлением привязок есть зазор, в который watcher может зарегистрировать дубликат.

Принцип: «Обновление session_id (temp → real) должно происходить **мгновенно** при первом обнаружении нового ID в потоке событий, а не по завершении обработки. Все модули, зависящие от session_id (watcher, registry, session_manager), должны быть уведомлены в одном атомарном шаге.»

**Место:** в раздел «Важные детали для разработки», после пункта про координацию watcher и handle_message

### 2. [DOC] Обновить BRD: сценарий «новая сессия с отложенным ID»

Добавить в CJM-07 (мониторинг сессий) описание граничного случая: watcher может увидеть файл сессии на диске раньше, чем бот обновит привязку. Это знание поможет при будущих ревью.

**Файл:** `dev/docs/brd/brd-user-journeys.md`

## Верификация выводов

- **Перечитал `session_watcher.py`** — подтвердилось: `pause_session()` принимает строку session_id, а `_check_session()` сравнивает с `_paused_sessions` по точному совпадению. Никакого механизма «этот temp ID связан с этим real UUID» нет
- **Перечитал `daily_session_registry.py`** — подтвердилось: `register_session()` ищет session_id среди значений текущего дня. Если temp ID ещё не заменён на real UUID, поиск не найдёт совпадение и зарегистрирует дубликат
- **Проверил данные:** 19 случаев дублирования за 3 дня — каждый раз один UUID под двумя номерами. Паттерн стабильный. Все дубликаты — для идущих подряд номеров (N и N+1), что подтверждает гонку в пределах одного цикла обработки

### Альтернативные объяснения

- **Альтернатива 1: Claude Agent tool создаёт дочерние сессии.** Отвергнута: если бы это были разные сессии (родительская и дочерняя), они имели бы разные UUID. В нашем случае UUID одинаковый — это одна сессия, зарегистрированная дважды
- **Альтернатива 2: Claude CLI возвращает чужой session_id.** Отвергнута: данные показывают, что дубликаты всегда идут парами (N, N+1) — это характерный паттерн для гонки «бот создаёт → watcher регистрирует», а не для ошибки протокола CLI
