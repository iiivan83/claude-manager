# Анализ первопричины: медленное переключение проектов

**Дата анализа:** 30-05-2026
**Проект:** Claude Manager
**Статус:** Первопричина найдена

## Симптом

Иван видит, что переключение между проектами в Telegram-боте занимает около 10 секунд, иногда больше. Это путь `/projects` -> `/pN`: пользователь выбирает проект, а бот должен быстро сменить рабочую папку и ответить в Telegram.

Важные термины:

- **`session_watcher`** — фоновый наблюдатель, который смотрит файлы сессий Claude/Codex и доставляет новые ответы в Telegram.
- **`pending`** — непрочитанные ответы, которые накопились в проекте, пока пользователь был в другом проекте.
- **`lookback`** — окно последних дней, которое ограничивает поиск файлов сессий. Сейчас это 2 дня: сегодня и вчера.
- **Codex rollout-файл** — JSONL-файл одной Codex-сессии в `~/.codex/sessions/YYYY/MM/DD/`.

Логи живого бота подтверждают, что задержка реальная и находится внутри бота, до ответа пользователю:

- 30-05 18:16:32 -> 18:16:39: 7 секунд, `bloger -> youtube-downloader`
- 30-05 18:19:22 -> 18:19:32: 10 секунд, `youtube-downloader -> bloger`
- 30-05 18:21:54 -> 18:22:11: 17 секунд, `bloger -> claude_manager`
- 30-05 18:23:36 -> 18:23:45: 9 секунд, `claude_manager -> bloger`
- 30-05 18:24:27 -> 18:24:34: 7 секунд, `bloger -> claude_manager`

## Цепочка причин

1. **Пользователь ждёт 7-17 секунд после `/pN`** -> потому что ->
2. **`project_manager.switch_project()` не отвечает сразу после смены проекта**. `project_manager.py` — модуль, который координирует переключение рабочей папки, сброс состояния и доставку непрочитанных. Он ждёт два тяжёлых участка: `session_watcher.reset_state()` и `_collect_pending_messages()` -> потому что ->
3. **Оба участка заново ищут Codex-сессии проекта через `list_all_session_files_for_project(..., lookback_days=2)`** -> потому что ->
4. **Codex хранит сессии не по проектам, а в общей папке `~/.codex/sessions/`**. Чтобы найти сессии одного проекта, код вынужден открыть каждый rollout-файл за lookback-окно, прочитать `session_meta` и проверить `payload.cwd` -> потому что ->
5. **За сегодня и вчера сейчас 3906 Codex rollout-файлов, и каждый project switch сканирует их несколько раз** -> потому что ->
6. **КОРНЕВАЯ ПРИЧИНА: горячий путь переключения проекта построен как повторный глобальный скан Codex-истории, а не как дешёвый per-project индекс. Предыдущие фиксы уменьшили область скана до 2 дней и распараллелили чтение метаданных, но не убрали саму O(N) работу по всем недавним Codex-файлам.**

## Корневая причина

Главный разрыв между архитектурой и реальностью находится на стыке трёх модулей:

- `src/claude_manager/project_manager.py` — переключает проект и перед ответом собирает pending.
- `src/claude_manager/session_watcher.py` / `src/claude_manager/coding_agent_session_file_poller.py` — пересобирают стартовые курсоры watcher-а после смены проекта.
- `src/claude_manager/codex_session_file_listing.py` — ищет Codex rollout-файлы, принадлежащие конкретному проекту.

В `project_manager._perform_switch()` проект меняется и затем сбрасываются state-модули. Внутри сброса `session_watcher.reset_state()` проходит по backend-ам последовательно: сначала Claude, потом Codex. Claude быстрый, потому что его файлы уже лежат по папке проекта. Codex медленный, потому что его файлы лежат глобально.

В `codex_session_file_listing.list_all_session_file_infos_for_project()` при `lookback_days=2` код сначала находит rollout-файлы за сегодня и вчера, а потом для каждого файла вызывает `_read_project_meta_pair()`. Эта функция открывает файл и читает `session_meta`, чтобы понять, относится ли файл к нужному проекту.

Практический эффект по локальным замерам:

- Всего Codex rollout-файлов: 14063.
- В lookback-окне 2 дня: 1977 файлов за 30-05 и 1929 файлов за 29-05, всего 3906.
- Codex listing для проекта, даже когда результат 0 файлов, занимает 2.3-3.3 секунды.
- Для `bloger`: Codex listing нашёл 60 файлов за 2.601 секунды.
- Для `claude_manager`: Codex listing нашёл 5 файлов за 3.109 секунды.
- Для проектов без Codex-сессий: listing всё равно занимает 2.3-3.3 секунды, потому что проверяются те же 3906 файлов.

Затем `project_manager._finalize_successful_switch()` перед ответом пользователю вызывает `_collect_pending_messages()`. Это второй тяжёлый проход. Он снова вызывает backend listing, а потом для сессий со snapshot-ом читает session snapshot, чтобы понять, появились ли новые сообщения.

Локальный замер показал:

- `bloger`: seeded snapshots для 56 Claude + 60 Codex сессий за 3.354 секунды; затем `collect_pending` вернул 0 pending за 3.916 секунды.
- `claude_manager`: seeded snapshots для 28 Claude + 5 Codex сессий за 3.059 секунды; затем `collect_pending` вернул 0 pending за 3.398 секунды.
- `youtube-downloader`: seeded snapshots для 0 Claude + 1 Codex сессии за 2.849 секунды; затем `collect_pending` вернул 0 pending за 2.964 секунды.

То есть даже когда непрочитанных нет, бот платит несколько секунд только за доказательство, что их нет.

## Почему предыдущие фиксы не решили проблему

28-05 и 29-05 были правильные, но недостаточные оптимизации:

1. **Параллельная инициализация курсоров** ускорила чтение уже найденных файлов, но не убрала поиск Codex-файлов среди глобальной истории.
2. **Operational lookback window** заменил полный scan по всей `~/.codex/sessions/` на scan последних 2 дней. Это помогло, пока файлов в окне было мало. Сейчас только за 2 дня уже 3906 файлов.
3. **Параллельное чтение meta-записей Codex** ускорило обработку, но каждый вызов всё равно открывает тысячи файлов.
4. Предыдущий вывод “`_collect_pending_messages` не bottleneck” оказался неверным для текущего состояния: код сохраняет snapshots для всех watcher-сессий, а не только для 1-2 активных. Поэтому при возврате в проект pending-сбор читает много сессий, даже когда итоговый pending равен 0.

Глубокая причина не в том, что “мало параллельности”. Проблема в том, что один и тот же дорогой global scan запускается повторно в горячем пользовательском пути.

## Почему проблема не была предотвращена

### Проверенные скиллы

- `root-cause-analysis` (скилл поиска первопричины) — использован. Он требует идти от симптома к системной причине и проверять альтернативы.
- `systematic-debugging` (скилл системной отладки) — использован как процесс: сначала воспроизведение и измерения, потом выводы.
- Проектных `.claude/skills/*/SKILL.md` в этом проекте нет. Поэтому локальный проектный скилл не мог предотвратить проблему.

### Проверенные документы

- `CLAUDE.md` (главные правила проекта) — описывает атомарность переключения и watcher, но не задаёт performance budget для `/pN`.
- `dev/docs/specs/realised/project_manager_spec.md` (спецификация переключения проектов) — описывает порядок reset и pending delivery, но не фиксирует, что backend listing в горячем пути должен быть O(число сессий проекта), а не O(все недавние Codex-файлы).
- `dev/docs/specs/realised/session_watcher_spec.md` (спецификация watcher-а) — сейчас устарела относительно split-фасада и Codex-backend модели; полезна как исторический контракт, но не описывает цену Codex listing.
- `dev/docs/adr/28.05_16.25-session-change-documenter-parallel-project-switch-init.md` — правильно нашёл последовательное чтение snapshots, но оставил Codex listing как отдельный следующий слой проблемы.
- `dev/docs/adr/28.05_22.29-session-change-documenter-operational-session-listing-lookback.md` — правильно ввёл lookback, но предположил, что 2-3 дня достаточно ограничат работу. Сейчас объём за 2 дня уже большой.
- `dev/docs/adr/29.05_00.25-session-change-documenter-poll-once-operational-lookback.md` — правильно синхронизировал baseline и poll, но тем самым сделал lookback scan постоянной горячей операцией.
- `dev/docs/adr/29.05_00.25-session-change-documenter-codex-session-listing-concurrent-meta-reads.md` — правильно распараллелил чтение meta, но не изменил алгоритмическую стоимость поиска.
- `dev/docs/logs/root-cause-reports/09-05_17-03_project-switch-message-flood.md` — связан по области project switching, но про другой симптом: массовую рассылку старых сообщений.

## Рекомендации по исправлению

1. [CODE] Добавить общий operational index для Codex-сессий за lookback-окно.
   Файлы: `codex_session_file_listing.py`, `codex_backend.py`, возможно новый модуль рядом с ними.
   Смысл: один раз за короткий TTL построить `project_dir -> list[SessionFileInfo]` из 3906 rollout-файлов и переиспользовать в `reset_state`, `poll_once` и pending-сборе. Сейчас каждый проект и каждый участок заново читает те же meta-записи.

2. [CODE] Сделать pending-сбор cursor/mtime-first.
   Файлы: `project_manager.py`, `unread_buffer.py`, backend-контракт.
   Смысл: хранить в snapshot не только `raw_record_count` и `last_delivered_idx`, но и `last_modified_at`. Если файл не менялся с момента ухода из проекта, не читать полный snapshot. Если менялся — сначала прочитать дешёвый cursor, и только при реальном росте читать полный snapshot для доставки текста.

3. [CODE] Не блокировать подтверждение переключения на pending-сборе.
   Файл: `project_manager.py` и обработчик `/pN` в `bot.py`.
   Смысл: пользователь должен быстро получить “переключено”, а pending можно дослать отдельной фоновой задачей. Это особенно важно, потому что pending — полезное дополнение, а не обязательная часть смены `WORKING_DIR`.

4. [CODE] Убрать повторный Codex listing внутри одного переключения.
   Файлы: `project_manager.py`, `session_watcher.py`.
   Смысл: если reset уже получил список файлов target-проекта, pending-сбор должен переиспользовать его или работать через общий индекс. Сейчас два независимых участка платят одну и ту же цену.

5. [CODE] Добавить bounded parallelism в `_collect_pending_for_backend`, но только как вторичный шаг.
   Файл: `project_manager.py`.
   Смысл: если после cursor/mtime-filter остались изменённые файлы, читать их параллельно с лимитом 16. Просто распараллелить нынешний полный pending-сбор недостаточно: это ускорит симптом, но оставит повторный global scan.

## Верификация выводов

Перепроверены альтернативные версии:

- **Telegram медленно доставляет ответ** — менее вероятно. Логи показывают, что задержка происходит до строки `Переключение проекта выполнено`, то есть до формирования результата для Telegram.
- **Проблема только в `session_watcher.reset_state`** — неполная версия. В логах после reset до `project_manager` остаётся ещё несколько секунд. Локальный замер `collect_pending` подтвердил отдельные 2.96-3.92 секунды даже при 0 pending.
- **Проблема только в большом проекте `bloger`** — неверно. Даже проекты с 0 Codex-сессий платят 2.3-3.3 секунды за Codex listing, потому что проверяются глобальные rollout-файлы за lookback-окно.
- **Достаточно увеличить параллельность** — недостаточно. Узкое место алгоритмическое: повторный scan 3906 файлов. Больше параллельности снизит стеночное время, но усилит нагрузку на thread pool и диск.

Корневая причина подтверждена: задержка объясняет все наблюдаемые симптомы, совпадает с live-логами и воспроизводится локальными замерами.

## Верификация решений

- **Рекомендация:** общий operational index/cache для Codex-сессий.
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** Убирает повторный global scan, решает корневую причину, а не только отдельный медленный цикл.
  - **Корректировка:** Индекс должен иметь короткий TTL или инвалидацию по дате/mtime, чтобы watcher не пропускал новые сессии.

- **Рекомендация:** cursor/mtime-first pending.
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** Убирает чтение полных JSONL-файлов, когда файл не менялся и pending заведомо нет.
  - **Корректировка:** Нужно расширить snapshot state аккуратно, без потери backend-aware ключа `(session_id, backend)`.

- **Рекомендация:** отправлять подтверждение переключения до pending.
  - **Вердикт:** ОДОБРЕНО С ЗАМЕЧАНИЯМИ
  - **Обоснование:** Даёт быстрый UX даже если pending тяжёлый, но требует аккуратной обработки ошибок фоновой задачи.
  - **Корректировка:** Сначала сделать быстрый индекс/cursor-filter, затем решать, нужен ли async pending. Иначе фоновая задача может продолжить молотить диск.

- **Рекомендация:** просто поднять semaphore выше 16.
  - **Вердикт:** ОТКЛОНЕНО
  - **Обоснование:** Это увеличит конкуренцию за диск и thread pool, но не устранит повторное чтение тысяч файлов.
  - **Корректировка:** Заменено на индекс и устранение повторных вызовов.
