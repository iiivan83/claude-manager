# Анализ первопричины: бот ломается при подключении второго пользователя

**Дата анализа:** 11-04-2026
**Проект:** Claude Manager (Telegram-бот для управления Claude Code с телефона)
**Статус:** Первопричина найдена

## Симптом

Когда к боту подключается второй аккаунт (Telethon-аккаунт для E2E тестирования), бот начинает вести себя непредсказуемо:

- **«Вылетает из проекта»** — текущий проект сбрасывается, привязки к сессиям исчезают
- **Сообщения приходят по 10 раз** — один ответ от Claude дублируется многократно
- **Параллельные цепочки повторных попыток** — бот запускает 2-3 retry-цикла одновременно для одной сессии
- **37 000+ фантомных предупреждений в логах** — watcher (фоновый наблюдатель за сессиями) каждые 2 секунды пытается прочитать несуществующие файлы

При работе с одним аккаунтом всё стабильно. Проблема проявляется только когда в `ALLOWED_USER_IDS` (белый список разрешённых Telegram-аккаунтов в `.env`) прописаны два разных пользователя.

## Цепочка причин

1. **Бот дублирует сообщения и теряет привязки к сессиям** → потому что →

2. **Watcher (фоновый наблюдатель, `session_watcher.py:174`) рассылает каждое сообщение ВСЕМ пользователям из белого списка** — цикл `for chat_id in config.ALLOWED_USER_IDS` безусловно отправляет каждый новый ответ Claude обоим аккаунтам. Плюс к этому `bot.py` отправляет прямой ответ тому, кто задал вопрос. Плюс при resume (снятии паузы) watcher перечитывает историю и шлёт повторно. Итого: одно сообщение доставляется 3-5 раз каждому пользователю → потому что →

3. **Все модули хранят состояние в глобальных словарях без изоляции по пользователям** — `config.WORKING_DIR` (текущий проект, `config.py:37`) — одна строка на весь процесс. `_paused_sessions` (набор приостановленных сессий, `session_watcher.py:45`) — привязан к `session_id`, а не к паре `(chat_id, session_id)`. `_busy_flags` (флаги занятости процессов, `process_manager.py:98`) — тоже по `session_id` без учёта кто именно отправил сообщение. `reset_state()` (сброс состояния при переключении проекта, `project_manager.py:175`) стирает привязки ВСЕХ пользователей → потому что →

4. **Архитектура спроектирована как строго однопользовательская** — BRD (документ бизнес-требований, `dev/docs/brd/brd-user-journeys.md`, строка 7) явно говорит: «Бот рассчитан на одного пользователя — управляет Claude Code на личном компьютере с телефона». `ALLOWED_USER_IDS` задуман как «несколько ID для одного человека с разных устройств» (строка 32 BRD). Все 11 пользовательских сценариев описывают одного пользователя. Все спецификации модулей, весь код, все тесты следуют этому допущению → потому что →

5. **КОРНЕВАЯ ПРИЧИНА: `ALLOWED_USER_IDS` принимает несколько ID без какой-либо защиты от реального multi-user** — поле конфигурации `ALLOWED_USER_IDS` в `.env` — это `set[int]`, который парсится из строки через запятую (`config.py:112`). Ничто не мешает добавить туда ID другого человека. E2E тестовая инфраструктура делает именно это: прописывает Telethon-аккаунт как второй ID в белом списке. В результате возникает многопользовательский сценарий, который архитектура явно не поддерживает. Нет ни программной защиты (runtime guard), ни предупреждения в документации, ни проверки в тестах.

## Корневая причина

Корень проблемы — **отсутствие архитектурного инварианта «один пользователь»**. BRD декларирует однопользовательность, но это знание осталось только в тексте документа. Оно не было:

- **Закреплено в коде** — нет runtime-проверки количества пользователей, нет предупреждения в логах, нет guard clause (проверки-барьера) при старте бота
- **Отражено в CLAUDE.md** — главный документ проекта не фиксирует ограничение «один пользователь» как архитектурный принцип
- **Учтено в E2E инфраструктуре** — Telethon-аккаунт добавляется в белый список наравне с основным пользователем, без анализа последствий

Конкретные точки в коде, через которые проблема проявляется:

- **`session_watcher.py:174-190`** — `_notify_chats()` (функция отправки уведомлений) итерирует по ВСЕМ `config.ALLOWED_USER_IDS` и шлёт каждому каждое сообщение. Понятия «владелец сессии» не существует
- **`config.py:37`** — `WORKING_DIR` (рабочая директория текущего проекта) — единственный глобальный скаляр. Переключение проекта через `/pN` одним пользователем мгновенно меняет проект для всех
- **`project_manager.py:175-179`** — `_reset_all_state_modules()` при переключении проекта стирает `session_manager._bindings` (привязки chat_id → session_id), `daily_session_registry._registry` (дневная нумерация), `session_watcher._seen_message_counts` (счётчики обработанных сообщений) — для ВСЕХ пользователей без исключения
- **`process_manager.py:95-101`** — `_busy_flags` (флаги «сессия занята, идёт обработка сообщения») индексируются по `session_id` без `asyncio.Lock`. При `concurrent_updates=256` (`bot.py:917`) два пользователя могут одновременно пройти проверку занятости и запустить параллельные retry-циклы
- **`session_watcher.py:45`** — `_paused_sessions` (приостановленные сессии) — глобальный `set` по `session_id`. Когда пользователь A отправляет сообщение и ставит сессию на паузу, watcher перестаёт отправлять уведомления о ней и пользователю B тоже

## Почему проблема не была предотвращена

### Проверенные скиллы

- **`validate-brd`** (глубокий аудит BRD перед разработкой) — направление 3 «Противоречия» формально покрывает случай «BRD говорит single-user, а конфиг допускает multi-user», но формулировки слишком общие. Нет явного пункта: «проверь, что архитектурные ограничения BRD не нарушены конфигурацией». **Вывод: пробел — скилл мог поймать, но инструкции недостаточно конкретны**

- **`spec-module`** (создание спецификаций модулей) — нет пункта про фиксацию архитектурных инвариантов в спецификации. Спеки `session_watcher`, `process_manager`, `config` не упоминают ограничение single-user. **Вывод: пробел — спецификации должны фиксировать ключевые допущения**

- **`implement-module`** (реализация модулей по спекам) — корректно реализует то, что написано в спеке. Проблема upstream (в спецификации). **Вывод: скилл корректен, пробел не его**

- **`review-code`** (4-проходное ревью кода) — проход 4 «Соответствие BRD» проверяет белый список, но чеклист не содержит пункта «если BRD = single-user, проверь что глобальное состояние не конфликтует при нескольких user ID». **Вывод: пробел — ревью кода могло поймать, но чеклист неполный**

- **`test-e2e`** (E2E тестирование через Telethon) — скилл добавляет второй user ID в `ALLOWED_USER_IDS` для тестирования. Нигде не анализирует последствия: второй пользователь разделит все глобальные структуры данных с первым. Нет предупреждения и нет механизма изоляции. **Вывод: прямой источник проблемы — E2E тестирование создаёт multi-user без защиты**

- **`test-integration`** (интеграционные тесты) — группа 5 «Конкурентный доступ» тестирует параллельные операции, но только для одного пользователя. Нет тест-кейса «два разных chat_id одновременно». **Вывод: корректно в рамках BRD (single-user), но при расширении до multi-user — пробел**

### Проверенные документы

- **`dev/docs/brd/brd-user-journeys.md`** (BRD) — явно декларирует single-user на строке 7. `ALLOWED_USER_IDS` описан как «для одного человека с разных устройств» на строке 32. Нет CJM для нескольких одновременных пользователей. **Вывод: BRD корректен, но ограничение осталось только текстом**

- **`CLAUDE.md`** (главный документ проекта) — нет раздела об архитектурном ограничении single-user. Упоминает `ALLOWED_USER_IDS` в контексте безопасности (белый список), но не фиксирует: «бот однопользовательский, несколько ID — только для одного человека». **Вывод: критический пробел — CLAUDE.md читается первым при каждой работе с проектом**

- **`dev/docs/adr/project_architecture.md`** (архитектурное решение) — описывает слоёную архитектуру, но не фиксирует ограничение single-user как архитектурный инвариант. **Вывод: пробел**

- **Спецификации модулей** (`dev/docs/specs/realised/`) — ни одна из проверенных спецификаций (session_manager, session_watcher, process_manager, config, project_manager) не фиксирует ограничение single-user и не описывает поведение при нескольких пользователях. **Вывод: системный пробел**

- **Предыдущие root-cause отчёты** (`dev/docs/logs/root-cause-reports/`) — ни один из 12+ отчётов не упоминает multi-user, второй пользователь или конфликт между chat_id. **Вывод: проблема не возникала ранее (или не была диагностирована)**

## Рекомендации по исправлению

1. **Добавить runtime-guard в `config.py`** — при загрузке конфигурации, если в `ALLOWED_USER_IDS` больше одного ID, выводить предупреждение в лог: «Бот рассчитан на одного пользователя. Несколько ID в ALLOWED_USER_IDS могут вызвать непредсказуемое поведение. Используйте несколько ID только для одного человека с разных устройств». Файл: `src/claude_manager/config.py`, функция `load_config()`

2. **Исправить watcher: отправлять уведомления только владельцу сессии** — в `session_watcher.py` заменить цикл по `config.ALLOWED_USER_IDS` на отправку только тому `chat_id`, который создал или подключился к данной сессии. Для этого нужно добавить в `session_manager` функцию `get_chat_id_for_session(session_id)` (обратный поиск по `_bindings`). Файлы: `src/claude_manager/session_watcher.py` (функция `_notify_chats`), `src/claude_manager/session_manager.py` (новая функция)

3. **Добавить `asyncio.Lock` в `process_manager`** — обернуть проверку и установку `_busy_flags` в один атомарный блок. Это предотвратит состояние гонки (race condition), когда два запроса одновременно проходят проверку занятости. Файл: `src/claude_manager/process_manager.py`, функции `send_message`, `_validate_process_ready`, `_prepare_for_send`

4. **Изолировать E2E тестирование** — добавить в `.env` отдельную переменную `E2E_TEST_USER_ID`. В коде бота обрабатывать этот ID отдельно: не отправлять ему watcher-уведомления в обычном режиме, только при явном запросе через E2E. Файлы: `.env.example`, `src/claude_manager/config.py`

5. **Очистить фантомные записи `_new_*` в daily_sessions.json** — добавить в `daily_session_registry` фильтрацию записей с префиксом `_new_` при загрузке с диска, если соответствующие файлы сессий не существуют. Файл: `src/claude_manager/daily_session_registry.py`, функция загрузки реестра

## Верификация выводов

- **Перечитанные документы:** BRD (строки 7, 32), CLAUDE.md (раздел «Архитектурные принципы»), спецификации session_watcher и process_manager, session_watcher.py (строки 174-190), config.py (строки 35-38)
- **Что увидел свежим взглядом:** BRD не просто описывает single-user — он явно объясняет зачем `ALLOWED_USER_IDS` множественный: «для одного человека с разных устройств». Это ключевое уточнение, которое никогда не попало в CLAUDE.md и не стало архитектурным инвариантом
- **Альтернативные объяснения:**
  - *«Может, это баг конкурентности, не связанный с multi-user?»* — частично верно (`concurrent_updates=256` и отсутствие `asyncio.Lock` усугубляют проблему), но триггером всегда является второй пользователь. Один пользователь не может вызвать broadcast в watcher на двух получателей
  - *«Может, бот перезапускается и переобрабатывает?»* — логи показывают рестарт в 08:59, но параллельные retry-цепочки (08:52-08:54) начались ДО рестарта, пока оба пользователя были активны
- **Подтверждение:** корневая причина подтвердилась. Все три симптома (дублирование, потеря привязок, параллельные retry) объясняются одной причиной — отсутствием изоляции при наличии двух пользователей

## Верификация решений

- **Рекомендация 1: Runtime-guard в config.py**
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** не меняет архитектуру, добавляет защиту. Соответствует BRD (single-user). Тривиальная реализация
  - **Корректировка:** без изменений

- **Рекомендация 2: Watcher отправляет только владельцу сессии**
  - **Вердикт:** ОДОБРЕНО С ЗАМЕЧАНИЯМИ
  - **Обоснование:** правильный долгосрочный фикс, но требует изменений в session_manager и session_watcher. Для немедленного исправления достаточно runtime-guard (рекомендация 1) + E2E изоляция (рекомендация 4)
  - **Корректировка:** приоритет снижен; реализовать после runtime-guard

- **Рекомендация 3: asyncio.Lock в process_manager**
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** race condition существует независимо от multi-user проблемы. Это баг, который нужно починить в любом случае
  - **Корректировка:** без изменений

- **Рекомендация 4: Изоляция E2E через отдельный E2E_TEST_USER_ID**
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** решает непосредственную проблему. E2E тесты получают доступ через отдельный механизм, не конфликтуя с основным пользователем
  - **Корректировка:** без изменений

- **Рекомендация 5: Очистка фантомных _new_* записей**
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** не связано с корневой причиной, но устраняет 37 000+ предупреждений в логах. Отдельная задача
  - **Корректировка:** без изменений
