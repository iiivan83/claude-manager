# Анализ первопричины: сессии чужого проекта попадают в дневной реестр

**Дата анализа:** 13-04-2026
**Проект:** Claude Manager (Telegram-бот для управления Claude Code)
**Статус:** Первопричина найдена

## Симптом

Бот бесконечно пишет WARNING "Файл сессии не найден" в error.log — 32 954 одинаковых
записи, лог раздулся до 9.5 МБ. Watcher (фоновый наблюдатель, который каждые 2 секунды
проверяет файлы сессий на диске) пытается прочитать файл сессии `dc96ec21...` (#45),
но файла нет.

Сессионный отчёт `dev/docs/session-reports/13-04/10-30_watcher-log-spam-diagnosis.md`
описал симптом и предложил решение (запоминать "мёртвые" сессии), но не ответил на
главный вопрос: **почему файл пропал?**

## Цепочка причин

1. **Watcher не находит файл сессии #45 в папке проекта claude_manager** — потому что

2. **Файл `.jsonl` этой сессии лежит в папке проекта budgets** (`~/.claude/projects/-Users-ivan-Desktop-claude-sandbox-budgets/dc96ec21...jsonl`) — потому что

3. **Сессия была создана Claude Code CLI, когда бот был переключён на проект budgets** — но при этом

4. **Запись о сессии попала в `daily_sessions.json` проекта claude_manager**, а не budgets — потому что

5. **КОРНЕВАЯ ПРИЧИНА: race condition при переключении проекта** — между моментом, когда `config.WORKING_DIR` меняется на новый путь (строка 195 в `project_manager.py`), и моментом, когда `daily_session_registry._registry_path` обновляется (внутри `reset_state()` → `load_registry()`, несколько `await` позже), watcher успевает проснуться, прочитать сессии из НОВОГО проекта и записать их в реестр СТАРОГО.

## Корневая причина

В `project_manager.py`, функция `_perform_switch()` (строка 182):

```python
config.WORKING_DIR = target_path          # строка 195: мгновенно
await _reset_all_state_modules()           # строка 197: три await подряд
```

`_reset_all_state_modules()` (строка 175) вызывает последовательно:
1. `await session_manager.reset_state()`  — первый await
2. `await daily_session_registry.reset_state()` — второй await (здесь `_registry_path` наконец обновится)
3. `await session_watcher.reset_state()` — третий await

Между строкой 195 и завершением пункта 2 существует **окно гонки**: `config.WORKING_DIR`
уже указывает на новый проект, но `daily_session_registry._registry_path` ещё указывает
на файл старого проекта. Если watcher (независимая asyncio-задача, которая опрашивает
файлы каждые 2 секунды) получит управление в этом окне, он:
- прочитает файлы сессий из НОВОГО `config.WORKING_DIR` (budgets)
- вызовет `daily_session_registry.register_session()`, который запишет в `_registry_path` — а это ещё СТАРЫЙ путь (claude_manager)

Результат: session_id из budgets сохраняется в `daily_sessions.json` проекта claude_manager.

**Масштаб проблемы:** Это не единичный случай. Проверка показала 6 пропавших файлов за 3 дня:
- **11 апреля:** #88, #89 (2 файла)
- **12 апреля:** #19, #22, #29 (3 файла)
- **13 апреля:** #45 (1 файл)

Все 6 файлов найдены в директории сессий проекта budgets. Все зарегистрированы в `daily_sessions.json` claude_manager.

## Почему проблема не была предотвращена

### Проверенные скиллы

- `feature-pipeline` (скилл, реализующий доработку функционала от задачи до коммита) — использовался для реализации фичи переключения проектов. Не содержит пункта о проверке race conditions между asyncio-задачами при переключении глобального состояния. **Вывод:** скилл пропустил этот аспект — нет требования проверять взаимодействие фоновых задач (watcher) с переключением состояния.
- `review-code` (глубокое ревью кода) — не выявил race condition при ревью feature-pipeline. **Вывод:** ревью фокусируется на качестве кода внутри одного модуля, но не проверяет межмодульные гонки в asyncio-контексте.
- `test-integration` (интеграционные тесты) — тесты на переключение проектов есть, но они синхронные по природе: вызывают `switch_project()` и проверяют результат. Не моделируют параллельную работу watcher. **Вывод:** интеграционные тесты не покрывают race conditions с фоновыми asyncio-задачами.

### Проверенные документы

- `CLAUDE.md` — описывает переключение проектов, упоминает что "Процессы Claude НЕ останавливаются", но не описывает требование атомарности переключения состояния для фоновых задач
- `dev/docs/specs/realised/session_reader_spec.md` — спецификация session_reader, не затрагивает взаимодействие с переключением проектов
- `dev/docs/brd/brd-user-journeys.md` — описывает сценарий переключения (CJM-09), но на уровне пользовательского опыта, без упоминания координации фоновых задач

## Рекомендации по исправлению

1. **Добавить глобальную паузу watcher при переключении проекта.** В `session_watcher.py` добавить флаг `_global_paused: bool = False` и функции `pause_all()` / `resume_all()`. В цикле `_poll_sessions()` проверять этот флаг перед опросом. В `_perform_switch()` (`project_manager.py`) вызывать `session_watcher.pause_all()` ПЕРЕД изменением `config.WORKING_DIR` и `session_watcher.resume_all()` ПОСЛЕ `_reset_all_state_modules()`. Это закрывает окно гонки полностью.

2. **Очистить существующие "чужие" записи из `daily_sessions.json`.** В `daily_session_registry.load_registry()` после загрузки файла добавить кросс-проверку: для каждого session_id проверить, существует ли `.jsonl` файл в текущем проекте. Записи без файлов — удалить. Это почистит уже накопившийся мусор и защитит от будущих аномалий.

3. **Дополнить решение из сессионного отчёта.** Предложенное в `10-30_watcher-log-spam-diagnosis.md` множество `_missing_file_sessions` по-прежнему полезно как защита второго уровня — но оно лечит симптом (спам в логах), а не причину (чужие записи в реестре). Реализовать оба исправления.

## Рекомендации по предотвращению

1. **[ARCHITECTURE]** Добавить в CLAUDE.md принцип атомарности переключения проекта:
   > **Атомарность переключения проекта.** Переключение проекта затрагивает глобальное состояние нескольких модулей (`config.WORKING_DIR`, `daily_session_registry._registry_path`, `session_manager`, `session_watcher`). Фоновые asyncio-задачи (watcher) не должны видеть промежуточное состояние — когда одни модули уже переключены, а другие ещё нет. Перед изменением `config.WORKING_DIR` фоновые задачи должны быть приостановлены, после полного сброса всех state-модулей — возобновлены.

   **Место:** раздел "Архитектурные принципы", после подраздела "Состояние в памяти".

2. **[DOC]** Обновить описание переключения проектов в CLAUDE.md — добавить упоминание глобальной паузы watcher как обязательного шага переключения.

3. **[CODE]** Реализовать исправления 1 и 2 из секции "Рекомендации по исправлению".

4. **[SKILL]** В `feature-pipeline` добавить в чеклист ревью пункт: "При изменении глобального состояния — проверить, какие фоновые asyncio-задачи могут увидеть частичное обновление".

## Верификация выводов

- **Перечитал `project_manager.py`**: подтвердил, что `config.WORKING_DIR` меняется на строке 195, а `_reset_all_state_modules()` вызывается на строке 197 — между ними нет никакой защиты от watcher
- **Перечитал `session_watcher.py`**: подтвердил, что `_poll_sessions()` использует `config.WORKING_DIR` напрямую (строки 111-112), а `_check_session()` вызывает `daily_session_registry.register_session()` без проверки проекта
- **Перечитал `daily_session_registry.py`**: подтвердил, что `register_session()` → `_save_registry()` записывает в `_registry_path`, который обновляется только при `load_registry()`
- **Кросс-проверка данных**: все 6 пропавших файлов найдены в `~/.claude/projects/-Users-ivan-Desktop-claude-sandbox-budgets/`. Сессия dc96ec21 зарегистрирована как #10 в budgets и #45 в claude_manager — двойная регистрация подтверждает гонку
- **Альтернативные объяснения:**
  - Ручное удаление файлов: исключено — файлы существуют, просто в другой папке
  - Очистка Claude Code CLI: исключено — `cleanupPeriodDays` = 30 дней (по умолчанию), а файлы сегодняшние
  - Баг в Claude Code CLI: исключено — CLI корректно записал файл в директорию проекта budgets

## Верификация решений

- **Рекомендация: глобальная пауза watcher при переключении**
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** Минимальное, точечное изменение. Механизм паузы уже есть для отдельных сессий — нужно добавить глобальный вариант. Не нарушает архитектуру (watcher и project_manager уже связаны через import). Пауза длится доли секунды — пользователь не заметит
  - **Корректировка:** без изменений

- **Рекомендация: кросс-проверка записей при загрузке реестра**
  - **Вердикт:** ОДОБРЕНО С ЗАМЕЧАНИЯМИ
  - **Обоснование:** Полезная защита от накопленного мусора. Замечание: при проверке не удалять записи с `_new_` префиксом — для них файлов нет по определению (temp ID ещё не заменён на UUID). Функция `_remove_phantom_entries()` уже обрабатывает `_new_` отдельно — кросс-проверку делать ПОСЛЕ неё
  - **Корректировка:** кросс-проверка пропускает записи с `_new_` префиксом

- **Рекомендация: принцип атомарности в CLAUDE.md**
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** Это архитектурная корневая причина — отсутствие явного принципа привело к тому, что при реализации проектного переключения не подумали о фоновых задачах. Принцип нужен чтобы такое не повторилось в будущем
  - **Корректировка:** без изменений

- **Рекомендация: множество `_missing_file_sessions` из сессионного отчёта**
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** Даже после исправления race condition, watcher должен уметь обрабатывать ситуацию "файл не найден" без спама. Это второй уровень защиты
  - **Корректировка:** без изменений
