# Анализ первопричины: Счётчик дневных сессий сбрасывается на #1 при перезапуске бота

**Дата анализа:** 30-03-2026
**Статус:** Первопричина найдена

## Симптом

Ты нажимаешь `/new` в боте и ожидаешь увидеть "Создана новая сессия #11" (или следующий номер по порядку). Но бот отвечает "Создана новая сессия #1" — как будто забыл, что сегодня уже были сессии #1 через #10.

На скриншоте видно: до этого бот нормально работал с сессиями #10 и #8, но после перезапуска следующая `/new` дала #1.

## Цепочка причин

1. **Бот отвечает "#1" вместо "#11"** — потому что функция `_next_day_number()` (в файле `daily_session_registry.py` — модуль, который хранит дневные номера сессий) не нашла ни одной записи за сегодня и вернула 1 → потому что →

2. **Реестр (словарь в памяти, где хранятся все сессии дня) оказался пустым** — потому что при запуске бота функция `load_registry()` (которая читает сохранённый файл `daily_sessions.json` с диска) не загрузила данные → потому что →

3. **`load_registry()` столкнулась с ошибкой чтения файла, которую не умеет обрабатывать** — код ловит только две конкретных ошибки: "файл не найден" и "файл повреждён (битый JSON)". Любая другая ошибка (например, проблема с правами доступа, ошибка файловой системы) пролетает мимо и поднимается выше → потому что →

4. **`post_init()` (функция инициализации бота в `bot.py`) ловит ВСЕ ошибки при загрузке состояния и молча продолжает работу с пустым реестром** — код выглядит так: "попробуй загрузить данные, а если не вышло — просто начни с чистого листа и запиши в лог предупреждение". Бот продолжает работать как будто ничего не случилось, но с полностью пустым счётчиком → потому что →

5. **КОРНЕВАЯ ПРИЧИНА: Нет защиты от перезаписи хороших данных пустыми.** Когда первая `/new` регистрирует сессию #1 в пустом реестре, функция `_save_registry()` записывает на диск файл с одной-единственной записью — и **перезаписывает** старый файл, где были все 10+ сессий. Данные **безвозвратно потеряны**.

## Корневая причина

Проблема в трёх связанных уязвимостях кода, которые по отдельности не страшны, но вместе создают "идеальный шторм":

**Уязвимость 1 — Неполный перехват ошибок при чтении файла**

Файл: [src/claude_manager/daily_session_registry.py](file:///Users/ivan/Desktop/claude-sandbox/claude_manager/src/claude_manager/daily_session_registry.py), строки 156-165

```python
try:
    content = await asyncio.to_thread(_registry_path.read_text, "utf-8")
    _registry = json.loads(content)
except FileNotFoundError:    # ← ловит только "файл не найден"
    _registry = {}
except json.JSONDecodeError:  # ← ловит только "файл битый"
    _registry = {}
# А что если PermissionError? OSError? UnicodeDecodeError? — Не поймано!
```

Любая "нестандартная" ошибка файловой системы улетает наверх незамеченной.

**Уязвимость 2 — Молчаливое проглатывание ошибки в `post_init()`**

Файл: [src/claude_manager/bot.py](file:///Users/ivan/Desktop/claude-sandbox/claude_manager/src/claude_manager/bot.py), строки 474-481

```python
try:
    await session_manager.load_bindings()
except Exception:
    logger.warning("Ошибка при восстановлении состояния — начинаю с чистого")
```

Если `load_registry()` бросает ошибку — `post_init()` её ловит, пишет в лог и **спокойно продолжает работу** с пустым реестром. Бот жив, но нумерация потеряна.

**Уязвимость 3 — Нет защиты от перезаписи**

Нигде в коде нет флага "реестр был загружен с диска". Функция `_save_registry()` не проверяет, загружался ли реестр успешно. Она просто пишет то, что есть в памяти — даже если в памяти пусто. Первая же регистрация сессии вызывает `_save_registry()`, которая перезаписывает старый файл (с 10+ сессиями) новым (с одной сессией #1).

**Что спровоцировало ошибку загрузки:**

По логам видно, что бот перезапускался несколько раз подряд (02:56, 02:57, 02:58). Между перезапусками — менее минуты. При быстрых перезапусках возможны ситуации, когда предыдущий процесс ещё не успел завершить запись файла, или `daily_sessions.json.tmp` (временный файл при атомарной записи) остался после аварийного завершения, или файловая система macOS вернула неожиданную ошибку при одновременном доступе.

Кроме того, LaunchAgent (автозапуск через macOS) и `watch_and_restart.sh` (автоперезапуск при изменениях кода) могут конфликтовать, потому что используют **разные PID-файлы** в разных папках:
- `watch_and_restart.sh` хранит PID в папке проекта: `.../claude_manager/bot.pid`
- `main.py` (код бота) хранит PID в рабочей папке: `.../su-main-master 2/bot.pid`

Это означает, что `watch_and_restart.sh` может не знать о боте, запущенном через LaunchAgent, и наоборот.

## Почему проблема не была предотвращена

### Проверенные скиллы

- `spec-module` (скилл, который пишет спецификации модулей) — спецификация `daily_session_registry_spec.md` описывает обработку двух ошибок чтения файла (файл не найден, битый JSON), но **не описывает** обработку прочих ошибок файловой системы. Также нет требования "не перезаписывать файл, если реестр не был успешно загружен". **Вывод:** пробел в спецификации — не описан сценарий "нестандартная ошибка при чтении"

- `implement-module` (скилл, который пишет код по спецификации) — реализовал ровно то, что написано в спецификации, включая обработку только двух конкретных ошибок. Скилл работает верно — он следует спеке. **Вывод:** скилл корректен, проблема в спецификации

- `review-code` (скилл ревью кода) — чеклист ревью включает "обработка ошибок", но формулировка общая ("внешние вызовы обёрнуты в try/except"). Нет конкретного пункта про "все ли типы ошибок файловой системы пойманы?" или "есть ли защита от перезаписи валидных данных пустыми?". **Вывод:** чеклист ревью недостаточно конкретен для этого типа проблем

### Проверенные документы

- [development/specs/realized/daily_session_registry_spec.md](file:///Users/ivan/Desktop/claude-sandbox/claude_manager/development/specs/realized/daily_session_registry_spec.md) (спецификация модуля дневной нумерации) — описывает `FileNotFoundError` и `JSONDecodeError`, но **не описывает** обработку `OSError`, `PermissionError`, `UnicodeDecodeError` при чтении. Также нет требования к защите от перезаписи данных при неудачной загрузке

- [development/docs/brd-user-journeys.md](file:///Users/ivan/Desktop/claude-sandbox/claude_manager/development/docs/brd-user-journeys.md) (пользовательские сценарии) — CJM-04 (создание новой сессии) описывает нумерацию, но не описывает сценарий "бот перезапущен — нумерация должна продолжиться"

- [CLAUDE.md](file:///Users/ivan/Desktop/claude-sandbox/claude_manager/CLAUDE.md) — описывает атомарную запись и восстановление состояния из файлов при перезапуске, но **нет архитектурного принципа** "не перезаписывай персистентные данные, если они не были успешно загружены"

## Рекомендации по исправлению

1. **Расширить обработку ошибок в `load_registry()`** — ловить все ошибки, а не только две конкретных.

   Файл: `src/claude_manager/daily_session_registry.py`, строки 156-165

   Добавить `except Exception` после `except json.JSONDecodeError`, который логирует ошибку и создаёт пустой реестр. Так бот не упадёт из-за неожиданной ошибки файловой системы.

2. **Добавить флаг `_loaded_from_disk`** — отслеживать, был ли реестр загружен с диска.

   Файл: `src/claude_manager/daily_session_registry.py`

   Новая переменная модуля `_loaded_from_disk: bool = False`. Устанавливается в `True` только при успешном чтении файла в `load_registry()`. Функция `_save_registry()` проверяет этот флаг: если `False` — лог предупреждение и отказ от записи (чтобы не перезатереть старые данные пустым реестром).

3. **Убрать тихое проглатывание ошибки в `post_init()`** — если загрузка состояния упала, бот должен хотя бы отметить это в реестре, а не молча работать с пустыми данными.

   Файл: `src/claude_manager/bot.py`, строки 474-481

## Верификация выводов

- **Перечитал `daily_session_registry.py` повторно** — подтвердил, что `except` блоки ловят только `FileNotFoundError` и `JSONDecodeError`. Любая другая ошибка чтения файла действительно пролетает вверх

- **Перечитал `bot.py` `post_init()`** — подтвердил, что `except Exception` ловит всё и продолжает с пустым состоянием. Реестр `_registry` остаётся `{}` (значение по умолчанию при импорте модуля)

- **Проверил логи бота** — логи LaunchAgent подтверждают множественные перезапуски (02:56, 02:57, 02:58) и обрыв логов после 03:00. Бот в 04:33 (момент бага) работал не через LaunchAgent, а через другой механизм (вероятно `watch_and_restart.sh`), чьи логи шли в терминал

- **Проверил `daily_sessions.json`** — файл сейчас содержит 13 записей за сегодня, начиная с #1. Запись #1 — реальный UUID (темповый ID `_new_0001` был обновлён). Записи #2 и #3 — `_new_0002` и `_new_0003` (неиспользованные темповые сессии). Это полностью совпадает с теорией: реестр был пуст → /new дал #1 → далее сессии накапливались заново

### Альтернативные версии и почему отвергнуты

- **"Баг в `_next_day_number()` — неправильно считает максимум"** — маловероятно: код `max(numbers) + 1` прост и покрыт тестом `test_number_after_gap_in_sequence`. Если записи есть — номер будет правильный

- **"Файл `daily_sessions.json` был удалён вручную"** — возможно, но не объясняет системность: баг может повториться при любом перезапуске. Кроме того, файл сейчас существует (был пересоздан после бага)

- **"`_ensure_today_registry()` затирает старые данные"** — неверно: код только добавляет пустой словарь для нового дня, если его нет. Старые записи не трогаются
