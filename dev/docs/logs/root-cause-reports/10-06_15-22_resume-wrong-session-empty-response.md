# Анализ первопричины: возобновление после «стоп» ушло в чужую сессию + повторяющийся «пустой ответ»

**Дата анализа:** 10-06-2026
**Проект:** Claude Manager — Telegram-бот для управления Claude Code с телефона
**Статус:** Первопричина найдена (по обоим симптомам)

## Симптом

Утром 10 июня в проекте `bloger` шли две параллельные сессии: **#1 «куратор»**
(session_id `3f4ca4d4-7168-45b7-aa3f-bd1980ed356e`, обсуждение фильтра новостных
поводов) и **#2 «форматы постов»** (session_id `235f7765-fea0-4b70-95c5-764e319d213e`,
обсуждение категорий постов из источников).

Пользователь послал «стоп», затем продолжил разговор — и продолжение попало в сессию
куратора, а не в форматы постов. Параллельно начала раз за разом вылезать ошибка про
пустой/отсутствующий ответ. В итоге две сессии перемешались: куратор внезапно ответил
на реплику про форматы постов.

Гонка retry-цикла со «стопом» (старый цикл повторов воскресал и переотправлял сообщение)
уже починена в предыдущей сессии. Этот отчёт разбирает два оставшихся вопроса:
**(1)** почему возобновление ушло в куратора и **(2)** откуда брался «пустой ответ».

## Восстановленная хронология инцидента

Хронология собрана из трёх независимых источников: лог бота
(`~/.local/state/claude-manager/claude-manager.log`), JSONL-файлы обеих сессий
(`~/.claude/projects/-home-ivan-claude-sandbox-bloger/`) и формулировка жалобы из
сессии `68f3bfd9` проекта claude_manager. Время местное (UTC+5).

- **С вечера 9 июня** активная привязка чата (запись «этот чат сейчас работает с этой
  сессией», хранится в `session_manager` и `sessions.json`) указывала на сессию
  куратора — она была создана через `/new` в 23:30 и с тех пор явно не переключалась.
- **Всё утро 10 июня** пользователь работал с тремя сессиями **только через
  Telegram-реплеи** (ответы на конкретные сообщения бота). Reply-роутинг доставляет
  текст в сессию-источник сообщения, но **активную привязку не меняет** — в логе за всё
  утро нет ни одной записи «привязан к сессии».
- **11:13:27** — сообщение «2» уходит в куратора. Claude отвечает («Принято — вариант 2
  (средний)…») и продолжает работать с инструментами (`stop_reason=tool_use` — ход не
  завершён).
- **11:15:03** — пользователь реплеем на сообщение форматов отправляет «Тут важная
  деталь…» в сессию #2. Она доставлена правильно и через минуту получает нормальный
  ответ.
- **11:15:10** — пользователь шлёт «стоп». Бот убивает процесс куратора посреди хода.
  Читатель событий видит обрыв без финального события и запускает **цикл повторов A**
  (текст «2»). Из-за уже починенной гонки стоп-флага цикл не останавливается.
- **11:15:29** — пользователь повторно отправляет «Тут важная деталь…» — уже **обычным
  сообщением, без реплея**. Оно уходит в активную привязку чата — то есть в куратора.
  Доказательство, что это был именно обычный путь (`send_to_claude_and_respond`), а не
  reply-роутинг: сторожевой таймер тишины (agent silence watchdog) сработал ровно через
  60 секунд, в 11:16:29 — его запускает только обычный путь, reply-роутинг таймер не
  ставит.
- **11:16:10** — цикл A, выждав минуту, делает повтор: **убивает чужой процесс**,
  занимающий общий слот сессии (это был процесс свежего сообщения «Тут важная
  деталь…»), и перезапускает свой. Убитое сообщение тоже падает «без финального
  события» и порождает **цикл повторов B** (текст «Тут важная деталь…»).
- **11:16–11:35** — циклы A и B по очереди, раз в ~60 секунд, убивают процессы друг
  друга через общий слот: каждый повтор начинается с остановки «старого процесса» по
  ключу сессии, не проверяя, чей он. Отсюда ~20 минут чередующихся ошибок «повтор
  N/10: Процесс завершился без финального события». Каждый перезапуск с `--resume`
  заставлял Claude CLI закрывать оборванный ход синтетической парой «Continue from
  where you left off.» → «No response requested.» — эти строки и насорили в файл
  сессии куратора.
- **11:35:26** — цикл A исчерпал 10 попыток → пользователю ушла финальная ошибка.
- **11:37:20** — последний перезапуск цикла B дожил до ответа (убивать его стало
  некому), и куратор ответил на реплику про форматы(«Записал: по строгости —
  вариант 2…»). Это и есть видимое «перемешивание» сессий.

## Цепочка причин — симптом 1 (возобновление ушло в куратора)

1. **Продолжение разговора про форматы постов оказалось в сессии куратора** → потому что →
2. **Пользователь отправил его обычным сообщением (не реплеем), а обычные сообщения
   идут в активную привязку чата** (`telegram_input_handlers.handle_message` →
   `claude_interaction.send_to_claude_and_respond` → `session_manager.get_active_session`)
   → потому что →
3. **Активная привязка всё ещё указывала на куратора со вчерашнего вечера**, хотя
   пользователь всё утро фактически работал в другой сессии → потому что →
4. **Reply-роутинг доставляет сообщение в сессию-источник, но не трогает активную
   привязку** (`reply_route_handler` шлёт через `process_manager.send_message`
   напрямую, `session_manager` не вызывает) → потому что →
5. **КОРНЕВАЯ ПРИЧИНА: в дизайне маршрутизации два понятия «текущей сессии» разошлись
   без обратной связи.** У пользователя «текущая» — это разговор, которого он касался
   последним; у бота — последняя явно привязанная сессия. Бот нигде не показывает,
   в какую сессию уйдёт обычное сообщение, и не даёт момента заметить расхождение:
   заголовок с номером сессии появляется только вместе с ответом, через десятки секунд.

Это пробел спецификации reply-routing-v1 (`dev/docs/specs/31.05_04.56-reply-routing-v1-spec.md`):
сценарий «пользователь долго работает только реплеями, потом пишет обычное сообщение»
там не рассмотрен.

## Цепочка причин — симптом 2 (повторяющийся «пустой ответ»)

1. **Раз за разом вылезала ошибка, что ответа нет / ответ пустой** → потому что →
2. **Каждую минуту очередной повтор завершался обрывом**: пользователю уходили
   сообщения «#1 Ошибка Claude, повтор N/10: Процесс завершился без финального
   события» (до ~20 штук за инцидент) → потому что →
3. **Два цикла повторов работали на одной сессии и убивали процессы друг друга**:
   `_execute_single_retry` перед перезапуском останавливает процесс из общего слота
   `_processes[(session_id, backend)]`, не проверяя, принадлежит ли тот ему → потому что →
4. **Второй цикл вообще не должен был существовать**: он родился, когда повтор цикла A
   убил процесс свежепринятого сообщения; а сам цикл A должен был умереть от «стопа»
   в 11:15:10 → потому что →
5. **КОРНЕВАЯ ПРИЧИНА (общая с уже починенной гонкой): у слота сессии не было понятия
   «владелец хода».** Процесс, busy-флаг и стоп-флаг жили в общих ячейках по ключу
   сессии; любой участник (новый send, чужой retry, /stop) читал и перетирал их вслепую.
   Починка предыдущей сессии (owner_stop_event + проверка тождества) закрывает
   стоп-флаг и дуэль циклов, но шаг «убей старый процесс из слота» в
   `_execute_single_retry` (process_retry.py:60-62) по-прежнему не проверяет владение —
   остаточный риск той же породы.

Дополнительный слой симптома — **строки про пустой ответ**. Claude CLI при `--resume`
сессии с оборванным ходом вставляет синтетическую пару «Continue from where you left
off.» → «No response requested.» (см. `dev/docs/claude-cli-stream-json-protocol.md`,
раздел «resume-placeholder»). За 20 минут дуэли таких пар в сессию куратора записалось
~18 штук. Бот умеет фильтровать этот маркер в watcher-пути
(`coding_agent_session_file_poller.NO_RESPONSE_MARKERS`), но в progress-пути прямого
запроса фильтра нет: если событие с этим текстом приходит по stdout, оно проходит через
`telegram_response_delivery._replace_empty_response` и превращается в сообщение
«Claude обработал запрос, но не дал текстовый ответ». Достоверно доказать по логам, что
пользователь получал именно эту формулировку (а не только «повтор N/10»), нельзя —
доставка сообщений не логируется; этот путь помечен как требующий эмпирической проверки
контракта stdout (правило CLAUDE.md «контракты проверяются эмпирически»).

## Почему проблема не была предотвращена

Это уже **третий** инцидент вокруг retry-цикла: 13-04 («retry loop создаёт десятки
новых сессий»), 29-05 («сообщение застряло в переполненной сессии #9, повтор-спам»),
10-06 (этот). Каждый раз чинился конкретный сценарий (обновление session_id в ретраях,
классификация постоянных ошибок, теперь — владение стоп-флагом), но общий источник —
общие ячейки состояния сессии без владельца хода — оставался. Модуль `process_manager`
вырос до 1226 строк (сегодня, уже после инцидента, распилен рефакторингом на
process_send/retry/stop/events/lifecycle/state) — классический признак того, что код
перерос архитектуру и точечные правки перестали помогать.

### Проверенные скиллы

- `root-cause-analysis` (поиск первопричин) — два прошлых отчёта по retry-циклу дали
  верные точечные рекомендации, и они были применены. Пункта «при 3-м инциденте в одном
  модуле эскалировать в архитектурную ретроспективу» в процессах не было — паттерн
  зафиксирован только сейчас.
- `development-process-retrospective` (ретроспектива процесса при повторных инцидентах) —
  в проекте лежит только в архиве (`.claude/skills/_archive/`), глобально не установлен.
  Автоматическая ретроспектива по паттерну «3+ отчёта об одном модуле» не запустилась.
- Скиллы разработки (test-driven-development, feature-pipeline и др.) — гонку
  конкурентного доступа они предотвратить не могли: сценарий «стоп + мгновенный повтор
  сообщения во время ретраев» не входил ни в один тест-план до вчерашнего дня. Вывод:
  пробел не в скиллах, а в отсутствии архитектурного инварианта.

### Проверенные документы

- `dev/docs/claude-cli-stream-json-protocol.md` (справочник протокола) — resume-placeholder
  «No response requested.» уже описан, включая правило «не считать его пустым ответом
  CLI». Не описан триггер «оборванный ход с tool_use» и появление пары при каждом
  перезапуске ретрая — дополнить.
- `dev/docs/specs/31.05_04.56-reply-routing-v1-spec.md` (спека reply-роутинга) — сценарий
  расхождения активной привязки с фактическим местом работы не рассмотрен.
- `CLAUDE.md` — описывает busy-флаги и stop-events как общие словари, но принципа
  «один владелец хода на сессию» в нём нет (фикс предыдущей сессии его фактически ввёл,
  текстом не закреплён).
- Отчёты `13-04_17-45_retry-loop-session-multiplication.md` и
  `29-05_11-26_session-overflow-retry-spam.md` — рекомендации применены, но оба отчёта
  останавливались на уровне конкретного сценария, zoom-out до «общего слота без
  владельца» не делали.

## Рекомендации по исправлению

1. **[CODE] Защитить остановку чужого процесса в ретрае.** В
   `process_retry._execute_single_retry` (src/claude_manager/process_retry.py:60-62)
   перед `_apply_backend_stop_strategy(old_process, ...)` проверять владение ходом
   (тот же owner-механизм, что уже добавлен для стоп-флага): если процесс в слоте
   принадлежит другому владельцу — прервать ретрай как superseded, а не убивать процесс.
2. **[CODE] Фильтровать resume-placeholder в progress-пути.** В
   `process_events._process_events` (или в
   `claude_code_backend.read_progress_text_from_event`) не передавать в progress текст,
   равный `EMPTY_RESPONSE_MARKER` («No response requested.») — зеркально фильтру
   `NO_RESPONSE_MARKERS` в poller. Иначе каждый перезапуск ретрая может показывать
   пользователю ложное «Claude обработал запрос, но не дал текстовый ответ».
3. **[CODE] Свести 4 дубля маркера в одну константу.** Сейчас «No response requested.»
   живёт в `claude_interaction.NO_RESPONSE_MARKER`, `process_types.EMPTY_RESPONSE_MARKER`,
   `claude_code_session_file_reader.EMPTY_RESPONSE_MARKER` и
   `coding_agent_session_file_poller.NO_RESPONSE_MARKERS`. Оставить один источник
   (логично — `process_types` или `coding_agent_backend`) и импортировать его всюду.
   После переименования прогнать `grep -r "No response requested"` по всему проекту и
   обновить каждое место использования.
4. **[CODE, решение за пользователем] Закрыть расхождение «текущей сессии».** Два
   варианта на выбор, нужно продуктовое решение:
   - при reply-роутинге обновлять активную привязку на сессию-цель («куда ответил —
     там и работаю»);
   - либо оставить привязку как есть, но давать мгновенную обратную связь для обычных
     сообщений — короткий ack вида «→ #1 куратор» сразу при приёме, чтобы ошибочную
     маршрутизацию было видно до ответа.

## Верификация выводов

Перепроверил хронологию по трём источникам и прогнал альтернативные объяснения:

- **Альтернатива «бот сам перекинул сообщение из #2 в #1 через remap session_id»** —
  отвергнута: в логе за окно инцидента нет ни одной записи «Session ID обновлён», а
  сообщение в #2 было доставлено и нормально отвечено. В файлы обеих сессий текст попал
  двумя независимыми отправками (11:15:03 и 11:15:29).
- **Альтернатива «пользователь ответил реплеем не на то сообщение (мис-тап)»** —
  отвергнута: отправка 11:15:29 шла через обычный путь — это доказывает сторожевой
  таймер тишины, сработавший ровно в 11:16:29 (его ставит только
  `send_to_claude_and_respond`; reply-роутинг таймер не запускает).
- **Альтернатива «Claude CLI при resume форкнул сессию #2 в #1»** — отвергнута: resume
  с форком давал бы новый UUID, а не существующий ID куратора; к тому же ответ в #2
  пришёл штатно.
- **Сверка таймингов** подтвердила интервал ретраев 60 секунд и точное чередование
  циклов A и B: каждый перезапуск в логе бота совпадает по секундам с парой
  «Continue …» → текст в JSONL куратора.
- Корневые причины устояли; уточнение по ходу верификации — «пустой ответ» в формулировке
  пользователя достоверно объясняется потоком ошибок «повтор N/10: Процесс завершился
  без финального события», а путь через «Claude обработал запрос, но не дал текстовый
  ответ» возможен, но не доказуем без логов доставки (отсюда рекомендация 4).

## Верификация решений

- **Рекомендация: owner-проверка перед kill в ретрае** —
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** прямое продолжение уже принятого в фиксе owner-механизма;
    соответствует принципу stale session_id из CLAUDE.md.
  - **Корректировка:** без изменений.
- **Рекомендация: фильтр resume-placeholder в progress** —
  - **Вердикт:** ОДОБРЕНО С ЗАМЕЧАНИЯМИ
  - **Обоснование:** зеркалит уже существующий фильтр poller; риск — заглушить настоящий
