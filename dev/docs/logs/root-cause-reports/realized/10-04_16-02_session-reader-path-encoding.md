# Анализ первопричины: бот не видит сессии и теряет сообщения после смены рабочей директории

**Дата анализа:** 10-04-2026
**Проект:** Claude Manager (Telegram-бот, который служит пультом управления Claude Code с телефона)
**Статус:** Первопричина найдена и подтверждена эмпирически

## Симптом

Пользователь написал: «Не все сообщения из проекта бюджет приходили в бота. Вообще он как-то сильно не стабильно работает. Я связываю это с костыльной системой запуска, но это возможно не так.»

Симптомы простыми словами:
- **Часть сообщений «пропадает»** — пользователь пишет в Telegram, а бот иногда молчит или отвечает через раз
- **Общее ощущение нестабильности** — работа идёт не предсказуемо

Гипотеза пользователя — проблема в системе запуска (три способа поднять бота: руками, через `watch_and_restart.sh` — скрипт-наблюдатель, который перезапускает бота при изменении `.py` файлов, и через LaunchAgent — механизм macOS для старта программы при входе в систему).

**Спойлер:** гипотеза отклонена. Настоящая причина — смешная одностроковая ошибка в функции, которая кодирует путь папки, и она проявилась именно сегодня после одного безобидного изменения в `.env`.

## Цепочка причин

1. **Пользователю кажется, что бот нестабилен и теряет сообщения** → потому что →

2. **Живой мониторинг сессий (`session_watcher` — модуль, который каждые 2 секунды читает файлы сессий и пересылает новые ответы Claude в Telegram) полностью сломан начиная с сегодняшнего 15:40.** Это значит: если Claude Code пишет промежуточные ответы, долго думает, а потом выдаёт финал — бот не видит этот живой поток и не пересылает его пользователю. Работает только «прямой» путь: «пользователь → Claude → первый ответ через стандартный вывод → Telegram» → потому что →

3. **Watcher не находит папку сессий на диске.** Бот ищет папку `~/.claude/projects/-Users-ivan-Desktop-claude-sandbox-claude_manager` (обрати внимание — в конце `claude_manager` через подчёркивание). А Claude Code CLI (внешний инструмент, который бот запускает) реально создаёт папку `~/.claude/projects/-Users-ivan-Desktop-claude-sandbox-claude-manager` (через дефис). Разница в одном символе — «_» vs «-». Папки с подчёркиванием нет, и watcher зря каждые 2 секунды пишет warning в лог → потому что →

4. **Функция `_encode_project_path` (в `src/claude_manager/session_reader.py:58-60` — модуль, который читает файлы сессий Claude Code с диска) превращает путь проекта в имя папки неверно.** Она заменяет только `/` и пробелы на `-`, но НЕ заменяет подчёркивание `_` (и, заодно, точки и другие символы). А настоящий Claude Code CLI меняет на `-` вообще все символы, кроме букв и цифр. Для пути `/Users/ivan/Desktop/claude-sandbox/claude_manager` функция бота даёт `-Users-ivan-Desktop-claude-sandbox-claude_manager`, а Claude Code CLI даёт `-Users-ivan-Desktop-claude-sandbox-claude-manager`. Всегда промах → потому что →

5. **Спецификация модуля (`dev/docs/specs/realised/session_reader_spec.md:62-64`) буквально описывает неполное правило.** Цитата из спеки: *«Кодирует путь к проекту в формат, который Claude Code использует для имён папок: слеши и пробелы заменяются дефисами»*. В разделе «Алгоритм» (строки 134-138) — два пункта: «Заменить слеши на дефисы» и «Заменить пробелы на дефисы». Скилл `implement-module` (скилл, который пишет код модулей строго по спецификации) реализовал ровно то, что написано в спеке. Код корректен относительно спеки, спека некорректна относительно реальности → потому что →

6. **[КОРНЕВАЯ ПРИЧИНА] Скилл `spec-module` (скилл, который пишет технические задания на модули перед реализацией) не требует проверять контракты с внешними системами эмпирически.** Разработчик (или агент), писавший спецификацию, **предположил** как именно Claude Code CLI формирует имя папки — по аналогии с тем, что он сам подумал. Это предположение никогда не было проверено: ни через запуск реального `claude` и наблюдение результата, ни через чтение исходников CLI (они, кстати, лежат в соседней папке `claude-code-sourcecode`). Когда контракт с внешней системой фиксируется по догадке, а не по эмпирическому факту — любая частная ошибка в догадке становится миной замедленного действия.

Баг живёт с самого первого дня реализации модуля (коммит `cc1c90e` — «pipeline(phase-3): слой 1 реализован»). Все 11+ дней он был латентным — потому что пользователь случайно работал в папке `budgets`, в пути которой нет подчёркиваний. Сегодня, 10 апреля в 15:40, рабочая директория была переключена на `claude_manager` — и мина сработала.

## Корневая причина

Если коротко: **функция, которая преобразует путь проекта в имя папки сессий Claude Code, реализует только часть правила санитации. Полный алгоритм Claude Code CLI заменяет на дефис ВСЕ символы, кроме букв и цифр. Наш код заменяет только слеш и пробел. Это было зафиксировано в спецификации как догадка и воспроизведено в коде без эмпирической проверки.**

Конкретные места:

- **`src/claude_manager/session_reader.py:58-60`** (модуль чтения файлов сессий с диска) — сама функция `_encode_project_path`:
  ```python
  def _encode_project_path(project_dir: str) -> str:
      """Кодирует путь проекта в формат имён папок Claude Code."""
      return project_dir.replace("/", "-").replace(" ", "-")
  ```

- **`dev/docs/specs/realised/session_reader_spec.md:62-64, 134-138`** (спецификация модуля session_reader, по которой его реализовали) — описывает неполный алгоритм буквально. Цитата:
  > Кодирует путь к проекту в формат, который Claude Code использует для имён папок: слеши и пробелы заменяются дефисами.

- **`tests/test_session_reader.py:65-83`** (юнит-тесты модуля) — класс `TestEncodeProjectPath` проверяет только happy-path:
  - `test_standard_path` — путь без подчёркиваний (!)
  - `test_path_with_spaces` — путь с пробелом
  - `test_root_path` — только слеш

  Случай «путь с подчёркиванием» не покрыт. Тесты зелёные, потому что они закрепляют буквальное поведение кода, а не настоящий контракт Claude Code CLI.

- **Реальный алгоритм Claude Code CLI** (эмпирически установлен из исходников, `claude-code-sourcecode/utils/sessionStoragePortable.ts:311`):
  ```typescript
  export function sanitizePath(name: string): string {
    const sanitized = name.replace(/[^a-zA-Z0-9]/g, '-')
    if (sanitized.length <= MAX_SANITIZED_LENGTH) {
      return sanitized
    }
    ...
  }
  ```
  Регулярка `/[^a-zA-Z0-9]/g` — «всё, что НЕ буква и НЕ цифра». Плюс есть ещё усечение длинных путей с добавлением хэша, но для нашего проекта этот кусок не срабатывает — путь короткий.

Эмпирическое подтверждение расхождения:

- **На диске существует** папка `/Users/ivan/.claude/projects/-Users-ivan-Desktop-claude-sandbox-claude-manager/` (с дефисом) — **142 файла**, самый свежий `13457bb5-17a9-4645-8794-e97b82b9c60d.jsonl` изменён 10 апреля 16:00 (это сессия текущего разговора)
- **На диске НЕ существует** папка `/Users/ivan/.claude/projects/-Users-ivan-Desktop-claude-sandbox-claude_manager/` (с подчёркиванием) — бот ищет именно её и никогда не находит
- **Файл логов `~/Library/Logs/claude-manager.error.log` разросся до 314 МБ**, внутри — 1 485 560 строк, большинство это повторяющийся WARNING `Папка сессий не найдена: .../claude_manager` раз в 2 секунды

## Почему проблема не была предотвращена

### Проверенные скиллы

- **`spec-module`** (скилл, который пишет технические задания на модули перед их реализацией) — в шаблоне спецификации есть раздел «Обработка ошибок», требование к типам, раздел «Зависимости». Но **нет ни одного пункта про контракты с внешними системами** и их эмпирическую верификацию. Скилл не требует проверить, как реально ведёт себя Claude Code CLI при формировании имени папки. Это **главный пробел**: скилл разрешает зафиксировать алгоритм по догадке.

- **`implement-module`** (скилл, который пишет код модулей строго по спецификации) — корректен. Он реализовал то, что было написано. Виновата не реализация, а спецификация.

- **`review-code`** (скилл 4-проходного ревью всего кода) — в нём есть Проход 1 (качество кода), Проход 2 (безопасность), Проход 3 (архитектура), Проход 4 (соответствие BRD). **Нет явного пункта про верификацию контрактов с внешними инструментами**. Ревью могло бы поймать расхождение, если бы чеклист явно требовал «для каждой функции, формирующей путь/имя для внешней системы, убедиться что правило воспроизведено целиком». Сейчас такого требования нет.

- **`root-cause-analysis`** (сам этот скилл, которым я сейчас пользуюсь) — в сессии 10-04 15:47 (`dev/docs/session-reports/10-04/15-47_switch-bot-working-dir.md`) предыдущий агент (я сам, в предыдущем запуске) **ошибочно диагностировал** warning-и как временное явление: *«После перезапуска в claude-manager.error.log каждые 2 секунды сыпется WARNING session_reader: Папка сессий не найдена... Не бага — это session_watcher опрашивает папку Claude CLI каждые 2 секунды, а папка появится только после первого запуска Claude CLI ботом»*. **Это было неверно.** Правильный первый шаг был: запустить `ls ~/.claude/projects/` и сравнить реальные папки с тем, что формирует `_encode_project_path`. Это минута работы. Скилл `root-cause-analysis` в тот момент не был вызван — агент просто принял симптом за «ожидаемое поведение». Процессный вывод: не полагаться на дружелюбные объяснения без фактической проверки.

### Проверенные документы

- **`CLAUDE.md`** (главный документ проекта) — есть раздел «Архитектурные принципы», «Асинхронность», «Состояние в памяти», «Защита персистентных данных от затирания», «Важные детали для разработки». Ни в одном разделе **нет принципа** «контракты с внешними системами проверяются эмпирически, а не по догадке». Это пробел, и это архитектурный пробел — он мог проявиться в любом модуле, работающем с внешним CLI или API, не только в session_reader.

- **`dev/docs/specs/realised/session_reader_spec.md`** — содержит ошибочное утверждение про «слеши и пробелы заменяются дефисами». Это прямая причина неверной реализации. Спека никогда не пересматривалась.

- **`tests/test_session_reader.py`** — тесты закрепили неверное правило, и теперь служат «гарантией» сохранения бага. Классическая ловушка: тесты валидируют код относительно ожиданий разработчика, а не относительно реального контракта внешней системы.

- **`dev/docs/session-reports/10-04/15-47_switch-bot-working-dir.md`** — самый свежий отчёт предыдущей сессии (за 15 минут до текущей). Зафиксировал симптом (WARNING в логе), дал ошибочное объяснение, пометил «незавершённым» только «пользователь ещё не протестировал работу в Telegram». Реальный корень остался незамеченным.

- **`dev/docs/logs/root-cause-reports/30-03_05-08_message-not-reaching-terminal-session.md`** — прошлый root-cause отчёт про похожий симптом «бот молчит». Он касался таймаутов на `readline` в `claude_runner`, что действительно было отдельной проблемой — и её починили (коммит `f5cc1dc`, `03-04_00-38_fix-thinking-and-readline-timeout.md`). Но корневую причину «неверное кодирование пути» тот отчёт не затрагивал — потому что тогда симптом был другой.

- **Исходники Claude Code CLI** (`/Users/ivan/Desktop/claude-sandbox/claude-code-sourcecode/`) — в проекте лежат **рядом**, в соседней папке. Функция `sanitizePath` в `utils/sessionStoragePortable.ts:311` однозначно описывает реальный алгоритм. Никто не заглядывал туда ни при написании спецификации, ни при реализации, ни при ревью.

## Рекомендации по исправлению

### 1. [CODE] Починить `_encode_project_path` — воспроизвести реальный алгоритм Claude Code CLI

**Файл:** `src/claude_manager/session_reader.py`, строки 58-60

**Что сделать:** заменить функцию на регулярное выражение, которое заменяет все не-буквенно-цифровые символы на дефис. Добавить в начало модуля `import re` (если ещё не импортирован), рядом с константами добавить скомпилированное регулярное выражение, в функции использовать его.

Как будет выглядеть:

```python
# В начале файла, рядом с другими константами:
# Регулярное выражение для санитации пути проекта — заменяет всё,
# что не буква и не цифра, на дефис. Повторяет sanitizePath() из
# Claude Code CLI (см. claude-code-sourcecode/utils/sessionStoragePortable.ts:311).
SANITIZE_PATH_PATTERN = re.compile(r"[^a-zA-Z0-9]")


def _encode_project_path(project_dir: str) -> str:
    """Кодирует путь проекта в формат имён папок Claude Code.
    
    Повторяет алгоритм sanitizePath() из Claude Code CLI: заменяет
    все не-буквенно-цифровые символы на дефис.
    """
    return SANITIZE_PATH_PATTERN.sub("-", project_dir)
```

**Почему так:** это точная копия алгоритма Claude Code CLI. В комментарии явно указан источник — чтобы будущий читатель мог сверить и обновить при изменении внешнего инструмента.

**Эффект после починки:** watcher снова начнёт видеть файлы сессий, команды `/sessions` и `/N` начнут работать корректно, WARNING перестанут сыпаться в лог. Симптомы 2, 3, 4 из цепочки причин исчезают автоматически.

## Рекомендации по предотвращению

Корневая причина архитектурная (проблема может повториться в любом другом модуле, который формирует путь/имя/идентификатор для внешней системы), поэтому приоритет — архитектурный принцип в CLAUDE.md. Порядок: `[ARCHITECTURE]` → `[DOC]` → `[SKILL]`.

### 1. [ARCHITECTURE] Добавить в `CLAUDE.md` принцип верификации контрактов с внешними системами

**Место:** раздел «Важные детали для разработки», новый пункт после упоминания протокола stream-json

**Что добавить** (формулировка принципа, не заплатка):

> **Контракты с внешними системами проверяются эмпирически, а не по догадке.** Когда код зависит от точного поведения внешнего инструмента или API (например, как Claude Code CLI формирует имя папки сессий, как Telegram Bot API экранирует HTML, как ОС обрабатывает файловые блокировки), это поведение должно быть либо получено из исходников внешнего инструмента (с указанием файла и строки в комментарии), либо проверено реальным вызовом и наблюдением результата. Предположения «скорее всего работает так» и «это очевидно» — неприемлемы для контрактов с внешними системами. Одна ошибка в догадке становится миной, которая срабатывает через месяцы.

### Потеря текста при гонке параллельных сообщений

**Файл:** `src/claude_manager/bot.py:396-404` — ветка обработки `process_manager.ProcessManagerError` в `_send_to_claude_and_respond`

Когда пользователь пишет два сообщения подряд в одной сессии, Telegram-клиент (через `python-telegram-bot` с `concurrent_updates(256)`) может запустить два параллельных обработчика. Первый возьмёт флаг занятости (`_busy_flags[session_id] = True`), второй получит `ProcessManagerError("уже занят")`. В ветке обработки этой ошибки **текст второго сообщения не сохраняется** — пользователь видит только уведомление «Claude ещё обрабатывает предыдущее сообщение. Подождите или /stop» и должен вручную отправить текст ещё раз. За 11 дней в логе 14 таких случаев — не массово, но это реальная потеря пользовательского ввода.

Возможные решения (каждое имеет свои trade-off):
- Очередь на уровне сессии: ставить сообщения в `asyncio.Queue`, отправлять по очереди. Плюс: никаких потерь. Минус: пользователь не видит «занят», не может выбрать `/stop`, поведение меняется.
- Сохранять текст в сообщении «занят» как «draft» и добавить кнопку «отправить сейчас»: пользователь видит что хотел отправить и может быстро повторить. Плюс: явное взаимодействие. Минус: нужна работа с inline-клавиатурами.
- Оставить как есть, но улучшить UX: показывать обратный отсчёт или очередь сообщений. Плюс: минимальные правки. Минус: всё равно теряется текст.

Рекомендую: отдельный root-cause или спек после основных фиксов.

### Проблема `_acquire_lock` в `main.py`

**Файл:** `src/claude_manager/main.py:50`

Функция `_acquire_lock()` открывает lock-файл в режиме `"w"`:
```python
lock_file = open(lock_path, "w")
```

Режим `"w"` использует флаг `O_TRUNC` — файл немедленно обнуляется. Только потом вызывается `fcntl.flock()`. Если в этот момент другая копия бота уже держит `flock` и PID в файле, вторая копия (обречённая упасть) всё равно успевает затереть PID первой. После неудачной `flock` в файле остаётся пусто.

Последствие: `watch_and_restart.sh`, который читает PID через `cat "$PID_FILE"`, может получить пустую строку и не суметь остановить правильный процесс. Это не теряет сообщения, но усложняет ручную диагностику.

Простой фикс — открыть в режиме `"a"` (append) или `"r+"` (read+write без truncate), проверить что файл открылся, после успешного `flock` вызвать `truncate(0)` и только тогда записать новый PID. Изменение на 4 строки.

### `watch_and_restart.sh` не знает про LaunchAgent

Если бот запущен через LaunchAgent (с `KeepAlive=true`), и пользователь руками запустит `watch_and_restart.sh`, скрипт пошлёт `SIGTERM` боту (прочитав PID из lock-файла). LaunchAgent через 10 секунд (`ThrottleInterval`) автоматически поднимет новый процесс, а `watch_and_restart.sh` параллельно попытается запустить свой. Получается гонка двух процессов за `flock` — один упадёт, но поведение непредсказуемо с точки зрения того, какой именно процесс обслуживает Telegram в итоге.

Правильный паттерн — `watch_and_restart.sh` должен сначала остановить LaunchAgent через `launchctl unload`, потом запустить свою версию, а при завершении — `launchctl load`. Это требует прав `launchctl`, которые не всегда есть.

Альтернатива: документировать в `CLAUDE.md` что `watch_and_restart.sh` и LaunchAgent взаимоисключающие — пользователь должен выбрать один способ и не смешивать.
