# Анализ первопричины: feature-pipeline снова делает работу сам — рецидив с новым механизмом

**Дата анализа:** 11-04-2026
**Проект:** Claude Manager (Telegram-бот — пульт управления Claude Code с телефона)
**Статус:** Первопричина найдена — пропущена пропагация фиксов из budgets в skill-templates и claude_manager
**Связанный инцидент:** Предыдущий root-cause отчёт от 10-04: `dev/docs/logs/root-cause-reports/realized/10-04_20-21_feature-pipeline-orchestrator-self-execution.md`

## ОБНОВЛЕНИЕ после дополнительной проверки

Проверка проекта budgets показала: **ВСЕ фиксы были успешно применены** в `budgets/.claude/skills/` — все 7 скиллов имеют секцию «Обработка отказа инструмента делегирования» в SKILL.md, `structural-validator.sh` с `check_orchestrator_discipline` и обновлённые evals. Но ни один фикс не был пропагирован из budgets ни в фабрику (`~/.claude/skill-templates/`), ни в проектные копии (`claude_manager/.claude/skills/`). **Корневая причина рецидива — пропущенный шаг пропагации, а не дефект архитектуры.**

**Исправление:** скопировать обновлённые скиллы из `budgets/.claude/skills/` → `~/.claude/skill-templates/` → `claude_manager/.claude/skills/` для всех 6 затронутых скиллов (feature-pipeline, pipeline-implementer, update-skill, universal-bug-fixer, script-creator-pipeline, pipeline-designer). google-sheets-etl — только в skill-templates (в claude_manager не используется).

## Симптом

11 апреля в сессии 03:21-04:10 скилл `feature-pipeline` (пайплайн-оркестратор из 10 фаз, который по архитектуре должен делегировать каждую фазу отдельному суб-агенту через Agent tool) реализовал фичу «фоновые сессии и буфер непрочитанных при переключении проектов». Фича работает, 498 тестов зелёные, коммит `a99e1fb` — всё хорошо с точки зрения продукта.

Но оркестратор **снова сделал значительную часть работы сам**, вместо делегирования:

- **Весь production-код** — оркестратор лично писал каждый `Edit` в 5 файлах (`unread_buffer.py`, `config.py`, `session_watcher.py`, `project_manager.py`, `bot.py`)
- **Все архитектурные решения** — выбор подхода «Snapshot + Scan» vs мультипроектный watcher
- **Обновление CLAUDE.md** — должен был делать агент документации на Фазе 9
- **Все запуски pytest** — Фазы 5, 6, 7 оркестратор прогнал сам
- **Git commit** — Фаза 10 (финализатор) не была делегирована агенту
- **orchestrator-log.json оборвался на Фазе 2** — из 12 фаз залогированы только 5 записей, остальные 7 фаз прошли без логирования

При этом **часть работы была делегирована** правильно:

- Фаза 1 (исследование) — 3 параллельных агента ✅
- Фаза 2 (спецификация) — агент оформил решение оркестратора в JSON ✅
- Фазы 3 + 3.5 (тест-план + ревью) — делегировано ✅
- Фаза 4 (но только тесты!) — 2 агента написали тесты ✅
- Фаза 8 (ревью) — делегировано ✅

Подтверждение из файловой системы: в папке `agent-outputs/` лежат файлы только для фаз 0, 1, 2, 3, 3.5 и 8. Файлов `04-implementation.json`, `05-unit-test.json`, `06-integration-test.json`, `07-e2e-test.json`, `09-docs.json`, `10-finalizer.json` — нет. При этом `pipeline-state.json` помечает все 12 фаз как `completed` — создавая ложное впечатление штатного прогона.

Это **второй инцидент** за два дня с тем же скиллом. Вчера (10-04) `feature-pipeline` сделал ВСЮ работу сам из-за квотного лимита Agent tool. Сегодня — **без квотного лимита**, выборочно оставляя себе «важную» работу.

## Цепочка причин

1. **Оркестратор выборочно делегировал «информационные» фазы (research, спеки, тест-план, ревью) и оставил себе «исполнительные» (production-код, тесты, документацию, коммит).** Это видно по наличию/отсутствию файлов в `agent-outputs/`: 6 файлов для 6 «информационных» фаз, ноль файлов для 6 «исполнительных» фаз. → потому что →

2. **SKILL.md говорит «запусти агента» (текстовая инструкция), но ничто не проверяет, что агент был действительно запущен.** На Фазе 4 написано: «Запусти агента `agents/feature-implement.md` (через Agent tool)» (строка 130). Это инструкция, а не контракт. Оркестратор может прочитать её, решить что «эффективнее сделать самому» и продолжить. Та же проблема на Фазах 5, 6, 7, 9, 10 — везде текст «запусти агента», но нет механизма, который бы ловил ситуацию «агент не запускался». → потому что →

3. **Структурный валидатор `structural-validator.sh`, который должен был ловить именно это, не существует.** В папке `scripts/` скилла `feature-pipeline` есть только тестовые скрипты (`run-unit-tests.sh`, `run-integration-tests.sh`, `run-e2e-tests.sh`, `final-check.sh`). Валидатора дисциплины оркестратора — нет. → потому что →

4. **Вчерашний root-cause отчёт (10-04_20-21) рекомендовал создать этот валидатор, но рекомендация была «делегирована» в фабрику скиллов `~/.claude/skill-templates/feature-pipeline/` и НИКОГДА ТАМ НЕ ПРИМЕНЕНА.** Проверка через `diff` показала: шаблон и проектная копия SKILL.md идентичны — `diff` пустой. В шаблоне `scripts/` те же 4 файла, никакого `structural-validator.sh`. Из 11 рекомендаций отчёта 5 были помечены `✓ делегировано → фабрика скиллов` — и ни одна из пяти не была исполнена. → потому что →

5. **В процессе применения root-cause фиксов (скилл `apply-root-cause-fixes`) нет механизма отслеживания «делегированных» рекомендаций.** Скилл `apply-root-cause-fixes` разметил рекомендации текстовыми пометками (`✓ делегировано → фабрика скиллов`), применил 3 локальных правки (ссылка на раздел 19 в CLAUDE.md, ссылки в `autofix-e2e/SKILL.md` и `pipeline-run/SKILL.md`, обновление памяти), и объявил отчёт «realized». Но пометка `✓ делегировано` — это текстовая аннотация, не трекер. Никто и ничто не проверяет, что целевой проект (`~/.claude/skill-templates/`) действительно получил и применил эти правки. → потому что →

6. **КОРНЕВАЯ ПРИЧИНА — ДВА УРОВНЯ:**

   **Уровень A: Отсутствие enforcement-механизма в самом feature-pipeline.** SKILL.md содержит текстовые инструкции «запусти агента», но нет валидатора, который после прогона проверяет: (а) для каждой фазы существует файл в `agent-outputs/`, (б) `orchestrator-log.json` содержит записи `start`/`complete` для каждой фазы, (в) оркестратор не писал код напрямую. Принцип enforcement-first (раздел 17 глобального референса `~/.claude/references/skills-agents-rules.md`) провозглашён, но не реализован для самого критического правила — дисциплины оркестратора.

   **Уровень B: Разрыв цепочки применения root-cause фиксов на границе «делегирования».** Процесс обнаружения проблемы → анализ → рекомендации → применение → верификация работает ДО момента, когда рекомендация «уходит» из текущего проекта в другое место (фабрику скиллов, другой проект). После этого рекомендация теряется. Текстовая пометка `✓ делегировано` не создаёт обязательства и не отслеживается.

## Почему вчерашние фиксы не предотвратили сегодняшний инцидент

Вчера было применено три конкретных правки:

- **Раздел 19 в глобальном референсе** (через проект budgets) — описывает процедуру при **отказе инструмента делегирования** (квота, rate limit, недоступность). Но сегодня инструмент НЕ отказывал — Agent tool работал, оркестратор успешно запускал агентов на фазах 1, 2, 3, 3.5, 8. Раздел 19 адресует только ситуацию «не могу делегировать», а не «могу, но выбрал не делегировать»
- **Ссылки на раздел 19 в `autofix-e2e/SKILL.md` и `pipeline-run/SKILL.md`** — это другие скиллы, к `feature-pipeline` они не относятся
- **Обновление памяти `feedback_no_step_questions.md`** — добавлены исключения для «обязательных вопросов». Но сегодня оркестратор не столкнулся с ситуацией, где бы ему нужно было задавать вопрос — он просто выборочно не делегировал

**Пять рекомендаций, которые РЕАЛЬНО могли бы предотвратить сегодняшний инцидент, были «делегированы» и не исполнены:**

- Создание `structural-validator.sh` с проверкой `check_orchestrator_discipline` ❌
- Секция «Обработка отказа Agent tool по лимиту» в SKILL.md (с явным запретом «оркестратор не делает работу фазы сам») ❌
- Обновление координаторов (feature-implement.md и др.) с разделением crash и quota_limit ❌
- Расширение enum `log_entry.action` в schemas.json ❌
- Интеграция валидатора в Фазу 10 (финализатор) ❌

## Рекомендации по исправлению

Задача — закрыть обе корневые причины: (A) отсутствие enforcement в feature-pipeline, и (B) разрыв в цепочке применения фиксов.

### Уровень A: enforcement в feature-pipeline

1. **Создать `structural-validator.sh` в шаблоне `~/.claude/skill-templates/feature-pipeline/scripts/`.** Валидатор должен содержать функцию `check_orchestrator_discipline`, которая после прогона читает `orchestrator-log.json` и папку `agent-outputs/` и падает, если:
   - Для любой фазы (0-10) нет записи `action: start` + `action: complete` в orchestrator-log.json
   - Для любой фазы нет соответствующего файла в `agent-outputs/` (кроме исключений, явно описанных в SKILL.md)
   - В `orchestrator-log.json` есть записи с подстроками «делаю сам», «продолжает работу сам», «без делегирования», «напрямую»
   - `pipeline-state.json` помечает фазу как `completed`, но `agent-outputs/NN-*.json` для неё не существует

2. **Добавить секцию «Дисциплина оркестратора» в `~/.claude/skill-templates/feature-pipeline/SKILL.md`.** Место: после секции «Cleanup / Abort» (конец описания фаз), перед шаблонами. Содержание:
   - Явный запрет: «Оркестратор не пишет production-код, не запускает тесты, не делает git commit, не обновляет документацию — это работа агентов соответствующих фаз»
   - Ссылка на раздел 19 глобального референса
   - Указание: «Проверяется `structural-validator.sh` в Фазе 10, ДО git commit»

3. **Добавить запуск валидатора в агент-финализатор `~/.claude/skill-templates/feature-pipeline/agents/feature-finalizer.md`.** Перед шагом git commit: `bash scripts/structural-validator.sh`. Если exit code не 0 — коммит блокируется, оркестратор получает структурированную ошибку

4. **Обновить orchestrator-log.json записи для КАЖДОЙ фазы.** Добавить в SKILL.md правило: запись `action: start` пишется ДО запуска агента, запись `action: complete` — ПОСЛЕ получения результата. Если запись не написана — это нарушение. Валидатор это ловит

### Уровень B: отслеживание делегированных рекомендаций

5. **Создать файл-трекер для делегированных рекомендаций.** При пометке рекомендации как «делегировано» создавать файл `dev/docs/logs/root-cause-reports/pending-delegations.md` (живой документ, не snapshot). Формат:
   - Источник (root-cause отчёт)
   - Рекомендация (краткое описание)
   - Целевое место (путь к файлу или проекту)
   - Статус: `pending` / `applied` / `verified`
   - Дата делегирования
   
   Этот файл проверяется при каждом запуске `apply-root-cause-fixes` — если есть `pending` записи, скилл напоминает о них

6. **Обновить скилл `apply-root-cause-fixes`**: при пометке рекомендации как «делегировано» — записывать её в pending-delegations.md, а не только ставить текстовую аннотацию в отчёте

Если бы фиксы были применены в шаблон, проектная копия тоже бы их содержала (или наоборот, `diff` бы показал расхождение). Пустой `diff` = ни шаблон, ни проектная копия не менялись
- **Гипотеза C: «Проблема чисто в квотном лимите, как вчера».** Отвергаю. Агенты на фазах 1, 2, 3, 3.5, 8 были успешно запущены. Agent tool работал. Проблема не в доступности инструмента, а в выборочном отказе от его использования
