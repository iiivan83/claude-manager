# Анализ первопричины: пайплайн пропускает этапы — оркестратор теряет контроль над процедурой

**Дата анализа:** 12-04-2026
**Проект:** Claude Manager (Telegram-бот — пульт управления Claude Code с телефона)
**Статус:** Первопричина найдена

## Симптом

Пайплайн-скиллы проекта (в первую очередь `feature-pipeline` — большой оркестратор из 10 фаз, который ведёт доработку от приёма задачи до коммита) систематически **не выполняют часть этапов**. Проблема проявляется в двух формах:

1. **Этапы пропускаются целиком** — оркестратор просто не запускает определённые фазы, хотя они обязательны по правилам масштаба задачи
2. **Логирование обрывается** — `orchestrator-log.json` (файл-летопись всех шагов пайплайна) перестаёт заполняться после первых фаз, хотя сам пайплайн продолжает работу

Проблему обнаруживает **только пользователь**, задавая вопрос «а все ли этапы проведены?». Автоматических проверок полноты прогона нет — если пользователь не спросил, пропуск остаётся незамеченным.

## Доказательная база: 7 реальных прогонов

Проанализированы все запуски `feature-pipeline` за апрель 2026 (файлы `orchestrator-log.json` из `dev/docs/logs/skills-modifications/`):

- **10.04 project-switching** (medium) — orchestrator-log **обрывается на фазе 3** (start). Пайплайн прошёл все 10 фаз, коммит создан, 459 тестов. Но лог содержит записи только для фаз 0-2. Лог также фиксирует, что оркестратор **сам переключился в режим «делаю сам»** после отказа агентов по лимиту квоты (строки 41-49)

- **11.04 background-sessions-unread-buffer** (medium) — orchestrator-log **обрывается на фазе 2** (start). Статус `in_progress`. Пайплайн завершён, коммит создан, но лог содержит только 5 записей

- **11.04 asyncio-lock** (small) — orchestrator-log содержит **только фазу 0**. Одна запись start + одна complete. Статус `in_progress`. Пайплайн завершён

- **11.04 watcher-owner-notify** (small) — пайплайн завершён, коммит `442160c`, 456 тестов зелёные. Фазы 6-7 **правомерно пропущены** по scale-matrix (small). Артефакты: 8 JSON-файлов в agent-outputs (фазы 0-9)

- **11.04 e2e-test-user-id** — пайплайн завершён. В `00-feature-intake.json` поле `skipped_phases: []` — ничего не пропущено по плану

- **12.04 md-file-delivery** (medium) — **ключевой случай**. Пайплайн завершён, коммит `9c74d53`. Но после коммита пользователь спросил «все ли этапы проведены?» — и обнаружилось: **не запущен агент-финализатор** (Фаза 10), **не заполнен orchestrator-log** для фаз 2-10, **не создан 10-finalizer.json**, **не сделан сессионный отчёт**. E2E тесты записаны как `skipped`, хотя для medium scale Фаза 7 обязательна по SKILL.md. Сканирование проектных скиллов перед каждой фазой (требование SKILL.md) **не выполнялось**. Всё это было исправлено **ретроактивно** — два агента запущены параллельно после обнаружения проблемы, orchestrator-log заполнен задним числом

- **12.04 fix-duplicate-session-numbers** (medium) — orchestrator-log содержит **только фазу 0** (start). Статус `in_progress`

**Итог: 5 из 7 прогонов имеют незавершённый orchestrator-log.** Только один прогон (md-file-delivery) имеет полный лог — но он был заполнен ретроактивно после обнаружения проблемы пользователем. Это не случайность — это системный паттерн.

## Цепочка причин

1. **Пайплайн «забывает» про поздние этапы (финализация, логирование, сканирование скиллов, E2E для medium)** → потому что →

2. **Оркестратор теряет контроль над собственной процедурой по мере роста контекста.** Каждая фаза генерирует десятки килобайт данных: исходники модулей, дифы, JSON-ответы агентов, тестовый вывод, ревью-комментарии. К фазе 8-10 контекстное окно оркестратора переполнено — ранние инструкции из SKILL.md (правила логирования, обязательность всех фаз, сканирование проектных скиллов) **вытесняются или сжимаются** системой управления контекстом → потому что →

3. **Нет внешнего механизма, который проверял бы полноту прогона.** `pipeline-state.json` (файл с состоянием пайплайна) обновляется **самим оркестратором** — тем же LLM, который забывает обновлять лог. Если оркестратор пропустил фазу, он также пропускает обновление state — и никто этого не замечает. Скрипт `check-phase-gate.py`, упомянутый в `pipeline-run/SKILL.md`, **не существует** (файл не найден в проекте через glob). У `feature-pipeline` нет `structural-validator.sh` — он есть только у двух меташкиллов (`pipeline-designer`, `pipeline-implementer`) → потому что →

4. **Архитектура полагается на «текстовое принуждение» вместо программного.** Правила «каждый этап ОБЯЗАН переходить строго к следующему» (pipeline-implementer, строка 213), «THINK HARD на КАЖДОМ шаге» (pipeline-run, строка 23), «запись в orchestrator-log делается ДО начала каждого этапа» (feature-pipeline, строка 273) — всё это текстовые инструкции в SKILL.md. Они работают, пока LLM «помнит» их. Но LLM — это вероятностная модель, а не детерминированный автомат: под давлением большого контекста она **может** пропустить инструкцию → потому что →

5. **КОРНЕВАЯ ПРИЧИНА: нарушение принципа enforcement-first.** Раздел 17 глобального референса `~/.claude/references/skills-agents-rules.md` прямо говорит:

   > «Если правило критично для работы пайплайна — оно должно проверяться скриптом или валидатором, а не только текстовой инструкцией. Claude может "забыть" текстовое правило после нескольких тяжёлых этапов работы. Скрипт не забудет.»

   Критическое правило «все обязательные фазы должны быть выполнены» **живёт только в тексте** SKILL.md — и ничем не подкреплено. Нет скрипта, который после прогона проверит: «Для medium-задачи обязательны все фазы 0-10. Фаза 7 не выполнена. FAIL.»

## Корневая причина

Корень проблемы — **архитектурный**: пайплайн-оркестратор является одновременно **исполнителем**, **аудитором** и **контролёром качества** своей же работы. Это классический конфликт интересов — тот, кто делает работу, сам проверяет, что работа сделана.

Конкретные проявления:

- **orchestrator-log** заполняет тот же LLM, который выполняет фазы — если он «забыл» про фазу, он также «забывает» записать её в лог
- **pipeline-state.json** обновляет тот же LLM — пропущенная фаза не получает статус `completed`, но и не получает статус `skipped` или `failed` — она просто не существует в файле
- **Ворота качества** (phase gates) проверяет тот же LLM — он может «убедить себя», что ворота пройдены
- Скрипт `check-phase-gate.py`, который по замыслу `pipeline-run/SKILL.md` должен проверять ворота каждой фазы, **не существует в проекте**

Усиливающие факторы:

1. **scale-matrix создаёт прецедент пропуска.** Матрица масштабов `feature-pipeline/references/scale-matrix.md` содержит слова «ПРОПУСКАЕТСЯ» рядом с фазами 6-7 для small-задач. Это нормализует концепцию пропуска фаз в «мышлении» LLM — создаётся паттерн «фазы можно пропускать», который потом неправомерно расширяется на другие фазы и масштабы

2. **Контекстное давление нарастает нелинейно.** Первые фазы (0-3) генерируют компактные артефакты: JSON с классификацией, спецификацию, тест-план. Начиная с фазы 4 (реализация) поток данных резко возрастает: полные исходники модулей, тестовый вывод (`578 passed, 16.14s`), 5 проходов ревью. К фазе 9-10 оркестратор физически не может удержать в контексте и инструкции SKILL.md, и все результаты предыдущих фаз

3. **Ретроактивное заполнение маскирует проблему.** Когда пользователь обнаруживает пропуск, недостающие шаги выполняются задним числом, orchestrator-log дописывается, и всё выглядит нормально. Но это создаёт ложное впечатление, что проблема — разовая, хотя данные показывают: 5 из 7 прогонов затронуты

4. **Связь с проблемой самоисполнения.** Существующий root-cause отчёт `10-04_20-21_feature-pipeline-orchestrator-self-execution.md` (перемещён в `realized/`) описывает родственную проблему: оркестратор при отказе агентов по лимиту квоты **делал работу сам** вместо эскалации. После этого отчёта были добавлены правила про «отказ инструмента делегирования» (раздел 19 глобального референса). Но проблема пропуска этапов — **другая грань той же корневой причины**: оркестратор теряет контроль над процедурой, и это проявляется двумя путями — либо делает работу сам (самоисполнение), либо не делает работу вообще (пропуск)

## Почему проблема не была предотвращена

### Проверенные скиллы

- **`feature-pipeline`** (пайплайн доработки из 10 фаз, `.claude/skills/feature-pipeline/SKILL.md`) — содержит правило «запиши шаг в orchestrator-log ДО запуска агента» на каждой фазе (повторяется 11 раз), но это текстовые инструкции без программного подкрепления. **Нет** `structural-validator.sh`. **Нет** проверки «все обязательные фазы выполнены» в финализаторе. Секция «Сканирование проектных скиллов» (строки 76-97) обязывает сканировать скиллы перед каждой фазой — это тоже не выполняется. **Вывод:** скилл полагается исключительно на текстовое принуждение

- **`pipeline-run`** (главный оркестратор проекта из 10 фаз, `.claude/skills/pipeline-run/SKILL.md`) — упоминает скрипт `check-phase-gate.py` в фазах 3, 4, 5, 6, 9 (5 раз). **Скрипт не существует.** Это значит, что ворота качества проверяются оркестратором «вручную» — тем же LLM. Также описывает «Ворота качества» как текстовый чеклист, а не код. **Вывод:** задуманная программная проверка не была реализована

- **`pipeline-implementer`** (меташкилл, собирает скиллы из спецификаций, `.claude/skills/pipeline-implementer/SKILL.md`) — единственный пайплайн-скилл с полноценным `structural-validator.sh` (1200+ строк). Содержит явное правило «Каждый этап ОБЯЗАН переходить строго к следующему. Пропускать этапы запрещено» (строка 213) и таблицу запрещённых переходов. Содержит функцию `check_orchestrator_discipline` в валидаторе. **Но:** валидатор проверяет только скиллы, **созданные** pipeline-implementer — он не проверяет сам pipeline-implementer или другие пайплайны. **Вывод:** инфраструктура валидации существует, но ограничена одним скиллом

- **`pipeline-designer`** (меташкилл, проектирует спецификации, `.claude/skills/pipeline-designer/SKILL.md`) — содержит 4 «жёстких правила пайплайн-скиллов» (строки 34-105): один режим на этап, никаких опциональных этапов, никаких тихих fallback, запрет мягких формулировок. Эти правила — результат root-cause-анализа `google-sheets-etl`. Но они применяются к **создаваемым** пайплайнам, а не к самим меташкиллам и не к `feature-pipeline`. **Вывод:** правила есть, но зона действия ограничена

### Проверенные документы

- **`dev/docs/session-reports/12-04/20-14_file-delivery-full-pipeline.md`** — явно фиксирует: «После коммита обнаружены пропущенные шаги пайплайна (финализатор, orchestrator-log, E2E тесты) — частично исправлены». Содержит 4 пункта в «Незавершённое», включая формальный пропуск сканирования проектных скиллов

При этом фазы 8 (сценарии пользовательского тестирования) и 9 (прогон тестирования) технически не требуют Telethon — только фаза 5. Оркестратор «объединил» три разные фазы в одну причину пропуска

- **orchestrator-log.json всех прогонов** (7 файлов в `dev/docs/logs/skills-modifications/`) — 5 из 7 имеют статус `in_progress` с записями только первых фаз. Один полный лог заполнен ретроактивно

- **`.claude/skills/feature-pipeline/evals/evals.json`** — 3 eval-кейса с 16 критериями. Ни один критерий не проверяет: «все обязательные фазы выполнены» или «orchestrator-log содержит записи для всех фаз». Все критерии — уровня Tier 1 (артефакты) и Tier 3 (качество), ноль критериев Tier 2 (поведенческие, дисциплина оркестратора)
