# Анализ первопричины: декоративный вызов Telegram API роняет обработчик сообщений

**Дата анализа:** 13-04-2026
**Проект:** Claude Manager — Telegram-бот для управления Claude Code с телефона
**Статус:** Первопричина найдена

## Симптом

Бот перестал отвечать на текстовые сообщения, фотографии и файлы. При этом команды (`/new`, `/stop`, `/56`) продолжали работать. Пользователь отправил "Ну как ты?" — тишина. Параллельно `error.log` разросся до ~1 МБ из-за повторяющихся ошибок.

## Цепочка причин

Обнаружены **три отдельных отклонения**, связанных общим контекстом. Каждое описано ниже со своей цепочкой.

### Отклонение 1: send_chat_action роняет обработчик (основное)

1. **Бот молчит на текстовые сообщения, фотографии и файлы** → потому что →
2. **Обработчики `handle_message`, `handle_photo`, `handle_document` падают целиком** — функция `_send_to_claude_and_respond` (отправляет текст в Claude и ждёт ответ) не вызывается → потому что →
3. **Вызов `send_chat_action(ChatAction.TYPING)` бросает исключение TimedOut** — это вызов, который показывает индикатор "печатает..." в Telegram. Чисто декоративный — пользователь и без него получит ответ → потому что →
4. **Этот вызов не обёрнут в try/except** — исключение поднимается на уровень фреймворка python-telegram-bot, который логирует "No error handlers are registered" и глотает ошибку молча → потому что →
5. **Спецификация бота (`bot_spec.md`) не разделяет вызовы API на критические и декоративные** — шаг 6 алгоритма `handle_message` просто говорит "включить индикатор печатает" без указания на обработку ошибок. Раздел "Обработка ошибок" подробно описывает обработку BadRequest, RetryAfter, TimedOut для _отправки сообщений_, но ни словом не упоминает `send_chat_action` → потому что →
6. **КОРНЕВАЯ ПРИЧИНА: в архитектуре проекта (CLAUDE.md) нет принципа разделения вызовов внешних API по критичности** — нет правила, которое бы говорило: "декоративный вызов не должен прерывать основную логику". Агент, создававший спецификацию, не мог знать об этом различии, потому что оно нигде не описано.

### Отклонение 2: отсутствие глобального error handler

1. **Ошибка send_chat_action не поймана ничем** → потому что →
2. **В боте не зарегистрирован глобальный обработчик ошибок** — функция `setup_bot()` (`bot.py`:989-1003) создаёт Application, регистрирует хэндлеры команд и сообщений, но не вызывает `application.add_error_handler(...)`. Библиотека python-telegram-bot предоставляет этот механизм как последний рубеж обороны для любых необработанных исключений в обработчиках → потому что →
3. **Ни спецификация бота, ни CLAUDE.md не упоминают необходимость глобального error handler** — при проектировании бота не учтён сценарий "необработанное исключение в любом handler должно быть поймано и залогировано, а не проглочено молча".

### Отклонение 3: session_watcher спамит error.log

1. **`error.log` растёт до ~1 МБ** → потому что →
2. **Watcher каждые 2 секунды пытается прочитать JSONL-файлы трёх сессий, которых нет на диске** — `session_reader.get_session_messages()` (`session_reader.py`:269-271) проверяет `os.path.exists()`, находит что файла нет, логирует `logger.warning("Файл сессии не найден: %s")` и возвращает пустой список → потому что →
3. **Функция `_get_sessions_to_monitor()` (`session_watcher.py`:105) объединяет два источника** — файлы с диска (через `session_reader.get_recent_sessions()`) и записи из дневного реестра (`daily_session_registry.get_all_today_sessions()`). Файлы с диска — актуальные, проблем нет. А вот реестр содержит записи о сессиях, чьи файлы уже удалены → потому что →
4. **Дневной реестр (`daily_session_registry`) не валидирует, существуют ли JSONL-файлы** — он хранит пары {номер: session_id} и не проверяет, жив ли файл на диске. Очистка stale-ключей в `_poll_sessions()` (строки 211-216) чистит только `_seen_message_counts`, но проблемные ID приходят из `_get_sessions_to_monitor()`, а не из `_seen_message_counts` → потому что →
5. **Нет механизма валидации или очистки реестра от записей с несуществующими файлами** — ни спецификация `session_watcher_spec.md`, ни `daily_session_registry_spec.md` не описывают этот сценарий.

## Корневая причина

Корень — **отсутствие архитектурного принципа** в CLAUDE.md:

**Нет разделения вызовов внешних API по критичности.** Когда обработчик сообщения вызывает Telegram API, код не различает: "этот вызов обязателен для результата" vs "этот вызов декоративный, его провал ничего не ломает". В `bot.py` уже есть обёртка `_send_telegram_message()` (строка 250) с полной обработкой ошибок — но она используется только для отправки сообщений пользователю. Декоративные вызовы (`send_chat_action`) идут напрямую через `context.bot.*` — без защиты.

Это не единичный баг — это системный пробел. Любой новый обработчик, который добавит `send_chat_action` или другой необязательный вызов API перед основной логикой, получит ту же уязвимость.

**Конкретные места в коде:**
- `bot.py`:887 — `handle_message`: `await context.bot.send_chat_action(chat_id, ChatAction.TYPING)` — без try/except
- `bot.py`:920 — `handle_photo`: аналогично
- `bot.py`:958 — `handle_document`: аналогично
- `bot.py`:989-1003 — `setup_bot()`: нет вызова `application.add_error_handler()`

## Что НЕ является проблемой (проверено)

- **Busy flag НЕ утекает.** Флаг занятости (busy flag) устанавливается внутри `process_manager.send_message()` (`process_manager.py`:445-447). Поскольку `send_chat_action` падает ДО вызова `_send_to_claude_and_respond` → `process_manager.send_message()` никогда не вызывается → busy flag не устанавливается → утечки нет. Событие "Процесс уже обрабатывает запрос" (23:50:12 в error.log) — отдельный инцидент, связанный с убитым процессом Claude CLI (23:49:43), не с send_chat_action.
- **Другие прямые вызовы Telegram API** (`_application.bot.send_message` на строках 206, 305; `_application.bot.send_document` на строке 316; `_application.bot.get_file` на строке 360) — вызываются только изнутри функций, которые уже обёрнуты в try/except на уровне вызывающего кода (`_send_to_claude_and_respond`, `_download_and_save_file`). Их падение не оставляет обработчик без защиты.

## Почему проблема не была предотвращена

### Проверенные скиллы

- **`spec-module`** (создаёт детальные спецификации модулей) — написал `bot_spec.md`. Шаг 6 алгоритма `handle_message` упоминает `send_chat_action`, но **нет пункта** про обработку ошибок декоративных вызовов. Раздел "Обработка ошибок" покрывает ошибки `_send_telegram_message`, но не `send_chat_action`. Тест-план содержит `test_typing_indicator_shown` (проверяет, что индикатор вызван), но **нет теста** "при ошибке send_chat_action обработчик продолжает работу". **Вывод:** спецификация не классифицирует API-вызовы по критичности — пробел.

- **`implement-module`** (реализует код по спецификации) — написал `bot.py`. Следовал алгоритму спецификации буквально: "шаг 6 — включить индикатор" → один `await` без try/except. **Вывод:** скилл действовал корректно — реализовал то, что было в спеке. Проблема была в спеке.

- **`review-code`** (4-проходное ревью) — использует чеклисты из `review-checklists.md`. В чеклистах **нет пункта** "декоративные вызовы API не должны ронять handler" или "все await-вызовы Telegram API в обработчиках защищены". **Вывод:** чеклист ревью не покрывает этот сценарий — пробел.

- **`feature-pipeline`** (полный цикл доработки) — имеет 5-проходное ревью, но использует те же чеклисты. **Тот же пробел.**

### Проверенные документы

- **`dev/docs/specs/realised/bot_spec.md`** (спецификация транспортного слоя бота) — раздел "Обработка ошибок" (строки 548-563) подробно описывает 12 типов ошибок, но все они относятся к отправке/получению сообщений. Ошибки `send_chat_action` не упоминаются. Глобальный error handler не упоминается.

- **`dev/docs/review-checklists.md`** (чеклисты ревью) — пункт про "визуальные индикаторы" (строка 53) проверяет только соответствие иконок BRD, а не устойчивость кода к ошибкам при их показе.

- **`CLAUDE.md`** — раздел "Обработка ошибок" говорит "Все ошибки от внешних систем (Telegram API, Claude CLI, файловая система) — ловятся и обрабатываются". Это правильное правило, но слишком общее — не различает критические и декоративные вызовы. Скилл, пишущий спецификацию, интерпретировал его для `_send_telegram_message` (критический), но пропустил `send_chat_action` (декоративный).

- **`dev/docs/specs/realised/session_watcher_spec.md`** (спецификация мониторинга сессий) — описывает `_get_sessions_to_monitor()` как объединение двух источников, но **не описывает** сценарий "что если ID из реестра ссылается на несуществующий файл".

## Рекомендации по исправлению

1. **Обернуть все 3 вызова `send_chat_action` в try/except** — `bot.py`:887, 920, 958. При ошибке — логировать на уровне debug (это нормальная ситуация при нагрузке), продолжить выполнение обработчика. Код:
   ```python
   try:
       await context.bot.send_chat_action(chat_id, ChatAction.TYPING)
   except Exception:
       logger.debug("Не удалось показать индикатор TYPING (chat_id=%d)", chat_id)
   ```

2. **Добавить глобальный error handler** — в `setup_bot()` (`bot.py`:1001, перед `return application`) вызвать `application.add_error_handler(error_handler)`. Обработчик должен логировать ошибку и отправить пользователю "Произошла внутренняя ошибка". Это последний рубеж для любых необработанных исключений.

3. **Валидировать сессии из реестра в watcher** — в `_get_sessions_to_monitor()` (`session_watcher.py`:105) для session_id из daily_registry проверять существование JSONL-файла на диске перед добавлением в список мониторинга. Если файла нет — не добавлять. Можно также понизить уровень лога в `session_reader.get_session_messages()` (`session_reader.py`:271) с `warning` до `debug` для "Файл сессии не найден" — это ожидаемая ситуация.

## Рекомендации по предотвращению

1. **[ARCHITECTURE]** Добавить в CLAUDE.md раздел "Разделение вызовов API по критичности":

   > **Критические vs декоративные вызовы внешних API.** Вызовы Telegram API (и любых внешних систем) в обработчиках делятся на два класса:
   > - **Критические** — результат нужен для продолжения работы (отправка сообщения, получение файла). Обрабатываются с ретраями через `_send_telegram_message()` или отдельный try/except с уведомлением пользователя.
   > - **Декоративные** — индикаторы, статусы, подсказки. Их провал не должен прерывать основную логику. Оборачиваются в try/except с логированием на уровне debug — без ретраев, без уведомления пользователя.
   >
   > Правило: необязательный вызов API **никогда** не ставится в основной поток без try/except. Глобальный error handler (`add_error_handler`) регистрируется всегда — как последний рубеж.

   **Место:** после раздела "Обработка ошибок" в "Правилах разработки".

2. **[DOC]** Обновить `dev/docs/specs/realised/bot_spec.md`:
   - В разделе "Обработка ошибок" добавить: "Ошибка `send_chat_action` — оборачивается в try/except, логируется на debug, обработка продолжается"
   - Добавить описание глобального error handler
   - В тест-плане добавить: `test_send_chat_action_failure_continues` — при ошибке send_chat_action обработчик продолжает работу

3. **[DOC]** Обновить `dev/docs/review-checklists.md`:
   - Добавить пункт в проход 2 (Безопасность): "Все await-вызовы внешних API в обработчиках: критические защищены через _send_telegram_message или try/except с ретраями; декоративные обёрнуты в try/except без ретраев"

4. **[SKILL]** В скилле `spec-module` — при описании алгоритма обработчика: "Перед описанием вызовов внешних API см. раздел 'Критические vs декоративные вызовы' в CLAUDE.md. Классифицируй каждый вызов и опиши обработку ошибок соответственно."

## Верификация выводов

- **Перечитал `process_manager.py`** — убедился, что busy flag устанавливается только внутри `send_message()` (строка 447: `_prepare_for_send(session_id)` → `_busy_flags[session_id] = True`). Если `send_chat_action` падает до вызова `_send_to_claude_and_respond`, busy flag не устанавливается. Подтверждено.

- **Перечитал `bot.py`:989-1003** — `setup_bot()` вызывает `ApplicationBuilder().token().post_init().concurrent_updates(256).build()`, затем `_register_handlers(application)`, но **нигде** не вызывает `add_error_handler`. Подтверждено.

- **Перечитал `session_watcher.py`:206-225** — `_poll_sessions()` действительно чистит `_seen_message_counts` от stale-ключей, но проблемные ID приходят из `_get_sessions_to_monitor()` (строка 208), которая сначала получает ID, а очистка происходит потом. Для ID из daily_registry, у которых нет файла, `_check_session` выбрасывает warning каждые 2 секунды. Подтверждено.

### Альтернативные объяснения

- **"Проблема в Telegram API — он слишком часто возвращает TimedOut"** — нет, TimedOut — нормальное поведение при нагрузке (одновременно шли E2E тесты, 21 запрос от одного токена). Код должен обрабатывать ожидаемые ошибки API, а не полагаться на идеальную сеть. Отклонено.

- **"Проблема в E2E тестах, которые создали нагрузку"** — E2E тесты были триггером (повышенная нагрузка на Telegram API), но не причиной. `send_chat_action` без try/except упадёт при любом сбое сети, таймауте или throttling — не только во время E2E тестов. Основная уязвимость существует независимо от E2E. Отклонено.

- **"Достаточно просто обернуть send_chat_action в try/except, без архитектурного принципа"** — это заплатка, не решение. Без принципа в CLAUDE.md следующий обработчик (или доработка существующего) может добавить другой декоративный вызов без защиты — и проблема повторится. Архитектурный принцип нужен, чтобы агенты знали правило заранее. Отклонено как недостаточное.
