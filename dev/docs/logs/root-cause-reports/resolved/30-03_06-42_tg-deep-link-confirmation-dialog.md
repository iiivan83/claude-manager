# Анализ первопричины: клик по номеру сессии показывает диалог подтверждения вместо переключения

**Дата анализа:** 30-03-2026
**Статус:** Первопричина найдена

## Симптом

Когда бот присылает сообщение из другой сессии (не из той, к которой ты сейчас подключён), номер сессии отображается как кликабельная ссылка — например, **#12**. По задумке, нажатие на эту ссылку должно мгновенно переключить тебя на ту сессию. Но вместо этого Telegram показывает диалоговое окно «Open tg://msg?text=/12?» с кнопками Yes/No. Даже если нажать Yes — текст `/12` лишь вставляется в поле ввода, но НЕ отправляется автоматически. Приходится ещё раз нажать кнопку отправки. Итого два лишних нажатия вместо мгновенного переключения.

## Цепочка причин

1. **Пользователь нажимает на номер сессии (#12) и видит окно подтверждения** → потому что →
2. **Ссылка использует формат `tg://msg?text=/12` — это специальный протокол Telegram (deep link), который ВСЕГДА показывает окно подтверждения** перед тем как что-то сделать. Это защита от случайных действий, встроенная в Telegram — обойти её нельзя → потому что →
3. **Функция `_format_clickable_session_number` в `bot.py` (модуль, который отправляет сообщения в Telegram) на строке 113 формирует ссылку именно в этом формате**: `<a href="tg://msg?text=/N">#N</a>` — красиво выглядит (`#3` как кликабельный текст), но работает неправильно → потому что →
4. **Спецификация модуля bot (`bot_spec.md`, строка 248) прямо предписывает именно этот формат** — автор спецификации выбрал `tg://msg?text=` не проверив, как это реально работает в Telegram-клиенте → потому что →
5. **КОРНЕВАЯ ПРИЧИНА: ни один этап процесса разработки (спецификация, реализация, ревью кода, E2E тесты) не проверяет, что механизмы Telegram API действительно работают так, как описано.** Формат `tg://msg?text=` был «изобретён» на этапе спецификации без проверки на реальном клиенте. При этом в том же самом проекте уже есть рабочий подход — команда `/sessions` отправляет номера как `/1`, `/2`, `/3` обычным текстом, и Telegram сам делает их кликабельными командами без всяких диалогов.

## Корневая причина

В файле `bot.py` (модуль, который отвечает за всё общение с Telegram) есть функция `_format_clickable_session_number` (строка 111-113). Она создаёт HTML-ссылку с протоколом `tg://msg?text=/N`:

```python
def _format_clickable_session_number(session_number: int) -> str:
    return f'<b><a href="tg://msg?text=/{session_number}">#{session_number}</a></b>'
```

Проблема в том, что `tg://msg?text=...` — это «глубокая ссылка» (deep link) Telegram, которая:
- Всегда показывает диалог подтверждения (это защита Telegram — обойти нельзя)
- Только вставляет текст в поле ввода, но НЕ отправляет его
- То есть вообще не делает того, что нужно

Забавно, что в этом же боте есть правильный подход. Команда `/sessions` (строка 580-584 в `bot.py`) отправляет список сессий как обычный текст с `/1`, `/2`, `/3` — без HTML-разметки. Telegram автоматически распознаёт `/N` как команды бота и делает их кликабельными. При нажатии — мгновенная отправка без диалогов.

Эта ошибка попала в код, потому что спецификация `bot_spec.md` (техническое задание на модуль bot, строка 248) прямо указывает использовать формат `tg://msg?text=`. Спецификация написана скиллом `spec-module` (инструмент, который создаёт технические задания на модули), и ни этот скилл, ни `implement-module` (который пишет код по спецификации), ни `review-code` (который проверяет готовый код) не требуют проверки Telegram-механизмов на реальном устройстве.

## Почему проблема не была предотвращена

История такая: когда писалась спецификация на модуль `bot.py`, автор (Claude через скилл `spec-module`) прочитал BRD (документ с описанием того, что бот должен делать). В BRD написано: «номер сессии оформлен как кликабельная ссылка — нажав на неё, пользователь переключается на эту сессию (в поле ввода подставляется `/номер` и сразу отправляется)». Автор спецификации нашёл формат `tg://msg?text=` и решил, что он подходит — ведь он действительно «подставляет текст в поле ввода». Но не проверил, что этот формат показывает диалог подтверждения и НЕ отправляет автоматически.

Дальше `implement-module` реализовал код строго по спецификации. `review-code` проверил соответствие BRD — но проверял только «есть ли кликабельная ссылка», а не «работает ли она правильно». E2E тесты (`test-e2e`) проверяли кликабельность через LLM-верификацию текста ответа, а не реальным кликом в Telegram.

### Проверенные скиллы

- `spec-module` (создаёт технические задания на модули) — нет пункта про проверку Telegram API-механизмов. Есть пункт про «визуальные индикаторы» (иконки, форматирование), но ссылки и deep link'и в нём не упоминаются. **Вывод: скилл пропустил проверку механизмов внешнего API**
- `implement-module` (пишет код по спецификации) — нет пункта про проверку форматов ссылок. Реализовал ровно то, что написано в спеке. **Вывод: скилл корректен — он должен следовать спецификации, а спецификация была неправильной**
- `review-code` (проверяет готовый код) — в чеклисте BRD-соответствия есть «визуальные индикаторы», но нет проверки «кликабельные ссылки работают корректно». **Вывод: скилл пропустил проверку кликабельных элементов**
- `test-e2e` (живое тестирование бота через Telegram) — упоминает «кликабельность номеров сессий» в LLM-верификации (строка 286), но проверяет только текст ответа, а не реальный клик. **Вывод: формулировка слишком расплывчатая — нужна проверка реальным кликом, а не только анализом текста**
- `create-user-test-scenarios` (создание сценариев тестирования) — упоминает «кликабельные ссылки» в контексте, но не имеет конкретного шага для их проверки. **Вывод: пробел — нет сценария для проверки клика по номеру сессии**

### Проверенные документы

- `development/docs/brd-user-journeys.md` (описание того, что бот должен делать) — описывает ожидаемое поведение: «сразу отправляется». Но не указывает конкретный технический механизм (deep link? inline button? plain text?). **Вывод: BRD корректно описывает ЖЕЛАЕМОЕ поведение, но не фиксирует технический способ его достижения**
- `development/specs/realized/bot_spec.md` (техническое задание на модуль bot) — строка 248 прямо предписывает формат `tg://msg?text=`. **Вывод: спецификация предписывает неработающий формат — это непосредственный источник бага**
- `CLAUDE.md` (главный документ проекта с правилами и архитектурой) — не содержит информации о форматах ссылок в Telegram. **Вывод: отсутствие не критично — это деталь реализации, а не архитектурный принцип**
- Отчёты сессии `29-03/22-21_brd-review-and-update.md` — зафиксировано решение: «формат `/номер` в тексте сообщения — Telegram автоматически делает их кликабельными». Это решение применено в `/sessions`, но не в watcher-сообщениях. **Вывод: принятое решение было реализовано непоследовательно**

## Рекомендации по исправлению

1. **Изменить функцию `_format_clickable_session_number`** в файле `bot.py` (строка 111-113). Вместо deep link'а `tg://msg?text=/N` использовать обычную команду `/N`, которую Telegram автоматически сделает кликабельной. Предлагаемый код:
   ```python
   def _format_clickable_session_number(session_number: int) -> str:
       """Форматирует номер сессии как кликабельную команду для Telegram."""
       return f"<b>/{session_number}</b>"
   ```
   Telegram распознаёт `/N` как команду бота даже внутри HTML-тегов — и при нажатии мгновенно отправляет её без диалогов.

2. **Обновить спецификацию** `development/specs/realized/bot_spec.md` (строка 248) — заменить пример формата с `tg://msg?text=` на `/N` и добавить пояснение, почему выбран именно этот подход.

3. **Обновить тесты** в `tests/test_bot.py` — тесты, проверяющие `_format_clickable_session_number`, нужно обновить под новый формат.

## Верификация выводов

- **Перечитал BRD** повторно, фокусируясь на всех местах, где упоминаются «кликабельные ссылки». В каждом случае BRD описывает желаемое поведение (мгновенное переключение), но не фиксирует механизм. Исключение: `/sessions` — там прямо сказано «Telegram автоматически подсвечивал `/1`, `/2`, `/3` как кликабельные команды». Это подтверждает, что для `/sessions` механизм выбран правильно, а для watcher-сообщений — нет.

- **Перечитал спецификацию bot_spec.md** — подтвердилось, что формат `tg://msg?text=` прописан явно (строка 248) и реализация ему соответствует. Баг — в спецификации, а не в реализации.

- **Альтернатива 1: это баг Telegram-клиента, а не нашего кода.** Отвергнута — `tg://msg?text=` ВСЕГДА показывает диалог подтверждения. Это штатное поведение Telegram, а не баг. Протокол `tg://msg` предназначен для отправки текста ЛЮБОМУ контакту, поэтому требует подтверждения.

- **Альтернатива 2: можно использовать другой deep link формат вместо `tg://msg?text=`.** Рассмотрена, но в Telegram нет deep link'а, который автоматически отправляет команду в текущий чат без подтверждения. Единственный способ «кликабельных команд без диалогов» — это native bot command detection, когда `/N` в тексте сообщения автоматически становится кликабельным.

- **Корневая причина подтвердилась** — проблема в выборе `tg://msg?text=` формата на этапе спецификации без проверки реального поведения.
