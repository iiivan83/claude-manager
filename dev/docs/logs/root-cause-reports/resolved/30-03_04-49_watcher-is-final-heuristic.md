# Анализ первопричины: эвристика is_final в watcher всегда выдаёт ✅

**Дата анализа:** 30-03-2026
**Статус:** Первопричина найдена

## Симптом

Когда ты в режиме `/all` (мониторинг всех сессий), все сообщения от Claude приходят с зелёной галочкой ✅ — даже те, где Claude явно ещё не закончил работу. Вот что видно на скриншоте:

- `#12 ✅ Ключевое отличие нашёл. Теперь нужно проверить конкретные вещи...` — Claude пишет «теперь нужно проверить», значит он ещё работает. Должно быть ⏳
- `#11 ✅ Отличная картина собрана. Теперь запускаю верификацию...` — то же самое, Claude продолжает. Должно быть ⏳

Это РОВНО та же проблема, которую уже чинили в коммите `ea3ec95`. Фикс был сделан, но не помог — эвристика (способ определения — финальное ли сообщение), которую выбрали для фикса, не работает в реальных условиях.

## Цепочка причин

1. **Все watcher-сообщения показывают ✅ вместо ⏳** → потому что →
2. **Эвристика «последнее в пачке = финальное»** — в `session_watcher.py` (модуль-наблюдатель, который каждые 2 секунды проверяет файлы сессий) на строке 148 написано: `is_final = (index == last_index)`. Это значит: если сообщение — последнее среди новых, считай его финальным. Но watcher проверяет файлы каждые 2 секунды, и обычно находит **одно** новое сообщение за цикл. Одно сообщение — это и «первое», и «последнее» в пачке одновременно → **всегда** `is_final = True` → **всегда** ✅ → потому что →
3. **Реализация не следовала рекомендации** — предыдущий root-cause отчёт (`30-03_03-06_progress-icon-checkmark.md`) рекомендовал проверять, «продолжает ли файл расти». Но реализация упростила идею до «последнее в пачке = финальное», что не учитывает, как реально работает поллинг → потому что →
4. **Фикс не был протестирован на живом боте** — юнит-тест `test_send_watcher_message_uses_correct_icon` проверяет, что функция отправки правильно выбирает иконку **по переданному параметру** `is_final`. Но НЕ проверяет, что эвристика **правильно определяет** этот параметр в реальных условиях (когда сообщения приходят по одному) → потому что →
5. **КОРНЕВАЯ ПРИЧИНА: watcher не может определить, закончил ли Claude работу, просто по количеству новых текстовых сообщений.** Нужно смотреть на типы событий в JSONL-файле (в файле, куда Claude записывает историю сессии). Там есть не только текстовые ответы — есть ещё мысли (thinking), вызовы инструментов (tool_use), индикаторы прогресса (progress). Если после последнего текстового ответа есть такие «рабочие» события — Claude ещё работает. Сейчас watcher эти события игнорирует.

## Корневая причина

В JSONL-файлах сессий (файлы, куда Claude Code записывает историю диалога — по одному на каждую сессию) события записываются в таком порядке:

- `user` → пользователь написал сообщение
- `assistant` с блоком `thinking` → Claude думает
- `assistant` с блоком `text` → Claude написал текстовый ответ (ЭТО watcher показывает пользователю)
- `assistant` с блоком `tool_use` → Claude вызывает инструмент (читает файл, пишет код и т.д.)
- `progress` → инструмент выполняется
- Потом снова thinking → text → tool_use → progress...
- `user` → следующее сообщение пользователя (или конец сессии)

Watcher (функция `_extract_message_text` в `session_watcher.py`, строка 53) извлекает **только** события типа `assistant` с блоком `text`. Все остальные типы он игнорирует — thinking, tool_use, progress, user.

Но эти «игнорируемые» события — это КЛЮЧ к определению: работает ли Claude ещё или уже закончил. Если после последнего текстового ответа в файле есть события `assistant(tool_use)` или `progress` — Claude точно ещё работает. Если файл заканчивается на `user` — предыдущий обмен завершён.

Текущая эвристика (`is_final = index == last_index`) этого не учитывает. Она считает только текстовые сообщения и не знает о существовании других типов событий в файле.

Вот что я увидел в реальных JSONL-файлах:

- **Файл сессии #1:** типы событий: `assistant: 32, progress: 85, user: 24, system: 2` — прогресс-событий в 2.5 раза больше, чем текстовых ответов
- **Файл сессии #2:** типы событий: `assistant: 40, progress: 112, user: 26` — ещё больше перекос
- **Последовательность:** `user → assistant → assistant → assistant → progress → user → assistant → progress → ...` — между текстовыми ответами всегда есть другие события

Это значит: когда watcher находит один новый текстовый ответ, в файле после него почти всегда есть ещё события (tool_use, progress). Но watcher их не видит и считает текст «последним» → ставит ✅.

**Конкретное место в коде:**

В `session_watcher.py`, функция `_check_session`, строка 148:
```python
is_final = (index == last_index)
```

Переменная `last_index` — это индекс последнего **текстового** сообщения. Но это НЕ то же самое, что «последнее событие в файле». Между текстами и после них есть десятки других событий, которые эвристика не учитывает.

## Рекомендации по исправлению

1. **Заменить эвристику определения `is_final` в `session_watcher.py`** — вместо «последнее в пачке текстов» проверять тип последнего события в JSONL-файле. Если файл заканчивается на «рабочее» событие (`assistant`, `progress`, `queue-operation`) — Claude ещё работает, все тексты промежуточные (⏳). Если на `user`, `system`, `last-prompt` — обмен завершён, последний текст финальный (✅)

   В `session_watcher.py`, функция `_check_session`, заменить строку 148:
   ```python
   # Было:
   is_final = (index == last_index)

   # Стало:
   ACTIVE_EVENT_TYPES = frozenset({"assistant", "progress", "queue-operation"})
   last_event_type = all_messages[-1].get("type", "") if all_messages else ""
   session_is_active = last_event_type in ACTIVE_EVENT_TYPES
   # ... в цикле:
   is_final = not session_is_active and (index == last_index)
   ```

2. **Добавить константу `ACTIVE_EVENT_TYPES`** — типы событий JSONL, которые означают «Claude ещё работает». Вынести в именованную константу с комментарием, чтобы было понятно откуда взялись эти значения

3. **Обновить тест `test_detects_new_assistant_message`** в `tests/test_session_watcher.py` — сейчас тест проверяет что callback вызывается с `is_final=True`, но это **валидирует неправильное поведение**. Нужно заменить на тест, который моделирует реальную ситуацию: файл заканчивается на `assistant` → `is_final=False`

4. **Добавить тест на реалистичный сценарий** — тест, где в JSONL-файле после текстового ответа есть ещё events (tool_use, progress). Проверить, что текст помечается как промежуточный

**Что перепроверил:**

- Прочитал реальные JSONL-файлы сессий проекта (2 файла). Подтвердил: типы событий — `assistant`, `progress`, `user`, `system`, `file-history-snapshot`, `queue-operation`, `last-prompt`. Событий `result` НЕТ (они существуют только в потоковом протоколе stdout, а не в файлах на диске)
- Проследил последовательность событий: `user → assistant(thinking) → assistant(text) → assistant(tool_use) → progress → user → ...`. Подтвердил: между текстовыми ответами всегда есть другие события
- Перечитал код `_check_session` и `_extract_assistant_messages`. Подтвердил: `current_count` считает ВСЕ события, `new_texts` — только тексты. При 2-секундном поллинге в пачке обычно 1 текст → `last_index = 0`, `index = 0`, `is_final = True`
- Проверил, что код фикса (коммит `ea3ec95`) действительно присутствует в файле session_watcher.py (строка 148: `is_final = (index == last_index)`) — фикс применён, но не работает

**Альтернативные версии:**

- **«Бот не был перезапущен после фикса»** — возможно, но ДАЖЕ если перезапустить, эвристика не работает. Это отягчающее обстоятельство, но не корневая причина. Проверить: посмотреть логи бота на наличие перезапуска после коммита `ea3ec95`

- **«Проблема в коде отправки, а не в определении is_final»** — отклонена. Функции `_format_session_header` и `send_watcher_message` корректно используют `is_final` для выбора иконки. Тесты подтверждают. Проблема именно в том, ЧТО приходит на вход — `is_final` почти всегда `True`

- **«JSONL-файлы содержат result-события, нужно их искать»** — отклонена. Проверил 2 реальных JSONL-файла: ни одного события с `type: "result"`. Результаты Claude записываются в JSONL как `assistant` с блоком `text`, без отдельного маркера завершения

**Корневая причина подтвердилась** — эвристика не учитывает типы событий в JSONL-файле и не может отличить промежуточное сообщение от финального.
