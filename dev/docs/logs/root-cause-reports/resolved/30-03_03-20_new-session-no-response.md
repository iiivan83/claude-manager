# Анализ первопричины: после /new ответ от новой сессии не приходит

**Дата анализа:** 30-03-2026 (обновлено после дополнительного расследования)
**Статус:** Первопричина найдена

## Симптом

Пользователь создал новую сессию командой `/new`, бот ответил «Создана новая сессия #1». Пользователь написал «Привет! Как дела?» — ответ от новой сессии **не пришёл совсем**. Вместо этого продолжились сообщения от старой сессии (#4, #6) через watcher (модуль мониторинга файлов сессий). Проблема воспроизводится при каждом `/new`.

## Цепочка причин

1. **Ответ от новой сессии не приходит** — пользователь пишет сообщение, бот молчит → потому что →

2. **Бот навечно завис в ожидании ответа от Claude CLI** — функция `_process_events` (`process_manager.py`, строка 179) вызывает `readline()` (чтение строки из выходного потока процесса Claude), которая **блокирует навсегда**, если Claude ничего не пишет → потому что →

3. **Процесс Claude CLI (PID 16620) застрял на инициализации** — он стартовал в 03:02, но за 34 минуты использовал всего 3 секунды CPU. Он читает файлы скиллов проекта (skill files) и, вероятно, ждёт ответа от API Claude для инициализации сессии. Он так и не дошёл до чтения сообщения пользователя из stdin (входного потока) → потому что →

4. **В коде нет таймаута на ожидание ответа от Claude CLI** — `read_events()` (`claude_runner.py`, строка 97) вызывает `await self.process.stdout.readline()` в бесконечном цикле без какого-либо ограничения по времени. Если Claude CLI зависает (инициализация, сетевая проблема, rate-limit от API) — бот зависает вместе с ним → потому что →

5. **КОРНЕВАЯ ПРИЧИНА: отсутствие таймаута в `read_events` / `_process_events`.** Ни спецификация `process_manager_spec.md`, ни спецификация `claude_runner_spec.md` не описывают поведение при зависании Claude CLI. Код был реализован без защиты от зависания внешнего процесса.

## Корневая причина — подробно

### Что показали логи

Настоящие логи бота (не launchctl, а задача Claude Code):

```
03:02:03 [INFO] session_manager: Создана новая сессия _new_0001 (#1) для чата REDACTED_OWNER_TELEGRAM_ID
03:02:03 [INFO] claude_runner: Процесс Claude запущен: PID=16620, cwd=...
03:02:03 [INFO] process_manager: Процесс Claude создан: session_id=_new_0001, тип=resume, PID=16620
```

**После этого — полная тишина.** Ни одной записи о получении сообщения, отправке в Claude, ответе или ошибке. Бот завис в `readline()`.

### Что показал живой процесс

Процесс Claude CLI (PID 16620):
- **Жив** — работает уже 34+ минут
- **Простаивает** — использовал 3 секунды CPU за всё время
- **Занимает мало памяти** — 57MB (рабочие сессии используют 200-500MB)
- **Читает файлы скиллов** — `lsof` показал открытые SKILL.md файлы проекта
- **stdin/stdout подключены через PIPE** — бот ожидает ответа на stdout

### Почему Claude CLI зависает

При запуске без `--resume` (новая сессия), Claude CLI:
1. Загружает конфигурацию проекта (CLAUDE.md, скиллы)
2. Подключается к API Claude для инициализации
3. Ждёт первое сообщение на stdin

Шаг 2 может занимать неопределённо долго — если API перегружен, если есть rate-limit от одновременных процессов (в терминалах параллельно работают 8+ сессий Claude), или если сеть нестабильна. Бот не отличает «Claude думает 5 секунд» от «Claude завис навсегда».

### Дополнительная находка: исправление `--resume _new_XXXX`

В закоммиченном коде (который работал ДО 02:48) была ещё одна ошибка — функция `create_process` передавала временный ID `_new_0001` как `--resume _new_0001` в Claude CLI. Claude CLI не мог найти такую сессию и падал. Это исправление (`process_manager.py`, diff: `start_process(cli_session_id)` вместо `start_process(session_id)`) было написано, но **не закоммичено**. После перезапуска бота в 03:00:45 исправление подхватилось — процесс стартует без `--resume`. Но зависание при инициализации осталось.

## Почему проблема не была предотвращена

### Проверенные скиллы

- `spec-module` (написание спецификаций) — спецификация `claude_runner_spec.md` не описывает поведение при зависании Claude CLI. Нет требования на таймаут в `read_events`. **Вывод: пробел в спецификации.**
- `implement-module` (реализация по спецификации) — код реализован по спецификации. Раз в спеке нет таймаута — его и нет в коде. **Вывод: скилл корректен, проблема выше.**
- `review-code` (ревью кода) — в чеклисте нет пункта «проверить наличие таймаутов при ожидании внешних процессов». **Вывод: пробел в чеклисте ревью.**
- `test-integration` — нет теста на сценарий «Claude CLI зависает при инициализации». **Вывод: пробел в тест-плане.**
- `validate-brd` — нет вопроса «что делает бот, если Claude CLI не отвечает?». **Вывод: пробел.**

### Проверенные документы

- `process_manager_spec.md` — описывает ретраи при ошибках, но не описывает поведение при зависании (когда Claude не возвращает ни результата, ни ошибки).
- `claude_runner_spec.md` — описывает `read_events` как бесконечный цикл до `result` или EOF. Нет упоминания таймаута.
- `brd-user-journeys.md` — BRD не рассматривает случай «Claude CLI завис».

## Рекомендации по исправлению

1. **Не запускать Claude CLI заранее в `/new`** — убрать `process_manager.create_process()` из `handle_new` (`bot.py`, строка 529). Вместо этого процесс запустится автоматически в `_ensure_process_running` при первом сообщении пользователя. Это устраняет гонку: Claude CLI получит сообщение сразу при старте, а не будет висеть пустым. Функция `_ensure_process_running` (`bot.py`, строка 329) уже умеет создавать процесс, если его нет — изменение минимальное.

2. **Добавить таймаут в `read_events`** — в `claude_runner.py` (строка 100), обернуть `readline()` в `asyncio.wait_for()` с таймаутом (120 секунд). Страховка: если Claude CLI всё равно зависнет при обработке — бот не зависнет вместе с ним, а отправит пользователю ошибку.

3. **Закоммитить исправление `--resume _new_XXXX`** — diff в `process_manager.py` (строки 313-333, 339-370) уже написан, но не закоммичен. Без этого исправления старая ошибка вернётся при следующем перезапуске бота.

## Рекомендации по предотвращению

1. **[ARCHITECTURE]** Добавить в CLAUDE.md архитектурный принцип: «Все вызовы внешних процессов (Claude CLI) должны иметь таймаут. Бесконечное ожидание внешнего процесса недопустимо.» Место: раздел «Архитектурные принципы», после «Асинхронность».

2. **[DOC]** Обновить спецификации `claude_runner_spec.md` и `process_manager_spec.md` — добавить описание таймаутов при ожидании ответа от Claude CLI.

3. **[CODE]** Реализовать таймаут в `read_events()` и `_process_events()` — `asyncio.wait_for(readline(), timeout=120)`.

4. **[CODE]** Закоммитить исправление `--resume _new_XXXX` в `process_manager.py`.

5. **[SKILL]** В скилле `review-code` добавить пункт в чеклист: «Проверить наличие таймаутов при ожидании ответа от внешних процессов (subprocess, API)».

6. **[SKILL]** В скилле `validate-brd` добавить вопрос: «Что делает система, если внешний процесс (Claude CLI) не отвечает?»

## Верификация выводов

### Прямые доказательства

- **Логи бота** — после создания процесса в 03:02:03 нет ни одной записи. Бот завис.
- **PID 16620** — жив 34+ минут, 3 секунды CPU, 57MB памяти. Не обрабатывает сообщение.
- **lsof PID 16620** — stdin/stdout подключены через PIPE к боту, Claude читает файлы скиллов.
- **Другие Claude-процессы** — работают нормально (200-500MB памяти, минуты CPU). Значит API доступен, проблема в конкретном процессе.

### Альтернативные объяснения

- **«Ответ приходит, но теряется среди watcher-сообщений»** — ОТКЛОНЕНО. Логи не содержат записи об отправке ответа. Процесс Claude использовал 3 секунды CPU — он не обработал сообщение.
- **«Ошибка --resume _new_XXXX»** — БЫЛО АКТУАЛЬНО ДО 02:48. После исправления процесс стартует без --resume. Но зависание при инициализации — другая проблема.
- **«Telegram API не работает»** — ОТКЛОНЕНО. Watcher успешно отправляет сообщения из других сессий. Бот подтверждает «Создана новая сессия #1».
