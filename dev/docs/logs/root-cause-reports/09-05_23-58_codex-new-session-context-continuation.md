# Анализ первопричины: новая Codex-сессия выглядела как продолжение старого диалога

**Дата анализа:** 09-05-2026
**Проект:** Claude Manager
**Связанный рабочий проект:** reviews-analyzer
**Статус:** Первопричина найдена

## Короткий вывод

Старый диалог физически не скопировался в новую сессию. Новая Codex-сессия #269 действительно была создана как новая: временный id `_new_0bbc75fb19cb` был заменён на новый id `019e0e00-defa-7922-89fd-33539d934762`.

Эффект "как будто скопировалось" возник из-за другого механизма: первое сообщение в новой сессии было `Опубликуй да`, то есть короткое продолжение без самостоятельного контекста. Агент увидел общий проектный каталог, нашёл сессионный отчёт `dev/docs/session-reports/09-05/23-12_load-all-orders-and-dishes.md` и восстановил намерение из файлов проекта. Сессионный отчёт - это markdown-документ для передачи контекста следующей сессии. Он специально содержит раздел "Контекст для следующей сессии" и открытые следующие шаги.

## Симптом

В проекте `reviews-analyzer` после переключения на новую сессию пользователь отправил:

- `Опубликуй да`

Новая сессия начала публиковать HTML-отчёт, будто знала предыдущее обсуждение про локальный HTML и внешнюю ссылку.

Ожидание пользователя: новая сессия должна быть пустой и не знать предыдущее сообщение.

Фактическое поведение системы: новая сессия не знала старый чат напрямую, но видела те же проектные файлы и документацию.

## Доказательства

1. В `daily_sessions.json` проекта `reviews-analyzer` есть отдельные записи:

- #268: `019e0df3-c88e-7ae2-bd05-d51e302921df`, backend `codex`.
- #269: `019e0e00-defa-7922-89fd-33539d934762`, backend `codex`.

Backend - это исполнитель, через который бот запускает агент: здесь Codex CLI, а не Claude CLI.

2. Лог Claude Manager подтверждает создание новой #269:

- `23:29:57` зарегистрирована `_new_0bbc75fb19cb` как #269.
- `23:29:58` запущен backend subprocess для `_new_0bbc75fb19cb`.
- `23:29:58` id обновлён `_new_0bbc75fb19cb -> 019e0e00-defa-7922-89fd-33539d934762`.

Subprocess - это отдельный системный процесс, который Claude Manager запускает для Codex CLI.

3. Код создания новой сессии не подставляет старый id:

- `src/claude_manager/bot.py:449` - `handle_new()` вызывает создание новой сессии.
- `src/claude_manager/session_manager.py:357` - `create_new_session()` создаёт временный id вида `_new_...`, регистрирует его и привязывает к чату.

4. Код запуска Codex различает новую сессию и продолжение старой:

- `src/claude_manager/codex_backend.py:117` - `compose_subprocess_command_args()` строит аргументы запуска.
- `src/claude_manager/codex_backend.py:131` - если id начинается с `_new_`, запускается `codex exec ... <prompt>`.
- `src/claude_manager/codex_backend.py:140` - иначе запускается `codex exec resume <session_id> ... <prompt>`.

`resume` означает "продолжить уже существующую сессию". Для первого запуска #269 использовался временный `_new_...`, значит `resume` старой #268 не применялся.

5. Файлы Codex-сессий #268 и #269 разные. Проверка `cmp` вернула код `1`, то есть файлы не идентичны.

Codex session file - это JSONL-файл, где Codex построчно записывает события диалога и вызовы инструментов.

6. Файл #269 начинается не со старого диалога #268, а с нового короткого сообщения:

- #269, строка 6: пользовательское сообщение `Опубликуй да`.
- #269, строка 38: агент перечислил файлы проекта, включая `dev/docs/session-reports/09-05/23-12_load-all-orders-and-dishes.md`.
- #269, строка 58: агент прочитал этот сессионный отчёт.
- #269, строки 68-69: агент написал, что "контекст прояснился" и что в прошлом отчёте оставлена опция опубликовать новую ссылку.

7. Старый диалог #268 действительно содержал недостающий человеческий контекст:

- #268, строки 230-231: агент объяснил, что локальный HTML обновлён, но актуальной внешней ссылки ещё нет.

Это делает фразу `Опубликуй да` понятной человеку, но она не была самостоятельной задачей для пустой сессии.

8. Сессионный отчёт `reviews-analyzer/dev/docs/session-reports/09-05/23-12_load-all-orders-and-dishes.md` прямо передавал следующий шаг:

- строка 34: раздел "Контекст для следующей сессии".
- строки 44-48: актуальные рабочие файлы данных.
- строки 50-52: открытая опция "перегенерировать HTML ... и опубликовать новую публичную ссылку".

## Цепочка причин

1. Пользователь увидел, что новая сессия #269 продолжила старую задачу -> потому что ->
2. Первое сообщение в #269 было `Опубликуй да`, то есть подтверждением без самостоятельного описания задачи -> потому что ->
3. Агент в новой сессии начал искать контекст в текущем проекте `reviews-analyzer` -> потому что ->
4. Проектный каталог общий для всех сессий, а сессионные отчёты специально созданы для передачи контекста новым сессиям -> потому что ->
5. В `dev/docs/session-reports/09-05/23-12_load-all-orders-and-dishes.md` была явная открытая опция опубликовать новую публичную ссылку -> потому что ->
6. Claude Manager показывает `/new` как "новая сессия", но не объясняет пользователю точную границу изоляции: изолируется история CLI-диалога, но не изолируются файлы проекта, сессионные отчёты и незакоммиченные рабочие артефакты.

## Корневая причина

Корневая причина - разрыв между пользовательским ожиданием и фактической моделью изоляции.

Пользователь ожидает, что `/new` означает "полностью чистый контекст". В реализации `/new` означает "новая CLI-сессия без истории старого CLI-чата", но агент всё равно работает в том же проектном каталоге. Проектный каталог - это папка с файлами проекта, документацией, временными результатами и отчётами. Эти файлы доступны новой сессии и могут быть использованы как источник контекста.

Вторая часть причины - отсутствие защиты от двусмысленного первого сообщения. Если первая фраза в новой сессии выглядит как продолжение, например `Опубликуй да` или `Продолжай`, Claude Manager сразу отправляет её в Codex. Нет правила, которое спросит пользователя: "Это новая самостоятельная задача или продолжить по последнему сессионному отчёту?"

## Что не является причиной

1. Это не `resume` старой сессии #268.

Код `codex_backend.py` использует `resume` только для id, который не начинается с `_new_`. В #269 первый запуск был через `_new_0bbc75fb19cb`, после чего Codex выдал новый id `019e0e00-defa-7922-89fd-33539d934762`.

2. Это не физическое копирование JSONL-файла #268 в #269.

Файлы #268 и #269 отличаются. В #269 нет полного предыдущего диалога #268 перед первым сообщением `Опубликуй да`.

3. Это не та же проблема, что массовая рассылка старых сообщений при переключении проекта.

Ранее был отдельный RCA `dev/docs/logs/root-cause-reports/09-05_17-03_project-switch-message-flood.md`. Там причиной был `session_watcher`, то есть фоновый наблюдатель за файлами сессий. В текущем случае #269 сама прочитала проектные файлы и продолжила задачу по ним.

4. Это не само переключение проекта в 23:37-23:38.

#269 была создана и начала работу в `reviews-analyzer` уже в 23:29:57-23:29:58. Более позднее переключение только снова привязало чат к уже существующей #269.

## Почему это не было предотвращено

1. Сообщение `/new` слишком короткое.

Сейчас `handle_new()` отвечает только "Создана новая сессия #N". В ответе нет предупреждения, что файлы проекта и сессионные отчёты остаются общим контекстом.

2. Нет первого-turn guard.

First-turn guard - это проверка первого пользовательского сообщения в свежей сессии. Она должна ловить фразы, которые зависят от прошлого контекста: `да`, `опубликуй да`, `продолжай`, `сделай`, `ок`, `давай`, `как говорил`.

3. Сессионный отчёт сработал ровно по назначению.

Проектный `session-report-creator` создаёт отчёт, чтобы новая сессия могла быстро продолжить работу. Поэтому сам отчёт не является багом. Проблема в том, что Claude Manager не различает намеренное продолжение по отчёту и случайное ожидание чистого старта.

## Итоговая формулировка

Новая сессия #269 не была копией предыдущей #268. Она стала похожа на продолжение, потому что получила короткую фразу `Опубликуй да`, увидела тот же проектный каталог и прочитала сессионный отчёт, который специально описывал следующий шаг. Главный дефект в Claude Manager - не в механике Codex-сессий, а в UX и защите первого сообщения: система не объясняет границу изоляции `/new` и не останавливает двусмысленные continuation-фразы в свежей сессии.
