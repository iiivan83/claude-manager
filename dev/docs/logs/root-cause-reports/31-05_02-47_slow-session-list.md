# Анализ первопричины: медленный список сессий

**Дата анализа:** 31-05-2026
**Проект:** Claude Manager
**Статус:** Первопричина найдена

## Симптом

Команда `/sessions` стала отвечать заметно дольше, хотя раньше список прошлых диалогов открывался быстро. Для Ивана это выглядит как простое ожидание в Telegram: бот получил команду, но список сессий долго не появляется.

Важные термины:

- **`/sessions`** — команда Telegram-бота, которая показывает последние диалоги и даёт перейти в нужную сессию через `/1`, `/2`, `/3`.
- **Codex rollout-файл** — JSONL-файл одной Codex-сессии в `~/.codex/sessions/YYYY/MM/DD/`.
- **`session_meta`** — служебная запись внутри rollout-файла. В ней лежит `payload.cwd`, то есть проект, к которому относится сессия.
- **preview** — короткий текст в списке сессий: первое настоящее сообщение пользователя.
- **operational index** — быстрый in-memory индекс Codex-файлов, добавленный 30-05-2026 для горячих внутренних путей вроде переключения проектов.

## Цепочка причин

1. **Иван ждёт список `/sessions` по 8-10 секунд** -> потому что ->
2. **обработчик `/sessions` в `bot.py` ждёт все backend-ы перед отправкой ответа**. `bot.py` — транспортный слой Telegram-команд; в `handle_sessions()` он вызывает `backend.list_session_files_for_project(config.WORKING_DIR)` для Claude и Codex -> потому что ->
3. **Claude быстрый, а Codex каждый раз сканирует глобальную историю за 30 дней**. У Claude файлы лежат в папке проекта, а у Codex все проекты лежат общей кучей в `~/.codex/sessions/` -> потому что ->
4. **чтобы понять, относится ли rollout-файл к текущему проекту, код открывает каждый файл и читает `session_meta.payload.cwd`** -> потому что ->
5. **в текущем профиле уже 14 106 Codex rollout-файлов за 30-дневное окно, и `/sessions` перечитывает их на каждый вызов** -> потому что ->
6. **КОРНЕВАЯ ПРИЧИНА: пользовательский список `/sessions` остался на старом O(N)-скане глобальной Codex-истории. Вчерашний быстрый `codex_session_index` исправил operational-пути, но `/sessions` был сознательно оставлен без индекса из-за необходимости строить preview. Предположение из спеки, что 30-дневный UI-скан будет дешёвым, больше не соответствует реальному объёму файлов.**

## Корневая причина

Главный участок — `src/claude_manager/codex_session_file_listing.py` (модуль, который ищет Codex-файлы сессий и строит метаданные для списка). Функция `list_session_file_infos_for_project()` делает так:

1. Берёт `LOOKBACK_DAYS_FOR_SESSION_LISTING = 30` из `src/claude_manager/codex_session_file_reader.py`.
2. Собирает все `rollout-*.jsonl` за эти 30 дней.
3. Для каждого файла читает `session_meta`, чтобы проверить `payload.cwd == project_dir`.
4. Сортирует найденные файлы текущего проекта.
5. Только для первых 15 строит preview.

Практический замер на реальных файлах:

- Всего Codex rollout-файлов в `~/.codex/sessions`: 14 106.
- За последние 4 дня: 4 804 файла, 116 МБ.
- Простое перечисление путей за 30 дней занимает 0.114 секунды, то есть тормозит не `find`/директории.
- Claude UI-list для текущего проекта: 0.140 секунды.
- Codex UI-list для `claude_manager`: 8.265-9.643 секунды.
- Codex UI-list для `budgets`: 7.875 секунды.
- Инструментированный вызов для `budgets`: 7.785 секунды, 14 106 чтений `session_meta`, 15 чтений preview.

Это подтверждает, что проблема не в Telegram и не в формировании 15 строк. Основная цена — чтение и JSON-парсинг `session_meta` во всех Codex-файлах за 30 дней.

Почему вчерашний индекс не спасает `/sessions`:

- `codex_session_index.py` работает только через `list_all_session_file_infos_for_project(..., lookback_days=N)`.
- `/sessions` вызывает другой API: `list_session_files_for_project(project_dir)`.
- Спецификация `dev/docs/specs/realised/codex_backend_spec.md` прямо говорит, что `/sessions` остаётся без operational index, потому что ему нужен preview.
- На практике preview нужен только для 15 видимых сессий, а не для всех 14 106 файлов. Значит, индексировать можно метаданные, а preview дочитывать точечно.

## Почему проблема не была предотвращена

### Проверенные скиллы

- `root-cause-analysis` (скилл поиска первопричины) — использован. Он заставил проверить документы, историю прошлых RCA, код, замеры и альтернативные объяснения.
- `systematic-debugging` (скилл системной отладки) — использован. Сначала была воспроизведена задержка и собраны факты, только потом сформулирована причина.
- Проектных `.claude/skills/*/SKILL.md` в этом репозитории сейчас нет. Поэтому локальный проектный скилл не мог поймать этот случай.

### Проверенные документы

- `CLAUDE.md` (главные правила проекта) — уже содержит принцип, что горячие пути не должны повторно сканировать внешнюю историю. Но там перечислены `/pN`, watcher, reset и pending; `/sessions` как пользовательский горячий путь туда не попал.
- `dev/docs/specs/realised/codex_backend_spec.md` (спецификация Codex backend-а) — фиксирует 30-дневное окно для `/sessions` и говорит, что 3000 файлов на проход выглядят приемлемо. Сейчас реальный объём — 14 106 файлов, и этот расчёт устарел.
- `dev/docs/specs/realised/coding_agent_backend_spec.md` (общий backend-контракт) — разделяет UI-метод `list_session_files_for_project` и operational-метод `list_all_session_files_for_project`, но не задаёт performance budget для UI-метода на backend-е с глобальным хранилищем.
- `dev/docs/logs/root-cause-reports/30-05_18-32_slow-project-switch.md` (вчерашний RCA по медленному переключению проектов) — нашёл ту же физическую проблему Codex: глобальная папка и чтение `session_meta`. Но последующая реализация ограничилась operational-путями.
- `dev/docs/adr/30.05_19.28-session-change-documenter-codex-operational-session-index.md` (ADR по operational index) — прямо зафиксировал ограничение: `/sessions` не использует индекс, потому что ему нужен preview. Это ограничение и стало местом текущей регрессии.
- `tests/test_bot.py` (тесты Telegram-команд) — проверяет объединение Claude и Codex через fake-backend, но не проверяет цену реального Codex listing.
- `tests/test_codex_backend.py` (тесты Codex backend-а) — проверяет корректность 30-дневного окна, фильтрацию и preview, но не проверяет регрессию на тысячах чужих rollout-файлов.

## Верификация выводов

Повторно проверены ключевые объяснения:

- **Telegram тормозит отправку** — маловероятно. Замеры выполнялись локально внутри backend-функций, без Telegram, и уже показывают 7.8-10.4 секунды.
- **Claude backend тормозит список** — отвергнуто. Claude list занимает 0.140 секунды, потому что его файлы уже лежат в project-specific папке.
- **Проблема в preview** — частично, но не корень. Preview читается только для 15 файлов. Инструментированный замер показал 14 106 meta-чтений и 15 preview-чтений.
- **Проблема в обходе директорий** — отвергнуто. Сбор путей за 30 дней занимает 0.114 секунды. Дорого именно открыть и распарсить JSONL каждого файла.
- **Operational index сломан** — отвергнуто. Он работает для `list_all_session_files_for_project(..., lookback_days=4)`: первый вызов строит индекс, повторные вызовы возвращаются за миллисекунды. Но `/sessions` этот API не использует.

Корневая причина подтверждена: текущий `/sessions` повторяет глобальный Codex meta-scan за 30 дней на каждый вызов, а реальный объём файлов вырос выше предположений спецификации.

## Верификация решений

- **Рекомендация:** user-facing index/cache для `/sessions`.
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** Решает корень: убирает повторное чтение `session_meta` во всех файлах. Preview можно строить точечно для 15 видимых сессий.
  - **Корректировка:** Не хранить текст сообщений как источник правды; cache preview инвалидировать по `mtime`.

- **Рекомендация:** просто уменьшить `LOOKBACK_DAYS_FOR_SESSION_LISTING` с 30 до 4.
  - **Вердикт:** ОТКЛОНЕНО
  - **Обоснование:** Ускорит симптом, но может скрыть валидные недавние сессии активного пользователя, если 15 свежих по проекту лежат старше 4 дней. Это меняет пользовательский контракт.
  - **Корректировка:** Допустимо как временная ручная настройка только после отдельного решения, но не как корневой фикс.

- **Рекомендация:** параллелить Claude и Codex в `handle_sessions`.
  - **Вердикт:** ОДОБРЕНО С ЗАМЕЧАНИЯМИ
  - **Обоснование:** Небольшое улучшение UX, но не убирает 14 106 meta-чтений Codex.
  - **Корректировка:** Делать после индекса или вместе с ним, но не считать это основным исправлением.

- **Рекомендация:** добавить performance regression test.
  - **Вердикт:** ОДОБРЕНО
  - **Обоснование:** Именно отсутствие теста на алгоритмическую цену позволило сохранить корректный, но медленный путь.
  - **Корректировка:** Проверять счётчик чтений/сканов, а не абсолютное время.
