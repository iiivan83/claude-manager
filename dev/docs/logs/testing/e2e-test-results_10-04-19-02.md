## Побочное открытие — новый баг (не связан с path encoding)

В рамках CJM-10-01 в логе бота обнаружена **ранее неизвестная ошибка в `claude_runner.py:112`**:

```
asyncio.exceptions.LimitOverrunError: Separator is found, but chunk is longer than limit

File "src/claude_manager/claude_runner.py", line 112, in read_events
    raw_bytes = await asyncio.wait_for(
File "asyncio/streams.py", line 571, in readline
    raise ValueError(e.args[0])
ValueError: Separator is found, but chunk is longer than limit
```

**Краткое объяснение.** `asyncio.StreamReader.readline()` имеет дефолтный лимит длины строки 64 KB. Когда Claude CLI пишет в stdout длинный JSON-chunk (например, событие `assistant.content` с большим текстом ответа), строка превышает этот лимит, и asyncio бросает исключение.

**Как повлияло на прогон.** Первое сообщение в `all_responses` для CJM-10-01 — «Произошла ошибка. Попробуйте ещё раз» (это механизм автоматического retry из CJM-02). Через ретрай пришёл корректный ответ `#2 ✅ В файле src/claude_manager/bot.py 799 строк.`. Общее время 18.6 сек включает первый провал и успешный повтор.

**Почему это важно.** Retry замаскировал реальный баг — пользователь в итоге получил правильный ответ, поэтому баг оставался скрытым. Но каждый длинный ответ Claude порождает такой сбой, и каждый раз бот тратит лишнюю минуту на retry. В error.log теперь регулярно копятся такие трейсбеки.

**Рекомендация.** Новый тикет на root-cause analysis и фикс: увеличить лимит `asyncio.StreamReader` (конструктор принимает `limit=<bytes>`) с дефолтных 64 KB до, например, 4 MB — это покроет даже очень длинные ответы Claude. Фикс на 3-5 строк в `claude_runner.py` (там где создаётся `create_subprocess_exec` с параметрами для stdout-pipe).
