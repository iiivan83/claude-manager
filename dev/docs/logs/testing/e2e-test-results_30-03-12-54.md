## Критический баг: Claude CLI stdout buffering

Claude CLI v2.1.81 использует block buffering (посимвольную буферизацию большими блоками) при запуске через subprocess (когда stdout привязан к pipe, а не к терминалу). В результате:

- CLI запускается, но не отдаёт init event на stdout
- Бот читает stdout.readline(), получает пустые байты, считает что процесс завершился
- Срабатывает ретрай-цикл, который повторяет запуск с тем же результатом
- Все 10 ретраев проваливаются одинаково

**Доказательство:**
- `echo '...' | claude -p --output-format stream-json --verbose` — работает (stdin закрыт EOF)
- subprocess с stdin.close() после отправки сообщения — работает
- subprocess без stdin.close() (как делает бот) — Claude CLI зависает, stdout пуст

**Вероятная причина:** Node.js (на котором написан Claude CLI) по умолчанию использует block buffering для stdout когда он является pipe. В интерактивном режиме (TTY) используется line buffering, и всё работает.

**Следствие:** бот не может получать ответы от Claude через subprocess. Watcher работает через чтение файлов с диска (JSONL), поэтому не затронут.

## Рекомендации

Баг Claude CLI stdout buffering критичен — бот не может получать ответы от Claude. Возможные решения:
- Использовать pseudo-TTY (pty) вместо pipe для stdout, чтобы Claude CLI использовал line buffering
- Закрывать stdin после каждого сообщения (создавая новый процесс для каждого запроса вместо поддержания долгоживущего процесса)
- Дождаться исправления в новой версии Claude CLI
