# Спецификация: reply-якоря для Telegram-ответов

## Контекст

Claude Manager — локальный Telegram-бот, который принимает сообщения Ивана в
Telegram, отправляет их в выбранный CLI-агент и возвращает ответ обратно в чат.

Сейчас ответы бота приходят обычными сообщениями. После этой доработки ответы
агента должны приходить как reply к тому Telegram-сообщению Ивана, которое
реально было принято в конкретную сессию агента.

Reply-якорь — это сохранённый `message_id` входящего Telegram-сообщения, к
которому бот потом привязывает ответ через Telegram reply-механику.

## Цель

Сделать так, чтобы ответы Claude/Codex, watcher-сообщения, `/all`-мониторинг и
pending-доставка могли отвечать reply к исходному пользовательскому запросу,
если такой якорь известен.

Если якорь неизвестен или сброшен, текущая доставка не должна ломаться: бот
отправляет обычное сообщение без reply.

## Решения v1

- Якоря хранятся только в памяти процесса.
- После перезапуска бота старые reply-связи теряются.
- Ключ якоря: `project_path + backend + session_id`.
- Якорь создаётся только для сообщения, которое реально принято в CLI-агент.
- Busy-ответы, monitoring mode, `/all`, `/sessions`, `/N`, `/pN`, `/3s12` и
  `/stop` не создают новый якорь.
- `/stop` сбрасывает якорь активного хода.
- Для Telegram-альбома якорем становится сообщение с caption, если оно есть;
  иначе первое сообщение альбома.
- В длинном ответе reply ставится только на первый Telegram-чанк.
- Если Telegram не принимает reply-параметры, бот повторяет отправку без reply.

## Архитектура

Нужно добавить модуль `reply_anchor_registry`.

Этот модуль хранит in-memory состояние reply-якорей и не зависит от Telegram API
или CLI-агентов. Его ответственность ограничена:

- записать активный якорь для пары проект/backend/сессия;
- вернуть активный якорь для доставки ответа;
- сбросить якорь после `/stop`;
- перенести якорь при замене временного session id на реальный.

Основные потребители:

- `bot.py` получает Telegram `message_id` из входящего update и передаёт его
  дальше вместе с пользовательским заданием;
- `claude_interaction.py` записывает якорь только после того, как запрос дошёл
  до `process_manager.send_message`;
- `telegram_sender.py` принимает optional reply-параметр и пробрасывает его в
  Telegram API;
- watcher, `/all` и pending-доставка читают якорь по источнику сообщения, но не
  создают его.

## Поток данных

1. Иван отправляет текст, одиночное фото, документ или альбом.
2. `bot.py` сохраняет `message_id` входящего Telegram-сообщения как кандидат на
   якорь, но ещё не записывает его в реестр.
3. Если сообщение отклонено из-за busy-состояния, monitoring mode или команды,
   реестр не меняется.
4. Когда запрос реально отправляется в CLI-агент, `claude_interaction.py`
   записывает якорь по ключу `project_path + backend + session_id`.
5. Если session id меняется с `_new_...` на реальный UUID, реестр переносит
   якорь на новый ключ.
6. При доставке ответа код ищет активный якорь:
   - прямой финальный ответ использует якорь текущего хода;
   - progress-сообщения используют тот же якорь;
   - watcher ищет якорь по своему project/backend/session;
   - `/all` ищет якорь по project/backend/session из all-mode link;
   - pending-доставка после переключения проекта использует якорь исходной
     сессии, если он ещё есть в памяти.
7. Если ответ разбит на несколько частей, reply получает только первая часть.

## Ошибки и крайние случаи

### Нет якоря

Бот отправляет обычное сообщение без reply. Это нормальный путь для старых
сессий, ручных сообщений из терминала, внешних источников и состояния после
перезапуска.

### `/stop`

Команда `/stop` сбрасывает активный якорь текущей сессии. Поздний хвост старого
хода, если он появится, не должен получить reply к старому или новому запросу.

### Новый запрос после `/stop`

Новый accepted-запрос создаёт новый якорь. Старый сброшенный якорь не
переиспользуется.

### Переключение проекта

Переключение проекта не чистит реестр. Если старый проект продолжит писать в
фоне, watcher или pending-доставка смогут найти якорь старого проекта.

### Переключение сессии

Команды переключения сессий не создают якорь. Следующее пользовательское
сообщение после переключения создаёт якорь уже для выбранной сессии.

### Telegram reply fallback

Если Telegram API возвращает ошибку при отправке reply, бот должен повторить
ту же отправку без reply-параметров. Пользовательский ответ не должен теряться
из-за невалидного или устаревшего `message_id`.

## Тестирование

### Юнит-тесты

Нужно покрыть:

- создание, чтение и сброс якоря в `reply_anchor_registry`;
- перенос якоря при `_new_... -> real session_id`;
- отсутствие обновления якоря для busy/monitoring/commands;
- выбор якоря для одиночного текста, фото, документа и альбома с caption;
- передачу reply-параметров в `telegram_sender`;
- fallback без reply при Telegram-ошибке;
- reply только на первом чанке длинного ответа;
- чтение якорей watcher-ом, `/all` и pending-доставкой без создания новых
  якорей.

### E2E-тесты

E2E-тесты обязательны для v1. E2E здесь означает реальный путь через Telegram:
тестовый пользователь отправляет сообщение боту через Telethon, а проверка
смотрит на фактическое сообщение, которое бот прислал обратно.

Нужно расширить `tests/e2e/test_client.py`, чтобы тест мог получить не только
текст ответа, но и Telegram message object. Для проверки reply достаточно
сравнить `reply_to_msg_id` ответа с `id` исходного сообщения пользователя.

Минимальный обязательный набор:

1. **Happy path**
   - создать новую сессию через `/new`;
   - отправить короткий текстовый запрос;
   - дождаться финального ответа текущей сессии;
   - проверить, что ответ reply к исходному пользовательскому сообщению.

2. **Busy не ворует якорь**
   - отправить долгий запрос A;
   - пока агент занят, отправить запрос B;
   - дождаться busy-ответа на B;
   - дождаться ответа по A;
   - проверить, что ответ по A reply к A, а не к B.

3. **Новый запрос после `/stop`**
   - отправить долгий запрос A;
   - отправить `/stop`;
   - отправить короткий запрос B;
   - дождаться ответа по B;
   - проверить, что ответ по B reply к B, а не к A и не к `/stop`.

## Acceptance criteria

- Ответы агента получают reply к последнему accepted Telegram-сообщению этой
  же пары project/backend/session.
- Busy, monitoring mode и команды не создают reply-якорь.
- `/stop` сбрасывает якорь остановленного хода.
- Переключение проекта и сессии не смешивает якоря разных источников.
- Длинный ответ ставит reply только на первый чанк.
- При ошибке Telegram reply-доставка повторяется без reply.
- После перезапуска бот продолжает доставлять ответы без reply к старым
  запросам.
- Юнит-тесты и обязательные E2E-сценарии проходят.
