# Детальный план этапа 4: UnicodeDecodeError во всех читателях JSONL (P2-23, P2-24, P3-50)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Закрыть механический кластер «UnicodeDecodeError» из реестра
`dev/docs/specs/04.07_13.12-project-wide-bug-review-findings.md`: P2-23 (Claude-читатели),
P2-24 (Codex-читатели) и P3-50 (`reply_routes.json`). Все три бага — один и тот же дефект в
разных файлах: файл сессии дописывается CLI на лету, читатель ловит хвост, оборванный посреди
многобайтного UTF-8 символа, и падает с `UnicodeDecodeError`. Ни один `except` этих читателей
такую ошибку не ловит.

**Architecture:** `UnicodeDecodeError` — подкласс `ValueError`, а НЕ `OSError`
(проверено эмпирически 05.07.2026: `issubclass(UnicodeDecodeError, OSError) == False`).
Все читатели ловят только `PermissionError`/`OSError`, поэтому декодная ошибка пролетает мимо
и роняет вызывающий код. Контент сессий русскоязычный (2-байтные UTF-8 символы), файлы
дописываются живьём — чтение `open(encoding="utf-8").readlines()` в момент, когда последний
символ дописан лишь наполовину, воспроизводимо падает:

```
UnicodeDecodeError: 'utf-8' codec can't decode byte 0xd0 in position 0: unexpected end of data
```

Три следствия одного корня:

- **P2-23 (Claude-читатели).** `claude_code_session_file_reader.py` (снапшот, курсор,
  метаданные списка сессий) и `session_reader.py` (`get_recent_sessions`,
  `get_session_messages`) — декодная ошибка пролетает мимо `except OSError`. Через голый
  `asyncio.gather` в путях сброса состояния (`reset_state`) одна битая сессия роняет
  переключение проекта (`/pN`) — с откатом и видимой пользователю ошибкой; повторная попытка
  срабатывает, когда CLI дописал символ.
- **P2-24 (Codex-читатели).** `codex_session_file_reader.py`,
  `codex_session_file_listing.py`, `codex_session_index.py`, `codex_session_metadata.py` —
  та же дыра. Усилитель: `asyncio.gather` в `_gather_optional_*` и в
  `_build_index_state` вызывается БЕЗ `return_exceptions=True`, поэтому один битый rollout,
  бросивший `UnicodeDecodeError`, обрывает весь index/листинг. Транзиентно и самолечится
  (watcher-цикл ретраит, `/all` показывает «временно недоступен»). Как только per-file
  чтение проглатывает `UnicodeDecodeError` и возвращает `None`, исключение до `gather` не
  доходит — усилитель обезврежен корневым фиксом (см. «Что НЕ входит», про
  `return_exceptions`).
- **P3-50 (`reply_routes.json`).** `reply_route_registry.load_routes` ловит
  `FileNotFoundError`/`JSONDecodeError` (порча JSON лечится штатно: чистое состояние,
  `_routes_loaded_from_disk = True`, следующая запись перезаписывает битый файл) и `OSError`
  (транзиентный сбой I/O: чистое состояние, но `_routes_loaded_from_disk = False` — запись
  заблокирована, чтобы не затереть валидный файл). Обрыв на многобайте (`read_text("utf-8")`
  → `UnicodeDecodeError`) не попадает ни в одну ветку: исключение пролетает наружу,
  `_routes_loaded_from_disk` остаётся `False`, персистентность маршрутов отключена до конца
  работы процесса.

**Единый фикс — явный `except UnicodeDecodeError` на границе каждого читателя.**
Декодная ошибка ловится ровно там, где уже ловятся `PermissionError`/`OSError`, и возвращается
тот же fallback, что для `OSError` в этом месте (`None` / `[]` / `empty_session_file_snapshot()`).
Для `reply_route_registry.load_routes` `UnicodeDecodeError` трактуется как порча контента —
идентично `JSONDecodeError` (чистое состояние, `_routes_loaded_from_disk = True`, битый файл
перезапишется при следующей записи). Логирование декодного случая — на уровне `debug`, а не
`warning`: это ожидаемое транзиентное состояние живого дописываемого файла (watcher опрашивает
раз в 2 секунды), и `warning` на каждый опрос во время записи создал бы шум в логах.

**Tech Stack:** Python 3.13, pytest (asyncio_mode=auto). Тесты юнит-уровня зеркалят структуру `src/`.

## Тестовый helper (общий для всех задач)

Добавляется локально в каждый тестовый файл, где нужен (по образцу уже существующих
per-file writer-helpers `write_jsonl_file` / `_write_jsonl_file` / `_write_rollout_file` —
проектная норма). Если по итогу helper понадобится в 3+ файлах и дублирование станет
заметным — поднять в `tests/conftest.py` и импортировать. Реализация:

```python
def write_jsonl_with_truncated_trailing_multibyte_char(
    file_path: Path, complete_records: list[dict[str, object]]
) -> None:
    """Пишет валидные JSONL-записи, затем строку, оборванную посреди 2-байтного UTF-8.

    Воспроизводит файл сессии, который CLI ещё дописывает: последняя строка кончается на
    первом байте 2-байтной последовательности (\\xd0 из 'П' = \\xd0\\x9f) без завершающего
    перевода строки, поэтому open(encoding='utf-8').readlines() бросает UnicodeDecodeError
    (это ValueError, НЕ OSError).
    """
    with file_path.open("wb") as file_handle:
        for record in complete_records:
            line = json.dumps(record, ensure_ascii=False) + "\n"
            file_handle.write(line.encode("utf-8"))
        # \xd0 — первый байт 2-байтной UTF-8 последовательности для 'П'; без второго байта
        # и без '\n' декодер падает на конце файла.
        file_handle.write(b'{"type":"user","message":{"role":"user","content":"\xd0')
```

Проверка воспроизведения (05.07.2026): такой файл роняет `open(encoding="utf-8").readlines()`
с `UnicodeDecodeError`; `except OSError` его НЕ ловит.

## Что НЕ входит в этот план (отложено / сознательно оставлено)

- **`asyncio.gather` без `return_exceptions=True` в `_gather_optional_*` и `_build_index_state`
  НЕ переписывается.** Корневой фикс (guard reads) убирает симптом P2-24 полностью: после того
  как per-file чтение проглатывает `UnicodeDecodeError` в `None`, исключение до `gather` не
  доходит, и один битый rollout больше не роняет весь листинг/индекс. Менять сам `gather` на
  `return_exceptions=True` сознательно НЕ делаем: текущие хелперы явно ловят `CancelledError`,
  отменяют соседние задачи и перевыбрасывают отмену; `return_exceptions=True` захватил бы
  `CancelledError` как обычный результат и сломал бы семантику отмены. Оборонительная изоляция
  каждой задачи от ЛЮБОГО неожиданного исключения (не только декодного) — отдельный заход, если
  понадобится; в рамках кластера UnicodeDecodeError она не требуется.
- **`errors="replace"` на уровне низкоуровневых читателей (`_read_file_lines_blocking`, прямые
  `open()`) НЕ применяется.** Такой подход убрал бы падение в 3–4 местах вместо 14, но менял бы
  семантику декодирования глобально: настоящая порча байтов в середине файла молча
  превращалась бы в U+FFFD и портила иначе валидные записи, без лога. Явный `except
  UnicodeDecodeError` точнее выражает намерение («файл ещё дописывается — пропускаем этот
  опрос, ретрайнем»), сохраняет исторические данные нетронутыми и логируется.
- **Дублирование `except`-веток.** 14 почти одинаковых веток — осознанное решение: они
  зеркалят уже принятый в этих же местах паттерн `except PermissionError`/`except OSError`
  (fallback у каждого места свой). Вынос в общий хелагер потребовал бы функций высшего порядка,
  которые скрыли бы поток управления и объединили бы разные fallback'ы и лог-сообщения. Держим
  идиому окружающего кода.
