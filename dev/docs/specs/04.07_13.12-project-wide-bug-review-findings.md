# Полное ревью проекта: реестр найденных багов

- **Дата:** 04.07.2026
- **Скоуп:** весь продакшен-код `src/claude_manager/` (60 модулей, ~15,5 тыс. строк), включая незакоммиченные правки рабочего дерева. Безопасность и уязвимости — вне скоупа (отдельный этап); здесь только баги и логические несоответствия.
- **Метод:** 9 параллельных агентов-ревьюеров (по группам модулей + сквозной по архитектурным инвариантам CLAUDE.md) → 98 находок → 6 агентов-верификаторов с адверсариальной проверкой каждой находки по коду (включая эмпирические репродукции: запуск функций модулей, проверка установленной python-telegram-bot 22.7, чтение реальных файлов сессий `~/.claude/projects` и `~/.codex/sessions`, логов бота) → дедупликация (9 дублей) → **89 уникальных багов**.
- **Вердикты верификации:** ни одна находка не отброшена целиком; 10 скорректированы (обычно понижена серьёзность из-за редкости триггера); у 2 находок ключевая предпосылка опровергнута частично (см. раздел «Скорректированные предпосылки»).

## Шкала приоритетов

- **P1** — потеря/дублирование сообщений в реалистичных повседневных сценариях; чинить в первую очередь.
- **P2** — неправильное поведение в правдоподобных, но менее частых сценариях; вводящие в заблуждение ошибки, блокирующие пользователя; порча данных с узким окном срабатывания.
- **P3** — edge cases, латентные баги без текущего продакшен-триггера, дрейф документации/спеки, медленная деградация.

## P1 — высокий приоритет

### P1-1. Курсор-заглушка `-1` из baseline попадает в unread_buffer — при входе в проект в чат вываливается ВСЯ история сессии — ✅ исправлено 04.07.2026 (этап 1 плана 04.07_21.27)

- **Где:** `coding_agent_session_file_poller.py:564-576` (baseline), `project_manager.py:253-268` (перенос в снапшот), `project_pending_delivery.py:219` (срез `messages[idx+1:]`), `all_projects_monitor.py:318, 536-544` (второй путь через /all).
- **Суть:** baseline после reset_state/включения /all строится из «курсорного» чтения (`read_session_file_cursor`), которое возвращает `messages=[]` → `last_delivered_idx = -1`. Это валидный sentinel только для raw-пути poller'а, но `SessionUnreadState` не имеет поля `parsed_message_count`, поэтому при сохранении снапшота в `unread_buffer` sentinel теряется. `_collect_pending_for_session_file` трактует `-1` как parsed-индекс и срезает `messages[0:]` — всю распарсенную историю.
- **Два пути входа:** (а) снапшот watcher-курсоров при уходе из проекта (`/pN`); (б) `_ensure_unread_snapshot` при первой доставке в /all (там же `last_modified_at=None` обходит mtime-гард `project_pending_delivery.py:200-204`).
- **Сценарий:** сессия с 50 сообщениями простаивает; пользователь ушёл в другой проект; терминальный CLI дописал один ход; пользователь вернулся — в Telegram уходит вся история сессии по одному сообщению.
- **Верификация:** CONFIRMED (оба пути, полная трасса), high.

### P1-2. /all: raw-курсор продвигается мимо придержанного «turn-active» сообщения — финальные ответы теряются навсегда — ✅ исправлено 04.07.2026 (этап 1 плана 04.07_21.27)

- **Где:** `all_projects_monitor.py:509-527` (`_candidate_indices` придерживает последнее сообщение при активном ходе), `all_projects_monitor.py:562` (`_next_state_from_snapshot` безусловно продвигает `raw_record_count`), `all_projects_monitor.py:677`.
- **Суть:** кандидаты выбираются по `raw_record_index > previous.raw_record_count`; при `is_turn_active` последнее сообщение придерживается, но raw-курсор всё равно продвигается за его индекс — на следующем поллинге придержанное сообщение уже не кандидат. Обычный watcher от этого защищён (`max(previous, len-2)` на hold, `poller:389-393`); /all — нет: parsed-курсор там вычисляется, но мёртв, потому что raw-путь всегда выигрывает.
- **Хуже заявленного (найдено верификатором):** если придержанное сообщение было единственным кандидатом, `deliverable` пуст → `_ensure_unread_snapshot` пропускается (`583-588`), а курсор продвигается (`677`) — ответ теряется и из pending-пути тоже, т.е. вообще навсегда.
- **Сценарий:** Codex-сессия в /all; поллинг попадает между записью финального ответа и `task_complete` — финал придержан, курсор ушёл вперёд; следующий поллинг финала не видит. Аналогично теряются промежуточные сообщения длинных ходов.
- **Верификация:** CONFIRMED, high.

## P2 — средний приоритет

### Доставка сообщений в Telegram

- **P2-1. Заголовок сессии добавляется ПОСЛЕ разбивки на 4096 — сообщение превышает лимит и теряется.** `telegram_response_delivery.py:280, 314, 349`. Единственная нерезаная часть ≤4096 + заголовок (~13-40 символов) = BadRequest; plain-text fallback для малоразмеченного текста тоже >4096 → второй BadRequest пробрасывается; в watcher-пути ошибка проглатывается, а курсор продвигается — сообщение потеряно навсегда. Воспроизведено (4108 символов). CONFIRMED, medium.
- **P2-2. Доставка файлов (`[SEND_FILE]`/`[SHOW_FILE]`) идёт мимо retry-логики и роняет весь финальный ответ.** `file_delivery.py:64-66, 75` шлют через голые `bot.send_message`/`bot.send_document` без обработки RetryAfter/TimedOut; маркеры обрабатываются ДО отправки текста ответа (`telegram_response_delivery.py:271→287`), так что одно исключение отменяет всю доставку. Замер: файл 300 КБ = 70 подряд идущих сообщений — флуд-контроль Telegram почти гарантирован. ADJUSTED (high→medium: мелкие файлы работают), medium.
- **P2-3. Regex курсива `\*(.+?)\*` съедает несвязанные звёздочки и портит текст.** `message_splitter.py:198` — нет граничных проверок (в отличие от правила `_` на строке 200). Воспроизведено: `SELECT * FROM a WHERE x * y` → `SELECT <i> FROM a WHERE x </i> y`. Ломает shell-глобы, SQL, арифметику в прозе (вне code-блоков). CONFIRMED, medium.
- **P2-4. Путь RetryAfter делает ровно одну повторную попытку и молча теряет часть сообщения.** `telegram_sender.py:143-152` безусловно выходит из retry-цикла; `handle_retry_after` (`88-94`) глотает любое исключение и возвращает None; вызывающие None игнорируют — в многочастном ответе молча выпадает середина. CONFIRMED, medium.
- **P2-5. `[SHOW_FILE]` большого файла замораживает весь бот на десятки секунд.** `file_delivery.py:39, 50` — синхронное чтение до 1 МБ и CPU-bound `telegramify_markdown.convert` прямо в event loop. Замер верификатора: 300 КБ markdown = 27 секунд полной заморозки (polling, watcher, watchdog). ADJUSTED (low→medium по замеру), medium.

- **P2-6. Редактирование сообщения пользователем роняет обработчик — правка никогда не обрабатывается.** `bot.py:176-206`: MessageHandler'ы без ограничения UpdateType принимают `edited_message` (проверено в установленной PTB 22.7), а обработчики безусловно разыменовывают `update.message` (`telegram_input_handlers.py:86` и др.) → AttributeError → «⚠️ Внутренняя ошибка». Редактирование сообщения — рутинное действие. CONFIRMED, medium.
- **P2-7. `/N` внутри глобального режима /all не гардится — бот попадает в гибридное полусостояние.** `telegram_session_handlers.py:307-335`: нет проверки `all_projects_monitor.is_enabled_for_chat` (в отличие от `/new` и `/pN`). После `/N` в /all: ввод текста открывается, но финальный ответ подавляется как «чат в /all» (`claude_interaction.py:594-617`), /all-поток продолжает сыпать сообщения всех проектов, обычный watcher остаётся на паузе. Если сессия не среди кандидатов /all (список заморожен на момент включения) — ответ не доставляется вообще до перезахода в проект. CONFIRMED (двумя ревьюерами независимо), medium.

### Переключение проектов и персистентность

- **P2-8. Переключение проекта не вызывает `daily_session_registry.reset_state()` — реестр дневных номеров может быть записан в файл ЧУЖОГО проекта.** `project_manager.py:229-232` (reset_state реестра отсутствует, вопреки CLAUDE.md), `daily_session_registry.py:559-611`: `load_registry` перенацеливает `_registry_path` на новый проект ДО awaited-чтения, без `_lock` и без сброса `_loaded_from_disk`; `_save_registry` (`205-208`) перечитывает глобальный `_registry_path` между двумя await — конкурентный `register_session` (реален при `concurrent_updates(256)`, например из /sessions-цикла) записывает реестр старого проекта в `daily_sessions.json` нового. Правильный паттерн (сброс флага под lock) есть в самом `reset_state` (`614-628`) — он просто не вызывается. Окно — миллисекунды; ущерб — порча нумерации (частично самолечится orphan-cleanup). CONFIRMED (двумя ревьюерами), medium.
- **P2-9. `deliver_pending_messages` очищает снапшот, не убедившись в доставке, и не изолирует ошибки по элементам.** `project_pending_delivery.py:89-115`: sender после исчерпанных ретраев возвращает None — снапшот всё равно очищается (сообщение потеряно); BadRequest с неудачным plain-text fallback пробрасывается и обрывает весь цикл, а следующий уход из проекта перезаписывает уцелевшие снапшоты курсорами «всё прочитано» — пропущенные сообщения становятся недоставляемыми навсегда. - **P2-13. Список кандидатов /all заморожен на момент включения и только сжимается.** `all_projects_monitor.py:477-478, 610-621, 693-704`: сессии, созданные после входа в /all, никогда не мониторятся (вопреки контракту модуля и CLAUDE.md); один транзиентный stat-сбой навсегда выкидывает живую сессию из мониторинга до перезахода в /all. ## Скорректированные предпосылки (важно для планирования фиксов)

- **«`claude --resume` мятит новый session_id на каждый ход» — неверно.** Новый id при resume — опциональное поведение (`--fork-session`, бот его не передаёт). По логу бота: 980 remap'ов temp→real против всего 23 UUID→UUID, сгруппированных в два инцидента ошибок/ретраев (16.06 и 25.06). Поэтому P2-18 — реальная, но редко срабатывающая дыра, а не повседневный отказ.
- **«/sessions постоянно перечитывает файлы из-за усечённых preview» — в основном неверно.** `PREVIEW_MAX_LENGTH=None` во всех читателях, усечения с `...` нет; остался узкий legacy-хвост (P3-54).
- **«resume_all поверх /all при неудачном выходе создаёт транзиентное окно» — неверно** (нет yield-точки между resume и повторной паузой); вместо этого подтверждено устойчивое multi-device нарушение (P3-16).
- Несколько находок помечены как **латентные** (сломанный код без текущего продакшен-вызова): P3-23, P3-24, P3-28, P3-34, P3-39 и часть P3-22. Они не влияют на пользователей сегодня, но взорвутся при первом же будущем использовании соответствующего API.
