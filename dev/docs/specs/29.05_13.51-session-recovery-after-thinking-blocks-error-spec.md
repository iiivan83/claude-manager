# Восстановление сессии после необратимой ошибки thinking-блоков

## Контекст

В проекте есть постоянная (невосстановимая) ошибка Claude Code CLI, которая ломает сессию насмерть:

```
API Error: 400 messages.N.content.M: thinking or redacted_thinking blocks in the latest assistant message cannot be modified
```

Первопричина разобрана в `dev/docs/session-reports/29-05/03-36_judge-tuner-bot-session-crash-diagnosis.md`. Кратко: бот на каждое сообщение запускает `claude -p --resume <session_id>` с зашитым `--effort max` (extended thinking включён всегда, `claude_runner.py:77`, `claude_code_backend.py`). Каждый блок размышлений несёт криптографическую подпись; правило API Anthropic — блоки размышлений в последнем сообщении ассистента переигрываются байт-в-байт. Когда длинный, насыщенный размышлениями ход обрывается или накапливает десятки подписанных блоков, повторный `--resume` подаёт на вход то же состояние JSONL и стабильно воспроизводит 400. Повтор бесполезен: каждый из 10 ретраев падает одинаково.

Что уже сделано (отчёт `dev/docs/session-reports/29-05/12-29_stop-resume-and-permanent-error-no-retry-fixes.md`, ADR `dev/docs/adr/29.05_12.29-session-change-documenter-backend-error-retryability-classification.md`): введена классификация ошибок backend по повторяемости. Есть enum `PermanentErrorKind` (значения `CONTEXT_OVERFLOW`, `USAGE_LIMIT`), метод backend `classify_permanent_error(error_text)`, гарды в `process_manager`, которые пропускают retry-цикл при постоянной ошибке, и человекочитаемые сообщения в `claude_interaction`. Ошибка thinking-блоков в эту классификацию **пока не входит** — она всё ещё уходит в 10 ретраев и спамит пользователя.

Эта спецификация добавляет два уровня поверх существующей машинерии:

1. **Распознавание** ошибки thinking-блоков как постоянной (по аналогии с overflow/limit).
2. **Восстановление**: бот предлагает пользователю перенести оборванную сессию в новую и, по подтверждению, заводит новую сессию, передавая ей ссылку на файл прошлой сессии — новая сессия сама читает историю и продолжает незавершённую работу.

## Цель и поведение для пользователя

Сценарий по шагам:

1. Сессия упирается в ошибку thinking-блоков. Бот распознаёт её как постоянную, **не уходит в ретраи**.
2. Бот проверяет, что файл прошлой сессии (`.jsonl` на диске) существует, и **спрашивает** пользователя одним сообщением: эта сессия не продолжается; напиши «да» — перенесу в новую; или /new — с чистого листа. Бот **ничего не пересказывает**.
3. Пользователь пишет **`да`** (обычным текстом, не командой). Бот заводит новую сессию, отправляет ей сообщение-затравку со **ссылкой** (путём к файлу) на прошлую сессию и инструкцией «прочитай и продолжи работу». Новая сессия сама читает файл и **сразу продолжает** незавершённую задачу.
4. Если вместо «да» пользователь пишет что-то другое, пока сессия в состоянии «ждём да» — бот **повторяет то же предложение** (напоминание) и **не трогает** сломанную сессию.
5. Если пользователь пишет `/new` — стандартное создание новой сессии, заметка о переносе сбрасывается.

Ключевые проектные решения (подтверждены с пользователем на этапе brainstorming):

- **Спрашиваем, не восстанавливаем молча.** Автоперенос без подтверждения отклонён.
- **Ссылка, а не пересказ.** Бот передаёт путь к файлу; новая сессия читает сама. Это намеренно: пересказ всего диалога снова упёрся бы в переполнение контекста.
- **Подтверждение словом «да», без команды.** Строгое распознавание — только само слово «да».
- **Чтение файла безопасно.** Ошибка 400 возникает только при `--resume` (переигрывание подписанных блоков в API). Чтение `.jsonl` с диска инструментом Read — это файловый ввод-вывод, переигрывания в API нет. Контракт подтверждён в отчёте-диагностике.

## Не-цели (out of scope)

- **Классификация thinking-ошибки для Codex.** Нет эмпирических образцов, как Codex сообщает об этой ситуации; проектное правило запрещает догадки о контрактах внешних систем. Codex наследует безопасный дефолт `classify_permanent_error → None` и в этот путь не попадает.
- **Изменение `--effort max`.** Пересмотр extended thinking — отдельная рекомендация №3 из отчёта-диагностики, не входит сюда.
- **Персистентность заметки о переносе на диск.** Заметка живёт только в памяти (см. «Риски»).
- **Остановка зависшего процесса прошлой сессии и расширение `/stop` на фоновые сессии.** Вторичные рекомендации RCA, радиус поражения шире, отложены.
- **Автопродолжение без участия пользователя.** Явно отклонено.

## Архитектурное решение

Распределение по слоям (границы из `CLAUDE.md` → «Слоёная архитектура»):

- **Контракт backend** (`coding_agent_backend.py`, `claude_code_backend.py`) — знает тексты ошибок конкретного CLI и классифицирует их. Здесь добавляется новый вид постоянной ошибки и его распознавание. `process_manager` остаётся backend-агностичным и менять его не нужно — существующие гарды уже пропускают retry для любой постоянной ошибки.
- **Состояние «ждём да»** (`pending_session_recovery_registry.py`, новый модуль) — in-memory заметка по чату, по образцу `silence_mode_registry` (но без диска). Не знает ни про Telegram, ни про процессы.
- **Транспортная оркестрация** (`claude_interaction.py`) — владеет текстами для пользователя, собирает затравку, распознаёт «да», решает, предлагать ли перенос, и выполняет перенос. Это расширение уже существующей ветки постоянных ошибок в `handle_claude_result`.
- **Точка входа** (`bot.py`) — тонкая: в обработчике текста один вызов оркестрации перехвата «да»/напоминания; в `/new` — сброс заметки.

Почему распознавание ошибки в контракте backend, а не в `process_manager`: формат ошибок специфичен для CLI-движка (Claude vs Codex), а `process_manager` backend-агностичен. Это тот же принцип, по которому уже размещены overflow/limit.

Почему тексты для пользователя в `claude_interaction`, а не в `process_manager`: соблюдение границ — инфраструктура не знает про Telegram и формулировки. Тот же принцип, что и `PERMANENT_ERROR_MESSAGES`.

## Новые сущности (имена проверены через ревьюера понятности)

- **`THINKING_BLOCKS_CANNOT_BE_MODIFIED`** — новое значение enum `PermanentErrorKind` (строковое значение `"thinking_blocks_cannot_be_modified"`). Файл: `coding_agent_backend.py`.
- **`CLAUDE_THINKING_BLOCKS_ERROR_MARKERS`** — константа-кортеж подстрок для распознавания текста ошибки. Файл: `claude_code_backend.py`.
- **`pending_session_recovery_registry`** — новый модуль: in-memory заметка по чату об отложенном переносе. Файл: `src/claude_manager/pending_session_recovery_registry.py`.
- **`PendingSessionRecovery`** — dataclass с полями заметки. В модуле выше.
- **`SESSION_RECOVERY_OFFER_MESSAGE`** — константа-сообщение для пользователя (предложение переноса, оно же напоминание). Файл: `claude_interaction.py`.
- **`build_session_recovery_prompt`** — функция: собирает сообщение-затравку для новой сессии (путь к файлу прошлой сессии + последний запрос + инструкция «прочитай и продолжи»). Файл: `claude_interaction.py`.
- **`is_recovery_confirmation`** — функция: проверяет, что текст — подтверждение «да». Файл: `claude_interaction.py`.
- **`try_handle_pending_session_recovery`** — функция: при входящем тексте проверяет заметку и привязку, выполняет перенос по «да» либо шлёт напоминание; возвращает `True`, если сообщение поглощено. Файл: `claude_interaction.py`.
- **`start_recovery_session_continuing_previous_work`** — функция: создаёт новую сессию, уведомляет пользователя, передаёт затравку, чистит заметку, запускает диалог новой сессии. Файл: `claude_interaction.py`.

## Граничные случаи

- **Файл прошлой сессии не найден.** Перенос невозможен — заметка не сохраняется, уходит fallback-сообщение из `PERMANENT_ERROR_MESSAGES` («файл не найден, /new»). На практике редко: сессия только что выдала result-событие, её `.jsonl` есть на диске.
- **Пользователь переключил проект или сессию, пока ждём «да».** Привязка чата больше не равна `broken_session_id`, `try_handle_pending_session_recovery` возвращает `False`, текст идёт обычным путём. Заметку не чистим намеренно: при возврате на сломанную сессию предложение снова валидно.
- **«Да» без ожидающего переноса.** Заметки нет → `False` → обычный текст в Claude. Это главная защита: «да» означает «переноси» только при заметке И совпадении привязки. А раз сломанная сессия мертва, живого Claude, ждущего ответ «да», там нет — спутать не с чем.
- **Новая сессия тоже сломалась.** Та же машинерия сработает заново: новое распознавание → новое предложение → новое «да». Зацикливания нет — каждый перенос требует ручного подтверждения.
- **Перезапуск бота во время ожидания «да».** Заметка в памяти теряется. Самовосстановление: привязки чата персистентны (`sessions.json`), следующее сообщение в ту же сессию снова упрётся в ту же ошибку, и бот снова предложит перенос.
- **Несколько сломанных сессий подряд в одном чате.** Заметка одна на чат, перезаписывается последней. Это корректно — пользователь работает в одной активной сессии за раз (однопользовательский инвариант).

### Ручная проверка через Telegram

Реальную 400-ошибку thinking-блоков воспроизвести по требованию трудно (нужна конкретная длинная сессия с обрывом). Поэтому:

- Классификацию покрываем юнит-тестами на реальном тексте ошибки из отчёта.
- Поток переноса проверяем вручную, временно подсунув маркер (например, отправив через тестовый стенд result-событие с текстом-ошибкой, либо включив маркер в тестовую сессию): убедиться, что приходит предложение, «да» создаёт новую сессию и она читает файл и продолжает, посторонний текст даёт напоминание, `/new` сбрасывает заметку.
- Ограничение фиксируется честно: на живой 400 от Claude поток проверяется при первом естественном повторении инцидента.

## Риски

1. **Текст ошибки изменится со стороны Anthropic.** Если API переформулирует сообщение, подстрока перестанет совпадать и классификация молча отключится (ошибка снова уйдёт в ретраи). Митигация: подстрока выбрана из стабильного хвоста без переменных индексов; источник задокументирован; кандидат — дозаписать ошибку в `dev/docs/claude-cli-stream-json-protocol.md`.
2. **Заметка в памяти теряется при рестарте.** Самовосстановление через повторное срабатывание (см. граничные случаи). Персистентность на диск — out of scope, добавляется при необходимости.
3. **Зависимость на `.jsonl` (Claude-only).** Допустима, потому что вид ошибки приходит только от Claude. Если в будущем понадобится перенос для другого backend — вынести вычисление пути в контракт backend отдельным методом.
4. **Новая сессия читает большой JSONL.** Чтение безопасно, но новая сессия может сама решить кратко пересказать историю — это поведение агента, вне нашего контроля. На первопричину 400 это не влияет (нет `--resume` старого состояния).
5. **Ложное «да».** Снято гардом «заметка + совпадение привязки»: у сломанной сессии нет живого процесса Claude, который ждал бы ответ «да».
