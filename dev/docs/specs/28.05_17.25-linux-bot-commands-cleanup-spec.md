# Linux-only: чистка macOS-хвоста в командах бота

## Контекст

Claude Manager (Telegram-бот) изначально написан под macOS, сейчас перевезён на Linux. В коде и shell-скриптах остался macOS-специфичный слой: вызовы `launchctl`, путь к логам в `~/Library/Logs/`, default `DEFAULT_PROJECTS_ROOT = /Users/ivan/Desktop/claude-sandbox`, три shell-скрипта, целиком написанных под `launchctl` и macOS-флаги `stat -f`. Команда `/restart` на Linux нерабочая.

Текущая инфраструктура на Linux:

- Systemd user service `~/.config/systemd/user/claude-manager.service` активен. `ExecStart` напрямую запускает entry point `claude-manager` из editable install в `.venv`. `Restart=always`, `RestartSec=10`, `TimeoutStopSec=30`.
- Венв и editable install находятся в `/home/ivan/claude-sandbox/claude_manager/.venv/`. macOS-овский принцип «venv вне TCC-зоны» на Linux нерелевантен (нет TCC, нет APFS-флага `UF_HIDDEN`).
- `.env` содержит `PROJECTS_ROOT_DIR` явно (проверено).

## Цель

Убрать весь macOS-зависимый код и инфраструктуру. После работы:

- `/restart` корректно перезапускает бота через systemd.
- Логи пишутся в XDG-стандартный путь.
- `PROJECTS_ROOT_DIR` обязателен — пустая переменная даёт `ConfigError` при старте.
- Shell-скрипты в корне работают только под Linux (`systemctl --user`, `inotifywait`, без `launchctl`, без `stat -f`).
- CLAUDE.md не содержит macOS-only принципов эксплуатации и macOS-команд разработки.
- `dev/docs/bot-launch-infrastructure.md` и `dev/docs/deployment-guide.md` описывают только Linux-инфраструктуру.

Совместимость с macOS не сохраняется — это явное проектное решение.

## Архитектурное решение для `/restart`

Подход — детач через `systemctl --user restart`. Это калька с macOS-схемы, но с заменой команды.

Поток:

1. Пользователь шлёт `/restart` в Telegram.
2. Handler пишет в чат «Перезапускаюсь через 2 сек».
3. Handler сохраняет `chat_id` в маркер-файл `/tmp/claude-manager-restart-chat-id` (логика остаётся неизменной).
4. Handler запускает отвязанный subprocess `bash -c "sleep 2 && systemctl --user restart claude-manager.service"` с `start_new_session=True`, `stdout/stderr=DEVNULL`. Бот не ждёт завершения subprocess.
5. Handler возвращает управление.
6. Через 2 секунды отвязанный bash вызывает `systemctl --user restart`. Systemd шлёт SIGTERM текущему процессу бота, даёт до 30 секунд на корректное завершение (`TimeoutStopSec=30`), запускает новый процесс через `ExecStart`.
7. В `post_init` нового процесса бот читает `/tmp/claude-manager-restart-chat-id`, отправляет «Перезапустился, готов к работе», удаляет маркер.

Отвязка через `start_new_session=True` нужна, чтобы команда `systemctl restart` пережила смерть бота-инициатора. Без отвязки systemd убил бы и сам subprocess как часть cgroup сервиса.

Альтернатива — `sys.exit(0)` + полагаться на `Restart=always`. Отклонена: даёт 10 секунд паузы (`RestartSec=10`), путает мониторинг systemd (выглядит как падение), требует править service-файл.

## Изменения — shell-скрипты (параллельная перезапись)

Старые скрипты в корне проекта удаляются. Новые пишутся с нуля, с теми же именами, под Linux. Это «параллельная разработка» в смысле логики (новая логика не наследует macOS-обёртку), а не файловая параллельность.

### Удаляем целиком

`start-claude-manager.sh` — wrapper для launchd с retry-логикой. На Linux systemd сам делает retry через `Restart=always` + `RestartSec=10`. Wrapper не нужен, `ExecStart` в service-файле уже указывает на entry point напрямую.

### Пишем заново

`restart-claude-manager.sh` — инструмент для запуска из терминала. Структура:

- Заголовок: `#!/bin/bash` + `set -e`.
- **Preflight**: проверка существования `.venv/bin/claude-manager` (файл), проверка `import claude_manager` через `.venv/bin/python -c`. При ошибке — вывод диагностики (`sys.executable`, версия Python), exit 1.
- **Рестарт**: `systemctl --user restart claude-manager.service`.
- **Post-flight**: `systemctl --user is-active claude-manager.service` (должен вернуть `active`), `pgrep -f "claude-manager"` (должен найти PID). При ошибке — вывод последних 30 строк `~/.local/state/claude-manager/claude-manager.log` и `journalctl --user -u claude-manager.service -n 30`, exit 1.

Принцип «verify-before-and-after» сохраняется. Скрипт безопасен для запуска из терминала, но не из самого бота — самоперезапуск идёт через `/restart` и отвязанный subprocess.

`watch_and_restart.sh` — наблюдатель за изменениями `.py`-файлов для разработки. Структура:

- Заголовок: `#!/bin/bash` + `set -e`.
- **Проверка зависимости**: `command -v inotifywait` — если нет, выводим подсказку «установите пакет inotify-tools» и exit 1.
- **Основной цикл**: `inotifywait -m -r -e modify,create,delete --include '.*\.py$' src/` пишет события в stdout. При каждом событии — `systemctl --user restart claude-manager.service`.
- Логика дебаунса: пропускать повторные события в течение 1 секунды после последнего рестарта (несколько подряд изменений при сохранении нескольких файлов в IDE).

## Контракты с внешними системами

`systemctl --user restart claude-manager.service` — стандартная команда systemd. Команда возвращает 0 при успешном принятии задания (не при успешном рестарте — это асинхронно). Реальный статус нужно проверять через `is-active` после паузы.

`inotifywait` из пакета `inotify-tools`. На Debian/Ubuntu — `sudo apt install inotify-tools`. Скрипт `watch_and_restart.sh` проверяет наличие команды на старте.

XDG Base Directory Specification: `~/.local/state/<app>/` — стандартное место для пользовательского состояния и логов. Папка создаётся ботом при первом запуске.

Systemd журнал: `stdout`/`stderr` бота под `Type=simple` без явного перенаправления `StandardOutput`/`StandardError` автоматически попадают в `journalctl --user -u claude-manager.service`. Это отдельный поток от `RotatingFileHandler`, который пишет в `~/.local/state/claude-manager/claude-manager.log`. Для диагностики нужны оба источника.

## Риски

1. **D-Bus user session отвалится** — `systemctl --user` перестанет работать. Маловероятно (бот сейчас запущен под `systemctl --user`, значит сессия живая). Если случится — пользователь увидит «Перезапускаюсь через 2 сек», но рестарт не произойдёт. Митигация: subprocess пишет stderr в `/tmp/claude-manager-restart.log` (опционально, если по итогам ручной проверки возникнет проблема).

2. **Удаление `DEFAULT_PROJECTS_ROOT` сломает запуск на чистой машине** — у пользователя в `.env` переменная уже задана, риск только для будущих развёртываний. Митигация: понятная ошибка в `ConfigError` плюс упоминание `PROJECTS_ROOT_DIR` в `.env.example`.

3. **Тесты могут оказаться завязаны на удалённые константы** — гипотетически, мы не проверяли. Митигация: после правок прогон `pytest tests/ -v`, исправление падений.

## Out of scope

- Перенос на system-level systemd unit (`/etc/systemd/system/`) — остаёмся на `--user`.
- Контейнеризация (Docker, podman) — не часть этой задачи.
- Поддержка дистрибутивов без systemd (Alpine с OpenRC и т.п.) — не нужна, у пользователя стандартный Ubuntu/Debian.
- Реструктуризация Python-кода — только точечные правки macOS-специфики.
- E2E-тесты — существующие прогоняются, новые под эту задачу не добавляем.
