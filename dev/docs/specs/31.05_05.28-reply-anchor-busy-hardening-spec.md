# Спецификация: hardening busy-сценариев для reply-anchor

## Контекст

Claude Manager - локальный Telegram-бот, который отправляет сообщения Ивана в CLI-агента и возвращает ответы в Telegram. Reply-anchor - это сохранённый `message_id` входящего сообщения Ивана, к которому бот прикрепляет будущий ответ агента.

31-05-2026 был найден баг: если Иван отправлял новое сообщение, пока Codex уже обрабатывал предыдущий запрос, busy-ветка могла стереть anchor активного хода. Из-за этого финальный ответ агента уходил обычным сообщением, без Telegram reply к исходному принятому запросу.

## Что исправлено точечным фиксом

Текущий маленький фикс должен закрыть главный пользовательский симптом:

- rejected busy-сообщение больше не стирает уже существующий anchor активного хода;
- быстрый busy-check выполняется до записи нового anchor;
- если между предварительной проверкой и `process_manager.send_message()` всё равно случилась гонка, прежний anchor восстанавливается;
- busy-ответ по-прежнему может уходить без reply к rejected-сообщению, потому что это отдельное улучшение с большим радиусом изменений.

## Зачем нужна отдельная доработка

Точечный фикс защищает финальный ответ активного хода, но не завершает всю модель поведения busy-сценариев. Сейчас в проекте параллельно реализуется reply-routing v1, где появляется другой реестр: `reply_route_registry`. Его нельзя смешивать с `reply_anchor_registry`.

Оставшиеся изменения лучше делать отдельной задачей, чтобы не смешать три разных смысла:

- **reply-anchor** - куда прикрепить ответ агента в Telegram;
- **busy-reply** - как ответить Ивану на rejected-сообщение, когда агент занят;
- **reply-route** - куда отправить новый текст Ивана, если он ответил reply на сообщение бота.

## Требования

### 1. Busy-ответ должен быть безопасным для anchor

Busy-сообщение не должно:

- создавать новый `reply_anchor_registry` anchor;
- перезаписывать anchor активного принятого хода;
- удалять anchor активного принятого хода;
- мешать переносу anchor при `_new_* -> real session_id`.

### 2. Busy-ответ может быть Telegram reply к rejected-сообщению

Если у rejected-сообщения есть входящий Telegram `message_id`, бот может отправить busy-ответ reply к нему. Это улучшает читаемость чата: Иван сразу видит, какое короткое сообщение было отвергнуто.

Ограничение: такой reply не должен попадать в `reply_anchor_registry`. Это одноразовый Telegram-параметр отправки busy-сообщения, а не anchor для будущего ответа агента.

### 3. Запись anchor должна происходить после принятия запроса

Идеальная модель:

1. Telegram handler получает входящий `message_id` и передаёт его как candidate.
2. `process_manager` атомарно принимает запрос, то есть проходит busy-gate и ставит busy-флаг.
3. Только после этого `claude_interaction` записывает candidate в `reply_anchor_registry`.
4. Progress и финальный ответ читают уже подтверждённый anchor.

Если текущий `process_manager` не даёт такого hook-а, допустим промежуточный вариант: предварительный busy-check плюс восстановление старого anchor при `ProcessManagerError`. Это уже сделано точечным фиксом, но остаётся архитектурно менее чистым.

### 4. Поведение должно быть одинаковым для основных входов

Проверить и привести к одному контракту:

- обычный текст;
- одиночное фото;
- документ;
- альбом фото;
- routed reply из будущего `reply_route_handler`;
- fast-path busy до скачивания файлов;
- race-path, когда предварительная проверка сказала “свободно”, но `send_message` получил busy.

## Нефункциональные требования

- Не смешивать `reply_anchor_registry` и `reply_route_registry`.
- Не добавлять постоянное хранение anchor в файлы.
- Не раздувать `claude_interaction.py`: файл уже больше 500 строк, новые общие helper-ы лучше выносить в маленькие модули или локальные приватные функции только при реальной необходимости.
- Не менять семантику `/stop`: `/stop` остаётся командой управления и может сбрасывать anchor остановленного хода.

## Тесты

Минимальный набор:

- busy rejected turn не стирает существующий anchor;
- busy rejected turn не заменяет anchor на свой `message_id`;
- busy-ответ на текст может быть reply к rejected-сообщению, если этот режим будет включён;
- busy fast-path для фото и документа не создаёт anchor;
- race-path через `ProcessManagerError("already busy")` восстанавливает старый anchor;
- финальный ответ активного хода остаётся reply к исходному принятому сообщению после одного или нескольких busy-сообщений;
- `/stop` по-прежнему сбрасывает anchor остановленного хода.

## Связь с reply-routing v1

Текущий план `31.05_05.02-reply-routing-v1-implementation-plan.md` должен продолжать отвечать только за маршрутизацию входящих reply Ивана к исходной сессии. Эта спецификация не заменяет тот план.

Если обе задачи выполняются рядом, порядок безопаснее такой:

1. Завершить точечный hotfix, который сохраняет существующий anchor при busy.
2. Продолжить reply-routing v1 без изменения смысла `reply_anchor_registry`.
3. Отдельно реализовать busy-hardening из этой спецификации, особенно если потребуется сделать busy-ответы настоящими Telegram reply.

## Критерий готовности

Задача считается готовой, когда busy-сообщения больше не могут испортить anchor активного хода, а все места отправки busy-ответов либо явно отправляют reply к rejected-сообщению, либо явно документируют, почему этот путь остаётся обычным сообщением.
