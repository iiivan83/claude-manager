# Детальный план этапа 3: жизненный цикл хода (P2-19, P2-20)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Закрыть кластер «жизненный цикл хода» из реестра `dev/docs/specs/04.07_13.12-project-wide-bug-review-findings.md`: P2-19 (флаг `handler_owns_final_delivery` протекает после `/stop`/ошибки → следующий внешний финал молча теряется) и P2-20 (finally первого хода сносит паузу и watchdog ВТОРОГО хода той же сессии). Оба бага живут в одном месте — finally-блоке `claude_interaction.send_to_claude_and_respond` — и лечатся одним механизмом владения ходом.

**Architecture:** Общий корень кластера — finally-блок безусловно сносит состояние наблюдателя за сессией (пауза, watchdog, флаг `handler_owns_final_delivery`), не проверяя, принадлежит ли это состояние ещё текущему ходу. Два следствия:

- **P2-20 (гонка двух ходов).** busy-флаг снимается в finally `process_backend_send` РАНЬШЕ, чем `send_to_claude_and_respond` доставит финал и сгенерирует summary (окно до 45 сек, `SESSION_SUMMARY_TIMEOUT_SECONDS`). Второе сообщение той же сессии в этом окне проходит `is_busy`, ставит свою паузу и свой watchdog. Затем finally первого хода вызывает `cancel_agent_silence_watchdog` и `resume_session` по тому же `session_id` — и сносит watchdog и паузу ВТОРОГО хода. У второго хода дублируются промежуточные сообщения (watcher + progress) и нет watchdog-защиты.
- **P2-19 (протечка флага).** `resume_session` снимает `handler_owns_final_delivery` только при чистом завершении хода (`not is_turn_active` и появилось новое сообщение). После `/stop` (SIGTERM) `is_turn_active` остаётся True, а нового финала нет — условие не срабатывает, флаг остаётся True. Одновременно finally обнуляет `paused_at`, из-за чего 120-секундный safety-автоклир в поллере (`PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS`, срабатывает только при `paused_at is not None`) никогда не сработает. Флаг живёт вечно; следующий внешний финал (терминальное продолжение той же сессии) придерживается как «handler-owned» и не доставляется.

**Единый механизм — токен владения паузой (`pause_owner_token`).** Каждый ход `send_to_claude_and_respond` создаёт уникальный токен-объект и передаёт его в `pause_session`. Второй ход перезаписывает токен в состоянии сессии своим. finally первого хода спрашивает наблюдателя `is_pause_owner(session_id, backend, token)` и трогает паузу/watchdog/флаг только если токен всё ещё его. Токен живёт в объекте `SessionWatcherState`, который при смене `session_id` (temp→real) переносится тем же объектом через `update_session_id`, поэтому переживает ремап. progress-события того же хода (`reset_watchdog_on_progress`) переставляют паузу тем же токеном.

**Tech Stack:** Python 3.13, pytest (asyncio_mode=auto), dataclasses. Тесты юнит-уровня зеркалят структуру `src/`.
