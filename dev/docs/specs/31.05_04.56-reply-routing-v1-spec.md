# Спецификация: адресные ответы на сообщения бота v1

## Контекст

Claude Manager — локальный Telegram-бот, который позволяет управлять CLI-агентами
из Telegram. Пользователь может работать в конкретном проекте и сессии или перейти в
режим `/all`, где бот показывает сообщения из всех проектов.

Сейчас в режиме `/all` обычный текст, фото и документы блокируются: бот просит сначала
перейти в проект и сессию. Новая возможность должна убрать лишний ручной шаг для
текста: если Иван отвечает reply на сообщение бота, бот должен понять, к какой
проектной сессии относилось это сообщение, и отправить текст туда.

Эта спецификация описывает обратное направление по сравнению с существующими
reply-якорями. Reply-якорь хранит, к какому сообщению Ивана привязать ответ агента.
Адресный ответ хранит, куда отправить новый текст Ивана, если он ответил на сообщение
бота.

## Цель

Сделать текстовый reply-routing v1:

- Иван отвечает текстом на сообщение бота.
- Бот находит исходную проектную сессию по `message_id` сообщения бота.
- Бот отправляет текст в эту сессию.
- Бот не переключает пользователя автоматически.
- Бот показывает короткое подтверждение или ошибку со ссылкой.

## Термины

- **Адресный ответ** — входящее Telegram-сообщение Ивана, которое является reply на
  сообщение бота и должно быть отправлено в сессию-источник этого сообщения.
- **Маршрут ответа** — запись вида `chat_id + bot_message_id -> project_path +
  session_id + backend + link metadata`.
- **Текущий режим** — место, где находится пользователь в момент входящего reply:
  `/all` или конкретный проект.
- **Ссылка** — Telegram-команда для ручного перехода: `/3s12` из `/all` или `/12`
  внутри проекта.

## Пользовательское поведение

### Успешный текстовый reply из `/all`

Если пользователь находится в `/all` и отвечает текстом на сообщение бота, бот:

1. Находит маршрут по `message_id` сообщения бота.
2. Отправляет текст в найденную проектную сессию.
3. Остаётся в `/all`.
4. Показывает короткое подтверждение:

```text
Передал в /3s12
```

В подтверждении не нужно писать backend, название проекта, `#12`, UUID сессии или
другие расширенные данные. Достаточно команды, по которой Иван может перейти вручную.

### Успешный текстовый reply внутри проекта

Если пользователь находится внутри конкретного проекта и отвечает текстом на сообщение
бота, которое относится к сессии этого же проекта, бот:

1. Отправляет текст в сессию-источник сообщения.
2. Не меняет активную сессию пользователя.
3. Показывает подтверждение:

```text
Передал в /12
```

### Reply внутри проекта на сообщение из другого проекта

Если пользователь находится внутри проекта, но отвечает на старое сообщение бота,
которое относится к другому проекту, короткая ссылка `/12` будет неоднозначной.
В этом случае бот должен использовать полный all-link:

```text
Передал в /3s12
```

Автоматически переключать пользователя в другой проект нельзя.

### Занятая целевая сессия

Если целевая сессия занята, реакция должна быть смыслово такой же, как текущая
реакция на сообщение в занятую активную сессию: сообщение не отправляется, пользователь
видит, что нужно подождать или остановить текущий ход.

Формат для `/all`:

```text
Не передал в /3s12: сессия занята. Подождите или /stop
```

Формат внутри проекта:

```text
Не передал в /12: сессия занята. Подождите или /stop
```

Backend-лейбл вроде `Codex` или `Claude` в этих сообщениях не нужен.

### Вложения в адресном reply

В v1 адресные ответы поддерживают только текст. Если пользователь отвечает на
сообщение бота и прикладывает фото, документ, файл или альбом, бот ничего не
отправляет в целевую сессию.

Формат для `/all`:

```text
Не передал в /3s12: ответы с вложениями пока не работают, этот функционал ещё не сделали
```

Формат внутри проекта:

```text
Не передал в /12: ответы с вложениями пока не работают, этот функционал ещё не сделали
```

Это ограничение относится именно к новому входящему сообщению Ивана с вложением. Если
Иван отвечает обычным текстом на сообщение бота, которое само было связано с файлом или
фото, это обычный текстовый адресный ответ.

### Неизвестный или устаревший маршрут

Если пользователь отвечает на сообщение бота, но маршрут не найден, бот не должен
угадывать цель по активной сессии.

Причины:

- бот перезапустился и in-memory реестр очистился;
- сообщение было отправлено до появления reply-routing;
- сообщение не является маршрутизируемым ответом агента;
- route registry был очищен в тестах или при сбросе состояния.

Сообщение пользователю:

```text
Не понял, куда передать ответ. Нажми ссылку на нужную сессию и отправь сообщение там
```

## Нецели v1

В первой версии не нужно делать:

- автоматическое переключение проекта или сессии;
- персистентный route registry на диске;
- поддержку фото, документов, файлов и альбомов в адресном reply;
- очередь сообщений для занятой сессии;
- маршрутизацию команд вроде `/stop`, `/new`, `/projects`;
- вывод backend-лейбла в подтверждениях и ошибках;
- расширенные подписи вида `project_name #12 Codex`;
- поддержку reply на файлы, отправленные через `[SEND_FILE:path]` или
  `[SHOW_FILE:path]`, если они не проходят через общий механизм регистрации
  маршрутизируемых Telegram-сообщений.

## Архитектурные решения

### Новый route registry

Нужно добавить отдельный in-memory модуль, например
`src/claude_manager/reply_route_registry.py`.

Он не должен заменять `reply_anchor_registry`. Эти реестры решают разные задачи:

- `reply_anchor_registry` — куда бот должен поставить Telegram reply при доставке
  ответа агента;
- `reply_route_registry` — куда отправить новый текст Ивана, если он ответил на
  сообщение бота.

Минимальная модель:

- ключ: `chat_id + bot_message_id`;
- значение: `project_path`, `session_id`, `backend`, `project_number`,
  `session_number`, `project_name` для диагностики и построения ссылок.

Хранение только в памяти процесса. После рестарта маршруты теряются, и это нормальное
поведение v1.

### Регистрация исходящих сообщений бота

Бот должен регистрировать route для сообщений, на которые Иван может ответить.

В v1 маршрутизируемыми должны быть:

- ответы текущей активной сессии;
- watcher-сообщения внутри проекта;
- сообщения из `/all`;
- pending-доставка после входа в проект, если известна исходная сессия.

Не должны регистрироваться:

- подтверждения `Передал в ...`;
- ошибки `Не передал ...`;
- служебные сообщения `/projects`, `/sessions`, `/new`, `/stop`;
- предупреждения monitoring mode;
- сообщения без известного `project_path + session_id + backend`.

### Получение `message_id` после отправки в Telegram

Сейчас низкоуровневый отправитель Telegram отправляет сообщение и не возвращает
`message_id`. Для reply-routing нужно изменить контракт отправки:

- отправка одного Telegram-сообщения должна возвращать отправленный Telegram message
  или минимум его `message_id`;
- отправка длинного ответа, разбитого на несколько частей, должна вернуть `message_id`
  каждой части;
- каждый chunk длинного ответа должен регистрироваться с одним и тем же route target,
  чтобы reply на любую часть попал в правильную сессию.

### Route target должен передаваться явно

Нельзя полагаться только на глобальный `config.WORKING_DIR` или активную сессию чата.
Для регистрации route delivery-слой должен явно знать источник сообщения:

- `project_path`;
- `session_id`;
- `backend`;
- `session_number`;
- для `/all` дополнительно `project_number`.

Если текущий delivery API не передаёт эти поля, его нужно расширить через optional
`route_target` или эквивалентную структуру.

### Обработка входящего reply

Обработчики входящего текста, фото и документов должны сначала проверить:

1. Есть ли `reply_to_message`.
2. Есть ли route для `reply_to_message.message_id`.
3. Какой сейчас режим пользователя: `/all` или проект.

Если это текстовый reply с найденным route:

- не выполнять обычную блокировку текста в `/all`;
- не вызывать `project_manager.switch_project`;
- не вызывать `session_manager.set_active_session`;
- отправить текст в целевую сессию напрямую по сохранённому `project_path`,
  `session_id` и `backend`;
- показать подтверждение или ошибку.

Если это reply с вложением и найденным route:

- не скачивать файл;
- не отправлять запрос в CLI-агент;
- показать сообщение о неподдержанном функционале с правильной ссылкой.

Если route не найден, оставить текущее поведение для обычного режима и показать
понятную ошибку для reply в `/all`.

### Отправка в целевую сессию без переключения пользователя

Адресный reply должен запускать отправку в CLI-агент по сохранённому target:

- `cwd=project_path`;
- `session_id=session_id`;
- `backend=backend`.

Глобальное состояние текущего проекта и активной сессии пользователя менять нельзя.
Это ключевой инвариант функции.

Результат агента не нужно дополнительно доставлять из reply-handler-а напрямую, если
его уже доставит существующий watcher или `/all`-monitor. Нужно избежать дублей:
подтверждение о передаче и ответ агента — разные сообщения.

## Формирование ссылок

Формат зависит от текущего режима пользователя и target:

- пользователь в `/all` — использовать `/{project_number}s{session_number}`;
- пользователь в проекте, target находится в текущем проекте — использовать
  `/{session_number}`;
- пользователь в проекте, target находится в другом проекте — использовать полный
  `/{project_number}s{session_number}`.

Подписи с backend, названием проекта и расширенным номером сессии не добавляются.

## Ошибки и крайние случаи

### Route найден, но целевая сессия занята

Сообщение не отправляется. Бот показывает `Не передал ...: сессия занята. Подождите
или /stop`.

### Route найден, но целевой проект больше не доступен

Сообщение не отправляется. Бот показывает:

```text
Не передал в /3s12: проект недоступен
```

Если пользователь находится внутри проекта и target был локальным, использовать
короткую ссылку:

```text
Не передал в /12: проект недоступен
```

### Route найден, но session_id больше не читается backend-ом

Сообщение не отправляется. Бот показывает:

```text
Не передал в /3s12: сессия недоступна
```

или:

```text
Не передал в /12: сессия недоступна
```

### CLI не запустился или вернул постоянную ошибку до принятия запроса

Бот показывает ошибку передачи со ссылкой. Текст ошибки должен быть коротким и не
должен содержать traceback.

### Пользователь ответил командой

Telegram-команды не маршрутизируются как адресные ответы. Существующие command handlers
остаются владельцами `/stop`, `/new`, `/projects`, `/all`, `/N`, `/PsS`.

### Пользователь быстро отправил два address-reply в одну цель

Первый accepted-запрос занимает сессию. Второй должен получить busy-сообщение по тем же
правилам, а не становиться в очередь.

### Пользователь вышел из `/all` после отправки address-reply

Это не должно отменять запрос. Ответ агента придёт обычным механизмом:

- если пользователь остался в `/all`, через all-project monitor;
- если пользователь перешёл в нужный проект, через обычный watcher или pending-доставку;
- если пользователь перешёл в другой проект, сообщение останется непрочитанным для
  исходного проекта по существующим правилам проекта.

## Тестирование

### Unit-тесты route registry

Покрыть:

- запись и чтение route по `chat_id + bot_message_id`;
- разные чаты с одинаковым `message_id`;
- очистку реестра;
- потерю route после `clear_all`;
- хранение `project_path`, `session_id`, `backend`, `project_number`,
  `session_number`.

### Unit-тесты Telegram sender и delivery

Покрыть:

- отправитель возвращает `message_id`;
- длинный ответ регистрирует route для каждого chunk-а;
- route не регистрируется для служебных подтверждений и ошибок;
- `/all`-сообщение регистрируется с full link metadata;
- watcher-сообщение внутри проекта регистрируется с local session link metadata.

### Unit-тесты входящих текстовых reply

Покрыть:

- текстовый reply из `/all` отправляется в target и не выключает `/all`;
- текстовый reply внутри проекта отправляется в target и не меняет активную сессию;
- reply внутри проекта на target другого проекта использует full all-link;
- unknown route в `/all` даёт понятную ошибку;
- busy target даёт busy-ошибку со ссылкой;
- два быстрых reply в одну target-сессию не создают очередь.

### Unit-тесты unsupported attachments

Покрыть:

- фото reply с route не скачивает файл и показывает unsupported-сообщение;
- документ reply с route не скачивает файл и показывает unsupported-сообщение;
- альбом reply с route не запускает media group aggregation;
- unsupported-сообщение использует `/PsS` в `/all` и `/N` внутри проекта.

### Интеграционные или E2E-сценарии

Минимальный E2E для v1:

1. Включить `/all`.
2. Дождаться сообщения из другой проектной сессии или подготовить тестовый route.
3. Ответить текстом на сообщение бота.
4. Проверить, что бот прислал `Передал в /PsS`.
5. Проверить, что пользователь остался в `/all`.

Дополнительный E2E:

1. Перейти в проект.
2. Получить сообщение watcher-а из другой сессии этого проекта.
3. Ответить текстом.
4. Проверить `Передал в /N`.
5. Проверить, что активная сессия пользователя не поменялась автоматически.

## Acceptance criteria

- Текстовый reply на маршрутизируемое сообщение бота отправляется в исходную
  `project_path + backend + session_id`.
- Reply-routing не переключает проект и не меняет активную сессию пользователя.
- Подтверждение из `/all` имеет вид `Передал в /PsS`.
- Подтверждение внутри того же проекта имеет вид `Передал в /N`.
- Cross-project target из проектного режима использует full all-link `Передал в /PsS`.
- Busy target не принимает сообщение и показывает `Не передал ...: сессия занята.
  Подождите или /stop`.
- Reply с фото, документом, файлом или альбомом не скачивает вложение и показывает
  unsupported-сообщение.
- Route registry хранится только в памяти.
- После рестарта старые Telegram-сообщения не маршрутизируются и дают понятное
  сообщение о неизвестном route.
- Длинные ответы регистрируют route для каждого Telegram chunk-а.
- Служебные подтверждения и ошибки не становятся новыми route targets.
- Backend label, название проекта и расширенный номер сессии не выводятся в
  подтверждениях и ошибках.

## Самопроверка спецификации

- Нет требований к автоматическому переключению пользователя.
- Нет требований к персистентному route registry.
- Фото, документы, файлы и альбомы явно вынесены из v1.
- Форматы сообщений не содержат backend label, имени проекта или `#N`.
- Существующий `reply_anchor_registry` не смешан с новым `reply_route_registry`.
- Реализация может быть выполнена через отдельные focused-модули, не раздувая
  `telegram_input_handlers.py` и `telegram_response_delivery.py`.
