# Feature Pipeline — Спецификация

Пайплайн для доработки существующего функционала проекта. Принимает задачу любого масштаба — от короткого промпта до подробного BRD — и проводит через полный цикл: исследование, спецификация, реализация, тестирование, ревью, документация.

**Дата:** 03-04-2026
**Статус:** Черновик

---

## Общие принципы

### Масштабирование

Пайплайн классифицирует задачу по масштабу и адаптирует набор фаз. Масштаб определяется НЕ количеством модулей, а глубиной и сложностью изменений — даже один модуль может потребовать крупных изменений:

- **Мелкая** — изолированное изменение: правка внутренней логики, исправление бага, добавление простой функции. Не меняет интерфейсы между модулями. Пропускает E2E, ADR, полное интеграционное тестирование
- **Средняя** — заметное изменение: новая функциональность, изменение API модуля, затрагивает взаимодействие между компонентами. Требует обновления тестов. Все фазы кроме ADR (если нет архитектурных изменений)
- **Крупная** — структурное изменение: новые модули, изменение архитектуры, переработка существующих интерфейсов, затрагивает множество компонентов или фундаментально меняет один. Все фазы без исключений

Критерии оценки:
- Сколько функций/API затронуто
- Меняются ли интерфейсы между модулями
- Добавляются ли новые концепции или абстракции
- Нужны ли новые тесты или только обновление существующих
- Есть ли риск регрессии в соседних модулях

### Цикл исправления багов (на каждой фазе)

Каждая фаза, которая может обнаружить ошибки (тестирование, ревью), работает по циклу:

1. Выполнить фазу
2. Если ошибки найдены → исправить
3. Повторить фазу (проверить что исправления работают)
4. Максимум 3 попытки исправления
5. Если после 3 попыток всё ещё ошибки → откат кода, возврат на Фазу 2 (спецификация) с обратной связью: что пробовали и почему не сработало
6. Перепланировать подход в спецификации и начать реализацию заново

Это правило действует на КАЖДОМ этапе тестирования, включая финальную проверку.

### Финальная петля качества

После завершения всех фаз — финальный прогон:
1. Все тесты (юнит + интеграция + E2E)
2. Если баги → исправление (максимум 3 попытки)
3. Если после 3 попыток всё ещё баги → откат кода, возврат на Фазу 2 (спецификация) с обратной связью, и весь круг заново
4. Полное ревью (каждый проход — отдельный агент, как в Фазе 8)
5. Если замечания → исправление → повторное ревью + тестирование
6. Повторять пока всё не будет зелёным

### Инструменты для создания пайплайна

При реализации (сборке) этого пайплайна — использовать существующие скиллы проекта:

- Для создания скриптов (если потребуются) → скилл `script-creator-pipeline`
- Для создания самого скилла feature-pipeline и его агентов → скилл `skill-creator`
- Для создания вспомогательных скриптов внутри пайплайна → скилл `script-creator-pipeline`

### Принцип использования проектных скиллов

Решение об использовании проектных скиллов принимает **оркестратор** (SKILL.md). На каждом этапе оркестратор проверяет: есть ли в проекте специализированный скилл для текущей задачи.

- **Для нетестовых фаз (0–4, 8–9):** если проектный скилл есть — оркестратор использует его **вместо** внутреннего агента пайплайна
- **Для тестовых фаз (5–7):** внутренний агент пайплайна и проектные скиллы работают **совместно** — оба участвуют в тестировании

Внутренние агенты создаются только для задач, которые не покрываются существующими проектными скиллами. Это правило применяется ко ВСЕМ фазам без исключений.

### Документация пайплайна

Все артефакты хранятся по структуре из AGENTS.md:
- Спецификации → `dev/docs/specs/`
- Сессионные отчёты → `dev/docs/session-reports/ДД-ММ/`
- Временные файлы → `dev/temp/`

Дополнительно для feature pipeline:
- Документация фичи → `dev/docs/features/{дата}-{имя}/`
- ADR → `dev/docs/adr/`
- Changelog → `dev/docs/changelog.md`

### Папка запуска пайплайна

Для каждого запуска пайплайна создаётся своя папка — по тем же канонам, что в скилле pipeline-designer:
- **Логи пайплайна** → `dev/docs/logs/{дата}-{имя}/` — сюда кладутся ВСЕ временные документы, связанные с выполнением этого запуска (промежуточные результаты, логи агентов, отчёты фаз)
- **Результаты тестирования** → `dev/docs/testing/{дата}-{имя}/` — результаты прогонов юнит-тестов, интеграционных, E2E для этого запуска
- **Сессионные отчёты** → `dev/docs/session-reports/{дата}-{имя}/` — также в папку с датой и именем запуска

### Принципы ведения документации

- **Immutable (неизменяемые)** — каждый раз НОВЫЙ файл:
  - ADR (запись архитектурного решения) — фиксирует решение в конкретный момент, менять нельзя
  - Сессионные отчёты — фиксируют что сделано в сессии
  - Результаты тестирования — снимок результатов прогона

- **Living (живые)** — ОДИН файл, обновляется:
  - Changelog — растёт вверх, новые записи добавляются в начало
  - BRD — обновляется при изменении/добавлении CJM
  - CLAUDE.md — обновляется при изменении структуры, модулей, команд
  - docs-index.md — обновляется при появлении новых документов

- **Per-feature bundle** — один набор файлов на фичу:
  - `task.md` — создаётся один раз при приёме задачи
  - `impact-analysis.md` — создаётся один раз при анализе влияния
  - `spec.md` — создаётся при спецификации, может обновляться при реализации
  - `test-plan.md` — создаётся при планировании тестов
  - `pipeline-state.json` — обновляется постоянно (для возобновления при прерывании)
  - `report.md` — создаётся в конце, итог всей работы

## Фазы пайплайна

### Фаза 0 — Приём задачи и классификация

**Что делает:**
- Принимает задачу от пользователя (промпт, описание, BRD)
- Классифицирует масштаб: мелкая / средняя / крупная
- Если всё понятно из задачи — сразу переходит к работе, БЕЗ вопросов
- Вопросы — когда решение важно уточнить, потому что оно будет влиять на работу или архитектуру
- Всё что можно понять из кода и документации — выводит самостоятельно

**Собственный агент:** `feature-intake.md`

**Результат:** `dev/docs/features/{дата}-{имя}/task.md` — формализованное описание + масштаб

---

### Фаза 1 — Исследование и анализ влияния

**Что делает:**
- Сканирует код, спецификации, тесты затронутых модулей
- Строит карту влияния — какие модули, функции, тесты затронет изменение
- Определяет зону риска — что может сломаться «по соседству»
- Проверяет совместимость с архитектурой (слои, зависимости)

**Параллельные суб-агенты:** анализ запускается параллельно по отдельному агенту на каждую область:
- Агент анализа кода — сканирует затронутые модули, функции, зависимости
- Агент анализа спецификаций — проверяет совместимость с существующими спеками
- Агент анализа тестов — определяет какие тесты затронуты и что нужно обновить

**Собственный агент:** `feature-impact-analysis.md` (координатор, запускает суб-агентов параллельно)

**Результат:** `dev/docs/features/{дата}-{имя}/impact-analysis.md`

---

### Фаза 2 — Спецификация изменений

**Что делает:**
- Для существующих модулей — diff-спецификация: что было → что стало
- Для новых модулей — полная спецификация (API, алгоритмы, зависимости)

**Собственный агент:** `feature-spec.md` (универсальный, поддерживает diff-формат)

**Результат:** `dev/docs/features/{дата}-{имя}/spec.md`

---

### Фаза 3 — Планирование тестов

**Что делает:**
- Анализирует impact-analysis и спецификацию
- Определяет: какие существующие тесты нужно обновить, какие добавить, какие удалить
- Планирует покрытие: юнит, интеграция, E2E (если масштаб >= средний)
- Использует стандартный подход к планированию тестов

**Собственный агент:** `feature-test-plan.md`

**Результат:** `dev/docs/features/{дата}-{имя}/test-plan.md`

---

### Фаза 3.5 — Верификация тест-плана

**Что делает:**
- Придирчивая проверка тест-плана отдельным агентом-ревьюером
- Роль: дотошный техлид, который ненавидит лишнюю работу
- Проверяет: нет ли лишних тестов, покрыты ли все критичные пути, нет ли дублирования
- Убирает всё, что не нужно: избыточные кейсы, тесты ради тестов, проверки очевидного
- Если тест-план плохой → возврат на Фазу 3 с конкретными замечаниями

**Собственный агент:** `feature-test-plan-review.md`

**Результат:** верифицированный тест-план (обновлённый `test-plan.md`)

---

### Фаза 4 — Реализация

**Что делает:**
- Код по спецификации
- Для новых модулей — полная реализация
- Для доработок — редактирование существующего кода
- **Каждая отдельная правка** (изменение в отдельном модуле/файле) — выполняется отдельным агентом
- Оркестратор может заменить внутренний агент проектным скиллом реализации (например `implement-module`)

**Собственный агент:** `feature-implement.md` (координатор, запускает отдельного агента на каждую правку)

**Цикл исправления:** прогон юнит-тестов после написания кода → исправление → максимум 3 попытки → если не получается — откат кода, возврат на Фазу 2 (спецификация) с обратной связью что не сработало

**Результат:** рабочий код

---

### Фаза 5 — Юнит-тестирование

**Что делает:**
- Обновление существующих тестов по test-plan
- Добавление новых тестов для нового кода
- Прогон ВСЕХ юнит-тестов (не только новых) — ловим регрессии
- Максимум 3 попытки исправления
- Если после 3 попыток всё ещё красное → откат кода, возврат на Фазу 2 (спецификация) с обратной связью

**Собственный агент:** `feature-unit-test.md` (работает совместно с проектными скиллами тестирования)

**Результат:** все юнит-тесты зелёные

---

### Фаза 6 — Интеграционное тестирование

**Условие:** масштаб >= средний

**Что делает:**
- Обновление/добавление интеграционных тестов по test-plan
- Прогон всех интеграционных тестов
- Максимум 3 попытки исправления
- Если после 3 попыток всё ещё красное → откат кода, возврат на Фазу 2 (спецификация) с обратной связью
- Проектные скиллы тестирования (например `test-integration`) работают совместно с внутренним агентом

**Собственный агент:** `feature-integration-test.md` (работает совместно с проектными скиллами)

**Результат:** все интеграционные тесты зелёные

---

### Фаза 7 — E2E тестирование

**Условие:** масштаб >= средний

**Что делает:**
- Прогон всех E2E тестов
- При падениях — автоматический цикл починки (максимум 3 попытки)
- Если после 3 попыток всё ещё красное → откат кода, возврат на Фазу 2 (спецификация) с обратной связью
- Проектные скиллы тестирования (например `test-e2e`, `autofix-e2e`) работают совместно с внутренним агентом

**Собственный агент:** `feature-e2e-test.md` (работает совместно с проектными скиллами)

**Результат:** все E2E тесты зелёные

---

### Фаза 8 — Ревью

**Что делает:**
- Полное 5-проходное ревью, **каждый проход — отдельный специализированный агент:**
  1. Агент качества кода → `feature-review-quality.md`
  2. Агент безопасности → `feature-review-security.md`
  3. Агент архитектуры → `feature-review-architecture.md`
  4. Агент соответствия спецификации → `feature-review-spec-compliance.md`
  5. Агент проверки на регрессию → `feature-review-regression.md`
- Автоисправление найденных проблем
- Прогон тестов после каждого исправления
- Повторное ревью после исправлений
- Если новые замечания → снова исправление → снова ревью + тестирование
- При провале тестов более 3 раз после цикла исправлений → возврат на Фазу 2 (спецификация) с обратной связью
- Оркестратор может заменить внутренних агентов проектным скиллом ревью (например `review-code`)

**Собственный агент:** `feature-review.md` (координатор, запускает 5 специализированных агентов)

**Результат:** все замечания исправлены, все тесты зелёные

---

### Фаза 9 — Документация

**Что делает — каждый документ обновляется отдельным агентом:**
- **Changelog entry** → `dev/docs/changelog.md` (Added / Changed / Fixed / Removed) — агент `feature-docs-changelog.md`
- **ADR** (если архитектурное решение) → `dev/docs/adr/{дата}-{имя}.md` — агент `feature-docs-adr.md`
- **Обновление CLAUDE.md** — если изменилась структура, модули, команды — агент `feature-docs-claude-md.md`
- **Обновление BRD** — если добавлен/изменён CJM — агент `feature-docs-brd.md`
- **Обновление docs-index.md** — если появились новые документы — агент `feature-docs-index.md`
- Оркестратор может заменить внутренних агентов проектными скиллами документации (например `update-docs`, `update-project-docs`)

**Собственный агент:** `feature-docs.md` (координатор, запускает отдельного агента на каждый документ)

**Результат:** вся документация актуальна

---

### Фаза 10 — Финальная проверка и коммит

**Что делает:**
1. Полный прогон всех тестов (юнит + интеграция + E2E)
2. Если баги → исправление (максимум 3 попытки)
3. Если после 3 попыток всё ещё баги → откат кода, возврат на Фазу 2 (спецификация) с обратной связью, и весь круг заново
4. Полное ревью (каждый проход — отдельный агент, как в Фазе 8)
5. Если замечания → исправление → повторное ревью + тестирование
6. Повторять петлю пока ВСЁ не зелёное
7. Проверка что документация обновлена
8. Git commit
9. Предложение запушить
10. **Сессионный отчёт** — в самом конце, через глобальный скилл `/session-report` (не внутренний агент)

**Результат:** готовый, протестированный, отревьюенный код с документацией + сессионный отчёт
