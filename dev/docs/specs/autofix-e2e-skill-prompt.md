# Промпт для создания скилла `autofix-e2e`

> Подробное ТЗ для skill-creator. Передать целиком при создании скилла.

---

Создай новый скилл `autofix-e2e` — полностью автоматический цикл починки E2E тестов с поиском первопричин, применением фиксов, откатом при неудаче и регрессионной проверкой.

## Общая архитектура

Скилл координирует 4 существующих инструмента через агентов:

- **`test-e2e`** (скилл) — прогон E2E тестов с живым ботом через Telethon
- **`root-cause-analysis`** (скилл) — глубокий поиск первопричины проблемы
- **`apply-root-cause-fixes`** (скилл) — применение исправлений из root-cause отчёта
- **git commit** — фиксация успешных исправлений

## Фазы работы

### Фаза 0 — Базовая линия (baseline)

- Прогнать ВСЕ тесты (юнит + интеграционные + E2E) командой `python -m pytest tests/ -v` и записать текущее состояние
- Это нужно чтобы отличить "тест сломался из-за нашего фикса" от "тест и так был сломан"
- Создать git-тег `autofix/baseline` — точка, к которой можно откатить всё
- Записать список изначально падающих тестов (baseline failures) — эти тесты НЕ считаются регрессией при проверке

### Фаза 1 — E2E прогон

- Запустить все E2E тесты через агента (скилл `test-e2e`)
- Собрать детальные результаты: какой тест упал, сообщение об ошибке, стектрейс, логи бота
- Если все E2E тесты зелёные — скилл завершается с сообщением "Все E2E тесты проходят, починка не нужна"

### Фаза 2 — Группировка ошибок

- Перед тем как чинить поштучно — проанализировать ВСЕ падения разом через агента
- Часто несколько тестов падают по одной причине (например, бот не стартует, или сессия не создаётся)
- Агент группирует падения по вероятной корневой причине
- Приоритизирует: сначала фундаментальные проблемы (старт бота, подключение), потом логику отдельных сценариев
- Результат: упорядоченный список групп для починки

### Фаза 3 — Цикл починки (для каждой группы ошибок)

```
Константы:
  MAX_ATTEMPTS_PER_GROUP = 3
  MAX_TOTAL_TIME = 60 минут

failed_approaches = []  # память неудачных попыток — ГЛОБАЛЬНАЯ на весь цикл

Для каждой группы падений:

  # --- Проверка каскадного эффекта ---
  # Предыдущий фикс мог починить зависимую группу.
  # Перед починкой группы — прогнать E2E тесты этой группы.
  # Если все прошли → пропустить группу, записать в отчёт "починилась каскадно".

  attempt = 0

  Пока тест не починен И attempt < MAX_ATTEMPTS_PER_GROUP И не превышен MAX_TOTAL_TIME:

    1. Запомнить текущий git HEAD (точка отката)

    2. Root-cause analysis через агента (скилл `root-cause-analysis`)
       → передать агенту: описание ошибки, логи, стектрейс
       → передать failed_approaches целиком: "уже пробовали X, Y, Z — не помогло, вот почему"
       → агент создаёт отчёт в dev/docs/logs/root-cause-reports/

    3. Apply fixes через агента (скилл `apply-root-cause-fixes`)
       → читает свежий root-cause отчёт и применяет исправления

    4. Перезапустить бота (если были изменения в .py файлах)
       → остановить текущий процесс бота
       → запустить заново через `python -m claude_manager`

    5. Прогнать упавшие E2E тесты этой группы

    ЕСЛИ тесты этой группы прошли:
      6. Прогнать ВСЕ тесты (юнит + интеграция + E2E)

      ЕСЛИ всё зелёное (или не хуже baseline):
        7. git commit с описанием: что починили, какая была причина
        8. Обновить baseline → перейти к следующей группе

      ИНАЧЕ (регрессия — сломали то, что раньше работало):
        9. Записать в failed_approaches:
           "фикс X починил тест Y, но сломал тест Z. Детали регрессии: ..."
        10. Откат: git checkout . && git clean -fd (сброс до последнего коммита)
        11. attempt++

    ИНАЧЕ (тест всё ещё падает):
      9. Записать в failed_approaches:
         "подход X не помог. Ошибка до: ... Ошибка после: ... Что делали: ..."
      10. Откат: git checkout . && git clean -fd
      11. attempt++

  ЕСЛИ attempt >= MAX_ATTEMPTS_PER_GROUP:
    Пометить группу как "не удалось починить за {MAX_ATTEMPTS_PER_GROUP} попыток"
    Записать в отчёт все попытки и их результаты
    Продолжить к следующей группе
```

### Фаза 4 — Финальная верификация

- Полный прогон ВСЕХ тестов с нуля (юнит + интеграция + E2E)
- Убедиться, что совокупность всех фиксов не создала проблем
- Сравнить с baseline: стало лучше или нет

### Фаза 5 — Отчёт

- Создать подробный отчёт в `dev/docs/session-reports/` с датой в формате ДД-ММ_ЧЧ-ММ
- Структура отчёта:
  - **Baseline:** сколько тестов было, сколько падало
  - **Для каждой группы:** что было, сколько попыток, что пробовали, что помогло (или не помогло)
  - **Финальный результат:** сколько починено, сколько осталось, регрессии
  - **Список всех коммитов**, сделанных в процессе
  - **Общее время работы**

## Ключевые механизмы

### 1. Память неудачных попыток (failed_approaches)

Самый важный механизм всего скилла. Без него root-cause-analysis будет предлагать одно и то же решение повторно, и цикл починки станет бесполезным.

Как работает:
- Каждая неудачная попытка записывается в список: что сделали, какой результат, почему не помогло
- При следующей попытке ВЕСЬ этот контекст передаётся в root-cause-analysis
- Память **глобальная** — накапливается между группами (фикс группы A может дать инсайт для группы B)
- Формат записи: `"Попытка N для группы X: применили фикс <описание>. Результат: <тест по-прежнему падает с ошибкой ... / тест прошёл, но сломался тест Z с ошибкой ...>. Вывод: <почему не помогло>."`

### 2. Изоляция через git

- Каждая попытка фикса начинается с чистого состояния (последний коммит)
- Откат — через `git checkout . && git clean -fd` (сброс до последнего коммита)
- Успешный фикс — сразу коммит, чтобы закрепить результат
- НЕ использовать ветки — работаем на текущей ветке, откатываемся к коммитам

### 3. Каскадный эффект

- После починки группы A группа B может починиться автоматически (зависела от A)
- Перед починкой каждой следующей группы — перепроверить, не починилась ли она уже
- Если починилась — пропустить, записать в отчёт "починилась каскадно после фикса группы A"

### 4. Перезапуск бота

- E2E тесты работают с живым ботом через Telethon
- Если код изменился (.py файлы) — бот ОБЯЗАТЕЛЬНО перезапускается перед тестированием
- Автоматизировано внутри цикла: остановить процесс → запустить заново → дождаться готовности

### 5. Сохранение состояния (state persistence)

- Состояние цикла сохранять в файл `dev/temp/autofix-state.json`
- Структура: какие группы починены, какие в процессе, номер текущей попытки, failed_approaches
- Если скилл прервали — при повторном запуске продолжить с места остановки
- После успешного завершения — удалить файл состояния

### 6. Последовательная починка (НЕ параллельная)

- Группы чинятся СТРОГО последовательно — изменения могут конфликтовать друг с другом
- Внутри одной группы попытки тоже последовательны
- Параллельность допустима только внутри фаз, где нет конфликтов (например, анализ ошибок в фазе 2)

### 7. Сравнение с baseline (не хуже)

- При проверке регрессии сравнивать с baseline, а не с "все тесты зелёные"
- Если тест падал ДО наших изменений и продолжает падать — это НЕ регрессия
- Регрессия — это когда тест РАБОТАЛ до наших изменений и СЛОМАЛСЯ после

---

## Режим dry-run (опциональный)

- Если пользователь запросит dry-run: проанализировать все падения, написать план починки с root-cause для каждой группы, но НЕ применять фиксы
- По умолчанию dry-run выключен — скилл работает в полном автоматическом режиме

---

## Участие человека

- НИКАКОГО участия человека. Скилл полностью автономный.
- Не спрашивать "продолжить?", не запрашивать подтверждение.
- Все решения принимаются автоматически по алгоритму.
- Единственный способ остановить — прервать скилл извне.

## Когда использовать (триггеры)

- "autofix e2e", "autofix-e2e", "автофикс e2e", "почини e2e тесты автоматически"
- "автоматическая починка e2e", "цикл починки тестов", "fix all e2e"
- "почини все e2e", "автофикс тестов", "fix e2e loop"
- Когда пользователь хочет автоматически починить все падающие E2E тесты с поиском первопричин и откатом при неудаче

## Когда НЕ использовать

- Для прогона E2E без починки — это `test-e2e`
- Для починки одного конкретного бага — это `root-cause-analysis` + `apply-root-cause-fixes`
- Для юнит-тестов или интеграционных тестов без E2E — это другие скиллы
