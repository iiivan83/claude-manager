Слой: 2 (зависит от слоёв 0-1)
Файл: `src/claude_manager/process_manager.py`

**Связанные спеки:**
- `coding_agent_backend_spec.md` — абстрактный интерфейс CLI-бэкенда (родитель Adapter pattern, источник `BackendName`, `UnifiedEvent`, `SendResult`-полей, `StopStrategy`, `TerminalStatus`).

- `codex_backend_spec.md` — реализация для Codex CLI (Adapter, используется через `get_backend(BackendName.CODEX)`).
- `current_backend_registry_spec.md` — персистентное хранилище имени активного бэкенда (`get_current()`). Используется потребителями (`bot.py`) перед вызовом `process_manager`, а не как fallback внутри backend-aware turn-а.
- `dev/docs/specs/realised/process_manager_spec.md` — старая Claude-only версия этой спеки. Сохранена в `realised/` для трейсабельности; новая спека — её надстройка с переходом на backend-абстракцию.

## Назначение

Управляет жизненным циклом subprocess-процессов CLI-бэкенда (Claude Code или Codex): на каждый turn поднимает новый subprocess через адаптер `CodingAgentBackend`, передаёт пользовательское сообщение через stdin (или включённое в args для Codex), читает поток событий, применяет ретраи при ошибке, останавливает процесс по `/stop` через backend-specific стратегию сигналов, и поддерживает координацию с `session_watcher` через флаг занятости. Модуль **backend-aware**: на каждый запрос бэкенд выбирается обработчиком `bot.py` и захватывается на старте запроса (по аналогии с захватом `WORKING_DIR`); все последующие операции (ретраи, остановка, ремаппинг session_id, логирование) используют захваченное значение, а не глобальный `current_backend_registry.get_current()`.

Модуль не знает деталей stream-json или JSON-протокола Codex — все парсинговые операции делегированы методам `CodingAgentBackend`. Для backend-aware пути он не запускает subprocess напрямую, а использует `claude_runner.start_subprocess_for_backend(args, cwd, stdin_bytes)` как тонкую обёртку над `asyncio.create_subprocess_exec`: `process_manager` формирует команду через `backend.compose_subprocess_command_args`, формирует stdin через `backend.encode_user_message_for_cli_stdin` и передаёт готовые значения в runner. Legacy `claude_runner.start_process(session_id, cwd)` остаётся только для старого Claude-only compatibility path.

## Расхождение с Claude-only версией спеки

В прежней Claude-only версии (`dev/docs/specs/realised/process_manager_spec.md`) словари состояния были по ключу `session_id: str`, `start_process` принимал только `session_id`, `terminate()` вызывал жёсткий `process.terminate()` (SIGTERM), а `SendResult` не имел поля `backend`. Эта спека сохраняет внешнюю семантику (CJM-02, CJM-03, CJM-04, CJM-08 не меняются с точки зрения пользователя), но переписывает внутренний контракт под backend-абстракцию. Каждое расхождение — техническое; имена основных публичных функций сохранены, добавлены новые обязательные параметры `backend`.

- **Ключ словарей состояния — composite `(session_id, backend)`.** В Claude-only версии ключом был только `session_id`. В backend-aware версии словари `_processes`, `_busy_flags`, `_stop_events` индексируются кортежем `(session_id, BackendName)`. Это нужно потому, что Claude и Codex теоретически могут вернуть UUID одного формата (Claude — v4, Codex — v7, но коллизии не исключены формально), и потому что одна сессия Codex и одна сессия Claude в один и тот же момент времени — это два разных физических ресурса (subprocess + файл сессии в разных директориях), которые нельзя адресовать одним ключом без неоднозначности. **Альтернатива** — `session_id` + параллельный маппинг `session_id → backend` — отвергнута: она вводит два инварианта поддержки (надо синхронно обновлять оба словаря при ремаппинге temp→real), а на каждом lookup-вызове требуется два чтения вместо одного. Composite key проще и атомарнее. Подробнее обоснование — в разделе «Контракт ключей и захват backend → Composite key».
- **`send_message` требует явный `backend` для backend-aware turn-а.** В Claude-only версии `send_message(session_id, text, callbacks)` неявно работал с Claude. В backend-aware версии потребитель обязан передать `backend: BackendName` явно: `send_message(session_id, text, *, backend=active_session.backend, image_paths=(), cwd=..., ...)`. Если backend-aware путь получает `backend=None`, это ошибка контракта и вызов завершается `ProcessManagerError`. Отсутствие аргумента `backend` допустимо только для старого Claude-only compatibility path, где вызов сохраняет прежнюю семантику ради обратной совместимости. Захват backend на старте запроса — по аналогии с захватом `cwd` (см. CLAUDE.md → «Привязка рабочей директории к запросу»): после захвата backend для всего жизненного цикла turn-а (включая ретраи и `/stop`) используется только захваченное значение, а не текущий глобальный.
- **`SendResult` содержит поле `backend: BackendName` и `error_text: str | None`.** Поле `backend` нужно потребителю (`bot.py` и `claude_interaction`) для записи в `daily_session_registry.register_session(date, session_id, backend)` через `session_id_callback`, для UI-отображения (`#3 ✅` с подписью `🤖 Claude` или `⚡ Codex`), и для диагностики в логах. Поле `error_text` — текст ошибки от бэкенда (Claude: `event["result"]` при `is_error: true`; Codex: `event["error"]["message"]` события `turn.failed`); используется в `bot.py` для показа причины пользователю при исчерпании ретраев. Источник полей — родительская спека `coding_agent_backend_spec.md` (см. шаг 10 раздела «Жизненный цикл turn-а»).
- **`stop_process(session_id, backend)` принимает обязательный `backend`.** Без этого нельзя выбрать правильную стратегию (Claude — SIGTERM→SIGKILL, Codex — SIGINT→SIGTERM→SIGKILL) и нельзя адресовать процесс в composite key. Старая сигнатура `stop_process(session_id)` отброшена.
- **Остановка процесса — `_apply_stop_strategy(process, strategy)` вместо `process.terminate()`.** В Claude-only коде `claude_runner.py:155-174` использовался `process.terminate()` (SIGTERM) с пятисекундным таймаутом перед SIGKILL. Это работало для Claude, но **некорректно для Codex** (Codex CLI обрабатывает только SIGINT, см. `coding_agent_backend_spec.md` → «Codex CLI — отсутствие subcommand для остановки turn, корректный сигнал — SIGINT»). Новый алгоритм: получить `strategy = backend.get_stop_strategy()`, последовательно применить каждый `StopSignalStep(signal_to_send, wait_seconds_before_next)` через `process.send_signal()` + `asyncio.wait_for(process.wait(), timeout=...)`. См. раздел «Алгоритм работы → _apply_stop_strategy».
- **`update_session_id(old, new, backend)` — добавлен параметр `backend`.** Ремаппинг ключа делается в одном backend-пространстве: `(temp_id, backend) → (real_id, backend)`, backend не меняется. Это согласовано с контрактом владения сессией (`coding_agent_backend_spec.md`): сессия принадлежит тому бэкенду, который её создал — temp→real не меняет backend.
- **`session_id_callback` — теперь `Callable[[str, str, BackendName], Awaitable[None]]`.** Старая сигнатура `(old, new) -> Awaitable[None]` расширена третьим аргументом `backend`. Это нужно `bot.py`, который через callback атомарно обновляет `session_manager.set_active_session(chat_id, new_id, backend)` и `daily_session_registry.register_session(date, new_id, backend)` — оба этих модуля переходят на backend-aware API (см. `coding_agent_backend_spec.md` → «Контракт владения сессией для потребителей»). Без `backend` потребитель не знал бы, под каким бэкендом регистрировать сессию.
- **Прямая адресация callback'ов через `chat_id` — закреплена явно.** В Claude-only версии progress/retry/session_id callback'и принимали `session_id` как параметр, и потребитель в `bot.py` искал `chat_id` по обратной привязке. После того, как session_id меняется в любой момент (temp→real), поиск по старому ID возвращает пустой результат — и сообщение пользователю теряется. Новый контракт: callback'и в `process_manager` определяются обработчиком `_send_to_claude_and_respond` в `bot.py` как **замыкания, захватывающие `chat_id` напрямую** на старте запроса. Внутри `process_manager` callback'и просто вызываются с информационными аргументами (`session_id`, `attempt`, `progress_text`, `error_reason`); адресат — уже зашит в замыкании. Это соответствует CLAUDE.md → «Принцип прямой адресации callback'ов» и закреплено сигнатурами callback-типов (см. раздел «Типы callback'ов»).
- **Принцип stale session_id — закреплён явно в алгоритме `_process_events` и `_retry_loop`.** После того как `session_id_callback` вызван (temp→real), все локальные использования оригинального `session_id`-параметра считаются stale. Внутренние словари `_processes/_busy_flags/_stop_events` ремаппятся через `update_session_id()` **в `_process_events` сразу при обнаружении нового ID**, до вызова `session_id_callback`, чтобы `/stop` из Telegram мог найти `stop_event` по новому ключу. См. CLAUDE.md → «Принцип stale session_id» и раздел «Алгоритм работы → _process_events».

## Обслуживаемые сценарии

- **CJM-02: Отправка текстового сообщения** — `bot.py` вызывает `send_message(session_id, text, backend=<захваченный>, cwd=<захваченный WORKING_DIR>, callbacks)`. Модуль на старте запроса:
  1. Проверяет, что backend передан явно; если backend-aware вызов пришёл с `backend=None`, возвращает ошибку контракта.
  2. Получает `backend_obj = get_backend(backend)`.
  3. Запускает новый subprocess через `start_subprocess_for_backend(backend_obj, session_id, cwd, text, image_paths)` (тонкая обёртка над `asyncio.create_subprocess_exec` с командой из `backend_obj.compose_subprocess_command_args`).
  4. Записывает stdin (для Claude) или сразу закрывает stdin (для Codex — `encode_user_message_for_cli_stdin` вернёт `b""`).
  5. Читает поток событий через `process.stdout.readline()` + `backend_obj.parse_stdout_line_into_event`.
  6. Извлекает `progress_text`, обновляет `last_assistant_text`, ловит `is_turn_complete_event` и `terminal_status` через методы бэкенда.
  7. При `terminal_status == FAILED` или EOF без финального события — переходит в retry-цикл.
  8. Возвращает `SendResult` с полями `text, session_id, backend, is_error, error_text, retries_used`.
- **CJM-03: Отправка фотографии или файла** — для `process_manager` алгоритм идентичен CJM-02. Путь к файлу включён в `text` модулем `claude_interaction`. Параметр `image_paths` пробрасывается в `backend.compose_subprocess_command_args` и `backend.encode_user_message_for_cli_stdin` для совместимости с интерфейсом, но в текущей версии оба бэкенда его игнорируют (см. родительскую спеку).
- **CJM-04: Создание новой сессии (/new)** — `bot.py` вызывает `send_message(session_id=_generate_temp_session_id(), text=..., backend=current_backend_registry.get_current(), ...)`. Backend для новой сессии берётся из глобального реестра. Модуль формирует команду без `--resume` (Claude) или без `resume <id>` (Codex), запускает subprocess. При получении первого события с реальным session_id — вызывает `session_id_callback(temp_id, real_id, backend)` и ремаппит ключи словарей. **`/new` НЕ создаёт subprocess заранее** — в backend-aware архитектуре новый процесс поднимается ровно на первое сообщение пользователя в этой сессии (см. CLAUDE.md → «новый процесс на каждый turn — оба CLI закрывают stdin после первого сообщения»). Это расхождение с прежней Claude-only спекой, где упоминалось «`/new` сразу запускает процесс». На уровне UX разницы нет — `bot.py` рассылает подтверждение «Создана сессия #N» сразу после генерации temp-id, а subprocess поднимается уже при первом текстовом сообщении в новую сессию.
- **CJM-08: Остановка Claude (/stop)** — `bot.py` определяет `(session_id, backend)` активной сессии чата через `session_manager.get_active_session(chat_id)` и вызывает `stop_process(session_id, backend)`. Модуль:
  1. Захватывает `_busy_lock`, читает `_processes[(session_id, backend)]` и `_busy_flags[(session_id, backend)]` (was_retrying — был ли активен запрос/ретрай).
  2. Устанавливает `_stop_events[(session_id, backend)].set()` — это прервёт ожидание ретрая в `_wait_with_stop_check` и проверку перед следующей попыткой в `_retry_loop`.
  3. Удаляет ключ `(session_id, backend)` из `_processes` и `_busy_flags` (но НЕ из `_stop_events` — флаг нужен retry-циклу до конца finally в `send_message`).
  4. Вне Lock: применяет `_apply_stop_strategy(process, backend.get_stop_strategy())` к процессу (для Claude — SIGTERM→SIGKILL, для Codex — SIGINT→SIGTERM→SIGKILL). Это даёт Codex шанс штатно записать `TurnInterrupt` в файл сессии — критично, иначе watcher не увидит маркера завершения.
  5. Возвращает `StopResult(was_running, was_retrying, backend)`.

## Обслуживаемые сценарии (косвенно через interface)

- **CJM-NEW: Переключение бэкенда (/agent)** — `bot.py` вызывает `current_backend_registry.set_current(name)`. На уже работающих процессах это **не отражается** — захваченный backend держит каждый запрос до его завершения. Только новые сессии (с `/new` после переключения) пойдут через новый бэкенд. Контракт владения сессией (см. родительскую спеку) гарантирует, что `/N` для существующих сессий продолжает работать через тот бэкенд, который их создал.

## Контракт ключей и захват backend

### Composite key `(session_id, BackendName)`

Все три словаря состояния используют один и тот же тип ключа:

```python
ProcessKey = tuple[str, BackendName]

_processes: dict[ProcessKey, ManagedProcess]     # запущенные subprocess'ы
_busy_flags: dict[ProcessKey, bool]              # флаги занятости
_stop_events: dict[ProcessKey, asyncio.Event]    # события отмены для /stop
```

Где `ManagedProcess = BackendSubprocess | ClaudeProcess`. Новый backend-aware flow использует `BackendSubprocess`; `ClaudeProcess` присутствует только из-за legacy Claude-only compatibility path и старых тестов.

**Обоснование composite key (vs. параллельный маппинг `session_id → backend`):**

- **Один lookup вместо двух.** При `/stop`: `_processes[(session_id, backend)]` против `_processes[session_id]` + `backend = _session_to_backend[session_id]`. Composite key избегает скрытой ошибки «забыл синхронизировать второй маппинг».
- **Атомарный ремаппинг при temp→real.** При обнаружении настоящего UUID `update_session_id(temp_id, real_id, backend)` за один захват `_busy_lock` обновляет ключ во всех трёх словарях: `(temp_id, backend) → (real_id, backend)`, backend остаётся прежним (контракт владения сессией). Параллельный маппинг потребовал бы дополнительного `_session_to_backend.pop(temp_id); _session_to_backend[real_id] = backend` — четыре операции вместо трёх.
- **Семантически точный.** Один subprocess — один файл сессии в одной CLI-директории. `(session_id, backend)` адресует именно эту тройку «процесс/файл/CLI». `session_id` без backend — неполный идентификатор: на одну UUID-строку могло бы быть два разных процесса, если бы оба CLI согласились использовать одинаковую UUID-схему (формально не запрещено).
- **Лёгкий рефакторинг.** Все три словаря индексируются одним и тем же `ProcessKey` — это позволяет ввести вспомогательную функцию `_remap_key(storage, old_key, new_key)` и применять её одинаково ко всем трём в `update_session_id` (см. реализацию в `_process_events`).

**Цена composite key:** в `send_message` приходится передавать `backend` явно (даже если `session_id` уже зарегистрирован). Это сознательный контракт: потребитель (`bot.py`) берёт backend из `session_manager.get_active_session(chat_id).backend`, а `process_manager` не пытается угадывать владельца сессии по глобальному состоянию.

### Захват backend на старте запроса

Аналогично захвату `cwd` (см. CLAUDE.md → «Привязка рабочей директории к запросу»):

1. **На входе в `send_message`:** определить `effective_backend: BackendName`. Для backend-aware пути backend должен быть передан явно. Если `backend is None`, это ошибка контракта: `process_manager` не читает `current_backend_registry` и не ищет владельца по существующим процессам. Отсутствие аргумента `backend` допускается только для legacy Claude compatibility path.
2. **Сохранить локально:** `effective_backend` — переменная функции, не глобальное состояние. Любой ретрай, любой `_apply_stop_strategy`, любая запись в `SendResult` использует именно её.
3. **Внутри `_retry_loop`:** при перезапуске процесса (`_restart_process`) backend берётся из переменной `effective_backend`, унаследованной через параметр. Это значит, что если пользователь между ретраями переключит глобальный бэкенд через `/agent` — наш текущий запрос продолжит работать с прежним бэкендом (как и должно быть для контракта владения сессией: сессия принадлежит её первоначальному бэкенду).
4. **В `_process_events`:** при ремаппинге `(temp_id, effective_backend) → (real_id, effective_backend)` backend остаётся прежним.

Этот принцип не оставляет дыр: глобальный `current_backend_registry.get_current()` читается верхним слоем (`bot.py`) только при создании новой сессии, а `process_manager` получает уже выбранный backend. Дальнейшие изменения в реестре не влияют на текущий жизненный цикл turn-а.

### Контракт владения сессией — backend в SendResult

`SendResult` всегда содержит `backend: BackendName` — потребитель использует его для:
- регистрации в `daily_session_registry.register_session(date, session_id, backend=result.backend)`;
- сохранения в `session_manager.set_active_session(chat_id, session_id, backend=result.backend)`;
- UI-отображения через `get_backend(result.backend).display_name` (`🤖 Claude` / `⚡ Codex`);
- логирования.

Этот контракт обязателен для **всех** возвратов из `send_message`, включая ошибочные (`is_error=True`). Backend в SendResult всегда равен `effective_backend`, захваченному на старте запроса, и не зависит от того, успешно завершился ли запрос.

## Публичный API

### Типы callback'ов

```python
type ProgressCallback = Callable[[str, str], Awaitable[None]]
# (session_id, progress_text) -> None
# Вызывается при поступлении промежуточного обновления от Claude/Codex.
# Не чаще одного раза в PROGRESS_THROTTLE_SECONDS.

type RetryCallback = Callable[[str, int, int, str], Awaitable[None]]
# (session_id, attempt, max_attempts, error_reason) -> None
# Вызывается перед каждой повторной попыткой ретрая.

type SessionIdCallback = Callable[[str, str, BackendName], Awaitable[None]]
# (old_session_id, new_session_id, backend) -> None
# Вызывается ровно один раз — при первом обнаружении настоящего session_id
# в потоке событий (когда был временный _new_XXXX). Потребитель обязан
# атомарно обновить session_manager и daily_session_registry с захваченным backend.
```

**Прямая адресация:** все три callback'а ничего не знают про `chat_id`. Адресат сообщения зашит в замыкании, которое формирует `bot.py._send_to_claude_and_respond` для каждого запроса. См. CLAUDE.md → «Принцип прямой адресации callback'ов».

### Dataclass `SendResult`

Результат отправки сообщения. Frozen.

```python
@dataclass(frozen=True)
class SendResult:
    text: str                  # финальный текст ответа (или текст ошибки при is_error)
    session_id: str            # актуальный session_id (после возможного temp→real ремаппинга)
    backend: BackendName       # бэкенд, под которым обрабатывался запрос (захвачен на старте)
    is_error: bool             # True если все ретраи исчерпаны или turn завершился с ошибкой
    error_text: str | None     # текст ошибки от бэкенда (None при is_error=False)
    retries_used: int          # количество использованных повторных попыток (0..MAX_RETRIES)
```

Поле `error_text` может отличаться от `text` — для Claude они совпадают (текст ошибки лежит в `event["result"]`), для Codex — `text` остаётся последним известным `agent_message` (или пустой), а `error_text` — `event["error"]["message"]` события `turn.failed`. Потребитель (`bot.py`) показывает `error_text` пользователю как причину при исчерпании ретраев.

### Dataclass `StopResult`

Результат остановки процесса. Frozen.

```python
@dataclass(frozen=True)
class StopResult:
    was_running: bool          # был ли запущен процесс на момент /stop
    was_retrying: bool         # был ли активен retry-цикл (busy_flag == True)
    backend: BackendName       # backend, под которым работал процесс
```

Поле `backend` нужно `bot.py` для логирования и для UI (например, "🤖 Claude остановлен" вместо абстрактного "Claude остановлен").

### Исключения

```python
class ProcessManagerError(Exception):
    """Общая ошибка модуля — например, не удалось запустить subprocess."""

class ProcessNotFoundError(Exception):
    """Для указанного (session_id, backend) нет запущенного процесса."""

class ProcessStoppedError(Exception):
    """Запрос прерван командой /stop — штатное поведение, не ошибка."""
```

### `async send_message(session_id: str, text: str, *, backend: BackendName | None = None, image_paths: list[str] | None = None, cwd: str | None = None, progress_callback: ProgressCallback | None = None, retry_callback: RetryCallback | None = None, session_id_callback: SessionIdCallback | None = None) -> SendResult`

Отправляет сообщение в CLI и ожидает финальный ответ. Поднимает новый subprocess на каждый вызов (оба CLI закрывают stdin после первого сообщения, повторное использование не работает).

**Аргументы:**
- `session_id` (`str`) — идентификатор сессии. Может быть временным (`_new_XXXX`) для новой сессии или реальным UUID для resume. **НЕ может быть `None`** — если сессии нет, потребитель должен сам сгенерировать `_generate_temp_session_id()` и передать его (или вызвать `register_new_session(backend)`, см. ниже).
- `text` (`str`) — текст сообщения пользователя.
- `backend` (`BackendName`) — бэкенд для запроса. Для backend-aware вызова обязателен и передаётся потребителем явно. `None` считается ошибкой контракта; отсутствие аргумента сохраняется только для legacy Claude compatibility path.
- `image_paths` (`list[str] | None`) — пути к изображениям; пробрасываются в `backend.compose_subprocess_command_args` и `backend.encode_user_message_for_cli_stdin`. По умолчанию пустой список (для текстовых сообщений). В текущей версии оба бэкенда аргумент игнорируют — путь к файлу уже включён в `text` модулем `claude_interaction`.
- `cwd` (`str | None`) — рабочая директория для subprocess. Если `None` — берётся `config.WORKING_DIR` на старте запроса. Захваченное значение передаётся во все ретраи (принцип «Привязка рабочей директории к запросу» из CLAUDE.md).
- `progress_callback` — см. «Типы callback'ов».
- `retry_callback` — см. «Типы callback'ов».
- `session_id_callback` — см. «Типы callback'ов». Вызывается ровно один раз.

**Возвращает:** `SendResult`. Поле `session_id` — актуальный (после возможного temp→real ремаппинга). Поле `backend` — `effective_backend`, захваченный на старте.

**Исключения:**
- `ProcessStoppedError` — если запрос или ретрай были прерваны через `stop_process(session_id, backend)`.
- `ProcessManagerError` — при невозможности запустить subprocess (например, бинарник CLI не найден; оборачивает `BackendBinaryNotFoundError` от backend).

### `async stop_process(session_id: str, backend: BackendName) -> StopResult`

Принудительно останавливает процесс CLI в указанной сессии и прерывает retry-цикл (если активен).

**Аргументы:**
- `session_id` (`str`) — идентификатор сессии (temp или real). Используется как часть composite key.
- `backend` (`BackendName`) — backend, под которым работает сессия (определяется потребителем через `session_manager.get_active_session(chat_id).backend`).

**Возвращает:** `StopResult`. `was_running=False, was_retrying=False, backend=backend` — если ключ `(session_id, backend)` не найден в `_processes` (нет процесса или сессия чужого бэкенда).

**Исключения:** не выбрасывает.

### `async stop_all_processes() -> int`

Останавливает все запущенные процессы во всех бэкендах (для shutdown бота, для админских команд). Внутри обходит все ключи в `_processes`, для каждого вызывает `stop_process(session_id, backend)`. Возвращает количество остановленных.

**Исключения:** не выбрасывает; ошибки при остановке отдельных процессов логируются `error`, остановка остальных продолжается.

### `def is_busy(session_id: str, backend: BackendName) -> bool`

Проверяет, обрабатывает ли процесс `(session_id, backend)` запрос прямо сейчас (включая ожидание ретрая). Используется **транспортным слоем**:
- `claude_interaction.py` — fast-path-проверка перед запуском нового сообщения (не дёргать `send_message`, если процесс уже занят).
- `bot.py.handle_stop` — определить, идёт ли retry-loop, при отсутствии запущенного процесса (когда `has_process=False`, но `is_busy=True` — значит мы между попытками).

**НЕ используется** `session_watcher` — watcher не зависит от `process_manager` (см. CLAUDE.md → «Слоёная архитектура» и `session_watcher_spec.md` → раздел «Зависимости» → «Не зависит»). Координация watcher ↔ process_manager идёт в обратном направлении: транспортный слой вызывает `pause_session`/`resume_session`/`update_session_id` у watcher-а, не watcher опрашивает `is_busy` (см. раздел «Координация с session_watcher» ниже).

**Аргументы:**
- `session_id` (`str`) — идентификатор сессии.
- `backend` (`BackendName`) — backend сессии.

**Возвращает:** `bool`. `False` для несуществующего ключа.

### `def has_process(session_id: str, backend: BackendName) -> bool`

Проверяет, есть ли запущенный процесс для `(session_id, backend)` (returncode is None).

**Возвращает:** `bool`.

### `async update_session_id(old_session_id: str, new_session_id: str, backend: BackendName) -> None`

Обновляет ключ во всех трёх словарях: `(old_session_id, backend)` → `(new_session_id, backend)`. Backend не меняется — это контракт владения сессией.

**Внутренне:** захватывает `_busy_lock`, для каждого storage в `(_processes, _busy_flags, _stop_events)` выполняет атомарный pop+set (если ключ существует). Ремаппинг должен происходить **до** того, как watcher успеет проснуться и обнаружить новый файл сессии — поэтому `_process_events` вызывает `update_session_id` сразу при первом обнаружении нового `session_id` в потоке, до вызова `session_id_callback` (см. CLAUDE.md → «Единство идентификатора сессии (temp → real)» и алгоритм `_process_events`).

**Возвращает:** ничего.

**Исключения:** не выбрасывает.

### `def register_new_session(backend: BackendName) -> str`

Генерирует временный `session_id` (формат `_new_XXXX`, где XXXX — hex-суффикс из `uuid.uuid4().hex[:12]`) и возвращает его. **НЕ создаёт subprocess** — просто резервирует идентификатор. Subprocess поднимется при первом `send_message(session_id=<temp>, backend=<backend>, ...)`.

**Аргументы:**
- `backend` (`BackendName`) — backend новой сессии (для логирования, не сохраняется во внутреннем состоянии до первого `send_message`).

**Возвращает:** `str` — temp session_id.

**Использование:** `bot.py` вызывает при `/new` для получения temp-id, регистрирует его в `session_manager.set_active_session(chat_id, temp_id, backend)` и `daily_session_registry.register_session(date, temp_id, backend)`, затем отправляет пользователю «Создана сессия #N». При первом текстовом сообщении пользователя — `send_message(session_id=temp_id, ...)` поднимает subprocess; в потоке событий приходит реальный UUID, `session_id_callback` обновляет привязки и реестр.

## Внутренние функции

### `def _generate_temp_session_id() -> str`

Возвращает строку `f"_new_{uuid.uuid4().hex[:12]}"` (например, `"_new_a3f9c2b1d4e5"`). Префикс `_new_` отличает temp-id от реального UUID — позволяет всем потребителям без специального флага решать, делать `--resume` или нет (через `session_id.startswith(TEMP_SESSION_PREFIX)`). 12-значный hex-суффикс — компромисс между читаемостью и уникальностью (16^12 = 2.8e14 — достаточно для процессов одного бота).

### `def _validate_effective_backend(session_id: str, backend: BackendName | None) -> BackendName`

Проверяет backend для backend-aware запроса:

1. Если `backend is not None` — вернуть `backend`.
2. Если `backend is None` — бросить `ProcessManagerError`: backend обязателен.
3. Не читать `current_backend_registry` и не искать backend в `_processes`: владелец сессии определяется верхним слоем до вызова `process_manager`.

### `async _start_subprocess_for_backend(backend: CodingAgentBackend, session_id: str, cwd: str, prompt_text: str, image_paths: list[str]) -> BackendSubprocess`

Поднимает новый subprocess через `claude_runner.start_subprocess_for_backend(args, cwd, stdin_bytes)` — это backend-aware обёртка над `asyncio.create_subprocess_exec`, возвращающая `BackendSubprocess` (см. раздел «Зависимости → claude_runner»). Все CLI-специфичные детали (флаги, формат stdin) формируются методами `backend`, не хардкодятся в `claude_runner`.

Алгоритм:

1. **Определить cli_session_id для команды.** `cli_session_id = None if session_id.startswith(TEMP_SESSION_PREFIX) else session_id`. Для resume передаётся реальный UUID, для temp — `None` (новая сессия). Параметр `session_id` в этой внутренней функции — `str` (никогда не `None`; вызывающая сторона уже получила или сгенерировала идентификатор).
2. **Сформировать команду через backend.** `args = backend.compose_subprocess_command_args(cli_session_id, cwd, prompt_text, image_paths)` — backend сам решает, как формируется команда (флаг `--resume` для Claude, subcommand `resume` для Codex; флаги `--output-format stream-json`, `--effort max` для Claude; `--json --dangerously-bypass-approvals-and-sandbox` для Codex).
3. **Закодировать stdin через backend.** `stdin_bytes = backend.encode_user_message_for_cli_stdin(prompt_text, image_paths)`. Для Claude — JSONL-сообщение в UTF-8; для Codex — `b""` (промпт уже в args).
4. **Поднять процесс через claude_runner.** `backend_process = await claude_runner.start_subprocess_for_backend(args=args, cwd=cwd, stdin_bytes=stdin_bytes)`. Внутри обёртка делает `asyncio.create_subprocess_exec(*args, stdin=PIPE, stdout=PIPE, stderr=PIPE, cwd=cwd, limit=STREAM_BUFFER_LIMIT_BYTES)`, записывает stdin (если `len(stdin_bytes) > 0`), закрывает stdin (EOF), оборачивает в `BackendSubprocess` и возвращает.
5. Вернуть `backend_process`.

**Исключения:**
- `BackendBinaryNotFoundError` (от backend) → перехватить и поднять `ProcessManagerError("Не удалось запустить CLI: бинарник не найден")`.
- `OSError` (от `create_subprocess_exec`) → перехватить и поднять `ProcessManagerError(f"Не удалось запустить subprocess: {e}")`.

### `async _process_events(backend_process: ManagedProcess, session_id: str, backend_obj: CodingAgentBackend, backend_name: BackendName, progress_callback: ProgressCallback | None, session_id_callback: SessionIdCallback | None) -> SendResult`

Читает поток событий из stdout subprocess'а через `process.stdout.readline()` + `backend_obj.parse_stdout_line_into_event`. Извлекает progress, обновляет `last_assistant_text`, ловит финальное событие.

Алгоритм:

1. Инициализация: `last_progress_time = 0.0`, `last_assistant_text = ""`, `final_session_id = session_id`, `terminal_status: TerminalStatus | None = None`, `terminal_event: UnifiedEvent | None = None`, `callback_fired = False`.
2. Цикл по событиям процесса. Для нового backend-aware path используется чтение stdout из `BackendSubprocess` с парсингом через `backend_obj.parse_stdout_line_into_event`; legacy `ClaudeProcess.read_events()` допустим только в Claude-only compatibility path.
   - **Проверка /stop:** `_check_stop_requested((final_session_id, backend_name))`. Если `_stop_events[(final_session_id, backend_name)].is_set()` — `raise ProcessStoppedError`.
   - **Извлечение session_id:** `event_sid = backend_obj.read_session_id_from_event(event)`. Если `event_sid is not None` и `event_sid != final_session_id`:
     - `old_final = final_session_id`
     - `final_session_id = event_sid`
     - **Ремаппинг ключей через `update_session_id(old_final, event_sid, backend_name)`** — атомарно, до вызова session_id_callback. Это гарантия, что watcher не проснётся раньше и не зарегистрирует «дубликата» сессии (см. CLAUDE.md → «Единство идентификатора сессии (temp → real)»).
     - Если `session_id_callback is not None and not callback_fired`:
       - `callback_fired = True`
       - `try: await session_id_callback(old_final, event_sid, backend_name)` (с ловлей и логированием любого исключения; не пробрасывать — callback не должен ломать turn).
   - **Progress:** `progress_text = backend_obj.read_progress_text_from_event(event)`. Если `progress_text is not None and progress_callback and _should_send_progress(last_progress_time)`:
     - `await progress_callback(final_session_id, progress_text)`
     - `last_progress_time = time.monotonic()`
   - **Assistant text:** `text = backend_obj.read_assistant_text_from_event(event)`. Если `text is not None`:
     - `last_assistant_text = text` (последнее ненулевое значение; оба бэкенда могут выдавать несколько `agent_message`, см. родительскую спеку).
   - **Финальное событие:** если `backend_obj.is_turn_complete_event(event)`:
     - `terminal_event = event`
     - `terminal_status = backend_obj.read_terminal_status_from_event(event)`
     - выйти из цикла.
3. После выхода из цикла:
   - Если `terminal_status is None` — поток завершился без финального события (subprocess упал, EOF без `result`/`turn.completed`/`turn.failed`). Прочитать stderr через `_read_stderr` (диагностика), логировать `warning`. Вернуть `SendResult(text="", session_id=final_session_id, backend=backend_name, is_error=True, error_text=stderr_text or "Процесс завершился без финального события", retries_used=0)`. Вызывающий код переходит в `_retry_loop`.
   - Если `terminal_status == TerminalStatus.FAILED`:
     - `error_text = backend_obj.read_error_text_from_event(terminal_event)`
     - вернуть `SendResult(text=last_assistant_text, session_id=final_session_id, backend=backend_name, is_error=True, error_text=error_text, retries_used=0)`. Вызывающий код переходит в `_retry_loop`.
   - Если `terminal_status == TerminalStatus.SUCCESS`:
     - вернуть `SendResult(text=last_assistant_text, session_id=final_session_id, backend=backend_name, is_error=False, error_text=None, retries_used=0)`.

### `async _retry_loop(session_id: str, text: str, backend_obj: CodingAgentBackend, backend_name: BackendName, cwd: str, image_paths: list[str], error_reason: str, progress_callback: ProgressCallback | None, retry_callback: RetryCallback | None, session_id_callback: SessionIdCallback | None) -> SendResult`

Цикл повторных попыток до `MAX_RETRIES = 10` раз с интервалом `RETRY_INTERVAL_SECONDS = 60` между попытками.

Алгоритм:

1. `last_result: SendResult | None = None`, `current_session_id = session_id`.
2. Цикл `for attempt in range(1, MAX_RETRIES + 1)`:
   - `_check_stop_requested((current_session_id, backend_name))`.
   - Если `retry_callback is not None`: `await retry_callback(current_session_id, attempt, MAX_RETRIES, error_reason)`.
   - Лог `warning`: «Повторная попытка attempt/MAX_RETRIES для (session_id, backend)».
   - `await _wait_with_stop_check((current_session_id, backend_name), RETRY_INTERVAL_SECONDS)` — ждать с проверкой stop_event каждую секунду.
   - `_check_stop_requested((current_session_id, backend_name))` (после ожидания).
   - **Перезапуск процесса:** `backend_process = await _restart_process(current_session_id, backend_obj, backend_name, cwd, text, image_paths)`. Это создаёт новый subprocess (старый убит при предыдущем падении или закрылся сам), записывает в `_processes[(current_session_id, backend_name)]`, ставит `_busy_flags=True`. **`_stop_events` не трогает** — Event создан один раз в `send_message` (пункт 3) и переиспользуется до конца turn-а. `_restart_process` лишь проверяет, не нажат ли он (двойной чек: до и после `_start_subprocess_for_backend`); при `is_set()=True` поднимается `ProcessStoppedError`, ретрай прерывается, новый subprocess убивается. Это закрывает race между `stop_process` и retry — см. Сценарий 2 в state machine.
   - **Чтение событий:** `result = await _process_events(claude_process, current_session_id, backend_obj, backend_name, progress_callback, session_id_callback)`. Внутри `_process_events` может произойти ремаппинг session_id — и тогда `result.session_id` будет новым.
   - Если `result.session_id != current_session_id`: `current_session_id = result.session_id` (синхронизируем локальную переменную).
   - Если `not result.is_error`: вернуть `SendResult(..., retries_used=attempt)` (успех).
   - Иначе: лог `warning`, `last_result = result`, `error_reason = (result.error_text or "неизвестная ошибка")[:200]`, продолжить цикл.
3. После исчерпания цикла: лог `error` «Все MAX_RETRIES ретраев исчерпаны для (session_id, backend)». Вернуть `SendResult(text=last_result.text if last_result else "", session_id=current_session_id, backend=backend_name, is_error=True, error_text=last_result.error_text if last_result else None, retries_used=MAX_RETRIES)`.

### `async _wait_with_stop_check(key: ProcessKey, duration_seconds: float) -> None`

Ждёт указанное время, проверяя `_stop_events[key].is_set()` каждые `_STOP_CHECK_INTERVAL_SECONDS = 1.0` секунды. Если флаг set — `raise ProcessStoppedError`.

### `def _check_stop_requested(key: ProcessKey) -> None`

Если `_stop_events.get(key) is not None and _stop_events[key].is_set()` — `raise ProcessStoppedError`. Используется в `_process_events` (между событиями) и в `_retry_loop` (перед/после ожидания).

### `async _restart_process(session_id: str, backend_obj: CodingAgentBackend, backend_name: BackendName, cwd: str, prompt_text: str, image_paths: list[str]) -> ManagedProcess`

Перезапускает процесс CLI для существующей сессии (для retry или для новой попытки после crash):

1. **Под `_busy_lock` — проверка отмены ДО любых действий (pre-check):**
   - `existing_event = _stop_events.get((session_id, backend_name))`.
   - Если `existing_event is None or existing_event.is_set()` → `raise ProcessStoppedError(f"Процесс {(session_id, backend_name)} остановлен до перезапуска")`. Случай `is_set()` — `stop_process` уже сработал; случай `None` — Event удалён извне (аномалия), безопаснее остановиться. Старый процесс при этом может ещё быть в `_processes` — его убьёт `stop_process` (если он сработал) или `send_message.finally`. На этом шаге `_restart_process` зомби сам не убирает, потому что отмена пришла; старый процесс — забота `stop_process`.
2. Если в `_processes[(session_id, backend_name)]` лежит старый процесс и он `is_running()` — остановить через `_apply_stop_strategy(old_process, backend_obj.get_stop_strategy())` (уборка зомби перед новым запуском).
3. `backend_process = await _start_subprocess_for_backend(backend_obj, session_id, cwd, prompt_text, image_paths)`. Между шагом 1 и этим шагом был `await` (yield-точка) — за это время мог прийти `/stop`. Поэтому ниже идёт повторная проверка.
4. **Под `_busy_lock` — повторная проверка отмены и запись (post-check + commit):**
   - `existing_event = _stop_events.get((session_id, backend_name))`.
   - Если `existing_event is None or existing_event.is_set()`:
     - Только что запущенный subprocess нужно убить — он не должен остаться зомби: `await _apply_stop_strategy(claude_process, backend_obj.get_stop_strategy())` (вне Lock'а, после release; либо синхронно `claude_process.process.kill()` под Lock — допустимо, kill быстрый).
     - `raise ProcessStoppedError(f"Процесс {(session_id, backend_name)} остановлен во время _restart_process")`.
   - Иначе — записать в словари по ключу `(session_id, backend_name)`:
     - `_processes[(session_id, backend_name)] = claude_process`
     - `_busy_flags[(session_id, backend_name)] = True`
     - **`_stop_events` НЕ трогаем** — Event создан в `send_message` пункт 3 и переиспользуется до конца turn-а (Event-singleton).
5. Лог `info`: «Процесс перезапущен: (session_id, backend_name), PID=…».
6. Вернуть `claude_process`.

**Инвариант (Event-singleton):** один `asyncio.Event` на весь turn. Создаётся в `send_message` пункт 3, проверяется (но не пересоздаётся) в `_restart_process`. Это закрывает race между `stop_process` и retry: `stop_process.set()` всегда попадает в тот же Event, который retry-цикл будет читать на следующей проверке. Старая логика «новый Event на каждом retry» порождала race — `stop_process` мог нажать старый Event, а `_restart_process` следом затереть его новым чистым Event, потеряв сигнал.

**Инвариант (двойная проверка отмены):** проверка `is_set()` делается под Lock'ом и до запуска subprocess (pre-check, шаг 1), и после (post-check, шаг 4). Между ними — `await _start_subprocess_for_backend`, в который может вклиниться `stop_process`. Без post-check пользовательский `/stop`, пришедший в этот момент, был бы потерян: `_restart_process` записал бы новый процесс в `_processes` и retry продолжился бы.

**Инвариант (Lock-atomicity):** запись в `_processes` и `_busy_flags` (шаг 4) происходит под одним `async with _busy_lock` — без этого перезапущенный процесс может оказаться неуправляемым (повторный `/stop` не найдёт процесс на ключе).

### `async _apply_stop_strategy(process: ManagedProcess, strategy: StopStrategy) -> None`

Применяет последовательность шагов остановки из `StopStrategy`. Замена `process.terminate()` (которая всегда отправляла SIGTERM).

Алгоритм:

1. Если `process.process.returncode is not None` — процесс уже завершился, ничего не делать.
2. Для каждого `step` из `strategy.steps[:-1]` (все кроме последнего):
   - `process.process.send_signal(step.signal_to_send)` — отправить сигнал.
   - `try: await asyncio.wait_for(process.process.wait(), timeout=step.wait_seconds_before_next); return` — если процесс завершился за указанное время, выйти.
   - `except asyncio.TimeoutError: continue` — перейти к следующему шагу.
3. Последний шаг (`strategy.steps[-1]`, обычно SIGKILL): `process.process.send_signal(step.signal_to_send); await process.process.wait()` — без таймаута. SIGKILL ОС не игнорирует.
4. Лог `info`: «Процесс остановлен через стратегию backend: PID=..., last_signal=..., returncode=...».

**Особый случай:** если `strategy.steps` пуст (теоретическая аномалия — backend вернул `StopStrategy(steps=())`) — `process.process.kill(); await process.process.wait()` (минимально безопасный fallback).

### `def _validate_process_ready(key: ProcessKey) -> ManagedProcess`

Под `_busy_lock` (вызывается из `send_message`):

1. `claude_process = _processes.get(key)`. Если `None` — `raise ProcessNotFoundError(f"Нет процесса для {key}")`. **Заметка:** для temp-id (новая сессия) ключ ещё не зарегистрирован — потребитель должен использовать `register_new_session()` или просто вызвать `send_message` с temp-id, и `_validate_process_ready` будет пропущен (см. ниже про разделение «есть процесс / нет процесса»).

В backend-aware send_message эту функцию заменяет более общая логика:

1. Если ключ `(session_id, backend)` есть в `_processes` и `_busy_flags=True` — `raise ProcessManagerError("Процесс занят другим запросом")`.
2. Если ключа нет — `_start_subprocess_for_backend` поднимает новый.
3. Если есть, но `_busy_flags=False` — поднять новый (старый закрыт после прошлого turn-а).

### `def _prepare_for_send(key: ProcessKey) -> None`

Под `_busy_lock`: `_busy_flags[key] = True`. Если `_stop_events[key]` уже существует — `_stop_events[key].clear()` (на случай несвежего флага от прошлого `/stop`).

### `async _read_stderr(backend_process: ManagedProcess) -> str`

Читает до 500 байт из stderr процесса (не блокирует — fire-and-forget с try/except). Используется для диагностики при падении turn-а без финального события. Возвращает строку UTF-8 с заменой невалидных байтов на `?`.

### `def _should_send_progress(last_progress_time: float) -> bool`

Возвращает `True` если `last_progress_time == 0.0` (первое обновление) или `time.monotonic() - last_progress_time >= PROGRESS_THROTTLE_SECONDS`. Иначе `False`.

## Алгоритм работы

### send_message (полный жизненный цикл turn-а)

1. **Захват backend, cwd и текущего session_id:**
   - `effective_backend = _validate_effective_backend(session_id, backend)` — проверяется и захватывается раз и навсегда для этого turn-а.
   - `effective_cwd = cwd if cwd is not None else config.WORKING_DIR` — захват рабочей директории.
   - `effective_image_paths = image_paths if image_paths is not None else []`.
   - `current_session_id = session_id` — локальная переменная, отслеживает актуальный session_id для cleanup в `finally`. Инициализируется ДО любого `try`, чтобы `finally` всегда имел корректный ключ — даже если исключение случилось до старта subprocess, до первого события с реальным UUID, до `_retry_loop` или внутри него. Обновляется при temp→real ремаппинге через tracking-обёртку над `session_id_callback` (см. п. 5). Не зависит от появления `result` — это критично для случаев `ProcessStoppedError`, `ProcessManagerError`, любого необработанного исключения до финального события.
2. **Получение backend-инстанса:**
   - `backend_obj = get_backend(effective_backend)` (singleton из `coding_agent_backend.get_backend`).
3. **Подготовка ключа и проверка занятости:**
   - `key = (session_id, effective_backend)`.
   - Под `_busy_lock`:
     - Если `_busy_flags.get(key, False)`: `raise ProcessManagerError(f"Процесс {key} уже занят")`.
     - `_busy_flags[key] = True` (помечаем busy ДО запуска subprocess, чтобы конкурентный send_message не успел вклиниться).
     - `_stop_events[key] = asyncio.Event()` — **единственное место создания Event для всего этого turn-а**. Если старый объект Event есть в словаре (от прошлого turn-а, finally которого по какой-то причине не отработал) — заменяется новым. На протяжении всего turn-а — включая `_process_events`, все ретраи в `_retry_loop` и каждый вызов `_restart_process` — Event этот же объект, **не пересоздаётся**. Это инвариант Event-singleton (см. `_restart_process` и Сценарий 2 в state machine). Без него возможна потеря `/stop`-сигнала: `stop_process` нажимает Event, а `_restart_process` следом за ним создаёт новый чистый Event и сигнал теряется.
4. **Запуск subprocess:**
   - `backend_process = await _start_subprocess_for_backend(backend_obj, session_id, effective_cwd, text, effective_image_paths)`.
   - Под `_busy_lock`: `_processes[key] = backend_process`. Это надо сделать **после** запуска — если запуск упал, ключ в `_processes` не появится, но `_busy_flags=True` останется до finally.
5. **Чтение событий:**
   - Перед вызовом `_process_events` оборачиваем `session_id_callback` так, чтобы при каждом ремаппинге обновлялась локальная `current_session_id`:
     ```python
     async def _tracking_session_id_callback(old_sid, new_sid, backend):
         nonlocal current_session_id
         current_session_id = new_sid
         if session_id_callback is not None:
             await session_id_callback(old_sid, new_sid, backend)
     ```
     Эта обёртка передаётся в `_process_events` (и далее в `_retry_loop`) вместо исходного `session_id_callback`. Гарантия: к моменту входа в `finally` переменная `current_session_id` всегда содержит актуальный ключ — даже если `result` так и не был сформирован. Обёртка — обычное замыкание над `nonlocal`-переменной, без глобального состояния, поэтому параллельные `send_message` не конфликтуют (каждый имеет свою `current_session_id`).
   - `try: result = await _process_events(claude_process, session_id, backend_obj, effective_backend, progress_callback, _tracking_session_id_callback)`.
   - `except ProcessStoppedError: raise` (пробросить наружу — bot.py обработает; `finally` отработает корректно по `current_session_id`).
   - `except BackendProtocolError as e: result = SendResult(text="", session_id=current_session_id, backend=effective_backend, is_error=True, error_text=str(e), retries_used=0)` — protocol error трактуется как ошибка turn-а (повод для retry); используем `current_session_id`, потому что ремаппинг мог произойти до падения.
6. **Если result.is_error → retry:**
   - `error_reason = (result.error_text or result.text or "неизвестная ошибка")[:200]`.
   - `current_session_id` **уже актуально** — обновлено tracking-обёрткой при ремаппинге (если он был). Дополнительно проверяем синхронность: `assert current_session_id == result.session_id` (инвариант для отладки; в production — `if current_session_id != result.session_id: current_session_id = result.session_id` как defensive sync, на случай если `_process_events` вернул более новый `result.session_id` без вызова callback).
   - `result = await _retry_loop(current_session_id, text, backend_obj, effective_backend, effective_cwd, effective_image_paths, error_reason, progress_callback, retry_callback, _tracking_session_id_callback)` — внутрь `_retry_loop` уходит та же tracking-обёртка, чтобы и при ремаппинге внутри ретраев `current_session_id` оставалось актуальным.
7. **finally — очистка:**
   - Под `_busy_lock`:
     - `final_key = (current_session_id, effective_backend)` — `current_session_id` поддерживается tracking-обёрткой (см. п. 1, 5) и **не зависит от существования `result`**. Это критично: `result` может не появиться вообще:
       - `ProcessStoppedError` от `/stop` — `_process_events` пробрасывает исключение до того, как `result` будет присвоен.
       - `ProcessManagerError` от падения запуска subprocess — `_start_subprocess_for_backend` упал, до `_process_events` дело не дошло.
       - Любое необработанное исключение (включая `RuntimeError`, `OSError` из I/O) до финального события.

       Использование `result.session_id` в этом блоке привело бы к `UnboundLocalError` — `finally` упал бы без cleanup, и ключ навсегда остался бы в `_busy_flags=True`. Сессия превратилась бы в «вечно занятую» — все следующие `send_message` для неё поднимали бы `ProcessManagerError("Процесс уже занят")`.
     - `if final_key in _busy_flags: _busy_flags[final_key] = False` (не удалять — `is_busy()` должен корректно вернуть False; следующий send_message переиспользует ключ).
     - `_stop_events.pop(final_key, None)` (удалить — flag отслужил, retry-цикл завершён).
     - **Не трогаем `_processes`** — старый процесс уже завершился сам (после `result`/`turn.completed`/EOF). Ключ останется до следующего send_message в той же сессии (там будет `_restart_process`, который заменит значение). Зомби-объекты `ManagedProcess` с `returncode != None` безвредны.
8. **Возврат:** `return result`.

### stop_process (обработка /stop)

1. `key = (session_id, backend)`.
2. Под `_busy_lock`:
   - `claude_process = _processes.get(key)` (может быть `None`).
   - `was_retrying = _busy_flags.get(key, False)`.
   - Если `key in _stop_events`: `_stop_events[key].set()` — это разбудит `_wait_with_stop_check` в retry-цикле и заставит `_check_stop_requested` поднять `ProcessStoppedError` в `_process_events`.
   - **Удалить из `_processes` и `_busy_flags`**: `_processes.pop(key, None); _busy_flags.pop(key, None)`. **НЕ удалять `_stop_events[key]`** — retry-цикл ещё может проверить флаг через `_check_stop_requested`. Очистка `_stop_events` — в finally `send_message`.
3. Вне Lock (terminate может занять секунды):
   - `was_running = False`.
   - Если `claude_process is not None and claude_process.is_running()`:
     - `was_running = True`.
     - `backend_obj = get_backend(backend)`.
     - `await _apply_stop_strategy(claude_process, backend_obj.get_stop_strategy())`.
4. Лог `info`: «Процесс остановлен: key=..., was_running=..., was_retrying=...».
5. Вернуть `StopResult(was_running, was_retrying, backend)`.

### State machine /stop в трёх сценариях

#### Сценарий 1: /stop во время штатной обработки сообщения

Ситуация: `send_message` запущен, идёт цикл `_process_events`, busy_flag=True, stop_event создан и не set.

Шаги:
1. `bot.py` ловит команду `/stop`, вызывает `stop_process(session_id, backend)`.
2. `stop_process` под Lock: устанавливает `_stop_events[key].set()`, удаляет ключ из `_processes` и `_busy_flags`.
3. `stop_process` вне Lock: `_apply_stop_strategy(process, strategy)` — отправляет SIGTERM (Claude) или SIGINT (Codex), ждёт штатного завершения, добивает SIGKILL при таймауте.
4. **Параллельно** `_process_events` либо успел увидеть `_stop_events[key].is_set() = True` через `_check_stop_requested` и поднял `ProcessStoppedError`, либо subprocess уже умер — `read_events` дойдёт до EOF и вернётся с `terminal_status=None`.
5. Если `_process_events` поднял `ProcessStoppedError` — `send_message` пробрасывает его наружу (после finally).
6. Если `_process_events` вернулся с ошибкой EOF — `send_message` идёт в `_retry_loop`, который проверяет `_stop_events[key].is_set()` через `_check_stop_requested` **сразу на входе** и поднимает `ProcessStoppedError`. Retry не выполнится.
7. `send_message.finally` под Lock: `_busy_flags[key] = False`, `_stop_events.pop(key, None)`.

Результат: пользователь получает «Claude остановлен», процесс мёртв, словари чистые.

#### Сценарий 2: /stop во время retry-цикла (новый процесс, новый busy_flag, **тот же** stop_event)

Ситуация: `_retry_loop` ждёт `_wait_with_stop_check(key, 60)` или начал новую попытку через `_restart_process`. После `_restart_process`:
- `_processes[key]` — НОВЫЙ процесс,
- `_busy_flags[key] = True` (свежий, поставлен в `_restart_process`),
- `_stop_events[key]` — **ТОТ ЖЕ Event**, что был создан в `send_message` пункт 3 (Event-singleton инвариант). `_restart_process` НЕ пересоздаёт Event.

Шаги:
1. `bot.py` вызывает `stop_process(session_id, backend)`.
2. `stop_process` под `_busy_lock` находит обновлённый `_processes[key]` (новый процесс), `_busy_flags[key]=True` (новый busy_flag — `was_retrying=True`), ставит `_stop_events[key].set()`. Удаляет ключ из `_processes` и `_busy_flags`. **НЕ удаляет `_stop_events[key]`** — retry-цикл и `_restart_process` ещё могут проверить флаг через `_check_stop_requested` или pre-check.
3. Применяет стратегию остановки к новому процессу — он умирает или штатно прерывается (Codex).
4. `_retry_loop` либо в `_wait_with_stop_check` — он каждую секунду проверяет `_stop_events[key].is_set()`, видит True, поднимает `ProcessStoppedError`. Либо в `_process_events` нового запроса — `_check_stop_requested` поднимает `ProcessStoppedError`. Либо в начале новой итерации `_retry_loop` — `_check_stop_requested` перед `retry_callback`. Либо на pre-check шага 1 в следующем `_restart_process` — поднимает `ProcessStoppedError` ДО запуска subprocess.
5. `send_message.finally` чистит словари (включая удаление `_stop_events[key]`).

#### Сценарий 2.1 (подсценарий): /stop пришёл во время самого `_restart_process`

Race-окно — между двумя захватами `_busy_lock` внутри `_restart_process`: pre-check (шаг 1) уже завершён, идёт `await _start_subprocess_for_backend` (шаг 3). Это yield-точка.

Шаги:
1. `_restart_process` шаг 1 (под Lock): `existing_event.is_set() == False` — Event не нажат. Lock освобождён. Старый процесс остановлен (шаг 2).
2. Идёт `await _start_subprocess_for_backend` (шаг 3) — создаётся новый subprocess. Это занимает время.
3. **В этом окне** `stop_process` захватывает Lock, выполняет `_stop_events[key].set()`, удаляет `_processes[key]` (старого процесса там может уже не быть — был убит на шаге 2 `_restart_process`) и `_busy_flags[key]`, освобождает Lock. Применяет стратегию остановки — но старый процесс уже мёртв, так что `was_running` зависит от состояния на момент set: если успели до шага 2 — был жив, иначе нет.
4. `_restart_process` завершает `_start_subprocess_for_backend`, получает новый subprocess. Захватывает Lock на шаге 4 (post-check).
5. `existing_event.is_set() == True` → `_restart_process` убивает только что запущенный subprocess через `_apply_stop_strategy` и поднимает `ProcessStoppedError`. Записи в `_processes` не происходит — нового зомби в словарях нет.
6. `_retry_loop` пробрасывает `ProcessStoppedError` наружу. `send_message` пробрасывает. `finally` чистит остатки.

Ключевой инвариант: Event один и тот же на весь turn. `stop_process.set()` всегда попадает в Event, который `_restart_process` будет читать на post-check'е. Race закрывается двойной проверкой `is_set()` под Lock'ом (pre-check на шаге 1, post-check на шаге 4) — между ними await-yield, в которой может вклиниться `stop_process`, но post-check ловит этот сигнал и убивает только что поднятый subprocess до того, как он окажется в словарях.

Альтернативный путь (если post-check каким-то образом пропустил сигнал — формально невозможно, asyncio однопоточен): `_check_stop_requested` в `_process_events` следующего вызова поднимет `ProcessStoppedError` на первой проверке. То есть на post-check'е сигнал ловится в худшем случае «рано» (новый subprocess убивается сразу), в лучшем — «вовремя» (subprocess прожил несколько событий, потом убит). В обоих случаях retry останавливается, утечек нет.

#### Сценарий 3: /stop после смены session_id (ремаппинг stop_event на новый ключ)

Ситуация: `_process_events` получил первое событие с реальным UUID, выполнил `update_session_id(temp_id, real_id, backend)`. Под Lock все три словаря ремапнуты: `(temp_id, backend) → (real_id, backend)`. Локальная переменная `final_session_id = real_id`. После этого `_process_events` продолжает читать события.

Параллельно `bot.py` ловит `/stop` от пользователя. У `bot.py` уже актуальный session_id (real_id), потому что `session_id_callback` (вызванный из `_process_events` после ремаппинга) обновил `session_manager`. `bot.py` вызывает `stop_process(real_id, backend)`.

Шаги:
1. `stop_process(real_id, backend)` под Lock.
2. `key = (real_id, backend)` — этот ключ существует в словарях (после ремаппинга).
3. `_stop_events[(real_id, backend)].set()` — флаг ставится на новом ключе.
4. `_check_stop_requested` в `_process_events` использует `(final_session_id, backend_name) = (real_id, backend_name)` — и видит `is_set() = True`. Поднимает `ProcessStoppedError`.
5. Дальше — как в Сценарии 1.

Ключевой инвариант: ремаппинг `update_session_id` происходит **под `_busy_lock`**, и `bot.py` (через `session_id_callback`, тоже под Lock?) узнаёт реальный session_id уже после ремаппинга. Если `/stop` пришёл *между* ремаппингом и `session_id_callback`, `bot.py` ещё не обновил `session_manager` и попытается остановить по старому temp_id. Чтобы такой race не оставлял зомби:
- `bot.py` при `/stop` сначала проверяет `session_manager.get_active_session(chat_id)` — получает (возможно, ещё temp) ID;
- если `stop_process(temp_id, backend)` находит ключ — отлично;
- если не находит — `_processes` уже ремапнут на real_id, но `bot.py` об этом не знает. Решение: **`stop_process` при отсутствии ключа `(input_session_id, backend)` делает fallback-проверку**: ищет в `_processes` любой ключ с тем же `chat_id`-связанным session_id через `session_manager.get_active_session` (но это ломает чистоту слоёв). Или: гарантирует, что между `update_session_id` и `session_id_callback` нет async-точки, в которой `bot.py` может вклиниться (callback вызывается **сразу** в том же task'е, где сделан ремаппинг).

В реализации (см. `process_manager.py:218-232` в существующем коде) callback вызывается сразу после ремаппинга, без `await` между ними кроме самого `await session_id_callback(...)`. Этого достаточно: race-окно в один await-yield, и если оно случилось — пользователь видит «Процесс не найден» и просто повторяет `/stop` (ничего не сломалось).

### update_session_id (ремаппинг ключей)

1. `old_key = (old_session_id, backend)`, `new_key = (new_session_id, backend)`.
2. Под `_busy_lock`:
   - Для каждого `storage` в `(_processes, _busy_flags, _stop_events)`:
     - `if old_key in storage: storage[new_key] = storage.pop(old_key)`.
3. Лог `info`: «Session ID обновлён: old → new (backend=...)».

Гарантии:
- Атомарность: все три словаря ремапятся в одном захвате Lock'а — никакая другая корутина не увидит «полу-ремаппинга» (когда `_processes` уже на новом ключе, а `_busy_flags` ещё на старом).
- Идемпотентность: если ключа нет в storage — операция skip'ается, не падает.

### _process_events (детально, см. также «Алгоритм работы → Жизненный цикл одного turn-а» в `coding_agent_backend_spec.md:720-753`)

Этот алгоритм — обвязка над интерфейсом backend. Главные delta-точки backend-aware:

- `event = await ...readline()` → `parse_stdout_line_into_event` (вместо прямого `json.loads` в Claude-only коде).
- Определение «ремаппинг ли это» — через `read_session_id_from_event(event) != final_session_id` (а не через прямую проверку `event["session_id"]`).
- Извлечение progress — через `read_progress_text_from_event` (а не через захардкоженный `_extract_progress_text`).
- Финальное событие — через `is_turn_complete_event` + `read_terminal_status_from_event` (а не `event["type"] == "result"`).
- Текст ответа — через `read_assistant_text_from_event` (Claude — `event["result"]`, Codex — `item.text` события `item.completed`).
- Текст ошибки — через `read_error_text_from_event` (Claude — `event["result"]`, Codex — `event["error"]["message"]`).

См. полный алгоритм в разделе «Внутренние функции → _process_events» выше.

## Зависимости

### От модулей проекта

- **`coding_agent_backend`** — `BackendName`, `CodingAgentBackend`, `UnifiedEvent`, `TerminalStatus`, `StopStrategy`, `StopSignalStep`, `get_backend`, `BackendBinaryNotFoundError`, `BackendProtocolError`. Используются для определения backend, парсинга событий, формирования команды subprocess, выбора стратегии остановки.
- **`current_backend_registry`** — не используется как fallback внутри backend-aware `process_manager`. Текущий backend читает верхний слой перед созданием новой сессии и передаёт его в `send_message`; существующие сессии используют backend из `session_manager`.
- **`claude_runner`** — новый backend-aware путь использует `start_subprocess_for_backend(args: list[str], cwd: str, stdin_bytes: bytes) -> BackendSubprocess`. `process_manager` формирует `args` через `backend.compose_subprocess_command_args(...)`, формирует `stdin_bytes` через `backend.encode_user_message_for_cli_stdin(...)` и передаёт готовые значения в runner. Захардкоженные Claude-флаги (`-p`, `--output-format stream-json`, `--input-format stream-json`, `--verbose`, `--dangerously-skip-permissions`, `--effort max`) живут в `ClaudeCodeBackend.compose_subprocess_command_args`, а не в `process_manager`.

  Legacy `ClaudeProcess`, `start_process(session_id: str | None, cwd: str | None)` и `ClaudeProcess.terminate()` могут временно оставаться в `claude_runner` как compatibility debt для старых callers и тестов. Это не считается нарушением backend-aware контракта, если новый Telegram/backend-aware flow использует только `start_subprocess_for_backend(...)` и `_apply_stop_strategy(process, backend.get_stop_strategy())`. `process_manager` не должен удалять legacy API в рамках этой спеки и не должен опираться на claude-specific parsing внутри legacy `ClaudeProcess`.
- **`config`** — `WORKING_DIR`. Используется только как fallback для `cwd`, когда потребитель не передаёт явно. На каждом ретрае используется захваченное значение, не `config.WORKING_DIR` повторно.

### Стандартная библиотека

- `asyncio` — `Lock`, `Event`, `wait_for`, `TimeoutError`, `subprocess.Process` (через claude_runner).
- `time` — `monotonic` для throttle progress.
- `uuid` — `uuid4().hex` для генерации temp session_id.
- `dataclasses` — `dataclass(frozen=True)` для `SendResult` и `StopResult`.

**Спецификации модулей-потребителей** (`bot.py`, частично `claude_interaction.py`) на момент написания этой спеки уже существуют в Claude-only версии (`dev/docs/specs/realised/bot_spec.md`). Для backend-aware они должны быть обновлены отдельной задачей: расширить `_send_to_claude_and_respond` параметром `backend` (захват из `session_manager.get_active_session(chat_id).backend`), обновить `handle_stop` для передачи `backend` в `stop_process`, обновить `session_id_callback` под трёхаргументную сигнатуру. Без обновлений потребителей реализация модуля не сломает текущий код (Claude по умолчанию), но не активирует Codex.

## Обработка ошибок

- **Бинарник CLI не найден (`BackendBinaryNotFoundError`)** — `_start_subprocess_for_backend` ловит, оборачивает в `ProcessManagerError("Не удалось запустить CLI: бинарник не найден; backend=...")`. Логируется `error`. Пользователь увидит сообщение от `bot.py`.
- **`OSError` при `create_subprocess_exec`** — те же действия, оборачивается в `ProcessManagerError`.
- **Невалидный JSON в stdout (`BackendProtocolError`)** — поймать в `send_message`, превратить в `SendResult(is_error=True, error_text=str(e))`, перейти в `_retry_loop` (это контрактное нарушение CLI; ретрай — единственная разумная реакция).
- **`turn.failed` (Codex) или `result.is_error=true` (Claude)** — `read_terminal_status_from_event` вернёт `TerminalStatus.FAILED`. `is_error=True`, `error_text` от `read_error_text_from_event`, переход в `_retry_loop`. Если ретраи исчерпаны — `SendResult(is_error=True, retries_used=MAX_RETRIES)` — `bot.py` показывает пользователю `error_text`.
- **Subprocess завершился без финального события (EOF без `result`/`turn.completed`/`turn.failed`)** — `terminal_status=None`. Прочитать stderr через `_read_stderr` для диагностики, лог `warning`, `is_error=True`, `error_text=stderr_text or "оборван без финального события"`, переход в `_retry_loop`.
- **`/stop` во время обработки** — `_check_stop_requested` поднимает `ProcessStoppedError`. `send_message` пробрасывает наружу. `bot.py` ловит и отвечает «Claude остановлен».
- **Зависший процесс при `_apply_stop_strategy`** — стратегия гарантирует SIGKILL последним шагом. Если даже SIGKILL не помог (zombie-процесс уровня ОС) — лог `error`, `_apply_stop_strategy` возвращается всё равно (бот не должен зависнуть из-за одной зомби-сессии).
- **`session_id_callback` упал с исключением** — поймать в `_process_events`, лог `error` с traceback, продолжить чтение событий (callback не должен ломать turn).
- **Конкурентный `send_message` для одного ключа** — `_busy_flags[key]=True` уже стоит, второй вызов под Lock увидит это и поднимет `ProcessManagerError("Процесс {key} уже занят")`. Это защита от race condition.
- **Транзиентный `EDEADLK` (errno 11) при чтении stderr** — `_read_stderr` ловит любое исключение и возвращает `""`. EDEADLK на macOS известен (см. CLAUDE.md → «Транзиентная ошибка EDEADLK»), здесь не критичен (диагностический read).

## Контракты с внешними системами

Этот модуль **не имеет прямых контрактов с внешними CLI** — все они инкапсулированы в `CodingAgentBackend` и его реализациях (`ClaudeCodeBackend`, `CodexBackend`). См. родительскую спеку `coding_agent_backend_spec.md` → «Контракты с внешними системами».

Единственный «контракт» этого модуля — с `asyncio.subprocess.Process`:

### POSIX-сигналы и `Process.send_signal`

**Источник правды:** Python stdlib `asyncio.subprocess.Process.send_signal(signum)` (`Lib/asyncio/subprocess.py`) — обёртка над `os.kill(pid, signum)` для POSIX-систем. На macOS значения `signal.SIGTERM`, `signal.SIGINT`, `signal.SIGKILL` стандартные (15, 2, 9 соответственно).

**Алгоритм `_apply_stop_strategy`:** для каждого шага `step` — `process.send_signal(step.signal_to_send)` + `await asyncio.wait_for(process.wait(), timeout=step.wait_seconds_before_next)`. На таймауте — переход к следующему шагу. Последний шаг (SIGKILL) — без таймаута.

**Эмпирический контракт:** SIGKILL ОС не игнорирует — `process.wait()` вернётся гарантированно (если процесс жив, он будет убит мгновенно; если уже мёртв, `wait` возвращается сразу). Тест-план обязан проверить, что `_apply_stop_strategy` корректно эскалирует через все шаги и НЕ зависает.

## Координация с session_watcher

При отправке сообщения через `bot.py`:
- ответ может прийти через `process_manager` (`send_message` читает stdout subprocess'а и возвращает `SendResult`),
- параллельно `session_watcher` опрашивает файлы сессий с диска каждые `WATCHER_POLL_INTERVAL_SECONDS = 2` секунды и видит те же финальные сообщения, записанные CLI в JSONL.

Чтобы пользователь не получил ответ дважды, используется **push-механизм координации из транспортного слоя в watcher** (вызовы делает `bot.py` / `claude_interaction.py`):

- **`session_watcher.pause_session(session_id, backend)`** — вызывается ПЕРЕД отправкой запроса в `process_manager.send_message`. С этого момента watcher для пары `(session_id, backend)` не доставляет обнаруженные обновления — он знает, что ответ придёт через `process_manager.SendResult`.
- **`session_watcher.resume_session(session_id, backend, last_delivered_idx)`** — вызывается ПОСЛЕ возврата `SendResult`. Транспортный слой передаёт индекс последнего сообщения, доставленного через `process_manager`, чтобы watcher не дублировал его при следующем опросе.
- **`session_watcher.update_session_id(old, new, backend)`** — вызывается при temp→real ремаппинге; синхронизировано с `process_manager.update_session_id` (оба обновляют свои внутренние ключи под общей координацией из транспортного слоя).

**Направление зависимостей.** `process_manager` (слой инфраструктуры) и `session_watcher` (слой мониторинга) живут в **разных слоях** (см. CLAUDE.md → «Слоёная архитектура»). Прямой связи между ними нет: `session_watcher` **не импортирует** `process_manager` и **не опрашивает** `is_busy`. Это сознательное архитектурное решение: цикл watcher → process_manager создал бы циклическую зависимость, что `session_watcher_spec.md` (раздел «Зависимости» → «Не зависит») явно называет «архитектурной регрессией». Координирует оба модуля транспортный слой, который стоит выше них в иерархии.

В backend-aware версии:
- watcher запускается **по одной инстанции на бэкенд** (см. `coding_agent_backend_spec.md` → «Контракт владения сессией → session_watcher»);
- транспортный слой передаёт `backend` в каждый из вызовов `pause_session`/`resume_session`/`update_session_id`, попадая в нужную инстанцию watcher-а.

**Внутренний `_busy_flags` этого модуля** используется НЕ для координации с watcher-ом, а для двух других целей:
- защита `send_message` от конкурентного запуска двух запросов в одну пару `(session_id, backend)` (поднимается `ProcessManagerError("Процесс уже занят")`);
- транспортный слой читает его через публичный `is_busy()` — для fast-path в `claude_interaction` и для определения активного retry-loop в `handle_stop` (см. описание `is_busy` в разделе «Публичный API»).

- **test_stop_during_normal_processing** — Сценарий 1: /stop во время штатной обработки.
  - Вход: `send_message` запущен в task A, MockBackend задерживает первое событие на 5с. В task B через 1с вызывается `stop_process(session_id, backend)`.
  - Ожидаемый результат: task A пробрасывает `ProcessStoppedError`. `_apply_stop_strategy` отправил `SIGTERM` на mock-процесс (для Claude), потом `SIGKILL` через 5с при таймауте. После завершения: `_processes[(session_id, backend)]` нет, `_busy_flags[(session_id, backend)]` нет, `_stop_events[(session_id, backend)]` нет.
  - Тип: integration (с реальной координацией двух tasks).

- **test_stop_during_retry_loop_same_event_persists** — Сценарий 2: /stop во время retry-цикла после `_restart_process` (новый процесс, новый busy_flag, **тот же** stop_event).
  - Вход: первая попытка завершается с `is_error=True` (turn.failed). `_retry_loop` ждёт RETRY_INTERVAL_SECONDS. Через 30с (внутри `_wait_with_stop_check`) — `stop_process(session_id, backend)`. ДО первой попытки захватить ссылку на `_stop_events[key]` в переменную `event_initial`.
  - Ожидаемый результат: `_wait_with_stop_check` каждую секунду проверяет `_stop_events[key].is_set()`, видит True на 31-й проверке, поднимает `ProcessStoppedError`. `send_message` пробрасывает наружу. **Ключевой ассерт: на момент `stop_process` (через 30с) `_stop_events[key] is event_initial`** — Event не пересоздан между попытками (Event-singleton инвариант). После finally ключ `(session_id, backend)` чист во всех словарях, включая `_stop_events`.
  - Тип: integration.

- **test_stop_during_restart_process_aborts_retry** — Сценарий 2.1: /stop пришёл во время самого `_restart_process` (race между запуском нового subprocess и записью в словари).
  - Вход: первая попытка завершается с `is_error=True`. RETRY_INTERVAL_SECONDS подменён на 0.01. `_retry_loop` входит в `_restart_process`. MockBackend подменяет `_start_subprocess_for_backend` на функцию с `await asyncio.sleep(2.0)` перед возвратом нового MockProcess. В отдельной task через 1с — `stop_process(session_id, backend)`: ставит `_stop_events[key].set()`. После возврата `_start_subprocess_for_backend` `_restart_process` захватывает Lock, выполняет post-check `is_set()` → True.
  - Ожидаемый результат: `_restart_process` поднимает `ProcessStoppedError` ДО записи нового MockProcess в `_processes`. Только что запущенный subprocess убит через `_apply_stop_strategy` (проверяется через mock — `send_signal` вызван). `_retry_loop` пробрасывает. `send_message` пробрасывает наружу. **Ключевые ассерты:** (1) `ProcessStoppedError` поднялся; (2) retry не запустился — `retry_callback` для попытки №2 не вызван; (3) новый MockProcess НЕ остался в `_processes[key]`; (4) после finally `_processes`, `_busy_flags`, `_stop_events` для `(session_id, backend)` чисты.
  - Тип: integration (race condition coverage).

- **test_stop_after_session_id_change_finds_remapped_event** — Сценарий 3: /stop после смены session_id.
  - Вход: `send_message(session_id="_new_x", backend=BackendName.CLAUDE)`. MockBackend сначала возвращает `system` (session_id="real-y") — это вызывает ремаппинг. Затем event с задержкой. В этот момент task B вызывает `stop_process("real-y", BackendName.CLAUDE)`.
  - Ожидаемый результат: `_stop_events[("real-y", BackendName.CLAUDE)].set()` сработал. `_check_stop_requested` в `_process_events` поднял `ProcessStoppedError`. Ключ `("real-y", BackendName.CLAUDE)` после finally чист, `("_new_x", BackendName.CLAUDE)` тоже чист (был ремапнут).
  - Тип: integration.

### Retry-цикл (3 кейса)

- **test_retry_on_process_crash** — retry при падении процесса (EOF без финального события).
  - Вход: первая попытка — MockBackend возвращает только `system`-событие, потом EOF. Вторая попытка — `system` + `result` (success).
  - Ожидаемый результат: `SendResult(text="Готово", is_error=False, retries_used=1)`. Лог `warning` про первую попытку с диагностикой stderr. `retry_callback` вызван 1 раз с `(session_id, 1, 10, "...stderr text...")`.
  - Тип: error.

- **test_retry_on_turn_failed_codex** — retry при `turn.failed` (Codex), `error_text` извлечён.
  - Вход: backend=Codex. Первая попытка — `thread.started`, `turn.failed` (`error.message="Connection reset"`). Вторая попытка — успех.
  - Ожидаемый результат: `SendResult(text="...", is_error=False, retries_used=1, backend=BackendName.CODEX)`. Внутри между попытками — `error_reason` в `retry_callback` содержит "Connection reset".
  - Тип: error.

- **test_all_retries_exhausted_returns_error_with_text** — все 10 ретраев упали.
  - Вход: MockBackend всегда возвращает `result(is_error=True, result="Service unavailable")`. RETRY_INTERVAL_SECONDS подменён на 0.01 для скорости теста.
  - Ожидаемый результат: `SendResult(text="Service unavailable" or last_assistant_text, is_error=True, retries_used=10, error_text="Service unavailable")`. `retry_callback` вызван 10 раз. Лог `error` про исчерпание.
  - Тип: error.

### Конкурентность (1 кейс)

- **test_concurrent_send_message_for_same_key_raises** — конкурентный `send_message` для одного ключа защищён `_busy_lock`.
  - Вход: task A — `send_message(session_id="_new_a", backend=BackendName.CLAUDE)` (висит на чтении событий 5с). Task B через 1с — `send_message(session_id="_new_a", backend=BackendName.CLAUDE)`.
  - Ожидаемый результат: task B сразу поднимает `ProcessManagerError("Процесс {key} уже занят")`. Task A продолжает работу и завершается успешно.
  - Тип: edge case.

### Раздельные словари по backend (1 кейс)

- **test_same_session_id_different_backends_separate_state** — один и тот же session_id для CLAUDE и CODEX — независимые ключи.
  - Вход: `send_message(session_id="uuid-shared", backend=BackendName.CLAUDE)` запущен в task A (висит). Параллельно `send_message(session_id="uuid-shared", backend=BackendName.CODEX)` в task B.
  - Ожидаемый результат: оба запроса работают одновременно. `_processes` содержит два ключа: `("uuid-shared", BackendName.CLAUDE)` и `("uuid-shared", BackendName.CODEX)`. `is_busy("uuid-shared", BackendName.CLAUDE) == True` и `is_busy("uuid-shared", BackendName.CODEX) == True`. После завершения — оба ключа чистые.
  - Тип: edge case.

### Backend-stop-strategy (2 кейса)

- **test_stop_uses_claude_strategy_sigterm_then_sigkill** — для Claude применяется SIGTERM → wait 5с → SIGKILL.
  - Вход: процесс под backend=CLAUDE, MockClaudeBackend.get_stop_strategy возвращает `StopStrategy(steps=(StopSignalStep(signal.SIGTERM, 5.0), StopSignalStep(signal.SIGKILL, 0.0)))`. Mock-процесс игнорирует SIGTERM (не завершается), завершается на SIGKILL. `stop_process(session_id, BackendName.CLAUDE)`.
  - Ожидаемый результат: `process.send_signal` вызван с `SIGTERM`, потом (через ~5с) с `SIGKILL`. Все ключи очищены, `was_running=True`.
  - Тип: integration.

- **test_stop_uses_codex_strategy_sigint_then_sigterm_then_sigkill** — для Codex применяется SIGINT → SIGTERM → SIGKILL.
  - Вход: backend=CODEX, стратегия — `(SIGINT, 5), (SIGTERM, 5), (SIGKILL, 0)`. Mock-процесс игнорирует SIGINT и SIGTERM, завершается на SIGKILL.
  - Ожидаемый результат: `process.send_signal` вызван последовательно с SIGINT, SIGTERM, SIGKILL. Между каждым — таймаут 5с. Финал — `was_running=True`, ключи очищены.
  - Тип: integration.

### Граничные случаи (4 кейса)

- **test_send_message_empty_response_returns_empty_text_no_error** — пустой ответ от Claude (маркер `"No response requested."`).
  - Вход: MockClaudeBackend возвращает `result(is_error=False, result="No response requested.")`. `read_assistant_text_from_event` возвращает `""` (см. контракт `claude_code_backend_spec.md` → `read_assistant_text_from_event`).
  - Ожидаемый результат: `SendResult(text="", is_error=False, retries_used=0)`. Не ретрай — пустой ответ при `is_error=False` валиден.
  - Тип: edge case.

- **test_stop_nonexistent_key_returns_safe_result** — `stop_process` для несуществующего ключа.
  - Вход: ключ `("never-existed", BackendName.CLAUDE)` не в словарях. Вызов `stop_process("never-existed", BackendName.CLAUDE)`.
  - Ожидаемый результат: `StopResult(was_running=False, was_retrying=False, backend=BackendName.CLAUDE)`. Никаких исключений, никаких сигналов.
  - Тип: edge case.

- **test_session_id_callback_exception_does_not_break_turn** — исключение в `session_id_callback` не ломает обработку.
  - Вход: `session_id_callback = AsyncMock(side_effect=ValueError("simulated"))`. MockBackend возвращает `system`+ремаппинг. После — `result(success)`.
  - Ожидаемый результат: `SendResult(is_error=False)` — turn завершился успешно. Лог `error` с traceback ValueError. Ремаппинг словарей всё равно произведён (он делается ДО callback).
  - Тип: edge case.

- **test_cleanup_uses_current_session_id_after_remap_then_exception** — `finally` чистит ключ по `current_session_id` (актуальный после ремаппинга), даже когда `result` не сформирован.
  - Вход (вариант A — голое исключение): `send_message(session_id="_new_x", backend=BackendName.CLAUDE)`. MockBackend сначала возвращает `system` (session_id="real-y") — `_process_events` делает ремаппинг через `update_session_id`, tracking-обёртка обновляет `current_session_id="real-y"`. Затем `read_events` бросает `RuntimeError("synthetic")` — `result` не создаётся вовсе, исключение пробрасывается наружу. Вариант B (`/stop` после ремаппинга): тот же ремаппинг, затем task B вызывает `stop_process("real-y", BackendName.CLAUDE)`, в `_process_events` поднимается `ProcessStoppedError`, `result` не создан.
  - Ожидаемый результат для обоих вариантов: после `finally` `_busy_flags[("real-y", BackendName.CLAUDE)] == False` (cleanup сработал по новому ключу), `_stop_events` не содержит `("real-y", BackendName.CLAUDE)`, ключ `("_new_x", BackendName.CLAUDE)` отсутствует во всех словарях (был ремапнут). Никакого `UnboundLocalError`/`NameError` в `finally`. Зависших `_busy_flags=True` нет ни по одному из двух ключей.
  - Тип: edge case.

### Резюме тест-плана

Все обязательные требования из задания покрыты:
- Базовый цикл send_message → receive response: `test_send_message_success_basic_flow`, `test_send_message_with_codex_backend`.
- /stop в трёх сценариях: `test_stop_during_normal_processing`, `test_stop_during_retry_loop_same_event_persists`, `test_stop_during_restart_process_aborts_retry`, `test_stop_after_session_id_change_finds_remapped_event`.
- Event-singleton (один `asyncio.Event` на весь turn, не пересоздаётся в `_restart_process`): `test_stop_during_retry_loop_same_event_persists` (ассерт `is`-identity), `test_stop_during_restart_process_aborts_retry` (ассерт race-устойчивости).
- Retry при падении процесса: `test_retry_on_process_crash`, `test_retry_on_turn_failed_codex`, `test_all_retries_exhausted_returns_error_with_text`.
- update_session_id с проверкой ремаппинга всех словарей: `test_update_session_id_remaps_all_three_dicts`.
- Конкурентный доступ через `_busy_lock`: `test_concurrent_send_message_for_same_key_raises`.
- send_message с backend=CLAUDE и backend=CODEX (раздельные словари): `test_send_message_with_codex_backend`, `test_same_session_id_different_backends_separate_state`.
- Захват backend на старте (по аналогии с WORKING_DIR): `test_send_message_captures_backend_at_start`, `test_send_message_rejects_missing_backend`.
- Использование `backend.get_stop_strategy()` вместо `process.terminate()`: `test_stop_uses_claude_strategy_sigterm_then_sigkill`, `test_stop_uses_codex_strategy_sigint_then_sigterm_then_sigkill`.
- session_id_callback с тремя аргументами и атомарным ремаппингом: `test_session_id_callback_called_once_with_backend`, `test_update_session_id_remaps_all_three_dicts`.
- SendResult.backend: проверяется во всех тестах через ассерт `result.backend == BackendName.X`.
- Прямая адресация callback'ов через chat_id: проверяется в спеке `bot.py` (где формируются замыкания); здесь тестируется тип сигнатуры (callback не принимает chat_id).
- Stale session_id и его обработка: `test_stop_after_session_id_change_finds_remapped_event`, `test_session_id_callback_called_once_with_backend`.
- Cleanup в `finally` без зависимости от `result`: `test_cleanup_uses_current_session_id_after_remap_then_exception`.

Контрактных интеграционных тестов с реальными CLI на уровне `process_manager` нет — они находятся в спеках бэкендов (`claude_code_backend_spec.md`, `codex_backend_spec.md`). Этот модуль тестируется на моках бэкендов.
