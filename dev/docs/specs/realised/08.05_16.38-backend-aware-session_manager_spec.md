# Спецификация модуля: session_manager (backend-aware версия)

Дата: 06-05-2026
Слой: 2 (зависит от слоёв 0–1)
Файл: `src/claude_manager/session_manager.py`

**Связанные спеки:**
- `coding_agent_backend_spec.md` — абстрактный интерфейс CLI-бэкенда (источник `BackendName`); раздел «session_manager» (строки 653–678) фиксирует контракт `ActiveSession`, формат файла и обязательную миграцию
- `current_backend_registry_spec.md` — глобальный регистр текущего backend; используется *внешними* потребителями (bot.py) для подстановки backend в `create_new_session`, но НЕ импортируется напрямую этим модулем
- `realised/daily_session_registry_spec.md` — старая спека дневной нумерации; в backend-aware пакете расширяется до `register_session(session_id, backend)`, `lookup_by_number(day_number) -> DailySessionEntry | None`, `get_backend_for_session(session_id) -> BackendName | None`
- `realised/session_reader_spec.md` — чтение видимых сессий с диска (для поиска при `/N` и для превью)
- `realised/session_manager_spec.md` — старая Claude-only версия, которая заменяется этой спекой

## Назначение

Хранит связку «Telegram-чат ↔ активная сессия» вместе с **именем backend-а** (Claude Code или Codex CLI), которому эта сессия принадлежит. Бот теперь умеет одновременно вести сессии в обоих CLI; чтобы команды `/N`, `/stop`, watcher и обработчик сообщений знали, какой бинарник запускать с `--resume`/`resume`, в каждой привязке сохраняется не голый `session_id`, а dataclass `ActiveSession(session_id, backend)`. Привязки сохраняются на диск в файл `sessions.json` под рабочей директорией текущего проекта; загрузка устойчива к транзиентным OS-ошибкам (EDEADLK на macOS), запись атомарна (write-rename), запись запрещена до успешной загрузки (защита от затирания валидного файла пустым словарём при первом запуске после переключения проекта).

Модуль продолжает обслуживать два состояния пользователя: «привязан к конкретной сессии» (есть запись `ActiveSession` для chat_id) и «режим мониторинга `/all`» (записи нет — chat_id не присутствует в словаре `_bindings`). Никакого нового третьего состояния backend-awareness не добавляет — backend хранится **внутри** записи о сессии, а не как отдельный режим.

## Расхождение с предыдущей Claude-only версией

В `realised/session_manager_spec.md` (от 30-03-2026, до внедрения двух-бэкендной архитектуры) хранится только `_bindings: dict[int, str]` — chat_id → строка session_id. Файл `sessions.json` имеет схему `{"chat_id": "uuid"}`. API: `bind_session`, `unbind_session`, `get_bound_session`, `get_chat_id_for_session`. Эта спека вводит четыре изменения, прямо вытекающие из решения «бот ведёт сессии в обоих CLI»:

1. **Тип значения меняется со `str` на `ActiveSession(session_id: str, backend: BackendName)`** — `frozen=True` dataclass, иммутабельный (никто из потребителей не должен мутировать поля привязки в обход модуля).
2. **Основные имена публичных функций обновляются** под backend-aware терминологию: `bind_session` → `set_active_session(chat_id, session_id, backend)`, `unbind_session` → `clear_active_session(chat_id)`, `get_bound_session` → `get_active_session(chat_id) -> ActiveSession | None`. Дополнительно вводится тонкий wrapper `get_active_session_id(chat_id) -> str | None` для тех вызывающих мест, где backend не нужен (например, проверка «есть ли вообще привязка»). `get_chat_id_for_session` переименовывается в `find_chat_by_session_id` — это имя явно говорит «обратный поиск владельца сессии», что согласуется с потребителями (watcher вызывает его для адресации уведомления владельцу).
3. **Формат файла `sessions.json` меняется** на `{"chat_id": {"session_id": "uuid", "backend": "claude" | "codex"}}`. При загрузке выполняется ленивая миграция: записи в старом формате (значение — строка, не объект) детектируются и конвертируются в `ActiveSession(session_id=строка, backend=BackendName.CLAUDE)`. Дефолт `CLAUDE` обоснован тем, что до внедрения backend-аware архитектуры в проекте существовал только Claude — все старые записи неизбежно принадлежат Claude-сессиям.
4. **`update_session_id(chat_id, old_session_id, new_session_id)` сохраняет backend без изменений.** Сценарий — замена временного `_new_XXXX` на реальный UUID, который CLI выдал в первом `system.init`/`session.created` событии. Backend сессии в этот момент не меняется (CLI не может «передать» свою сессию другому CLI), поэтому `_bindings[chat_id] = ActiveSession(session_id=new_session_id, backend=existing_backend)` — берём существующий backend из текущей записи.

Новые потребители используют backend-aware имена. Старые имена остаются как compatibility wrappers: `bind_session`, `unbind_session`, `get_bound_session` и `get_chat_id_for_session` делегируют в новый API с безопасным Claude-default там, где старый контракт не умел передавать backend. Это сохраняет обратную совместимость старых тестов и legacy-вызовов, но не является рекомендуемым API для нового кода.

Для пользователя миграция бесшовна: один раз при первом запуске новой версии бот прочитает старый `sessions.json`, превратит каждую запись в `ActiveSession(uuid, BackendName.CLAUDE)`, при первой же операции записи перепишет файл в новом формате. Никаких ручных действий не требуется.

## Обслуживаемые сценарии

- **CJM-02: Отправка текстового сообщения** — `bot.py` вызывает `get_active_session(chat_id)`, чтобы узнать `(session_id, backend)` текущей привязки. Backend передаётся в `process_manager.send_message(..., backend=backend)` для запуска правильного subprocess (Claude CLI или Codex CLI). Если `get_active_session` вернул `None` — пользователь в режиме `/all`, бот отвечает «нет активной сессии».
- **CJM-03: Отправка фотографии или файла** — то же, что CJM-02; backend нужен для составления правильных subprocess args (Codex и Claude по-разному принимают пути к файлам).
- **CJM-04: Создание новой сессии (/new)** — `bot.py` определяет backend через `current_backend_registry.get_current()`, передаёт в `create_new_session(chat_id, backend)`. Модуль генерирует временный `_new_XXXX`, записывает `ActiveSession(temp_id, backend)`, регистрирует в дневном реестре через `daily_session_registry.register_session(temp_id, backend)`, возвращает `NewSessionResult(session_id=temp_id, day_number=N)`.
- **CJM-05: Просмотр списка сессий (/sessions)** — `bot.py` для каждой сессии в списке выясняет, к какой привязан текущий чат (метить «текущую» в выводе). Использует `get_active_session_id(chat_id)` (backend не нужен — нужен только session_id для сравнения).
- **CJM-06: Переключение на сессию (/N)** — `switch_to_session(chat_id, day_number)`. Алгоритм: ищет в дневном реестре через `daily_session_registry.lookup_by_number(day_number) -> DailySessionEntry | None`. Если entry найден — backend известен из `entry.backend`, привязка создаётся как `ActiveSession(entry.session_id, entry.backend)`. Если не найден в реестре — fallback в видимые сессии через `_find_session_among_visible(day_number)`, который для каждой найденной сессии берёт backend из контекста (текущий backend project-а — Claude, потому что `session_reader` пока сканирует только Claude-папки; полная backend-aware enumeration приходит позже с переписанным `session_reader`).
- **CJM-07: Мониторинг всех сессий (/all)** — `clear_active_session(chat_id)` переводит чат в режим мониторинга, удаляет запись из `_bindings`, переписывает файл. Watcher (фоновый наблюдатель за сессиями) при доставке уведомления вызывает `find_chat_by_session_id(session_id, backend) -> int | None`, чтобы определить владельца — поиск идёт по паре `(session_id, backend)`, согласован с архитектурой проекта (см. описание метода в разделе «Публичный API»).

## Публичный API

### `@dataclass(frozen=True) class ActiveSession`

Иммутабельная пара «session_id + backend». Используется как тип значения во внутреннем словаре привязок и как возвращаемый тип `get_active_session`.

**Поля:**
- `session_id: str` — идентификатор сессии. Может быть либо реальным UUID, который выдал CLI (например, `"eb5ac5bc-2ac6-45ad-8ca9-bb3a1a741f1e"`), либо временным идентификатором вида `"_new_<12 hex>"` (например, `"_new_a1b2c3d4e5f6"`), сгенерированным `_generate_temp_session_id()` для свежесозданных сессий до получения первого ответа от CLI. Тип строго `str`, не `Optional` — пустой `session_id` запрещён.
- `backend: BackendName` — имя CLI-бэкенда, которому принадлежит эта сессия. Импортируется из `claude_manager.coding_agent_backend`. Допустимые значения — все варианты enum (`BackendName.CLAUDE`, `BackendName.CODEX`).

**Иммутабельность:** `frozen=True` запрещает мутацию полей после создания. Если потребителю нужно «обновить session_id» — он создаёт новый объект `ActiveSession(new_id, old.backend)`, не правит существующий. Это страховка от случайной мутации внутреннего состояния модуля кодом снаружи (получивший `ActiveSession` из `get_active_session` не сможет изменить хранимую привязку, не вызвав публичный API).

### `@dataclass class SwitchResult`

Результат переключения чата на сессию по дневному номеру через `switch_to_session`. Не `frozen` — это transient DTO, возвращается одним вызовом и сразу обрабатывается, иммутабельность не даёт практической пользы.

**Поля:**
- `found: bool` — `True`, если сессия с указанным `day_number` найдена (либо в дневном реестре, либо среди видимых сессий на диске); `False` — если не найдена нигде.
- `session_id: str` — идентификатор найденной сессии или пустая строка `""`, если `found == False`.
- `backend: BackendName` — backend найденной сессии или `BackendName.CLAUDE` (бессмысленный заполнитель), если `found == False`. Потребитель обязан проверять `found` перед чтением `backend`.
- `day_number: int` — номер сессии, как был передан в `switch_to_session` (повторяется в результате для удобства логирования и формирования сообщения пользователю).
- `preview: str` — превью первого сообщения сессии (до 120 символов), полученное через `session_reader`. Пустая строка `""`, если превью недоступно или сессия не найдена.

### `@dataclass class NewSessionResult`

Результат создания новой сессии через `create_new_session`. Не `frozen` (см. рассуждение для `SwitchResult`).

**Поля:**
- `session_id: str` — временный идентификатор новой сессии (всегда вида `"_new_<12 hex>"`).
- `backend: BackendName` — backend новой сессии (тот, что был передан в `create_new_session`).
- `day_number: int` — присвоенный дневной номер (например, `4`).

### `async def set_active_session(chat_id: int, session_id: str, backend: BackendName) -> int`

Привязывает Telegram-чат к сессии конкретного backend-а. Если чат уже был привязан к другой сессии — перезаписывает. Регистрирует сессию в дневном реестре, сохраняет привязку на диск. Это backend-aware замена старого `bind_session(chat_id, session_id)`.

**Аргументы:**
- `chat_id` (`int`) — идентификатор Telegram-чата (например, `123456789`).
- `session_id` (`str`) — идентификатор сессии (UUID или `_new_XXXX`). Не должен быть пустой строкой; нарушение этого инварианта на уровне модуля не проверяется (тип-подсказка и контракт документации считаются достаточными), но валидный `session_id` обязан передать вызывающий.
- `backend` (`BackendName`) — backend сессии. Передаётся вызывающим явно. Для команды `/new` — обычно `current_backend_registry.get_current()` (значение определяется в `bot.py`, не здесь). Для команды `/N` — `entry.backend` из `daily_session_registry.lookup_by_number(N)`.

**Возвращает:** дневной номер сессии (`int`), полученный от `daily_session_registry.register_session(session_id, backend)`. Например, `3` означает третью сессию за сегодня.

**Исключения:**
- `OSError` — если запись `sessions.json` на диск провалилась (диск переполнен, права отозваны, raid disconnect и т.п.). Состояние `_bindings` в памяти при этом ОБНОВЛЕНО (запись добавлена) — атомарного rollback памяти этот модуль не делает; вызывающий должен либо ретраить, либо смириться с временным дрейфом «память впереди диска» до следующей успешной записи. В отличие от `current_backend_registry`, где «диск + память либо вместе, либо никак», здесь главное — не потерять привязку, поэтому память остаётся обновлённой.
- `RuntimeError` — если `_bindings_path is None` (`load_bindings` не был вызван). Запись отказывается работать до загрузки. Это страховка от затирания валидного файла на диске пустым состоянием.

### `def get_active_session(chat_id: int) -> ActiveSession | None`

Возвращает активную сессию (`ActiveSession`) привязанного чата или `None`, если чат в режиме `/all`. Синхронная функция, не делает I/O (читает только in-memory словарь).

**Аргументы:**
- `chat_id` (`int`) — идентификатор Telegram-чата.

**Возвращает:** `ActiveSession(session_id, backend)`, если чат привязан; `None`, если в режиме `/all`. Возвращаемый объект — иммутабельный (`frozen=True`), мутировать его нельзя.

**Исключения:** не выбрасывает.

### `def get_active_session_id(chat_id: int) -> str | None`

Возвращает `session_id` привязанной сессии без backend-а. Тонкий wrapper вокруг `get_active_session`, упрощающий код вызывающих, которым нужно только сравнить идентификатор (например, в `bot.py` при формировании списка `/sessions` — пометить «текущую» сессию).

**Аргументы:**
- `chat_id` (`int`) — идентификатор Telegram-чата.

**Возвращает:** `session_id` (`str`) или `None`, если чат не привязан.

**Исключения:** не выбрасывает.

### `async def clear_active_session(chat_id: int) -> None`

Отвязывает чат от сессии (переводит в режим `/all` мониторинга). Если чат не был привязан — ничего не делает (идемпотентность). Сохраняет состояние на диск. Backend-aware замена старого `unbind_session(chat_id)`.

**Аргументы:**
- `chat_id` (`int`) — идентификатор Telegram-чата.

**Возвращает:** `None`.

**Исключения:**
- `OSError` — не удалось записать `sessions.json` (см. подробности у `set_active_session`).
- `RuntimeError` — `_bindings_path is None`.

### `def find_chat_by_session_id(session_id: str, backend: BackendName) -> int | None`

Обратный поиск: по паре `(session_id, backend)` находит `chat_id` владельца. Используется watcher-ом для адресации уведомлений (только владельцу сессии, не всем пользователям из `ALLOWED_USER_IDS`) — каждый watcher запущен под свой backend и передаёт `self.backend.name` в этот вызов. Backend-aware замена старого `get_chat_id_for_session(session_id)`.

Поиск идёт **по паре**, а не по одному `session_id`. Обоснование:

1. **Согласованность с архитектурой проекта.** Состояние во всём проекте ключуется парой `(session_id, backend)` — `process_manager._processes`/`_busy_flags`/`_stop_events`, `unread_buffer`, инстанции `SessionWatcher` (по одной на бэкенд). `find_chat_by_session_id` без `backend` был бы единственным исключением и усложнял ментальную модель.
2. **Семантическая точность.** Контракт владения сессией формально привязан к паре (см. `coding_agent_backend_spec.md` → раздел «Контракт владения сессией»). Поиск владельца только по `session_id` означал бы «при коллизии отдадим первого попавшегося»; поиск по паре — детерминированный.
3. **Защита от не-UUID идентификаторов.** Сейчас `session_id` — UUID4, и физическая коллизия между Claude и Codex имеет вероятность ≈2⁻¹²⁸ (астрономически невозможна). Но если в будущем формат идентификаторов изменится (например, один из бэкендов перейдёт на детерминированные ID), backend-фильтр заранее закрывает этот класс багов.

Стоимость добавления параметра — нулевая: структура `_bindings: dict[chat_id, ActiveSession]` уже хранит `backend` в DTO `ActiveSession`. В алгоритме итерация по `_bindings.items()` просто получает дополнительное условие `and active.backend == backend` к существующему сравнению.

**Аргументы:**
- `session_id` (`str`) — идентификатор сессии (UUID или `_new_XXXX`).
- `backend` (`BackendName`) — backend, к которому относится сессия. Передаётся вызывающим (watcher знает свой backend по `self.backend.name`).

**Возвращает:** `chat_id` (`int`) владельца, либо `None`, если **пара** `(session_id, backend)` не привязана ни к одному чату. Совпадение только по `session_id` (без совпадения backend) трактуется как «не найдено».

**Исключения:** не выбрасывает.

### Compatibility wrappers старого API

Эти функции сохранены только для обратной совместимости и старых тестов. Новый код должен использовать backend-aware API выше.

- `async bind_session(chat_id: int, session_id: str) -> int` — вызывает `set_active_session(chat_id, session_id, BackendName.CLAUDE)`.
- `async unbind_session(chat_id: int) -> None` — вызывает `clear_active_session(chat_id)`.
- `def get_bound_session(chat_id: int) -> str | None` — вызывает `get_active_session_id(chat_id)`.
- `def get_chat_id_for_session(session_id: str) -> int | None` — legacy-обратный поиск только для Claude; вызывает `find_chat_by_session_id(session_id, BackendName.CLAUDE)`.

Wrapper-ы не должны использоваться для Codex-сессий: в старом API нет места для backend, поэтому он не может безопасно адресовать не-Claude сессии.

### `def is_monitoring_mode(chat_id: int) -> bool`

Проверяет, находится ли чат в режиме `/all` (мониторинг без привязки к сессии). Эквивалент `get_active_session(chat_id) is None`.

**Аргументы:**
- `chat_id` (`int`) — идентификатор Telegram-чата.

**Возвращает:** `True`, если чат в режиме мониторинга (`/all`); `False`, если привязан к конкретной сессии.

**Исключения:** не выбрасывает.

### `async def switch_to_session(chat_id: int, day_number: int) -> SwitchResult`

Переключает чат на сессию по дневному номеру. Сначала ищет в дневном реестре через `daily_session_registry.lookup_by_number(day_number) -> DailySessionEntry | None`. Если найден — backend известен из `entry.backend`, привязка создаётся с этим backend-ом. Если не найден — fallback в видимые сессии через `_find_session_among_visible(day_number)`.

**Аргументы:**
- `chat_id` (`int`) — идентификатор Telegram-чата.
- `day_number` (`int`) — дневной номер сессии (например, `3` для команды `/3`).

**Возвращает:** `SwitchResult` с полями `found`, `session_id`, `backend`, `day_number`, `preview`. При `found == False` поля `session_id` и `preview` — пустые строки, `backend` — заполнитель (потребитель обязан сначала проверить `found`).

**Исключения:**
- `OSError` — при ошибке записи `sessions.json` через `set_active_session`.
- `RuntimeError` — `_bindings_path is None`.

### `async def update_session_id(chat_id: int, old_session_id: str, new_session_id: str) -> None`

Обновляет `session_id` в привязке чата и в дневном реестре. Используется в сквозном механизме «temp → real» — когда CLI возвращает реальный UUID вместо временного `_new_XXXX` в первом событии (`system.init` для Claude или `session.created` для Codex).

**Backend сессии при этом сохраняется** — `_bindings[chat_id] = ActiveSession(new_session_id, current_active.backend)`. Это инвариант: temp и real принадлежат одному CLI, потому что temp создаётся по команде `/new` через `create_new_session(chat_id, backend)`, а real — это первое уведомление того же CLI о том, какой UUID он присвоил своей сессии.

**Аргументы:**
- `chat_id` (`int`) — идентификатор Telegram-чата, привязанного к обновляемой сессии.
- `old_session_id` (`str`) — текущий (временный) идентификатор сессии (например, `"_new_a1b2c3d4e5f6"`).
- `new_session_id` (`str`) — новый (реальный) идентификатор сессии от CLI.

**Возвращает:** `None`.

**Исключения:**
- `OSError` — при ошибке записи `sessions.json`.
- `RuntimeError` — `_bindings_path is None`.

**Примечание о граничном случае.** Если чат не привязан к `old_session_id` (например, пользователь успел переключиться на другую сессию между отправкой `/new` и приходом первого ответа от CLI) — привязка чата НЕ обновляется (нечего обновлять), но `daily_session_registry.update_session_id(old, new)` вызывается всегда (реестр должен знать новый UUID, иначе watcher и `/sessions` будут видеть «дубль»). Лог уровня `debug`.

### `async def create_new_session(chat_id: int, backend: BackendName) -> NewSessionResult`

Создаёт новую сессию: генерирует временный идентификатор, регистрирует в дневном реестре с указанным backend-ом, привязывает чат к новой сессии. Возвращает `NewSessionResult` с информацией о созданной сессии (включая backend — для удобства потребителя).

**Аргументы:**
- `chat_id` (`int`) — идентификатор Telegram-чата.
- `backend` (`BackendName`) — backend новой сессии. Передаётся вызывающим явно. В нормальном сценарии `bot.py` определяет backend через `current_backend_registry.get_current()` и передаёт сюда. Этот модуль НЕ импортирует `current_backend_registry` напрямую — иначе зависимости разрастаются ради удобства, и тесты `session_manager` приходится мокать с лишним модулем.

**Возвращает:** `NewSessionResult(session_id=temp_id, backend=backend, day_number=N)`, где `temp_id` имеет формат `"_new_<12 hex>"`.

**Исключения:**
- `OSError` — при ошибке записи `sessions.json` (через `set_active_session`).
- `RuntimeError` — `_bindings_path is None`.

### `async def load_bindings() -> None`

Загружает привязки из `sessions.json` в память при запуске бота. Также инициирует загрузку дневного реестра (`daily_session_registry.load_registry()`). Устанавливает `_bindings_path` — после успешного вызова запись разрешена (`set_active_session`, `clear_active_session`, etc. перестают выбрасывать `RuntimeError`).

При чтении файла применяет ленивую миграцию: записи в старом формате (значение — строка вместо объекта) детектируются и конвертируются в `ActiveSession(session_id=строка, backend=BackendName.CLAUDE)`. Запись в файл в новом формате произойдёт при первой же модифицирующей операции (`set_active_session`, `clear_active_session`, `update_session_id`).

Устойчива к транзиентным OS-ошибкам (`EDEADLK` errno 11 на macOS) — выполняет до `LOAD_RETRY_COUNT` попыток с экспоненциальным backoff (база `LOAD_RETRY_BACKOFF_BASE_SECONDS`).

**Аргументы:** нет.

**Возвращает:** `None`.

**Исключения:** не выбрасывает в нормальных сценариях. Все ошибки чтения логируются:
- Файл не существует — `info`, `_bindings = {}`, `_bindings_path` установлен (запись разрешена).
- Файл повреждён (невалидный JSON или неожиданная структура) — `warning`, `_bindings = {}`, `_bindings_path` установлен.
- Все `LOAD_RETRY_COUNT` попыток упали с транзиентным `OSError` — `error`, `_bindings = {}`, `_bindings_path` установлен. Это компромисс: оставлять `_bindings_path = None` (как `current_backend_registry`) защитило бы от затирания, но навсегда заблокировало бы `set_active_session` до перезапуска бота. Здесь приоритет — пользователь должен иметь возможность создать новую привязку в новом проекте, даже если старый файл нечитаем (и согласован с тем, как реагирует старая версия).

Если будущая надёжность потребует «не затирать», флаг `_loaded_from_disk: bool` (как в `silence_mode_registry`) — путь миграции (см. также раздел «Обработка ошибок → Дополнительные решения»).

### `def get_all_bindings() -> dict[int, ActiveSession]`

Возвращает копию всех текущих привязок. Нужна `session_watcher` для определения, какие сессии сейчас активно используются (для приоритизации опроса) и какому chat_id принадлежит каждая сессия.

**Аргументы:** нет.

**Возвращает:** словарь `{chat_id: ActiveSession}`. Чаты в режиме `/all` НЕ включаются (отсутствие `chat_id` в словаре = режим мониторинга). Если привязок нет — пустой словарь `{}`. Возвращается *копия* (`dict(_bindings)`), чтобы внешний код не мог изменить внутреннее состояние; `ActiveSession` сами по себе иммутабельны.

**Исключения:** не выбрасывает.

### `async def reset_state() -> None`

Сбрасывает все привязки и путь к файлу, перезагружает данные из нового `config.WORKING_DIR`. Используется при переключении проекта (команда `/projects` / `/pN` в `bot.py`).

**Контракт с переключением проекта.** Перед вызовом `reset_state` watcher уже приостановлен через `session_watcher.pause_all()`, и `config.WORKING_DIR` уже изменён на путь нового проекта. Этот метод просто очищает in-memory состояние и инициирует повторный `load_bindings`, который пересчитает `_bindings_path` из текущего `config.WORKING_DIR`. После возврата — состояние согласовано с новым проектом.

**Аргументы:** нет.

**Возвращает:** `None`.

**Исключения:** не выбрасывает (ошибки внутри `load_bindings` обрабатываются там и в исключения не превращаются).

## Внутренние функции

### `def _generate_temp_session_id() -> str`

Генерирует уникальный временный идентификатор сессии вида `"_new_<12 hex chars>"`, например `"_new_a1b2c3d4e5f6"`. Использует первые 12 символов hex-представления `uuid.uuid4()` — этого достаточно для уникальности в рамках одного процесса бота (вероятность коллизии 12-символьного hex — порядка 2⁻⁴⁸, что для нескольких десятков активных сессий за день пренебрежимо мало).

**Аргументы:** нет.

**Возвращает:** строку вида `"_new_a1b2c3d4e5f6"`.

### `async def _save_bindings() -> None`

Атомарно записывает `_bindings` в `sessions.json` через паттерн «tmp + os.replace». Конвертирует in-memory словарь `{chat_id: ActiveSession}` в JSON-сериализуемый словарь `{str(chat_id): {"session_id": session.session_id, "backend": session.backend.value}}`. Перед записью проверяет `_bindings_path is not None` — если путь не задан, выбрасывает `RuntimeError` (защита от затирания валидного файла на диске пустым словарём при первом запуске после переключения проекта, когда `set_active_session` мог бы быть вызван до `load_bindings`).

**Аргументы:** нет.

**Возвращает:** `None`.

**Исключения:**
- `RuntimeError` — `_bindings_path is None` (load_bindings не вызывался). Сообщение: `"Путь к файлу привязок не задан — вызовите load_bindings() перед записью"`.
- `OSError` — от `Path.write_text` или `os.replace`. Пробрасывается наружу для обработки вызывающим (`set_active_session`, `clear_active_session` и т.п. — они логируют и пробрасывают дальше; финальный обработчик — `bot.py`, который сообщает пользователю в Telegram).

### `async def _find_session_among_visible(day_number: int) -> tuple[str, BackendName, str] | None`

Ищет сессию по дневному номеру среди всех видимых сессий (через `session_reader.get_recent_sessions(config.WORKING_DIR)`), когда дневной реестр не содержит этот номер. Регистрирует найденные сессии в дневном реестре через `daily_session_registry.register_session(session_id, backend)` и проверяет, совпадает ли присвоенный номер с запрашиваемым.

**Backend для регистрации.** На текущей итерации `session_reader.get_recent_sessions` возвращает только Claude-сессии (он сканирует `~/.claude/projects/`, не Codex-директории). Поэтому в этом fallback-пути backend всегда `BackendName.CLAUDE`. Когда `session_reader` будет обновлён до backend-aware версии (отдельная задача из пакета backend) — функция получит backend от `session_reader` напрямую (через расширенный `SessionInfo.backend`); до этого жёсткий `CLAUDE` — сознательное допущение.

**Аргументы:**
- `day_number` (`int`) — дневной номер сессии.

**Возвращает:** кортеж `(session_id, backend, preview)` если сессия найдена, или `None` если нет.

### `async def _get_preview_for_session(session_id: str) -> str`

Получает превью первого сообщения сессии из списка видимых сессий (`session_reader.get_recent_sessions`). Используется в `switch_to_session` после нахождения сессии в дневном реестре — реестр хранит только UUID и backend, не превью.

**Аргументы:**
- `session_id` (`str`) — идентификатор сессии.

**Возвращает:** превью (`str`, до 120 символов) или пустую строку `""`, если сессия не найдена в списке видимых.

## Алгоритм работы

### set_active_session

1. Захватить `_lock` (asyncio.Lock — защита от параллельной записи в `_bindings` и в файл).
2. Сформировать запись: `active = ActiveSession(session_id=session_id, backend=backend)`.
3. Записать в словарь: `_bindings[chat_id] = active`.
4. Зарегистрировать сессию в дневном реестре: `day_number = await daily_session_registry.register_session(session_id, backend)`. Это присвоит дневной номер (или вернёт существующий, если сессия уже зарегистрирована).
5. Вызвать `await _save_bindings()` для записи на диск.
6. Залогировать `info`: `"Чат %d привязан к сессии %s (#%d, backend=%s)"` с подстановкой значений.
7. Освободить `_lock` (через `async with`, автоматически).
8. Вернуть `day_number`.

### get_active_session

1. Вернуть `_bindings.get(chat_id)` — `ActiveSession`-объект или `None`, если ключа нет. Без захвата `_lock` (это чтение из dict — атомарная операция Python, гонок не возникает).

### get_active_session_id

1. Получить активную запись: `active = get_active_session(chat_id)`.
2. Если `active is None` — вернуть `None`.
3. Иначе — вернуть `active.session_id`.

### clear_active_session

1. Захватить `_lock`.
2. Удалить ключ: `_bindings.pop(chat_id, None)` — `pop` с default не вызывает `KeyError`, если ключа нет (идемпотентность).
3. Вызвать `await _save_bindings()` для записи на диск.
4. Залогировать `info`: `"Чат %d переведён в режим мониторинга (/all)"`.
5. Освободить `_lock`.

### find_chat_by_session_id

1. Итерация по `_bindings.items()`: для каждой пары `(chat_id, active)` проверить **обе** части: `active.session_id == session_id and active.backend == backend`. Поиск по паре — соответствие только по `session_id` без совпадения backend трактуется как «не та сессия».
2. Если найдена — вернуть `chat_id`.
3. Если ни одна не подошла — вернуть `None`. Без захвата `_lock` (итерация по dict в чтении — потокобезопасна для asyncio при условии, что нет одновременных модификаций; модификации захватывают `_lock`, но чтение под Lock не помещаем — это ухудшит производительность watcher-а, который вызывает функцию каждые 2 секунды).

### is_monitoring_mode

1. Вернуть `chat_id not in _bindings`.

### switch_to_session

1. **Поиск в дневном реестре.** Вызвать `entry = await daily_session_registry.lookup_by_number(day_number)` — возвращает `DailySessionEntry(session_id, backend) | None`.
2. **Если найден в реестре** — `entry.backend` известен. Получить превью: `preview = await _get_preview_for_session(entry.session_id)`. Привязать чат: `await set_active_session(chat_id, entry.session_id, entry.backend)`. Вернуть `SwitchResult(found=True, session_id=entry.session_id, backend=entry.backend, day_number=day_number, preview=preview)`.
3. **Если не найден в реестре** — fallback: `visible_result = await _find_session_among_visible(day_number)`.
4. **Если найдена среди видимых** — `(found_session_id, found_backend, found_preview) = visible_result`. Привязать: `await set_active_session(chat_id, found_session_id, found_backend)`. Вернуть `SwitchResult(found=True, session_id=found_session_id, backend=found_backend, day_number=day_number, preview=found_preview)`.
5. **Сессия не найдена нигде** — вернуть `SwitchResult(found=False, session_id="", backend=BackendName.CLAUDE, day_number=day_number, preview="")`. Заполнитель `BackendName.CLAUDE` для `backend` — потребитель обязан проверить `found` перед чтением `backend`.

### _find_session_among_visible

1. Получить список видимых сессий: `sessions = await session_reader.get_recent_sessions(config.WORKING_DIR)`.
2. Для каждой `session` из списка зарегистрировать в дневном реестре: `number = await daily_session_registry.register_session(session.session_id, BackendName.CLAUDE)` (на текущей итерации `session_reader` возвращает только Claude-сессии — см. примечание в описании функции).
3. Если `number == day_number` — вернуть `(session.session_id, BackendName.CLAUDE, session.preview)`.
4. Если ни одна сессия не получила нужный номер — вернуть `None`.

### update_session_id

1. Захватить `_lock`.
2. Получить текущую запись: `current_active = _bindings.get(chat_id)`.
3. **Если** `current_active is not None` **и** `current_active.session_id == old_session_id` — обновить привязку: `_bindings[chat_id] = ActiveSession(session_id=new_session_id, backend=current_active.backend)`. Backend из существующей записи — temp и real принадлежат одному backend-у.
4. **Иначе** — залогировать `debug`: `"Чат %d не привязан к сессии %s — привязка не обновлена"` (нормальная ситуация, если пользователь переключился между `/new` и приходом первого ответа CLI).
5. Вызвать `await daily_session_registry.update_session_id(old_session_id, new_session_id)` — реестр обновляется ВСЕГДА (даже если привязка чата не обновлена), потому что иначе watcher и `/sessions` увидят «дубль» (старый UUID и новый UUID одновременно).
6. Вызвать `await _save_bindings()`.
7. Залогировать `info`: `"Session ID обновлён в привязках: %s → %s"`.
8. Освободить `_lock`.

### create_new_session

1. Сгенерировать временный ID: `temp_session_id = _generate_temp_session_id()` (синхронно, не требует Lock — функция чистая).
2. Вызвать `day_number = await set_active_session(chat_id, temp_session_id, backend)`. `set_active_session` сам захватывает Lock, регистрирует в реестре с переданным backend, сохраняет на диск.
3. Залогировать `info`: `"Создана новая сессия %s (#%d) для чата %d, backend=%s"`.
4. Вернуть `NewSessionResult(session_id=temp_session_id, backend=backend, day_number=day_number)`.

### load_bindings

1. Установить путь: `_bindings_path = Path(config.WORKING_DIR) / BINDINGS_FILENAME` (модифицирует module-level переменную через `global _bindings_path` или эквивалентную конструкцию).
2. Инициировать загрузку дневного реестра: `await daily_session_registry.load_registry()` (реестр — отдельный модуль; его загрузка координируется отсюда, потому что оба используются вместе при старте бота).
3. Прочитать файл с retry: цикл по `attempt in range(1, LOAD_RETRY_COUNT + 1)`:
   - `try: content = await asyncio.to_thread(_bindings_path.read_text, "utf-8")`.
   - `raw_data = json.loads(content)`. Прервать цикл (`break`) при успехе.
   - `except FileNotFoundError`: `_bindings = {}`, лог `info` `"Файл привязок не найден, начинаю с чистого состояния"`, `return` (выход из функции, не из цикла).
   - `except json.JSONDecodeError`: `_bindings = {}`, лог `warning` `"Файл привязок повреждён (невалидный JSON), начинаю с чистого состояния"`, `return`.
   - `except OSError as error`: лог `warning` `"Попытка чтения привязок %d/%d: %s"`. Если `attempt < LOAD_RETRY_COUNT` — `await asyncio.sleep(LOAD_RETRY_BACKOFF_BASE_SECONDS * (2 ** (attempt - 1)))` (экспоненциальный backoff: 0.1, 0.2, 0.4, 0.8, 1.6 секунды для `LOAD_RETRY_BACKOFF_BASE_SECONDS = 0.1` и `LOAD_RETRY_COUNT = 5`). Иначе — выйти из цикла без `break`.
4. Если все попытки упали (вышли из цикла без `break` и без `return`) — `_bindings = {}`, лог `error` `"Не удалось прочитать привязки после %d попыток, начинаю с чистого состояния"`. Возврат — путь `_bindings_path` уже установлен на шаге 1, запись разрешена.
5. Распарсить `raw_data` (`dict[str, str | dict]`):
   - Инициализировать `_bindings = {}`.
   - Для каждой пары `(raw_key, raw_value)` в `raw_data.items()`:
     - Сконвертировать ключ: `try: chat_id = int(raw_key)`. При `ValueError` — лог `warning` `"Невалидный ключ chat_id в sessions.json: '%s' — пропущен"`, `continue`.
     - Распарсить значение через миграцию (см. ниже): `active = _parse_binding_value(raw_value)`. При неудаче парсинга — лог `warning`, `continue`.
     - Записать: `_bindings[chat_id] = active`.
6. Залогировать `info`: `"Загружено %d привязок из sessions.json"` с количеством.

### _parse_binding_value (вспомогательная функция миграции)

Не публичный API, но описывается отдельно для ясности алгоритма миграции. Принимает `raw_value: str | dict`, возвращает `ActiveSession`.

1. **Если** `isinstance(raw_value, str)` — старый формат (только UUID-строка). Вернуть `ActiveSession(session_id=raw_value, backend=BackendName.CLAUDE)`. Дефолт CLAUDE — обоснован тем, что до внедрения backend-aware архитектуры в проекте существовал только Claude.
2. **Если** `isinstance(raw_value, dict)` — новый формат. Извлечь `session_id = raw_value["session_id"]` (`KeyError` пробросить — это сломанная запись, должна быть отловлена вызывающим). Извлечь `raw_backend = raw_value["backend"]`. Сконвертировать в Enum: `backend = BackendName(raw_backend)` (`ValueError` пробросить, если значение не из enum, например `"gemini"`). Вернуть `ActiveSession(session_id=session_id, backend=backend)`.
3. **Иначе** (например, `raw_value` — `int`, `list`, `None`) — выбросить `ValueError("Неожиданный тип значения в sessions.json: ...")`.

Вызывающий (`load_bindings`) ловит `KeyError` и `ValueError` от этой функции в своём общем `except` для пропуска сломанной записи с логом.

### get_all_bindings

1. Вернуть `dict(_bindings)` — копию словаря. Без захвата `_lock` (см. рассуждение для `find_chat_by_session_id` про чтение под Lock).

### reset_state

1. Захватить `_lock`.
2. Очистить состояние: `_bindings = {}`, `_bindings_path = None`. Сброс пути критичен — иначе следующий `load_bindings` или `_save_bindings` использовал бы старый путь к файлу старого проекта.
3. Освободить `_lock`.
4. Вызвать `await load_bindings()` — пересчитает `_bindings_path` из текущего (уже изменённого вызывающим) `config.WORKING_DIR` и загрузит новый файл. `load_bindings` сам не использует `_lock` — он только инициализирует состояние, а не модифицирует его в гонках с другими операциями (вызывающий обязан гарантировать, что во время `reset_state` никто параллельно не делает `set_active_session` — это обеспечивается тем, что `bot.py` приостанавливает обработку сообщений на время переключения проекта; см. CLAUDE.md → «Атомарность переключения проекта»).
5. Залогировать `info`: `"Состояние session_manager сброшено и перезагружено"`.

### _save_bindings

1. **Guard «загружено ли с диска».** Если `_bindings_path is None` — выбросить `RuntimeError("Путь к файлу привязок не задан — вызовите load_bindings() перед записью")`. Никаких изменений на диске.
2. Сериализовать в новый формат: `serializable = {str(chat_id): {"session_id": active.session_id, "backend": active.backend.value} for chat_id, active in _bindings.items()}`. JSON не поддерживает числовые ключи — конвертируем `chat_id` в строки. `active.backend.value` (а не `repr` или `.name`) — `BackendName.CLAUDE.value == "claude"`, что и нужно для последующего чтения через `BackendName(raw_backend)`.
3. Сформировать JSON: `json_content = json.dumps(serializable, indent=2, ensure_ascii=False)`.
4. Атомарная запись:
   - `temp_path = _bindings_path.with_name(BINDINGS_FILENAME + BINDINGS_TEMP_SUFFIX)` (например, `sessions.json.tmp`).
   - `await asyncio.to_thread(temp_path.write_text, json_content, "utf-8")`. Запись блокирующая — выносим в поток.
   - `await asyncio.to_thread(os.replace, str(temp_path), str(_bindings_path))`. Атомарное переименование на macOS APFS — POSIX rename.

## Зависимости

**От модулей проекта:**

- **`config`** — `WORKING_DIR` (`str`/`Path`) — путь к рабочей директории проекта, используется для определения пути к `sessions.json` (`{WORKING_DIR}/sessions.json`) и передачи в `session_reader.get_recent_sessions()`.
- **`coding_agent_backend`** — `BackendName` (Enum). Импортируется для типизации поля `ActiveSession.backend`, для создания дефолтного значения при миграции (`BackendName.CLAUDE`), и для конверсии строк из JSON-файла в Enum (`BackendName(raw_backend)`). Спека `coding_agent_backend_spec.md` определяет enum как `BackendName.CLAUDE = "claude"` и `BackendName.CODEX = "codex"`. На момент написания этой спеки модуль `coding_agent_backend` ещё может быть не реализован — в этом случае реализация `session_manager` должна идти после или вместе с реализацией `coding_agent_backend`. Согласовано в порядке реализации в pipeline-state.json.
- **`daily_session_registry`** (backend-aware версия) — четыре функции:
  - `register_session(session_id: str, backend: BackendName) -> int` — регистрация сессии с указанным backend, возврат дневного номера. Расширение старой сигнатуры (раньше принимала только `session_id`).
  - `lookup_by_number(day_number: int) -> DailySessionEntry | None` — поиск по номеру; возвращает entry с обоими полями (`session_id`, `backend`), не голую строку. Замена старого `get_session_id_by_number`.
  - `update_session_id(old_session_id: str, new_session_id: str) -> None` — без изменений в backend-aware версии.
  - `load_registry() -> None` — без изменений.
- **`session_reader`** — `get_recent_sessions(working_dir) -> list[SessionInfo]` — получение списка видимых сессий для поиска при `/N` (когда нет в дневном реестре) и для превью при переключении. На текущей итерации возвращает только Claude-сессии; backend-aware enumeration придёт в отдельной задаче.

**Стандартная библиотека:**
- `asyncio` — `asyncio.Lock` (защита от параллельной записи), `asyncio.to_thread` (вынос блокирующего I/O в поток), `asyncio.sleep` (backoff между retry).
- `dataclasses` — `@dataclass(frozen=True)` для `ActiveSession`, `@dataclass` для `SwitchResult` и `NewSessionResult`.
- `json` — `json.dumps` / `json.loads` для (де)сериализации `sessions.json`.
- `logging` — `logger = logging.getLogger(__name__)` для логирования событий.
- `os` — `os.replace` (атомарное переименование).
- `pathlib.Path` — пути к файлам.
- `uuid` — `uuid.uuid4` для генерации временных идентификаторов.

**Сторонние пакеты:** нет.

## Обработка ошибок

Категории сгруппированы по поведению модуля при разных классах сбоев.

**Категория «Чтение sessions.json при `load_bindings`»**

- **Файл не существует (первый запуск).** Штатно. `_bindings = {}`, `_bindings_path` установлен (запись разрешена). Лог `info`: `"Файл привязок не найден, начинаю с чистого состояния"`.
- **Файл повреждён (невалидный JSON).** `_bindings = {}`, `_bindings_path` установлен. Лог `warning`. Повреждённый файл будет перезаписан в новом формате при первой же модифицирующей операции.
- **Транзиентный `OSError` (включая `EDEADLK` errno 11 на macOS).** До `LOAD_RETRY_COUNT = 5` попыток с экспоненциальным backoff (0.1, 0.2, 0.4, 0.8, 1.6 с). При успехе — продолжить. При исчерпании всех попыток — `_bindings = {}`, `_bindings_path` установлен, лог `error`. Это компромисс: оставлять `_bindings_path = None` навсегда заблокировало бы запись до перезапуска, что для активного пользователя хуже потенциального затирания (которое всё равно произойдёт при следующей записи). Защита `_bindings_path is not None` срабатывает только в окне между импортом модуля и первым `load_bindings`.
- **`PermissionError` при чтении.** Обрабатывается тем же путём, что и `OSError` (`PermissionError` — подкласс `OSError`). Лог `error`, `_bindings = {}`, `_bindings_path` установлен. Для production-сценария это аномалия (бот должен иметь права на свой проектный каталог), но не блокируется.

**Категория «Парсинг отдельной записи в `sessions.json`»**

- **Невалидный ключ chat_id (`int(raw_key)` бросает `ValueError`).** Запись пропускается с логом `warning` `"Невалидный ключ chat_id в sessions.json: '%s' — пропущен"`. Остальные записи загружаются нормально.
- **Значение записи не `str` и не `dict` (например, `int`, `list`, `None`).** `_parse_binding_value` бросает `ValueError`, `load_bindings` ловит и пропускает запись с логом `warning`.
- **Новый формат, но отсутствует ключ `"session_id"` или `"backend"`.** `_parse_binding_value` бросает `KeyError`, `load_bindings` ловит и пропускает запись с логом `warning` `"Сломанная запись в sessions.json для chat_id=%d: %s"`.
- **Новый формат, но `backend` — неизвестное значение (например, `"gemini"`).** `_parse_binding_value` бросает `ValueError` от `BackendName(raw_backend)`. Запись пропускается с логом `warning`. Чат окажется в режиме `/all` после загрузки — пользователь должен будет переподключиться через `/N` или `/new`.

**Категория «Запись `sessions.json` при модифицирующих операциях»**

- **`OSError` при `_save_bindings` (`write_text` или `os.replace`).** Пробрасывается наверх через `set_active_session` / `clear_active_session` / `update_session_id` / `create_new_session` без отката in-memory состояния. Память остаётся обновлённой (новая привязка добавлена / старая удалена). Финальный обработчик — `bot.py`, который сообщает пользователю в Telegram «не удалось сохранить привязку: <причина>», и логирует ошибку. Дрейф «память впереди диска» сохраняется до следующей успешной записи.
- **`RuntimeError` от `_save_bindings` (`_bindings_path is None`).** Возникает, если `set_active_session` или другая модифицирующая функция вызвана до `load_bindings`. Программная ошибка вызывающей стороны (порядок инициализации). `bot.py` гарантирует через `post_init`, что `load_bindings` вызывается до подключения к Telegram polling — в production это исключение не должно возникать.

**Категория «Параллельный доступ»**

- **Защита через `_lock: asyncio.Lock`.** Все модифицирующие операции (`set_active_session`, `clear_active_session`, `update_session_id`, `reset_state`) захватывают Lock на время изменения `_bindings` и записи на диск. Чтения (`get_active_session`, `find_chat_by_session_id`, `is_monitoring_mode`, `get_all_bindings`) Lock НЕ захватывают — это сознательный выбор для минимизации задержек частых операций (watcher вызывает `find_chat_by_session_id` каждые 2 секунды на каждую активную сессию). Поскольку Python dict atomic-операции для одного ключа потокобезопасны в asyncio (нет настоящего параллелизма потоков, только переключение задач), читать без Lock безопасно: гонка может проявиться только как «прочитал чуть устаревшее значение», что для UI не критично.
- **Конкурентные операции на разные `chat_id` (несколько Telegram-чатов одновременно).** При архитектурно-однопользовательском боте (см. CLAUDE.md → «Однопользовательский инвариант») разные `chat_id` могут принадлежать одному человеку с разных устройств. Lock сериализует все модификации — параллельная запись двух разных привязок не повредит файл (атомарный `os.replace` гарантирует целостность).

**Категория «Граничные сценарии»**

- **`set_active_session` с тем же session_id что и текущая привязка.** Перезаписывает запись в словаре (с тем же значением — no-op для памяти, но `_save_bindings` всё равно вызывается). Лог `info` появится, но никакого визуального эффекта для пользователя.
- **`clear_active_session` для несуществующего chat_id.** `pop(chat_id, None)` — no-op, никаких ошибок. `_save_bindings` всё равно вызывается (записывает текущее состояние, которое не изменилось). Это сознательная идемпотентность.
- **`update_session_id` для chat_id, не привязанного к `old_session_id`.** Привязка чата не обновляется (нечего обновлять). `daily_session_registry.update_session_id(old, new)` вызывается ВСЕГДА. Лог `debug`. Это покрывает сценарий «пользователь сменил сессию через `/N` после отправки `/new`, но до прихода первого ответа CLI с реальным UUID».

**Дополнительные решения:**

- Модуль НЕ реализует `_loaded_from_disk` флаг (как `silence_mode_registry`). Защита от затирания валидного файла обеспечивается через `_bindings_path is not None`. После любой неудачной загрузки путь всё равно устанавливается, и следующая запись пройдёт — это сознательный выбор приоритетов: «пользователь должен иметь возможность создать новую привязку, даже если предыдущий файл нечитаем» важнее, чем «не затереть нечитаемый файл». Если в будущем накопится опыт инцидентов с потерями, миграция к `_loaded_from_disk` — путь.
- Модуль НЕ откатывает in-memory изменения при `OSError` записи (в отличие от `current_backend_registry`). Причина: для привязок дрейф «память впереди диска» — меньшее зло, чем потеря созданной пользователем привязки. Пользователь увидит сообщение об ошибке в Telegram и может решить, что делать дальше; внутренне привязка остаётся рабочей до перезапуска.
- Модуль НЕ делает `isinstance(backend, BackendName)` рантайм-проверку в `set_active_session` и `create_new_session`. Тип-подсказка `BackendName` и контракт документации считаются достаточными. Если вызывающий передаст строку — `_save_bindings` упадёт на `active.backend.value` с `AttributeError` (у строки нет атрибута `value`), что явно укажет на ошибку контракта.

## Контракты с внешними системами

Модуль НЕ работает с внешним CLI (Claude или Codex напрямую), Telegram API или сетью. Внешние контракты — два:

### Файл `sessions.json` — формат и миграция

**Источник правды:** эта спецификация (модуль и пишет, и читает файл — он же диктует формат). Расположение: `{config.WORKING_DIR}/sessions.json`.

**Новый формат (после внедрения backend-aware версии):**

```json
{
  "123456789": {"session_id": "eb5ac5bc-2ac6-45ad-8ca9-bb3a1a741f1e", "backend": "claude"},
  "987654321": {"session_id": "_new_a1b2c3d4e5f6", "backend": "codex"}
}
```

Поля каждой записи:
- `session_id` (string, обязательно) — UUID реальной сессии или временный `_new_<12 hex>` для новой сессии.
- `backend` (string, обязательно) — `"claude"` или `"codex"`. Соответствует значениям enum `BackendName` (используется `BackendName(raw_backend)` для конверсии).

**Старый формат (Claude-only, до внедрения):**

```json
{"123456789": "eb5ac5bc-2ac6-45ad-8ca9-bb3a1a741f1e"}
```

**Алгоритм миграции (lazy, при загрузке):** для каждой записи:
- Если значение — `str` → `ActiveSession(session_id=str, backend=BackendName.CLAUDE)`.
- Если значение — `dict` → распарсить как новый формат (`session_id`, `backend`).
- Иначе → пропустить запись с логом `warning`.

При первой же модифицирующей операции (`set_active_session`, `clear_active_session`, `update_session_id`) `_save_bindings` перезапишет файл в новом формате. Старый формат безопасно сосуществует с новым в одном файле во время миграции, потому что разбор идёт пер-record.

**Тест-план для контракта:** unit-тесты на миграцию (входной файл со старым форматом → корректный `ActiveSession` с `BackendName.CLAUDE`), на смешанный формат (часть записей старая, часть новая → все корректно загружены), на запись в новом формате (после `set_active_session` файл содержит структуру с объектом-значением).

### Файловая система — атомарное переименование

**Источник правды:** POSIX `rename(2)`, реализация в Python — `os.replace()` (`Lib/os.py`). На macOS APFS `os.replace` гарантирует атомарность переименования: либо старый файл целиком, либо новый файл целиком. Сторонний читатель не увидит промежуточного состояния.

**Алгоритм:**
1. Записать содержимое в `<file>.tmp` (`Path.write_text` через `asyncio.to_thread`).
2. Переименовать `<file>.tmp` → `<file>` через `os.replace` (`asyncio.to_thread`).

**Тест-план:** unit-тест с моками `Path.write_text` и `os.replace`, проверяющий, что прямой записи в `<file>` нет — только в `<file>.tmp`, затем `os.replace`.

## Константы

- `BINDINGS_FILENAME: str = "sessions.json"` — имя файла привязок. Вынесено в константу для единообразия с `daily_session_registry` (`daily_sessions.json`) и для возможности изменения без правок в нескольких местах.
- `BINDINGS_TEMP_SUFFIX: str = ".tmp"` — суффикс временного файла при атомарной записи. Финальный путь временного файла: `{BINDINGS_FILENAME}{BINDINGS_TEMP_SUFFIX}` = `"sessions.json.tmp"`.
- `TEMP_SESSION_PREFIX: str = "_new_"` — префикс временных session_id. Используется при генерации (`_generate_temp_session_id`) и может использоваться внешним кодом для проверки `session_id.startswith(TEMP_SESSION_PREFIX)` (является ли ID временным).
- `TEMP_SESSION_HEX_LENGTH: int = 12` — длина hex-части временного ID (`uuid.uuid4().hex[:12]`). 12 символов (48 бит) даёт коллизию ~2⁻²⁴ при ~16 млн ID, что для нескольких десятков сессий в день пренебрежимо.
- `LOAD_RETRY_COUNT: int = 5` — количество попыток чтения файла привязок при транзиентной OS-ошибке (EDEADLK на macOS). Значение унаследовано от текущей реализации.
- `LOAD_RETRY_BACKOFF_BASE_SECONDS: float = 0.1` — база экспоненциального backoff между попытками. Задержки: 0.1, 0.2, 0.4, 0.8, 1.6 секунды. Суммарно — около 3.1 секунды максимум, что приемлемо для блокирующего старта бота.

**Module-level state (не публичные константы, но описаны для полноты):**

- `_bindings: dict[int, ActiveSession]` — словарь привязок. Чаты в режиме `/all` отсутствуют в словаре.
- `_lock: asyncio.Lock` — блокировка для защиты от параллельной модификации.
- `_bindings_path: Path | None` — путь к `sessions.json`. `None` до первого вызова `load_bindings`. Запись (`_save_bindings`) проверяет `is not None` и отказывается работать, если путь не задан.

### Тесты ошибок

- **test_save_bindings_raises_runtime_error_when_path_not_loaded** — запись отказывается работать без `load_bindings`.
