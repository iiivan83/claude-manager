# ТЗ: скилл distribute-skill-changes

**Назначение документа:** техническое задание для `pipeline-designer`. Содержит всё необходимое, чтобы в отдельной сессии спроектировать полноценную спецификацию пайплайна. Интервьюер дизайнера может пропустить заново проговаривание согласованных пунктов — они закреплены в разделе «Согласованные решения».

**Дата:** 2026-04-16

---

## 1. Цель и контекст

### Что

Скилл, который берёт обновлённую версию одного конкретного скилла в одном из проектов и раскатывает изменения в остальные проекты — так, чтобы:

- правка уехала в единый источник правды (`budgets`), если её сделали не там
- новая версия появилась в глобальных темплейтах (`~/.claude/skill-templates/`)
- копии скилла в каждом целевом проекте получили изменения **без затирания локальной проектной специфики**
- связанные документы и ссылки (индекс скиллов, `~/.claude/CLAUDE.md`, глобальные референсы) обновились в том же прогоне

### Зачем

Сейчас синхронизация скиллов между проектами делается руками: после правки скилла в `budgets` нужно вручную выгрузить его в `~/.claude/skill-templates/`, а оттуда руками разнести по всем проектам, где этот скилл лежит как рабочая копия. Документы вроде `skill-index.md` и упоминания в `~/.claude/CLAUDE.md` при этом легко забыть. Результат — проекты расходятся между собой, локальные правки теряются, часть документации устаревает. Скилл закрывает этот разрыв: одна команда — согласованное обновление везде.

### Для кого

Оператор проекта `budgets` — человек, который ведёт фабрику скиллов и отвечает за то, чтобы все клиенты скиллов получали свежую версию. Сейчас это один пользователь.

### Ключевые особенности

- Скилл работает с **одним скиллом за прогон** — никаких массовых апдейтов сразу всех скиллов
- Исходник может быть **в любом из проектов**, не только в `budgets`
- Новая версия **не копируется целиком** — применяется только дельта изменений через трёхстороннее слияние. Это нужно, чтобы не затереть проектную специфику в целевых копиях
- При конфликте слияния пайплайн **останавливается** и эскалирует пользователю — скилл ничего сам не разруливает
- Дефолтный список целевых проектов — `budgets` и `claude-manager`. Пользователя каждый раз не спрашиваем

## 2. Входные данные

### Что пользователь указывает на старте

- **Имя скилла** — строка, например `skill-creator`. Пользователь называет сам, никакого автообнаружения «что изменилось»
- **Проект-источник** — путь к проекту, в котором лежит новая версия скилла. Может быть любой из известных проектов. Если не указан — поведение должно быть определено дизайнером (варианты на выбор в разделе «Вопросы для дизайнера»)

### Примеры запуска

Пользователь в Claude Code говорит что-то вроде:

- «Раскатай обновление скилла `skill-creator` из `budgets` в остальные проекты»
- «Обнови скилл `update-skill` во всех проектах, исходник в `claude-manager`»
- «Разнеси изменения в `pipeline-implementer`» (источник не указан — скилл спрашивает или действует по дефолту)

### Откуда скилл сам берёт данные

- **Путь к проектам-целям** — из конфигурационного файла `~/.claude/skill-distribution-targets.json` (создаётся дизайнером). Первый раз — дефолтный список `budgets` + `claude-manager`
- **Базу для трёхстороннего слияния (снимок прошлого sync)** — из `~/.claude/skill-sync-snapshots/<имя-скилла>/`. Если снимка нет — это первый sync, нужно предупредить пользователя, что защиты локальных правок не будет
- **Список файлов скилла для слияния** — сканирование папки скилла в источнике (`SKILL.md`, `agents/**`, `scripts/**`, `references/**` и т. д.)
- **Упоминания имени скилла в связанных документах** — Grep по `~/.claude/references/*.md`, `~/.claude/CLAUDE.md`, `<каждый-проект>/CLAUDE.md`, `<каждый-проект>/.claude/skills/**/SKILL.md`, `<каждый-проект>/.claude/skills/AGENTS.md`, `<каждый-проект>/dev/docs/skill-index.md`

## 3. Выходные данные

### Что скилл производит

- **Обновлённые копии скилла** во всех целевых проектах, прошедших слияние без конфликтов
- **Свежая версия скилла в `~/.claude/skill-templates/<имя>/`** — становится новым мастер-источником для следующих sync
- **Новый снимок в `~/.claude/skill-sync-snapshots/<имя>/`** — база для следующего трёхстороннего слияния
- **Обновлённые связанные документы**:
  - `dev/docs/skill-index.md` в каждом целевом проекте (при необходимости)
  - `~/.claude/CLAUDE.md` — если имя скилла там упомянуто и автоматическая правка безопасна (варианты решения — в разделе «Вопросы для дизайнера»)
  - `~/.claude/references/*.md` — автоправка ссылок на внутренние файлы скилла, если они изменились
- **Отчёт о прогоне** — итоговый сессионный отчёт с перечислением: что изменилось, куда уехало, где были конфликты, какие документы обновлены

### Формат отчёта

Стандартный session-report по правилам `~/.claude/references/document-naming-and-placement.md` (раздел про session-reports).

В отчёте должны быть:

- Имя скилла и проект-источник
- Список целевых проектов и статус по каждому (успех / конфликт / пропущен)
- Список обновлённых связанных документов
- Список упоминаний имени скилла, которые скилл не смог автоматически разобрать и оставил пользователю на ручную проверку
- Пути к резервным копиям

## 4. Этапы пайплайна (скелет)

Ниже — каркас этапов. Полную детализацию (входы-выходы, агенты/скрипты, контракты JSON) разрабатывает `pipeline-designer`. Названия этапов осмысленные — каждое сразу говорит, за что отвечает.

### Схема потока

```
════════════════════════════════════════════════════════════════
 Вход: имя скилла + (опционально) путь к проекту-источнику
       если путь не указан — дефолт = budgets
════════════════════════════════════════════════════════════════
                                │
                                ▼
────────────────────────────────────────────────────────────────
 1. accept-skill-name-and-source
    валидация имени скилла и пути к источнику
    ветка: скилл проектно-специфичный? ──→ СТОП с отказом
           (budget-analyzer, google-sheets-etl,
            signer-orchestrator — не раскатываем)
────────────────────────────────────────────────────────────────
                                │
                                ▼
────────────────────────────────────────────────────────────────
 2. enforce-budgets-as-source-of-truth
    источник == budgets?  ──→ пропуск шага
    иначе: 3-way merge правки из источника в budgets-копию
           (база = снимок, левая = источник, правая = budgets)
    ветка: конфликт ──→ СТОП, эскалация пользователю
    после шага: budgets становится новым источником
────────────────────────────────────────────────────────────────
                                │
                                ▼
────────────────────────────────────────────────────────────────
 3. locate-previous-snapshot
    читаем ~/.claude/skill-sync-snapshots/<имя-скилла>/
    как базу для трёхстороннего слияния в этапе 7
    ветка: снимка нет ──→ предупредить пользователя,
                         запросить явное подтверждение
                         (защиты локальных правок не будет)
────────────────────────────────────────────────────────────────
                                │
                                ▼
────────────────────────────────────────────────────────────────
 4. analyze-impact
    Grep по связанным документам и скиллам:
      - ~/.claude/CLAUDE.md
      - ~/.claude/references/*.md
      - <цель>/CLAUDE.md
      - <цель>/.claude/skills/**/SKILL.md  (cross-refs)
      - <цель>/dev/docs/skill-index.md
    собрать карту: где упомянуто, в каком контексте,
    нужна ли автоправка или ручная проверка
────────────────────────────────────────────────────────────────
                                │
                                ▼
────────────────────────────────────────────────────────────────
 5. present-plan-and-confirm
    показать пользователю:
      - какие файлы скилла изменятся
      - в какие проекты поедет обновление
      - какие связанные документы будут править
      - что останется «на ручную проверку»
    ветка: отказ пользователя ──→ чистая остановка
                                   (ничего не применено)
────────────────────────────────────────────────────────────────
                                │
                                ▼
────────────────────────────────────────────────────────────────
 6. backup-targets
    tar -czf всех целевых папок скилла и связанных документов
    → ~/.claude/skill-distribution-backups/<timestamp>/
────────────────────────────────────────────────────────────────
                                │
                                ▼
────────────────────────────────────────────────────────────────
 7. merge-skill-into-targets

    для каждой целевой копии в проектах (budgets, claude-manager):
      для каждого файла скилла:
        git merge-file <цель> <снимок-база> <источник>
        ветка: конфликт ──→ СТОП только на этой цели;
                            бэкап этой цели остаётся;
                            остальные цели продолжают работать

    для ~/.claude/skill-templates/<имя>/:
      chmod u+w для read-only файлов
      (skill-creator/SKILL.md — Apache License)
      перезапись как мастер-копия
────────────────────────────────────────────────────────────────
                                │
                                ▼
────────────────────────────────────────────────────────────────
 8. update-related-documents
    автоправки по карте из этапа 4:
      - skill-index.md в каждом целевом проекте
      - автоправка ссылок в ~/.claude/references/
        (например, переименование файла внутри скилла)
    неочевидные упоминания ──→ в отчёт на ручную проверку
────────────────────────────────────────────────────────────────
                                │
                                ▼
────────────────────────────────────────────────────────────────
 9. save-fresh-snapshot
    cp -r <budgets>/.claude/skills/<имя>/
         → ~/.claude/skill-sync-snapshots/<имя>/
    становится базой для следующего sync
────────────────────────────────────────────────────────────────
                                │
                                ▼
────────────────────────────────────────────────────────────────
 10. validate-and-report
     structural-validator.sh на обновлённой копии в budgets
     diff -rq источник vs целевые копии
       (разница допустима только там, где в цели была
        локальная специфика, не пересекающаяся с правкой)
     session-report: статус по каждой цели,
                     список мест на ручную проверку,
                     пути к бэкапам
────────────────────────────────────────────────────────────────
                                │
                                ▼
════════════════════════════════════════════════════════════════
 Выход:
   - обновлённые копии скилла во всех целях,
     локальная специфика сохранена
   - мастер-копия в ~/.claude/skill-templates/<имя>/
   - новый снимок в ~/.claude/skill-sync-snapshots/<имя>/
   - резервные копии в skill-distribution-backups/<ts>/
   - session-report
════════════════════════════════════════════════════════════════
```
