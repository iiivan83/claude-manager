## Назначение

Фоновый мониторинг файлов сессий обоих CLI-бэкендов в реальном времени. Каждые `POLL_INTERVAL_SECONDS = 2` секунды модуль обходит файлы сессий каждого backend-а, обнаруживает новые сообщения от ассистента и доставляет их пользователю в Telegram через callback. Координируется с обработчиком сообщений (`bot._send_to_claude_and_respond`) через механизм паузы — чтобы один и тот же ответ не пришёл и от watcher, и от прямого stdout-чтения.

В новой архитектуре, поддерживающей два бэкенда (Claude Code CLI и Codex CLI) одновременно в одном процессе бота, watcher состоит из двух **независимых инстанций** — по одной на каждый `BackendName`. Каждая инстанция работает только со своими файлами через operational-интерфейс `CodingAgentBackend.list_all_session_files_for_project` и `CodingAgentBackend.read_session_file_snapshot` своего бэкенда. Пользователь видит единый поток сообщений — фасад из module-level функций маршрутизирует команды (`pause_session`, `resume_session`, `update_session_id`) в нужную инстанцию через явный параметр `backend: BackendName`. Глобальные операции (`pause_all`, `resume_all`, `reset_state`) применяются ко всем инстанциям одновременно.

**Buffer-and-hold семантика финальности.** Раньше watcher отправлял каждое assistant-сообщение из JSONL-файла сразу, выставляя `is_final = (last_event_type not in busy_set) and (это последнее в пачке)`. Для Claude этот эвристический критерий работал корректно — финал turn-а определяется типом записи `result`, который не относится к busy-множеству. Для Codex же финальный assistant-ответ записывается в JSONL до прихода terminal-записи `event_msg.payload.type == "task_complete"`: пока `task_complete` не пришёл, `snapshot.is_turn_active == True`, и старая эвристика помечала бы такое сообщение как промежуточное (`is_final=False`). Последствия — silence mode подавлял бы финальный ответ, а маркеры `[SEND_FILE]`/`[SHOW_FILE]` не срабатывали бы (`bot.send_watcher_message` обрабатывает их только при `is_final=True`).

Решение — **buffer-and-hold для последнего сообщения**: пока `snapshot.is_turn_active == True`, последнее `assistant`-сообщение из `snapshot.messages` НЕ доставляется (буферизуется). Все остальные новые сообщения turn-а доставляются как промежуточные (`is_final=False`) — это сохраняет живой стриминг прогресса (CJM-10). Когда в snapshot впервые появляется terminal record (`backend.is_turn_terminal_session_record(snapshot.last_record) == True` → `snapshot.is_turn_active == False`), watcher доставляет всё, что накопилось «недостаточно последним» новые промежуточные с `is_final=False`, плюс самое последнее сообщение turn-а с `is_final=True`. Это гарантирует: финальное сообщение никогда не доходит с `is_final=False`, silence mode его не подавляет, маркеры `[SEND_FILE]`/`[SHOW_FILE]` срабатывают.

## Обслуживаемые сценарии

- **CJM-02 (Отправка текстового сообщения)** — модуль обеспечивает координацию через `pause_session`/`resume_session`. `bot._send_to_claude_and_respond` вызывает `pause_session(session_id, backend)` перед отправкой запроса в `process_manager.send_message`, чтобы watcher не дублировал ответ, который придёт в `bot.py` через прямой stdout-канал. После завершения обработки (или по watchdog тишины Agent через 60 секунд) — `resume_session(session_id, backend)`. Также `update_session_id(old, new, backend)` уведомляется при первом получении реального UUID от CLI (callback из `process_manager`).
- **CJM-03 (Отправка фото/файла)** — то же, что CJM-02, плюс watcher доставляет финальный ответ (с возможными маркерами `[SEND_FILE]`/`[SHOW_FILE]`) через тот же callback.
- **CJM-07 (Мониторинг всех сессий /all)** — основной сценарий watcher. Пользователь в режиме `/all` (не привязан ни к какой сессии). Watcher видит, что Claude/Codex дописывает в JSONL-файлы сессий, извлекает текст ответов и доставляет пользователю с заголовком `#N ✅ ответ` (для финальных) или `#N ⏳ ...` (для промежуточных). Номер сессии (`day_number`) для **сторонних** сессий оформляется как кликабельная ссылка `/N` (это делает не watcher, а обработчик callback в `bot.py`).
- **CJM-10 (Живой стриминг промежуточных через watcher)** — критический сценарий, проверяющий контракт с CLI. Watcher должен находить файлы сессий в правильной директории (для Claude — `~/.claude/projects/<sanitized>`, для Codex — `~/.codex/sessions/YYYY/MM/DD/`), парсить их формат, извлекать новые сообщения и доставлять промежуточные обновления ДО финального ответа. Контракт делегирован конкретному бэкенду через `list_all_session_files_for_project` и `read_session_file_snapshot`; watcher не делает прямых файловых операций.
- **CJM-11 (Переключение между проектами)** — watcher участвует в транзакционной семантике переключения. Перед сменой `config.WORKING_DIR` обработчик `bot.handle_switch_project` вызывает `pause_all()` (паузит обе инстанции). После сброса всех state-модулей и загрузки данных нового проекта — `resume_all()` в `finally`-блоке. Также `reset_state()` вызывается между паузой и снятием для очистки накопленного состояния обеих инстанций.
- **CJM-15 (Silence mode)** — watcher не реализует silence mode напрямую (это ответственность `bot.send_watcher_message`), но buffer-and-hold семантика гарантирует, что финальный ответ никогда не уйдёт с `is_final=False` и не будет молча подавлен. Полная связка: watcher выставляет `is_final` корректно → `bot.send_watcher_message` смотрит на `is_final` и `silence_mode_registry.is_enabled()` → промежуточные при включённом silence пропускаются, финальные проходят всегда.

## Расхождение с предыдущей версией

Текущая реализация (`src/claude_manager/session_watcher.py` на 03-05-2026) — module-level функции с одним глобальным состоянием для всех сессий. Поддерживает только Claude (вызывает `session_reader.get_session_messages` напрямую, читает `~/.claude/projects/<sanitized>/<id>.jsonl`). Backend-агностичность отсутствует — модуль ничего не знает о Codex.

В новой версии устранены три источника технического долга и введены два архитектурных усиления:

- **Две независимые инстанции вместо одной.** Раньше `_seen_message_counts: dict[str, int]` хранил счётчики только Claude-сессий. Теперь — класс `SessionWatcher` с per-instance state, по одному singleton-инстансу на каждый `BackendName`. Каждая инстанция работает через свой `backend: CodingAgentBackend`. Module-level функции — фасад, маршрутизирующий команды по `backend: BackendName` параметру. **Причина:** UUID-формат session_id одинаков у Claude и Codex; одного словаря недостаточно для безопасной адресации (Codex-сессия с UUID `abc-123` и Claude-сессия с тем же UUID — теоретически возможно). Изоляция по инстанциям — простейшая защита.
- **Buffer-and-hold для последнего сообщения вместо «is_final по эвристике last_event_type».** Раньше `is_final = (last_event_type not in ACTIVE_EVENT_TYPES)`. Это работало для Claude, ломалось для Codex (`response_item` финального ответа приходит до `event_msg.task_complete`). Теперь watcher не доставляет последнее `assistant`-сообщение пока `snapshot.is_turn_active == True`. Когда turn завершён — последнее сообщение уходит с `is_final=True`, остальные накопленные — с `is_final=False`. **Причина:** silence mode и маркеры файлов опираются на `is_final`; ошибочный `is_final=False` для финала был бы скрытой регрессией.
- **Дельта-чтение через `SessionFileSnapshot` вместо `get_session_messages` + `len()`.** Раньше watcher читал ВСЕ сообщения файла через `session_reader`, считал длину и сравнивал с прошлым счётчиком. Теперь — `await backend.read_session_file_snapshot(file_path)`: snapshot содержит `messages`, `raw_record_count` (стабильный счётчик сырых JSONL-строк), `last_record` и `is_turn_active`. Дельта вычисляется по индексу последнего доставленного сообщения (`last_delivered_idx`), а не по длине списка — это даёт корректный buffer-and-hold для последнего элемента. **Причина:** snapshot — единый контракт интерфейса, watcher не должен лазить мимо него во внутренности `session_reader`.
- **Координация по `(session_id, backend)`-ключу через явный параметр.** Раньше `pause_session(session_id)` адресовал глобальный словарь. Теперь все методы координации (`pause_session`, `resume_session`, `update_session_id`) принимают `backend: BackendName` и маршрутизируют в нужную инстанцию. **Причина:** обработчики (`bot._send_to_claude_and_respond`, `process_manager.session_id_callback`) знают `backend`, потому что они получают его из активной сессии (`session_manager.get_active_session(chat_id).backend`) или передают вручную. Дополнительный параметр — мизерная стоимость, выгода — невозможность перепутать backend.
- **Регистрация сессий через backend-aware `daily_session_registry.register_session(session_id, backend=self.backend.name)`.** Раньше watcher вызывал `register_session(session_id)`, реестр считал любой UUID Claude-сессией. Теперь — обязательный параметр `backend`; registry хранит `DailySessionEntry(session_id, backend)`. Когда watcher обнаруживает новый файл — он знает свой backend по принадлежности к инстанции и передаёт его в registry. **Причина:** `/N` обработчик должен открывать сессию её родным CLI (контракт владения сессией, см. `coding_agent_backend_spec.md` → раздел «Контракт владения сессией для потребителей»).

**Что НЕ меняется:**
- Интервал опроса `POLL_INTERVAL_SECONDS = 2` — из BRD CJM-07.
- Двухслойная защита от затемнения при работе Agent: watchdog в `bot.py` (60 сек) и `PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS = 120` в watcher (см. CJM-10 в BRD).
- Уведомление только владельца сессии (а не broadcast по `ALLOWED_USER_IDS`) с fallback на белый список, если владелец не найден. Однопользовательский инвариант (см. CLAUDE.md → «Однопользовательский инвариант») сохраняется — `ALLOWED_USER_IDS` всё ещё означает «один человек с разных устройств».
- Устойчивость к транзиентному `OSError` (включая `EDEADLK`, errno 11) при чтении файлов. Реализуется на уровне backend (`list_all_session_files_for_project` и `read_session_file_snapshot` ловят ошибки и пропускают файл с `warning`-логом, не падают), плюс на уровне watcher — внешний `try/except` вокруг каждой сессии в `_poll_sessions`.
- Single sequential pass по списку файлов внутри одного опроса. Watcher НЕ запускает обработку файлов через `asyncio.gather` — только `for session_id in session_ids: await self._check_session(...)`. Это сохраняет предсказуемый порядок доставки и снимает риск гонок при параллельной обработке нескольких файлов одного бэкенда.

## Публичный API

### Тип-алиасы для callback-функций

```python
# callback(chat_id, session_id, backend, day_number, message_text, is_current_session, is_final)
MessageCallback = Callable[
    [int, str, BackendName, int, str, bool, bool],
    Awaitable[None],
]

# get_current_session(chat_id) -> ActiveSession | None
CurrentSessionGetter = Callable[[int], Awaitable["ActiveSession | None"]]
```

`MessageCallback` принимает `backend: BackendName` дополнительным аргументом — потребитель (`bot.send_watcher_message`) использует его для отображения иконки бэкенда (🤖/⚡) и для формирования кликабельной ссылки `/N` с правильным backend-routing (на стороне `bot.py` ссылка резолвится через `daily_session_registry.lookup_by_number(today, N)`, который вернёт `DailySessionEntry(session_id, backend)`).

`CurrentSessionGetter` возвращает `ActiveSession` (dataclass из `session_manager_spec.md`), а не голый `session_id`, чтобы потребитель watcher-а мог сравнить ОБА поля (UUID + backend) при определении `is_current_session`. Сравнение только по `session_id` могло бы дать ложно-положительный «это текущая сессия», если у Claude и Codex случайно совпадут UUID.

### Dataclass `SessionWatcherState`

In-memory состояние одной отслеживаемой сессии внутри одной инстанции `SessionWatcher`.

```python
@dataclass
class SessionWatcherState:
    raw_count: int = 0              # последний известный snapshot.raw_record_count
    last_delivered_idx: int = -1    # индекс последнего доставленного сообщения в snapshot.messages (-1 если ничего не доставлено)
    paused_at: float | None = None  # time.monotonic(), если сессия на паузе; None если активна
```

`raw_count` — сырой счётчик строк JSONL-файла. Сравнивается со `snapshot.raw_record_count` для быстрого отбрасывания «файл не менялся» (если совпадает — пропускаем сессию без парсинга `messages`).

`last_delivered_idx` — индекс последнего assistant-сообщения, которое уже доставлено пользователю (или явно пропущено как `is_empty_response`). На следующем опросе новые сообщения берутся как `snapshot.messages[last_delivered_idx + 1:]`. Buffer-and-hold реализуется тем, что при `is_turn_active == True` индекс **не доходит** до последнего сообщения в `snapshot.messages` — последнее остаётся буферизованным до прихода terminal record.

`paused_at` — время постановки на паузу (через `pause_session`). Если `time.monotonic() - paused_at > PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS` — пауза автоматически снимается (защита от утечки, если обработчик упал и не вызвал `resume_session`).

Не `frozen=True`, потому что инстанс watcher изменяет поля in-place (через `_states[session_id].raw_count = ...`); внешним потребителям dataclass не отдаётся.

### Класс `SessionWatcher`

Один инстанс на один backend. Хранит state всех сессий своего бэкенда и реализует цикл опроса.

```python
class SessionWatcher:
    """Мониторинг сессий одного backend-а. Один инстанс на BackendName."""

    def __init__(self, backend: CodingAgentBackend) -> None: ...

    @property
    def backend_name(self) -> BackendName: ...

    async def poll_once(
        self,
        callback: MessageCallback,
        get_current_session: CurrentSessionGetter,
    ) -> None: ...

    async def reset_state(self) -> None: ...

    def pause_session(self, session_id: str) -> None: ...

    async def resume_session(self, session_id: str) -> None: ...

    def pause_all(self) -> None: ...

    def resume_all(self) -> None: ...

    def update_session_id(self, old_session_id: str, new_session_id: str) -> None: ...

    def get_seen_counts_snapshot(self) -> dict[str, SessionUnreadState]: ...

    def is_session_paused(self, session_id: str) -> bool: ...
```

#### `__init__(backend: CodingAgentBackend) -> None`

Создаёт инстанс. Сохраняет `backend` как поле `self.backend`. Инициализирует пустые `self._states: dict[str, SessionWatcherState]` и `self._missing_files: set[str]`, флаг `self._global_paused: bool = False`.

**Аргументы:**
- `backend` (`CodingAgentBackend`) — конкретный бэкенд, на который завязана инстанция (Claude или Codex). Singleton, получается через `coding_agent_backend.get_backend(name)`.

**Исключения:** не выбрасывает.

#### Свойство `backend_name`

```python
@property
def backend_name(self) -> BackendName:
    return self.backend.name
```

Возвращает `BackendName` для удобной диагностики и для маршрутизации в module-level фасаде. Read-only.

#### `poll_once(callback, get_current_session) -> None`

Один цикл опроса всех файлов сессий бэкенда. Вызывается module-level функцией `start()` каждые `POLL_INTERVAL_SECONDS` секунд для каждой инстанции по очереди.

**Аргументы:**
- `callback` (`MessageCallback`) — функция, которой watcher отдаёт обнаруженное сообщение.
- `get_current_session` (`CurrentSessionGetter`) — функция, через которую watcher узнаёт активную сессию пользователя для определения `is_current_session`.

**Возвращает:** `None`.

**Исключения:** не выбрасывает (все исключения внутри ловятся; невосстановимые проблемы логируются как `error`).

#### `reset_state() -> None`

Сбрасывает накопленное состояние и инициализирует счётчики для сессий нового проекта. Вызывается в обработчике `/projects` (через module-level фасад) между `pause_all` и `resume_all`. Без этого вызова watcher в новом проекте увидел бы ВСЕ исторические сообщения как «новые» (потому что `last_delivered_idx == -1` для всех сессий).

**Аргументы:** нет.

**Возвращает:** `None`.

**Исключения:** не выбрасывает (см. Обработка ошибок).

#### `pause_session(session_id: str) -> None`

Помечает сессию как paused — на следующих опросах `_check_session` пропускает её до момента `resume_session` (или истечения `PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS`).

**Аргументы:**
- `session_id` (`str`) — UUID сессии или временный ID `_new_XXXX`.

**Возвращает:** `None`.

**Исключения:** не выбрасывает. Если сессия неизвестна watcher-у — создаёт пустой `SessionWatcherState` со `paused_at = time.monotonic()` (это нормально: процесс может вызвать pause до первого poll).

#### `resume_session(session_id: str) -> None`

Снимает паузу. Перечитывает snapshot файла, обновляет `raw_count` и `last_delivered_idx` до текущего «всё видели» — чтобы НЕ отправить пользователю сообщения, которые уже доставил `bot.py` через прямой stdout-канал.

**Аргументы:**
- `session_id` (`str`) — UUID сессии.

**Возвращает:** `None`.

**Исключения:** не выбрасывает.

#### `pause_all() -> None`

Глобальная пауза всей инстанции — `poll_once` сразу выходит без обхода файлов. Используется при переключении проекта.

**Аргументы:** нет.

**Возвращает:** `None`.

#### `resume_all() -> None`

Снимает глобальную паузу. Сбрасывать state не нужно (за это отвечает `reset_state`, который вызывается между `pause_all` и `resume_all` в обработчике `/projects`).

**Аргументы:** нет.

**Возвращает:** `None`.

#### `update_session_id(old_session_id: str, new_session_id: str) -> None`

Переносит state со старого session_id на новый. Вызывается из `process_manager.session_id_callback`, когда CLI выдаёт первый реальный UUID вместо временного `_new_XXXX`. Обновляются ОБА словаря: `_states` и `_missing_files`.

**Аргументы:**
- `old_session_id` (`str`) — старый ID (обычно `_new_XXXX`).
- `new_session_id` (`str`) — новый реальный UUID.

**Возвращает:** `None`.

**Исключения:** не выбрасывает. Если `old_session_id` не найден в `_states` — пропускает без ошибки (это нормально: callback может прилететь для сессии, которая ещё не попала в watcher).

#### `get_seen_counts_snapshot() -> dict[str, SessionUnreadState]`

Возвращает копию текущего unread-state для всех сессий инстанции. Используется `unread_buffer.save_snapshot()` при переключении проекта.

**Возвращает:** `dict[str, SessionUnreadState]` — `{session_id: SessionUnreadState(raw_record_count=state.raw_count, last_delivered_idx=state.last_delivered_idx)}`.

**Исключения:** не выбрасывает.

#### `is_session_paused(session_id: str) -> bool`

Проверяет, на паузе ли сессия (учитывая `PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS`).

**Возвращает:** `bool`.

### Module-level фасад

Все потребители watcher-а (`bot.py`, `process_manager.py`, обработчики команд) работают **только** через module-level функции. Класс `SessionWatcher` не экспортируется в публичный API — он деталь реализации. Это сохраняет совместимость с привычной module-level моделью, которой следует весь проект (см. `silence_mode_registry`, `daily_session_registry` — те же паттерны).

#### `start(callback, get_current_session) -> None`

Запускает бесконечный цикл опроса для **всех** инстанций. По очереди (последовательно, не через `asyncio.gather`) вызывает `poll_once` каждой инстанции, затем `await asyncio.sleep(POLL_INTERVAL_SECONDS)`.

```python
async def start(
    callback: MessageCallback,
    get_current_session: CurrentSessionGetter,
) -> None: ...
```

**Аргументы:** те же, что у `SessionWatcher.poll_once`.

**Возвращает:** не возвращает (бесконечный цикл).

**Исключения:** `asyncio.CancelledError` — единственный способ корректного завершения (вызывается при `task.cancel()` в `bot.shutdown`).

**Алгоритм инициализации:**
1. Получить список всех инстанций через `_get_all_watchers()`. Этот список фиксируется на момент старта — добавление новых бэкендов в рантайме не предусмотрено (для этого нужен рестарт бота).
2. Для каждой инстанции — выполнить первоначальное сканирование файлов: вызвать `await instance.reset_state()` (без предварительного `pause_all` — это первый запуск). Это инициализирует `_states` так, что начальные сообщения существующих файлов **не** отправятся как «новые» (см. CJM-10: при перезапуске бота старые сообщения не должны лавиной валиться в чат).
3. Войти в бесконечный цикл `while True`, в каждой итерации поочерёдно вызывать `await instance.poll_once(callback, get_current_session)` для каждой инстанции, затем `await asyncio.sleep(POLL_INTERVAL_SECONDS)`. На любую `Exception` (кроме `asyncio.CancelledError`) — логировать `error` с `exc_info=True`, ждать `ERROR_RETRY_DELAY_SECONDS = 10` секунд, продолжать.

#### `pause_all() -> None`

Глобально приостанавливает мониторинг **всех** инстанций (Claude и Codex). Синхронная.

**Алгоритм:** для каждой инстанции из `_get_all_watchers()` — вызвать `instance.pause_all()`. Затем залогировать `info`: «Мониторинг всех сессий приостановлен (глобальная пауза)».

#### `resume_all() -> None`

Глобально возобновляет мониторинг **всех** инстанций. Синхронная.

**Алгоритм:** для каждой инстанции — `instance.resume_all()`. Залогировать `info`.

#### `reset_state() -> None`

Очищает накопленное состояние **всех** инстанций. Вызывается в обработчике переключения проекта между `pause_all` и `resume_all`.

**Аргументы:** нет.

**Возвращает:** `None` (асинхронная — внутри идут `await backend.list_all_session_files_for_project(...)` и `await backend.read_session_file_snapshot(...)`).

**Алгоритм:** для каждой инстанции — `await instance.reset_state()`. Затем залогировать `info`: «Состояние session_watcher сброшено для переключения проекта».

#### `pause_session(session_id: str, backend: BackendName) -> None`

Маршрутизирует pause в инстанцию указанного бэкенда. Синхронная.

**Аргументы:**
- `session_id` (`str`) — UUID сессии или `_new_XXXX`.
- `backend` (`BackendName`) — какой бэкенд владеет сессией.

**Алгоритм:** `_get_watcher(backend).pause_session(session_id)`.

#### `resume_session(session_id: str, backend: BackendName) -> None`

Маршрутизирует resume.

```python
async def resume_session(session_id: str, backend: BackendName) -> None: ...
```

**Алгоритм:** `await _get_watcher(backend).resume_session(session_id)`.

#### `update_session_id(old_session_id: str, new_session_id: str, backend: BackendName) -> None`

Маршрутизирует ремаппинг ID в инстанцию указанного бэкенда. Синхронная.

**Аргументы:**
- `old_session_id` (`str`) — старый ID.
- `new_session_id` (`str`) — новый.
- `backend` (`BackendName`) — backend сессии.

**Алгоритм:** `_get_watcher(backend).update_session_id(old_session_id, new_session_id)`.

#### `get_seen_counts_snapshot(backend: BackendName) -> dict[str, SessionUnreadState]`

Возвращает snapshot unread-state ОДНОГО бэкенда. Используется `unread_buffer.save_snapshot()`, который вызывается отдельно для каждого backend.

**Аргументы:**
- `backend` (`BackendName`) — для какого бэкенда нужно unread-state.

**Возвращает:** `dict[str, SessionUnreadState]`.

#### `is_session_paused(session_id: str, backend: BackendName) -> bool`

Делегат `_get_watcher(backend).is_session_paused(session_id)`. Используется в тестах и для диагностики.

## Внутренние функции

### Module-level

#### `_get_watcher(backend: BackendName) -> SessionWatcher`

Возвращает singleton-инстанс watcher-а для данного бэкенда. Создаёт его при первом обращении.

**Алгоритм:**
1. Если `backend not in _watchers` — `_watchers[backend] = SessionWatcher(coding_agent_backend.get_backend(backend))`. Использует фабрику родительского интерфейса, не импортирует конкретные классы напрямую.
2. Вернуть `_watchers[backend]`.

#### `_get_all_watchers() -> list[SessionWatcher]`

Возвращает список всех инстанций. Создаёт недостающие на лету (при первом обращении).

**Алгоритм:**
1. Для каждого `name in BackendName` — `_get_watcher(name)`.
2. Вернуть собранный список (длина равна количеству значений `BackendName`, на 06-05-2026 — 2).

Используется в `start`, `pause_all`, `resume_all`, `reset_state`.

### Внутри класса `SessionWatcher`

#### `_get_sessions_to_monitor() -> list[str]`

Возвращает список `session_id` всех сессий бэкенда, которые надо опросить в текущем цикле.

**Алгоритм:**
1. `files = await self.backend.list_all_session_files_for_project(config.WORKING_DIR)` — полный operational-список `SessionFileInfo`. Контракт: backend сам обрабатывает `OSError`/`PermissionError`, возвращает пустой список при сбое.
2. `session_ids = [info.session_id for info in files]`.
3. Объединить с сессиями из дневного реестра, у которых `entry.backend == self.backend.name` (это могут быть сессии без файла на диске — например, только что зарегистрированные через temp ID; они уже отфильтрованы orphan-cleanup при загрузке реестра, но добавляются здесь для подстраховки):
   - `today = await daily_session_registry.get_all_today_sessions()` — метод вызывается **без** аргумента `backend`. Возвращает `dict[int, DailySessionEntry]`, где `DailySessionEntry` — DTO с полями `session_id: str` и `backend: BackendName` (см. `coding_agent_backend_spec.md` → раздел DTO). Registry намеренно не принимает аргумент `backend`: это single-responsibility модуль нумерации, а не индекса по атрибутам.
   - Фильтрация на стороне watcher: `today_filtered = [entry.session_id for entry in today.values() if entry.backend == self.backend.name]`. Принцип «фильтр живёт у потребителя» — watcher единственный, кому нужна выборка по бэкенду; остальные потребители (`bot.py`, интеграционные тесты) хотят весь список.
   - Добавить уникальные `session_id` из `today_filtered` в `session_ids`.
4. Удалить session_id, помеченные как `_missing_files` (для них файла нет — не пытаемся читать). Защита от спама `WARNING` в логах.
5. Вернуть.

#### `_check_session(session_id: str, callback, get_current_session) -> None`

Ядро алгоритма — обработка одной сессии в одном опросе.

**Алгоритм:** см. подробное описание в разделе «Алгоритм работы → жизненный цикл одного опроса».

#### `_deliver_message(session_id, day_number, text, is_final, callback, get_current_session) -> None`

Доставляет одно сообщение через callback, выбирая получателя.

**Алгоритм:**
1. Определить владельца: `owner_chat_id = session_manager.find_chat_by_session_id(session_id, backend=self.backend.name)`. Если функция найдёт привязку — `owner_chat_id` это `int`, иначе `None`.
2. Если `owner_chat_id is not None` — `target_chat_ids = [owner_chat_id]`. Иначе — fallback на белый список: `target_chat_ids = [cid for cid in config.ALLOWED_USER_IDS if cid != config.E2E_TEST_USER_ID]`. E2E-тест-аккаунт исключается из broadcast-fallback (он подписан только на свои прямые сессии — иначе сломается тестовая изоляция, см. CLAUDE.md → «E2E тестирование и изоляция»).
3. Для каждого `chat_id` в `target_chat_ids`:
   - `current_session = await get_current_session(chat_id)` (`ActiveSession | None`).
   - `is_current = current_session is not None and current_session.session_id == session_id and current_session.backend == self.backend.name`.
   - `await callback(chat_id, session_id, self.backend.name, day_number, text, is_current, is_final)` — обернуто в `try/except Exception` с `error`-логом и `exc_info=True`. Сбой одного callback не должен останавливать обработку других сессий.

## Алгоритм работы

### `SessionWatcher.poll_once`

1. Если `self._global_paused == True` — выйти без обработки. Это защита от того, что между `pause_all` и `reset_state` event loop успеет выполнить poll и записать сессии нового проекта в state старого.
2. `session_ids = await self._get_sessions_to_monitor()`.
3. Удалить из `self._states` записи, которые исчезли из `session_ids` (сессии, удалённые с диска или ставшие сиротами в реестре). Это предотвращает утечку state при долгой работе бота.
4. Для каждого `session_id` из `session_ids` (СТРОГО ПОСЛЕДОВАТЕЛЬНО, НЕ через `asyncio.gather`):
   - `try: await self._check_session(session_id, callback, get_current_session) except Exception: logger.error(...exc_info=True)`. Внешний guard на случай, если backend нарушит контракт «не выбрасываю ошибок».

**Почему последовательно, а не параллельно:** один backend (особенно Codex) может в `read_session_file_snapshot` сам внутри запускать параллельные задачи — это его дело. Но watcher НЕ должен накладывать поверх ещё один уровень параллелизма. Параллельная обработка нескольких файлов одного бэкенда даёт три проблемы: (а) непредсказуемый порядок доставки сообщений (важно для silence mode и для пользовательского восприятия), (б) гонка при обновлении `self._states` — дельта-чтение перестаёт работать (concurrent `last_delivered_idx`), (в) умножение нагрузки на диск и event loop без реальной выгоды (опросы редкие, по 2 секунды, файлов мало). Последовательный обход в одном цикле проще, безопаснее и достаточен для масштаба бота.

### `SessionWatcher._check_session` (жизненный цикл одного опроса одной сессии)

1. **Проверка паузы.** Если `self._states.get(session_id)` имеет `paused_at is not None`:
   - Если `time.monotonic() - paused_at > PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS` — снять паузу принудительно (`paused_at = None`), залогировать `warning`: «Пауза сессии %s превысила %d сек — автоматическое снятие (вероятная утечка)».
   - Иначе — выйти без обработки (сессия легитимно на паузе).
2. **Проверка отсутствующего файла.** Если `session_id in self._missing_files` — выйти. Это защита от ежесекундного попыток прочитать файл, которого нет (orphan).
3. **Получение пути к файлу.** `files = await self.backend.list_all_session_files_for_project(config.WORKING_DIR)`. Найти `SessionFileInfo` с `info.session_id == session_id`. Если нет — `self._missing_files.add(session_id)`, выйти. Это означает, что сессия зарегистрирована в дневном реестре, но файла на диске нет (или backend не видит — Codex отфильтровал по `payload.cwd`).

   **Оптимизация:** результат `list_all_session_files_for_project` можно закэшировать на одну итерацию `poll_once`, передавая в `_check_session` через параметр; тогда вызов backend.list делается ровно один раз за poll. Это отдельная задача оптимизации; в первой версии — простой повторный вызов на каждую сессию. Backend сам решает, насколько дешёв этот вызов: для Claude — `os.listdir` одной папки (быстро), для Codex — обход директорий по дате (дороже, кэширование осмысленно). При реализации — **передавать кэш через параметр**, чтобы избежать повторного полного обхода для Codex. См. примечание в разделе «Тест-план → Производительность».

4. **Чтение snapshot.** `snapshot = await self.backend.read_session_file_snapshot(info.file_path)`. Контракт: backend сам обрабатывает `OSError`/`PermissionError` (включая `EDEADLK`), возвращает snapshot с пустыми полями при сбое.
5. **Проверка изменения файла.** `previous = self._states.get(session_id, SessionWatcherState())`. Если `snapshot.raw_record_count == previous.raw_count` — файл не менялся, выйти без обработки. Но если сессии не было в `_states` (новый файл) — она впервые попала на радар, идём дальше.
6. **Если snapshot пустой.** Если `snapshot.raw_record_count == 0` (файл пуст или backend вернул пустой snapshot из-за ошибки чтения) — `self._missing_files.add(session_id)` (помечаем как «нет файла», на следующих опросах не пытаемся), залогировать `debug`, выйти.
7. **Регистрация в дневном реестре.** Перед доставкой сообщения — гарантировать, что сессия пронумерована в текущем дне. `try: day_number = await daily_session_registry.register_session(session_id, backend=self.backend.name) except Exception: logger.error(...exc_info=True); return`. **Важно:** registry уже знает сессию (если бот уже отправлял через эту сессию или она есть в файле реестра), `register_session` идемпотентен — возвращает существующий номер.
8. **Buffer-and-hold алгоритм.**
   - `last_idx = len(snapshot.messages) - 1` (-1 если messages пусто).
   - Если `last_idx <= previous.last_delivered_idx` — нет новых сообщений (но `raw_record_count` мог измениться за счёт служебных строк, например только что появилась `event_msg.token_count` без assistant-текста). Обновить `previous.raw_count = snapshot.raw_record_count`, выйти.
   - `is_turn_active = snapshot.is_turn_active`.
   - **Расчёт количества доставляемых сообщений:**
     - Если `is_turn_active == True`:
       - Доставляемая часть: `to_deliver_intermediate = snapshot.messages[previous.last_delivered_idx + 1 : last_idx]` (всё кроме самого последнего; если новых ровно одно — пусто).
       - Самое последнее (`snapshot.messages[last_idx]`) — БУФЕРИЗУЕТСЯ, не доставляется в этом цикле.
       - `new_last_delivered_idx = last_idx - 1` (всё кроме последнего считается доставленным).
       - Все доставляемые — с `is_final=False`.
     - Если `is_turn_active == False`:
       - `to_deliver_intermediate = snapshot.messages[previous.last_delivered_idx + 1 : last_idx]` (всё кроме самого последнего).
       - `to_deliver_final = snapshot.messages[last_idx:last_idx + 1]` (последнее, как одноэлементный список).
       - `new_last_delivered_idx = last_idx`.
       - Промежуточные — с `is_final=False`. Финальное — с `is_final=True`.
9. **Доставка.** Перебор `to_deliver_intermediate` затем `to_deliver_final`:
   - Для каждого `msg`:
     - Если `msg.role != "assistant"` — пропустить (user-сообщения watcher не доставляет; у пользователя они уже есть).
     - Если `msg.is_empty_response == True` — пропустить (служебный маркер вроде `"No response requested."`; backend сам проставляет этот флаг через `text_markers_indicating_empty_response()`).
     - Иначе — `await self._deliver_message(session_id, day_number, msg.text, is_final, callback, get_current_session)`.
10. **Обновление state.** `self._states[session_id] = SessionWatcherState(raw_count=snapshot.raw_record_count, last_delivered_idx=new_last_delivered_idx, paused_at=previous.paused_at)`.

**Edge case: `previous.last_delivered_idx > last_idx`.** Может произойти, если `update_session_id` пропустил перенос или после `reset_state` файл стал короче (не должно быть, но защищаемся). Алгоритм трактует как «отрицательный новый список» — выйдет на шаге 8 без действий, что безопасно.

**Edge case: `is_turn_active == False` И последнее сообщение — это user, а не assistant.** Маловероятно для Claude (`result` всегда последняя запись после user/assistant), но возможно для Codex (если turn штатно завершился без agent_message — например, пользователь сказал «ничего не делай»). Алгоритм пометит это user-сообщение `is_final=True` и пропустит на шаге 9 по `msg.role != "assistant"`. Финального сообщения пользователю не уйдёт — это корректно.

### `SessionWatcher.reset_state`

1. Получить актуальный список сессий: `session_ids = await self._get_sessions_to_monitor()`.
2. Для каждого session_id — `await self.backend.read_session_file_snapshot(file_path)` (через тот же путь, что в `_check_session`), запомнить `snapshot.raw_record_count` и `len(snapshot.messages) - 1`.
3. **Атомарная подмена:** между `_states.clear()` и `_states.update(new_states)` НЕ должно быть `await`, иначе event loop может запустить `poll_once` в момент пустого словаря и watcher отправит ВСЕ исторические сообщения как новые. Сбор данных (через `await`) делается ДО `clear()`, в локальную переменную.

```python
new_states: dict[str, SessionWatcherState] = {}
for session_id in session_ids:
    info = ...  # найти SessionFileInfo
    snapshot = await self.backend.read_session_file_snapshot(info.file_path)
    new_states[session_id] = SessionWatcherState(
        raw_count=snapshot.raw_record_count,
        last_delivered_idx=len(snapshot.messages) - 1,
        paused_at=None,
    )

# Атомарный блок без await
self._states.clear()
self._states.update(new_states)
self._missing_files.clear()
# self._global_paused НЕ трогаем — управляется pause_all/resume_all отдельно
```

3. Залогировать `info`: «Состояние session_watcher (%s) сброшено: %d сессий», %s — `self.backend.name.value`.

### `SessionWatcher.pause_session`

1. `state = self._states.setdefault(session_id, SessionWatcherState())`. Создаёт пустой state, если сессии ещё не было.
2. `state.paused_at = time.monotonic()`.
3. Залогировать `debug`: «Watcher (%s): сессия %s на паузе», %s/%s — backend и session_id.

### `SessionWatcher.resume_session`

1. `state = self._states.get(session_id)`. Если нет — выйти (нечего возобновлять).
2. `state.paused_at = None`.
3. **Перечитать snapshot**, чтобы обновить `raw_count` и `last_delivered_idx` до текущего «всё видели». Это критично: пока сессия была на паузе, `bot._send_to_claude_and_respond` уже доставил пользователю финальный ответ через прямой stdout. Если не обновить state — следующий poll увидит дельту и доставит то же сообщение второй раз.
4. Получить `info` через `list_all_session_files_for_project` и path. Если файла нет — оставить `last_delivered_idx` как есть (state и так корректное), выйти.
5. `snapshot = await self.backend.read_session_file_snapshot(info.file_path)`.
6. `state.raw_count = snapshot.raw_record_count`.
7. `state.last_delivered_idx = len(snapshot.messages) - 1`.
8. Залогировать `debug`: «Watcher (%s): сессия %s снята с паузы, last_delivered_idx=%d», %s/%s/%d.

### `SessionWatcher.update_session_id`

1. Если `old_session_id not in self._states` И `old_session_id not in self._missing_files` — пропустить (нечего переносить, см. Edge case в `update_session_id` ниже).
2. Если `old_session_id in self._states`:
   - `self._states[new_session_id] = self._states.pop(old_session_id)`.
3. Если `old_session_id in self._missing_files`:
   - `self._missing_files.remove(old_session_id)`. 
   - Новый ID **НЕ добавляем** в `_missing_files` — для нового UUID файл может уже существовать на диске; даём шанс обнаружить его на следующем поллинге.
4. Залогировать `info`: «Watcher (%s): session_id обновлён %s → %s», %s/%s/%s.

### `SessionWatcher.pause_all` / `resume_all`

1. `self._global_paused = True` / `False`.

Этих двух операций достаточно: `poll_once` на шаге 1 проверяет `_global_paused` и выходит. `pause_all` не сбрасывает per-session паузы (`paused_at`) — они независимы и продолжают работать корректно, когда `resume_all` вернёт `_global_paused = False`.

### `SessionWatcher.is_session_paused`

1. `state = self._states.get(session_id)`. Если нет — `False`.
2. Если `state.paused_at is None` — `False`.
3. Если `time.monotonic() - state.paused_at > PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS` — `False` (пауза истекла, но ещё не сброшена в `_check_session`).
4. Иначе — `True`.

### `start` (module-level)

1. Получить все инстанции через `_get_all_watchers()`.
2. Для каждой — `await instance.reset_state()` (первоначальная инициализация — все исторические сообщения помечаются как «уже видели»). Залогировать `info` с количеством сессий каждой инстанции.
3. Войти в бесконечный цикл:
   ```python
   try:
       while True:
           try:
               for instance in _get_all_watchers():
                   await instance.poll_once(callback, get_current_session)
           except asyncio.CancelledError:
               raise
           except Exception:
               logger.error("Непредвиденная ошибка в цикле мониторинга", exc_info=True)
               await asyncio.sleep(ERROR_RETRY_DELAY_SECONDS)
               continue
           await asyncio.sleep(POLL_INTERVAL_SECONDS)
   except asyncio.CancelledError:
       logger.info("Мониторинг сессий остановлен")
   ```

## Зависимости

**От модулей проекта:**

- **`config`** — читает `WORKING_DIR` (текущий проект, передаётся в `backend.list_all_session_files_for_project`), `ALLOWED_USER_IDS` и `E2E_TEST_USER_ID` (для broadcast fallback в `_deliver_message`). Эти константы могут меняться во время работы бота (`WORKING_DIR` — при `/projects`); watcher читает их при каждом обращении (не кэширует).
- **`coding_agent_backend`** — `BackendName` (Enum), `CodingAgentBackend` (тип параметра `__init__`), `SessionFileSnapshot`, `SessionMessage`, `SessionFileInfo` (типы возвратов backend-методов). Фабрика `get_backend(name)` — для получения singleton-инстансов в `_get_watcher`.
- **`daily_session_registry`** — `register_session(session_id, backend=...)` (регистрация при первом обнаружении файла; backend-aware, потому что регистрация должна сохранять метку бэкенда сразу), `get_all_today_sessions() -> dict[int, DailySessionEntry]` (возвращает все записи дня без фильтрации; watcher фильтрует по `entry.backend` сам — см. `_get_sessions_to_monitor`). Метод `get_all_today_sessions` намеренно НЕ принимает параметр `backend`: registry остаётся универсальным реестром нумерации, а специфичную выборку делает только тот потребитель, кому она нужна (на момент написания спеки — только watcher).
- **`session_manager`** — `find_chat_by_session_id(session_id, backend=...) -> int | None` (определение владельца сессии для адресной доставки). Backend-aware API — обновлённая версия `session_manager`. На момент написания спеки текущая реализация называется `get_chat_id_for_session(session_id) -> int | None` (без backend, см. `src/claude_manager/session_manager.py:120`); функция будет переименована и расширена при реализации backend-абстракции `session_manager`. **Если на момент реализации watcher API session_manager ещё не обновлён** — использовать текущее имя `get_chat_id_for_session` без `backend` параметра, отметить TODO в коде, поднять дискуссию с авторами. Спека session_watcher не должна блокироваться этим — backend известен по принадлежности инстанции, и одно и то же UUID-значение в текущей кодовой базе всегда означает Claude-сессию.

**Стандартная библиотека:**
- `asyncio` — `asyncio.sleep`, `asyncio.CancelledError`.
- `logging` — `logging.getLogger(__name__)`.
- `time` — `time.monotonic()` для меток `paused_at` (НЕ `time.time()` — monotonic монотонна и не зависит от системных часов).
- `dataclasses` — `@dataclass` для `SessionWatcherState`.
- `collections.abc` — `Awaitable`, `Callable` (для типов callback).

**Не зависит:**
- `session_reader` — больше не нужен. Раньше watcher читал JSONL через `session_reader.get_session_messages` напрямую; теперь — через `backend.read_session_file_snapshot`. Это изоляция: watcher не лазит мимо backend-интерфейса в реализацию чтения файлов.
- `process_manager`, `claude_runner` — нет ни прямой, ни обратной зависимости. `process_manager` импортирует `session_watcher` (для координации pause/resume и update_session_id), но не наоборот. Нарушение этой инверсии — архитектурная регрессия (см. CLAUDE.md → «Слоёная архитектура»).

## Обработка ошибок

- **Файл сессии исчез между listing и snapshot.** Backend (`read_session_file_snapshot`) возвращает пустой snapshot. Watcher на шаге 6 алгоритма помечает `session_id` в `_missing_files`, на следующем опросе пропускает.
- **Ошибка backend.list_all_session_files_for_project.** Backend по контракту не выбрасывает (возвращает пустой список с `warning`-логом). Watcher на этой итерации не увидит новых сессий — нормально, дождётся следующей.
- **Ошибка backend.read_session_file_snapshot.** Backend по контракту не выбрасывает. Возвращает пустой snapshot. Watcher на шаге 6 трактует как «файла нет», помечает session_id в `_missing_files`. Если ошибка транзиентная (`EDEADLK`) — нужно дать шанс перечитать. Решение: периодически очищать `_missing_files` через `_evict_stale_missing_files()` каждые `MISSING_FILES_TTL_SECONDS = 60` секунд (см. Константы). Без этого транзиентный сбой при первом чтении приведёт к тому, что сессия навсегда выпадет из мониторинга в этой инстанции.
- **Ошибка `daily_session_registry.register_session`.** Watcher логирует `error` с `exc_info=True` и пропускает доставку этой сессии в текущем цикле (`return` из `_check_session` на шаге 7). На следующем опросе попробует снова. State `_states[session_id]` НЕ обновляется — это нужно, чтобы при следующем успешном вызове registry watcher всё-таки доставил сообщения (которые в этом цикле остались недоставленными).
- **Ошибка callback (отправка в Telegram).** Watcher логирует `error` с `exc_info=True`, продолжает с другими `chat_id` и другими сессиями. State обновляется как обычно — повторных попыток отправки этого сообщения не будет (это сознательное решение: callback сам должен делать retry внутри, см. `bot.send_watcher_message` → разбивка на части + retry на сетевые ошибки + fallback на plain text). Если callback падает раз за разом — это уже проблема Telegram-канала, не watcher.
- **Транзиентный `OSError` (включая `EDEADLK`).** Делегируется backend-ам (см. контракт `read_session_file_snapshot`/`list_all_session_files_for_project` в родительской спеке: backend сам ловит `OSError` и логирует `warning`, пропуская конкретный файл). Внешний `try/except Exception` в `_poll_sessions` — последний рубеж.
- **`asyncio.CancelledError` в цикле `start`.** Пробрасывается дальше — единственный способ корректного завершения watcher-а при `bot.shutdown`.
- **Изменение `config.WORKING_DIR` посреди цикла `poll_once`.** Watcher НЕ должен видеть промежуточное состояние, когда `WORKING_DIR` уже изменён, но `reset_state` ещё не вызван. Защита — `pause_all` ставится ДО смены `WORKING_DIR` в `bot.handle_switch_project`. Если из-за бага `pause_all` не сработал и watcher запустил `poll_once` в момент рассинхрона — backend всё равно вернёт ошибку «папка не существует» или пустой список, и `_check_session` помечает сессии как `_missing_files`. После `reset_state` `_missing_files` очищается, и сессии нового проекта подцепляются на следующем опросе.
- **Утечка `paused_at`.** Если `bot._send_to_claude_and_respond` упал между `pause_session` и `resume_session` без `try/finally` (баг), сессия остаётся в `_states[id].paused_at` навсегда. Защита: `_check_session` на шаге 1 принудительно снимает паузу через `PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS = 120` секунд с `warning`-логом.

## Контракты с внешними системами

Watcher НЕ имеет прямых контрактов с внешними системами (CLI, файловой системой, Telegram API). Все такие контракты делегированы:

- **Имя папки сессий, формат файла JSONL, расположение** — контракт между `CodingAgentBackend` и реальным CLI. См. `claude_code_backend_spec.md` → «Контракты с внешними системами» (sanitize path, формат stream-json, формат result event), `codex_backend_spec.md` → «Контракты с внешними системами» (формат RolloutLine, расположение `~/.codex/sessions/YYYY/MM/DD/`, формат `event_msg.task_complete`).
- **Telegram API** — контракт `bot.send_watcher_message`, который вызывается через `MessageCallback`. Watcher не знает про `MAX_MESSAGE_LENGTH = 4096`, HTML-экранирование, retry на `RetryAfter`. Эта инкапсуляция — преднамеренная: watcher остаётся transport-agnostic.
- **Файловая система macOS / `EDEADLK`** — обрабатывается на уровне backend (`read_session_file_snapshot` с `try/except OSError`). Watcher не делает прямого I/O.

Поэтому раздел «Контракты с внешними системами» этого модуля минимален: единственный внутренний контракт — между `session_watcher` и `coding_agent_backend.CodingAgentBackend`. Этот контракт зафиксирован в родительской спеке и проверяется параметризованными тестами (`@pytest.fixture(params=[BackendName.CLAUDE, BackendName.CODEX])`) — см. родительский тест-план.

## Константы

- `POLL_INTERVAL_SECONDS = 2` — интервал между обходами всех инстанций. Источник: BRD CJM-07 «каждые 2 секунды проверяет файлы сессий».
- `ERROR_RETRY_DELAY_SECONDS = 10` — задержка после непредвиденной ошибки в цикле `start` перед следующей итерацией. 10 секунд — достаточно, чтобы транзиентная проблема (например, перегруз диска) разрешилась, но не настолько долго, чтобы пользователь жаловался на «бот молчит».
- `PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS = 120` — максимальная длительность паузы сессии. Если истекло — пауза автоматически снимается. 120 сек покрывает: 60 сек watchdog тишины Agent в `bot.py` + 60 сек запас на медленный turn. Источник: CLAUDE.md → «Защита от утечки пауз» и инцидент 22-04-2026 (5 минут 29 секунд тишины).
- `MISSING_FILES_TTL_SECONDS = 60` — TTL для записей в `_missing_files`. Раз в `MISSING_FILES_TTL_SECONDS` секунд `poll_once` очищает `_missing_files` (целиком) — это даёт шанс перечитать файлы, которые временно пропадали (transient `OSError`/`EDEADLK`). Реализация: метка `_missing_files_last_evicted_at: float` в инстанции, проверяется в начале `poll_once`. Без этого механизма транзиентный сбой при первом чтении файла приведёт к перманентному выпадению сессии до рестарта бота.

Внутренние module-level переменные (не предмет публичного API, но детали реализации):

- `_watchers: dict[BackendName, SessionWatcher] = {}` — кэш singleton-инстанций. Заполняется лениво в `_get_watcher`. Очистка не предусмотрена за время жизни процесса (бэкенды не пересоздаются).

- **test_user_messages_skipped_in_delivery** — user-сообщения watcher не доставляет.

- **test_pause_leak_safety_auto_unpauses_after_timeout** — пауза дольше `PAUSE_LEAK_SAFETY_TIMEOUT_SECONDS` снимается автоматически.

- **test_missing_file_session_skipped_until_ttl_eviction** — сессия с отсутствующим файлом пропускается до истечения TTL.
  - Вход: первый poll — backend возвращает пустой snapshot (`raw_record_count=0`) для session_id. Watcher помечает в `_missing_files`. Подменить time.monotonic так, чтобы прошло > 60 сек. Backend теперь возвращает корректный snapshot.
  - Ожидаемый результат: до TTL — backend.read_session_file_snapshot НЕ вызывается на каждом poll (сессия пропущена). После TTL `_missing_files` очищается, backend снова вызывается, сообщение доставляется.

- **test_session_registered_in_daily_registry_but_no_file_yet** — сессия в дневном реестре без файла на диске (свеже-зарегистрированный temp ID после `/new`).
  - Вход: `daily_session_registry.get_all_today_sessions()` возвращает `{1: DailySessionEntry(session_id="_new_AAA", backend=BackendName.CLAUDE)}` (зарегистрирован при `/new`, файла ещё нет). Watcher не пытается читать файл и не помечает в `_missing_files` (это бы заблокировало доставку после `update_session_id` → реальный UUID). - **test_get_sessions_to_monitor_filters_by_backend** — `_get_sessions_to_monitor` фильтрует записи реестра по своему бэкенду; чужие игнорирует.

Никаких pause/missing_files.
  - Ожидаемый результат: `_get_sessions_to_monitor` возвращает только Claude-сессии — `["uuid-claude", "_new_AAA"]` (порядок не важен; Codex-запись `"uuid-codex"` отфильтрована). `daily_session_registry.get_all_today_sessions` вызвана **без** аргумента `backend`. Тест защищает от регрессии «watcher одного бэкенда обрабатывает чужие сессии», подтверждает контракт «фильтр живёт у потребителя».
  - Тип: edge case.

- **test_temp_session_id_replaced_via_update_session_id_then_delivers** — полный цикл: temp ID → реальный UUID → доставка.

Вызвать poll.

State не содержит `_new_AAA` (перенесён в `uuid-real`). `register_session` вызван с `(uuid-real, BackendName.CLAUDE)`. Тип: edge case (защита от race condition «двойная регистрация», которую решает session_id callback из process_manager).

### Тесты ошибок

- **test_transient_oserror_does_not_crash_poll_loop** — backend.read_session_file_snapshot не выбрасывает (по контракту), но если внезапно выбросит — внешний guard ловит.
  - Вход: `FakeBackend.read_session_file_snapshot` поднимает `OSError(11, "Resource deadlock avoided")` (имитация EDEADLK). Watcher вызывает `poll_once`.
  - Ожидаемый результат: исключение поймано в `_poll_sessions` через `try/except Exception`, в логах `error` с `exc_info=True`. Цикл продолжается со следующей сессией. Watcher не падает.
  - Тип: error.

- **test_register_session_failure_skips_delivery_but_keeps_state** — ошибка `daily_session_registry.register_session` не приводит к доставке без номера.
  - Вход: `register_session` поднимает `Exception("registry broken")`. Snapshot с новым assistant-сообщением.
  - Ожидаемый результат: callback НЕ вызван (доставка пропущена), в логах `error`. State `_states[session_id]` НЕ обновляется (на следующем poll попробуем снова).
  - Тип: error.

- **test_callback_failure_continues_with_other_chats** — ошибка callback для одного chat_id не блокирует доставку другим.
  - Вход: `find_chat_by_session_id` возвращает None, два получателя в `ALLOWED_USER_IDS`. Callback падает на первом, успешен на втором.
  - Ожидаемый результат: оба вызова сделаны, ошибка первого залогирована `error` с `exc_info`. State обновлён нормально (поведение не идемпотентно: повтора попытки не будет).
  - Тип: error.

- **test_global_pause_blocks_polling_during_project_switch** — `_global_paused == True` останавливает обработку.
  - Вход: `pause_all()`, затем `await poll_once()`. backend методы не должны быть вызваны.
  - Ожидаемый результат: backend.list_all_session_files_for_project НЕ вызван. callback НЕ вызван. `_states` не изменился.
  - Тип: error (защита от race при переключении проекта).

- **test_reset_state_atomic_clear_and_update** — между `clear()` и `update()` нет `await` (защита от concurrent poll).
  - Вход: подменить `_states.clear` и `_states.update` так, чтобы вызвать `await asyncio.sleep(0)` между ними — и проверить, что НЕ запускается параллельный `poll_once`. Это структурный тест: если автор реализации добавит `await` между clear и update — тест упадёт. (Альтернативно: проверить, что между clear и update event loop не получает control: например, поставить `state['claude'] = ...` после `await self.backend.read_session_file_snapshot(...)`, потом сделать атомарный `clear() + update()`).
  - Ожидаемый результат: реализация атомарна; конкурентный poll, начатый между clear и update, видит уже подмененный state, а не пустой.
  - Тип: error (защита от потери state при `reset_state` с конкурентным poll).
