# Спецификация модуля: daily_session_registry

Дата: 06-05-2026
Слой: 1 (зависит от слоя 0 — `coding_agent_backend` для `BackendName`/`get_backend`, `config` для пути к директории)
Файл: `src/claude_manager/daily_session_registry.py`

**Связанные спеки:**
- `coding_agent_backend_spec.md` — абстрактный интерфейс CLI-бэкенда (источник `BackendName`, `DailySessionEntry`-контракта, фабрики `get_backend`)
- `claude_code_backend_spec.md` — конкретный бэкенд Claude (используется orphan cleanup'ом для backend=CLAUDE)
- `codex_backend_spec.md` — конкретный бэкенд Codex (используется orphan cleanup'ом для backend=CODEX)
- `current_backend_registry_spec.md` — глобальное хранилище имени текущего бэкенда (используется потребителями `register_session` для определения backend новой сессии — НЕ внутри этого модуля)
- `realised/daily_session_registry_spec.md` — старая спека (формат «UUID как строка»), на которую опирается текущая реализация. Эта спека её **заменяет**.

## Назначение

Реестр дневной нумерации сессий: присваивает каждой существующей или вновь созданной сессии порядковый номер в рамках текущего дня (#1, #2, #3...), сбрасывает счётчик в полночь по локальному времени, позволяет находить сессию по её дневному номеру и хранит метаданные сессии — её UUID **и** имя бэкенда (Claude или Codex), который её создал. Хранит реестр в файле `daily_sessions.json` рядом с `sessions.json` в корне проекта (значение `config.WORKING_DIR` на момент `load_registry()`), записывает атомарно (`tmp` → `os.replace`).

**Backend-aware расширение.** Каждая запись хранит структуру `DailySessionEntry(session_id, backend)`, не голый UUID. Это нужно потому, что в одном проекте могут одновременно существовать сессии разных CLI-бэкендов: Claude хранит файлы в `~/.claude/projects/<sanitized>/<uuid>.jsonl`, Codex — в `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl` с фильтрацией по `cwd`. Без поля `backend` обработчик команды `/N` не знает, какой subprocess запустить (`claude --resume` или `codex exec resume`), а orphan cleanup не знает, в какой папке искать .jsonl-файл сессии.

**Сквозная нумерация через два бэкенда.** Счётчик дневных номеров общий для обоих бэкендов: если первая сессия дня создана через Claude (`#1`), вторая — через Codex (`#2`), третья — снова через Claude (`#3`). Номера не повторяются и не пересекаются. Это сознательное продуктовое решение: пользователь видит одну линейку номеров вместо двух (`#1c`, `#1x`), что сохраняет уже сложившийся UX команды `/N`.

**Самовосстановление формата файла.** Файл на диске может оказаться в одном из трёх состояний: целиком старый формат (значения — строки UUID, до миграции), целиком новый (значения — словари `{session_id, backend}`), гибридный (часть строк, часть словарей — например, после прерванной миграции или ручной правки). Любое из этих состояний модуль читает корректно, конвертирует в единое представление в памяти и при первой же записи закрепляет результат на диске в новом формате. Старые записи получают `backend=BackendName.CLAUDE` по умолчанию — это исторически верно: до внедрения двух-бэкендной архитектуры существовал только Claude, и любой UUID без указания бэкенда — Claude-сессия.

## Расхождение с текущей реализацией

Эта спека заменяет `realised/daily_session_registry_spec.md`. Ниже — точечный список изменений, чтобы будущий имплементатор и ревьюер понимали границу миграции.

- **Тип значения в реестре.** Было: `str` (UUID). Стало: `DailySessionEntry(session_id: str, backend: BackendName)`. Внутреннее представление: `_registry: dict[str, dict[str, DailySessionEntry]]` (ключ верхнего уровня — `"YYYY-MM-DD"`, ключ нижнего уровня — номер строкой `"1"`, `"2"`, значение — entry).
- **Сигнатура `register_session`.** Было: `register_session(session_id: str) -> int`. Стало: `register_session(session_id: str, backend: BackendName) -> int`. Backend передаётся вызывающей стороной — для новых сессий это `current_backend_registry.get_current()`, для уже зарегистрированных watcher-инстанций — `self.backend` инстанции (см. `coding_agent_backend_spec.md` → «session_watcher запускается по одной инстанции на каждый бэкенд»).
- **Возврат `lookup_by_number` (backend-aware замена `get_session_id_by_number`).** Было: `get_session_id_by_number(day_number) -> str | None`. Стало: `lookup_by_number(day_number) -> DailySessionEntry | None`. Возврат entry, а не строки — потому что обработчик `/N` после получения entry должен знать backend, чтобы запустить правильный subprocess. Старое имя `get_session_id_by_number` сохранено как compatibility wrapper: оно вызывает `lookup_by_number(day_number)` и возвращает только `entry.session_id` или `None`. Новые потребители используют `lookup_by_number`, но wrapper остаётся для старых тестов и совместимых вызовов.
- **Возврат `get_all_today_sessions`.** Было: `dict[int, str]`. Стало: `dict[int, DailySessionEntry]`. Имя метода сохранено (контракт «всё за сегодня» не изменился, изменился только тип значения).
- **Новая функция `get_backend_for_session(session_id) -> BackendName | None`.** Обратный поиск: по UUID сессии вернуть её backend. Нужна потребителям, которые получают `session_id` без явного `backend` (например, `session_manager` при чтении `sessions.json` старого формата, обработчик `/switch <uuid>` в будущем). Поиск идёт по всем дням реестра — UUID мог быть зарегистрирован вчера и до сих пор используется.
- **Backend-aware orphan cleanup.** Было: `_remove_orphan_entries()` синхронно проверяет существование `<sessions_path>/<session_id>.jsonl`, где `sessions_path` всегда был папкой Claude (`session_reader.build_sessions_path`). Стало: `_remove_orphan_entries()` стал async, для каждой записи смотрит `entry.backend`, через `coding_agent_backend.get_backend(entry.backend)` получает инстанс бэкенда и вызывает `await backend.session_file_exists_for_project(entry.session_id, config.WORKING_DIR)`. Проверка точечная и не зависит от UI-окна `MAX_RECENT_SESSIONS = 15`.
- **Удалена зависимость на `session_reader`.** Старая `_remove_orphan_entries` импортировала `session_reader.build_sessions_path` для построения пути к папке сессий Claude. Теперь путь к файлам каждого бэкенда — внутренняя деталь самого бэкенда (Claude и Codex по-разному устроены), и модулю-реестру правильно делегировать это бэкенду через `session_file_exists_for_project`. Импорт `session_reader` из реестра удаляется.
- **Появилась зависимость на `coding_agent_backend`.** Импорт `BackendName` (для type hints и парсинга строки из JSON в Enum) и `get_backend` (для orphan cleanup'а).
- **Защита `_loaded_from_disk`, retry на чтение, удаление фантомных и дублированных записей** — сохраняются как сейчас, без изменений.
- **`update_session_id(old, new)`** — сигнатура и поведение сохраняются: ищет старый UUID, заменяет на новый. **Backend записи остаётся прежним** (не меняется при подмене UUID — это та же сессия, тот же CLI её создал, просто временный `_new_XXXX` заменён на реальный UUID, выданный CLI).

## Канонический API без `date_str`

В этой спеке и во всех потребителях параметра `date_str` **нет**: дата вычисляется внутри модуля через `_get_today_key()` (текущий день, локальное время). Это канонический контракт пакета backend-aware миграции:

- `register_session(session_id, backend) -> int`
- `lookup_by_number(day_number) -> DailySessionEntry | None`
- `get_all_today_sessions() -> dict[int, DailySessionEntry]`

Такое решение принято по двум причинам:

- **Семантика «нумерация сегодняшнего дня».** Все потребители вызывают `register_session` в момент обработки события (входящего сообщения, обновления от watcher, ответа Claude/Codex). Для них «сегодня» — естественный и единственный осмысленный контекст. Регистрация сессии во вчерашний день не имеет продуктового смысла — вчерашняя нумерация уже зафиксирована, новая сессия из сегодняшнего сообщения должна получить сегодняшний номер. Добавление `date_str` параметра усложнило бы вызовы у `session_watcher._check_session`, `session_manager.bind_session`, `bot._handle_command_new` без выигрыша.
- **Совместимость с текущей реализацией.** В файле `src/claude_manager/daily_session_registry.py` сигнатура сейчас именно такая: `register_session(session_id) -> int`, `get_session_id_by_number(day_number) -> str | None`. Сохранение этой формы (только с расширением типов и новым параметром `backend`) минимизирует ширину миграционного фронта в потребителях — каждый из них меняет один параметр или один тип возврата, не структуру вызова.

Тестируемость даты сохраняется: тесты могут мокать `datetime.date.today()` через `monkeypatch` для проверки midnight reset (этот паттерн уже используется в `tests/test_daily_session_registry.py::TestEdgeCases::test_midnight_reset_creates_new_day`).

Если в будущем понадобится регистрация в произвольный день (например, импорт исторических данных или batch-операции), это будет отдельным изменением API и всех потребителей. В текущем пакете миграции добавлять второй вариант сигнатуры запрещено: два способа регистрации одной дневной сессии создают риск несовместимых тестов и расхождения `/N`, watcher-а и `session_manager`.

## Обслуживаемые сценарии

Из `dev/docs/brd/brd-user-journeys.md` и трейсабельности в `dev/docs/specs/module-dependency-graph.md`:

- **CJM-02: Отправка текстового сообщения** — при получении ответа Claude/Codex модуль регистрирует сессию (если ещё не зарегистрирована) и возвращает её дневной номер. Backend сообщается вызывающей стороной (это бэкенд, через который сейчас работает subprocess).
- **CJM-03: Отправка фотографии или файла** — то же, что CJM-02, поведение реестра не отличается (фото — лишь часть полезной нагрузки, не меняет контракт нумерации).
- **CJM-04: Создание новой сессии (`/new`)** — резервирует новый дневной номер для сессии, создаваемой через текущий глобальный бэкенд (`current_backend_registry.get_current()`). Вызывающая сторона (`bot._handle_command_new`) передаёт этот backend параметром в `register_session`. Если в этот же день уже были сессии другого бэкенда — общий счётчик продолжается, не сбрасывается.
- **CJM-05: Просмотр списка сессий (`/sessions`)** — для каждой видимой в `/sessions` сессии (от Claude и от Codex, обе ветки) реестр обеспечивает дневной номер. Идемпотентность действует по паре `(session_id, backend)`: повторный `register_session` для той же пары возвращает существующий номер, а тот же UUID под другим backend получает отдельный номер. Пользователь видит строку вида `/3 ⚡ Посмотри файл main.py`: номер кликабельный, иконка backend-а добавляется UI-обработчиком, а не самим реестром.
- **CJM-06: Переключение на сессию (`/N`)** — `lookup_by_number(N)` отдаёт `DailySessionEntry` с UUID и backend; обработчик `bot._handle_command_N` передаёт обе компоненты в `process_manager.send_message(session_id=entry.session_id, backend=entry.backend, ...)`. Если сессия не найдена — возвращается `None`, обработчик ищет дальше среди видимых сессий через `session_reader` (поведение из старой спеки, сохраняется).
- **CJM-07: Мониторинг всех сессий (`/all`)** — `get_all_today_sessions()` отдаёт `dict[int, DailySessionEntry]`; watcher по каждому entry знает, какой бэкенд читать (Claude — JSONL в папке проекта, Codex — JSONL в `~/.codex/sessions`). В формате `#3 ⏳` (промежуточные) и `#3 ✅` (финальные) номер берётся из реестра, иконка бэкенда — отдельный визуальный индикатор (см. BRD строки 833, 424, 880), формирование которого — обязанность watcher-форматтера, не реестра.

## Публичный API

### Dataclass `DailySessionEntry`

Одна запись реестра: UUID сессии и имя бэкенда, который её создал. Иммутабельный (`frozen=True`) — реестр не должен мутировать entry после регистрации; единственный способ изменить — заменить целиком (например, в `update_session_id`).

```python
@dataclass(frozen=True)
class DailySessionEntry:
    session_id: str
    backend: BackendName
```

**Поля:**
- `session_id` (`str`) — UUID сессии. Формат: для Claude — UUID v4 от `claude` CLI (например, `"abc12345-def6-7890-abcd-ef1234567890"`), для Codex — UUID v7 от `codex` CLI (например, `"01910e8e-3a2c-7000-89ab-cdef01234567"`). Также допустим временный ID `"_new_XXXX"` в момент создания сессии до получения первого события с реальным UUID.
- `backend` (`BackendName`) — имя бэкенда из enum `coding_agent_backend.BackendName`. Допустимые значения: `BackendName.CLAUDE`, `BackendName.CODEX`.

### `async register_session(session_id: str, backend: BackendName) -> int`

Регистрирует сессию в реестре текущего дня и возвращает её дневной номер. Идемпотентна по паре `(session_id, backend)`: повторный вызов с той же парой в тот же день возвращает существующий номер без изменения записи. Тот же `session_id` с другим `backend` считается другой сессией и получает отдельный дневной номер.

**Аргументы:**
- `session_id` (`str`) — UUID сессии или временный `"_new_XXXX"`.
- `backend` (`BackendName`) — бэкенд, через который сессия создаётся (для новых) или существует (для уже известных). Пара `(session_id, backend)` является логическим идентификатором записи: backend не «перебивает» уже существующую запись другого backend, а создаёт отдельную запись.

**Возвращает:** дневной номер (`int`, начиная с 1).

**Исключения:**
- `OSError` — не удалось записать `daily_sessions.json` (диск заполнен, нет прав, raid disconnect). Ошибка пробрасывается из `_save_registry`. После исключения данные в памяти **уже изменены** (новый номер выдан и сохранён в `_registry`); это компромисс производительности (alternative — два прохода с rollback in-memory — усложнил бы код без практической пользы для бота с одним пользователем).

### `async lookup_by_number(day_number: int) -> DailySessionEntry | None`

Ищет сессию по дневному номеру в реестре **текущего дня** и возвращает её entry или `None`, если номер не найден. Поиск только по сегодняшним записям — это совпадает с поведением старой `get_session_id_by_number` и с UX команды `/N` (пользователь работает с дневными номерами, которые сбрасываются в полночь).

**Аргументы:**
- `day_number` (`int`) — дневной номер сессии.

**Возвращает:** `DailySessionEntry | None`. Если сессия с таким номером не зарегистрирована сегодня — `None`. Вызывающая сторона (`bot._handle_command_N`) при `None` ищет дальше среди видимых сессий через `session_reader` — это поведение из BRD CJM-06, сохраняется без изменений.

**Исключения:** не выбрасывает.

### `async get_session_id_by_number(day_number: int) -> str | None`

Compatibility wrapper для старого Claude-only API. Вызывает `lookup_by_number(day_number)`, возвращает `entry.session_id`, если запись найдена, иначе `None`. Backend через этот wrapper не возвращается, поэтому новые потребители не должны использовать его для запуска subprocess или `/N`.

**Назначение:** сохранить обратную совместимость старых callers/tests на время backend-aware миграции без дублирования логики поиска.

**Исключения:** не выбрасывает.

### `async get_backend_for_session(session_id: str) -> BackendName | None`

Обратный поиск: по `session_id` находит запись в реестре (по всем дням, не только сегодняшнему — UUID мог быть выдан вчера и до сих пор использоваться) и возвращает её backend. `None`, если UUID не зарегистрирован или неоднозначен.

**Аргументы:**
- `session_id` (`str`) — UUID сессии.

**Возвращает:** `BackendName | None`. Если `session_id` найден ровно в одной записи — возвращает её backend. Если тот же UUID есть под несколькими backend — возвращает `None` и логирует `warning`, потому что по одному UUID нельзя безопасно выбрать CLI.

**Исключения:** не выбрасывает.

**Назначение.** Используется потребителями, которые получают «голый» UUID (без явного backend) и должны определить, какой CLI запустить. Примеры: `session_manager.load_bindings` при чтении `sessions.json` старого формата; будущий обработчик `/switch <uuid>`; диагностические команды. Если UUID не зарегистрирован или неоднозначен — потребитель решает сам, что делать (отказать пользователю, попросить создать новую сессию, использовать дефолтный бэкенд). Реестр здесь — справочник «уже виденных» сессий, не директива «как использовать неизвестное».

### `async update_session_id(old_session_id: str, new_session_id: str) -> None`

Заменяет временный UUID на реальный во всех записях реестра (по всем дням). Backend записи **сохраняется** — это та же самая сессия, просто получила настоящий UUID от CLI вместо плейсхолдера `"_new_XXXX"`. Идемпотентна: если `old_session_id` не найден, функция молча завершается без ошибки (повторное обновление, обновление сессии, не зарегистрированной в реестре).

**Аргументы:**
- `old_session_id` (`str`) — текущий UUID (часто временный `"_new_XXXX"`).
- `new_session_id` (`str`) — новый UUID, выданный CLI.

**Возвращает:** `None`.

**Исключения:**
- `OSError` — при записи на диск (см. `register_session`).

**Поиск по всем дням.** Алгоритм перебирает все дни реестра, не только сегодня. Причина: сессия могла быть создана вчера в 23:59 с временным `_new_XXXX`, а реальный UUID пришёл сегодня в 00:01 — обновление должно найти запись во вчерашнем дне. На практике это редко, но дешевле проверить все дни (их единицы), чем потерять обновление.

### `async get_all_today_sessions() -> dict[int, DailySessionEntry]`

Возвращает копию словаря всех зарегистрированных сегодня сессий. Ключ — дневной номер (`int`), значение — `DailySessionEntry`.

**Аргументы:** нет.

**Возвращает:** `dict[int, DailySessionEntry]`. Если сегодня сессий нет — пустой словарь `{}`.

**Контракт «копия».** Возвращаемый словарь — независимая копия; модификация словаря потребителем не отражается на внутреннем состоянии модуля. Это критично для `session_watcher`, который может итерироваться по результату пока другие задачи (`register_session`) добавляют новые записи.

### `def is_registry_loaded() -> bool`

Возвращает значение флага `_loaded_from_disk`. Синхронная функция, не делает I/O.

**Аргументы:** нет.

**Возвращает:** `bool`. `True` если `load_registry()` отработал успешно (включая «файла нет» и «файл повреждён» — оба пути считаются успешной загрузкой и разрешают запись). `False` — если `_read_registry_file` вернул `None` после исчерпания всех retry на непредвиденной ошибке (`PermissionError`, `OSError` с EDEADLK, прочие).

**Назначение.** `bot._notify_user_on_failed_registry_load` или подобный нотификатор может проверить флаг после `load_registry` и сообщить пользователю в Telegram, что нумерация недоступна до перезапуска. Также используется в тестах.

### `async load_registry() -> None`

Загружает реестр из файла `<config.WORKING_DIR>/daily_sessions.json` в память. Вызывается один раз при запуске бота (через `session_manager.load_bindings` или напрямую из `main.post_init`) и при переключении проектов (через `reset_state`).

**Аргументы:** нет.

**Возвращает:** `None`.

**Исключения:** не выбрасывает. Все ошибки чтения логируются, флаги выставляются согласно сценариям в разделе «Обработка ошибок».

**Алгоритм после чтения.** После успешной загрузки модуль вызывает три cleanup-прохода по записям (миграция формата → удаление фантомных `_new_` → удаление дубликатов пары `(session_id, backend)` → backend-aware orphan cleanup) и при необходимости сохраняет очищенный реестр на диск. Подробности — в разделе «Алгоритм работы → load_registry».

### `async reset_state() -> None`

Сбрасывает реестр и путь к файлу, перезагружает данные из `<config.WORKING_DIR>/daily_sessions.json`. Используется при переключении проектов (команда `/projects`, `/pN`).

**Аргументы:** нет.

**Возвращает:** `None`.

**Исключения:** не выбрасывает (как `load_registry`).

**Поведение.** Под `_lock` сбрасывает `_registry = {}`, `_registry_path = None`, `_loaded_from_disk = False`. Затем (вне lock-секции, чтобы не блокировать `_save_registry` внутри `load_registry`) вызывает `await load_registry()`, который пересчитает `_registry_path` от нового `config.WORKING_DIR` и запустит cleanup-проходы.

## Внутренние функции

### `def _get_today_key() -> str`

Возвращает ключ текущего дня для словаря `_registry`. Формат: `"YYYY-MM-DD"` (`DATE_FORMAT`), часовой пояс — локальный (`date.today()`).

### `def _ensure_today_registry() -> None`

Проверяет наличие секции для текущего дня в `_registry`. Если нет (первый запуск, новый день после полуночи) — создаёт пустой словарь под этим ключом. Старые секции **не удаляются** — они нужны для `update_session_id` (вчерашний `_new_` может обновиться сегодня) и для `get_backend_for_session` (вчерашний UUID может использоваться сегодня).

### `def _next_day_number() -> int`

Возвращает следующий свободный дневной номер: `max(existing_numbers_today) + 1`, или `1`, если сегодня записей ещё нет. Сквозной счётчик через два бэкенда — функция не различает Claude и Codex, считает все entries текущего дня одинаково.

### `async _save_registry() -> None`

Атомарно записывает `_registry` в `<config.WORKING_DIR>/daily_sessions.json`. Сериализация — через явный конвертер `_serialize_registry_to_json_dict(_registry)` в словарь с типами JSON-совместимыми (см. ниже), затем `json.dumps(..., indent=2, ensure_ascii=False)`. Запись через `tmp` → `os.replace` (см. «Контракты с внешними системами»).

**Защита `_loaded_from_disk`.** Если флаг `False` — функция молча выходит, логирует `warning`. Это защищает данные на диске от перезаписи пустым реестром, если загрузка упала на непредвиденной ошибке.

**Защита `_registry_path is None`.** Если путь не задан (вызов до `load_registry` или после `reset_state` до завершения перезагрузки) — выбрасывает `OSError("Путь к файлу реестра не задан — вызовите load_registry()")`.

### `def _serialize_registry_to_json_dict(registry: dict[str, dict[str, DailySessionEntry]]) -> dict[str, dict[str, dict[str, str]]]`

Конвертирует in-memory представление в JSON-сериализуемое: каждая `DailySessionEntry` превращается в `{"session_id": entry.session_id, "backend": entry.backend.value}`. Результат — словарь, готовый для `json.dumps`.

### `def _deserialize_one_value_to_entry(raw_value: object) -> DailySessionEntry | None`

Конвертирует одно сырое значение из JSON в `DailySessionEntry`. Принимает `object`, потому что JSON может вернуть `str` (старый формат), `dict` (новый формат) или мусор (повреждённая запись). Возвращает `DailySessionEntry`, или `None`, если запись повреждена и не подлежит восстановлению.

**Поведение по типу:**
- `isinstance(raw_value, str)` — старый формат. Возвращает `DailySessionEntry(session_id=raw_value, backend=BackendName.CLAUDE)`. Логирует `info` при первой такой конверсии в проходе («Миграция формата записи: %s → claude»).
- `isinstance(raw_value, dict)` — новый формат. Извлекает `session_id` (строка), `backend` (строка для конверсии в `BackendName`). Если `session_id` отсутствует или не строка — возвращает `None` (повреждённая запись, удаляется из реестра). Если `backend` отсутствует, не строка или не валиден — возвращает `None` с логом `warning`. Fallback на `BackendName.CLAUDE` разрешён только для старого строкового формата, где backend исторически отсутствовал.
- Любой другой тип (`list`, `int`, `None`) — возвращает `None`, лог `warning`.

### `def _migrate_registry_to_new_format(raw_registry: dict[str, dict[str, object]]) -> tuple[dict[str, dict[str, DailySessionEntry]], bool]`

Проходит по всему сырому реестру, конвертирует каждое значение через `_deserialize_one_value_to_entry`, отбрасывает `None`-результаты. Возвращает кортеж `(migrated_registry, had_migrations)`, где `had_migrations = True`, если хотя бы одна запись была старым форматом ИЛИ хотя бы одна была отброшена. Используется в `load_registry` для решения «нужно ли сразу записать очищенный реестр на диск».

### `def _remove_phantom_entries() -> int`

Удаляет из `_registry` все entries, у которых `entry.session_id.startswith("_new_")`. Это временные плейсхолдеры, для них нет файлов на диске; если они остались в реестре после рестарта — это призраки от прерванной регистрации. Возвращает количество удалённых.

### `def _remove_duplicate_entries() -> int`

Внутри каждого дня находит entries с одинаковым `session_id` (под разными номерами — race condition между watcher и обработчиком). Оставляет запись с наименьшим номером, остальные удаляет. **Backend учитывается:** дубликат — это совпадение `(session_id, backend)`, не только `session_id` (теоретически невозможно, чтобы один UUID был и Claude, и Codex, но защита от мусорных данных). Возвращает количество удалённых.

### `async _remove_orphan_entries() -> int`

Backend-aware orphan cleanup. Удаляет записи, для которых на диске нет соответствующего .jsonl-файла в папке сессий **их бэкенда**. Возвращает количество удалённых.

**Алгоритм:**
1. Для каждого дня и каждой записи в реестре:
   - Пропустить временные `_new_` (для них файлов нет по определению, они отдельно ловятся `_remove_phantom_entries`).
   - Получить инстанс через `coding_agent_backend.get_backend(entry.backend)`.
   - Вызвать `await backend.session_file_exists_for_project(entry.session_id, config.WORKING_DIR)`.
   - Если метод вернул `False` — пометить на удаление, лог `info`.
   - Если метод выбросил неожиданное исключение — запись оставить, залогировать `warning` (консервативное поведение: лучше оставить лишнюю запись, чем удалить валидную).
2. Удалить помеченные записи. Вернуть число.

**Почему не `list_session_files_for_project`.** Listing ограничен `MAX_RECENT_SESSIONS = 15` и предназначен для UI `/sessions`. Orphan cleanup — операция сохранности данных: она должна проверять конкретный UUID без recent-окна. Поэтому контракт использует отдельный backend-метод `session_file_exists_for_project`.

### `async _read_registry_file() -> dict | None`

Читает `_registry_path` с диска с retry. Сохраняется как сейчас, без изменений по существу:
- До `LOAD_RETRY_COUNT = 5` попыток.
- Между попытками — `LOAD_RETRY_DELAY_SECONDS = 1` секунда.
- `FileNotFoundError` → возврат `{}` сразу, без retry (нормально при первом запуске).
- `json.JSONDecodeError` → возврат `{}` сразу, без retry (битый файл, перезапись только улучшит).
- Любая другая `Exception` (включая `PermissionError` и `OSError(11, "Resource deadlock avoided")` — EDEADLK на macOS, см. CLAUDE.md) — повтор, лог `warning`. После исчерпания — лог `error`, возврат `None`.

Возвращает «сырой» словарь до миграции — JSON-структура `dict[str, dict[str, str | dict]]`, где значение нижнего уровня может быть строкой (старый формат) или словарём (новый). Конверсия в `DailySessionEntry` — отдельным шагом в `load_registry`.

## Алгоритм работы

### register_session

1. Захватить `_lock`.
2. Вызвать `_ensure_today_registry()` — гарантировать наличие секции под текущий день.
3. Получить `today_entries = _registry[_get_today_key()]`.
4. Поиск по существующим: для каждого `(number_str, existing_entry)` из `today_entries`, если `existing_entry.session_id == session_id and existing_entry.backend == backend` — вернуть `int(number_str)` без изменений (идемпотентность по паре). Если UUID совпал, но backend другой — это отдельная запись.
5. Если не найдено — получить `day_number = _next_day_number()`.
6. Создать `entry = DailySessionEntry(session_id=session_id, backend=backend)`.
7. Записать `today_entries[str(day_number)] = entry`.
8. Вызвать `await _save_registry()` (защита `_loaded_from_disk` внутри).
9. Лог `info`: `"Сессия %s (%s) зарегистрирована как #%d"`, args: `(session_id, backend.value, day_number)`.
10. Освободить lock, вернуть `day_number`.

### lookup_by_number

1. Захватить `_lock`.
2. Вызвать `_ensure_today_registry()`.
3. Получить `today_entries = _registry[_get_today_key()]`.
4. Вернуть `today_entries.get(str(day_number))` (`None` если ключа нет).
5. Освободить lock.

### get_backend_for_session

1. Захватить `_lock`.
2. Создать `matches: set[BackendName] = set()`.
3. Для каждого `day_entries` в `_registry.values()`:
   - Для каждого `entry` в `day_entries.values()`:
     - Если `entry.session_id == session_id` — добавить `entry.backend` в `matches`.
4. Если `len(matches) == 1` — освободить lock и вернуть единственный backend.
5. Если `len(matches) > 1` — залогировать `warning` про неоднозначный UUID, освободить lock, вернуть `None`.
6. После всех дней — освободить lock, вернуть `None`.

### update_session_id

1. Захватить `_lock`.
2. Флаг `found = False`.
3. Внешний цикл по `day_entries` в `_registry.values()`:
   - Внутренний цикл по `(number_str, current_entry)` в `day_entries.items()`:
     - Если `current_entry.session_id == old_session_id` — создать новый `entry = DailySessionEntry(session_id=new_session_id, backend=current_entry.backend)` (backend сохранён!), записать `day_entries[number_str] = entry`, выставить `found = True`, прервать внутренний цикл.
   - Если `found` — прервать внешний цикл.
4. Если `not found` — лог `debug`: `"Session ID %s не найден в реестре — пропускаем"`, освободить lock, выйти.
5. Если `found` — `await _save_registry()`, лог `info`: `"Session ID обновлён: %s → %s (backend остался %s)"`.
6. Освободить lock.

### get_all_today_sessions

1. Захватить `_lock`.
2. Вызвать `_ensure_today_registry()`.
3. Получить `today_entries = _registry[_get_today_key()]`.
4. Сформировать копию: `{int(number_str): entry for number_str, entry in today_entries.items()}`. Так как `DailySessionEntry` иммутабелен (`frozen=True`), shallow copy на уровне словаря достаточна — потребитель не сможет изменить поля entry.
5. Освободить lock, вернуть копию.

### load_registry

1. Глобально: `_registry_path = Path(config.WORKING_DIR) / REGISTRY_FILENAME`.
2. Прочитать файл: `raw = await _read_registry_file()`.
3. Если `raw is None` (исчерпали retry на непредвиденной ошибке):
   - `_registry = {}`, `_loaded_from_disk = False`.
   - Завершить функцию (без cleanup'а — реестр пуст, чистить нечего).
4. Иначе (получили `raw_registry: dict`):
   - Запустить миграцию: `(migrated, had_migrations) = _migrate_registry_to_new_format(raw)`.
   - `_registry = migrated`, `_loaded_from_disk = True`.
   - Лог `info`: `"Реестр дневных сессий загружен из %s (%d дней, %d записей)"` (если `raw` непустой).
5. Cleanup-проходы (только если `_loaded_from_disk == True` — иначе чистить нечего):
   - `phantom_count = _remove_phantom_entries()`. Если `> 0` — лог `info`.
   - `duplicate_count = _remove_duplicate_entries()`. Если `> 0` — лог `info`.
   - `orphan_count = await _remove_orphan_entries()`. Если `> 0` — лог `info`.
6. Если `_loaded_from_disk == True` И (`had_migrations` или `phantom_count > 0` или `duplicate_count > 0` или `orphan_count > 0`):
   - `await _save_registry()` — закрепить мигрированный/очищенный реестр в новом формате на диске.
7. `_ensure_today_registry()` — гарантировать наличие секции для сегодня.

**Порядок cleanup-проходов важен.** Phantom удаляются первым — после этого `_remove_orphan_entries` не запускает поиск файлов для `_new_` UUID (они никогда не имеют файлов). Duplicate — вторым: дубликаты одной пары `(session_id, backend)` должны схлопнуться до orphan-проверки. Одинаковый UUID под разными backend не является дублем и остаётся двумя записями. Orphan — последним: он async и потенциально дорогой, его выполняют на уже почищенном реестре.

### reset_state

1. Захватить `_lock` для атомарного сброса:
   - `_registry = {}`.
   - `_registry_path = None`.
   - `_loaded_from_disk = False`.
2. Освободить lock.
3. `await load_registry()` — пересчитать путь от нового `config.WORKING_DIR`, запустить cleanup на новом проекте.
4. Лог `info`: `"Состояние daily_session_registry сброшено и перезагружено для проекта %s"`.

**Lock не удерживается во время `load_registry`.** Внутри `load_registry` есть свой захват lock (через `_save_registry` и cleanup), вложенный захват `asyncio.Lock` не блокирует, но логика проще без вложенности. Сценарии race: пока reset_state в `load_registry`, другая задача может вызвать `register_session` — она увидит `_loaded_from_disk = False` (мы только что сбросили) и `_save_registry` молча выйдет, новая запись не попадёт ни на диск, ни в перезагружаемый реестр. Это приемлемо: переключение проектов — редкая операция, потеря одной регистрации в этом окне (миллисекунды) допустима, а пользователь всё равно получит непрочитанные сообщения через `unread_buffer`.

## Формат файла daily_sessions.json

### Целевой формат (после миграции)

```json
{
  "2026-05-06": {
    "1": {"session_id": "abc12345-def6-7890-abcd-ef1234567890", "backend": "claude"},
    "2": {"session_id": "01910e8e-3a2c-7000-89ab-cdef01234567", "backend": "codex"},
    "3": {"session_id": "fghi5678-jklm-9012-nopq-rstuvwxyz789", "backend": "claude"}
  },
  "2026-05-05": {
    "1": {"session_id": "old-session-uuid", "backend": "claude"}
  }
}
```

**Структура:**
- Ключ верхнего уровня — дата `"YYYY-MM-DD"`.
- Значение — словарь `{дневной_номер_строкой: entry_dict}`.
- `entry_dict` — JSON-объект с двумя обязательными полями:
  - `session_id` — UUID сессии (строка) или `"_new_XXXX"` (временный, в норме не сохраняется на диск, но если попал — фильтруется на загрузке через `_remove_phantom_entries`).
  - `backend` — `"claude"` или `"codex"` (значения `BackendName.value`).

### Старый формат (до миграции)

```json
{
  "2026-05-05": {
    "1": "old-session-uuid",
    "2": "another-uuid"
  }
}
```

**Структура:**
- Тот же двойной словарь, но значение нижнего уровня — строка (UUID), а не словарь.

### Гибридный формат (промежуточное состояние)

```json
{
  "2026-05-05": {
    "1": "old-uuid-as-string",
    "2": {"session_id": "new-uuid-as-dict", "backend": "codex"}
  }
}
```

Возникает после прерванной миграции, ручной правки, или смешанной записи в момент апгрейда бота. **Поведение модуля:**
- При загрузке детектирует тип каждого значения отдельно (`isinstance(value, str)` — старый, `isinstance(value, dict)` — новый).
- Каждая старая запись конвертируется в `DailySessionEntry(session_id=value, backend=BackendName.CLAUDE)`.
- Каждая новая запись парсится как есть.
- При первой записи на диск (если был cleanup или `had_migrations`) — весь реестр сохраняется в новом формате; гибридное состояние больше не воспроизводится.

### Кодировка и форматирование

- UTF-8, без BOM.
- `ensure_ascii=False` (читаемая кириллица в логах ручных правок, если они будут).
- `indent=2` (двухпробельный, чтобы файл был читаем глазами).
- Перенос строки после `}` финального — стандартное поведение `json.dumps` (без trailing newline).

## Внутреннее состояние модуля

- `_registry: dict[str, dict[str, DailySessionEntry]] = {}` — основное состояние. Ключ верхнего уровня — `"YYYY-MM-DD"`, ключ нижнего уровня — номер строкой `"1"`, `"2"` (строкой потому, что сериализация JSON всегда делает ключи строками; в API возвращается уже как `int`), значение — entry.
- `_lock: asyncio.Lock = asyncio.Lock()` — защита от параллельных операций.
- `_registry_path: Path | None = None` — путь к `daily_sessions.json`. Заполняется в `load_registry`, обнуляется в `reset_state`.
- `_loaded_from_disk: bool = False` — защита от затирания. См. логику в `load_registry` и `_save_registry`.

## Обработка ошибок

**При загрузке (`_read_registry_file` и `load_registry`):**
- **Файл не существует** — `_registry = {}`, `_loaded_from_disk = True` (запись разрешена; ничего не теряем — файла никогда не было). Лог `info`: «Файл реестра не найден, создаю пустой реестр».
- **Невалидный JSON** (`json.JSONDecodeError`) — `_registry = {}`, `_loaded_from_disk = True` (битый файл, перезапись только улучшит). Лог `warning`.
- **`PermissionError` или непредвиденный `OSError` (включая EDEADLK errno 11 на macOS)** — retry до 5 раз с интервалом 1 секунда. Если все попытки исчерпаны — `_registry = {}`, `_loaded_from_disk = False` (запись заблокирована — данные на диске могут быть валидными, не затираем). Лог `error`. Бот через `is_registry_loaded()` может уведомить пользователя в Telegram.
- **Запись с повреждённой структурой** (например, значение нижнего уровня — `int` или `null`) — `_deserialize_one_value_to_entry` возвращает `None`, запись отбрасывается с логом `warning`. Цельность реестра не страдает (другие записи остаются).

**При записи (`_save_registry`):**
- **`_loaded_from_disk == False`** — функция выходит молча с логом `warning`. Никакой записи на диск, никакого исключения наверх.
- **`_registry_path is None`** — `OSError("Путь к файлу реестра не задан — вызовите load_registry()")` пробрасывается. Это программный баг (`_save_registry` вызвали до `load_registry`), а не штатная ошибка ввода-вывода.
- **`OSError` от `write_text` или `os.replace`** — пробрасывается из `_save_registry` в `register_session`/`update_session_id` и далее в вызывающий код. Внутреннее состояние `_registry` уже изменено (новая запись или обновление сделаны до записи на диск); это компромисс, в условиях бота с одним пользователем — рассогласование между памятью и диском исчезнет на следующей удачной записи.

**Backend в записях:**
- **Значение `backend` не из enum** (например, `"gemini"` после ручной правки или мусор) — `_deserialize_one_value_to_entry` ловит `ValueError` от `BackendName(...)`, логирует `warning` и возвращает `None`. Запись отбрасывается при загрузке. Fallback на Claude здесь запрещён: это уже новый формат с явным, но неизвестным backend, и тихая подмена может открыть сессию неправильным CLI.
- **Backend в записи отсутствует (только `session_id`)** — запись нового формата считается повреждённой, `_deserialize_one_value_to_entry` возвращает `None`, лог `warning`. Fallback на `BackendName.CLAUDE` применяется только к старому формату, где всё значение — строка UUID.

**Идемпотентность:**
- `register_session` повторно с той же парой `(session_id, backend)` в тот же день — возвращает существующий номер, ничего не пишет. Тот же UUID с другим backend получает отдельный номер.
- `update_session_id` с несуществующим `old_session_id` — выходит без ошибки (лог `debug`).

**Конкурентность:**
- Все публичные функции захватывают `_lock`, что сериализует операции. Параллельные `register_session` не теряют данные.

## Контракты с внешними системами

Модуль работает с двумя «внешними» системами: файловая система macOS (атомарное переименование) и интерфейс `coding_agent_backend` для backend-aware orphan cleanup. Контрактов с самим CLI (Claude Code, Codex) у модуля нет — все CLI-специфичные знания делегированы бэкенду через интерфейс.

### Файловая система — атомарное переименование

**Источник правды:** POSIX `rename(2)`, реализация — Python `os.replace()`. На macOS APFS `os.replace` атомарен: сторонний читатель видит либо старое содержимое, либо новое — без промежуточного состояния (полузаписанного JSON).

**Алгоритм** (тот же, что используется во всех state-модулях проекта):
1. Записать новое содержимое в временный файл `<file>.tmp`.
2. `os.replace(<file>.tmp, <file>)` — атомарное переименование.

**Тест-план:** `test_save_uses_atomic_rename` мокает `Path.write_text` и `os.replace`, проверяет что `write_text` вызван на `<file>.tmp`, `os.replace` вызван с парой `(<tmp>, <file>)`. Реальная атомарность POSIX отдельно не тестируется (зафиксированное стандартом поведение).

### Файл `daily_sessions.json` — формат и контракт

**Источник правды:** эта спецификация. Модуль читает и пишет файл сам, формат полностью под его контролем.

**Формат:** см. раздел «Формат файла daily_sessions.json» выше — целевой, старый, гибридный.

**Расположение:** `<config.WORKING_DIR>/daily_sessions.json`. Меняется при переключении проектов через `reset_state` (новое значение `config.WORKING_DIR` → новый путь).

**Кодировка:** UTF-8 без BOM, `ensure_ascii=False`, `indent=2`.

### Интерфейс `coding_agent_backend.session_file_exists_for_project`

**Источник правды:** `coding_agent_backend_spec.md` — сигнатура `async session_file_exists_for_project(session_id: str, project_dir: str) -> bool`. Метод проверяет конкретный UUID и не ограничивается `MAX_RECENT_SESSIONS`, потому что orphan cleanup — не UI-listing.

**Алгоритм использования в `_remove_orphan_entries`:** см. раздел «Внутренние функции → _remove_orphan_entries → Алгоритм».

**Контракт сохранности:** если файл существует, но не попадает в список 15 свежайших `/sessions`, метод всё равно должен вернуть `True`.

**Тест-план для этого контракта:** `test_orphan_cleanup_keeps_session_existing_outside_recent_list` — мокает `coding_agent_backend.get_backend`, возвращает фейковый бэкенд, у которого `list_session_files_for_project()` не содержит UUID, но `session_file_exists_for_project(uuid, project_dir)` возвращает `True`; запись должна остаться.

## Константы

- `REGISTRY_FILENAME = "daily_sessions.json"` — имя файла реестра.
- `REGISTRY_TEMP_SUFFIX = ".tmp"` — суффикс временного файла при атомарной записи.
- `DATE_FORMAT = "%Y-%m-%d"` — формат ключей даты, ISO 8601.
- `LOAD_RETRY_COUNT = 5` — количество попыток чтения файла при непредвиденной OS-ошибке (EDEADLK на macOS и подобные транзиентные сбои).
- `LOAD_RETRY_DELAY_SECONDS = 1` — пауза между попытками чтения (секунды).
- `DEFAULT_BACKEND_FOR_LEGACY_ENTRIES = BackendName.CLAUDE` — backend по умолчанию для записей старого формата (когда значение в JSON — строка). Выбор: до внедрения двух-бэкендной архитектуры существовал только Claude, любая «безымянная» запись — Claude.

  - Тип: edge case (приоритет cleanup-проходов).

### Тесты ошибок

- **test_load_retry_on_transient_oserror** — устойчивость к транзиентному EDEADLK.
  - Вход: на диске валидный файл, но `Path.read_text` мокнут так, что первые 2 вызова бросают `OSError(11, "Resource deadlock avoided")`, третий вызов проходит и возвращает валидный JSON.
  - Ожидаемый результат: `load_registry` не падает, реестр загружен, `_loaded_from_disk == True`. В логах 2 warning-сообщения о попытках, нет error.
  - Тип: error.

- **test_load_exhausts_retries_blocks_writes** — после 5 неудачных попыток запись запрещена.
  - Вход: `Path.read_text` бросает `PermissionError("denied")` на каждый вызов (включая все 5 retry).
  - Ожидаемый результат: после `load_registry`: `_loaded_from_disk == False`, `is_registry_loaded() == False`. Любая последующая попытка `await register_session("uuid", BackendName.CLAUDE)` завершается без `OSError` (новая запись добавлена в память), но `_save_registry` молча выходит — файл на диске не создан. Лог `warning` от `_save_registry`.
  - Тип: error.

- **test_save_blocked_when_not_loaded_from_disk** — защита `_loaded_from_disk == False`.
  - Вход: вручную выставить `_loaded_from_disk = False` (через monkeypatch). Вызов `await register_session("uuid", BackendName.CLAUDE)`.
  - Ожидаемый результат: `register_session` отрабатывает (новый номер выдан, в памяти entry появилась), но `_save_registry` молча вышел; файла на диске нет (либо остался прежний). Лог `warning` от `_save_registry` с фразой про «не были загружены с диска».
  - Тип: error.

- **test_save_raises_on_oserror_during_write** — ошибка записи пробрасывается.
  - Вход: `_loaded_from_disk = True`, мокнуть `Path.write_text` на `OSError(28, "No space left on device")`.
  - Ожидаемый результат: `await register_session("uuid", BackendName.CLAUDE)` бросает `OSError(28, "No space left on device")`. Запись в памяти **уже добавлена** до момента вызова `_save_registry` — это согласуется с разделом «Обработка ошибок → При записи».
  - Тип: error.

- **test_reset_state_reloads_from_new_working_dir** — переключение проекта.
  - Вход: загрузить реестр из проекта A (несколько записей). Подменить `config.WORKING_DIR` на проект B, на диске B — другой `daily_sessions.json` с другими записями. Вызов `await reset_state()`.
  - Ожидаемый результат: `_registry_path` теперь указывает на путь в проекте B. `_registry` содержит записи из B, не из A. `_loaded_from_disk == True` (если файл B читался успешно).
  - Тип: error / state-management.
