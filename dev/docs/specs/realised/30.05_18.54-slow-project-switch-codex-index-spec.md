## Контекст

Claude Manager — локальный Telegram-бот для управления Claude Code и Codex с телефона. В сценарии `/projects` -> `/pN` пользователь выбирает другой проект, а бот должен сменить рабочую директорию, сбросить наблюдатели сессий и вернуть непрочитанные сообщения, накопившиеся в проекте.

Проблема из RCA: переключение занимает 7-17 секунд, потому что горячий пользовательский путь повторно сканирует глобальную историю Codex. Codex хранит rollout-файлы не внутри проекта, а в общей папке `~/.codex/sessions/YYYY/MM/DD/`, поэтому текущий код открывает тысячи JSONL-файлов, читает `session_meta` и проверяет `payload.cwd`.

Термины, важные для реализации:

- **Operational index** — in-memory индекс последних Codex rollout-файлов: карта `project_dir -> list[SessionFileInfo]`. Индекс хранит только местоположение и лёгкие метаданные файлов, а не текст сообщений.
- **Hot path** — путь, который блокирует ответ пользователю в Telegram. Для этой задачи это `/pN`, `session_watcher.reset_state()` и `project_manager._collect_pending_messages()`.
- **Pending** — непрочитанные ответы из проекта, пока пользователь был в другом проекте.
- **Cursor** — лёгкое состояние файла сессии: количество записей, последний record и активность turn-а, без полного парсинга всех сообщений.
- **mtime** — время последнего изменения файла. Если файл не менялся с момента сохранения unread snapshot, pending заведомо отсутствует и полный JSONL читать не надо.

## Решения из handoff

Эта спека принимает решения, уже зафиксированные после RCA:

- Не менять UX на раннее подтверждение. Бот по-прежнему отвечает «переключено» после штатной pending-проверки. Идея «сначала ответить, потом дослать pending фоном» в этой задаче не реализуется.
- Lookback для Codex operational index — 4 дня. Старые дни выпадают из индекса при пересборке окна.
- Индекс является картой файлов, а не источником содержимого. Preview, summary, pending-дельта и отчёты читают реальные файлы.
- Актуализация индекса не должна держаться только на TTL. Нужны проверка date-директорий, точечное обновление известной `session_id` и обновление `mtime` для возвращаемых файлов.
- Новый объём кода не добавлять в уже раздутые модули. Основная логика индекса живёт в новом модуле.

## Цели

1. Убрать повторный global scan Codex rollout-файлов из одного переключения проекта.
2. Сделать повторные вызовы `CodexBackend.list_all_session_files_for_project(..., lookback_days=4)` дешёвыми, пока окно не изменилось.
3. Сделать pending no-op дешёвым: если файл сессии не менялся, не читать полный snapshot.
4. Сохранить текущий пользовательский контракт `/pN`: сообщение о переключении и pending-доставка остаются в том же порядке.
5. Добавить тесты, которые ловят не только время, но и алгоритмическую регрессию: два-три повторных скана вместо одного.

Целевой практический результат: обычное переключение проекта должно вернуться к диапазону около 1-2 секунд вместо 7-17 секунд. Точное время зависит от диска и числа реальных сессий, поэтому автотесты должны проверять число сканов и чтений, а живой замер — подтверждать стеночное время.

## Не цели

- Не переносить pending-сбор в фон.
- Не создавать persistent cache на диске.
- Не добавлять filesystem watcher или зависимость на inotify.
- Не менять формат файлов Codex CLI.
- Не переписывать режим `/all`; если он получит пользу от нового индекса позже, это отдельная задача.
- Не делать крупный рефакторинг `project_manager.py` или `coding_agent_session_file_poller.py` внутри этой задачи без явного решения по размеру файлов.

## Архитектура решения

### Новый модуль `codex_session_index.py`

Добавить новый модуль `src/claude_manager/codex_session_index.py`. Он отвечает только за operational-индекс Codex rollout-файлов и не знает о Telegram, watcher, pending-delivery или project switching.

Публичный API модуля:

```python
async def list_project_session_file_infos(
    sessions_root: str,
    project_dir: str,
    lookback_days: int,
    today: date | None = None,
) -> list[SessionFileInfo]:
    """Return lightweight Codex session files for one project from the operational index."""


async def refresh_known_session(
    sessions_root: str,
    session_id: str,
    project_dir: str | None = None,
    today: date | None = None,
) -> SessionFileInfo | None:
    """Refresh one known Codex session in the operational index when its file can be located."""


def clear_cache() -> None:
    """Clear the in-memory Codex operational index, primarily for tests."""
```

Допускается добавить `list_project_session_file_infos_by_project(...)`, если реализация сразу подключает bulk-сценарий для нескольких проектов. Если bulk-сценарий не нужен для `/pN`, его лучше не добавлять в первой итерации.

Внутренние структуры:

- `_IndexKey` — `sessions_root`, `lookback_days`, `window_end_date`.
- `_IndexState` — `built_at_monotonic`, `directory_signature`, `files_by_project`.
- `_IndexedRollout` — `session_id`, `file_path`, `project_dir`, `last_modified_at`.

`files_by_project` должен хранить только лёгкие объекты. `preview` в `SessionFileInfo` для operational-пути остаётся пустой строкой.

### Построение индекса

Алгоритм полной пересборки:

1. Построить окно дат от `today` назад на `lookback_days` дней.
2. Прочитать только существующие директории `~/.codex/sessions/YYYY/MM/DD/` внутри окна.
3. Собрать `rollout-*.jsonl`.
4. Для каждого файла прочитать только `session_meta`, извлечь `payload.cwd` и `payload.id`.
5. Если `payload.cwd` является строкой, добавить файл в `files_by_project[payload.cwd]`.
6. Если `payload.id` отсутствует, извлечь UUID из имени rollout-файла.
7. Получить `mtime` через `os.path.getmtime`.
8. Для каждого проекта отсортировать список по `last_modified_at` по убыванию.

Чтение meta-записей должно идти с bounded parallelism. Лимит оставить в том же порядке, что сейчас: 16 одновременных file-read операций.

### Инвалидация индекса

Перед переиспользованием индекса модуль проверяет подпись date-директорий. Подпись должна быть дешёвой и не читать JSONL:

- путь date-директории;
- `st_mtime_ns` директории;
- количество rollout-файлов в директории.

Если подпись изменилась — пересобрать весь индекс для окна. Это покрывает создание, удаление и переименование rollout-файлов.

Если подпись не изменилась — индекс можно переиспользовать, но перед возвратом файлов конкретного проекта нужно обновить `mtime` только для файлов этого проекта:

1. `os.path.getmtime(file_path)` для каждого файла проекта;
2. удалить из результата исчезнувшие файлы;
3. вернуть список с актуальными `last_modified_at`;
4. пересортировать список, если `mtime` изменился.

TTL нужен только как страховка от редких случаев, когда файловая система не обновила mtime директории. Рекомендуемое значение: `CODEX_SESSION_INDEX_TTL_SECONDS = 30`. Истечение TTL не должно быть единственным способом заметить новые файлы.

При смене календарного дня cache key меняется через `window_end_date`, поэтому старые дни выпадают из окна естественно при следующем обращении.

### Точечное обновление известной session_id

`refresh_known_session(...)` нужен для случаев, когда код уже знает `session_id` и не должен ждать полной пересборки:

1. Построить вероятные пути через UUIDv7 timestamp, как уже делает `_candidate_paths_from_uuid_v7`.
2. Если файл найден, прочитать его `session_meta`.
3. Если `project_dir` передан, принять файл только при совпадении `payload.cwd == project_dir`.
4. Обновить или добавить запись в текущий индекс, если файл попадает в 4-дневное окно.
5. Вернуть актуальный `SessionFileInfo` или `None`.

Если UUIDv7-путь не найден, полный `os.walk` запускать нельзя. Такой fallback оставляем существующему `session_file_exists_for_project`, потому что он не должен попадать в горячий `/pN` путь.

### Интеграция с Codex backend

`src/claude_manager/codex_session_file_listing.py` остаётся модулем совместимости для текущего API листинга. Основная новая логика не добавляется туда.

Изменения:

- `list_all_session_file_infos_for_project(..., lookback_days=N)` при `lookback_days is not None` делегирует в `codex_session_index.list_project_session_file_infos(...)`.
- Ветка `lookback_days is None` сохраняет прежний полный scan для совместимости.
- `list_session_file_infos_for_project(...)` для пользовательского `/sessions` может остаться без индекса, потому что ему нужен preview и другой лимит свежих сессий.
- `session_file_exists_for_project(...)` может использовать `refresh_known_session(...)` перед текущим fallback-сканом, но не обязан блокировать первую итерацию.

`src/claude_manager/codex_backend.py` сохраняет публичный контракт. Его `list_all_session_files_for_project` продолжает пробрасывать `lookback_days`; поведение меняется внутри listing-модуля.

`config.OPERATIONAL_SESSION_LOOKBACK_DAYS` обновить с 2 до 4. Старое значение 2 появилось как защита от постоянного full scan в poll/reset/pending. После индекса горячий путь больше не платит global scan на каждый вызов, поэтому выбранное в handoff 4-дневное окно становится допустимым.

### Pending mtime/cursor-first

Сейчас `project_manager._collect_pending_for_session_file` читает полный snapshot для каждой сессии, если в `unread_buffer` есть старый cursor. Это дорого, когда pending нет.

Нужно расширить cursor-состояние:

- `SessionUnreadState` в `coding_agent_backend.py` получает поле `last_modified_at: float | None = None`.
- `SessionWatcherState` в `session_file_polling_cursors.py` получает поле `last_modified_at: float | None = None`.
- `unread_buffer.save_snapshot(...)` принимает `last_modified_at: float | None = None`.
- `session_watcher.get_seen_counts_snapshot(...)` возвращает `SessionUnreadState` с `last_modified_at`.

Watcher должен записывать `last_modified_at` из `SessionFileInfo` в состояние при:

- `reset_state`;
- `poll_once` после успешного чтения snapshot;
- `resume_session`.

Новый алгоритм pending для одного файла:

```python
old_state = unread_buffer.restore_snapshot(file_info.session_id, backend)
if old_state is None:
    return []

if (
    old_state.last_modified_at is not None
    and file_info.last_modified_at <= old_state.last_modified_at
):
    return []

cursor = await backend_adapter.read_session_file_cursor(file_info.file_path)
if cursor.raw_record_count <= old_state.raw_record_count:
    return []

snapshot = await backend_adapter.read_session_file_snapshot(file_info.file_path)
if not _has_new_messages(old_state, snapshot):
    return []

delta = snapshot.messages[old_state.last_delivered_idx + 1:]
return _build_pending_items_from_delta(...)
```

Почему нельзя сравнивать с `saved_at` из `unread_buffer`: файл мог получить новый ответ прямо перед переключением, но watcher ещё не успел его прочитать. Тогда `file_mtime <= saved_at`, а старый cursor всё ещё не содержит новое сообщение. Нужен именно `last_modified_at`, который watcher видел вместе с сохранённым cursor-ом.

### Bounded parallelism для pending

После mtime/cursor-фильтра `_collect_pending_for_backend` должен обрабатывать session files параллельно с лимитом 16. Это вторичный слой оптимизации: сначала убрать лишние полные чтения, затем параллелить только оставшиеся проверки.

Требование: ошибки одного файла логируются и не отменяют pending-сбор остальных файлов.

### Cancellation warning в Codex listing

RCA зафиксировал риск в `_gather_optional_results_with_concurrency_limit`: сейчас в него передаются заранее созданные coroutine objects. При отмене часть coroutine может остаться unawaited.

В рамках реализации нужно исправить этот паттерн одним из способов:

- заменить helper так, чтобы он принимал zero-arg async factories и создавал coroutine только внутри semaphore;
- или создавать tasks для всех входов и при `CancelledError` явно cancel/drain через `asyncio.gather(..., return_exceptions=True)`.

Нельзя чинить это большим дописыванием в `codex_session_file_listing.py`, потому что файл уже на границе 500 строк.

## Ограничения по размеру файлов

Перед кодовым проходом важно явно учесть проектное правило размеров:

- `src/claude_manager/project_manager.py` — 597 строк, уже выше error-порога 500.
- `src/claude_manager/coding_agent_session_file_poller.py` — 653 строки, уже выше error-порога 500.
- `src/claude_manager/codex_session_file_listing.py` — 494 строки, почти у error-порога 500.

- `src/claude_manager/codex_backend.py` — 319 строк, выше warning-порога 300.

Практическое правило для реализации:

- Вся логика индекса — только в новом `codex_session_index.py`.
- В `project_manager.py` и `coding_agent_session_file_poller.py` допустимы только точечные изменения wiring/state полей. Если изменение начинает добавлять заметные блоки логики, сначала нужен отдельный разрез файла или явное решение владельца проекта.
- В `codex_session_file_listing.py` нельзя просто дописать индекс. Нужно делегировать в новый модуль или заменить существующий код без роста за 500 строк.

## Тест-план

### Новый `tests/test_codex_session_index.py`

Добавить тесты:

- **index_groups_rollouts_by_project** — несколько rollout-файлов разных проектов попадают в разные списки.
- **index_keeps_only_sliding_lookback_window** — при `lookback_days=4` файлы старше окна не возвращаются.
- **repeated_calls_reuse_index_without_meta_reread** — два вызова подряд для разных проектов не перечитывают `session_meta` всех rollout-файлов повторно.

- **known_session_refresh_updates_one_file** — `refresh_known_session` обновляет конкретную сессию по UUIDv7-пути.
- **concurrent_callers_share_one_rebuild** — параллельные вызовы к холодному индексу не запускают несколько одинаковых пересборок.
- **cancelled_rebuild_does_not_leave_unawaited_coroutines** — отмена не оставляет предупреждений про never-awaited coroutine.

### `tests/test_project_manager.py`

Добавить тесты на pending:

- pending-сбор нескольких изменившихся файлов идёт с bounded parallelism и не падает целиком при ошибке одного файла.

## Критерии готовности

- Повторные вызовы Codex operational listing в рамках одного свежего окна не перечитывают `session_meta` всех rollout-файлов.
- Новые rollout-файлы появляются через directory signature или точечный refresh, без ожидания одного только TTL.
- Pending no-op не читает полный snapshot неизменившихся файлов.
- UX `/pN` не изменён: подтверждение переключения остаётся после pending-проверки.
- Тесты покрывают индекс, pending mtime/cursor-first, watcher `last_modified_at` и отсутствие повторного global scan.
- Новая логика не раздувает существующие файлы за пороги без явного сигнала владельцу проекта.
- Живой замер показывает существенное снижение задержки переключения; целевой ориентир — около 1-2 секунд.
