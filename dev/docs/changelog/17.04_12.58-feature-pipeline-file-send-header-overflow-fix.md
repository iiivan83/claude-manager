# Changelog: Фикс переполнения первого чанка при отправке текстового файла

## Дата: 17.04.2026

## Тип: Исправление бага (small)

## Что изменилось

Исправлен баг в отправке текстовых файлов через маркеры `[SEND_FILE:path]`. Суть проблемы: в функции `_send_text_file` (обработчик отправки файла в `bot.py`) заголовок с именем файла приклеивался к первому чанку ПОСЛЕ разбивки текста на части. Разбивка через `split_entities` (функция из библиотеки `telegramify-markdown`, которая делит длинный текст на куски до 4096 символов) не знала про заголовок — и итоговый первый чанк мог превысить лимит Telegram в 4096 символов. Telegram отвечал ошибкой `BadRequest: Message is too long`.

Решение — резервирование места под заголовок ДО разбивки. Функция `render_file_for_telegram` (утилита в `file_sender.py`, которая рендерит содержимое файла через telegramify-markdown) получила новый опциональный параметр `first_chunk_reserve`. Это количество символов (в единицах UTF-16), которое нужно зарезервировать для заголовка. Все чанки становятся чуть короче (на величину резерва), зато первый гарантированно помещается с заголовком.

Потеря места в чанках 2+ минимальна — около 2-3% от лимита при типичном имени файла.

## Затронутые файлы

- **src/claude_manager/file_sender.py** — функция `render_file_for_telegram` получила параметр `first_chunk_reserve: int = 0`. Вычисляет `effective_limit = TELEGRAM_MESSAGE_LIMIT - first_chunk_reserve` и передаёт его в `split_entities`. При абсурдном значении резерва (больше или равно лимиту) — fallback на `TELEGRAM_MESSAGE_LIMIT // 2` с warning в лог
- **src/claude_manager/bot.py** — в `_send_text_file` заголовок формируется ДО вызова `render_file_for_telegram`. Длина заголовка в UTF-16 передаётся как `first_chunk_reserve`
- **tests/test_file_sender.py** — 3 новых теста: резерв уменьшает лимит чанков, резерв 0 не ломает обратную совместимость, абсурдный резерв использует fallback
- **tests/test_bot.py** — 2 новых теста в классе `TestSendTextFile`: проверка что `render_file_for_telegram` вызывается с корректным резервом, проверка корректного расчёта для длинного имени файла (200 символов)
