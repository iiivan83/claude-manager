## Направление 1 — Конфигурация и инфраструктура

### Проблема 1.1: Архитектурный документ удалён, но pipeline-spec.md ссылается на него
- **Где:** pipeline-spec.md, строки 20, 154, 284-288, 335, 355, 372
- **Суть:** Pipeline-spec ссылается на «Архитектурный документ» как входной артефакт для скиллов validate-brd, brd-decompose и spec-module. Документ был удалён. CLAUDE.md написан по старой спецификации.
- **Решение:** Не использовать ни CLAUDE.md, ни архитектурный документ как источник правды. BRD — единственный первоисточник. При реализации принимать оптимальные решения на основе BRD, результаты вписывать в CLAUDE.md. Pipeline-spec.md исправить — убрать ссылки на архитектурный документ.
- **Принял:** пользователь

### Проблема 1.2: CLAUDE.md пишет «Два основных документа», а остался только BRD
- **Где:** CLAUDE.md, секция «Техническая документация»
- **Суть:** Написано «Два основных документа описывают проект», но перечислен только один (BRD).
- **Решение:** Исправится автоматически в фазе 7 (обновление документации). CLAUDE.md будет переписан по реальному коду.
- **Принял:** аудитор

### Проблема 1.3: CLAUDE.md описывает целевую структуру, а не текущую
- **Где:** CLAUDE.md, секция «Структура проекта»
- **Суть:** Описаны 10 файлов в src/claude_manager/, но реально существуют только 3 (__init__.py, __main__.py, main.py). Также не упомянуты __init__.py и __main__.py.
- **Решение:** Ожидаемо для проекта до реализации. Исправится автоматически в фазе 7. При написании спецификаций (фаза 2) — ориентироваться на BRD, а не на CLAUDE.md.
- **Принял:** аудитор

### Проблема 1.4: main.py использует print() вместо logging
- **Где:** src/claude_manager/main.py
- **Суть:** Заглушка с print("Hello from claude_manager!") нарушает правило «Логирование через logging — не через print()».
- **Решение:** Оставить — project-setup (фаза 1) перепишет main.py.
- **Принял:** аудитор

### Проблема 1.5: .env.example не содержит Telethon-переменных
- **Где:** .env.example, pipeline-spec.md (S1)
- **Суть:** Pipeline-spec требует TELETHON_API_ID, TELETHON_API_HASH, TELETHON_PHONE, TELETHON_BOT_USERNAME. В .env.example их нет.
- **Решение:** Скилл project-setup добавит в фазе 1.
- **Принял:** аудитор

### Проблема 1.6: Отсутствуют тестовые зависимости
- **Где:** pyproject.toml, requirements.txt
- **Суть:** pytest, pytest-asyncio, pytest-mock, telethon не указаны. Команда python -m pytest tests/ из CLAUDE.md не работает.
- **Решение:** Скилл project-setup добавит в фазе 1.
- **Принял:** аудитор

## Направление 3 — Противоречия

### Проблема 3.1: Pipeline-spec исключает фазу 6 из циклов починки, но review-code содержит циклы

## Направление 4 — Полнота сценариев

### Проблема 4.1: Сообщение без активной сессии теряется
- **Где:** BRD, CJM-02 (строки 82-86)
- **Суть:** BRD не описывает, что происходит с сообщением, когда пользователь пишет без активной сессии. Неявно — оно теряется.
- **Решение:** Явно указать в BRD: сообщение не сохраняется, пользователю нужно написать заново после создания сессии.
- **Принял:** пользователь

### Проблема 4.2: Карта состояний — убрать Состояние 1
- **Где:** BRD, «Карта состояний пользователя» (строки 352-385)
- **Суть:** Пользователь видит 3 состояния, но Состояние 1 («Чистый лист») не имеет практического смысла — бот должен сразу что-то делать.
- **Решение:** Убрать Состояние 1. Оставить 2 состояния: «Подключён к сессии» и «/all (мониторинг)». При первом запуске и при перезапуске без сохранённой привязки — бот автоматически входит в режим /all.
- **Принял:** пользователь

### Проблема 4.3: Что происходит после 10 неудачных ретраев
- **Где:** BRD, CJM-02 (строка 123), «Обработка ошибок Claude» (строки 443-447)
- **Суть:** Не описан финальный результат после исчерпания 10 попыток.
- **Решение:** Добавить: «Не удалось получить ответ от Claude после 10 попыток. Попробуйте снова.» Команда /stop прерывает цикл ретраев.
- **Принял:** пользователь

### Проблема 4.4: /stop когда Claude не работает
- **Где:** BRD, CJM-08 (строки 302-328)
- **Суть:** Не описано поведение /stop, когда Claude не обрабатывает запрос.
- **Решение:** Если Claude не работает — бот отвечает «Claude сейчас не работает, нечего останавливать». /stop также прерывает цикл ретраев.
- **Принял:** пользователь

### Проблема 4.5: Когда /new запускает процесс Claude
- **Где:** BRD, CJM-04 (строки 177-182)
- **Суть:** Не описано, создаётся ли процесс Claude сразу при /new или только при первом сообщении.
- **Решение:** /new сразу запускает процесс Claude с working directory из CLAUDE_WORKING_DIR. Процесс ждёт первого сообщения.
- **Принял:** пользователь

### Проблема 4.6: Состояние после перезапуска
- **Где:** BRD, «Восстановление после перезапуска» (строки 449-454)
- **Суть:** Не описано, в каком состоянии пользователь после перезапуска — особенно если был в /all.
- **Решение:** При перезапуске: если была привязка к сессии — восстановить (Состояние 2). Иначе — автоматически войти в /all.
- **Принял:** пользователь

### Проблема 4.7: Сообщение при попытке написать в режиме /all
- **Где:** BRD, CJM-07 (строка 297)
- **Суть:** Нет конкретного текста ошибки для попытки написать в режиме /all.
- **Решение:** Добавить: «Вы в режиме мониторинга. Для отправки сообщений подключитесь к сессии — нажмите на номер сессии или отправьте /new».
- **Принял:** аудитор

### Проблема 4.8: Промежуточные обновления — первые 30 секунд тишины
- **Где:** BRD, CJM-02 (строки 93-94, 113-114)
- **Суть:** Если Claude отвечает быстрее 30 секунд, промежуточных обновлений не будет. Пользователь видит только «печатает...».
- **Решение:** Оставить как есть. Индикатор «печатает...» достаточен для первых 30 секунд.
- **Принял:** аудитор

### Проблема 6.1: Конфликт PID-файлов между watch_and_restart.sh и ботом

### Ключевые решения, влияющие на реализацию

1. **BRD — единственный первоисточник.** CLAUDE.md и архитектурный документ написаны по старой спецификации, не использовать для принятия решений.
2. **Два состояния вместо трёх.** Убрать «Чистый лист». Бот всегда либо подключён к сессии, либо в режиме /all.
3. **/new сразу запускает Claude.** Процесс создаётся немедленно, не ждёт первого сообщения.
4. **/N ищет везде.** Не только в дневном реестре, но и среди всех видимых сессий.
5. **/stop прерывает ретраи.** Пользователь может остановить цикл повторных попыток.
6. **Автоочистка received_files/.** Файлы старше 7 дней удаляются при запуске бота.
7. **При перезапуске — /all по умолчанию.** Если привязки к сессии нет — автоматически в мониторинг.
