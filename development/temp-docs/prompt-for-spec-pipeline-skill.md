# Промпт для skill-creator: создание скилла spec-pipeline

Этот промпт передаётся скиллу skill-creator для создания нового скилла.

---

THINK HARD. Используй максимальную глубину мышления на каждом шаге.

ВАЖНО: В SKILL.md созданного скилла укажи инструкцию — при выполнении скилл обязан использовать THINK HARD (максимальную глубину мышления) на каждом шаге.

Создай скилл `spec-pipeline`.

## Назначение

Универсальный скилл для создания детальных спецификаций инженерных пайплайнов. Принимает на вход примерный план пайплайна и спецификацию (BRD) проекта, на выходе даёт готовую к реализации спеку пайплайна с двумя частями: инструментарий (что создать или что изменить) и процесс (как оркестратор это использует).

Скилл должен быть **универсальным** — работает в любом проекте, не привязан к конкретной технологии или архитектуре.

## Когда использовать

Когда пользователь хочет: создать спецификацию пайплайна, спроектировать пайплайн реализации, «сделай спеку на пайплайн», «спроектируй пайплайн», «spec pipeline», «pipeline spec», планирует систематическую реализацию проекта через автоматизированный пайплайн.

НЕ использовать: для создания отдельных скиллов (есть skill-creator), для написания кода, для обновления существующих спецификаций.

## Входные данные

Скилл ожидает от пользователя:

1. **Примерный план пайплайна** — какие фазы/этапы пользователь представляет (может быть черновым, скилл доработает)
2. **Спецификация для реализации** — BRD, техническое описание, архитектурный документ — то, что пайплайн будет реализовывать
3. **Контекст проекта** — скилл сам читает CLAUDE.md, существующие документы, структуру проекта

Если пользователь не предоставил что-то — попросить.

## Критическое правило: ЗАДАВАТЬ ВОПРОСЫ

Это финальный этап перед реализацией. Если при создании спеки возникают вопросы или непонимание как что-то реализовать технически — ОБЯЗАТЕЛЬНО спросить пользователя. Не додумывать. Не оставлять на потом. Если техническое решение неясно — оно должно быть разрешено в спеке, иначе спека бесполезна.

Обязательно спрашивать:

### Вопросы по структуре проекта
- **Куда складывать файлы** — для каждого типа артефактов (спецификации, документы тестирования, отчёты, скрипты) предложить варианты размещения через AskUserQuestion. Не решать за пользователя.
- **Именование файлов** — предложить конвенцию именования и спросить подтверждение

### Вопросы по технической реализации
- **Технологии и решения** — если из кода и документации НЕ понятно какие технологии используются, какие библиотеки, какие протоколы, какие форматы данных — ОБЯЗАТЕЛЬНО спросить у пользователя. Не угадывать. Примеры: «Какой протокол общения между компонентами?», «Какая библиотека используется для X?», «Какой формат у файлов сессий?», «Как именно запускается внешний процесс?»
- **Как тестировать** — какой фреймворк для тестов (pytest, jest, go test...), мокать или живое тестирование, какие инструменты для E2E
- **Как делать E2E** — если есть внешние системы (API, Telegram, база данных) — как с ними взаимодействовать в тестах? Мокать или реальные? Если реальные — какой инструмент?
- **Кто делает ревью** — Claude автоматически или человек?
- **Уровень автоматизации оркестратора** — полный автомат, полуавтомат или ручной?
- **Зависимости от существующих скиллов** — новые скиллы самостоятельные или используют существующие?
- **Неясные технические детали** — любое место в спецификации, где непонятно КАК реализовать — спросить. Спека бесполезна если в ней есть шаги которые технически непонятны

### Вопросы по процессу
- **Что на входе пайплайна** — готовый BRD или пайплайн должен создавать BRD?
- **Какие фазы нужны** — после анализа предложить структуру фаз и спросить подтверждение
- **Дополнительные фазы** — нужно ли пользовательское тестирование отдельной фазой? Нужна ли фаза документации?

- Определить: язык, фреймворки, внешние системы, зависимости (package.json / pyproject.toml / go.mod)

**3. Существующие скиллы:**

**4. Существующие документы, спеки, скрипты:**
- Просканировать папки документации — что уже описан��
- Просканировать папки спецификаций — есть ли готовые ТЗ на модули
- Просканировать папки скриптов — есть ли вспомогательные утилиты
- Определить: какие артефакты уже существуют и пайплайн должен их ОБНОВ��ЯТЬ, а не создавать заново

**5. Тесты:**

**6. Результат анализа:**
Составить список:
- **Существует и актуально** — оставить как есть, пайплайн использует
- **Существует но нужно обновить** — пайплайн обновит в соответствующей фазе
- **Не существует — нужно создать** — пайплайн создаёт с нуля

### Шаг 2: Глубокий технический аудит — проверка ВСЕГО проекта

THINK HARD.

Это НЕ просто чтение текста BRD. Это полный аудит проекта — документация, конфигурация, код, файлы. Пайплайн реализует то что написано в документах, и если в них есть дыры, противоречия или сломанные конфиги — всё это перейдёт в код.

Скилл должен найти ВСЕ возможные проблемы. Ниже — направления поиска (не закрытый список, а ориентиры). Если найдена проблема, которая не попадает ни в одну категорию — всё равно зафиксировать и спросить.

**Направление 1 — Конфигурация и инфраструктура:** всё что связано с запуском, сборкой, зависимостями. Неправильные зависимости, версии языка, отсутствие точки входа, сломанные скрипты, битые симлинки, отсутствие .env.example.

**Направление 2 — Целостность файлов и ссылок:** каждый файл/документ упомянутый где-либо — существует ли? Каждая ссылка ведёт куда нужно?

**Направление 3 — Противоречия:** любой факт упомянутый более чем в одном месте — совпадает ли? Числа, имена, пути, поведение. Документ противоречит сам себе?

**Направление 4 — Полнота сценариев:** каждый путь завершён? Каждая ошибка обработана? Каждый переход покрыт? Какие граничные ситуации пропущены?

**Направление 5 — Технические пробелы:** где описано ЧТО но не КАК. Открытые вопросы, TODO, абстрактные формулировки. Есть ли более простые решения?

**Направление 6 — Всё остальное:** любая проблема, несоответствие, подозрительное место. Если сомневаешься — лучше спросить пользователя чем промолчать.

**Что делать с найденным:**
1. Составить полный список по ВСЕМ 5 категориям
2. Для каждой проблемы — спросить пользователя через AskUserQuestion с вариантами решения
3. Зафиксировать решения — они войдут в спеку пайплайна как уточнения
4. НЕ ПЕРЕХОДИТЬ к следующему шагу пока ВСЕ проблемы не разрешены

Этот шаг критичен: каждое неразрешённое расхождение = баг в коде, который найдётся только при тестировании или у пользователя.

### Шаг 3: Вопросы по пайплайну

THINK HARD.

Проанализировать входные данные и задать ВСЕ необходимые вопросы через AskUserQuestion:

1. **Структура фаз** — предложить набор фаз на основе анализа и спросить подтверждение
2. **Технический стек тестирования** — определить конкретные инструменты для каждого уровня тестирования
3. **Размещение файлов** — для каждого типа (спеки, тесты, доки, скрипты) предложить варианты
4. **Автоматизация** — полный автомат / полуавтомат / ручной
5. **Ревью** — Claude / человек / оба
6. **Самостоятельность скиллов** — зависят от существующих или нет

Не переходить к шагу 4, пока ВСЕ технические вопросы не разрешены.

### Шаг 4: Проектирование инструментария (Часть 1)

THINK HARD.

Используя результаты анализа из Шага 1 (что существует / что обновить / что создать), для каждой фазы определить:
- Какие **скиллы** нужно создать с нуля, а какие уже есть и их можно использовать
- Какие **документы** будут создаваться, а какие уже есть и их нужно обновить
- Какие **скрипты** нужны, а какие уже существуют

Для каждого инструмента явно указать: **создать** / **обновить** / **использовать как есть**

Правила именования:
- Названия скиллов отражают что они делают, без лишних префиксов
- Если скилл универсальный — НЕ добавлять имя пайплайна в название
- Только оркестратор может содержать слово «pipeline» в названии
- Документы с возможными дублями получают суффикс даты-времени `_ДД-ММ-ЧЧ-ММ`

### Шаг 5: Проектирование процесса (Часть 2)

THINK HARD.

1. Построить схему процесса (ASCII-диаграмму потока фаз)
2. Для каждой фазы написать чеклист с воротами качества
3. Описать циклы исправления (выполнить → проверить → починить → перепроверить → до чистоты)
4. Описать обработку ошибок
5. Написать ТЗ на скилл-оркестратор

### Шаг 6: Проверка технической полноты (ДО написания)

THINK HARD.

Перед тем как писать спеку — пройтись по КАЖДОМУ скиллу/инструменту из Шагов 3-4 и для каждого проверить:
- Понятно ли КАК он будет реализован технически?
- Какие библиотеки/инструменты/протоколы используются?
- Есть ли в проекте примеры аналогичной реализации?

Если хоть один технический вопрос остался без ответа — **ОСТАНОВИТЬСЯ и спросить пользователя**. Не писать спеку с пробелами.

Цель: в готовой спеке каждый шаг каждого скилла должен быть реализуем без дополнительных вопросов. Разработчик (или Claude) берёт спеку и реализует — не гуглит, не спрашивает, не додумывает.

### Шаг 8: Верификация

THINK HARD.

Перечитать созданную спеку и проверить:
1. Каждая фаза имеет конкретные инструменты из Части 1
2. Каждый скилл имеет промпт для создания через skill-creator
3. Каждый промпт содержит THINK HARD + инструкцию что созданный скилл тоже должен THINK HARD

8. Циклы исправления описаны для каждой фазы
9. После любого исправления — прогон ВСЕХ тестов (регрессионная защита)

## Обязательные правила для спецификации

Скилл ОБЯЗАН соблюдать при создании спеки:

1. **Нет «будущих» фич** — если фича нужна, она в спеке сейчас. Слова «потом», «позже», «в будущем» запрещены
2. **Нет ручных действий** — для каждого действия есть скилл или скрипт. Если нет подходящего скилла — определить что нужно создать
3. **Конкретные технические решения** — не «тестирование», а «pytest + pytest-asyncio + pytest-mock с фикстурами mock_bot, mock_update». Не «E2E через API», а «Telethon/Playwright/Cypress/конкретный инструмент»
4. **Все промпты с THINK HARD** — каждый промпт для создания скилла начинается с THINK HARD и содержит инструкцию что созданный скилл тоже должен THINK HARD
5. **Реализованные спеки → realized/** — скиллы, реализующие спецификации, переносят их в подпапку `realized/` после реализации
6. **Дата в именах дублируемых документов** — `_ДД-ММ-ЧЧ-ММ` (день-месяц-час-минута) в конце имени
7. **Документы тестирования — в отдельной папке** — путь согласуется с пользователем
8. **Универсальные имена скиллов** — без привязки к проекту. Только оркестратор содержит слово «pipeline»
9. **Циклы починки в каждой фазе** — выполнить → проверить → починить → ВСЕ тесты → повторить (макс 5 итераций)
10. **Регрессионная защита** — после ЛЮБОГО исправления прогоняются ВСЕ тесты предыдущих фаз
11. **Ворота качества** — чёткий чеклист между фазами, нельзя перейти дальше пока не пройден
12. **Состояние в JSON** — оркестратор хранит прогресс в JSON-файле для возобновления после прерывания
13. **Git-коммиты** — коммит после каждого завершённого артефакта (спека, модуль, исправление). Пуш — только после подтверждения пользователя
14. **Параллелизм через агентов** — независимые задачи одного слоя выполняются параллельно через Agent tool
15. **Интерактивные шаги** — оркестратор останавливается и спрашивает пользователя при: авторизации, пуше, вопросах по BRD
16. **Очистка после тестов** — отдельный шаг/фаза для удаления тестовых данных после живого тестирования
17. **Возврат спеки** — если спека модуля оказалась неправильной при реализации, она возвращается из realized/ в specs/, исправляется и коммитится

## Пример вопросов, которые скилл должен задать

Через AskUserQuestion (не текстом в чате):

- «Оркестратор — полный автомат, полуавтомат или ручной?»
- «Кто делает code review — Claude автоматически или ты?»
- «Куда складывать спецификации модулей?» — варианты: development/specs/, docs/specs/, specs/
- «Куда складывать документы тестирования?» — варианты: development/docs/testing/, tests/reports/, docs/testing/
- «Какой фреймворк для тестов?» — варианты зависят от языка проекта
- «E2E тестирование — с моками или живое?»
- «Если живое — какой инструмент для взаимодействия с внешними системами?» — варианты зависят от системы
- «Скиллы пайплайна зависят от существующих скиллов проекта или самостоятельные?»
- «Нужна ли отдельная фаза пользовательского тестирования в конце?»
- «Пушить изменения в remote-репозиторий автоматически или спрашивать каждый раз?»
- «Нужна ли фаза очистки тестовых данных отдельно или как шаг внутри тестовых фаз?»
- «Есть ли интерактивные авторизации (Telethon, OAuth, etc.) которые нужно выполнить до E2E?»
