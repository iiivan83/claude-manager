# Спецификация: Инженерный пайплайн реализации проекта

## Контекст

Спецификация на создание полностью автоматизированного пайплайна, который ведёт проект от готового BRD до production-ready кода. Пайплайн сам идёт от этапа к этапу, останавливается только при ошибке.

Дата: 29-03-2026

## Обзор

Два раздела:

**Часть 1 — Инструментарий:** какие скиллы, документы и скрипты нужно создать, чтобы пайплайн работал. Для каждой сущности — промпт и инструмент для её создания.

**Часть 2 — Процесс:** схема работы + чеклист + ТЗ на скилл-оркестратор (`pipeline-run`), который объединяет все инструменты из Части 1 и запускает их в логичном порядке. Полный автомат.

## Входные данные пайплайна

- Готовый BRD с описанием пользовательских сценариев
- Документ архитектуры проекта (модули, слои, зависимости)
- CLAUDE.md с правилами проекта

## Принципы

1. **Спецификации ДО кода** — каждый модуль специфицируется перед реализацией
2. **Снизу вверх** — реализация по слоям зависимостей
3. **Тесты рядом с кодом** — ю��ит-тесты пишутся вместе с модулем
4. **Циклы починки** — в каждой фазе: выполнить → проверить → починить → перепроверить → до чистоты
5. **Регрессионная защита** — после любого исправления перегоняются ВСЕ тесты
6. **Ворота качества** — нельзя перейти к следующей фазе, пока не пройдена текущая
7. **Полная автомат��зация** — оркестратор сам идёт от этапа к этапу
8. **LLM-верификация** — Claude проверяет ответы и результаты на соответствие BRD

## Технический стек тестирования

Три уровня тестирования используют разные инструменты:

### Юнит и интеграционные тесты (Фазы 2, 3) — с моками

Тесты быстрые (миллисекунды), детерминированные (один и тот же результат при каждом запуске), не требуют сети.

**Фреймворк:**
- **pytest** — запуск тестов
- **pytest-asyncio** — поддержка async/await в тестах (бот полностью асинхронный)
- **pytest-mock** — удобные моки (подмена реальных объектов фейковыми)

**Что мокается (подменяется фейковым):**

- **Telegram Bot API** — фейковые объекты `Update` (входящее сообщение), `Message`, `Chat`, `User`. Фейковый `Bot.send_message()`, который записывает отправленные сообщения вместо реальной отправки. Фейковый `Bot.get_file()` для тестов скачивания файлов
- **Claude CLI** — фейковый subprocess (процесс), который принимает сообщения через stdin и отдаёт заранее подготовленные stream-json события через stdout. Имитирует задержки и различные типы ответов (текст, tool_use, ошибки)
- **Файловая система** — временные папки (`tmp_path` фикстура pytest) для сессий, JSON-файлов, скачанных файлов

**Тестовые фикстуры (готовые объекты для тестов):**
- `mock_bot` — фейковый Telegram-бот, записывает все вызовы send_message
- `mock_update` — генератор фейковых входящих сообщений с настраиваемым текстом, chat_id, user_id
- `mock_claude_process` — фейковый процесс Claude, отвечает заранее заданными stream-json событиями
- `allowed_user_id` — ID пользователя из белого списка (для тестов авторизации)
- `session_dir` — временная папка, имитирующая `~/.claude/projects/`

### E2E и пользовательское тестирование (Фазы 4, 8) — живое

Тесты работают с реальным Telegram и реальным Claude. Используют реальный аккаунт пользователя.

**Инструмент: Telethon**
- Python-библиотека для работы с Telegram через пользовательский аккаунт (не через Bot API)
- Подключается от имени реального пользователя — бот не может отличить Telethon от обычного Telegram-приложения
- Может: отправлять сообщения ботам, читать ответы, отправлять фото/файлы, читать кнопки

**Авторизация Telethon (одноразовая):**
1. Получить `API_ID` и `API_HASH` на https://my.telegram.org (раздел API development)
2. При первом запуске — ввести номер телефона и код из SMS
3. Telethon сохраняет сессию в файл `telethon_test.session` — повторная авторизация не нужна
4. Файл сессии добавляется в `.gitignore` (содержит токен авторизации)

**Переменные окружения для E2E:**
- `TELETHON_API_ID` — ID приложения с my.telegram.org
- `TELETHON_API_HASH` — Hash приложения с my.telegram.org
- `TELETHON_PHONE` — номер телефона для авторизации
- `TELETHON_BOT_USERNAME` — username бота для отправки сообщений (например `@my_claude_bot`)

**Тестовый клиент (обёртка над Telethon):**
- `TelegramTestClient` — класс, который упрощает работу с Telethon в тестах:
  - `send_message(text)` — отправить сообщение боту
  - `send_photo(path, caption)` — отправить фото боту
  - `wait_for_response(timeout=30)` — дождаться ответа от бота
  - `get_last_response()` — получить текст последнего ответа
  - `send_command(command)` — отправить команду (например `/new`, `/sessions`)
- Этот класс создаётся как часть тестовой инфраструктуры в `tests/e2e/test_client.py`

**Как работает E2E тест:**
1. `TelegramTestClient` подключается к Telegram через Telethon
2. Отправляет сообщение боту: `client.send_message("Привет")`
3. Ждёт ответ: `response = client.wait_for_response(timeout=60)`
4. Claude (LLM-верификация) сравнивает `response` с ожиданием из BRD
5. Результат: pass или fail

### S10: run-user-testing

**Назначение:** Прогоняет все сценарии пользовательского тестирования. Записывает результаты. При ошибке — исправляет и повторно прогоняет конкретный сценарий.

**Вход:**
- `development/docs/testing/user-test-scenarios_*.md` (последний файл)
- Рабочий бот
- `.env` с реальными данными

**Выход:**
- `development/docs/testing/user-test-results_ДД-ММ-ЧЧ-ММ.md` — результаты
- Все сценарии пройдены (pass)

**Алгоритм:**
1. Прочитать файл сценариев (последний по дате)
2. Запустить бота: `python -m claude_manager`
3. Проходить сценарии по порядку:
   - Выполнить шаги сценария
   - Проверить результат по критериям pass/fail
   - **LLM-верификация:** Claude оценивает ответ бота — соответствует ли он ожидаемому поведению из BRD
   - Записать результат: pass или fail + что именно не сошлось
4. **Цикл починки:** если сценарий fail:
   - Остановить бота
   - Найти и исправить баг в коде
   - Запустить ВСЕ автоматические тесты (юнит + интеграционные) — регрессионная проверка
   - Перезапустить бота
   - **Повторить только тот сценарий, который не прошёл** (не все заново)
   - Если снова fail → ещё одна итерация починки (макс 5)
5. После прохождения всех сценариев — финальная проверка: все автотесты зелёные
6. Записать результаты в `development/docs/testing/user-test-results_ДД-ММ-ЧЧ-ММ.md`
7. Обновить pipeline-state.json

**Создание — инструмент:** skill-creator
**Промпт для создания:**

> THINK HARD. Используй максимальную глубину мышления на каждом шаге.
>
> ВАЖНО: В SKILL.md созданного скилла укажи инструкцию — при выполнении скилл обязан использовать THINK HARD (максимальную глубину мышления) на каждом шаге.
>
> Создай скилл `run-user-testing`.
>
> Назначение: прогон пользовательских тест-сценариев с живым ботом + LLM-верификация + цикл починки. Самостоятельный скилл.

> 1. Запускает бота (`python -m claude_manager`)
> 2. Подключается к Telegram через TelegramTestClient (Telethon, `tests/e2e/test_client.py`)
> 3. Проходит каждый сценарий: `client.send_message()` / `client.send_command()` → `client.wait_for_response()` → LLM-верификация → pass/fail
> 4. При fail: останавливает бота → исправляет код → прогоняет ВСЕ автотесты → перезапускает → повторяет ТОЛЬКО упавший сценарий (макс 5 попыток)

> 6. Записывает в `development/docs/testing/user-test-results_ДД-ММ-ЧЧ-ММ.md`
> 7. Обновляет `pipeline-state.json`
>
> Пользователь видит все сообщения в Telegram в реальном времени.

## Управление состоянием

Оркестратор использует `pipeline-state.json` для:

- **Возобновление** — если сессия прервалась, оркестратор при следующем запуске читает файл и продолжает с места остановки
- **Прозрачность** — в любой момент видно текущую фазу и задачу
- **Диагностика** — `fix_cycles` показывает проблемные места

**Жизненный цикл:**
1. project-setup создаёт файл при первом запуске
2. Каждый скилл обновляет свои задачи: `pending` → `in_progress` → `completed`/`failed`
3. Оркестратор проверяет ворота качества при завершении фазы
4. При `failed` — оркестратор останавливается с диагностикой

---

## Циклы исправления

Каждая фаза (кроме 0 и 7) содержит цикл исправления:

```
Выполнить задачу
    |
    v
Проверить результат
    |
    +--> Зелёный? ---> Да ---> Следующая задач��
    |
    +--> Нет
           |
           v
      Проанализировать ошибку
           |
           v
      Исправить код/тест
           |
           v
      Перезапустить ВСЕ тесты (регрессионная защита)
           |
           v
      Итерация < 5? ---> Да ---> Вернуться к "Проверить результат"
           |
           +--> Нет ---> FAIL, остановить пайплайн
```

**Ключевое правило:** после ЛЮБОГО исправления перегоняются ВСЕ тесты предыдущих фаз, а не только сломанный тест. Это защита от регрессий.

**Особенность Фазы 9 (прогон пользовательского тестирования):** при fail повторно прогоняется ТОЛЬКО упавший сценарий, а не все сценарии заново. Но ВСЕ автотесты (юнит + интеграционные) перегоняются полностью.

## Обработка ошибок

**Задача не выполняется (5 циклов починки исчерпаны):**
1. Записать в pipeline-state.json: `status = "failed"`, `error = "описание"`
2. Остановить пайплайн
3. Вывести: что сломалось, что было сделано, что нужно для продолжения

**Регрессия (починка одного ломает другое):**
1. Цикл: починка → ВСЕ тесты → до зелёного (макс 5 итераций)
2. Если не удалось → откат через `git checkout`, запись ошибки, остановка

**E2E / пользовательское тестирование (бот не запускается, Telegram не отвечает):**
1. Диагностика: проверить `.env`, проверить сеть, проверить логи
2. Записать ошибку в pipeline-state.json
3. Остановка пайплайна

## ТЗ на скилл pipeline-run (оркестратор)

**Назначение:** Главный скилл пайплайна. Объединяет все инструменты из Части 1 и запускает их в правильном порядке. Полный автомат.

**Ключевые правила оркестратора:**
- **Все задачи выполняются через агентов (Agent tool).** Оркестратор НЕ выполняет работу сам — он запускает агентов. Каждая задача (спецификация модуля, реализация модуля, прогон тестов) = отдельный агент.
- **Параллелизм через агентов.** Независимые задачи одного слоя запускаются параллельно (несколько Agent tool вызовов в одном сообщении). Пример: модули config, message_splitter, session_reader — слой 0 → три агента параллельно.
- **Git-коммит после каждого артефакта.** Агент завершил спеку → коммит. Агент реализовал модуль → коммит. Агент исправил баг → коммит. Это создаёт точки отката.
- **Пуш — только по подтверждению пользователя.** Перед git push оркестратор останавливается и спрашивает через AskUserQuestion.
- **Интерактивные шаги.** Оркестратор останавливается и спрашивает пользователя при: первой авторизации Telethon (SMS-код), git push, вопросах из validate-brd.
- **Возврат спеки.** Если агент-реализатор обнаружил ошибку в спеке → спека возвращается из realized/ в specs/, исправляется, коммитится, реализация продолжается.

**Аргументы:**
- `brd_path` — путь к BRD (обязательный при первом запуске)

**Алгоритм:**
1. Проверить `pipeline-state.json`:
   - Если существует → прочитать, определить текущую фазу и задачу (режим возобновления)
   - Если не существует → начать с фазы 0
2. Определить следующее действие:
   - Найти первую задачу со статусом `pending` или `in_progress` в текущей фазе
   - Если все задачи фазы `completed` → запустить `check-phase-gate.py` → перейти к следующей фазе
3. Запустить агента (Agent tool) для выполнения задачи:
   - Фаза 0 → агент: validate-brd
   - Фаза 1 → агент: project-setup
   - Фаза 2 → агент: brd-decompose, затем агенты: spec-module для каждого модуля (параллельно в рамках слоя)
   - Фаза 3 → агенты: implement-module для каждого модуля (параллельно в рамках слоя, последовательно между слоями)
   - Фаза 4 → агент: test-integration
   - Фаза 5 → агент: test-e2e (интерактивный — может потребовать SMS для Telethon)
   - Фаза 6 → агент: review-code
   - Фаза 7 → агент: update-project-docs
   - Фаза 8 → агент: create-user-test-scenarios
   - Фаза 9 → агент: run-user-testing
4. После завершения агента → git commit
5. Обновить `pipeline-state.json`
6. Если ошибка → обработать (см. «Обработка ошибок»)
7. Если не ошибка → шаг 2
8. Когда все фазы `completed` → вывести финальный отчёт

**Финальный отчёт:**
- Сколько модулей реализовано
- Сколько тестов (юнит, интеграционных, E2E, пользовательских)
- Сколько проблем найдено и исправлено при ревью
- Общее количество циклов починки
- Статус: ПРОЕКТ ГОТОВ

**Создание — инструмент:** skill-creator
**Промпт для создания:**

> THINK HARD. Используй максимальную глубину мышления на каждом шаге.
>
> ВАЖНО: В SKILL.md созданного скилла укажи инструкцию — при выполнении скилл обязан использовать THINK HARD (максимальную глубину мышления) на каждом шаге.
>
> Создай скилл `pipeline-run`.
>
> Назначение: полностью автоматический оркестратор инженерного пайплайна. Самостоятельный скилл.
>
> Это главный скилл. Он управляет всем процессом, но НЕ выполняет работу сам — запускает агентов (Agent tool):
> 1. Читает или создаёт `pipeline-state.json`
> 2. Определяет текущую фазу и следующую задачу
> 3. Запускает агента (Agent tool) для задачи. Независимые задачи одного слоя — параллельно (несколько Agent вызовов в одном сообщении)
> 4. После завершения агента — git commit
> 5. Проверяет результат, обновляет состояние
> 6. При завершении фазы — проверяет ворота качества (`check-phase-gate.py`)
> 7. Переходит к следующей фазе
> 8. На интерактивных шагах (Telethon SMS, git push) — останавливается и спрашивает пользователя через AskUserQuestion
> 9. Если спека модуля оказалась неправильной — возвращает из realized/ в specs/, исправляет, коммитит
> 10. Останавливается только при ошибке (5 неудачных циклов починки)
>
> Фазы: 0-Валидация BRD → 1-Фундамент → 2-Спецификации → 3-Реализация → 4-Интеграция → 5-E2E → 6-Ревью → 7-Документация → 8-Сценарии пользовательского тестирования → 9-Прогон пользовательского тестирования
>
> Скиллы: validate-brd, project-setup, brd-decompose, spec-module, implement-module, test-integration, test-e2e, review-code, update-project-docs, create-user-test-scenarios, run-user-testing.
>
> При возобновлении: читает `pipeline-state.json`, продолжает с места остановки.
>
