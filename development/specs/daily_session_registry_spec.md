# Спецификация модуля: daily_session_registry

Дата: 30-03-2026
Слой: 1 (зависит от слоя 0)
Файл: `src/claude_manager/daily_session_registry.py`

## Назначение

Управляет дневной нумерацией сессий Claude (#1, #2, #3...) — присваивает каждой сессии порядковый номер в рамках текущего дня, сбрасывает счётчик в полночь по локальному времени и позволяет находить сессию по её дневному номеру. Хранит реестр в файле `daily_sessions.json` с атомарной записью и защитой от параллельного доступа.

## Обслуживаемые сценарии

- **CJM-02: Отправка текстового сообщения** — привязывает дневной номер к сессии, в которую отправлено сообщение (при получении ответа Claude)
- **CJM-03: Отправка фотографии или файла** — аналогично CJM-02: привязывает дневной номер к сессии
- **CJM-04: Создание новой сессии (/new)** — резервирует новый дневной номер для только что созданной сессии
- **CJM-05: Просмотр списка сессий (/sessions)** — регистрирует все показываемые сессии в дневном реестре и возвращает их номера для отображения
- **CJM-06: Переключение на сессию (/N)** — ищет сессию по дневному номеру (а если не нашёл — отдаёт None, чтобы вызывающий код искал дальше, среди всех видимых сессий)

## Публичный API

### `async register_session(session_id: str) -> int`

Регистрирует сессию в дневном реестре и возвращает её дневной номер. Если сессия уже зарегистрирована сегодня — возвращает существующий номер без создания нового.

**Аргументы:**
- `session_id` (str) — уникальный идентификатор сессии Claude (например, `"abc123-def456"` или временный `"_new_0001"`)

**Возвращает:** дневной номер сессии (int, начиная с 1). Например, `3` означает третью сессию за сегодня.

**Исключения:**
- `OSError` — не удалось записать файл реестра на диск (нет прав, диск заполнен)

### `async get_session_id_by_number(day_number: int) -> str | None`

Ищет сессию по дневному номеру в реестре текущего дня. Возвращает идентификатор сессии или None, если номер не найден.

**Аргументы:**
- `day_number` (int) — дневной номер сессии (например, `3` для третьей сессии за день)

**Возвращает:** идентификатор сессии (str) или None, если сессия с таким номером не зарегистрирована сегодня.

**Примечание:** по решению из валидации BRD (проблема 3.3), команда /N ищет сессию не только здесь. Если эта функция вернула None, вызывающий код (session_manager или bot) должен искать дальше — среди всех видимых сессий через session_reader.

### `async update_session_id(old_session_id: str, new_session_id: str) -> None`

Заменяет временный идентификатор сессии на реальный во всех записях реестра. Нужна для сквозного механизма «обновление session_id» — когда Claude возвращает настоящий ID вместо временного `_new_XXXX`.

**Аргументы:**
- `old_session_id` (str) — текущий (временный) идентификатор сессии (например, `"_new_0001"`)
- `new_session_id` (str) — новый (реальный) идентификатор сессии от Claude (например, `"abc123-def456"`)

**Возвращает:** ничего.

**Исключения:**
- `OSError` — не удалось записать файл реестра на диск

**Примечание:** если old_session_id не найден в реестре — функция молча завершается без ошибки (идемпотентность). Это нормальная ситуация: обновление может прийти повторно или для сессии, которая не была зарегистрирована.

### `async get_all_today_sessions() -> dict[int, str]`

Возвращает все сессии, зарегистрированные сегодня. Нужна модулю session_watcher для отображения номеров и модулю bot для формирования списка `/sessions`.

**Аргументы:** нет.

**Возвращает:** словарь `{дневной_номер: session_id}`. Например: `{1: "abc123", 2: "def456", 3: "_new_0001"}`. Если сегодня не было сессий — пустой словарь `{}`.

### `async load_registry() -> None`

Загружает реестр из файла `daily_sessions.json` в память. Вызывается один раз при запуске бота (из main.py через session_manager) для восстановления состояния после перезапуска.

**Аргументы:** нет.

**Возвращает:** ничего.

**Исключения:**
- Файл не существует — молча создаёт пустой реестр (первый запуск)
- Файл повреждён (невалидный JSON) — логирует warning, создаёт пустой реестр

## Внутренние функции

### `_get_today_key() -> str`

Возвращает ключ текущего дня для хранения в реестре. Формат: `"YYYY-MM-DD"` по локальному времени.

**Аргументы:** нет.

**Возвращает:** строку с датой, например `"2026-03-30"`.

### `_ensure_today_registry() -> None`

Проверяет, что в памяти есть записи для текущего дня. Если дата изменилась (наступила полночь) — создаёт новую пустую секцию для нового дня. Старые записи сохраняются в файле (для возможности поиска вчерашних сессий при восстановлении), но не используются при выдаче новых номеров.

**Аргументы:** нет.

**Возвращает:** ничего.

### `_next_day_number() -> int`

Возвращает следующий свободный номер для текущего дня. Считает как `max(существующие_номера) + 1`, или `1` если сессий за сегодня ещё нет.

**Аргументы:** нет.

**Возвращает:** целое число (int), начиная с 1.

### `async _save_registry() -> None`

Сохраняет реестр из памяти в файл `daily_sessions.json`. Использует атомарную запись: сначала пишет во временный файл `.tmp`, затем переименовывает в основной.

**Аргументы:** нет.

**Возвращает:** ничего.

**Исключения:**
- `OSError` — не удалось записать или переименовать файл

## Алгоритм работы

### register_session

1. Захватить asyncio Lock (защита от параллельной записи — если два обработчика одновременно регистрируют сессии)
2. Вызвать `_ensure_today_registry()` — убедиться, что есть записи для текущего дня (если наступила полночь — создать новую секцию)
3. Проверить, есть ли session_id уже в записях сегодняшнего дня
4. Если есть — вернуть существующий номер (идемпотентность)
5. Если нет — вызвать `_next_day_number()` для получения нового номера
6. Добавить запись `{номер: session_id}` в реестр текущего дня
7. Вызвать `_save_registry()` для записи на диск
8. Залогировать через logging.info: `"Сессия {session_id} зарегистрирована как #{номер}"`
9. Освободить Lock
10. Вернуть номер

### get_session_id_by_number

1. Захватить asyncio Lock (защита от чтения во время записи)
2. Вызвать `_ensure_today_registry()` — убедиться, что работаем с текущим днём
3. Найти в записях текущего дня запись с указанным номером
4. Освободить Lock
5. Вернуть session_id или None

### update_session_id

1. Захватить asyncio Lock
2. Найти old_session_id во всех записях реестра (не только сегодня — ID мог быть зарегистрирован вчера)
3. Если не найден — залогировать через logging.debug, освободить Lock, выйти
4. Если найден — заменить old_session_id на new_session_id в этой записи
5. Вызвать `_save_registry()` для записи на диск
6. Залогировать через logging.info: `"Session ID обновлён: {old_session_id} → {new_session_id}"`
7. Освободить Lock

### get_all_today_sessions

1. Захватить asyncio Lock
2. Вызвать `_ensure_today_registry()`
3. Скопировать словарь записей текущего дня, конвертируя ключи из строк в int (в JSON ключи — строки `"1"`, `"2"`, а API возвращает `{1: "...", 2: "..."}`)
4. Освободить Lock
5. Вернуть копию с числовыми ключами

### load_registry

1. Получить из config путь к директории данных бота (там же, где хранится sessions.json). Построить полный путь: `{директория_данных}/daily_sessions.json`. Сохранить в `_registry_path`
2. Попытаться прочитать файл через `asyncio.to_thread()` (чтение с диска — блокирующая операция)
3. Если файл не существует — залогировать info `"Файл реестра не найден, создаю пустой реестр"`, инициализировать пустой словарь
4. Если файл существует, но JSON невалидный — залогировать warning `"Файл реестра повреждён, создаю пустой реестр"`, инициализировать пустой словарь
5. Если файл прочитан успешно — загрузить данные в внутренний словарь `_registry`
6. Вызвать `_ensure_today_registry()` для подготовки записей текущего дня

### _save_registry

1. Сериализовать словарь `_registry` в JSON с отступами (indent=2) и ensure_ascii=False (чтобы кириллица была читаемой)
2. Записать во временный файл `daily_sessions.tmp` через `asyncio.to_thread()`
3. Переименовать `daily_sessions.tmp` в `daily_sessions.json` через `os.replace()` (атомарная операция на macOS)

## Формат файла daily_sessions.json

```json
{
  "2026-03-30": {
    "1": "abc123-def456",
    "2": "ghi789-jkl012",
    "3": "_new_0001"
  },
  "2026-03-29": {
    "1": "mno345-pqr678",
    "2": "stu901-vwx234"
  }
}
```

**Структура:**
- Ключ верхнего уровня — дата в формате `"YYYY-MM-DD"`
- Значение — словарь `{дневной_номер_строкой: session_id}`
- Дневные номера хранятся как строки (ключи JSON всегда строки), но в API модуля используются как int
- Хранятся записи за несколько дней (не удаляются автоматически) — это нужно для поиска сессий при восстановлении после перезапуска и для работы update_session_id с вчерашними сессиями

## Внутреннее состояние модуля

- `_registry: dict[str, dict[str, str]]` — словарь реестра в памяти. Ключ — дата `"YYYY-MM-DD"`, значение — словарь `{номер_строкой: session_id}`
- `_lock: asyncio.Lock` — блокировка для защиты от параллельного чтения/записи
- `_registry_path: Path` — путь к файлу `daily_sessions.json` (определяется при `load_registry()`)

## Обработка ошибок

- **Файл daily_sessions.json не существует** — нормальная ситуация при первом запуске. Создаётся пустой реестр. Логирование: info
- **Файл daily_sessions.json повреждён (невалидный JSON)** — создаётся пустой реестр. Логирование: warning. Повреждённый файл будет перезаписан при следующей операции записи
- **Ошибка записи на диск (OSError)** — ошибка проксируется наверх (в вызывающий код). Данные в памяти остаются консистентными. Вызывающий код (session_manager или bot) должен залогировать ошибку и сообщить пользователю
- **Параллельный доступ** — защита через asyncio.Lock. Два одновременных вызова register_session не потеряют данные
- **Смена дня (полночь)** — обрабатывается через `_ensure_today_registry()` при каждом обращении. Старые записи не теряются
- **update_session_id с несуществующим old_session_id** — функция молча завершается (идемпотентность). Это нормальная ситуация при повторном обновлении
- **Запись во временный файл прошла, но переименование не удалось** — временный файл остаётся на диске. При следующей записи он будет перезаписан. Основной файл остаётся нетронутым (целостность данных)

## Константы

- `REGISTRY_FILENAME = "daily_sessions.json"` — имя файла реестра. Вынесено в константу для единообразия и возможности изменения
- `REGISTRY_TEMP_SUFFIX = ".tmp"` — суффикс временного файла при атомарной записи. Временный файл: `daily_sessions.json.tmp`
- `DATE_FORMAT = "%Y-%m-%d"` — формат даты для ключей реестра. ISO 8601 — однозначный, сортируемый

  - Вход: `register_session("abc123")`

### Граничные случаи

- **test_midnight_reset_creates_new_day** — после полуночи нумерация начинается с 1
  - Вход: реестр с сессиями #1-#5 за 2026-03-29. Мокнуть `date.today()` на 2026-03-30. Вызвать `register_session("new-session")`
  - Ожидаемый результат: возвращает `1` (новый день). Старые записи за 29-е число сохранены в реестре
  - Тип: edge case

- **test_yesterday_sessions_preserved_after_midnight** — вчерашние записи не удаляются при смене дня
  - Вход: реестр с сессиями за 2026-03-29. Мокнуть дату на 2026-03-30. Вызвать `register_session("new")`
  - Ожидаемый результат: `_registry` содержит записи и за 2026-03-29, и за 2026-03-30
  - Тип: edge case

- **test_update_session_id_in_yesterday_registry** — обновление ID в записях предыдущего дня
  - Вход: реестр с `_new_0001` за вчера. Вызвать `update_session_id("_new_0001", "real-id")`
  - Ожидаемый результат: запись за вчера обновлена на `"real-id"`
  - Тип: edge case

- **test_update_nonexistent_session_id_no_error** — обновление несуществующего ID не вызывает ошибку
  - Вход: `update_session_id("nonexistent-id", "new-id")`
  - Ожидаемый результат: функция завершается без ошибки, реестр не изменён
  - Тип: edge case

- **test_concurrent_register_sessions** — две параллельные регистрации не теряют данные
  - Вход: `asyncio.gather(register_session("aaa"), register_session("bbb"))`
  - Ожидаемый результат: обе сессии зарегистрированы с разными номерами (1 и 2), ни одна не потеряна
  - Тип: edge case

- **test_register_session_with_temporary_id** — регистрация сессии с временным ID формата `_new_XXXX`
  - Вход: `register_session("_new_0001")`
  - Ожидаемый результат: возвращает номер, сессия с временным ID нормально зарегистрирована
  - Тип: edge case

- **test_number_after_gap_in_sequence** — номер выдаётся правильно если предыдущие номера не идут подряд (теоретически невозможно при нормальной работе, но защита от повреждённых данных)
  - Вход: реестр с номерами 1, 2, 5 (вручную загружен). Вызвать `register_session("new")`
  - Ожидаемый результат: возвращает `6` (max + 1)
  - Тип: edge case
