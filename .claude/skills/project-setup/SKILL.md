# Настройка инфраструктуры проекта

Ты настраиваешь проект с нуля: зависимости, тесты, конфигурация, фикстуры, E2E клиент, логирование. После твоей работы проект готов к написанию кода — любой разработчик может клонировать репозиторий, запустить `pip install` и начать работать.

Этот скилл — фундамент, на котором строится весь остальной пайплайн. Ошибка здесь каскадно ломает все последующие фазы: неправильные зависимости — не запустятся тесты, отсутствующие фикстуры — не напишутся моки, сломанный pytest — встанет CI. Поэтому каждый шаг проверяется перед переходом к следующему.

**На каждом шаге используй THINK HARD** — максимальную глубину мышления. Инфраструктурные ошибки дорого исправлять позже.

Аргументы (если есть): $ARGUMENTS

## Шаг 4: tests/conftest.py

**THINK HARD** — фикстуры определяют, насколько удобно будет писать тесты во всём проекте. Сделай их универсальными и понятными.

Создай `tests/conftest.py` с фикстурами, описанными в pipeline-spec.md (раздел «Технический стек тестирования»). Каждая фикстура — готовый инструмент для тестов.

### Фикстура mock_bot

Фейковый Telegram-бот, который записывает все вызовы `send_message` вместо реальной отправки. Тесты проверяют, что бот отправил правильные сообщения, не подключаясь к Telegram.

Что должен уметь:
- `send_message(chat_id, text, **kwargs)` — записывает вызов в список `sent_messages`
- `sent_messages` — список всех отправленных сообщений (каждый элемент — словарь с `chat_id`, `text` и прочими параметрами)
- Все методы — async (бот полностью асинхронный)

Реализация: используй `AsyncMock` из `unittest.mock`. Настрой `side_effect` у `send_message`, чтобы он записывал вызовы в список и возвращал фейковый объект `Message`.

### Фикстура mock_update

Фабрика (функция-генератор) фейковых входящих сообщений Telegram. Каждый вызов создаёт новый объект `Update` с настраиваемыми параметрами.

Параметры:
- `text` (str) — текст сообщения (по умолчанию "test message")
- `chat_id` (int) — ID чата (по умолчанию 12345)
- `user_id` (int) — ID пользователя (по умолчанию 12345)

Возвращает объект, у которого есть:
- `message.text` — текст сообщения
- `message.chat.id` и `message.chat_id` — ID чата
- `message.from_user.id` — ID отправителя
- `effective_chat.id` — ID чата (используется python-telegram-bot)
- `effective_user.id` — ID пользователя
- `message.reply_text(text, **kwargs)` — AsyncMock для ответа

Реализация: функция-фабрика, которую тест вызывает с нужными параметрами. Внутри — `MagicMock` / `AsyncMock` с правильно настроенными атрибутами.

### Фикстура mock_claude_process

Фейковый процесс Claude CLI (subprocess), который принимает сообщения через stdin и отдаёт заранее подготовленные stream-json события через stdout.

Что должен уметь:
- Принимать данные через `stdin.write()` / `stdin.drain()` (AsyncMock)
- Отдавать заранее заданные JSON-строки через `stdout.readline()` — каждый вызов возвращает следующую строку из заданного списка
- Имитировать завершение процесса: `wait()` возвращает 0, `returncode` = 0

Параметры при создании:
- `responses` (list[dict]) — список stream-json событий, которые процесс «отдаёт». Каждый dict сериализуется в JSON-строку + `\n`

Реализация: `MagicMock` с async-атрибутами для stdin/stdout/stderr. `stdout.readline` — `AsyncMock` с `side_effect`, который по очереди отдаёт закодированные JSON-строки, затем пустую строку `b""` (сигнал конца потока).

### Фикстура session_dir

Временная папка, имитирующая директорию сессий Claude (`~/.claude/projects/`). Создаётся через `tmp_path` (встроенная фикстура pytest) — автоматически удаляется после теста.

### Фикстура allowed_user_id

ID пользователя из белого списка — целое число, которое проходит проверку авторизации. Используется в тестах, чтобы имитировать разрешённого пользователя.

Возвращает просто `int` (например 12345).

### Общие требования к фикстурам

- Все фикстуры — на уровне файла conftest.py (доступны всем тестам в папке `tests/`)
- Импорты: стандартная библиотека, pytest, unittest.mock
- Комментарий-docstring у каждой фикстуры — что она делает и как её использовать
- Type hints для возвращаемых значений
- Никаких зависимостей от продакшен-кода (conftest.py должен работать даже если src/ пуст)

## Шаг 5: E2E тестовый клиент

Создай два файла:

### tests/e2e/__init__.py

Пустой файл (делает папку Python-пакетом).

### tests/e2e/test_client.py

Класс `TelegramTestClient` — обёртка над библиотекой Telethon, упрощающая написание E2E тестов. Тесты будут использовать этот клиент для отправки сообщений реальному боту и проверки ответов.

**API класса** (из pipeline-spec.md):

- `__init__(api_id, api_hash, phone, bot_username, session_name="telethon_test")` — создаёт клиент Telethon, сохраняет параметры
- `async connect()` — подключается к Telegram. Если сессия не авторизована — запрашивает код
- `async disconnect()` — отключается
- `async send_message(text: str)` — отправляет текстовое сообщение боту
- `async send_command(command: str)` — отправляет команду боту (например "/new", "/sessions")
- `async send_photo(path: str, caption: str = "")` — отправляет фото боту с подписью
- `async wait_for_response(timeout: int = 30) -> str` — ждёт ответ от бота, возвращает текст. Если за `timeout` секунд ответ не получен — бросает TimeoutError
- `async get_last_response() -> str | None` — возвращает текст последнего ответа от бота (или None, если ответов не было)

**Реализация:**

- Использует `TelegramClient` из telethon
- `wait_for_response` — подписывается на новые сообщения от бота через event handler, ждёт с asyncio.wait_for
- Все методы — async
- Логирование через `logging` (не print)
- Type hints для всех публичных методов
- Комментарий-docstring у каждого метода

**Важно:** этот файл не содержит тестов — только клиент. Тесты, использующие его, будут созданы позже (скиллом test-e2e).

Проверь `src/claude_manager/main.py`. Если в нём нет настройки логирования — добавь базовую конфигурацию:
