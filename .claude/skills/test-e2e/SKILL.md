# Живое E2E тестирование по сценариям BRD

**THINK HARD** на каждом шаге выполнения этого скилла. Используй максимальную глубину мышления при принятии каждого решения — от генерации тест-плана до LLM-верификации ответов и диагностики ошибок.

Ты проводишь живое end-to-end тестирование Telegram-бота Claude Manager. E2E тесты — это проверка системы глазами реального пользователя: ты отправляешь сообщения боту в настоящем Telegram и проверяешь, что ответы соответствуют BRD (документу с описанием ожидаемого поведения). Никаких моков и подмен — бот работает по-настоящему, Claude отвечает по-настоящему, Telegram доставляет сообщения по-настоящему.

Это последний рубеж качества перед ревью: если E2E тесты прошли — бот работает как задумано.

## Предварительные проверки

**THINK HARD** — без этих проверок тестирование невозможно. Каждый пропущенный шаг приведёт к загадочным ошибкам, которые будет сложно отследить.

Вместо этого используй отдельную переменную `E2E_TEST_USER_ID` в `.env`. Если Telethon user ID обнаружен в `ALLOWED_USER_IDS` — **СТОП**, это создаст многопользовательский сценарий и сломает бота.

## Шаг 1: Генерация тест-плана из BRD

**THINK HARD** — тест-план определяет, что именно будет проверяться. Пропущенный сценарий = непроверенная функциональность = потенциальный баг в продакшене.

### 1.1 Чтение BRD

Прочитай `dev/docs/brd/brd-user-journeys.md` — это документ со всеми пользовательскими сценариями (CJM). Каждый CJM описывает, что делает пользователь, что происходит внутри, что пользователь видит в ответ, и что может пойти не так.

### 1.2 Формирование тест-сценариев

Для каждого CJM из BRD создай тест-сценарий. У каждого сценария должны быть:

- **ID** — CJM-XX-YY (XX — номер CJM, YY — номер сценария внутри CJM)
- **Название** — краткое описание, что проверяем
- **Предусловие** — что должно быть настроено до начала (есть сессия, нет сессии, бот запущен и т.д.)
- **Шаги** — конкретные действия: какие сообщения отправить, какие команды выполнить
- **Ожидаемый результат** — что должен ответить бот (на основе раздела «Что видит пользователь» в BRD)
- **Способ проверки** — как LLM-верификатор будет определять pass/fail

Разбивай CJM на несколько сценариев, когда BRD описывает разные ветки поведения. Например, для CJM-02 (текстовое сообщение) будут отдельные сценарии для:
- Обычное сообщение с ответом
- Сообщение без активной сессии (ожидаем подсказку)
- Длинный ответ (ожидаем разбивку на несколько сообщений)

Сценарии, описанные в разделе «Что может пойти не так», тоже нужно покрыть, если их можно воспроизвести в E2E (например, «пустой ответ» — можно попросить Claude сделать действие без текстового вывода).

**Порядок тестирования критически важен.** Сценарии зависят друг от друга: нельзя проверить отправку сообщения (CJM-02), не создав сначала сессию (CJM-04). Расположи сценарии в порядке, который обеспечивает корректные предусловия:

1. CJM-04 (создание сессии через /new) — сначала нужна сессия
2. CJM-02 (текстовое сообщение) — теперь есть куда отправлять
3. CJM-03 (фото/файлы) — расширение CJM-02
4. CJM-05 (список сессий /sessions) — проверяем, что созданная сессия видна
5. CJM-06 (переключение /1, /2...) — переключаемся на сессию из списка
6. CJM-07 (мониторинг /all) — проверяем режим наблюдения
7. CJM-08 (остановка /stop) — останавливаем работу Claude

CJM-01 (первый запуск) и CJM-09 (защита от двойного запуска) тестируются отдельно, так как требуют перезапуска бота.

### 1.3 Запись тест-плана

Определи текущую дату и время, затем запиши тест-план:


Создай файл `dev/docs/logs/testing/e2e-test-plan_ДД-ММ-ЧЧ-ММ.md` (подставь реальную дату-время). Убедись, что папка `dev/docs/logs/testing/` существует (создай если нет).

```markdown
  1. Отправить команду `/new`
```

## Шаг 3: Подключение к Telegram

**THINK HARD** — авторизация Telethon может потребовать SMS-код от пользователя. Это интерактивный шаг — скилл должен остановиться и подождать ввода.

### 3.1 Инициализация TelegramTestClient

Прочитай `tests/e2e/test_client.py` и разберись в API клиента. Затем напиши и выполни Python-скрипт для подключения:

```python
import asyncio
import sys
sys.path.insert(0, '/Users/ivan/Desktop/claude-sandbox/claude_manager')

from tests.e2e.test_client import TelegramTestClient
from dotenv import load_dotenv
import os

load_dotenv('/Users/ivan/Desktop/claude-sandbox/claude_manager/.env')

async def connect():
    client = TelegramTestClient(
        api_id=int(os.getenv('TELETHON_API_ID')),
        api_hash=os.getenv('TELETHON_API_HASH'),
        phone=os.getenv('TELETHON_PHONE'),
        bot_username=os.getenv('TELETHON_BOT_USERNAME')
    )
    await client.connect()
    return client

client = asyncio.run(connect())
```

### 3.2 Обработка авторизации Telethon

При **первом** запуске Telethon попросит SMS-код. Если это произошло:

1. **Останови выполнение** и спроси пользователя: «Telethon запрашивает код авторизации. Telegram отправил SMS-код на номер [номер из .env]. Введи код:»
2. Дождись ответа пользователя
3. Введи код в Telethon
4. После успешной авторизации Telethon сохранит сессию в файл `*.session` — при следующих запусках авторизация не потребуется

Если файл `*.session` уже существует — авторизация пройдёт автоматически, без SMS.

## Шаг 4: Прогон тест-сценариев

**THINK HARD** — каждый сценарий требует внимательного выполнения и осмысленной верификации. Механический прогон без анализа ответов пропустит тонкие баги.

Проходи сценарии строго в порядке, заданном в тест-плане. Для каждого сценария:

### 4.1 Выполнение шагов

Для каждого шага сценария используй соответствующий метод TelegramTestClient:

- **Текстовое сообщение:** `client.send_message("текст")` → `response = client.wait_for_response(timeout=60)`
- **Команда:** `client.send_command("/new")` → `response = client.wait_for_response(timeout=60)`
- **Фото:** `client.send_photo("/путь/к/файлу", "подпись")` → `response = client.wait_for_response(timeout=60)`

Таймаут 60 секунд — Claude может думать долго, особенно при первом запросе в сессии. Если ответ не пришёл за 60 секунд — это fail.

### 4.2 LLM-верификация ответа

После получения ответа бота выполни LLM-верификацию. Это ключевое отличие от обычных тестов: вместо точного сравнения строк Claude анализирует **смысл** ответа.

Для каждого ответа сформулируй промпт верификации:

```
Ты верификатор E2E теста. Сравни ответ бота с ожидаемым поведением.

**Сценарий:** [название сценария]
**Что отправил пользователь:** [сообщение/команда]
**Ожидаемое поведение (из BRD):** [описание из тест-плана]
**Фактический ответ бота:** [полный текст ответа]

Вынеси вердикт:
- **PASS** — ответ соответствует ожидаемому поведению (допускаются различия в формулировках, но суть и структура должны совпадать)
- **FAIL** — ответ не соответствует ожидаемому поведению

Формат ответа:
VERDICT: PASS или FAIL
REASON: краткое обоснование (1-2 предложения)
```

**THINK HARD** при верификации. Проверяй не только текст ответа, но и:
- Наличие номера сессии в заголовке (если ожидается)
- Правильное форматирование (HTML-разметка отрендерилась)
- Кликабельность номеров сессий (если описана в BRD)
- Полноту ответа (не обрезан, все части доставлены)

### 4.3 Запись результата

Для каждого сценария записывай:
- ID сценария
- Вердикт: PASS / FAIL
- Обоснование LLM-верификатора
- Полный текст ответа бота (для отладки при fail)
- Время выполнения (сколько секунд ждали ответ)

## Шаг 5: Цикл починки при FAIL

**THINK HARD** — правильная диагностика причины провала критически важна. Неверный диагноз → неверное исправление → новые баги.

Если хотя бы один сценарий завершился FAIL:

### 5.1 Диагностика

Проанализируй упавший сценарий:

1. **Перечитай ответ бота** — что именно он ответил? Это ошибка, неправильный ответ, или отсутствие ответа?
2. **Проверь лог бота** — `tail -100 /tmp/claude_manager_e2e.log` — есть ли ошибки, трейсбеки, предупреждения?
3. **Определи источник проблемы:**
   - Баг в коде бота (ответ не соответствует BRD)
   - Баг в тестовом клиенте (сообщение не доставлено или ответ не прочитан)
   - Проблема с Claude (таймаут, пустой ответ)
   - Проблема с сетью/Telegram

### 5.2 Остановка бота

```bash
pkill -f "python -m claude_manager"
```

Подожди 2 секунды, убедись что процесс завершился.

### 5.3 Исправление кода

**THINK HARD** — прочитай код, который отвечает за упавший сценарий. Сверься с BRD и спецификациями модулей из `dev/docs/specs/realised/`.

Внеси исправления в соответствующие файлы в `src/claude_manager/`.

### 5.4 Регрессионная проверка

После исправления — запусти ВСЕ автоматические тесты:

Проверь, существует ли скрипт запуска:
```bash

```

Если существует:
```bash
cd /Users/ivan/Desktop/claude-sandbox/claude_manager && ./dev/scripts/run-all-tests.sh
```

Если не существует:
```bash

```

Это обязательный шаг: исправление E2E бага может сломать юнит-тесты или интеграционные тесты. Если автотесты стали красными — исправь регрессию перед продолжением.

### 5.5 Перезапуск бота


Подожди 5 секунд для инициализации.

### 5.6 Повтор упавшего сценария

Повтори **только** упавший сценарий (не все сценарии с начала). Если он снова FAIL — вернись к шагу 5.1.

**Максимум 5 попыток** на один сценарий. Если после 5 попыток сценарий всё ещё FAIL:
1. Зафиксируй статус: сценарий не пройден, описание проблемы, что было испробовано
2. Продолжи со следующим сценарием — не блокируй весь прогон из-за одного бага
3. В итоговом отчёте отметь этот сценарий как FAIL с подробным описанием

### 5.7 Продолжение прогона

После успешной починки (или исчерпания попыток) продолжи прогон со следующего сценария.
