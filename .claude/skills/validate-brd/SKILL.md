# Валидация BRD и аудит проекта

Ты проводишь полный технический аудит проекта перед началом реализации. Это не поверхностная проверка — ты сканируешь каждый файл, сверяешь каждый факт между документами, проходишь каждый сценарий шаг за шагом. Цель: обнаружить ВСЕ проблемы до того, как они станут багами в коде.

Используй THINK HARD (максимальную глубину мышления) на каждом шаге этого скилла. Каждое направление проверки требует тщательного анализа — не торопись, думай глубоко.

Почему это важно: ошибка, пойманная на этапе документации, исправляется за минуту. Та же ошибка, обнаруженная после реализации, стоит часы отладки и переписывания кода. Аудит — это инвестиция, которая окупается многократно.

## Вход

Аргументы: $ARGUMENTS

Откуда брать данные:
- **BRD** — путь из аргументов, или из `pipeline-state.json` (поле `brd_path`), или стандартный `development/docs/brd-user-journeys.md`
- **CLAUDE.md** — корень проекта
- **Весь проект** — все файлы: конфиги, код, скрипты, документы, тесты

Прочитай `development/docs/docs-index.md` — это оглавление всех документов проекта. По нему определи, какие ещё документы существуют (архитектурные спецификации, пайплайн-спеки, и т.д.), и прочитай все релевантные.

## Выход

Файл отчёта: `development/docs/brd-validation-report_ДД-ММ-ЧЧ-ММ.md`

Формат даты: день-месяц-часы-минуты (например, `brd-validation-report_29-03-14-30.md`).

Отчёт содержит по каждой найденной проблеме:
- Направление (1-6)
- Описание проблемы
- Варианты решения, которые были предложены
- Выбранное решение
- Кто принял решение (пользователь / аудитор)

Также обновляется `pipeline-state.json` — фаза валидации = completed.

## Алгоритм

### Фаза 1: Сбор информации (параллельно)

Информацию можно собирать одновременно из разных источников — они не зависят друг от друга. Используй Agent tool для параллельного сбора.

**Агент 1 — Документы:**
Прочитать все документы проекта целиком: BRD, CLAUDE.md, все файлы из `development/docs/`, все спецификации из `development/specs/`. Составить список всех утверждений (факты, числа, имена, пути, поведение), запомнить где какое утверждение встречается.

**Агент 2 — Код и конфиги:**
Просканировать все файлы проекта: `pyproject.toml`, `requirements.txt`, `.env.example`, `watch_and_restart.sh`, весь `src/`, весь `tests/`, все скрипты из `development/scripts/`. Для каждого файла: что он делает, какие зависимости, какие импорты.

**Агент 3 — Структура проекта:**
Построить полное дерево файлов проекта. Для каждого файла упомянутого в любом документе — проверить существование на диске. Проверить все симлинки. Собрать все пути и ссылки из всех документов.

После завершения всех агентов — объединить результаты и перейти к проверкам.

### Фаза 2: Шесть направлений проверки

THINK HARD на каждом направлении. Не пропускай ни одной проверки.

#### Направление 1 — Конфигурация и инфраструктура

Проверить каждый конфигурационный файл на корректность:

- **pyproject.toml** — версия Python совпадает с CLAUDE.md? Точка входа указана? Зависимости совпадают с requirements.txt? Метаданные заполнены?
- **requirements.txt** — все пакеты существуют? Версии совместимы между собой? Нет лишних или пропущенных зависимостей?
- **.env.example** — все переменные из документации перечислены? Описания понятны? Нет секретов в примере?
- **Скрипты** (`watch_and_restart.sh`, скрипты из `development/scripts/`) — исполняемые? Шебанг правильный? Пути внутри скриптов валидны?
- **Точка входа** — `python -m claude_manager` работает? `__main__.py` существует и корректен?
- **Структура пакета** — `__init__.py` на месте? Импорты разрешимы?

#### Направление 2 — Целостность файлов и ссылок

Для каждого файла или пути, упомянутого в любом документе проекта:
- Существует ли файл на диске?
- Если это ссылка на раздел документа — существует ли этот раздел?
- Если это путь к модулю — корректен ли путь для Python-импорта?

Также проверить:
- Код импортирует модули, которые существуют?
- Документация ссылается на файлы, которые существуют?
- CLAUDE.md описывает структуру, которая совпадает с реальной?

#### Направление 3 — Противоречия

Это самое трудоёмкое направление. Каждый факт, упомянутый более чем в одном месте, нужно сверить.

Метод: для каждого утверждения из BRD найти все места, где тот же факт упоминается в других документах (CLAUDE.md, спецификации, код, комментарии). Сравнить формулировки. Искать расхождения в:
- Числах (лимиты, таймауты, размеры)
- Именах (модулей, функций, переменных, файлов)
- Поведении (что происходит при ошибке, порядок операций)
- Технологиях (версии, библиотеки, протоколы)

Также проверить: нет ли противоречий внутри одного документа? Начало документа говорит одно, конец — другое?

#### Направление 4 — Полнота сценариев

Для каждого CJM (пользовательского пути) из BRD:
1. Пройти путь шаг за шагом от начала до конца
2. На каждом шаге спросить: «Что если здесь произойдёт ошибка? Описано ли это?»
3. Проверить граничные случаи: пустой ввод, очень длинный ввод, одновременные действия, потеря соединения
4. Проверить: каждый переход между состояниями покрыт?
5. Проверить: каждое сообщение об ошибке конкретное и понятное?

Типичные пропуски:
- Что если пользователь отправит сообщение пока предыдущее ещё обрабатывается?
- Что если процесс Claude завершится неожиданно?
- Что если файл, на который ссылается пользователь, не существует?
- Что если Telegram API вернёт ошибку?

#### Направление 5 — Технические пробелы

Найти все места, где описано ЧТО делается, но не описано КАК:
- Слова-маркеры: «TODO», «TBD», «будет определено позже», «нужно решить»
- Абстрактные формулировки: «обрабатывает ошибки», «восстанавливает состояние» — без конкретики
- Открытые вопросы, отмеченные вопросительным знаком или словами «может быть», «возможно»

Также проверить: существуют ли более простые технические решения, чем описанные? Например:
- Можно ли заменить сложный механизм флагом CLI или параметром API?
- Нет ли готовой библиотеки для того, что предлагается писать вручную?
- Не переусложнена ли архитектура для масштаба проекта?

#### Направление 6 — Всё остальное

Любая проблема, не попавшая в категории 1-5:
- Стилистические несоответствия между документами
- Устаревшая информация
- Неясные формулировки, которые можно трактовать двояко
- Подозрительные места, где «что-то не так, но не могу точно сказать что»
- Пропущенные нефункциональные требования (производительность, безопасность, ограничения)

Правило: если сомневаешься — лучше спросить пользователя, чем промолчать. Ложная тревога стоит 30 секунд на ответ. Пропущенная проблема стоит часы отладки.

### Фаза 3: Разрешение проблем

THINK HARD. Это ключевая фаза — от качества вопросов зависит качество проекта.

1. Собрать ВСЕ проблемы из всех шести направлений в один список
2. Для каждой проблемы подготовить:
   - Краткое описание (что не так)
   - Где обнаружено (файл, строка, раздел)
   - Почему это проблема (какие последствия если не исправить)
   - Варианты решения (2-3 варианта с пояснениями плюсов и минусов)
3. Спросить пользователя через инструмент ввода (prompt / ask) по каждой проблеме. Подождать ответа.
4. Не переходить к следующей проблеме, пока текущая не разрешена
5. Записать решение по каждой проблеме

Группировка: если несколько проблем связаны между собой (например, одно и то же имя модуля написано по-разному в трёх местах), объединить их в один вопрос — так пользователю проще принять согласованное решение.

### Фаза 4: Отчёт и завершение

1. Записать отчёт в `development/docs/brd-validation-report_ДД-ММ-ЧЧ-ММ.md`

Структура отчёта:
```markdown
# Отчёт валидации BRD

**Дата:** ДД.ММ.ГГГГ ЧЧ:ММ
**BRD:** путь к файлу
**Проверенные документы:** список
**Проверенные файлы:** количество

## Направление 1 — Конфигурация и инфраструктура

### Проблема 1.1: краткое описание
- **Где:** файл, строка/раздел
- **Суть:** что не так
- **Решение:** что решили
- **Принял:** пользователь / аудитор

### Проблема 1.2: ...

## Направление 2 — Целостность файлов и ссылок
...

## Направление 3 — Противоречия
...

## Направление 4 — Полнота сценариев
...

## Направление 5 — Технические пробелы
...

## Направление 6 — Прочие замечания
...

## Итого
- Найдено проблем: N
- Решено: N
- Критических: N
```

2. Обновить `pipeline-state.json`:

Если файл существует — обновить фазу 0 (валидация BRD):
```json
{
  "phases": {
    "0": {
      "name": "Валидация BRD",
      "status": "completed",
      "completed_at": "ISO-дата",
      "report": "development/docs/brd-validation-report_ДД-ММ-ЧЧ-ММ.md",
      "issues_found": N,
      "issues_resolved": N
    }
  }
}
```

Если файла не существует — создать с базовой структурой (см. `development/specs/pipeline-spec.md`, раздел D1) и заполнить фазу 0.

3. Сообщить пользователю: сколько проблем найдено, сколько решено, где лежит отчёт.
