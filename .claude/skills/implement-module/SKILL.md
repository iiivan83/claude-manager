# Реализация модуля по спецификации

THINK HARD. На каждом шаге этого скилла используй максимальную глубину мышления. Это обязательное требование для каждого шага без исключений.

Ты реализуешь один модуль проекта Claude Manager (Telegram-бот, управляющий Claude Code с телефона). Твоя задача: прочитать спецификацию, написать код, написать тесты, добиться зелёных тестов, проверить качество кода.

Модуль — это один Python-файл в `src/claude_manager/`, который выполняет конкретную задачу в системе. Спецификация модуля — это подробное ТЗ, описывающее публичный API, алгоритм каждой функции, зависимости и тест-план.

## Аргумент

Имя модуля — передаётся как аргумент (например, `config`, `message_splitter`, `session_manager`).

Если имя модуля не передано — спроси у пользователя: «Какой модуль нужно реализовать? Список модулей и порядок реализации — в `development/specs/module-dependency-graph.md`.»

Имя модуля = `$ARGUMENTS`

## Шаг 1: Сбор контекста

THINK HARD.

Прочитай все источники параллельно (через Agent tool, если доступен):

### 1.1 Спецификация модуля

Прочитай `development/specs/{module}_spec.md`. Это главный документ — вся реализация строится на нём.

Извлеки и запомни:
- **Публичный API** — все функции с сигнатурами (имена, аргументы, типы, возвраты)
- **Внутренние функции** — приватные функции с сигнатурами
- **Алгоритм работы** — пошаговая логика каждой функции
- **Зависимости** — какие модули импортирует и какие функции из них вызывает
- **Константы** — именованные константы с описаниями
- **Обработка ошибок** — что может пойти не так и как реагировать
- **Тест-план** — конкретные тест-кейсы с входами и ожидаемыми результатами

Если файл спецификации не найден — сообщи пользователю: «Спецификация `development/specs/{module}_spec.md` не найдена. Сначала создай её через скилл spec-module.»

### 1.2 CLAUDE.md

Прочитай `CLAUDE.md` (корень проекта). Извлеки правила, которые влияют на код:
- **Python стиль** — Python 3.13, type hints обязательны, docstrings одной строкой, snake_case
- **Асинхронность** — async/await, блокирующие операции через `asyncio.to_thread()`
- **Обработка ошибок** — logging (не print), конкретные сообщения об ошибках
- **Безопасность** — секреты в .env, html.escape() для пользовательских данных
- **Архитектура** — слоёная архитектура, правило зависимостей (верхний слой вызывает нижний, не наоборот)

### 1.3 Код модулей-зависимостей

Для каждого модуля из раздела «Зависимости» спецификации прочитай его реализацию `src/claude_manager/{dependency}.py`.

Зачем: ты будешь вызывать функции этих модулей — нужно знать их реальные сигнатуры и поведение, а не только то, что написано в спецификации. Реальный код может немного отличаться от спецификации (допустимые отклонения при реализации).

Если файл зависимости не найден — это проблема порядка реализации. Сообщи пользователю: «Модуль-зависимость `{dependency}` ещё не реализован. Он должен быть реализован раньше.»

### 1.4 Тестовая инфраструктура

Прочитай `tests/conftest.py` — узнай какие фикстуры доступны и как они работают. Тесты будут использовать эти фикстуры:
- `mock_bot` — фейковый Telegram-бот, записывающий отправленные сообщения
- `mock_update` — фабрика фейковых входящих Telegram-сообщений
- `mock_claude_process` — фейковый процесс Claude CLI
- `session_dir` — временная папка для сессий
- `allowed_user_id` — ID пользователя из белого списка

Если `conftest.py` не найден — сообщи пользователю: «Тестовая инфраструктура не настроена. Сначала запусти скилл project-setup.»

## Шаг 2: Написание кода модуля

THINK HARD.

Создай файл `src/claude_manager/{module}.py` строго по спецификации.

### Порядок написания

1. **Импорты** — стандартная библиотека, затем сторонние пакеты, затем локальные модули. Группы разделены пустой строкой
2. **Константы** — все именованные константы из спецификации, с комментариями
3. **Публичные функции** — в том порядке, в каком они описаны в спецификации
4. **Внутренние функции** — приватные функции (с префиксом `_`), сгруппированные рядом с публичными функциями, которые их вызывают

### Правила кода

Эти правила — не формальность, а защита от реальных проблем. Каждое основано на опыте поддержки кода.

- **Type hints обязательны** для всех публичных функций. Используй встроенные типы Python 3.13: `str | None`, `list[str]`, `dict[str, int]` — не `typing.Optional`, не `typing.List`
- **Docstring** — одна строка для каждой публичной функции: `"""Что делает функция."""`
- **Функции до 20 строк**. Если функция длиннее — разбей на вспомогательные. Длинная функция — это функция, которая делает больше одной вещи, и при чтении приходится держать в голове слишком много контекста
- **Максимум 2 уровня вложенности** (if/for внутри if/for). Если глубже — вынеси внутренний блок в отдельную функцию. Глубокая вложенность затрудняет понимание потока выполнения
- **Никаких magic numbers** — каждое числовое или строковое значение должно быть именованной константой с понятным именем, или иметь комментарий, объясняющий его смысл
- **Никакого дублирования** — если одна и та же логика появляется в двух местах, вынеси в общую функцию
- **Именование** — snake_case для функций и переменных, UPPER_SNAKE для констант. Имя должно описывать назначение без необходимости читать код функции
- **Async** — если функция работает с I/O (файлы, сеть, процессы), она должна быть async. Блокирующие вызовы — через `asyncio.to_thread()`
- **Обработка ошибок** — ловить ошибки от внешних систем (Telegram API, Claude CLI, файловая система), логировать через `logging`, отдавать понятные сообщения
- **Импорт зависимостей** — импортируй ровно то, что указано в спецификации. Не добавляй лишних зависимостей

### Верность спецификации

Код должен реализовывать ровно то, что описано в спецификации. Не больше (не добавляй функции, которых нет в спеке) и не меньше (не пропускай описанные функции).

Если при написании кода обнаружишь, что спецификация содержит ошибку или пробел — запиши это, но реализуй максимально близко к спецификации. Расхождения отмети в финальном отчёте.

## Шаг 3: Написание юнит-тестов

THINK HARD.

Создай файл `tests/test_{module}.py` по тест-плану из спецификации.

### Структура тестового файла

1. **Импорты** — pytest, unittest.mock, модуль под тестом, зависимости для моков
2. **Тесты** — каждый тест-кейс из спецификации становится отдельной функцией `test_*`

### Правила написания тестов

- **Каждый тест-кейс из спецификации** — отдельная тестовая функция. Не объединяй несколько кейсов в один тест
- **Используй фикстуры из conftest.py** — `mock_bot`, `mock_update`, `mock_claude_process`, `session_dir`, `allowed_user_id`. Не создавай собственные моки для того, что уже есть в фикстурах
- **Arrange-Act-Assert** — каждый тест состоит из трёх блоков: подготовка данных, вызов тестируемой функции, проверка результата. Разделяй блоки пустой строкой для читаемости
- **Один тест = одна проверка**. Тест проверяет одно конкретное поведение. Если нужно проверить несколько аспектов — создай несколько тестов
- **Моки для внешних зависимостей** — Telegram API, subprocess, файловая система. Тесты не должны зависеть от внешних сервисов
- **Async тесты** — для асинхронных функций используй `async def test_*`. pytest-asyncio (настроен в pyproject.toml с `asyncio_mode = "auto"`) автоматически запускает их
- **Конкретные данные** — используй реальные значения из тест-плана спецификации, а не абстрактные плейсхолдеры

### Покрытие

Тестовый файл должен содержать тесты для всех кейсов из спецификации:
- Юнит-тесты (основное поведение)
- Граничные случаи (edge cases)
- Тесты ошибок (error cases)

## Шаг 4: Запуск тестов модуля

THINK HARD.

Запусти тесты только текущего модуля:

```bash
cd <корень_проекта> && python -m pytest tests/test_{module}.py -v
```

### Если тесты зелёные

Переходи к шагу 5 (регрессионная проверка).

### Если тесты красные — цикл починки

Максимум 5 итераций. На каждой итерации:

1. **Прочитай вывод pytest** — какие тесты упали и с какими ошибками
2. **THINK HARD** — определи причину: баг в коде модуля или баг в тесте. Это важное разделение:
   - Если тест ожидает правильное поведение, а код делает неправильное — баг в коде
   - Если код работает правильно, а тест ожидает неправильное — баг в тесте
   - Если ошибка импорта или TypeError — скорее всего несовпадение сигнатур
3. **Исправь** — внеси минимальное изменение, которое исправляет проблему. Не переписывай всё — точечная починка
4. **Перезапусти тесты модуля** — `pytest tests/test_{module}.py -v`
5. Если снова красные — следующая итерация

Если после 5 итераций тесты всё ещё красные — остановись и сообщи пользователю. Покажи:
- Какие тесты падают
- Какие ошибки
- Что ты пробовал исправить
- В чём, по-твоему, корневая проблема

## Шаг 5: Регрессионная проверка

THINK HARD.

Запусти ВСЕ тесты проекта — не только текущего модуля.

Сначала проверь, существует ли скрипт `development/scripts/run-all-tests.sh`. Если да — используй его:

```bash
cd <корень_проекта> && ./development/scripts/run-all-tests.sh
```

Если скрипта нет — запусти pytest напрямую:

```bash
cd <корень_проекта> && python -m pytest tests/ -v
```

Это критический шаг. Новый модуль мог сломать существующие тесты — например, если изменил общий модуль-зависимость или если тесты разных модулей конфликтуют.

### Если все тесты зелёные

Переходи к шагу 6 (самопроверка).

### Если есть красные тесты — цикл починки регрессий

Красные тесты других модулей — это регрессия, которую ты ввёл. Её нужно исправить, не ломая тесты текущего модуля.

На каждой итерации:

1. **Определи** какие тесты сломались и к какому модулю они относятся
2. **THINK HARD** — пойми связь между твоими изменениями и сломавшимися тестами. Что именно ты изменил, что повлияло на другой модуль?
3. **Исправь** — минимальное изменение, не ломающее тесты текущего модуля
4. **Перезапусти ВСЕ тесты** — не только сломавшиеся, а все. Это ключевое правило: каждая починка может создать новую регрессию, поэтому проверяем всё целиком

Повторяй до полностью зелёного состояния. Если не удаётся добиться зелёного за 5 итераций — остановись и сообщи пользователю.

## Шаг 6: Самопроверка по чеклисту

THINK HARD.

Перечитай написанный код (`src/claude_manager/{module}.py`) и тесты (`tests/test_{module}.py`). Проверь каждый пункт:

### Чеклист кода модуля

- [ ] **Type hints** — все публичные функции имеют полные type hints для аргументов и возвратов
- [ ] **Docstrings** — каждая публичная функция имеет однострочный docstring
- [ ] **Нет magic numbers** — каждое числовое/строковое значение — именованная константа или имеет комментарий
- [ ] **Нет дублирования** — одинаковая логика не повторяется в двух+ местах
- [ ] **Именование понятно** — имена функций и переменных описывают назначение без необходимости читать код
- [ ] **Функции до 20 строк** — ни одна функция не превышает 20 строк
- [ ] **Максимум 2 уровня вложенности** — нет if/for внутри if/for внутри if/for
- [ ] **Async корректен** — I/O функции асинхронны, блокирующие вызовы через `asyncio.to_thread()`
- [ ] **Обработка ошибок** — внешние вызовы обёрнуты в try/except, ошибки логируются
- [ ] **Импорты правильные** — порядок: стандартная библиотека, сторонние, локальные

### Чеклист тестов

- [ ] **Все тест-кейсы из спецификации реализованы**
- [ ] **Фикстуры из conftest.py используются** (а не самодельные моки)
- [ ] **Каждый тест проверяет одно поведение**
- [ ] **Конкретные данные** из тест-плана спецификации

Если нашёл нарушение — исправь и перезапусти ВСЕ тесты (не только модуля). Каждое исправление может создать регрессию — контролируй это.
