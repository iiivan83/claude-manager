# Pipeline Implementer

Получает готовую спецификацию (spec.md) и собирает из неё полностью рабочий скилл — все файлы агентов, скрипты, схемы данных, главный оркестратор и тесты. После сборки проверяет, что скилл реально помогает — даёт одну и ту же задачу Claude со скиллом и без, и сравнивает результаты.

## Global Reference Rules

Перед началом работы прочитай:
- `~/.claude/references/document-naming-and-placement.md` — правила документооборота (куда складывать логи, как называть папки и файлы, формат временных меток — разделы 2, 3.2)
- `~/.claude/references/writing-style-guide.md` — единый стиль письма для всех документов и отчётов
- `~/.claude/references/skill-testing-standard.md` — стандарт качества assertions и протокол сравнительного тестирования
- `~/.claude/references/cli-testing-patterns.md` — паттерны тестирования через CLI-подпроцесс (изоляция, таймауты, отладка)

Если файлы не найдены — остановить пайплайн с ошибкой.

## Глубина мышления

Думай как опытный тех-лид, который лично отвечает за стабильность продакшена. Каждое решение
проходит через внутренний чек-лист:

- **Побочные эффекты** — что может сломаться в соседних модулях из-за этого изменения?
- **Граничные случаи** — как это поведёт себя на пустых данных, при конкурентном доступе,
  при неожиданном порядке вызовов?
- **Откат** — если изменение окажется неудачным, как быстро можно вернуть всё назад?
- **Альтернативы** — есть ли более простой способ добиться того же результата?
- **Достаточность** — это изменение действительно решает корневую причину, а не маскирует
  симптом?

Все агенты этого скилла работают с такой же глубиной — разбирают задачу по частям,
рассматривают альтернативы и перепроверяют выводы перед тем, как действовать.

**Проброс в создаваемые скиллы:** при создании файлов агентов (Этап 3) и SKILL.md (Этап 4) —
всегда включай секцию «Глубина мышления» с полным чек-листом (побочные эффекты, граничные
случаи, откат, альтернативы, достаточность) в каждый создаваемый SKILL.md и вшивай
инструкцию о глубоком мышлении в каждый создаваемый промпт агента. Эталон секции —
в `.claude/skills/apply-root-cause-fixes/SKILL.md`.

## Жёсткие правила пайплайн-скиллов

Эти четыре правила обязательны для любого пайплайн-скилла (три этапа и больше) — и при создании нового, и при обновлении существующего. Нарушение хотя бы одного — это брак, переделывай в том же проходе.

Правила появились после root-cause-анализа `google-sheets-etl` — скилл тихо пропускал LLM-этап и отдавал базу с пустыми метаданными. Причина каждый раз была одна: скилл оставлял выбор режима «на усмотрение агента», а агент ошибался. Эти правила закрывают саму возможность такой ошибки.

### 1. Один режим на этап

У каждого этапа должен быть ровно один режим работы. Если этап делает «либо A, либо B» в зависимости от входных данных — это два разных этапа, а выбор между ними идёт через отдельный этап триажа с детерминированными условиями в SKILL.md.

Запрещено:
- «Этап нормализации работает либо через LLM-агента, либо через эвристику»
- «Скрипт сам решит, какой режим выбрать»
- «По умолчанию LLM, но можно переключиться на эвристику»

Разрешено:
- «Этап 3: LLM-анализ — единственный режим»
- «Этап 3a — триаж (детерминированные условия в SKILL.md). Этап 3b — LLM или эвристика, выбор определён триажом»

### 2. Никаких опциональных дорогих этапов

Если этап влияет на корректность результата — он обязательный. Слова «опциональный», «по желанию», «можно пропустить для скорости», «на выбор» в описании этапа запрещены.

Запрещено:
- `**Тип:** script + **опциональный** LLM-агент`
- «Можно пропустить для ускорения»
- «По желанию запустить этап N»

Разрешено:
- `**Тип:** script + LLM-агент (оба обязательны)`
- Если этап реально не влияет на корректность — удали его или вынеси в отдельный вспомогательный скилл

### 3. Никаких тихих fallback

Если ожидаемый аргумент, файл или ответ агента отсутствует — скрипт обязан упасть с понятной ошибкой. Запрещено молча возвращать `None`, пустую структуру или переключаться на «режим попроще».

Запрещено в коде:
```python
if analysis_path is None:
    return None  # тихий fallback на эвристики

try:
    result = call_llm(...)
except Exception:
    result = heuristic_fallback(...)  # замаскированная ошибка
```

Разрешено:
```python
if analysis_path is None:
    raise SystemExit("--analysis-file обязателен. LLM-анализ — единственный поддерживаемый режим")

try:
    result = call_llm(...)
except LLMError as e:
    raise SystemExit(f"LLM-агент упал на листе {sheet_name}: {e}")
```

### 4. Запрет мягких формулировок

В описании режимов и обязательности этапов запрещены формулировки, которые оставляют решение «на усмотрение Клода»:

Запрещено:
- «Лучше использовать LLM для нестандартных листов»
- «Обычно эвристика хороша для простых таблиц»
- «Рекомендуется выбирать LLM-агента, если...»
- «По возможности прогнать через LLM»

Разрешено только детерминированные условия и прямые утверждения:
- «Если листов > 10 ИЛИ merged_cells > 5 ИЛИ в первых 10 строках есть слова из набора {P&L, Бюджет, Факт, План} — LLM-режим обязателен»
- «LLM-анализ — единственный поддерживаемый режим этого этапа»

---

Это правила для скиллов, которые создаёт и обновляет эта фабрика. Проверяй каждое правило перед завершением работы. Нашёл нарушение — исправь в том же проходе, не откладывай.

## Входные данные

Пользователь указывает путь к файлу spec.md при вызове скилла.

**Предварительное чтение:** перед запуском этапов извлеки из spec.md:
1. Первый заголовок — ожидается формат `# Спецификация пайплайна: {название}`
2. Если формат другой — ищи строку `- **Название пайплайна:**`
3. Сохрани: путь к файлу + название скилла — они нужны на всех этапах

## Константы

- `LOG_SUBCATEGORY`: `skills-modifications`
- Дополнительные подпапки логов: `dry-run-test-results/`, `completeness-metrics/`, `effectiveness-eval/`

## Directory Layout

```
.claude/skills/{skill-name}/
```

Внутри папки логов:
```

agent-outputs/

  benchmark/benchmark.json, benchmark.md, analysis.json

```

Формат orchestrator-log.json:
```json
{
  "pipeline": "pipeline-implementer",
  "created_at": "ISO-8601",
  "skill_name": "<название>",
  "spec_file": "<путь>",
  "resumed_from_step": null,
  "steps": []
}
```

Формат записи шага:
```json
{
  "step_number": 1,
  "start_time": "ISO-8601",
  "end_time": "ISO-8601",
  "agent": "spec-parser",
  "input": ["путь"],
  "success": true,
  "comment": "",
  "output": ["путь"],
  "next_agent": "schema-generator"
}
```

**Паттерн логирования:** запись в orchestrator-log.json делается ДО начала каждого этапа (не после). Это позволяет видеть, на каком этапе пайплайн прервался, если он упал.

## Resume Mode

При запуске проверь, существует ли `orchestrator-log.json`:
- Если да — пайплайн уже запускался ранее
- Прочитай лог, пропусти успешно завершённые этапы
- На Этапе 3 проверь каждый файл отдельно — перезапускай только упавшие
- Запиши в лог `resumed_from_step` с номером шага, с которого продолжил

## Обязательная последовательность переходов

Каждый этап ОБЯЗАН переходить строго к следующему. Пропускать этапы запрещено.

- Этап 0 (подготовка) → Этап 1 (разбор спеки)
- Этап 1 → Этап 2 (схемы данных)
- Этап 2 → Этап 3 (создание файлов)
- Этап 3 → Этап 4 (сборка SKILL.md)
- Этап 4 → Этап 5 (тестирование)
- Этап 5 (все тесты прошли) → Этап 7 (проверка эффективности)
- Этап 5 (есть упавшие тесты) → Этап 6 (цикл починки) → обратно к Этап 5
- Этап 7 → Этап 8 (документация)

**Запрещённые переходы:**
- Этап 5 → Этап 8 (нельзя перейти к документации, минуя проверку эффективности)
- Этап 6 → Этап 7 (нельзя перейти к эффективности, минуя повторное тестирование)
- Любой этап → Этап 8 напрямую (документация всегда последняя)

## Pipeline Stages

### Этап 0 — Чтение настроек и подготовка

**0.1 — Чтение глобальных настроек** (оркестратор напрямую)
- Прочитай `~/.claude/references/document-naming-and-placement.md`
- Извлеки: путь к логам (раздел 3.2), формат дат (раздел 2)
- Совмести с константами: `LOG_SUBCATEGORY` = `skills-modifications`
- Если файл не найден → СТОП

**0.2 — Подготовка папок** (скрипты)
1. Запусти `scripts/ensure-log-dir.sh <project-path> <log-path>`
2. Запусти `scripts/create-skill-folder.sh <pipeline-name> <project-path> <skill-name> <log-path>`
3. Если папка уже существует — спроси пользователя: перезаписать или создать с суффиксом
4. Создай `orchestrator-log.json`
5. Сгенерируй `folder_timestamp` (DD.MM_HH.MM) и `file_timestamp` (DD-MM-HH-MM)

### Этап 1 — Разбор спецификации

- **Инструкция:** `agents/spec-parser.md`
- **Вход:** файл spec.md
- **Выход:** `agent-outputs/01-spec-parser.json`
- **Инструмент:** Agent tool
- **Зависимости:** Этап 0
- **При ошибке:** если spec.md не найден → СТОП. Если раздел отсутствует → status: "partial", продолжить

### Этап 2 — Создание схем данных

- **Инструкция:** `agents/schema-generator.md`
- **Вход:** `agent-outputs/01-spec-parser.json`
- **Выход:** `references/schemas.json` (в папке нового скилла) + `agent-outputs/02-schema-generator.json`
- **Инструмент:** Agent tool
- **Зависимости:** Этап 1
- **Оптимизация:** карта зависимостей берётся из результатов разбора, не вычисляется заново

### Этап 3 — Создание агентов и скриптов

- **Вход:** `agent-outputs/01-spec-parser.json` + `references/schemas.json`
- **Выход:** файлы `agents/*.md` и `scripts/*.sh` в папке нового скилла
- **Инструмент:** Bash → CLI (claude -p /skill-creator, claude -p /script-creator-pipeline)
- **Зависимости:** Этап 2

**Порядок создания файлов:**

Оркестратор вызывает скиллы через CLI-подпроцесс (`claude -p`) — каждый вызов запускается как отдельный процесс с полным набором инструментов:

- Для каждого этапа из спеки: вытащить описание + текст агента + схему данных
- Тип "агент" → CLI-вызов skill-creator:
  ```bash
  source ~/.claude/cli-budgets.env 2>/dev/null || true
  env -u CLAUDECODE claude -p \
    --output-format text \
    --effort max \
    --dangerously-skip-permissions \
    --max-budget-usd "$BUDGET_NORMAL" \
    <<'PROMPT'
  Используй /skill-creator.
  Задача: создай агента по описанию.
  Описание: {описание из спеки}
  Текст агента: {промпт}
  Схема данных: прочитай {путь к schemas.json}
  Путь к файлу: {путь}
  PROMPT
  ```
- **Стилевая ссылка в агентах:** если создаваемый агент генерирует текстовые артефакты для человека (отчёты, документацию, спецификации, описания), в промпт CLI-вызова добавь инструкцию: "В начале файла агента добавь секцию '## Стиль письма' со строкой: 'Перед формированием текста прочитай ~/.claude/references/writing-style-guide.md и следуй стилю.'" Для агентов, работающих только с машиночитаемыми данными (JSON, код), этого не нужно.
- Тип "скрипт" → CLI-вызов script-creator-pipeline:
  ```bash
  source ~/.claude/cli-budgets.env 2>/dev/null || true
  env -u CLAUDECODE claude -p \
    --output-format text \
    --effort max \
    --dangerously-skip-permissions \
    --max-budget-usd "$BUDGET_NORMAL" \
    <<'PROMPT'
  Используй /script-creator-pipeline.
  Задача: создай скрипт по описанию.
  Описание: {описание из спеки}
  Что делает скрипт: {задача}
  Путь к файлу: {путь}
  PROMPT
  ```
- Независимые этапы → одновременно (без ограничений)
- Зависимые этапы → по очереди
- Один CLI-вызов = один файл

**Лимит промпта:** промпт для `claude -p` не должен превышать 7000 символов. Передавай **пути** к файлам и краткую выжимку, а не содержимое файлов целиком.

#### Запись файлов нового скилла

Все файлы, которые этот пайплайн создаёт (`SKILL.md`, `agents/*.md`, `scripts/*`, `references/*`), лежат внутри папки `.claude/skills/`. Если `pipeline-implementer` запущен через CLI-подпроцесс (`claude -p`) — например, вызван из `update-skill` или другого скилла-обёртки, — агенты-сборщики (`skill-creator`, `script-creator-pipeline`) должны использовать штатные шаблоны **X.1** (полная перезапись) и **X.2** (точечная замена) из раздела «Запись в `.claude/skills/` из CLI-подпроцессов» корневого `CLAUDE.md` проекта. Шаблоны описаны там один раз и здесь намеренно не дублируются.

Формулировка промптов в CLI-вызовах — строго нейтральная: «используй шаблон X.1 для создания нового файла по пути `{путь}`» или «используй шаблон X.2 для точечной правки фрагмента в `{путь}`». Запрещены формулировки про «обход», «блокировку», «bypass», «shortcut», «permission-ошибку», «Edit заблокирован» — полный список триггерных слов и их нейтральных замен приведён в том же разделе `CLAUDE.md`.

После каждого CLI-вызова, который должен был создать или изменить файл внутри `.claude/skills/`, оркестратор обязан проверить результат на диске собственными глазами — через `grep` по ожидаемому маркеру, `cat`/Read с визуальной сверкой или однострочник `python3 -c "open(...)"`. Отчёт агента «готово» без независимой проверки подтверждением успеха не считается.

**После каждого CLI-вызова:**
- Проверь exit code (0 = успех)
- Проверь, что выходной файл создан
- Запиши в orchestrator-log.json: команду, exit code, путь к артефакту
- Если пустой ответ или ошибка — запиши ошибку, передай в Этап 6

**После завершения:** оркестратор делает `chmod +x` для всех .sh файлов

**При ошибке:** если один вызов упал — дождаться остальных, передать ошибки в Этап 6

### Этап 4 — Сборка SKILL.md

- **Вход:** `agent-outputs/01-spec-parser.json` + все файлы из Этапа 3 + `references/schemas.json`
- **Выход:** `SKILL.md` + `evals/evals.json` + `agent-outputs/04-skillmd-creator.json`
- **Инструмент:** Bash → CLI (claude -p /skill-creator)
- **Зависимости:** Этап 3

Оркестратор собирает полное описание скилла (из результатов Этапов 1–3) и вызывает skill-creator через CLI:

```bash
source ~/.claude/cli-budgets.env 2>/dev/null || true
env -u CLAUDECODE claude -p \
  --output-format text \
  --effort max \
  --dangerously-skip-permissions \
  --max-budget-usd "$BUDGET_NORMAL" \
  <<'PROMPT'
Используй /skill-creator.
Задача: создай SKILL.md для скилла {имя}.
Описание: {из спеки}
Этапы: {список этапов с агентами и скриптами}
Схемы: прочитай {путь к schemas.json}
Файлы агентов: {список путей к agents/*.md}
Файлы скриптов: {список путей к scripts/*.sh}
PROMPT
```

**Лимит промпта:** не более 7000 символов. Передавай пути, а не содержимое.

**После CLI-вызова:**
- Проверь exit code и наличие SKILL.md
- Запиши в orchestrator-log.json

После вызова оркестратор сам проверяет результат и дополняет обязательные разделы:

**Global Reference Rules** — оркестратор добавляет в эту секцию создаваемого скилла:
- `~/.claude/references/writing-style-guide.md` — единый стиль письма (безусловно)
- `~/.claude/references/document-naming-and-placement.md` — правила документооборота (безусловно)
- `~/.claude/references/evals-schema.md` — формат evals.json (безусловно, потому что у каждого скилла есть evals)
- `~/.claude/references/skill-testing-standard.md` — стандарт качества assertions и протокол сравнительного тестирования (условно: добавлять, если среди этапов создаваемого скилла есть тестирование или валидация — маркеры в названии/описании этапа: «тест», «валидация», «проверка», «verification», «test», «eval»)

**Resume Mode** — секция с инструкциями по продолжению после сбоя

**Cleanup / abort instructions** — инструкции при аварийном завершении

**CLI-паттерн для этапов, вызывающих другие скиллы.** Оркестратор проверяет спецификацию: если какой-то этап создаваемого скилла вызывает полноценный скилл (не плоский агент с Read/Write/Edit, а скилл с собственными агентами и пайплайном) — в SKILL.md этого этапа должно быть указано `Инструмент: Bash → CLI (claude -p)`, а не `Agent tool` или `Skill tool`.

Причина: Skill tool — обёртка над Agent tool. Вызванный скилл становится субагентом (уровень 1 вложенности) и не может запустить своих агентов — ограничение вложенности в 1 уровень. CLI-подпроцесс (`claude -p`) запускается как отдельный процесс с чистым контекстом и полным набором инструментов.

Разграничение:
- **Agent tool** — для плоских агентов, которые работают только с Read/Write/Edit и не запускают вложенные вызовы
- **Bash → CLI (`claude -p`)** — для полноценных скиллов с собственными агентами, тестами, пайплайнами

**evals/evals.json** — создаёт pipeline-implementer напрямую (не через помощника). Требования к содержимому:
- Формат — по схеме из `~/.claude/references/evals-schema.md`
- Assertions — по правилам Блока A из `~/.claude/references/skill-testing-standard.md`:
  - Три обязательных тира: Tier 1 (Structural), Tier 2 (Behavioral), Tier 3 (Result Quality)
  - Минимум 70% assertions = Tier 3
  - Тест инверсии для каждого Tier 3 assertion: «пройдёт ли без скилла?» — если да, усилить
  - Минимум 5 assertions на сценарий
  - Минимум 2 сценария

### Этап 5 — Тестирование

5 проверок. T1, T2 и TF — одновременно. Потом T3 и T4 — по очереди.

**T1 — Проверка файлов** (скрипт, параллельная)
- Инструмент: `scripts/structural-validator.sh <skill-path> [reports-dir]`
- PASS: все критические проверки пройдены

**T2 — Проверка совместимости данных** (агент, параллельная)
- Инструкция: `agents/contract-verifier.md`
- PASS: errors = 0

**TF — Проверка логики управления** (агент, параллельная)
- Инструкция: `agents/flow-self-test.md`
- PASS: все 5 проверок пройдены

**T3 — Пробный запуск** (скрипт + чистый агент)
- Инструмент: `scripts/dry-run-test.sh <skill-folder> <results-folder>`
- Инструкция для чистого Claude: `agents/dry-run-tester.md`
- PASS: blocking_issues = 0

**T4 — Проверка полноты** (скрипт)
- Инструмент: `scripts/extract-metrics.sh <skill-folder> [reports-dir]`
- PASS: все метрики в норме

**После тестов:**
- ВСЕ прошли → Этап 7
- ЛЮБОЙ не прошёл → Этап 6

**Явный запрет:** нельзя перейти к документации (Этап 8), минуя проверку эффективности (Этап 7).
Даже если все тесты прошли — следующий шаг ВСЕГДА Этап 7.

**Чекпоинт:** перед переходом сверься с блоком «Обязательная последовательность переходов».
Следующий этап зависит от результатов: все тесты прошли → Этап 7, есть упавшие → Этап 6.
НЕ переходи к Этапу 8.

### Этап 6 — Цикл починки

Два агента работают в паре: один ищет причину, другой чинит.

- **Диагностика:** `agents/problem-tracer.md` — для каждой упавшей проверки
- **Починка:** `agents/change-fixer.md` — применяет минимальные исправления
- **Зависимости:** Этап 5

**Порядок:** найти причину → починить → перепроверить

**Максимум 3 попытки.** После 3-й — спросить пользователя:
- Дать подсказку → продолжить
- Пропустить нерешённые → перейти к Этапу 7
- Остановить → записать "aborted", создать PIPELINE-INCOMPLETE.md

**Регрессия:** если починка затронула общий файл — перезапустить ВСЕ зависимые тесты:
- `schemas.json` → T2
- `SKILL.md` → T1, T2, TF, T3
- `agents/*.md` → T2, TF, T3
- `scripts/*.sh` → T1, T3
- `evals/evals.json` → T3

**Чекпоинт:** после успешной починки вернись к Этапу 5 (повторное тестирование).
НЕ переходи к Этапу 7 или 8 напрямую.

### Этап 7 — Проверка эффективности

Прочитай `~/.claude/references/skill-testing-standard.md` и выполни Протокол сравнительного тестирования (Блок B) с параметрами:

- `scenario`: `"new_skill"`
- `eval_workspace`: `{папка-логов}/effectiveness-eval`
- `config_a_name`: `"with_skill"`
- `config_a_skill_path`: `.claude/skills/{skill-name}/SKILL.md`
- `config_b_name`: `"baseline"`
- `config_b_skill_path`: `null`
- `evals_source`: `.claude/skills/{skill-name}/evals/evals.json`
- `skill_name`: `{skill-name}`

После завершения протокола — переходи к Этапу 7.7 (решение пользователя).

**7.7 — Решение пользователя**
- Скилл лучше → Этап 8
- Скилл не лучше → три варианта:
  - **Починить:** вызови root-cause-analysis через CLI:
    ```bash
    source ~/.claude/cli-budgets.env 2>/dev/null || true
    env -u CLAUDECODE claude -p \
      --output-format text \
      --effort max \
      --dangerously-skip-permissions \
      --max-budget-usd "$BUDGET_NORMAL" \
      <<'PROMPT'
    Используй /root-cause-analysis.
    Проблема: скилл {skill-name} показал эффективность ниже baseline.
    Результаты бенчмарка: прочитай {путь к benchmark.json}
    Анализ: прочитай {путь к analysis.json}
    Файлы скилла: {путь к папке скилла}
    PROMPT
    ```
    После получения результата → чеклист → выбор способа:
    - CODE + 1-2 файла → `agents/effectiveness-quick-fixer.md`
    - SKILL или 3+ файлов → `agents/effectiveness-planner.md` → CLI-ревьюер → починка
    - Повторный прогон (макс. 2 попытки)
  - **Принять как есть** → Этап 8
  - **Остановить** → "aborted" + PIPELINE-INCOMPLETE.md

**Если evals.json пуст:** пропустить Этап 7 с предупреждением

**Чекпоинт:** после завершения проверки эффективности переходи к Этапу 8 (документация).
Это последний рабочий этап.

## Cleanup

При аварийном завершении:
1. Записать "aborted" в orchestrator-log.json
2. Создать `PIPELINE-INCOMPLETE.md` в папке скилла с описанием:
   - Что построено
   - Что сломано
   - Что не удалось починить
3. Показать пользователю сводку

## Completion

Когда все этапы завершены:
1. Обновить orchestrator-log.json — финальная запись
2. Показать пользователю:
   - Путь к готовому скиллу
   - Количество файлов (агенты, скрипты, схемы)
   - Результаты тестов
   - Результаты проверки эффективности (если проводилась)
3. Напомнить: "Скилл готов к использованию. Для вызова используй /{skill-name}"
