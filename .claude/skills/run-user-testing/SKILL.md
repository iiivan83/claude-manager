# Прогон пользовательских тест-сценариев

**THINK HARD** на каждом шаге выполнения этого скилла. Используй максимальную глубину мышления при принятии каждого решения — от чтения сценариев до LLM-верификации ответов и диагностики ошибок. Каждое решение здесь влияет на качество бота, который будет использоваться ежедневно.

Ты прогоняешь готовые пользовательские тест-сценарии на живом боте. Сценарии уже написаны скиллом create-user-test-scenarios — твоя задача выполнить их, проверить ответы бота через LLM-верификацию и починить всё, что не работает. Это последний рубеж качества перед завершением проекта.

Ключевое отличие от E2E-тестов (test-e2e): E2E тесты генерируют тест-план из BRD на лету; пользовательские тесты используют **заранее написанные сценарии** с конкретными шагами, ожидаемыми результатами и критериями pass/fail. Сценарии учитывают результаты предыдущего E2E тестирования и покрывают граничные случаи.

## Предварительные проверки

**THINK HARD** — без этих проверок тестирование невозможно. Каждая пропущенная проверка приведёт к загадочным ошибкам, которые будет трудно отследить.

## Шаг 1: Разбор сценариев

**THINK HARD** — правильный разбор сценариев определяет весь дальнейший прогон. Пропущенный шаг или неверно понятый критерий = некорректная верификация.

Прочитай файл сценариев и для каждого сценария выдели:

- **ID** — идентификатор сценария (например, CJM-04-01)
- **Название** — что тестируется
- **Предусловия** — что должно быть настроено до начала
- **Шаги** — конкретные действия: какие сообщения отправить, какие команды выполнить, какие фото послать
- **Ожидаемый результат** — что должен ответить бот на каждом шаге
- **Критерии pass/fail** — однозначные условия прохождения
- **Связь с BRD** — какой CJM и какие требования проверяются

Составь внутренний список сценариев в порядке прогона. Порядок важен — сценарии могут зависеть друг от друга (нельзя тестировать отправку сообщения, если сессия не создана).

## Шаг 3: Подключение к Telegram

**THINK HARD** — авторизация Telethon может потребовать SMS-код от пользователя. Это интерактивный шаг — скилл должен остановиться и подождать ввода.

### 3.1 Инициализация TelegramTestClient

Прочитай `tests/e2e/test_client.py` и разберись в API клиента. Затем напиши и выполни Python-скрипт для подключения:

```python
import asyncio
import sys
sys.path.insert(0, '/Users/ivan/Desktop/claude-sandbox/claude_manager')

from tests.e2e.test_client import TelegramTestClient
from dotenv import load_dotenv
import os

load_dotenv('/Users/ivan/Desktop/claude-sandbox/claude_manager/.env')

async def connect():
    client = TelegramTestClient(
        api_id=int(os.getenv('TELETHON_API_ID')),
        api_hash=os.getenv('TELETHON_API_HASH'),
        phone=os.getenv('TELETHON_PHONE'),
        bot_username=os.getenv('TELETHON_BOT_USERNAME')
    )
    await client.connect()
    return client

client = asyncio.run(connect())
```

### 3.2 Обработка авторизации Telethon

При **первом** запуске Telethon попросит SMS-код. Если это произошло:

1. **Останови выполнение** и спроси пользователя: «Telethon запрашивает код авторизации. Telegram отправил SMS-код на номер [номер из .env]. Введи код:»
2. Дождись ответа пользователя
3. Введи код в Telethon
4. После успешной авторизации Telethon сохранит сессию в файл `*.session` — при следующих запусках авторизация не потребуется

Если файл `*.session` уже существует — авторизация пройдёт автоматически, без SMS.

## Шаг 4: Прогон сценариев

**THINK HARD** — каждый сценарий требует внимательного выполнения и осмысленной верификации. Механический прогон без анализа ответов пропустит тонкие баги.

Проходи сценарии строго в порядке из файла сценариев. Для каждого сценария:

### 4.1 Проверка предусловий

Перед каждым сценарием проверь, что его предусловия выполнены. Например, если сценарий требует «активная сессия» — убедись, что сессия была создана в предыдущем сценарии. Если предусловие не выполнено — выполни подготовительные шаги (создай сессию, переключись и т.д.).

### 4.2 Выполнение шагов

Для каждого шага сценария используй соответствующий метод TelegramTestClient:

- **Текстовое сообщение:** `client.send_message("текст")` затем `response = client.wait_for_response(timeout=60)`
- **Команда:** `client.send_command("/new")` затем `response = client.wait_for_response(timeout=60)`
- **Фото:** `client.send_photo("/путь/к/файлу", "подпись")` затем `response = client.wait_for_response(timeout=60)`

Таймаут 60 секунд — Claude может думать долго, особенно при первом запросе в сессии. Если ответ не пришёл за 60 секунд — это fail.

Пользователь видит все сообщения в Telegram в реальном времени — ты отправляешь от имени его аккаунта, и бот отвечает в тот же чат.

### 4.3 LLM-верификация ответа

После получения ответа бота выполни LLM-верификацию. Это главная особенность пользовательского тестирования: вместо точного сравнения строк Claude анализирует **смысл** ответа и сверяет его с критериями pass/fail из сценария и с ожидаемым поведением из BRD.

**THINK HARD** при верификации. Для каждого ответа:

1. Прочитай BRD — раздел, соответствующий CJM этого сценария: `development/docs/brd-user-journeys.md`
2. Сопоставь фактический ответ бота с:
   - Ожидаемым результатом из сценария
   - Критериями pass/fail из сценария
   - Ожидаемым поведением из BRD (раздел «Что видит пользователь»)
3. Вынеси вердикт:

```
Ты верификатор пользовательского теста. Сравни ответ бота с ожиданиями.

**Сценарий:** [ID и название]
**Шаг:** [какой шаг выполнялся]
**Что отправил пользователь:** [сообщение/команда]
**Ожидаемый результат (из сценария):** [описание из файла сценариев]
**Критерии pass/fail (из сценария):** [конкретные критерии]
**Ожидаемое поведение (из BRD):** [описание из BRD]
**Фактический ответ бота:** [полный текст ответа]

Вынеси вердикт:
- **PASS** — ответ соответствует ожиданиям (допускаются различия в формулировках, но суть, структура и все обязательные элементы должны присутствовать)
- **FAIL** — ответ не соответствует ожиданиям

Формат ответа:
VERDICT: PASS или FAIL
REASON: краткое обоснование (1-2 предложения)
DETAILS: что именно совпало / не совпало (если FAIL — что конкретно не так)
```

Дополнительно проверяй:
- Наличие номера сессии в заголовке (если ожидается по BRD)
- Правильное форматирование (HTML-разметка отрендерилась)
- Кликабельность номеров сессий (если описана в BRD)
- Полноту ответа (не обрезан, все части доставлены)
- Корректность разбивки длинных сообщений (если ответ длинный)

### 4.4 Запись результата сценария

Для каждого сценария запоминай:
- ID и название сценария
- Вердикт: PASS или FAIL
- Обоснование LLM-верификатора
- Полный текст ответа бота (для отладки при fail)
- Время выполнения (сколько секунд ждали ответ)
- Номер попытки (1 = первый раз, 2+ = после починки)

Если сценарий PASS — переходи к следующему сценарию.
Если сценарий FAIL — переходи к шагу 5 (цикл починки).

## Шаг 5: Цикл починки при FAIL

**THINK HARD** — правильная диагностика причины провала критически важна. Неверный диагноз приведёт к неверному исправлению и новым багам. Не торопись — лучше потратить время на анализ, чем на исправление не того.

Если сценарий завершился FAIL:

### 5.1 Диагностика

Проанализируй упавший сценарий:

1. **Перечитай ответ бота** — что именно он ответил? Это ошибка, неправильный ответ, или отсутствие ответа?
2. **Проверь лог бота:**
   ```bash
   tail -100 /tmp/claude_manager_user_testing.log
   ```
   Ищи ошибки, трейсбеки, предупреждения.
3. **Определи источник проблемы:**
   - Баг в коде бота (ответ не соответствует BRD)
   - Баг в тестовом клиенте (сообщение не доставлено или ответ не прочитан)
   - Проблема с Claude (таймаут, пустой ответ)
   - Проблема с сетью/Telegram
4. **Прочитай код**, который отвечает за упавший сценарий — файлы в `src/claude_manager/`. Сверься с BRD и спецификациями модулей из `development/specs/realized/`.

### 5.2 Остановка бота

```bash
pkill -f "python -m claude_manager"
```

Подожди 2 секунды, убедись что процесс завершился.

### 5.3 Исправление кода

**THINK HARD** — прочитай весь релевантный код, пойми причину, исправь. Не ставь «заплатку» — исправь корневую причину.

Внеси исправления в соответствующие файлы в `src/claude_manager/`.

### 5.4 Регрессионная проверка — ВСЕ автотесты

После каждого исправления — запусти **ВСЕ** автоматические тесты (юнит + интеграционные). Это обязательный шаг: исправление одного бага может сломать другой модуль.

Проверь, существует ли скрипт запуска:
```bash
ls /Users/ivan/Desktop/claude-sandbox/claude_manager/development/scripts/run-all-tests.sh
```

Если существует:
```bash
cd /Users/ivan/Desktop/claude-sandbox/claude_manager && ./development/scripts/run-all-tests.sh
```

Если не существует:
```bash
cd /Users/ivan/Desktop/claude-sandbox/claude_manager && python -m pytest tests/ -v
```

Если автотесты стали красными — это регрессия. Исправь её перед продолжением. Не продолжай с красными автотестами.

### 5.5 Перезапуск бота

```bash
cd /Users/ivan/Desktop/claude-sandbox/claude_manager && nohup python -m claude_manager > /tmp/claude_manager_user_testing.log 2>&1 &
echo $!
```

Подожди 5 секунд для инициализации. Проверь, что бот жив.

### 5.6 Повтор ТОЛЬКО упавшего сценария

Повтори **только** тот сценарий, который не прошёл. Не прогоняй все сценарии заново — это ключевое правило. Ранее пройденные сценарии уже PASS, их не нужно повторять.

Если сценарий снова FAIL — вернись к шагу 5.1.

**Максимум 5 попыток** на один сценарий. Если после 5 попыток сценарий всё ещё FAIL:
1. Зафиксируй статус: сценарий не пройден, описание проблемы, что было испробовано
2. Продолжи со следующим сценарием — не блокируй весь прогон из-за одного бага
3. В итоговом отчёте отметь этот сценарий как FAIL с подробным описанием

### 5.7 Продолжение прогона

После успешной починки (или исчерпания 5 попыток) продолжи прогон со следующего сценария.
