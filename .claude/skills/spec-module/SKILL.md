# Спецификация модуля проекта

THINK HARD. На каждом шаге этого скилла используй максимальную глубину мышления.

Ты создаёшь детальное техническое задание на один модуль проекта Claude Manager (Telegram-бот, управляющий Claude Code с телефона). Спецификация — это фундамент, по которому модуль будет реализован. Ошибка в спецификации порождает каскад ошибок в коде и тестах, поэтому качество здесь критически важно.

## Аргумент

Имя модуля — передаётся как аргумент (например, `config`, `message_splitter`, `session_manager`).

Если имя модуля не передано — спроси у пользователя: «Какой модуль нужно специфицировать? Полный список модулей есть в `dev/docs/specs/module-dependency-graph.md`.»

## Шаг 1: Сбор контекста

THINK HARD.

Прочитай все источники информации параллельно (через Agent tool, если доступен, или последовательно):

### 1.1 Граф зависимостей

Прочитай `dev/docs/specs/module-dependency-graph.md`. Найди в нём:
- **Модуль** — в каком слое он находится, какие CJM обслуживает
- **Зависимости модуля** — от каких модулей он зависит (вызывает их функции)
- **Зависимые от него** — какие модули вызывают его функции (это определяет его публичный API — он должен предоставить то, что от него ждут)

### 1.2 BRD

Прочитай `dev/docs/brd/brd-user-journeys.md`. Найди конкретные CJM, которые обслуживает модуль (номера CJM указаны в графе зависимостей). Для каждого CJM извлеки:
- Что пользователь делает (входные действия)
- Что происходит внутри (логика, относящаяся к этому модулю)
- Что может пойти не так (ошибки, которые этот модуль должен обработать)

### 1.3 CLAUDE.md

Прочитай `CLAUDE.md` (корень проекта). Извлеки:
- Описание модуля из структуры проекта (что модуль делает в одну строку)
- Архитектурный слой модуля и правила зависимостей
- Правила кода: type hints, docstrings, именование, обработка ошибок, асинхронность
- Технологии, которые модуль использует (Telegram API, subprocess, fcntl и т.д.)

### 1.4 Спецификации модулей-зависимостей

Для каждого модуля, от которого зависит текущий (из графа зависимостей), прочитай его спецификацию `dev/docs/specs/{dependency_name}_spec.md`. Из каждой спецификации извлеки:
- Публичный API — какие функции/классы экспортирует
- Типы аргументов и возвратов — чтобы вызовы из текущего модуля точно совпадали

Если спецификация зависимости ещё не создана — запиши этот факт. В разделе «Зависимости» укажи, что интерфейс зависимости предварительный и должен быть согласован после создания её спецификации.

### 1.5 Спецификации модулей-потребителей

Для каждого модуля, который зависит от текущего (вызывает его функции), проверь — есть ли уже его спецификация `dev/docs/specs/{consumer_name}_spec.md`. Если есть — прочитай, какие функции текущего модуля он ожидает. Публичный API текущего модуля должен предоставить всё, что от него ожидают потребители.

## Шаг 2: Написание спецификации

THINK HARD.

Напиши спецификацию по следующей структуре. Каждый раздел обязателен. Конкретика вместо абстракций — реальные имена функций, реальные типы, реальные значения.

### Структура спецификации

```markdown
# Спецификация модуля: {module_name}

Дата: {ДД-ММ-YYYY}
Слой: {слой из графа зависимостей}
Файл: `src/claude_manager/{module_name}.py`

## Назначение

{1-2 предложения: что делает модуль и зачем он нужен в системе}

## Обслуживаемые сценарии

{Список CJM из BRD, которые этот модуль обслуживает, с кратким описанием что именно модуль делает в каждом сценарии}

- **CJM-XX: {название}** — {что модуль делает в этом сценарии}

## Публичный API

{Для каждой публичной функции/класса:}

### `function_name(arg1: type1, arg2: type2) -> return_type`

{Одно предложение — что делает функция}

**Аргументы:**
- `arg1` (type1) — {описание}
- `arg2` (type2) — {описание}

**Возвращает:** {описание возвращаемого значения}

**Исключения:**
- `SomeError` — {когда выбрасывается}

## Внутренние функции

{Приватные функции, которые нужны для реализации публичного API. Та же структура, что и публичный API, но с префиксом `_` в имени}

## Алгоритм работы

{Для каждой публичной функции — пошаговое описание того, что происходит внутри}

### function_name

1. {Шаг 1 — что делается и зачем}
2. {Шаг 2 — ...}
3. ...

## Зависимости

{Какие модули импортирует и какие конкретно функции/классы из них использует}

- **{module_name}** — `function1()`, `function2()` — {зачем}

## Обработка ошибок

{Что может пойти не так и как модуль должен реагировать}

- **{Ситуация}** — {что делает модуль: логирует, выбрасывает исключение, возвращает значение по умолчанию}

## Константы

{Именованные константы, которые использует модуль}

- `CONSTANT_NAME = value` — {зачем эта константа, почему именно такое значение}

## Тест-план

{Конкретные тест-кейсы. Каждый кейс — это реальный тест, который можно написать без дополнительных уточнений}

### Юнит-тесты

- **test_name** — {что проверяет}
  - Вход: {конкретные входные данные}
  - Ожидаемый результат: {конкретный результат}
  - Тип: unit

### Граничные случаи

- **test_name** — {что проверяет}
  - Вход: {конкретные входные данные}
  - Ожидаемый результат: {конкретный результат}
  - Тип: edge case

### Тесты ошибок

- **test_name** — {что проверяет}
  - Вход: {конкретные входные данные, провоцирующие ошибку}
  - Ожидаемый результат: {какая ошибка, какое сообщение}
  - Тип: error
```

### Правила написания

- **Типы** — используй встроенные типы Python 3.13: `str | None`, `list[str]`, `dict[str, int]`. Не используй `typing.Optional`, `typing.List`
- **Async** — если модуль работает с I/O (файлы, сеть, процессы), функции должны быть `async`. Указывай это в сигнатурах
- **Конкретные значения** — в тест-плане используй реальные данные, а не плейсхолдеры. Вместо `"some text"` пиши `"Посмотри файл main.py"`
- **Полнота API** — публичный API должен покрывать все потребности модулей-потребителей (тех, кто вызывает этот модуль). Сверься со спецификациями потребителей
- **Согласованность вызовов** — если модуль A вызывает `module_b.do_thing(x, y)`, то в спецификации module_b функция `do_thing` должна принимать именно `x` и `y` с теми же типами. Перекрёстная проверка критически важна

## Шаг 3: Проверка покрытия BRD

THINK HARD.

Для каждого CJM, который обслуживает модуль, пройдись по секциям:
- «Что происходит внутри» — каждый шаг, относящийся к модулю, должен быть покрыт функцией в публичном API
- «Что может пойти не так» — каждый сценарий ошибки должен быть в разделе «Обработка ошибок» и иметь тест-кейс типа `error`
- «Что видит пользователь» и «Механизмы, которые работают во всех сценариях» — если модуль формирует или отправляет сообщения пользователю, проверь в BRD все визуальные индикаторы (иконки, форматирование, статусные префиксы) для этих сообщений. Каждый такой индикатор должен быть явно указан в алгоритме соответствующей функции и вынесен в именованную константу — BRD описывает конкретное визуальное поведение, которое легко потерять, если не зафиксировать его в спецификации

Если нашёл непокрытое требование — добавь в спецификацию.

## Шаг 4: Итеративная самопроверка

Спецификация — это фундамент реализации. «Написал и готово» недопустимо.

### Итерация 1: критический взгляд

THINK HARD.

Перечитай написанную спецификацию и проверь:

1. **Полнота** — все CJM покрыты? Все ошибки описаны? Все функции имеют алгоритм?
2. **Точность типов** — типы аргументов и возвратов совпадают с тем, как функции вызываются из других модулей?
3. **Согласованность** — нет ли противоречий между разделами (например, функция описана в API, но отсутствует в алгоритме)?
4. **Тест-покрытие** — каждая публичная функция имеет хотя бы один юнит-тест? Каждый сценарий ошибки имеет тест?
5. **Конкретика** — нет ли абстрактных описаний, которые не помогут при реализации?

Исправь найденные проблемы.

### Итерация 2: сверка с реальным кодом и документами

THINK HARD.

Перечитай спецификацию ещё раз. На этот раз сверяй каждый факт с источниками:
- Типы — совпадают с тем, что написано в спецификациях зависимостей?
- Константы — значения совпадают с BRD (например, лимит 4096 символов, интервал 30 секунд)?
- Алгоритм — последовательность шагов совпадает с описанием в BRD?

Если нашёл расхождения — исправь.

Минимум две итерации проверки обязательны. Если вторая итерация нашла проблемы — сделай третью.
