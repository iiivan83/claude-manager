# Pipeline Explorer

Оркестратор-скилл, который итеративно исследует задачу обработки данных через цикл «попробовал — проверил — исправил». Результат работы — два продукта:

- **Основной:** ТЗ в `dev/docs/specs/{DD.MM_HH.MM}-{task-name}-tz.md` — подробное ТЗ для передачи в pipeline-designer
- **Побочный:** финальный скилл-оркестратор в папке `{skill-name}-final/` — собранный рабочий пайплайн как переиспользуемый скилл

## Global Reference Rules

Перед началом работы прочитай:
- `~/.claude/references/writing-style-guide.md` — единый стиль письма для всех документов и отчётов

## Глубина мышления

Думай как опытный тех-лид, который лично отвечает за стабильность продакшена. Каждое решение
проходит через внутренний чек-лист:

- **Побочные эффекты** — что может сломаться в соседних модулях из-за этого изменения?
- **Граничные случаи** — как это поведёт себя на пустых данных, при конкурентном доступе,
  при неожиданном порядке вызовов?
- **Откат** — если изменение окажется неудачным, как быстро можно вернуть всё назад?
- **Альтернативы** — есть ли более простой способ добиться того же результата?
- **Достаточность** — это изменение действительно решает корневую причину, а не маскирует
  симптом?

Все агенты этого скилла работают с такой же глубиной — разбирают задачу по частям,
рассматривают альтернативы и перепроверяют выводы перед тем, как действовать.

## Основные принципы

- **Всё через агентов** — каждый этап выполняется как отдельный агент (Agent tool), никогда в основном контексте
- **Промежуточные данные — JSON**, финальное ТЗ — MD. Файлы агентов (`agents/*.md`) — тоже MD
- **Перед запуском любого агента — прочитать его инструкцию** из `agents/*.md` через Read tool и передать содержимое как промпт агенту
- **Логировать каждую операцию** — обновлять `orchestrator-log.json` ДО начала каждого шага (не после). Это позволяет видеть, на каком шаге пайплайн прервался. Также логировать решения и переходы между фазами
- **Логировать каждую неудачу** — обновлять `failure-log/failure-log.json` после каждого запуска diagnostician
- **Перед любым ретраем — обязательна диагностика** — слепые повторы запрещены
- **Лог неудач передаётся** при каждом ретрае и при каждом перезапуске стратегии
- **Язык общения с пользователем — русский**; имена файлов, переменных и код — английский
- **Black-box тесты НИКОГДА не передаются агентам разработки** — путь `tests/blackbox/` не упоминается ни в одном промпте агентов фазы 2 и 3

## Управление бюджетом

Пайплайн работает в рамках жёстких бюджетов:

- **max_total_iterations = 5** — максимальное количество полных проходов фазы 3 → фаза 4 → возврат
- **max_strategy_restarts = 3** — максимальное количество перезапусков стратегии
- **max_attempts_per_stage = 3** — максимальное количество попыток реализации одного этапа
- При исчерпании любого бюджета — СТОП с подробным отчётом пользователю

## Сообщения пользователю о прогрессе

**Формат:** `[pipeline-explorer] Фаза {N}: {описание} — {статус}`

Каждый шаг в `orchestrator-log.json` содержит поле `user_message` с текстом показанного сообщения.

# Папка архива (создаётся в фазе 7.1):
dev/docs/logs/skills-modifications/{ДД.ММ_ЧЧ.МИН}-pipeline-explorer-{task-name}/

    orchestrator-log.json

### Путь к ТЗ

ТЗ всегда сохраняется в: `dev/docs/specs/{DD.MM_HH.MM}-{task-name}-tz.md`

Где `{DD.MM_HH.MM}` — временна́я метка начала пайплайна (день.месяц_час.минута, например `03.04_14.30`). Сгенерировать один раз в фазе 0.1 и переиспользовать во всех этапах.

## Формат логирования

### orchestrator-log.json

```json
{
  "pipeline": "pipeline-explorer",
  "task_name": "<название задачи>",
  "created_at": "<ISO-8601>",
  "budget": {
    "max_total_iterations": 5,
    "max_strategy_restarts": 3,
    "used_iterations": 0,
    "used_strategy_restarts": 0
  },
  "current_phase": "<имя текущей фазы>",
  "current_strategy": 1,
  "steps": [],
  "decisions": []
}
```

### failure-log/failure-log.json

```json
{
  "failures": [
    {
      "strategy": 1, "stage": 2, "attempt": 1,
      "timestamp": "<ISO-8601>",
      "approach": "<что пробовали>",
      "error": "<что пошло не так>",
      "diagnosis": "<почему не вышло>",
      "lesson": "<что из этого следует>"
    }
  ]
}
```

## Фазы пайплайна

### Фаза 0 — Инфраструктура и сбор требований

#### 0.1 Создание папок (оркестратор, Bash)
Создать все поддиректории, инициализировать orchestrator-log.json и failure-log.json. Сгенерировать временну́ю метку `{DD.MM_HH.MM}` (текущие день.месяц_час.минута, например `03.04_14.30`) и обеспечить существование `dev/docs/specs/` (`mkdir -p`).

#### 0.2 Сбор требований (agent: requirements-collector)
Прочитать `agents/requirements-collector.md`, запустить агента. Единственный агент, который общается с пользователем. Собирает описание задачи, критерии приёмки, золотые примеры. Выход: `agent-outputs/00-requirements.json`.

### Фаза 1 — Генерация тестов (параллельно)

#### 1.1 Black-box тесты (agent: test-generator-blackbox)
Прочитать `agents/test-generator-blackbox.md`. Передать ТОЛЬКО requirements. НЕ передавать стратегию. Выход: `tests/blackbox/`, `agent-outputs/01-test-generator-blackbox.json`.

#### 1.2 Open тесты (agent: test-generator-open)
Прочитать `agents/test-generator-open.md`. Передать requirements. Выход: `tests/open/`, `agent-outputs/02-test-generator-open.json`.

**АБСОЛЮТНОЕ ПРАВИЛО:** путь `tests/blackbox/` НИКОГДА не передаётся агентам разработки.

### Фаза 2 — Стратегия и тесты этапов

#### 2.1 Планирование стратегии (agent: strategy-planner)
Прочитать `agents/strategy-planner.md`. Передать requirements, open-тесты, golden-examples, failure-log. Выход: `agent-outputs/03-strategy-planner.json`, `tests/contracts/contracts.json`.

#### 2.2 Тесты этапов (agent: stage-test-creator × N, параллельно)
Прочитать `agents/stage-test-creator.md`. Запустить для КАЖДОГО этапа параллельно. Выход: `tests/stage-tests/stage-{N}/`.

### Фаза 3 — Итеративная разработка

#### 3.N.1 Реализация (agent: stage-implementer)
Прочитать `agents/stage-implementer.md`. Передать описание этапа, тесты, контракты, результаты предыдущих этапов, failure-log (при ретрае). **НИКОГДА** не передавать blackbox тесты.

#### 3.N.2 Запуск и проверка (оркестратор, Bash)
Изолированная временная директория. Contract pre-check после backtrack. Запуск реализации, проверка контракта, запуск тестов этапа.

#### 3.N.3 Диагностика (agent: diagnostician)
Прочитать `agents/diagnostician.md`. Передать ошибку, тесты, код, описание этапа, failure-log. Дерево решений:
- `current_stage` + attempt < 3 → RETRY
- `current_stage` + attempt >= 3 → STRATEGY RESTART
- `previous_stage` (N-1) → BACKTRACK (1 уровень, цепные запрещены)
- глубже N-1 → STRATEGY RESTART

### Фаза 4 — Финальная проверка

Запуск ВСЕХ тестов из `tests/blackbox/` в изолированной директории. Все прошли → фазы 5+6. Часть не прошла → diagnostician для атрибуции → возврат в фазу 3. Бюджет 5 итераций.

### Фаза 5 — Генерация ТЗ (параллельно с фазой 6)

Agent: tz-generator. Прочитать `agents/tz-generator.md`. Передать все данные. Выход: `dev/docs/specs/{DD.MM_HH.MM}-{task-name}-tz.md` с 10 секциями.

### Фаза 6 — Сборка оркестратора (параллельно с фазой 5)

Agent: orchestrator-assembler. Прочитать `agents/orchestrator-assembler.md`. Выход: `{skill-name}-final/`.

### Фаза 7 — Архивация и очистка

#### 7.1 Архивация результатов (оркестратор, Bash)
Сохранить полный снимок результатов работы в папку логов. Запустить скрипт `scripts/archive-pipeline-run.sh`:

```bash
bash scripts/archive-pipeline-run.sh "<task-name>" "<project-root>" "<workspace-path>" "<pipeline-path>"
```

Скрипт создаёт папку в `dev/docs/logs/skills-modifications/` с именем в формате:
`{ДД.ММ_ЧЧ.МИН}-pipeline-explorer-{task-name}`

Где `ДД.ММ_ЧЧ.МИН` — день.месяц_час.минута начала архивации (например, `03.04_14.30`).

Внутрь копируются три вещи:
- `pipeline/` — рабочий пайплайн (готовый код)
- `logs/` — все логи для быстрого доступа (orchestrator-log.json, failure-log/, agent-outputs/)
- `workspace/` — полная копия рабочей папки pipeline-workspace со всем содержимым

**ВАЖНО:** архивация выполняется ДО очистки, чтобы ничего не потерять.

#### 7.2 Очистка промежуточных данных
Удалить промежуточные папки нефинальных стратегий. Автоматически, без подтверждения.

## Правила изоляции тестов

1. `TEMP_DIR=$(mktemp -d)`
2. Скопировать входные файлы и реализацию
3. Запустить внутри TEMP_DIR
4. Забрать результаты
5. `rm -rf "$TEMP_DIR"`

## Сессионный отчёт

Запусти агента для создания сессионного отчёта (обязательно для пайплайнов, правило из AGENTS.md).

Промпт агенту:
```
Используй /session-report.
Контекст: пайплайн pipeline-explorer завершён.
Задача: {описание задачи}
Папка логов: {папка логов}
```

## Правила переходов между фазами

```
Фаза 0 → Фаза 1        : всегда
Фаза 1 → Фаза 2        : всегда
Фаза 2 → Фаза 3        : всегда
Фаза 3 → Фаза 4        : все этапы пройдены
Фаза 4 → Фазы 5+6      : ВСЕ blackbox тесты пройдены
Фаза 4 → Фаза 3        : часть тестов не пройдена, бюджет есть
Фаза 4 → СТОП           : бюджет исчерпан (5/5)
Фаза 3 → Фаза 2        : strategy restart
Фаза 2 → СТОП           : бюджет перезапусков исчерпан (3/3)
Фазы 5+6 → Фаза 7      : обе завершены
Фаза 7 → Завершение     : всегда
```
