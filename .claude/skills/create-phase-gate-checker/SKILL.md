# Генерация скрипта check-phase-gate.py

**THINK HARD** на каждом шаге выполнения этого скилла. Используй максимальную глубину мышления при принятии каждого решения.

Ты создаёшь Python-скрипт `dev/scripts/check-phase-gate.py` — автоматическую проверку ворот качества пайплайна. Скрипт нужен оркестратору (`pipeline-run`) для принятия решения: можно ли перейти к следующей фазе, или текущая фаза ещё не завершена. Без этого скрипта оркестратор был бы вынужден вручную проверять статусы задач и запускать тесты, рискуя пропустить невыполненные условия.

## Контекст: что такое ворота качества

Пайплайн состоит из 10 фаз (0-9). Между фазами стоят «ворота качества» — набор условий, которые должны быть выполнены, прежде чем можно двигаться дальше. Ворота защищают от ситуации, когда непроверенный код попадает в следующую фазу и порождает каскад проблем.

## Шаг 1: Изучи спецификацию

**THINK HARD** — скрипт должен точно соответствовать спецификации пайплайна.

Прочитай `dev/docs/specs/pipeline-spec.md` и извлеки:

1. **Раздел D1** — формат `pipeline-state.json`. Тебе нужна структура файла: где хранятся фазы, задачи, статусы.
2. **Часть 2, Чеклист по фазам** — ворота качества для каждой фазы. Для каждой фазы описано, что должно быть выполнено.

Запомни эту информацию — она потребуется для генерации скрипта.

## Шаг 2: Проверь текущее состояние

**THINK HARD** — пойми контекст перед генерацией.

1. Проверь, существует ли уже файл `dev/scripts/check-phase-gate.py`. Если да — предупреди пользователя, что файл будет перезаписан, и спроси подтверждение.
2. Убедись, что папка `dev/scripts/` существует. Если нет — создай её.
3. Проверь, существует ли `dev/scripts/run-all-tests.sh` — скрипт, который будет вызываться для проверки тестов. Если нет — сообщи пользователю, что для полной работы check-phase-gate.py нужен run-all-tests.sh (его создаёт скилл `create-test-runner`).

## Шаг 3: Сгенерируй скрипт

**THINK HARD** — скрипт должен быть надёжным, покрывать все 10 фаз, и давать понятные сообщения об ошибках.

Создай файл `dev/scripts/check-phase-gate.py` со следующей логикой:

### Структура скрипта

```python
#!/usr/bin/env python3
"""
check-phase-gate.py — проверка ворот качества пайплайна.

Читает pipeline-state.json, проверяет что все условия текущей фазы
выполнены, и выдаёт PASS (можно переходить) или FAIL (что не выполнено).

Использование:
    python dev/scripts/check-phase-gate.py          # проверить текущую фазу
    python dev/scripts/check-phase-gate.py --phase 3  # проверить конкретную фазу

Exit codes: 0 = PASS, 1 = FAIL
"""
```

### Ключевые компоненты скрипта

**1. Разбор аргументов командной строки:**
- Без аргументов: скрипт определяет текущую фазу из `pipeline-state.json` (поле `current_phase`)
- С аргументом `--phase N`: проверяет указанную фазу N (число от 0 до 9)

**2. Чтение pipeline-state.json:**
- Путь: `pipeline-state.json` в корне проекта (скрипт сам определяет корень проекта — на 2 уровня выше от своего расположения)
- Если файл не найден: FAIL с сообщением «pipeline-state.json не найден»
- Если файл невалидный JSON: FAIL с сообщением о синтаксической ошибке

**3. Проверка задач фазы:**
- Для указанной фазы получить список задач из `phases.{N}.tasks`
- Проверить, что каждая задача имеет статус `completed`
- Если у фазы нет массива `tasks` (фазы 0, 1 имеют только `status`) — проверить `phases.{N}.status == "completed"`
- Собрать список незавершённых задач для отчёта

**4. Запуск тестов:**
- Выполнить `dev/scripts/run-all-tests.sh`
- Проверить exit code: 0 = тесты пройдены, не-0 = есть провалы
- Если скрипт run-all-tests.sh не найден: вывести предупреждение, но не считать это провалом ворот (тесты могут быть ещё не настроены в ранних фазах)

**5. Проверка ворот качества конкретной фазы:**

Для каждой фазы — свой набор проверок. Скрипт должен содержать словарь (dictionary), который описывает дополнительные проверки для каждой фазы (помимо базовой проверки статусов задач). Эти проверки основаны на чеклисте из pipeline-spec.md, Часть 2:

- **Фаза 0 (Валидация BRD):** проверить существование файла `dev/docs/brd/brd-validation-report_*.md` (через glob-паттерн)
- **Фаза 1 (Фундамент):** проверить существование `pyproject.toml`, `.env.example`, `tests/conftest.py`, папки `dev/docs/logs/testing/`, файла `pipeline-state.json`
- **Фаза 2 (Спецификации):** проверить существование `dev/docs/specs/module-dependency-graph.md`; для каждого модуля из `pipeline-state.json["modules"]` проверить существование `dev/docs/specs/{module}_spec.md`
- **Фаза 3 (Реализация):** для каждого модуля проверить `src/claude_manager/{module}.py` и `tests/test_{module}.py`; все юнит-тесты зелёные (через run-all-tests.sh)
- **Фаза 4 (Интеграционные тесты):** проверить что `tests/integration/` содержит файлы тестов; все тесты (юнит + интеграционные) зелёные
- **Фаза 5 (E2E тестирование):** проверить существование `dev/docs/logs/testing/e2e-test-plan_*.md` и `dev/docs/logs/testing/e2e-test-results_*.md`; все автотесты зелёные
- **Фаза 6 (Ревью):** проверить существование `dev/docs/review-report_*.md`; все тесты зелёные
- **Фаза 7 (Документация):** проверить что `CLAUDE.md` и `dev/docs/docs-index.md` существуют
- **Фаза 8 (Сценарии пользовательского тестирования):** проверить существование `dev/docs/logs/testing/user-test-scenarios_*.md`
- **Фаза 9 (Прогон пользовательского тестирования):** проверить существование `dev/docs/logs/testing/user-test-results_*.md`; все автотесты зелёные

**6. Вывод результатов:**

Скрипт должен выводить понятные сообщения для каждой проверки:
```
=== Проверка ворот качества: Фаза 3 (Реализация) ===

[ПРОВЕРКА] Статусы задач фазы...
  [OK] T2.1 (implement config) — completed
  [OK] T2.2 (implement message_splitter) — completed
  [FAIL] T2.3 (implement session_reader) — in_progress

[ПРОВЕРКА] Файлы модулей...
  [OK] src/claude_manager/config.py существует
  [OK] src/claude_manager/message_splitter.py существует
  [FAIL] src/claude_manager/session_reader.py не найден

[ПРОВЕРКА] Файлы тестов...
  [OK] tests/test_config.py существует
  [FAIL] tests/test_message_splitter.py не найден

[ПРОВЕРКА] Запуск тестов...
  [OK] Все тесты пройдены

=== РЕЗУЛЬТАТ: FAIL ===
Не пройдено: 3 из 10 проверок
- T2.3 (implement session_reader) — статус in_progress, ожидается completed
- src/claude_manager/session_reader.py не найден
- tests/test_message_splitter.py не найден
```

### Требования к коду

- Используй `argparse` для разбора аргументов
- Используй `pathlib.Path` для работы с путями
- Используй `subprocess.run()` для запуска run-all-tests.sh
- Используй `glob` (модуль Python) для поиска файлов по паттерну (например `brd-validation-report_*.md`)
- Все строки с путями — через константы в начале файла
- Все сообщения — на русском языке
- Type hints для всех функций
- Exit code 0 = PASS, 1 = FAIL

### Структура функций скрипта

Скрипт должен быть разбит на маленькие функции (каждая делает одно):

- `parse_arguments()` — разбор аргументов CLI
- `find_project_root()` — определение корня проекта
- `load_pipeline_state(path)` — загрузка и валидация pipeline-state.json
- `get_current_phase(state)` — определение текущей фазы из состояния
- `check_task_statuses(phase_data)` — проверка статусов задач фазы
- `check_files_exist(file_patterns, project_root)` — проверка существования файлов
- `run_tests(project_root)` — запуск run-all-tests.sh
- `get_phase_checks(phase_number, state, project_root)` — получение списка проверок для конкретной фазы
- `print_results(checks, phase_number, phase_name)` — вывод форматированного отчёта
- `main()` — точка входа, координирует все шаги

```bash

```

Если проверка выявит ошибки — исправь их и повтори проверку. Не завершай работу, пока скрипт не пройдёт `py_compile` без ошибок.
