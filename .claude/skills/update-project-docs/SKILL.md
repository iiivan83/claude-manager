# Массовое обновление документации проекта по реальному коду

THINK HARD. На каждом шаге этого скилла используй максимальную глубину мышления. Это обязательное требование для каждого шага без исключений. Документация -- фундамент, по которому работают все будущие сессии Claude. Ошибка в CLAUDE.md или docs-index.md каскадно влияет на качество работы в каждой сессии, поэтому точность здесь критически важна.

## Чем этот скилл отличается от update-docs

Два скилла решают разные задачи, и их важно не путать:

- **update-docs** -- пользователь вставил текст или беседу, скилл интегрирует эту информацию в один или несколько существующих документов. Источник правды -- текст от пользователя.
- **update-project-docs** (этот скилл) -- источник правды -- реальный код в `src/`. Скилл сканирует весь код, сравнивает его с документацией и обновляет документацию, чтобы она отражала реальное состояние проекта.

## Аргументы

$ARGUMENTS

Если аргументов нет -- выполнять полный цикл обновления (все шаги).

## Шаг 1: Сбор данных из кода (параллельно)

THINK HARD. Это фундамент всей работы -- от полноты сбора зависит качество обновления.

Используй Agent tool для параллельного сбора информации из трёх независимых источников.

### Агент 1 -- Код проекта

Прочитать все Python-файлы в `src/claude_manager/`:
- Список всех модулей (файлов .py) с однострочным описанием каждого (из docstring или из содержания)
- Публичные функции и классы каждого модуля (имя + краткое назначение)
- Импорты между модулями -- какой модуль от какого зависит
- Используемые внешние библиотеки (из import)

### Агент 2 -- Конфигурация и инфраструктура

Прочитать:
- `pyproject.toml` -- зависимости, точка входа, версия Python
- `requirements.txt` -- список пакетов
- `.env.example` -- переменные окружения
- `watch_and_restart.sh` -- что делает
- `tests/` -- какие тесты существуют, структура
- `__main__.py` -- как запускается пакет

### Агент 3 -- Текущая документация

Прочитать:
- `CLAUDE.md` -- целиком
- `dev/docs/docs-index.md` -- целиком
- `dev/docs/brd/brd-user-journeys.md` -- обзор (первые 50 строк для понимания CJM-списка)
- Все остальные документы из `dev/docs/` (кроме session-reports) -- заголовки и структура

После завершения всех агентов -- объединить результаты и перейти к анализу.

## Шаг 2: Обновление CLAUDE.md

THINK HARD. CLAUDE.md -- самый важный файл проекта для Claude. Каждая ошибка здесь будет повторяться в каждой сессии.

### 2.1 Сравнение с реальным кодом

Взять результаты Агента 1 и Агента 2, сверить с каждым разделом CLAUDE.md:

**Раздел «Структура проекта»:**
- Все ли файлы из `src/claude_manager/` перечислены? Нет ли лишних (удалённых)?
- Комментарии к файлам соответствуют реальному содержимому?
- Структура `dev/`, `tests/` актуальна?

**Раздел «Технологии»:**
- Версии библиотек совпадают с `requirements.txt` и `pyproject.toml`?
- Нет ли новых зависимостей, которые не упомянуты?
- Нет ли упомянутых зависимостей, которых больше нет?

**Раздел «Архитектурные принципы»:**
- Слоёная архитектура -- модули правильно распределены по слоям?
- Реальные зависимости между модулями не нарушают описанные правила?

**Раздел «Команды разработки»:**
- Команды реально работают с текущей структурой?
- Нет ли новых важных команд?

**Раздел «Важные детали для разработки»:**
- Все ли детали актуальны?
- Нет ли новых важных деталей из кода?

**Раздел «Техническая документация»:**
- Все ли перечисленные документы существуют?
- Нет ли важных документов, которые не упомянуты?

### 2.2 Применение изменений

Для каждого найденного расхождения -- исправить в CLAUDE.md. Правила:
- Обновлять только то, что реально устарело. Не переписывать ради переписывания.
- Сохранять существующий стиль и формат документа.
- Если раздел полностью устарел (например, описывает модули, которые ещё не написаны, но уже запланированы) -- оставить его, но убедиться что он помечен как план, а не как факт.
- Если появились новые модули -- добавить их в структуру с однострочным комментарием.
- Если модули удалены -- убрать из структуры.

### 2.3 Итеративная самопроверка CLAUDE.md

После внесения изменений -- перечитать обновлённый CLAUDE.md целиком и сверить с результатами Агентов ещё раз. Проверить:
1. Каждый файл из `src/` упомянут в структуре
2. Ни один упомянутый файл не отсутствует на диске
3. Версии технологий точны
4. Описания модулей соответствуют реальному коду
5. Команды разработки работоспособны

Минимум две итерации проверки -- даже если первая не нашла проблем.

## Шаг 3: Создание deployment-guide.md

THINK HARD. Этот гайд -- единственный документ, по которому можно поднять бота с нуля. Пропущенный шаг = неработающий бот.

Создать файл `dev/docs/deployment-guide.md` -- пошаговый гайд для запуска бота с чистой системы.

### Структура гайда

```markdown
# Гайд по запуску Claude Manager

Пошаговая инструкция: от чистой системы до работающего бота.

## Что нужно заранее

(Предусловия: Python, Claude Code CLI, macOS, Telegram-бот)

## Шаг 1: Установка

(Клонирование, виртуальное окружение, зависимости)

## Шаг 2: Настройка .env

(Какие переменные, где взять значения, формат)

## Шаг 3: Запуск

(Команды запуска, что должно появиться в консоли)

## Шаг 4: Проверка работы

(Как убедиться что бот работает: отправить /start в Telegram, ожидаемый ответ)

## Шаг 5: Запуск с автоперезапуском

(watch_and_restart.sh -- для разработки)

## Решение проблем

(Типичные ошибки и их решения)
```

### Правила написания гайда

- Каждый шаг самодостаточен -- читатель может выполнить его, не перескакивая в другие документы.
- Конкретные команды, а не абстрактные описания. «Выполните `pip install -r requirements.txt`», а не «установите зависимости».
- Ожидаемый результат после каждого шага -- как понять, что шаг выполнен успешно.
- Информацию о переменных окружения брать из реального `.env.example`, а не выдумывать.
- Информацию о командах запуска -- из реального `pyproject.toml` и `__main__.py`.

Если файл `deployment-guide.md` уже существует -- обновить его по реальному коду, а не создавать заново.

## Шаг 4: Обновление docs-index.md

THINK HARD. docs-index.md -- навигатор по документации. Пропущенный документ = потерянный документ.

### 4.1 Сканирование

Просканировать все файлы в `dev/docs/` (кроме подпапки `session-reports/`). Для каждого файла проверить:
- Есть ли запись в docs-index.md?
- Если есть -- описание актуально?
- Условия «Читать если» всё ещё верны?

### 4.2 Обновление

- Добавить записи для новых документов (включая только что созданный deployment-guide.md).
- Удалить записи для документов, которые больше не существуют.
- Обновить описания, если содержимое документа существенно изменилось.
- Сохранить существующий формат (заголовок H2, блок «Читать если» со списком условий).

### 4.3 Проверка CLAUDE.md

Если раздел «Техническая документация» в CLAUDE.md ссылается на документы, которые были добавлены или удалены из docs-index.md -- обновить и этот раздел тоже.

## Шаг 5: Обновление pipeline-state.json

Если файл `pipeline-state.json` существует в корне проекта -- обновить фазу 7:

```json
{
  "phases": {
    "7": {
      "name": "Обновление документации",
      "status": "completed",
      "completed_at": "ISO-дата",
      "updated_files": ["CLAUDE.md", "dev/docs/deployment-guide.md", "dev/docs/docs-index.md"]
    }
  }
}
```

Не перезаписывать другие фазы -- только обновить/добавить фазу 7.

Если файла не существует -- пропустить этот шаг (pipeline-state.json создаётся на более ранних фазах пайплайна).

## Шаг 6: Итеративная верификация

THINK HARD. Последняя линия защиты от ошибок.

Перечитать все изменённые файлы и проверить:

1. **CLAUDE.md** -- структура проекта в документе полностью совпадает с реальной файловой системой
2. **deployment-guide.md** -- каждый шаг выполним, команды корректны, переменные окружения совпадают с `.env.example`
3. **docs-index.md** -- каждый упомянутый документ существует на диске, каждый документ на диске упомянут в индексе
4. **Перекрёстные ссылки** -- если один документ ссылается на другой, ссылка ведёт на существующий файл

Минимум две итерации проверки. Если вторая итерация нашла проблемы -- исправить и запустить третью.
