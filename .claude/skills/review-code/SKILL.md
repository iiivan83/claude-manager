# 4-проходное ревью кода с автоисправлением

**THINK HARD.** На каждом шаге этого скилла используй максимальную глубину мышления. Это обязательное требование для каждого шага без исключений. Перед каждым решением — анализ, перед каждым исправлением — обоснование, перед каждой оценкой — перепроверка.

Ты проводишь глубокое ревью всего кода проекта Claude Manager (Telegram-бот, управляющий Claude Code с телефона). Это не поверхностный просмотр — ты методично проходишь по каждому файлу, проверяя его по чеклисту, исправляешь найденные проблемы и убеждаешься, что исправления не сломали существующий код.

Ревью состоит из 4 проходов, каждый с отдельным фокусом. После каждого прохода ты исправляешь найденные проблемы и прогоняешь все тесты. После 4 проходов — повторная полная проверка всего кода, чтобы убедиться, что исправления не создали новых проблем.

## Шаг 0: Сбор контекста

Прочитай всё, что нужно для ревью. Используй параллельные агенты для ускорения — каждый агент читает свою группу файлов.

**Агент 1 — Код проекта:**
- Прочитай все `.py` файлы из `src/claude_manager/`
- Для каждого файла запомни: публичные функции, импорты, зависимости

**Агент 2 — Тесты:**
- Прочитай все файлы из `tests/`
- Запомни: какие модули покрыты тестами, какие фикстуры используются

**Агент 3 — Документация:**
- Прочитай `development/docs/brd-user-journeys.md` (BRD со всеми сценариями)
- Прочитай `development/docs/review-checklists.md` (чеклисты для 4 проходов)
- Прочитай `CLAUDE.md` (архитектура и правила проекта)

После сбора контекста у тебя должна быть полная картина: весь код, все тесты, все требования, все чеклисты.

## Шаг 1: Проход 1 — Качество кода

Цель этого прохода — убедиться, что код читается легко, написан единообразно, и его может понять разработчик, впервые открывший проект.

Открой чеклист из `development/docs/review-checklists.md`, раздел «Проход 1: Качество кода». Пройди по каждому файлу из `src/claude_manager/` и проверь каждый пункт чеклиста.

**На что обращать внимание:**

- **Именование:** переменная `d` ничего не говорит — `delay_seconds` говорит всё. Проверь каждое имя: можно ли понять его назначение без чтения контекста? Исключение: `i` в простых циклах `for i in range(...)` допустимо
- **Размер функций:** если функция больше 20 строк — разбей на более мелкие. Каждая функция должна делать одну вещь, описуемую одним предложением
- **Вложенность:** `if` внутри `for` внутри `if` — это 3 уровня, а допустимо 2. Вынеси внутреннюю логику в отдельную функцию
- **Magic numbers:** `if len(text) > 4096` — что такое 4096? Должна быть константа `TELEGRAM_MESSAGE_LIMIT = 4096` с комментарием
- **Дублирование:** одна и та же логика в двух местах — вынеси в общую функцию. Одна и та же строка в двух местах — сделай константой
- **Type hints:** каждая публичная функция должна иметь аннотации типов параметров и возвращаемого значения. Используй встроенные типы (`list`, `dict`, `str | None`), не `typing.Optional`
- **Запрет `any`:** в коде не должно быть типа `any` — все типы должны быть конкретными. Если встретился `any` — определи правильный тип и замени
- **Docstrings:** каждая публичная функция — одна строка `"""Что делает."""`

**Запись проблем:** для каждой найденной проблемы записывай:
- Файл и строку
- Что не так
- Как исправить

**Исправление:** после проверки всех файлов — исправь все найденные проблемы.

**Тесты:** после исправлений запусти:
```bash
python -m pytest tests/ -v
```

Если тесты упали — исправь и запусти снова. Максимум 5 циклов починки. Если после 5 циклов тесты не зелёные — запиши проблему и продолжай.

## Шаг 2: Проход 2 — Безопасность

Цель — убедиться, что бот защищён от типичных атак и утечек. Бот работает локально на компьютере пользователя, но всё равно получает данные из Telegram, которые могут содержать что угодно.

Открой чеклист «Проход 2: Безопасность». Проверь каждый файл.

**На что обращать внимание:**

- **Секреты в коде:** поищи строки, похожие на токены, ключи, пароли прямо в `.py` файлах. Все секреты должны загружаться из `.env` через `config.py`
- **Белый список:** найди все точки входа (обработчики команд в `bot.py`). Каждый обработчик должен проверять `user_id in ALLOWED_USER_IDS` до выполнения любой логики
- **HTML escaping:** любые данные от пользователя (текст сообщений, имена файлов), которые включаются в HTML-ответ, должны проходить через `html.escape()`. Без этого пользователь может сломать форматирование или (теоретически) выполнить инъекцию
- **Command injection:** если где-то пользовательский ввод подставляется в shell-команду (через `subprocess`, `os.system` и т.п.), это критическая уязвимость. Используй список аргументов `subprocess.run(["cmd", arg])`, а не строку `subprocess.run(f"cmd {arg}", shell=True)`
- **Path traversal:** если пользователь может указать путь к файлу — проверь, что путь не выходит за допустимую директорию (`..` в пути, символические ссылки)

**Исправления и тесты:** аналогично Проходу 1 — исправь, запусти `pytest tests/ -v`, цикл починки до зелёного.

## Шаг 3: Проход 3 — Архитектура

Цель — убедиться, что код организован по слоям с правильным направлением зависимостей, все операции ввода-вывода асинхронны, и нет потенциальных проблем с блокировками.

Открой чеклист «Проход 3: Архитектура». Проверь каждый файл.

**На что обращать внимание:**

- **Направление зависимостей:** проверь `import` в каждом файле. Правило: верхний слой вызывает нижний, но не наоборот
  - Транспортный слой (`bot.py`) может импортировать бизнес-логику и инфраструктуру
  - Бизнес-логика (`session_manager`, `daily_session_registry`) НЕ должна импортировать `bot`, `telegram`, `process_manager`, `claude_runner`
  - Инфраструктура (`process_manager`, `claude_runner`) НЕ должна импортировать бизнес-логику
  - Утилиты (`message_splitter`, `session_reader`) не импортируют ничего из проекта (только стандартную библиотеку)

- **Async/await:** все операции ввода-вывода (сеть, файлы, процессы) должны быть async. Файловые операции — через `asyncio.to_thread()`. Нигде не должно быть `time.sleep()` — только `asyncio.sleep()`

- **Атомарность файлов:** запись в JSON-файлы (`sessions.json`, `daily_sessions.json`, `pipeline-state.json`) должна быть атомарной: сначала пишем во временный файл (`.tmp`), затем переименовываем (`os.rename()`). Это защищает от повреждения файла при аварийном завершении

- **Deadlocks:** проверь места, где используются блокировки (`asyncio.Lock`, `fcntl.flock`). Убедись, что нет ситуации, когда два ресурса блокируются в разном порядке

**Исправления и тесты:** аналогично — исправь, запусти `pytest tests/ -v`, цикл починки до зелёного.

## Шаг 4: Проход 4 — Соответствие BRD

Цель — убедиться, что каждый пользовательский сценарий из BRD реализован в коде, каждый edge case обработан, и сквозные механизмы работают.

Открой чеклист «Проход 4: Соответствие BRD». Открой BRD (`development/docs/brd-user-journeys.md`).

**На что обращать внимание:**

- **Покрытие CJM:** для каждого CJM в BRD (CJM-01, CJM-02 и т.д.) найди соответствующий код. Каждый шаг сценария должен быть реализован. Если что-то пропущено — допиши
- **Edge cases:** в каждом CJM есть раздел «что может пойти не так». Для каждого пункта проверь, что в коде есть обработка этой ситуации (try/except, проверка условия, fallback-логика)
- **Карта состояний:** пользователь бота находится в одном из 3 состояний (нет сессии / подключён / мониторинг). Проверь, что переходы между состояниями работают как описано в BRD
- **Сквозные механизмы:**
  - Проверка доступа по белому списку — при каждом действии
  - Промежуточные обновления (прогресс ответа Claude) — не чаще раза в 30 секунд
  - Координация мониторинга и прямых ответов — нет дублирования сообщений
  - Одновременная работа через Telegram и терминал
  - Защита от двойного запуска (файл-замок)
  - Восстановление состояния при перезапуске бота
  - Дневная нумерация сессий (#1, #2, #3...)

**Исправления и тесты:** аналогично — исправь, запусти `pytest tests/ -v`, цикл починки до зелёного.

## Шаг 5: Повторная полная проверка

Это критически важный шаг. Исправления в предыдущих проходах могли создать новые проблемы — переименование переменной могло сломать другой файл, добавление проверки безопасности могло нарушить логику, рефакторинг мог испортить тесты.

**Алгоритм:**
1. Перечитай весь код `src/claude_manager/` заново — с нуля, как будто видишь его впервые
2. Быстро пройди по всем 4 чеклистам (все проходы), но теперь фокус на: не создали ли исправления новых проблем?
3. Если нашёл новые проблемы — исправь и запусти тесты
4. Если не нашёл — отлично, переходи к отчёту

## Шаг 7: Обновление pipeline-state.json

Прочитай `pipeline-state.json` из корня проекта. Обнови фазу 6 (Ревью):

- `status`: `"completed"` (если все проблемы решены и тесты зелёные) или `"failed"` (если остались нерешённые проблемы)
- `completed_at`: текущая дата и время в формате ISO 8601

Запиши файл атомарно (через временный файл + переименование).

## Важные правила

- **Всегда все тесты:** после каждого исправления запускай `pytest tests/ -v` — все тесты, не только связанные с изменённым файлом. Регрессия (когда починка одного ломает другое) — главный враг ревью
- **Максимум 5 циклов починки** на каждый проход. Если после 5 циклов тесты не зелёные — запиши проблему в отчёт и продолжай следующий проход
- **Не меняй тесты без причины.** Если тест падает после исправления — сначала разберись: тест правильный, а код сломан? Или тест тестировал неправильное поведение? Исправляй то, что действительно неправильно
- **Атомарность записи:** все JSON-файлы пишутся через `.tmp` + `os.rename()`
- **Логирование:** если добавляешь обработку ошибок — используй `logging`, не `print()`
