# Спецификация внутреннего скрипта для разработки

Ты создаёшь подробные технические задания на вспомогательные скрипты — инструменты, которые помогают в разработке и миграции проекта, но не являются частью продакшен-кода.

Такие скрипты решают конкретные задачи: проанализировать код, сравнить данные, сгенерировать файлы, проверить соответствие, собрать статистику. Хорошее ТЗ позволяет реализовать скрипт без дополнительных вопросов.

## Главное правило: задавай вопросы

Прежде чем писать спецификацию — убедись, что ты понимаешь задачу. Если что-то неясно, спроси у пользователя. Не додумывай.

Ситуации, когда нужно обязательно спросить:
- Неясен масштаб: скрипт для одного модуля или для всего проекта?
- Неясен формат вывода: в консоль, в файл, в какой формат?
- Неясны входные данные: какие файлы, откуда берутся, в каком формате?
- Есть несколько способов решить задачу — спроси, какой предпочтительнее
- Пользователь описал результат, но не описал процесс — уточни ожидания

Лучше задать 2-3 вопроса, чем написать ТЗ, которое потом придётся переделывать.

## Шаг 1: Пойми задачу

**THINK HARD** — это самый важный шаг.

### Что нужно выяснить

1. **Цель** — какую проблему решает скрипт? Зачем он нужен?
2. **Контекст** — какой этап работы, какой модуль затрагивает, чем вызвана потребность
3. **Входные данные** — что скрипт принимает (файлы, аргументы, пути, данные из БД)
4. **Выходные данные** — что скрипт выдаёт (файл, вывод в консоль, отчёт, изменённые файлы)
5. **Критерии успеха** — как понять, что скрипт отработал правильно

### Исследуй кодовую базу

Изучи файлы проекта, которые относятся к задаче скрипта. Это нужно для:
- Понимания структуры данных, с которыми скрипт будет работать
- Определения зависимостей (какие файлы/таблицы/модули задействованы)
- Обнаружения крайних случаев (нестандартные форматы, исключения, пустые данные)
- Определения реального масштаба задачи (сколько файлов/записей придётся обработать)

Прочитай индекс документации `development/docs/docs-index.md` — там может быть информация, полезная для контекста задачи.

## Шаг 2: Спланируй размещение результатов скрипта

Скрипт, как правило, создаёт выходные файлы — отчёты, JSON, markdown-документы, сгенерированный код. Прежде чем проектировать алгоритм, нужно определить, куда именно скрипт будет сохранять свои результаты. Если результаты окажутся не в том месте, другие скиллы и люди не смогут их найти — структура папок проекта стандартизирована и все инструменты опираются на неё.

### Как определить правильный путь

1. **Прочитай `development/docs/target-folder-structure.md`** — там зафиксирована целевая структура папок проекта с комментариями, какой тип файлов куда складывается.

2. **Определи тип результата скрипта** — какие файлы создаёт скрипт и к какой категории они относятся:
   - Отчёт сессии → `development/docs/session-reports/DD-MM/`
   - Отчёт анализа первопричин → `development/docs/root-cause-reports/`
   - Спецификация скрипта → `development/script-specs/`
   - Временные/промежуточные файлы LLM → `development/temp-docs/`
   - Общая документация проекта → `development/docs/`

3. **Если тип результата не вписывается ни в одну категорию** — спроси у пользователя, куда лучше разместить. Не придумывай новые папки без согласования.

### Подтверждение пути цитатой

Выбранный путь нужно подтвердить цитатой из `target-folder-structure.md`. Найди в этом файле строку дерева папок, которая соответствует выбранному пути, и процитируй её в спецификации. Это структурная проверка: без реального чтения файла цитату не получится написать, а если раздел пуст — сразу видно, что путь не был сверен с целевой структурой.

Если подходящей строки в `target-folder-structure.md` нет — значит, для этого типа результатов ещё не определено место. В таком случае спроси пользователя, куда разместить результаты, и предложи дополнить `target-folder-structure.md`.

### Что зафиксировать

В разделе «Выходные данные» спецификации укажи:
- Полный путь, куда скрипт сохраняет результат (с шаблонами `{module}`, `{date}` и т.д., если путь зависит от параметров)
- **Цитата из `target-folder-structure.md`** — строка дерева, подтверждающая выбранный путь
- Формат именования выходного файла
- Почему выбран именно этот путь (к какой категории целевой структуры относится результат)

## Шаг 3: Назови спецификацию

Имя файла — первое, что видит человек (или LLM) при выборе спеки. Хорошее имя сразу говорит, что делает скрипт, без необходимости открывать файл.

### Определи порядковый номер

Посмотри существующие файлы в `development/script-specs/` (включая подпапку `realized/`) и возьми следующий свободный номер. Номер нужен для хронологического порядка — чтобы было видно, в какой последовательности проектировались скрипты.

### Правила именования

- Формат: `{NNN}-{описательное-имя}.md` — трёхзначный номер, затем дефис, затем имя
- Строчные буквы, слова через дефис
- Имя должно отвечать на вопрос: **«Что конкретно делает скрипт?»** — описывай действие и объект, а не абстрактную роль
- Избегай общих слов вроде «manager», «handler», «processor», «orchestrator», «helper» — они не объясняют назначение. Вместо этого называй по действию: что скрипт делает и с чем

### Примеры

| Скрипт | Хорошее имя | Плохое имя (и почему) |
|---|---|---|
| Анализ PHP-модулей через LLM | `002-llm-analyze-module.md` | `002-llm-orchestrator.md` (не говорит что анализирует) |
| Сравнение ответов PHP и Django | `004-php-django-response-comparator.md` | `004-response-handler.md` (что за handler?) |
| Генерация Django-моделей из MySQL | `005-mysql-to-django-model-generator.md` | `005-model-tool.md` (какие модели? какой tool?) |
| Подсчёт зависимостей между модулями | `006-module-dependency-counter.md` | `006-dependency-processor.md` (processor ничего не объясняет) |

### Принципы написания

- **Конкретика вместо абстракций** — вместо «обрабатывает файлы» пиши «читает все `.php` файлы из `protected/controllers/`»
- **Реальные примеры** — в разделе «Пример вывода» показывай реальные данные из проекта, а не выдуманные
- **Алгоритм без воды** — каждый шаг должен быть действием, а не описанием намерения
- **Зависимости от реального кода** — указывай конкретные пути к файлам, которые ты проверил в кодовой базе

### Скрипты, которым нужен LLM

Иногда проектируемый скрипт предполагает обращение к языковой модели (LLM) — например, для анализа текста, классификации, генерации кода или извлечения структурированных данных из неструктурированных. В таких случаях LLM-логика не должна жить внутри скрипта — промпты и инструкции для модели выносятся в отдельные скиллы.

Причина: скрипт — это детерминированный инструмент с понятным вводом и выводом. Встраивание промптов и обработки LLM-ответов прямо в скрипт делает его хрупким: промпт требует итеративной доработки, а скрипт — стабильного алгоритма. Разделение позволяет улучшать логику скрипта и качество промптов независимо друг от друга, а также использовать инструментарий skill-creator (eval-тесты, оценка качества, итерации) для доводки LLM-части.

#### Архитектура: скрипт как оркестратор

Скрипт выступает оркестратором — управляет потоком данных и вызывает LLM через скиллы. Механизм работы:

1. **Промпты живут в скиллах.** Для каждого LLM-шага создаётся отдельный скилл через `skill-creator`. Скилл содержит инструкции для модели в файле `SKILL.md`. Скиллы, предназначенные для программного вызова из скриптов (не пользователями), размещаются в `.claude/skills/{имя-скрипта}/{имя-скилла}/SKILL.md` — сгруппированные по скрипту, к которому относятся. Имя скрипта берётся из имени спецификации без числового префикса (например, `002-llm-analyze-module.md` → `llm-analyze-module`). Группировка нужна, чтобы при росте количества скриптов скиллы не превращались в плоскую кучу — было понятно, какие скиллы принадлежат какому скрипту. Пользовательские скиллы размещаются в `.claude/skills/{имя-скилла}/SKILL.md` (например, `.claude/skills/llm-analyze-module/SKILL.md`).

2. **Скрипт читает SKILL.md и передаёт как системный промпт.** При вызове LLM скрипт читает содержимое нужного `SKILL.md` и передаёт его как `--system-prompt` в команду `claude` CLI. Таким образом данные передаются автоматически — никакого ручного копирования.

3. **Данные между шагами — через временные файлы в `development/temp-docs/`.** Для каждого запуска цепочки LLM скрипт создаёт отдельную подпапку в формате `{ДД-ММ}_{название-задачи}/` — название должно кратко описывать, какую задачу решает вся цепочка. Все промежуточные файлы (JSON, markdown) этого запуска складываются в эту подпапку. Следующий шаг читает нужный файл оттуда как входные данные. Такая структура не даёт результатам разных запусков смешиваться и позволяет легко найти нужный набор данных.

#### Что описывать в спецификации

Для каждого шага, требующего LLM, в спецификации нужно описать:

- **Какой шаг алгоритма требует LLM** и что именно модель должна сделать
- **Имя скилла**, который нужно создать (например, `pass1-map`, `pass2-analyze`)
- **Интерфейс скилла:**
  - Вход — что скрипт передаёт скиллу (формат, структура данных, откуда берутся)
  - Выход — что скилл возвращает (формат ответа, структура, критерии качества)
  - Формат обмена — JSON-схема или markdown-шаблон для входных и выходных данных
- **Модель:** для LLM-вызовов указывай Claude Opus 4.6 (`claude-opus-4-6`) — самая мощная модель для задач анализа, генерации и классификации

В разделе «Технические решения» укажи, что скиллы создаются через `skill-creator` до реализации скрипта — скрипт зависит от готовых скиллов.

#### Порядок реализации

Спецификация скрипта с LLM-шагами предполагает цепочку: (1) написать ТЗ скрипта с описанием скиллов → (2) создать скиллы через skill-creator → (3) реализовать скрипт через dev-script-implement. Каждый шаг зависит от предыдущего, поэтому в спецификации стоит явно указать эту последовательность.

Спецификация сохраняется в `/development/script-specs`

## Шаг 6: Верификация

**THINK HARD** — перечитай спецификацию и проверь:

1. **Полнота** — можно ли по этому ТЗ написать скрипт без дополнительных вопросов?
2. **Точность** — пути к файлам, имена таблиц, форматы данных — всё проверено в кодовой базе?
3. **Алгоритм** — каждый шаг понятен и выполним?
4. **Примеры** — примеры запуска и вывода реалистичны?
5. **Крайние случаи** — учтены ситуации с пустыми данными, ошибками, нестандартным форматом?

Выведи краткий отчёт:
```
Верификация:
- Полнота: ✅/⚠️
- Точность: ✅/⚠️
- Алгоритм: ✅/⚠️
- Примеры: ✅/⚠️
- Крайние случаи: ✅/⚠️
```
