# Оркестратор инженерного пайплайна

**THINK HARD.** На КАЖДОМ шаге этого скилла используй максимальную глубину мышления. Это обязательное требование без исключений. Перед каждым решением — глубокий анализ. Перед каждым запуском агента — проверка предусловий. Перед каждым переходом между фазами — верификация ворот. Ты управляешь всем проектом от начала до конца — цена ошибки оркестратора выше, чем у любого отдельного скилла, потому что ошибка здесь может каскадно сломать весь процесс.

## Что это за скилл

Ты — дирижёр оркестра. Каждый музыкант (скилл) умеет играть свою партию, но без дирижёра они не знают когда вступать, в каком порядке и что делать при ошибке. Ты координируешь 11 скиллов через 10 фаз, ведя проект от документации до готового кода.

**Главный принцип: ты НЕ выполняешь работу сам.** Каждая задача делегируется агенту (Agent tool), который вызывает соответствующий скилл. Ты определяешь что нужно делать, в каком порядке, и координируешь результаты. Дирижёр не играет на скрипке.

## Аргументы

`$ARGUMENTS`

Ожидаемый аргумент: `brd_path` — путь к BRD-документу.

- При первом запуске (нет `pipeline-state.json`) аргумент `brd_path` обязателен. Если не передан — спроси: «Укажи путь к BRD-документу, например: `dev/docs/brd/brd-user-journeys.md`»
- При возобновлении (есть `pipeline-state.json`) аргумент не нужен — путь уже сохранён в файле состояния

## Шаг 0: Определение режима — новый запуск или возобновление

THINK HARD.

Прочитай `pipeline-state.json` в корне проекта.

### Файл не найден — новый запуск

1. Убедись, что `brd_path` передан в аргументах. Если нет — спроси у пользователя
2. Проверь, что файл BRD существует по указанному пути. Если нет — сообщи пользователю
3. Переходи к Фазе 0

### Файл найден — возобновление

1. Прочитай файл целиком
2. Определи `current_phase` — номер текущей фазы
3. Внутри текущей фазы найди первую задачу со статусом `pending` или `in_progress`
4. Сообщи пользователю: «Пайплайн обнаружен. Текущая фаза: {N} — {name}. Продолжаю с задачи {task_id}.»
5. Переходи к соответствующей фазе и задаче

Формат `pipeline-state.json` описан в `dev/docs/specs/pipeline-spec.md`, раздел D1.

## Ключевые правила оркестрации

Эти правила действуют на протяжении всего пайплайна. Они здесь, а не внутри отдельных фаз, потому что каждое правило применяется многократно и единообразно.

### Правило 1: Все задачи — только через агентов

Каждая задача выполняется через Agent tool. Оркестратор никогда не пишет код, не создаёт файлы, не запускает тесты напрямую. Единственное исключение — чтение и обновление `pipeline-state.json`, которое делает сам оркестратор (чтобы агенты не перезаписывали файл состояния параллельно).

Как формулировать задачу агенту — шаблон:

> Ты — агент оркестратора пайплайна. Твоя задача: вызвать скилл `{skill-name}` с аргументами `{arguments}`.
> Контекст проекта: Claude Manager — Telegram-бот, управляющий Claude Code с телефона.
> После завершения скилла верни: краткий результат (что создано/изменено) и статус (успех/ошибка).
> Если скилл задаёт вопросы пользователю — передавай их через AskUserQuestion, не отвечай сам.

### Правило 2: Параллелизм для независимых задач

Когда несколько задач не зависят друг от друга — запускай агентов параллельно (несколько Agent tool вызовов в одном сообщении). Это критически экономит время: три модуля слоя 0, реализуемые параллельно, занимают столько же времени, сколько один.

Где безопасно параллелить:
- Спецификации модулей одного слоя (фаза 2) — модули одного слоя не зависят друг от друга по определению
- Реализация модулей одного слоя (фаза 3) — аналогично
- Чтение нескольких документов — всегда безопасно

Где нельзя параллелить:
- Реализация модулей разных слоёв — слой 1 зависит от слоя 0
- Последовательные фазы — фаза 3 зависит от результатов фазы 2
- Задачи, которые модифицируют одни и те же файлы

### Правило 3: Git-коммит после каждого артефакта

После каждого завершённого агента — git-коммит. Это создаёт точки отката: если следующий шаг сломает что-то, можно откатиться к последнему рабочему состоянию.

Формат коммита: `pipeline({phase}): {что сделано}`

Примеры:
- `pipeline(phase-0): валидация BRD завершена, отчёт создан`
- `pipeline(phase-2): спецификация модуля config создана`
- `pipeline(phase-3): модуль message_splitter реализован, тесты зелёные`
- `pipeline(phase-6): ревью проход 2 — исправлены 3 проблемы безопасности`

Как коммитить: запусти агента с задачей сделать коммит. Агент должен:
1. `git add` — только файлы, созданные/изменённые текущей задачей (не `git add -A`)
2. `git commit -m "pipeline({phase}): {описание}"`

### Правило 4: Пуш — только по подтверждению

Перед `git push` оркестратор ВСЕГДА останавливается и спрашивает пользователя через `AskUserQuestion`: «Все коммиты фазы {N} готовы. Запушить в remote?» Пуш выполняется только при явном подтверждении. Это защита от случайного попадания незавершённой работы в общий репозиторий.

### Правило 5: Интерактивные точки

Оркестратор останавливается и спрашивает пользователя в следующих ситуациях:
- **Фаза 0** — вопросы из validate-brd (расхождения в BRD требуют решения пользователя)
- **Фаза 5** — авторизация Telethon (SMS-код для E2E тестирования). Это одноразовая процедура — после первой авторизации Telethon сохраняет сессию в файл
- **Фаза 9** — интерактивное тестирование (пользователь видит сообщения бота в Telegram в реальном времени)
- **Любой git push** — перед отправкой в remote
- **Любая критическая ошибка** — перед остановкой пайплайна

### Правило 6: Возврат спеки при ошибке реализации

Если агент-реализатор (фаза 3) обнаружил ошибку в спецификации, которую невозможно исправить на уровне кода:
1. Спецификация возвращается из `dev/docs/specs/realised/` в `dev/docs/specs/`
2. Запускается агент для исправления спецификации (скилл spec-module)
3. Коммит с исправленной спекой
4. Реализация модуля продолжается с обновлённой спекой

Это случается редко, но механизм нужен: спека могла не учесть edge case, который обнаружился при написании кода.

## Фаза 0: Валидация BRD

THINK HARD.

**Цель:** убедиться, что BRD и вся документация проекта непротиворечивы и полны перед началом реализации.

**Скилл:** validate-brd

### Действия

1. Запусти агента с задачей вызвать скилл `validate-brd`:

   > Вызови скилл validate-brd. Путь к BRD: `{brd_path}`. Для каждого найденного расхождения или пробела — спрашивай пользователя через AskUserQuestion, предлагая варианты решения. Зафиксируй все решения в отчёте.

2. Дождись завершения. Агент будет задавать вопросы пользователю — это нормально и ожидаемо.

3. Git-коммит: `pipeline(phase-0): валидация BRD завершена, отчёт создан`

4. Обнови `pipeline-state.json`:
   - Фаза 0: status = `completed`, completed_at = текущее время
   - current_phase = 1

### Ворота качества

- Файл `dev/docs/brd/brd-validation-report_*.md` существует
- Все расхождения разрешены (ответы пользователя зафиксированы)

Проверь ворота вручную (прочитай отчёт и убедись, что нет открытых вопросов). Если всё чисто — переход к фазе 1.

## Фаза 2: Спецификации

THINK HARD.

**Цель:** декомпозировать проект на модули и написать спецификацию для каждого.

**Скиллы:** brd-decompose, spec-module

### Шаг 2.1: Декомпозиция

1. Запусти агента:

   > Вызови скилл brd-decompose. BRD: `{brd_path}`. Создай граф зависимостей модулей с разбивкой по слоям.

2. Дождись завершения — результат: `dev/docs/specs/module-dependency-graph.md`

3. Git-коммит: `pipeline(phase-2): декомпозиция на модули, граф зависимостей создан`

4. Прочитай `module-dependency-graph.md` и извлеки:
   - Список всех модулей
   - Разбивку по слоям (0, 1, 2, 3)
   - Обнови `pipeline-state.json` — заполни поля `modules` и `layers`

### Шаг 2.2: Спецификации модулей

Для каждого слоя (0, 1, 2, 3), для модулей внутри слоя — параллельно:

1. Запусти агентов параллельно (один агент на модуль одного слоя):

   > Вызови скилл spec-module с аргументом `{module_name}`. Создай детальную спецификацию модуля.

   Пример для слоя 0 — три агента одновременно: spec-module config, spec-module message_splitter, spec-module session_reader (и другие модули слоя 0, если они есть).

2. После завершения каждого агента — git-коммит: `pipeline(phase-2): спецификация модуля {module} создана`

3. Обнови `pipeline-state.json` — задача спецификации модуля = completed

4. Когда все модули слоя специфицированы — переходи к следующему слою. Между слоями можно не ждать (спецификации не зависят от порядка слоёв, в отличие от реализации), поэтому допустимо запускать все модули всех слоёв параллельно.

### Ворота качества

Запусти агента-верификатора:

> Проверь ворота фазы 2:
> 1. Файл `dev/docs/specs/module-dependency-graph.md` существует
> 2. Для каждого модуля из графа существует `dev/docs/specs/{module}_spec.md`
> 3. Прочитай BRD и список спецификаций — каждый CJM должен быть покрыт хотя бы одной спекой
> Верни: PASS или FAIL + что не покрыто

Если PASS — обнови `pipeline-state.json` (фаза 2 = completed, current_phase = 3).

## Фаза 3: Реализация

THINK HARD.

**Цель:** реализовать все модули по спецификациям, снизу вверх по слоям зависимостей.

**Скилл:** implement-module

Это самая длинная фаза. Модули реализуются по слоям: сначала слой 0 (модули без зависимостей), потом слой 1 (зависит от слоя 0), и так далее. Внутри одного слоя — модули реализуются параллельно.

### Алгоритм

Для каждого слоя (0, 1, 2, 3) последовательно:

1. **Запуск агентов слоя** — для каждого модуля в текущем слое запусти агента параллельно:

   > Вызови скилл implement-module с аргументом `{module_name}`. Реализуй модуль по спецификации, напиши юнит-тесты, добейся зелёных тестов. После успешной реализации перенеси спеку в `dev/docs/specs/realised/`.

2. **После завершения каждого агента:**
   - Git-коммит: `pipeline(phase-3): модуль {module} реализован, тесты зелёные`
   - Обнови `pipeline-state.json` — задача реализации модуля = completed

3. **После завершения всех модулей слоя** — запусти агента для регрессионной проверки:

   > Запусти все тесты проекта: `dev/scripts/run-all-tests.sh` (если существует, иначе `python -m pytest tests/ -v --tb=short`). Все тесты должны быть зелёными. Если есть красные — верни список упавших тестов.

4. **Если регрессия обнаружена** (тесты, которые были зелёными, стали красными):
   - Запусти агента для починки
   - После починки — снова полный прогон тестов
   - Git-коммит с исправлением
   - Максимум 5 циклов починки на одну регрессию

5. **Переход к следующему слою** — только после того, как все модули текущего слоя реализованы и все тесты зелёные

### Обработка ошибки в спецификации

Если агент-реализатор сообщил, что спецификация содержит ошибку:

1. Верни спеку из `dev/docs/specs/realised/` в `dev/docs/specs/` (если она уже была перенесена)
2. Запусти агента: `Вызови скилл spec-module с аргументом {module_name}. Исправь спецификацию: {описание проблемы}`
3. Git-коммит: `pipeline(phase-3): спецификация {module} исправлена`
4. Запусти реализацию модуля заново

### Ворота качества

Запусти `dev/scripts/check-phase-gate.py --phase 3` (через агента).

Дополнительно проверь:
- Все модули из графа реализованы (файлы в `src/claude_manager/` существуют)
- Для каждого модуля есть `tests/test_{module}.py`
- Все юнит-тесты зелёные
- Все спеки в `dev/docs/specs/realised/`

Если PASS — обнови `pipeline-state.json` (фаза 3 = completed, current_phase = 4).

## Фаза 5: E2E тестирование

THINK HARD.

**Цель:** протестировать бота вживую — с реальным Telegram и реальным Claude.

**Скилл:** test-e2e

**ИНТЕРАКТИВНАЯ ФАЗА:** может потребовать SMS-код для авторизации Telethon. Если агент сообщит, что нужна авторизация — передай запрос пользователю.

### Действия

1. Перед запуском проверь: файл `.env` содержит `TELETHON_API_ID`, `TELETHON_API_HASH`, `TELETHON_PHONE`, `TELETHON_BOT_USERNAME`. Если переменных нет — спроси пользователя.

2. Запусти агента:

   > Вызови скилл test-e2e. Проведи E2E тестирование бота с реальным Telegram. Если потребуется SMS-код для Telethon — спроси через AskUserQuestion. Для каждого CJM: отправь сообщение боту, получи ответ, проверь через LLM-верификацию.

3. Git-коммит: `pipeline(phase-5): E2E тестирование завершено, все CJM pass`

4. Обнови `pipeline-state.json`

### Ворота качества

Запусти `dev/scripts/check-phase-gate.py --phase 5` (через агента).

Проверь:
- `dev/docs/logs/testing/e2e-test-plan_*.md` существует
- `dev/docs/logs/testing/e2e-test-results_*.md` — все сценарии = pass
- Все автотесты зелёные

Если PASS — обнови `pipeline-state.json` (фаза 5 = completed, current_phase = 6).

## Фаза 6: Ревью

THINK HARD.

**Цель:** 4-проходное ревью кода с автоисправлением. Качество, безопасность, архитектура, соответствие BRD.

**Скилл:** review-code

## Завершение пайплайна: финальный отчёт

THINK HARD.

Когда все 10 фаз завершены (все статусы = completed), собери финальный отчёт.

### Что собрать

Прочитай `pipeline-state.json` и соответствующие файлы, чтобы собрать статистику:

- **Модули** — сколько модулей реализовано (количество задач типа `implement` в фазе 3)
- **Тесты** — количество по типам:
  - Юнит-тесты (количество файлов `tests/test_*.py`)
  - Интеграционные тесты (количество файлов в `tests/integration/`)
  - E2E тесты (результаты из `dev/docs/logs/testing/e2e-test-results_*.md`)
  - Пользовательские сценарии (результаты из `dev/docs/logs/testing/user-test-results_*.md`)
- **Ревью** — сколько проблем найдено и исправлено (из `dev/docs/review-report_*.md`)
- **Циклы починки** — суммарное количество `fix_cycles` по всем задачам в `pipeline-state.json`
- **Время** — общее время пайплайна (от started_at фазы 0 до completed_at фазы 9)

### Формат отчёта

Выведи пользователю:

```
ПАЙПЛАЙН ЗАВЕРШЁН

Модулей реализовано: {N}
Тестов написано: {unit} юнит + {integration} интеграционных + {e2e} E2E + {user} пользовательских = {total} всего
Проблем найдено при ревью: {N}, все исправлены
Циклов починки: {N}
Статус: ПРОЕКТ ГОТОВ
```

## Обработка ошибок

### Циклы починки — максимум 5

Каждая задача может пройти до 5 циклов починки (попытка → ошибка → исправление → повтор). Если после 5 циклов задача всё ещё не зелёная:

1. Запиши в `pipeline-state.json`: задача status = `failed`, error = описание проблемы
2. Выведи пользователю диагностику:
   - Какая задача упала
   - Какие ошибки возникали в каждом цикле
   - Последний лог ошибки
3. Останови пайплайн

### Регрессия

Если после исправления одного модуля сломались тесты другого:

1. Запиши в `pipeline-state.json` информацию о регрессии
2. Попробуй откатить через `git checkout` до последнего зелёного коммита
3. Сообщи пользователю: какой коммит вызвал регрессию, что сломалось
4. Останови пайплайн — регрессия требует ручного разбора

### E2E / пользовательское тестирование — бот не запускается

Если бот не может запуститься для E2E или пользовательского тестирования:

1. Проверь `.env` — все ли переменные заполнены
2. Проверь логи бота — что пишет при запуске
3. Проверь сеть — доступен ли Telegram API
4. Сообщи пользователю конкретную проблему
5. Останови пайплайн

## Обновление pipeline-state.json

После каждого действия оркестратор обновляет файл состояния. Это позволяет продолжить с места остановки при следующем запуске.

### Что обновлять

- `current_phase` — номер текущей фазы
- Статус фазы: `pending` → `in_progress` → `completed` (или `failed`)
- Статус задач внутри фазы: `pending` → `in_progress` → `completed` (или `failed`)
- `fix_cycles` — сколько циклов починки прошла задача
- `completed_at` — время завершения (для фаз со статусом `completed`)
- `error` — описание ошибки (для задач со статусом `failed`)

### Когда обновлять

1. Перед запуском задачи: статус задачи = `in_progress`
2. После успешного завершения задачи: статус = `completed`
3. При ошибке: статус = `failed`, error = описание
4. При завершении фазы: статус фазы = `completed`, current_phase += 1
5. При каждом цикле починки: fix_cycles += 1

Формат файла — JSON, как описано в `dev/docs/specs/pipeline-spec.md`, раздел D1.
