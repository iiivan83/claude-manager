# Декомпозиция проекта на модули с графом зависимостей

THINK HARD. На каждом шаге этого скилла используй максимальную глубину мышления. Это критически важно — ошибка в графе зависимостей каскадно влияет на все последующие фазы: спецификации, реализацию, тестирование. Неправильный слой или пропущенная зависимость приведёт к тому, что модуль будет реализован раньше, чем его зависимости, и не скомпилируется.

## Что делает этот скилл

Ты анализируешь BRD (документ с пользовательскими сценариями) и архитектурный документ проекта, чтобы создать полную карту модулей: что от чего зависит, какие пользовательские сценарии обслуживает каждый модуль, в каком порядке модули нужно реализовывать. Результат — файл `development/specs/module-dependency-graph.md`, который станет основой для фазы спецификаций и реализации.

## Шаг 1: Сбор исходных данных

### 1.1 Определи пути к документам

Проверь, передал ли пользователь пути через аргументы (`$ARGUMENTS`). Если нет — прочитай `pipeline-state.json` в корне проекта и возьми пути оттуда.

Нужны три источника:
- **BRD** — документ с пользовательскими сценариями (CJM). Путь по умолчанию: `development/docs/brd-user-journeys.md`
- **CLAUDE.md** — корневой файл проекта с архитектурой, структурой и принципами
- **pipeline-state.json** — текущее состояние пайплайна (если существует)

Если `pipeline-state.json` не найден — это нормально, значит скилл запущен до того, как пайплайн был инициализирован. Используй пути по умолчанию.

### 1.2 Прочитай все три документа

THINK HARD. Запусти параллельные агенты (Agent tool) для чтения каждого документа — это ускорит сбор данных. Каждый агент читает свой документ и возвращает структурированную выжимку.

**Агент 1 — BRD:**
- Прочитай BRD целиком
- Извлеки список всех CJM (пользовательских сценариев) с номерами и кратким описанием
- Для каждого CJM выпиши: какие технические действия описаны (проверка доступа, создание сессии, отправка в Telegram, чтение файлов с диска и т.д.)
- Извлеки раздел «Механизмы, которые работают во всех сценариях» — это сквозная функциональность, которая затрагивает несколько модулей

**Агент 2 — CLAUDE.md:**
- Прочитай CLAUDE.md целиком
- Извлеки: список модулей из «Структура проекта», описание слоёв из «Архитектурные принципы», правило зависимостей
- Для каждого модуля: имя файла, короткое описание из комментария, к какому архитектурному слою относится

**Агент 3 — pipeline-state.json (если существует):**
- Прочитай файл
- Извлеки: текущую фазу, список модулей (если уже заполнен), слои (если уже заполнены)
- Если файла нет — верни: «файл не найден, используем данные из BRD и CLAUDE.md»

## Шаг 2: Извлечение CJM

THINK HARD. На основе данных от Агента 1 составь полный пронумерованный список CJM.

Формат каждого CJM:
- **Номер** (CJM-01, CJM-02, ...)
- **Название** (одна фраза — о чём сценарий)
- **Технические действия** — что система должна делать (не что видит пользователь, а что происходит внутри)

Также выдели **сквозные механизмы** — функциональность, которая работает во всех или нескольких сценариях (проверка доступа, промежуточные обновления, координация watcher/handler и т.д.). Каждый сквозной механизм пометь: какие CJM он затрагивает.

## Шаг 3: Извлечение модулей и зависимостей

THINK HARD. На основе данных от Агента 2 составь полный список модулей.

Для каждого модуля определи:
- **Имя** (имя файла без расширения: `config`, `bot`, `message_splitter`)
- **Назначение** (одно предложение — что делает)
- **Архитектурный слой** (транспортный, бизнес-логика, инфраструктура, утилиты)
- **Зависимости** — от каких других модулей проекта зависит

Как определить зависимости:
- Прямые: модуль A импортирует функции из модуля B → A зависит от B
- Из описания архитектуры: «bot.py вызывает process_manager» → bot зависит от process_manager
- Из правила зависимостей: верхний слой может вызывать нижний → транспортный слой зависит от бизнес-логики и инфраструктуры
- Сквозные зависимости: если модуль использует конфигурацию → зависит от config

Не добавляй зависимости от стандартной библиотеки или сторонних пакетов — только от других модулей проекта.

## Шаг 4: Трейсабельность CJM → модули

THINK HARD. Для каждого модуля определи, какие CJM он обслуживает. Это критически важная связь — она показывает, зачем каждый модуль существует и какие требования он реализует.

Как определить связь CJM → модуль:
- Пройдись по каждому CJM. Для каждого технического действия в сценарии спроси: «Какой модуль это делает?»
- Пример: CJM-02 описывает «Проверка текущего состояния — есть ли активная сессия» → это делает session_manager
- Пример: CJM-02 описывает «Конвертирует ответ из Markdown в HTML, разбивает на части» → это делает message_splitter
- Сквозные механизмы привязываются к нескольким CJM

Проверка полноты: каждый CJM должен быть покрыт хотя бы одним модулем. Каждый модуль должен обслуживать хотя бы один CJM. Если модуль не привязан ни к одному CJM — это сигнал: либо пропущена связь, либо модуль лишний.

## Шаг 5: Построение графа зависимостей и слоёв

THINK HARD. Это ключевой шаг — от него зависит порядок реализации всего проекта.

### 5.1 Построй граф зависимостей

Для каждого модуля ты уже знаешь его зависимости (из шага 3). Теперь визуализируй это как граф:
- Каждый модуль — вершина
- Каждая зависимость — направленное ребро (A → B означает «A зависит от B»)

### 5.2 Проверь граф на циклы

Циклическая зависимость (A → B → C → A) — это ошибка проектирования. Если нашёл цикл:
1. Укажи его явно в результате
2. Предложи способ разрешения (обычно: вынести общую логику в отдельный модуль или инвертировать зависимость через callback/интерфейс)

### 5.3 Назначь слои реализации

Слои определяются автоматически по зависимостям:
- **Слой 0** — модули без зависимостей от других модулей проекта. Их можно реализовать первыми
- **Слой 1** — модули, которые зависят только от модулей слоя 0
- **Слой 2** — модули, которые зависят от модулей слоёв 0 и/или 1
- **Слой N** — модули, которые зависят от модулей слоёв 0..N-1

Алгоритм (топологическая сортировка):
1. Помести все модули без зависимостей в слой 0
2. Убери их из графа
3. Теперь некоторые модули остались без зависимостей — помести их в слой 1
4. Повторяй, пока все модули не будут распределены
5. Если остались модули с неразрешёнными зависимостями — это цикл (см. шаг 5.2)

### 5.4 Определи порядок реализации

На основе слоёв:
- Модули одного слоя можно реализовывать **параллельно** (они не зависят друг от друга в рамках слоя)
- Слои реализуются **последовательно** — слой N только после слоя N-1
- Если в слое есть модуль, который используется остальными модулями слоя — пометь его как «реализовать первым в слое»

## Шаг 6: Итеративная самопроверка

Этот скилл генерирует спецификацию, на которой строится вся последующая работа. Ошибка здесь размножается каскадно. Поэтому — обязательный цикл проверки.

### Проверка 1 (сразу после составления черновика)

THINK HARD. Перечитай результат критическим взглядом. Проверь:

1. **Полнота CJM**: все ли CJM из BRD извлечены? Пересчитай и сравни с BRD
2. **Полнота модулей**: все ли модули из CLAUDE.md учтены? Сравни список с «Структура проекта»
3. **Корректность зависимостей**: для каждой зависимости A → B проверь — действительно ли A использует функциональность B? Нет ли лишних зависимостей?
4. **Полнота трейсабельности**: каждый CJM покрыт хотя бы одним модулем? Каждый модуль привязан хотя бы к одному CJM?
5. **Корректность слоёв**: слой каждого модуля корректен? Модуль в слое N действительно зависит только от слоёв 0..N-1?
6. **Отсутствие циклов**: граф ацикличен?

Исправь все найденные проблемы.

### Проверка 2 (после исправлений)

THINK HARD. Даже если первая проверка не нашла проблем — проведи вторую. Перечитай результат заново, сверяясь с BRD и CLAUDE.md. Вторая проверка нужна для подтверждения — она ловит ошибки, пропущенные при первом проходе из-за «замыленного взгляда».

Повторяй цикл, пока проверка не выявит ноль замечаний. Минимум — две итерации проверки.

## Шаг 7: Запись результата

Запиши файл `development/specs/module-dependency-graph.md` в точном формате ниже.

### Формат файла

```markdown
# Граф зависимостей модулей

Дата генерации: ГГГГ-ММ-ДД
Источники: BRD (путь), CLAUDE.md

## Список CJM

- **CJM-01** — Первый запуск и настройка
- **CJM-02** — Отправка текстового сообщения
- ...

## Слой 0 (нет зависимостей)

- **config** — загрузка настроек из .env. CJM: 01. Зависимости: нет
- **message_splitter** — разбивка длинных сообщений. CJM: 02, 03. Зависимости: нет

## Слой 1 (зависит от слоя 0)

- **claude_runner** — обёртка для запуска Claude CLI. CJM: 02, 03, 04. Зависимости: config

## Слой 2 (зависит от слоёв 0-1)

- **process_manager** — управление процессами Claude. CJM: 02, 03, 08. Зависимости: claude_runner, config

...

## Порядок реализации

1. [параллельно] config, message_splitter, session_reader, daily_session_registry
2. [параллельно] claude_runner, session_manager
3. [параллельно] process_manager, session_watcher
4. [последовательно] bot → main

## Сквозные механизмы

- **Проверка доступа** — CJM: все. Модули: bot
- **Промежуточные обновления** — CJM: 02, 03. Модули: process_manager, bot
- ...
```

Каждый модуль описывается одной строкой:
- **имя** — описание. CJM: перечисление номеров. Зависимости: перечисление модулей или «нет»

## Шаг 8: Обновление pipeline-state.json

Если `pipeline-state.json` существует — обнови его:

1. Заполни поле `modules` — полный список имён модулей
2. Заполни поле `layers` — словарь: номер слоя → список модулей
3. Если есть задача с `"type": "decompose"` в текущей фазе — обнови её статус на `"completed"` и добавь `"completed_at"` с текущей датой-временем в формате ISO 8601

Если файла нет — создай его с минимальной структурой:
```json
{
  "project": "имя-проекта-из-CLAUDE.md",
  "brd_path": "путь-к-BRD",
  "current_phase": 2,
  "modules": ["список", "модулей"],
  "layers": {
    "0": ["модули", "слоя", "0"],
    "1": ["модули", "слоя", "1"]
  },
  "phases": {
    "2": {
      "name": "Спецификации",
      "status": "in_progress",
      "tasks": [
        {"id": "T1.1", "type": "decompose", "status": "completed", "completed_at": "ISO-дата"}
      ]
    }
  }
}
```

## Чего этот скилл НЕ делает

- **Не пишет спецификации модулей** — для этого есть скилл `spec-module`. Этот скилл только определяет ЧТО это за модули и в каком порядке их специфицировать
- **Не пишет код** — для этого есть скилл `implement-module`
- **Не валидирует BRD** — для этого есть скилл `validate-brd`. Этот скилл предполагает, что BRD уже валиден
- **Не принимает архитектурных решений** — он извлекает структуру из существующих документов, а не проектирует новую
