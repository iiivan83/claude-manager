# Системное исправление по отчёту о первопричинах

Этот скилл — оркестратор. Он читает root-cause отчёт, проверяет каждую рекомендацию через
верификатор (`agents/verifier.md`), применяет одобренные изменения через агентов
и контролирует весь процесс.

Скилл универсален — работает в любом проекте с любым стеком. Контекст проекта (правила,
стандарты, архитектура) обнаруживается автоматически.

Главный принцип: **прагматизм** — лучше 3 точных исправления, чем 10 перестраховочных.

## Global Reference Rules

Перед началом работы прочитай `~/.claude/references/writing-style-guide.md` и следуй стилю
во всех создаваемых документах и отчётах.

## Глубина мышления

Думай как опытный тех-лид, который лично отвечает за стабильность продакшена. Каждое решение
проходит через внутренний чек-лист:

- **Побочные эффекты** — что может сломаться в соседних модулях из-за этого изменения?
- **Граничные случаи** — как это поведёт себя на пустых данных, при конкурентном доступе,
  при неожиданном порядке вызовов?
- **Откат** — если изменение окажется неудачным, как быстро можно вернуть всё назад?
- **Альтернативы** — есть ли более простой способ добиться того же результата?
- **Достаточность** — это изменение действительно решает корневую причину, а не маскирует
  симптом?

Все агенты этого скилла работают с такой же глубиной — разбирают задачу по частям,
рассматривают альтернативы и перепроверяют выводы перед тем, как действовать.

---

## Входные данные

Путь к root-cause отчёту. Варианты получения:
- Пользователь указывает файл явно
- Пользователь говорит «исправь по последнему отчёту» → найди свежие файлы с "root-cause" в имени через Glob
- Пользователь только что запустил анализ первопричин → используй путь из его результата

---

## Режим работы

- **Обычный** (по умолчанию) — показывает результаты верификации, ждёт подтверждения пользователя
- **Pipeline mode** (`pipeline_mode=true` в аргументах) — автономный: без подтверждений

## Фаза 0: Парсинг отчёта и инициализация

### Инициализация логирования

1. Сгенерируй временную метку `{DD.MM_HH.MM}` (текущее время)
2. Определи имя отчёта из имени файла (например, `infrastructure-gaps` из
   `08-04_20-54_apply-root-cause-fixes-infrastructure-gaps.md`)
3. Создай папку логов:
   ```
   dev/docs/logs/root-cause-fixes/{DD.MM_HH.MM}-apply-root-cause-fixes-{report-name}/
   ```
4. Создай `orchestrator-log.json` в этой папке:
   ```json
   {
     "pipeline": "apply-root-cause-fixes",
     "created_at": "ISO-8601",
     "report_file": "<путь к root-cause отчёту>",
     "steps": []
   }
   ```

**Паттерн логирования:** запись делается ДО начала каждой фазы (не после) — если пайплайн прервётся, лог покажет, на какой фазе это произошло.

Записывай шаг в `steps` массив `orchestrator-log.json`:
```json
{
  "phase": "phase-0",
  "start_time": "ISO-8601",
  "end_time": "ISO-8601",
  "success": true,
  "comment": "краткое описание результата"
}
```

### Парсинг

Прочитай root-cause отчёт целиком. Извлеки ВСЕ рекомендации — ищи в секциях вроде:
- «Рекомендации по исправлению», «Recommendations», «Fixes», «What to fix»
- «Рекомендации по предотвращению», «Prevention», «How to prevent»
- Любые другие секции с конкретными предложениями по изменениям

Отчёт может быть в свободном формате — не требуй жёсткой структуры. Главное — найти
рекомендации и понять их контекст.

Для каждой рекомендации определи тег по содержанию:
- `[CODE]` — изменение в коде (исходники, конфиги, скрипты)
- `[SKILL]` — изменение в скилле Claude (файлы в `.claude/skills/`)
- `[DOC]` — документация
- `[ARCHITECTURE]` — архитектурные документы
- `[OPTIONAL]` — необязательное улучшение

Если теги уже расставлены в отчёте — используй их.

Результат: пронумерованный список всех рекомендаций с тегами.

Запиши шаг `phase-0` в orchestrator-log.json.

## Фаза 1: Верификация рекомендаций

Цель — проверить каждую рекомендацию на адекватность и соответствие правилам проекта
ДО применения. Это экономит время и предотвращает каскадные ошибки.

### Запуск верификатора

Вызови Agent tool с `model: "opus"`. Агент должен прочитать инструкции из
`.claude/skills/apply-root-cause-fixes/agents/verifier.md` и выполнить их.

В промпте агенту передай:

```
Прочитай инструкции из `.claude/skills/apply-root-cause-fixes/agents/verifier.md`
и выполни верификацию.

Рекомендации для верификации:
{пронумерованный список рекомендаций с тегами из Фазы 0}

Контекст проблемы:
{краткое описание проблемы и корневой причины из отчёта}

Путь к root-cause отчёту: {путь}
```

### Интерпретация результатов

Сопоставь вердикты верификатора со статусами:
- **ОДОБРЕНО** → ПРИНЯТО — применять как есть
- **ОДОБРЕНО С ЗАМЕЧАНИЯМИ** → ПРИНЯТО — применять с учётом замечаний
- **ОТКЛОНЕНО** → ОТКЛОНЕНО — не применять

### Контрольная точка (только обычный режим)

> В pipeline_mode этот шаг пропускается — результаты принимаются автоматически.

Покажи пользователю результаты верификации:

```
## Верификация рекомендаций

**Исходный отчёт:** {путь}
**Всего:** N

### ПРИНЯТО (N шт.)
1. [TAG] {описание} — {вердикт}: {краткое обоснование}

### ОТКЛОНЕНО (N шт.)
2. [TAG] {описание} — {причина отклонения}
```

Жди подтверждения. Пользователь может:
- Согласиться и продолжить
- Вернуть отклонённые пункты
- Убрать принятые
- Изменить приоритеты

Без подтверждения к Фазе 2 не переходи.

Запиши шаг `phase-1` в orchestrator-log.json (с количеством принятых/отклонённых).

## Фаза 2: Применение исправлений

Каждое принятое изменение = отдельный агент через Agent tool.

### Промпт агента

Каждый агент-исполнитель получает:
- Конкретную задачу (цитата рекомендации из отчёта)
- Контекст проблемы (из секции «Корневая причина»)
- Пути к файлам для изменения
- Замечания верификатора (если ОДОБРЕНО С ЗАМЕЧАНИЯМИ)

### Работа с файлами внутри `.claude/skills/`

Если целевой файл правки находится внутри папки `.claude/skills/`, добавь в промпт
агента-исполнителя инструкцию использовать штатные шаблоны X.1 (полная перезапись)
или X.2 (точечная замена) из раздела «Запись в `.claude/skills/` из CLI-подпроцессов»
корневого `CLAUDE.md` проекта. Формулировка должна быть нейтральной — как штатная
процедура, а не исключительный случай. После применения оркестратор обязан проверить
результат на диске (grep, cat, python3 -c "open").

### Параллелизм

**Параллельно** — если изменения независимы:
- Разные файлы, разные модули
- Документ и код в разных местах
- Два [CODE] изменения в разных файлах

**Последовательно** — если есть зависимость:
- Архитектурный документ → код, который на него ссылается
- Спецификация → реализация
- Исправление кода → тесты

### Маршрутизация по тегам

**[CODE]** — агент правит код напрямую:
- Читает файл, вносит точечное изменение, проверяет что не сломал контекст
- Если в отчёте указаны строки — использует как ориентир (код мог измениться)
- Если целевой файл лежит внутри `.claude/skills/` — применяется правило из подраздела
  «Работа с файлами внутри `.claude/skills/`» выше

**[DOC]** — делегируется агентам-документалистам:
- Перед запуском определи, существует ли целевой файл (проверь путь через Glob)
- **Существующий файл** → запусти Agent tool. Агент читает инструкции из
  `agents/doc-update-agent.md`. Передай в промпте:
  - Путь к документации проекта и docs-index.md (найди через `**/docs-index.md`)
  - Цитату рекомендации из root-cause отчёта
  - Описание корневой причины
  - Путь к целевому файлу
  - Замечания верификатора (если есть)
- **Новый файл** → запусти Agent tool. Агент читает инструкции из
  `agents/doc-create-agent.md`. Передай в промпте:
  - Путь к документации проекта и docs-index.md
  - Цитату рекомендации
  - Описание корневой причины
  - Замечания верификатора (если есть)

**[SKILL]** — агент обновляет скилл:
- **Обычный режим:** вносит изменения в SKILL.md и связанные файлы. Целевые пути лежат
  внутри `.claude/skills/`, поэтому применяется правило из подраздела «Работа с файлами
  внутри `.claude/skills/`» выше
- **Pipeline mode:** записывает чек-лист со статусом «ОЖИДАЕТ РУЧНОГО ПРИМЕНЕНИЯ» —
  автоматическое изменение скиллов из пайплайна слишком рискованно, потому что затрагивает
  все будущие запуски

**[ARCHITECTURE]** — маршрутизируется как [DOC]:
- Архитектурные документы — это документы. Используй ту же логику: `doc-update-agent.md`
  для существующих, `doc-create-agent.md` для новых.
- Если рекомендация требует создания ADR — передай формат из
  `~/.claude/references/agent-document-triggers.md` в промпте агента.

**[OPTIONAL]** — маршрутизация по содержанию рекомендации

### Ошибки

Если агент не справился → запиши ошибку в orchestrator-log.json → продолжай с остальными.
Один сбой не останавливает весь процесс.

Запиши шаг `phase-2` в orchestrator-log.json (с деталями по каждому агенту).

## Фаза 3: Верификация результатов

После завершения всех агентов:

1. **Полнота** — все ПРИНЯТЫЕ изменения применены? Сравни список из Фазы 1 со статусами.
   Если что-то пропущено — сообщи.

2. **Консистентность** — изменения не противоречат друг другу? Если несколько агентов
   правили связанные вещи — прочитай ключевые файлы и сверь значения.

3. **Тесты** — если были [CODE] изменения и в проекте есть тесты:
   - Определи фреймворк: pytest, jest, go test, cargo test, и т.д.
   - Запусти тесты для затронутых модулей
   - Если тестов нет — пропусти, не выдумывай

Запиши шаг `phase-3` в orchestrator-log.json.

## Фаза 4: Документирование, итоги и коммит

### Документалист (условный)

Если среди применённых изменений были рекомендации с тегом `[ARCHITECTURE]`:

Вызови Agent tool. Агент читает инструкции из `agents/documentalist.md` и выполняет их.
Передай в промпте:
- Список применённых [ARCHITECTURE] изменений
- Путь к orchestrator-log.json
- Путь к root-cause отчёту

Если [ARCHITECTURE] изменений не было — пропусти этот шаг.

### Итоговый отчёт

Покажи пользователю:

```
## Результат: {название отчёта}

**Верификация:** N принято, N отклонено из N всего
**Исправления:** N успешно, N ошибок
**Тесты:** пройдены / не пройдены / не запускались

### Детали
1. [TAG] {описание} — УСПЕХ — {что сделано, какие файлы затронуты}
2. [TAG] {описание} — ОШИБКА — {что пошло не так}
```

### Коммит

1. `git status` — если нет изменений, пропусти
2. `git add` для каждого файла, изменённого в Фазе 2 (НЕ `git add -A`)
3. Коммит: `fix(root-cause): {краткое описание из названия отчёта}`
4. Сообщи пользователю о коммите и затронутых файлах

### Архивация отчёта

Если все исправления применены успешно (ноль ошибок) — перенеси исходный root-cause отчёт
в папку `realized/` рядом с ним:

```bash
mv "{путь к отчёту}" "$(dirname "{путь к отчёту}")/realized/"
```

Если были ошибки — отчёт остаётся на месте (проблема ещё не решена полностью).

После переноса добавь перемещённый файл в коммит:
```bash
git add "{старый путь}" "$(dirname "{путь к отчёту}")/realized/{имя файла}"
git commit --amend --no-edit
```

### Финализация лога

Запиши финальный шаг `phase-4` в orchestrator-log.json. Добавь итоговую запись:
```json
{
  "phase": "completed",
  "end_time": "ISO-8601",
  "total_recommendations": 0,
  "accepted": 0,
  "rejected": 0,
  "applied_successfully": 0,
  "errors": 0
}
```

### Сессионный отчёт

Запусти агента для создания сессионного отчёта (обязательно для пайплайнов, правило из AGENTS.md).

Промпт агенту:
```
Используй /session-report.
Контекст: пайплайн apply-root-cause-fixes завершён.
Исходный отчёт: {путь к root-cause отчёту}
Результат: {N принято, N отклонено, N успешно применено, N ошибок}
Папка логов: {путь к папке логов}
```

## Принципы

- **Прагматизм.** Root-cause отчёты часто рекомендуют больше, чем реально нужно.
  Цель — минимум изменений с максимальным эффектом. Если 3 точных изменения решают
  проблему — остальные 5 «на всякий случай» не нужны.

- **Не переследуй.** Отчёт уже содержит расследование. Если указаны файлы и строки —
  используй их, не запускай повторный анализ.

- **Один агент = одна задача.** Не мешай несколько изменений в одном агенте.
  Это упрощает отчётность и позволяет параллелить.

- **Спроси, если непонятно.** Если рекомендация неоднозначна или код изменился после
  анализа — спроси пользователя, а не додумывай.

- **Универсальность.** Скилл не привязан к конкретному проекту — адаптируется
  к любому через автоматическое обнаружение контекста.
