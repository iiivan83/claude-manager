# Системное исправление по отчёту о первопричинах

Этот скилл — **оркестратор**. Он не исправляет проблемы сам, а читает вердикты из root-cause отчёта, маршрутизирует каждое изменение в подходящий скилл через отдельного агента и контролирует весь процесс. Главный принцип: **прагматизм** — лучше 3 точных исправления, чем 10 перестраховочных.

## Входные данные

Путь к root-cause отчёту. Варианты получения:
- Пользователь указывает файл явно: `development/docs/root-cause-reports/30-03_03-06_progress-icon-checkmark.md`
- Пользователь говорит «исправь по последнему отчёту» → найди самый свежий файл в `development/docs/root-cause-reports/` (не в `resolved/`, не в `fix-process/`)
- Пользователь только что запустил root-cause-analysis → используй путь из его результата

---

## Фаза 0: Чтение и разбор отчёта

Прочитай root-cause отчёт целиком. Извлеки ВСЕ рекомендации из двух мест:

1. **«Чек-лист исправлений»** — основной источник. Содержит конкретные задачи с тегами:
   - `[CODE]` — изменение в коде (src/claude_manager/, tests/, скрипты)
   - `[SKILL]` — изменение в скилле Claude
   - `[DOC]` — изменение в документации (development/docs/, development/specs/, CLAUDE.md)

2. **«Верификация решений»** — проверь вердикты каждой рекомендации:
   - **ОДОБРЕНО** → статус **ПРИНЯТО**
   - **ОДОБРЕНО с оговоркой** → статус **ПРИНЯТО** (учти оговорки при маршрутизации)
   - **ОТКЛОНЕНО** → статус **ОТКЛОНЕНО** — пропускаем

Составь пронумерованный список ВСЕХ рекомендаций с тегами и вердиктами. Если в отчёте рекомендации не размечены тегами — расставь теги сам, исходя из содержания (какой файл затрагивается).

---

## Фаза 1: Подтверждение пользователем

Покажи пользователю результаты разбора:

```
## План исправлений

**Исходный отчёт:** {путь}
**Всего рекомендаций:** N

### ПРИНЯТО (N шт.)
1. [CODE] {описание} — ОДОБРЕНО: {обоснование из отчёта}
2. [SKILL] {описание} — ОДОБРЕНО с оговоркой: {обоснование + оговорка}

### ОТКЛОНЕНО (N шт.)
3. [CODE] {описание} — ОТКЛОНЕНО: {обоснование из отчёта}
```

**Жди подтверждения.** Пользователь может согласиться, вернуть отклонённые пункты, добавить свои или изменить приоритеты. Без подтверждения к Фазе 2 не переходи.

---

## Фаза 2: Маршрутизация и параллельное исправление

### Шаг 2.1: Создай отчёт fix-process

Перед запуском агентов создай файл:
```
development/docs/root-cause-reports/fix-process/DD-MM_HH-MM_{название-отчёта}.md
```
Запиши результаты Фазы 0–1. Этот файл обновляется после каждого шага.

### Шаг 2.2: Определи маршрут для каждого изменения

#### Маршрутизация по тегам

**[SKILL]** — обновление скилла Claude:
- Запусти `/update-skill` через агента
- Передай: имя скилла, что изменить, цитату из root-cause отчёта с контекстом
- Один скилл = один агент

**[DOC]** — обновление или создание документации:
- Существующий документ → `/update-docs`
- Новый документ → `/create-doc`
- Спецификация модуля (development/specs/) → прямое редактирование агентом
- Один документ = один агент

**[CODE]** — изменение в коде:

- **Простое точечное исправление** (добавить параметр, изменить значение, поправить логику в одной функции):
  Прямое изменение через агента без скилла. Передай агенту точные инструкции из root-cause отчёта: какой файл, какая функция, что изменить.

- **Новый тест или изменение теста** (добавить тест, обновить тест-кейс):
  Прямое изменение через агента. Передай: какой файл теста, что тестировать, ожидаемое поведение.

- **Существенное изменение модуля** (переработка логики, добавление нового функционала, изменение API модуля):
  Если изменение затрагивает спецификацию — сначала обнови спеку через [DOC] агента, затем запусти `/implement-module` для перереализации.

### Шаг 2.3: Запусти агентов

Правила запуска:

1. **Каждое изменение = отдельный агент.** Не мешай два изменения в одном — это усложняет отладку и отчётность.

2. **Параллельно** — если изменения независимы (разные файлы, разные скиллы):
   - Обновление скилла A и обновление скилла B → параллельно
   - Обновление документа и правка кода → параллельно
   - Два [CODE] изменения в разных файлах → параллельно

3. **Последовательно** — если есть зависимость:
   - Обновление спецификации → реализация модуля (для одного модуля)
   - Изменение кода → добавление теста на это изменение (если тест зависит от нового API)

4. **Модель: Opus.** Все агенты запускаются с `model: "opus"`.

5. **Промпт агента** должен содержать:
   - Конкретную задачу (цитата рекомендации из отчёта)
   - Контекст проблемы (из секции «Корневая причина» отчёта)
   - Путь к файлу, который нужно изменить
   - Какой скилл использовать (вызвать через Skill tool), если применимо

### Если скилл не подходит

Если ни один скилл не покрывает нужное изменение — **ОСТАНОВИСЬ и спроси пользователя:**

```
Для изменения #{N} ({описание}) подходящий скилл не найден.
Варианты:
A) Сделать вручную (прямое редактирование агентом)
B) Создать новый скилл для этого типа задач
C) Пропустить это изменение
```

Не принимай решение сам — дай выбор пользователю.

## Фаза 3: Отчётность

После завершения каждого агента **сразу обновляй** отчёт в fix-process/. Не жди завершения всех — записывай результат каждого по мере готовности.

Итоговый формат отчёта:

```markdown
# Fix Process: {название root-cause отчёта}

**Дата запуска:** DD-MM-YYYY HH:MM
**Исходный отчёт:** {полный путь к root-cause отчёту}

## Фаза 1: Разбор и верификация

- **Всего рекомендаций:** N
- **Принято:** N
- **Отклонено:** N

### Детали
1. **[TAG] {описание}** — ПРИНЯТО/ОТКЛОНЕНО
   - Обоснование: {текст}

## Фаза 2: Исправления

### Изменение 1: {описание}
- **Тег:** [CODE/SKILL/DOC]
- **Скилл:** update-skill / update-docs / direct edit / ...
- **Статус изменения:** УСПЕХ / ОШИБКА / ТРЕБУЕТ ВНИМАНИЯ
- **Что сделано:** {краткое описание выполненной работы}
- **Файлы затронуты:** {список путей к изменённым файлам}

### Изменение 2: ...

## Фаза 3: Итоги

- **Успешно исправлено:** N из N
- **Ошибки:** N (если есть — описание каждой)
- **Требуют ручного вмешательства:** N (если есть — что именно)
```

---

## Фаза 4: Верификация

После завершения всех агентов проверь:

1. **Полнота** — все ли ПРИНЯТЫЕ изменения выполнены? Сравни список из Фазы 1 со статусами в Фазе 2. Если что-то пропущено — сообщи пользователю.

2. **Согласованность** — не противоречат ли изменения друг другу? Если один агент обновил что-то в скилле, а другой — в документе, убедись что значения совпадают. Прочитай ключевые файлы и сверь.

3. **Тесты** — если среди изменений были [CODE] правки в `src/claude_manager/` или `tests/`, запусти тесты:
   ```bash
   python -m pytest tests/ -v
   ```
   Убедись что все тесты проходят. Если тесты падают — исправь и перезапусти (до 3 попыток).

4. **Обнови отчёт** — запиши финальные статусы в отчёт fix-process/.

5. **Перемести отчёт в resolved/**:
   Если все исправления успешны, автоматически перемести исходный root-cause отчёт в `development/docs/root-cause-reports/resolved/` (создай папку `resolved/`, если не существует). Запиши в отчёт fix-process, что исходный отчёт перемещён. Сообщи пользователю.

   Если хотя бы одно изменение не выполнено — отчёт остаётся открытым. Сообщи пользователю: "Изменения применены частично — отчёт остаётся открытым."

---

## Фаза 4.5: Коммит изменений

Этот шаг обязателен. Без коммита изменения существуют только в рабочей копии и могут потеряться.

### Шаги

1. **Проверь что есть что коммитить** — запусти `git status`. Если нет изменённых файлов — пропусти эту фазу.

2. **Добавь изменённые файлы** — `git add` для каждого файла, затронутого агентами в Фазе 2. Не используй `git add -A` — добавляй только файлы, которые были целенаправленно изменены (список есть в отчёте fix-process/).

3. **Создай коммит** с сообщением формата:
   ```
   fix(root-cause): применены исправления из отчёта {краткое-название-отчёта}
   ```

4. **Сообщи пользователю** что коммит создан и какие файлы в него вошли.

---

## Фаза 5: Сессионный отчёт

Последний шаг — зафиксировать всю проделанную работу для будущих сессий.

Вызови скилл `/session-report` через Skill tool. Скилл сам проанализирует весь контекст диалога и создаст отчёт в `development/docs/session-reports/`.

## Важные принципы

- **Прагматизм превыше всего.** Root-cause анализ часто рекомендует больше, чем реально нужно. Цель — минимум изменений с максимальным эффектом. Если 3 точных изменения решают проблему — не нужно делать ещё 5 «на всякий случай».

- **Не переследуй.** Root-cause отчёт уже содержит расследование. Если в нём указаны конкретные файлы и строки — используй их, не запускай повторный анализ.

- **Один агент = одна задача.** Не смешивай несколько изменений в одном агенте. Это упрощает отчётность и позволяет параллелить.

- **Формат дат:** `DD-MM_HH-MM` для имён файлов, `DD-MM-YYYY HH:MM` для заголовков документов.

- **Спроси, если непонятно.** Если рекомендация из отчёта непонятна, двусмысленна или не соответствует текущему коду (мог измениться после анализа) — спроси пользователя, а не додумывай.
