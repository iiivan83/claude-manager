# Обновление существующих документов

Ты обновляешь существующие документы проекта, интегрируя новую информацию из двух возможных источников:
- **Структурированный текст** — пользователь вставил готовый документ, таблицу, конфиг
- **Беседа/переписка** — пользователь вставил лог чата или описал что-то в свободной форме

Эти два типа входных данных обрабатываются по-разному — подробности ниже.

## Шаг 1: Определи тип входных данных

**THINK HARD** — от правильного определения типа зависит, сколько информации сохранится.

### Структурированный текст
Признаки: заголовки, списки, таблицы, конфиги, чёткая организация, технические данные без диалога.

Правило: **НИЧЕГО не упускать.** Каждый факт, число, имя, параметр — всё должно попасть в документ. Это уже отфильтрованная информация, пользователь хочет сохранить её целиком.

### Беседа/переписка
Признаки: реплики разных людей, вопросы-ответы, обсуждение вариантов, пробы и ошибки.

Правило: **Извлечь только ценное.** Из переписки нужны:
- Финальные решения и выводы (не промежуточные варианты)
- Конкретные факты: числа, адреса, имена сервисов, команды
- Настройки и конфигурации, которые реально используются
- Проблемы и их решения (если нашли ответ)

Убрать:
- Промежуточные рассуждения и гипотезы, которые не подтвердились
- Вопросы без ответов
- Повторения одного и того же разными словами
- Приветствия, благодарности, «ок», «понял» и прочую воду
- Тестирование гипотез, которые оказались неверными

## Шаг 2: Найди ВСЕ целевые документы

**THINK HARD** — это критически важный шаг. Частая ошибка: найти один «главный» документ и забыть про остальные.

Прочитай индекс документации `development/docs/docs-index.md`. Затем **пройди по КАЖДОМУ документу из индекса** и задай вопрос: «Есть ли во входных данных информация, которая относится к этому документу?»

Составь таблицу. Для каждого документа определи его тип: **справочник** (хранит факты) или **гайд** (описывает процесс по шагам).

```
| Документ | Тип | Новые факты? | Шаг процесса? (только для гайдов) | Что добавить |
|----------|-----|-------------|----------------------------------|-------------|
| migration_strategy.md | справочник | нет | — | — |
| kubernetes_cluster_config.md | справочник | да | — | SSL, DNS |
| namespace_creation_guide.md | гайд | нет | да — SSL является шагом создания namespace | шаг: настройка DNS и SSL-сертификата |
| ... | ... | ... | ... | ... |
```

**Правило для гайдов (ОБЯЗАТЕЛЬНО):**

Для каждого документа с типом «гайд» ты ОБЯЗАН заполнить колонку «Шаг процесса?». Для этого:
1. Определи: какой процесс описывает этот гайд?
2. Является ли новая информация **шагом этого процесса**?
3. Если да → документ нужно обновить, даже если «Новые факты?» = нет

Документ нужно обновить если «Новые факты?» = да **ИЛИ** «Шаг процесса?» = да.

Формат шага в гайде: кратко, но самодостаточно. Читатель должен смочь выполнить шаг, не переходя в другой документ. Стиль — как у остальных шагов этого гайда.

### Правило множественного обновления

Большинство задач затрагивают **несколько документов**. Типичные комбинации:
- **Инфраструктурные задачи** (деплой, K8s, SSL, CI/CD) → `kubernetes_cluster_config.md` (текущее состояние, статусы) + `django_project_setup.md` (как настроено, как воспроизвести)
- **Миграция модуля** → `migration_strategy.md` (прогресс) + `codebase-analysis-auto.md` (зависимости) + `django_project_setup.md` (новое приложение)
- **Отладка/фиксы** → документ по теме + `kubernetes_cluster_config.md` (если касается инфраструктуры)

**Различай два типа информации:**
- **Факт-статус** («SSL работает», «отчёт проверен») → документ текущего состояния (`kubernetes_cluster_config.md`)
- **Факт-инструкция** («SSL настроен через cert-manager, а не Cloudflare Proxy», «авторизация через curl требует HTTPS из-за secure cookies») → документ настройки/воспроизведения (`django_project_setup.md`, `migration_strategy.md`)

Оба типа фактов нужно сохранить, каждый в свой документ.

### Итог шага

Покажи пользователю список документов, которые планируешь обновить, и что именно в каждый добавишь. Попроси подтверждение.

Если тема не совпадает ни с одним существующим документом из индекса — спроси у пользователя:
> «Я не нашёл существующий документ по этой теме. Создать новый документ или добавить информацию в один из существующих: [список]?»

## Шаг 3: Прочитай ВСЕ целевые документы

Прочитай **каждый** целевой документ целиком с помощью Read (читай параллельно через несколько вызовов Read). Для каждого пойми:
- Какие разделы (заголовки H2, H3) уже есть
- Какой стиль форматирования используется (таблицы, списки, код-блоки)
- Где логически должна находиться новая информация
- Какая информация из входных данных уже есть в этом документе (чтобы не дублировать)

Также **бегло просмотри остальные документы** из `development/docs/`, чтобы убедиться, что не дублируешь информацию.

## Шаг 4: Интегрируй информацию

Новую информацию нужно **вплести в существующую структуру**, а не просто дописать в конец файла. Это критически важно для поддержания качества документации.

### Правила интеграции

1. **Если в документе уже есть раздел по теме** — дополни его. Например, если добавляешь новый сервис в Kubernetes, добавь строку в существующую таблицу сервисов.

2. **Если раздела нет, но тема вписывается в логику документа** — создай новый раздел (H2 или H3) в подходящем месте. Не в конце файла, а там, где он логически должен быть.

3. **Если новая информация противоречит старой** — замени старую на новую. Предупреди пользователя: «Заменил [старое значение] на [новое] в разделе [X], потому что новые данные более актуальны.»

4. **Сохраняй стиль документа.** Если в документе таблицы — добавляй в таблицы. Если списки — добавляй в списки. Не меняй формат существующего контента.

5. **Обнови дату** в разделе «Контекст», если он есть в документе, на текущую дату.

### Краткость — критически важное правило

**THINK HARD** — документация должна быть справочником, а не учебником.

Пиши **факты**, а не **инструкции и туториалы**. Документ — это место, куда заглядывают за конкретной информацией, а не руководство для новичка.

**Правила:**
- Один факт = одна строка или одна короткая строка в таблице/списке
- НЕ раздувай простой факт в целый раздел с примерами
- НЕ добавляй примеры команд, если факт и так понятен
- НЕ объясняй, что означает факт и зачем он нужен — это знает тот, кто читает документ
- НЕ пиши «это позволяет...», «благодаря этому...», «это значит, что...» — это вода

**Плохо** (раздуто):
```
## Доступ Claude к кластеру
Claude может напрямую из терминала выполнять любые kubectl-команды...
Примеры того, что Claude может сделать:
- Проверить состояние подов: `kubectl get pods -n sueda1ru-dj`
- Посмотреть логи: `kubectl logs deployment/...`
(ещё 4 примера)
Это позволяет Claude самостоятельно диагностировать проблемы...
```

**Хорошо** (кратко):
```
- Claude имеет доступ к kubectl (kubeconfig настроен на локальной машине)
```

**Тест на краткость:** перед добавлением каждого блока спроси себя — «Можно ли выразить это одной строкой?» Если да — напиши одну строку.

### Антидублирование — ключевое правило

Каждый факт в документации проекта должен существовать **ровно в одном месте**. Перед добавлением каждого пункта задай себе два вопроса:

1. **Эта информация уже есть в целевом документе?** → Не добавляй, только обнови если значение изменилось.
2. **Эта информация уже есть в другом документе проекта?** → Не добавляй. Если нужна ссылка — добавь «см. `имя_файла.md`», но не копируй содержимое.

Пример: пользователь даёт информацию про новый сервис в Kubernetes и упоминает, что он мигрирован из PHP-модуля. Сервис → добавляем в `kubernetes_cluster_config.md`. Факт миграции модуля → это тема `migrated-modules.md`, туда не пишем из этого обновления (только если пользователь явно попросил обновить оба документа).

**ИСКЛЮЧЕНИЕ — документы-гайды:**

Правило «один факт = одно место» НЕ применяется к **шагам процесса** в документах-гайдах. Проверка гайдов выполняется в Шаге 2 (колонка «Шаг процесса?»). Если шаг найден — добавь его кратко и самодостаточно, в стиле остальных шагов гайда.

### Релевантность — что НЕ добавлять

Добавляй в документ только то, что **соответствует теме и назначению** этого конкретного документа. Каждый документ имеет чёткую область ответственности (описана в `docs-index.md`).

Не добавляй:
- Информацию, которая по теме относится к другому документу
- Общие рассуждения, не несущие конкретных фактов
- Информацию, которая уже есть в этом или другом документе проекта (кроме шагов в гайдах — см. Шаг 2)
- Вводные фразы вроде «Ниже приведена обновлённая информация»

Если из входных данных только часть относится к целевому документу — добавь только эту часть. Остальное или игнорируй (если оно уже задокументировано), или предложи пользователю обновить соответствующий документ отдельно.

### Прочие ограничения

- Не менять структуру или порядок существующих разделов без необходимости
- Не удалять существующую информацию, если она не устарела и не противоречит новой

## Шаг 5: Верификация — сверка с источником

**THINK HARD** — это критически важный шаг.

Перечитай исходный материал и обновлённый документ. Проверь:

1. **Полнота** — вся ценная информация из источника попала в документ?
   - Для структурированного текста: ВСЁ должно быть перенесено
   - Для переписки: все выводы и факты извлечены
2. **Точность** — числа, адреса, имена, команды перенесены без искажений?
3. **Нет выдумок** — ничего не добавлено от себя?
4. **Интеграция** — новая информация вписана в структуру, а не просто дописана в конец?
5. **Целостность** — существующая информация не повреждена?
6. **Нет дублирования** — добавленная информация не повторяет то, что уже есть в этом или других документах проекта? (кроме шагов в гайдах — см. Шаг 2)
7. **Релевантность** — всё добавленное относится к теме именно этого документа, а не другого?

Выведи краткий отчёт:
```
Верификация:
- Полнота: ✅/⚠️ (что пропущено, если есть)
- Точность: ✅/⚠️
- Нет выдумок: ✅/⚠️
- Интеграция: ✅/⚠️
- Целостность: ✅/⚠️
- Нет дублирования: ✅/⚠️
- Релевантность: ✅/⚠️
```

## Шаг 6: Проверка названия документа

После обновления содержимого документа его название (заголовок внутри файла и имя файла) может перестать отражать суть. Например, документ назывался `namespace_creation_guide.md`, а после обновления он стал описывать весь процесс деплоя в Kubernetes — название уже не покрывает содержимое.

Для каждого обновлённого документа задай себе вопрос: **«Если бы я создавал этот документ с нуля с его текущим содержимым — я бы дал ему такое же название?»**

Если нет — предложи пользователю новое название файла и заголовок. Дождись подтверждения.

### При переименовании — обнови все ссылки

Переименование файла без обновления ссылок ломает навигацию по документации. Поэтому после переименования нужно найти и обновить все упоминания старого имени файла:

1. **Найди все упоминания** старого имени файла во всех документах проекта — `docs-index.md`, все файлы в `development/docs/`, `CLAUDE.md` и любые другие места, где может быть ссылка.
2. **Замени** каждое упоминание на новое имя файла.
3. **Контрольная проверка** — после всех замен пройди по всем документам ещё раз с помощью Grep и убедись, что старое имя файла больше нигде не встречается. Это важно, потому что ссылки легко пропустить (например, в примерах, комментариях или таблицах).

Покажи пользователю список всех мест, где была обновлена ссылка.

## Шаг 7: Проверка на стратегическую важность (обновление CLAUDE.md)

**THINK HARD** — этот шаг выполняется ТОЛЬКО в исключительных случаях. Порог очень высокий.

После обновления документа задай себе вопрос: **«Есть ли в добавленной информации что-то, без чего Claude будет работать НЕПРАВИЛЬНО почти в каждой сессии?»**

### Критерии для добавления в CLAUDE.md (должны выполняться ВСЕ три):

1. **Фундаментальность** — это меняет базовые правила работы с проектом (например: сменился фреймворк, изменилась структура БД, появился новый обязательный процесс для всего кода)
2. **Универсальность** — это актуально для 80%+ задач, а не для одного модуля или одной темы
3. **Невыводимость** — это НЕЛЬЗЯ узнать, просто прочитав код или документы в `development/docs/` — нужно знать заранее, до начала работы

### Примеры того, что НАДО добавить в CLAUDE.md:
- Смена основного фреймворка или языка (например: «Бэкенд теперь на Django, а не на Yii»)
- Принципиальное изменение архитектуры (например: «Все API теперь идут через API Gateway»)
- Новое глобальное правило для всего кода (например: «Все новые модули пишутся только на Python»)

### Примеры того, что НЕ НАДО добавлять в CLAUDE.md:
- Детали конкретного модуля — для этого есть документы в `development/docs/`
- Решения по конкретной задаче — для этого есть документ задачи
- Прогресс миграции — для этого есть `migrated-modules.md`
- Всё, что можно узнать из кода или существующих документов

### Если информация подходит:

1. Прочитай текущий `CLAUDE.md`
2. Найди подходящий раздел (или создай новый, если нет подходящего)
3. Добавь **1-2 строки** — максимально кратко, только суть
4. Убедись, что не дублируешь то, что уже есть в CLAUDE.md
5. Спроси у пользователя подтверждение: «Эта информация настолько важна, что я хочу добавить её в CLAUDE.md (файл, который читается в каждой сессии): [текст]. Добавить?»

**Важно:** если сомневаешься — НЕ добавляй. Лучше пропустить, чем засорить CLAUDE.md. Этот файл должен оставаться компактным.

## Шаг 8: Итоговый отчёт

Для КАЖДОГО обновлённого документа:

```
📄 Обновлён: development/docs/имя_файла.md
📝 Что добавлено: [краткое описание]
🔍 Верификация: пройдена
```

В конце:
```
📌 CLAUDE.md: обновлён / не требуется
⚠️ Документы НЕ обновлены (информация не найдена): [список документов из индекса, для которых не нашлось новой информации]
```

Если обновлён только один документ, а входные данные содержали инфраструктурную или кросс-доменную информацию — **перепроверь**, не упущен ли второй документ. Это частая ошибка.

## Шаг 9: Запусти скилл `session-report` через Skill tool.
