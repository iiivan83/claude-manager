Переименование любой сущности проекта с автоматическим исправлением всех ссылок на неё.

Ссылки могут прятаться в неожиданных местах — markdown-документы, frontmatter, конфиги, индексы, CLAUDE.md, MEMORY.md, другие скиллы. Поэтому после переименования скилл прогоняет поиск по всем текстовым файлам в целевых директориях и исправляет найденное. Затем проверяет ещё раз — потому что исправление одной ссылки иногда порождает новую (например, при обновлении индекса появляется новый путь, который тоже нужно обновить).

## Вход

Аргументы: $ARGUMENTS

Из аргументов и контекста беседы извлечь:
- **Старое имя** — что переименовать (путь к файлу/папке, имя скилла и т.д.)
- **Новое имя** — как должно называться после переименования

Если из контекста непонятно что именно переименовать или как — спросить пользователя.

## Область действия

Поиск ссылок ведётся только в этих директориях:
- `dev/` — документация, спецификации, отчёты сессий, скрипты
- `.claude/` — скиллы, память, конфиги Claude

Бинарные и скомпилированные файлы пропускаются (`.min.css`, `.min.js`, картинки, архивы, `__pycache__/`).

## Алгоритм

### 1. Определить тип сущности

- **Файл** — путь к конкретному файлу
- **Папка** — путь к директории
- **Скилл** — имя скилла (директория в `.claude/skills/`)
- **Другое** — любая именованная сущность (спецификация, документ и т.д.)

Если пользователь указал неполный путь или нечёткое имя — найти сущность через Glob и уточнить при неоднозначности.

### 2. Переименовать сущность

Использовать `git mv` для переименования — это сохраняет историю файла в git. Если git недоступен или файл не отслеживается git-ом, переименовать через `mv`.

Для скиллов: переименовать директорию `.claude/skills/<старое-имя>/` → `.claude/skills/<новое-имя>/`, а также обновить поле `name:` в frontmatter файла SKILL.md внутри скилла.

### 3. Поиск и замена ссылок в содержимом файлов

Главная задача — найти и исправить упоминания старого имени **внутри текстовых файлов**: в ссылках, путях, конфигах, frontmatter, индексах, документации. Поиск по именам файлов уже выполнен на шаге 2, здесь фокус именно на содержимом.

Через Grep искать **точное совпадение** старого имени по содержимому всех текстовых файлов в `dev/` и `.claude/`:

1. Найти все файлы, содержащие старое имя
2. Для каждого найденного файла — прочитать, заменить старое имя на новое через Edit
3. Записать: какой файл, сколько замен

### 4. Повторные проверки (цикл до полной чистоты)

После каждого прохода исправлений — снова запустить поиск старого имени по содержимому файлов в тех же директориях.

- **Ничего не найдено** → задача выполнена, перейти к отчёту
- **Найдены ссылки** → исправить и повторить проверку

Цикл продолжается до тех пор, пока поиск не вернёт 0 результатов. Это важно, потому что исправление одного файла может обнажить или создать новые ссылки (например, обновление индекса может содержать старое имя в другом контексте). Количество проходов не ограничено — скилл работает до полной чистоты.

### 5. Отчёт

Вывести итог:

```
## Результат переименования

**Сущность:** <тип> `<старое-имя>` → `<новое-имя>`

**Затронутые файлы:**
- `путь/к/файлу.md` — N замен
- `путь/к/другому.md` — N замен

**Проходов проверки:** M
**Всего замен:** N

(если есть неисправленные)
**Не удалось исправить:**
- `путь/к/файлу` — причина
```

## Важные детали

- Искать именно точное совпадение строки, а не подстроку. Например, при переименовании `review` не трогать `review-code` или `review-migration-docs`.
- При замене в markdown-ссылках вроде `[текст](путь/старое-имя.md)` — обновлять и путь, и текст ссылки если текст совпадает со старым именем.
- Если переименовывается папка — обновлять все пути, которые содержат эту папку как часть пути.
- Не трогать файлы за пределами `dev/` и `.claude/`.
