# Генерация отчёта о сессии

Ты создаёшь отчёт о всей работе, проделанной в текущем диалоге. Отчёт предназначен
для Claude — чтобы в следующей сессии быстро понять что было сделано, какие решения
приняты и что осталось незавершённым.

Единственный источник данных — **контекст текущего диалога**. Ты помнишь все сообщения,
все tool calls, все изменения файлов, все обсуждения. Извлекай информацию оттуда.
Не запускай git-команды для сбора данных — всё уже есть в твоей памяти диалога.

Аргументы пользователя (если есть): $ARGUMENTS

## Принцип: сухие факты для машины

Отчёт читает Claude, не человек. Значит:
- Никакой воды, вежливых оборотов, вступлений
- Каждая строка несёт новый факт
- С первых строк понятно что изменилось
- Конкретные пути, имена, числа — без «некоторые файлы» или «были внесены изменения»

## Шаг 1: Извлеки факты из диалога

Пройдись по всему диалогу и выдели:

1. **Файлы** — какие файлы создавались, редактировались, удалялись (из tool calls: Write, Edit, Bash)
2. **Что именно менялось** — конкретные изменения в каждом файле (из содержимого tool calls)
3. **Коммиты** — если в диалоге были `git commit`, запомни хэши и сообщения
4. **Решения** — что обсуждалось, какие варианты рассматривались, что выбрали и почему
5. **Проблемы** — что не получилось, какие ошибки возникали, как решились
6. **Незавершённое** — о чём говорили но не сделали, что отложили на потом
7. **Команды** — какие bash-команды запускались (установка пакетов, сборка, тесты и т.д.)

## Шаг 2: Сформируй отчёт

Используй **строго** этот шаблон (пустые секции пропускай):

```markdown
# Сессия DD-MM: краткое-описание-главной-темы

## Резюме

1-2 предложения: главная цель сессии и достигнута ли она.

## Изменённые файлы

| Файл | Действие | Что сделано |
|------|----------|-------------|
| `путь/к/файлу` | создан/изменён/удалён | конкретное описание изменений |

## Коммиты

- `hash` — сообщение коммита

## Выполненные команды

Только значимые команды (установка зависимостей, настройка инфраструктуры, запуск тестов).
Не включай тривиальные вроде `ls`, `cat`, `git status`.

- `команда` — зачем выполнялась

## Решения

- **Решение**: краткое описание. **Причина**: почему так решили.

## Проблемы и решения

- **Проблема**: описание. **Решение**: как решили.

## Незавершённое

- [ ] Что осталось сделать и почему не завершено

## Контекст для следующей сессии

Что нужно знать Claude чтобы продолжить работу.
Текущее состояние, зависимости, ограничения, на чём остановились.
```

### Правила формирования

- **Пустые секции пропускай** — не пиши секцию если в ней нечего написать
- **Таблица файлов** — основная ценность отчёта, заполняй максимально подробно
- **Решения** — только те, которые влияют на дальнейшую работу
- **TODO** — только конкретные задачи, не абстрактные пожелания
- **Не выдумывай** — если чего-то не было в диалоге, не пиши

## Шаг 3: Определи имя файла

### Получение реального времени (ОБЯЗАТЕЛЬНО)

Claude не знает текущее время — его нужно узнать через системную команду.
Выполни команду:

```bash
date +"%H-%M"
```

Используй результат этой команды как время в имени файла. **Никогда не угадывай
и не придумывай время** — только результат команды `date`.

### Формат имени

Формат: `ЧЧ-ММ_краткое-описание.md`

Где `ЧЧ-ММ` — реальное время из команды `date` выше.

Имя файла начинается с времени создания отчёта (часы-минуты), потому что за день
может быть несколько сессий, и по времени легко восстановить хронологию работы.
Дату отдельно не добавляй — она уже в имени папки.

Краткое описание — 2-4 слова на английском, отражающие главную тему сессии.

### Суффикс `-planning` для стратегических сессий

Если основное содержание сессии — обсуждение дальнейших шагов, стратегии, планирование
(а не выполнение конкретной работы вроде написания кода, исправления багов, настройки
инфраструктуры), добавь `-planning` в конец описательной части имени файла.
Это помогает при просмотре папки сразу отличить сессии с реальными изменениями от сессий
с планированием, где главная ценность — принятые решения и намеченные шаги.

Примеры:
- `14-30_setup-k8s-deploy.md` — реальная работа
- `09-15_courier-module-migration.md` — реальная работа
- `17-45_fix-auth-bug.md` — реальная работа
- `11-00_create-skills.md` — реальная работа
- `16-20_migration-strategy-planning.md` — обсуждение стратегии
- `10-00_next-modules-planning.md` — выбор и планирование следующих шагов

Если в папке дня уже есть отчёт с таким именем — добавь суффикс: `ЧЧ-ММ_topic-2.md`

## Шаг 4: Сохрани отчёт

1. Определи текущую дату, выполнив команду `date +"%d-%m"` (не угадывай дату — бери из системы)
2. Папка для сохранения: `development/docs/session-reports/DD-MM/` (сессионные отчёты — хронологические артефакты, они всегда сохраняются в session-reports/, а не в methods/)
3. Создай директорию если нет: `mkdir -p {выбранная папка}`
4. Проверь нет ли уже отчёта с таким именем в этой папке (через Glob)
5. Сохрани файл через Write в `{выбранная папка}/имя-файла.md`

Отдельные отчёты в development/docs/docs-index.md  **не добавляй** — достаточно ссылки на папку.

## Шаг 5: Итог

Кратко сообщи пользователю:
```
Отчёт сохранён: development/docs/session-reports/DD-MM/имя-файла.md
```
