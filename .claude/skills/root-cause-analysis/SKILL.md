# Глубокий анализ первопричины проблемы

Перед ответом включи максимально глубокий режим размышлений (extended thinking).

Ты — следователь-аналитик. Твоя задача — докопаться до самой глубокой, настоящей
причины проблемы. Не остановиться на первом подозрительном месте, а пройти всю
цепочку до конца. Потом проверить себя и пройти ещё раз.

Аргументы пользователя: $ARGUMENTS

## Принцип: копай до дна

Поверхностная причина — это не ответ. Если ошибка в том, что пакет не установлен —
нужно понять, почему его не установили. Если спецификация пропустила зависимость —
нужно понять, почему скилл, создававший спецификацию, не проверил зависимости.
Каждый ответ «потому что X» порождает следующий вопрос «а почему X?».

Останавливайся только когда доходишь до:
- Пробела в процессе (скилл/документация/чеклист не покрывает этот случай)
- Человеческого решения (кто-то сознательно решил не делать X)
- Системного ограничения (технология не позволяет)

## Фаза 1: Сбор контекста проблемы

### 1.1 Пойми проблему

Прочитай описание проблемы от пользователя. Если пользователь дал путь к отчёту
сессии — прочитай этот отчёт целиком. Если описал текстом — зафиксируй:
- Что именно сломалось (симптом)?
- Когда это проявляется?
- Какой ожидаемый результат vs фактический?

### 1.2 Найди связанные отчёты сессий

Используй инструмент Grep, чтобы найти все отчёты в `development/docs/session-reports/`,
которые упоминают ключевые слова проблемы. Часто проблема обсуждалась в нескольких
сессиях, и в ранних сессиях могут быть подсказки о корневой причине.

Прочитай ВСЕ найденные отчёты — не только последний.

## Фаза 2: Глубокий анализ документации

Прочитай индекс документации `development/docs/docs-index.md` и определи, какие
документы могут быть связаны с проблемой.

Обязательно проверь:

- **Спецификации модулей** — если проблема связана с конкретным модулем, найди
  его спецификацию в `development/specs/`.
  Проверь: описана ли в спеке работа с тем, что сломалось?

- **BRD (пользовательские сценарии)** — `development/docs/brd-user-journeys.md`:
  проверь ожидаемое поведение по описанию сценария (CJM).

- **CLAUDE.md** — главный документ проекта с архитектурными принципами и правилами:
  не нарушены ли архитектурные решения?

- **Deployment guide** — `development/docs/deployment-guide.md`: если проблема
  связана с инфраструктурой запуска бота.

- **Состояние пайплайна** — `pipeline-state.json`: показывает на какой фазе пайплайна
  находится проект, какие задачи завершены, а какие нет. Если проблема возникла в ходе
  пайплайна — этот файл подскажет, на каком этапе что-то пошло не так.

- **Отчёты ревью кода** — `development/docs/review-report_*.md`: содержат уже найденные
  и исправленные проблемы. Возможно, текущая проблема связана с тем, что ревью пропустило
  или исправило не до конца.

- **Чеклисты ревью** — `development/docs/review-checklists.md`: критерии, по которым
  проверялся код. Если проблема не покрыта чеклистом — это пробел в процессе ревью.

- **Отчёт валидации BRD** — `development/docs/brd-validation-report_*.md`: принятые
  решения по спорным вопросам при валидации BRD. Неверное решение на этапе валидации
  может каскадировать в код.

- **Результаты тестирования** — `development/docs/testing/`: результаты E2E тестов
  и пользовательских сценариев. Если проблема не была поймана тестами — важно понять
  почему: тест-план неполный, сценарий пропущен, или тест был зелёный, но проверял
  не то.

Для каждого прочитанного документа запиши: что нашёл и что НЕ нашёл (отсутствие
информации тоже важно).

## Фаза 3: Анализ скиллов

Это критически важная фаза. Проблемы часто возникают из-за того, что скилл,
создававший код или документацию, пропустил какой-то аспект.

### 3.1 Определи связанные скиллы

Подумай: какие скиллы участвовали в создании кода/документации, где возникла проблема?

Типичные связи:
- Код модулей → `implement-module`
- Спецификации модулей → `spec-module`
- Декомпозиция на модули → `brd-decompose`
- Валидация BRD → `validate-brd`
- Ревью кода → `review-code`
- Интеграционные тесты → `test-integration`
- E2E тесты → `test-e2e`
- Сценарии пользовательского тестирования → `create-user-test-scenarios`
- Прогон пользовательских тестов → `run-user-testing`
- Документация → `create-doc`, `update-docs`, `update-project-docs`
- Отчёты сессий → `session-report`
- Настройка инфраструктуры → `project-setup`
- Обновление скиллов → `update-skill`
- Оркестрация → `pipeline-run`

### 3.2 Прочитай каждый связанный скилл

Для каждого скилла из `.claude/skills/{skill-name}/SKILL.md`:

1. Прочитай SKILL.md **целиком**
2. Проверь: есть ли в скилле пункт, который должен был предотвратить эту проблему?
3. Если пункт есть — почему он не сработал? (неточная формулировка? пропущен случай?)
4. Если пункта нет — это пробел в скилле, который нужно зафиксировать

### 3.3 Запиши результат по каждому скиллу

Для отчёта запиши в формате:
- `skill-name` — есть/нет пункт про [что именно]. Вывод: [скилл пропустил X / скилл корректен / формулировка неточна]

## Фаза 4: Анализ кода

Найди и прочитай код, связанный с проблемой. Код проекта находится в `src/claude_manager/`.

### 4.1 Найди точное место ошибки

- Используй grep/glob чтобы найти файлы по ключевым словам из описания проблемы
- Прочитай каждый найденный файл целиком — не только строку с ошибкой, а весь контекст
- Проследи вызовы: кто вызывает проблемный код? Что вызывает он?

### 4.2 Сравни код со спецификацией

Если проблема в модуле, для которого есть спецификация — ищи её в
`development/specs/realized/` (реализованные спеки перемещаются в подпапку `realized/`
после имплементации модуля). Если спеки там нет — проверь корень `development/specs/`:
- Прочитай спецификацию модуля
- Сравни: что описано в спеке, но не реализовано в коде? Что реализовано не так?
- Расхождение спеки и кода = потенциальная корневая причина

### 4.3 Проверь инфраструктуру

Если проблема может быть инфраструктурной:
- Конфигурация: `.env.example`, `pyproject.toml`, `requirements.txt`
- Запуск: `watch_and_restart.sh`, `src/claude_manager/__main__.py`
- LaunchAgent: `~/Library/LaunchAgents/com.ivan.claude-manager.plist`

### 4.4 Проверь тестовую инфраструктуру

Если проблема проявляется в тестах или могла быть поймана тестами:
- `tests/conftest.py` — общие фикстуры (заготовки) для тестов. Неправильная фикстура
  может маскировать баги или создавать ложные срабатывания
- `tests/integration/` — интеграционные тесты, проверяющие совместную работу модулей.
  Если баг межмодульный, здесь может быть подсказка
- Юнит-тесты `tests/test_*.py` — по одному на каждый модуль из `src/`

### 4.5 Обрати внимание на типичные проблемные места проекта

В этом проекте есть несколько мест, где баги возникают чаще всего. При анализе стоит
проверить, не связана ли проблема с одним из них:

- **Координация watcher и handler** — `session_watcher.py` и `bot.py` (обработчик
  сообщений) координируются через механизм pause/resume, чтобы не дублировать ответы.
  Нарушение этой координации — частый источник проблем

- **Смена session_id** — Claude может вернуть новый session_id в любом событии.
  Все словари, где ключ — session_id, должны обновлять свои ключи при смене.
  Если хотя бы один словарь не обновился — состояние рассинхронизируется

- **Stream-json протокол** — общение с Claude Code CLI идёт через потоковый JSON
  (строки JSON через stdin/stdout). Ошибки парсинга, неожиданные типы событий или
  обрыв потока — потенциальные источники проблем

- **Разбивка длинных сообщений** — `message_splitter.py` разбивает ответы на части
  до 4096 символов (ограничение Telegram) с починкой незакрытых HTML-тегов.
  Ошибки в починке тегов могут ломать форматирование

- **Файловая блокировка** — `main.py` использует `fcntl.flock()` для защиты от
  запуска двух копий бота. Проблемы с блокировкой могут мешать запуску или
  приводить к зависанию

## Фаза 5: Построение цепочки причин

Теперь собери всё в цепочку. Начни с симптома и задавай вопрос «почему?» на каждом
шаге:

```
Симптом: [что видит пользователь]
  ↓ потому что
Причина 1: [что именно сломалось в коде]
  ↓ потому что
Причина 2: [почему это оказалось сломано]
  ↓ потому что
Причина 3: [что привело к этому]
  ↓ потому что
КОРНЕВАЯ ПРИЧИНА: [самая глубокая причина — пробел в процессе/решение/ограничение]
```

Цепочка должна содержать минимум 3 звена. Если меньше — ты не докопался до дна.

## Фаза 6: ВЕРИФИКАЦИЯ (обязательная)

После того как ты построил цепочку причин и нашёл корневую причину — ОСТАНОВИСЬ
и пройди весь цикл заново. Это не формальность, а защита от ошибок.

### 6.1 Перечитай ключевые документы

Вернись к самым важным документам и перечитай их свежим взглядом. Теперь, когда у тебя
есть гипотеза, ты можешь заметить то, что пропустил в первый раз.

### 6.2 Проверь выводы на противоречия

- Объясняет ли найденная корневая причина ВСЕ симптомы? Если нет — причина не корневая
- Нет ли фактов, которые противоречат твоей теории?
- Нет ли более простого объяснения?

### 6.3 Рассмотри альтернативные объяснения

Придумай минимум 2 альтернативные версии и объясни, почему они менее вероятны.
Это заставляет тебя критически оценить свою основную версию.

### 6.4 Запиши результат верификации

Для отчёта запиши:
- Что перепроверил
- Какие альтернативы рассмотрел и почему отверг
- Подтвердилась ли корневая причина или изменилась

## Фаза 6.5: Самостоятельная верификация решений

После двойной проверки выводов (Фаза 6) и перед формированием финального отчёта —
предварительные рекомендации проходят самостоятельную верификацию. Это нужно, потому что
анализатор первопричин может предложить решения, которые выглядят разумно, но
противоречат архитектуре проекта или являются перестраховкой. Критический взгляд
помогает отфильтровать лишнее до того, как рекомендации попадут в финальный отчёт.

### 6.5.1 Подготовка предварительных рекомендаций

Сформируй черновой список всех рекомендаций по исправлению и предотвращению.
Каждая рекомендация должна содержать:
- Описание: что именно сделать
- Тег: `[ARCHITECTURE]`, `[DOC]`, `[CODE]` или `[SKILL]`
- Затрагиваемые файлы
- Обоснование: зачем это нужно

### 6.5.2 Верификация каждой рекомендации

Проверь каждую рекомендацию по 5 критериям:
- **Архитектурное соответствие** — не нарушает ли принципы, описанные в `CLAUDE.md`
- **Соответствие BRD** — не противоречит ли `development/docs/brd-user-journeys.md`
- **Кодовые стандарты** — соответствует ли правилам кодирования из `CLAUDE.md`
- **Необходимость** — решает ли это реальную проблему, или это перестраховка
- **Осуществимость** — можно ли это сделать сейчас, нет ли блокирующих зависимостей

Для каждой рекомендации вынеси вердикт:
- **ОДОБРЕНО** — рекомендация проходит все проверки
- **ОДОБРЕНО С ЗАМЕЧАНИЯМИ** — в целом годится, но нуждается в корректировке
- **ОТКЛОНЕНО** — перестраховка, не соответствует архитектуре или невыполнимо

### 6.5.3 Фильтрация

1. **ОДОБРЕНО** → рекомендация идёт в финальный отчёт без изменений
2. **ОДОБРЕНО С ЗАМЕЧАНИЯМИ** → скорректировать рекомендацию по замечаниям
3. **ОТКЛОНЕНО** → убрать из финального отчёта или заменить альтернативным решением,
   которое устраняет ту же проблему другим способом

### 6.5.4 Фиксация результатов верификации

Для каждой рекомендации запиши:
- Вердикт
- Обоснование вердикта
- Была ли корректировка и какая

Эта информация войдёт в секцию «Верификация решений» финального отчёта.

## Фаза 7: Создание отчёта

Отчёт читает живой человек — владелец проекта, который понимает бизнес-логику, но не
разбирается в структуре кода. Поэтому отчёт пишется в том же стиле, в котором ты
общаешься с пользователем в диалоге — на «ты», дружелюбно, как будто объясняешь
другу за кофе, что произошло и почему.

### Стиль отчёта

**Язык:** Разговорный, дружелюбный. Никакого канцелярита и «птичьего языка».
Вместо «Обнаружен дефект маршрутизации запросов» — «Оказалось, бот отправлял
сообщение в Claude, но не дожидался ответа — процесс падал на полпути».

**Сущности кода:** При первом упоминании любой сущности (класс, метод, файл,
команда, настройка) поясни что это и за что отвечает. Пример: «`claude_runner.py`
(модуль, который запускает Claude Code CLI и читает его ответы)». При повторном
упоминании в том же отчёте пояснение не нужно.

**Технические термины:** Сначала понятное объяснение, потом техническое название
в скобках. Пример: «потоковый JSON-протокол (stream-json)» вместо
«stream-json protocol».

**Без таблиц:** Не используй markdown-таблицы — они ломаются при пересылке
в мессенджеры. Вместо таблиц — буллеты с **жирными** метками.

**Структура объяснения:** В каждом разделе придерживайся логики:
- Что проверялось / делалось — простыми словами
- Что пошло не так — объяснить проблему понятно
- Что работает нормально — хорошие новости для баланса
- Контекст — что ещё важно знать

**Так писать нельзя:**
- Сухие статусы без объяснений: «Шаг 1: FAIL, Шаг 2: ERROR»
- Сущности кода без пояснения: «Проверил `SessionManager.get_active_session()`»
- Аббревиатуры без расшифровки: «SM», «PM», «CR»
- Формальные обороты: «В ходе анализа было установлено, что...»

### 7.1 Формат отчёта

Используй **строго** этот шаблон (структура разделов обязательна, но текст внутри —
в дружелюбном разговорном стиле с пояснением каждой сущности):

```markdown
# Анализ первопричины: [краткое, понятное название проблемы]

**Дата анализа:** DD-MM-YYYY
**Статус:** [Первопричина найдена / Требуется дополнительное исследование]

## Симптом

Расскажи простыми словами, что именно сломалось — так, чтобы человек без знания кода
понял ситуацию. Каждую сущность кода поясняй при первом упоминании. Пример хорошего
описания: «Бот перестал отвечать на сообщения в Telegram — пишешь ему, а он молчит,
хотя процесс запущен и ошибок в логах нет».

## Цепочка причин

Пошаговая цепочка от симптома до корневой причины. Каждое звено — понятным языком,
с пояснением технических деталей в скобках:

1. **[Симптом простыми словами]** → потому что →
2. **[Промежуточная причина — с пояснением что это]** → потому что →
3. **[Ещё глубже — что именно пошло не так]** → потому что →
4. **[КОРНЕВАЯ ПРИЧИНА — почему так получилось]**

## Корневая причина

Расскажи подробно, в чём корень проблемы — как будто объясняешь коллеге, который
знает продукт, но не лезет в код. Укажи конкретные файлы и строки, но поясни что
каждый файл делает и за что отвечает. Объясни почему именно это — корень, а не
промежуточное звено.

## Почему проблема не была предотвращена
```

### 7.1.1 Обязательная проверка: нужно ли обновить CLAUDE.md?

После написания раздела "Рекомендации по предотвращению" в отчёте — ОСТАНОВИСЬ и ответь
на вопрос: **является ли корневая причина отсутствием архитектурного правила или принципа?**

Признаки архитектурной корневой причины:
- Проблема повторяется в разных модулях
- Агент принял решение, которое не описано ни в одном документе (и выбрал неверно)
- Несколько скиллов пришлось обновить одной и той же "заплаткой"
- Вопрос "когда/зачем/нужно ли" не имеет ответа в `CLAUDE.md`

**Если ДА — корневая причина архитектурная:**

1. Сформулируй ПРИНЦИП (не заплатку):
   - Плохо: "Добавить Step 4 проверки в implement-module"
   - Хорошо: "Watcher и handler координируются через pause/resume. Записать в CLAUDE.md"

2. Укажи ТОЧНОЕ место в `CLAUDE.md`, куда вставить принцип:
   - Раздел (например: "Архитектурные принципы")
   - После какой строки или перед каким заголовком

3. Пометь рекомендацию тегом по типу изменения:
   - `[ARCHITECTURE]` — обновить CLAUDE.md
   - `[DOC]` — обновить документацию (спецификации, гайды в development/docs/)
   - `[CODE]` — исправить код
   - `[SKILL]` — обновить скилл (ссылкой на CLAUDE.md, не дублированием правила)

4. Порядок приоритетов в рекомендациях по предотвращению:
   `[ARCHITECTURE]` → `[DOC]` → `[CODE]` → `[SKILL]`.
   Сначала архитектурные принципы (фундамент), затем документация,
   потом код, и в последнюю очередь скиллы:
   ```
   ## Рекомендации по предотвращению

   1. [ARCHITECTURE] Добавить в CLAUDE.md раздел "X": {формулировка принципа}
      Место: после раздела Y, перед разделом Z

   2. [DOC] Обновить спецификацию модуля: добавить описание {чего не хватало}
      Файл: development/specs/{module}_spec.md

   3. [CODE] Исправить {что именно} в {каком файле}

   4. [SKILL] В скилле implement-module добавить ссылку:
      "Перед реализацией см. раздел X в CLAUDE.md"
   ```

**Если НЕТ — корневая причина не архитектурная:**

Это нормально — не каждая проблема требует архитектурного принципа. Продолжай
с рекомендациями по обновлению скиллов.

**Правило:** рекомендации по предотвращению должны включать обновление
CLAUDE.md, если проблема архитектурная. Обновление только скиллов —
недостаточно, потому что скиллы — это инструкции "как делать", а CLAUDE.md —
знание "как устроена система". Агент читает CLAUDE.md первым.

### 7.2 Сохранение отчёта

1. Узнай текущую дату и время: `date +"%d-%m_%H-%M"`
2. Сформулируй краткое описание проблемы для имени файла — 2-4 слова через дефис, на русском или английском (как уместнее). Описание должно отражать суть исследуемой проблемы, чтобы по имени файла сразу было понятно, о чём отчёт.
3. Определи папку для сохранения:
   - Если в аргументах указан `output_dir=путь/к/папке` — использовать эту папку. Это позволяет вызывающему скиллу направить отчёт в нужное место.
   - Иначе — всегда использовать `development/docs/root-cause-reports`
4. Создай директорию если нет: `mkdir -p {выбранная папка}`
5. Имя файла: `{DD-MM_HH-MM}_краткое-описание-проблемы.md` (день/месяц и время впереди, затем суть проблемы через подчёркивание)
6. Сохрани через Write в `{выбранная папка}/{DD-MM_HH-MM}_краткое-описание-проблемы.md`

## Важные правила

- **Не останавливайся на поверхности** — «пакет не установлен» это не корневая причина.
  Корневая причина — почему пакет не оказался в requirements.txt или в pyproject.toml.
- **Читай файлы целиком** — не ленись, контекст вокруг ошибки часто важнее самой ошибки
- **Проверяй скиллы обязательно** — это уникальная возможность найти системные пробелы
  в процессах, а не просто одноразовый баг
- **Двойная проверка обязательна** — фаза верификации не пропускается никогда
- **Конкретика, не абстракции** — «файл X, строка Y, функция Z» вместо «в одном из файлов»
- **Запускай по одному агенту на каждую фазу** — используй параллельные агенты для
  независимых задач (например, поиск по отчётам сессий и поиск по коду одновременно)
