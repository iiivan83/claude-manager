# Автоматический цикл починки E2E тестов

**THINK HARD** на каждом шаге. Каждое решение здесь влияет на код проекта и может создать или предотвратить регрессию. Думай глубоко перед каждым действием.

Ты — автономный оркестратор, который находит и чинит все падающие E2E тесты. Твои инструменты — три существующих скилла (`test-e2e`, `root-cause-analysis`, `apply-root-cause-fixes`) и git для изоляции изменений. Главная ценность — не просто починка, а **системный подход**: группировка ошибок, глубокий анализ причин, память неудачных попыток и защита от регрессий.

Скилл полностью автономный — никаких вопросов пользователю, никаких подтверждений. Все решения по алгоритму. Единственный способ остановить — прервать скилл извне.

Аргументы пользователя: $ARGUMENTS
(Если аргумент `dry-run` — режим анализа без применения фиксов, см. секцию в конце)

## Константы

- **MAX_ATTEMPTS_PER_GROUP** = 3 (максимум попыток починки одной группы ошибок)
- **MAX_TOTAL_TIME** = 60 минут (общий лимит времени на весь цикл)
- **STATE_FILE** = `dev/temp/autofix-state.json`

## Фаза 0: Базовая линия (baseline)

Baseline нужен чтобы отличить «тест сломался из-за нашего фикса» от «тест и так был сломан». Без этого невозможно обнаружить регрессию.

### 0.1 Проверка сохранённого состояния

Прочитай `dev/temp/autofix-state.json`. Если файл существует — это прерванный предыдущий запуск. Прочитай состояние и продолжи с того места, где остановились (номер текущей группы, attempt, failed_approaches). Сообщи: «Обнаружен прерванный запуск. Продолжаю с группы N, попытка M.»

Если файла нет — это новый запуск, продолжай ниже.

### 0.2 Полный прогон всех тестов

Запусти все тесты (юнит + интеграционные + E2E) и запиши текущее состояние:

```bash
cd /Users/ivan/Desktop/claude-sandbox/claude_manager && python -m pytest tests/ -v --tb=short 2>&1 | tee /tmp/autofix_baseline.log
```

Из результатов извлеки:
- Общее количество тестов
- Список пройденных тестов (PASSED)
- Список падающих тестов (FAILED) — это **baseline failures**
- Список пропущенных (SKIPPED)

Baseline failures НЕ считаются регрессией при дальнейших проверках.

### 0.3 Создание git-тега

Создай точку отката — тег, к которому можно вернуть всё:

```bash
git tag -f autofix/baseline
```

### 0.4 Засечь время старта

Запомни время начала (для лимита MAX_TOTAL_TIME и для отчёта):

```bash
date +%s
```

### 0.5 Инициализация состояния

Создай `dev/temp/autofix-state.json`:

```json
{
  "phase": 0,
  "start_time_epoch": <timestamp>,
  "baseline": {
    "total_tests": N,
    "passed": ["test1", "test2"],
    "failed": ["test3"],
    "skipped": ["test4"]
  },
  "e2e_failures": [],
  "groups": [],
  "current_group_index": 0,
  "failed_approaches": [],
  "commits": [],
  "results": {}
}
```

## Фаза 1: Прогон E2E тестов

Цель — собрать актуальный список падающих E2E тестов с деталями.

### 1.1 Запуск E2E через агента

Запусти агента с задачей выполнить E2E тестирование в режиме **только диагностики**:

> Ты — агент оркестратора autofix-e2e. Твоя задача: вызвать скилл `test-e2e` через Skill tool.
> Контекст: это диагностический прогон — нам нужны только результаты тестирования.
> Если скилл test-e2e начнёт цикл починки (Шаг 5) — это нормально, пусть работает.
> После завершения верни: полный список результатов по каждому сценарию (ID, вердикт PASS/FAIL, ответ бота, ошибка если есть, время).
> Если скилл задаёт вопросы (например, SMS-код Telethon) — передай через AskUserQuestion.

Используй `model: "opus"` для агента.

### 1.2 Оценка результатов

Если все E2E сценарии PASS — скилл завершается:

> «Все E2E тесты проходят, починка не нужна. Удаляю файл состояния.»

Удали `dev/temp/autofix-state.json` и тег `autofix/baseline`.

Если есть FAIL — сохрани список падений в state файл и переходи к Фазе 2.

### 1.3 Фиксация изменений от test-e2e

test-e2e мог внести исправления в код (его встроенный цикл починки). Проверь `git status` — если есть изменения, зафиксируй:

```bash
git add -A && git commit -m "autofix(phase-1): изменения от диагностического прогона test-e2e"
```

Обнови baseline — эти изменения теперь часть нового базового состояния.

## Фаза 2: Группировка ошибок

Несколько тестов часто падают по одной причине (бот не стартует, сессия не создаётся, Telegram API недоступен). Чинить их поштучно неэффективно — одно исправление может починить целую группу.

### 2.1 Анализ через агента

Запусти агента для группировки:

> Проанализируй все E2E падения и сгруппируй их по вероятной корневой причине.
>
> Падения:
> {список всех FAIL с деталями: ID, ошибка, стектрейс, ответ бота}
>
> Для каждой группы определи:
> - Название группы (кратко, 3-5 слов)
> - Список тестов в группе
> - Предполагаемая корневая причина
> - Приоритет (1 = фундаментальные проблемы вроде старта бота, 2 = проблемы сессий, 3 = логика отдельных сценариев)
>
> Отсортируй группы по приоритету (сначала фундаментальные).
> Верни результат как JSON-список групп.

### 2.2 Сохранение групп

Запиши группы в state файл. Каждая группа:

```json
{
  "name": "Бот не стартует",
  "priority": 1,
  "tests": ["CJM-04-01", "CJM-02-01"],
  "suspected_cause": "Ошибка инициализации в main.py",
  "status": "pending",
  "attempts": 0,
  "fixed_by": null
}
```

## Фаза 3: Цикл починки

Группы чинятся **строго последовательно** — изменения могут конфликтовать. Внутри одной группы попытки тоже последовательны.

Перед началом каждой итерации проверяй: не превышен ли MAX_TOTAL_TIME (60 минут от старта). Если превышен — переходи к Фазе 4.

### Для каждой группы (по приоритету):

#### 3.0 Проверка каскадного эффекта

Предыдущий фикс мог починить текущую группу. Прогони E2E тесты этой группы:

```bash
cd /Users/ivan/Desktop/claude-sandbox/claude_manager && python -m pytest tests/e2e/ -v -k "{test_ids}" --tb=short
```

Если все тесты группы прошли — пометь группу как `"status": "cascaded"`, `"fixed_by": "группа N"` и перейди к следующей.

#### 3.1 Цикл попыток (attempt < MAX_ATTEMPTS_PER_GROUP)

**Шаг 1: Запомнить точку отката**

```bash
git rev-parse HEAD
```

Сохрани хеш — к нему откатимся при неудаче.

**Шаг 2: Root-cause analysis через агента**

Запусти агента (model: "opus"):

> Вызови скилл `root-cause-analysis` через Skill tool с аргументами:
> «Проблема: {описание группы}. Падающие тесты: {список}. Ошибки: {стектрейсы и логи}.
>
> ВАЖНО — уже пробовали следующие подходы, и они НЕ помогли:
> {содержимое failed_approaches — ВЕСЬ список целиком}
>
> Учти эти неудачные попытки и предложи ДРУГОЙ подход к решению.
> output_dir=dev/docs/logs/root-cause-reports»
>
> Это автоматический цикл починки — не задавай вопросов пользователю, действуй самостоятельно.

Запомни путь к созданному отчёту.

**Шаг 3: Применение фиксов через агента**

Запусти агента (model: "opus"):

> Вызови скилл `apply-root-cause-fixes` через Skill tool с аргументом: путь к отчёту {путь}.
>
> ВАЖНО: это автоматический цикл починки (autofix-e2e). Автоматически подтверждай ВСЕ
> пункты со статусом ОДОБРЕНО — не запрашивай подтверждение пользователя.
> Пропускай фазу «Подтверждение пользователем» — считай все ОДОБРЕНО пункты подтверждёнными.
> Не создавай сессионный отчёт (пропусти Фазу 5 скилла apply-root-cause-fixes).
> Не задавай вопросов — действуй самостоятельно.

**Шаг 4: Перезапуск бота (если менялся код)**

Проверь, были ли изменены .py файлы:

```bash
git diff --name-only HEAD~1 HEAD | grep '\.py$' || echo "no py changes"
```

Если да — перезапусти бота:

```bash
pkill -f "python -m claude_manager" 2>/dev/null; sleep 2
cd /Users/ivan/Desktop/claude-sandbox/claude_manager && nohup python -m claude_manager > /tmp/claude_manager_autofix.log 2>&1 &
echo $!
sleep 5
```

Проверь, что бот запустился (kill -0 PID).

**Шаг 5: Прогон E2E тестов группы**

Прогони только тесты текущей группы и проверь результат.

**ЕСЛИ тесты группы прошли:**

**Шаг 6: Регрессионная проверка — ВСЕ тесты**

```bash
cd /Users/ivan/Desktop/claude-sandbox/claude_manager && python -m pytest tests/ -v --tb=short
```

Сравни с baseline: регрессия — это когда тест РАБОТАЛ в baseline и СЛОМАЛСЯ сейчас. Тесты из baseline failures не считаются регрессией.

**Если нет регрессии (или не хуже baseline):**

```bash
git add -A && git commit -m "autofix: починена группа '{название}' — {краткое описание причины}"
```

Обнови baseline, пометь группу `"status": "fixed"`, запиши коммит в state файл. Переходи к следующей группе.

**Если регрессия:**

Записать в failed_approaches:
```
"Попытка {N} для группы '{название}': применили фикс {описание из RCA отчёта}.
Результат: тесты группы прошли, но сломался тест {имя} с ошибкой {текст}.
Вывод: фикс создаёт регрессию в {описание}."
```

Откат:

```bash
git checkout . && git clean -fd
```

attempt++. Обнови state файл.

**ЕСЛИ тесты группы НЕ прошли:**

Записать в failed_approaches:
```
"Попытка {N} для группы '{название}': применили подход {описание}.
Результат: тест по-прежнему падает с ошибкой {текст}.
Ошибка до: {оригинальная ошибка}. Ошибка после: {новая ошибка}.
Вывод: {почему подход не сработал}."
```

Откат:

```bash
git checkout . && git clean -fd
```

attempt++. Обнови state файл.

#### 3.2 Исчерпание попыток

Если attempt >= MAX_ATTEMPTS_PER_GROUP — пометь группу `"status": "unfixed"`. Запиши в state файл все попытки и причины неудач. Продолжай к следующей группе.

## Фаза 4: Финальная верификация

Полный прогон ВСЕХ тестов с чистого состояния:

```bash
cd /Users/ivan/Desktop/claude-sandbox/claude_manager && python -m pytest tests/ -v --tb=short 2>&1 | tee /tmp/autofix_final.log
```

Сравни с baseline:
- Сколько тестов починено (было FAIL, стало PASS)
- Сколько осталось FAIL
- Есть ли регрессии (было PASS, стало FAIL)

Если обнаружены регрессии, которых не было в baseline — это серьёзная проблема. Запиши детали в отчёт.

## Ключевые механизмы

### Память неудачных попыток (failed_approaches)

Самый важный механизм скилла. Без него root-cause-analysis будет предлагать одно и то же решение снова и снова, превращая цикл починки в бесконечную петлю.

Как работает:
- Глобальный список — накапливается между группами (инсайты из группы A полезны для группы B)
- Каждая запись содержит: что сделали, какой результат, почему не помогло
- При каждом вызове root-cause-analysis ВЕСЬ список передаётся целиком
- Формат записи: «Попытка N для группы X: применили {описание}. Результат: {что произошло}. Вывод: {почему не помогло}.»

### Git-изоляция

Каждая попытка фикса начинается с чистого состояния (последний коммит). Это гарантирует, что неудачный фикс не накапливается:
- Перед попыткой: запомнить `git rev-parse HEAD`
- Неудача: `git checkout . && git clean -fd` (откат всех изменений)
- Успех: `git commit` (фиксация результата)
- НЕ использовать ветки — работаем на текущей ветке

### Каскадный эффект

После починки группы A группа B может починиться автоматически (если зависела от A). Перед починкой каждой группы — прогнать её тесты заново. Если прошли — пропустить, записать «починилась каскадно». Это экономит время и избегает ненужных изменений.

### Сравнение с baseline (не хуже)

При проверке регрессии сравниваем с baseline, а НЕ с «все тесты зелёные»:
- Тест падал ДО наших изменений и продолжает падать = НЕ регрессия
- Тест РАБОТАЛ до наших изменений и СЛОМАЛСЯ = РЕГРЕССИЯ
- Только регрессия вызывает откат

### Сохранение состояния

Состояние записывается в `dev/temp/autofix-state.json` после каждого значимого шага (завершение группы, попытка, откат). При прерывании скилла — следующий запуск продолжит с места остановки. После успешного завершения файл удаляется.

## Режим dry-run

Если пользователь передал аргумент `dry-run`:

1. Выполни Фазы 0-2 как обычно (baseline, E2E прогон, группировка)
2. Для каждой группы запусти root-cause-analysis (Шаг 2 из Фазы 3) — но НЕ применяй фиксы
3. Создай отчёт-план: для каждой группы — предполагаемая причина и рекомендации из root-cause отчёта
4. Не делай git-коммитов, не изменяй код
5. Сообщи пользователю план починки

Это полезно когда хочется сначала посмотреть, что autofix-e2e собирается делать, прежде чем запускать в полностью автоматическом режиме.
