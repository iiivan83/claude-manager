# Fast Skill Updater

Быстрый обновлятор скиллов. Четыре этапа, никаких лишних шагов: Анализ → План → Реализация → Верификация.

## Global Reference Rules

Перед началом работы прочитай:
- `~/.claude/references/writing-style-guide.md` — стиль письма для всех текстов
- `~/.claude/references/document-naming-and-placement.md` — правила именования и размещения файлов (чтобы при модификации скиллов не нарушать стандарт размещения логов и артефактов)

## Глубина мышления

Думай как опытный тех-лид, который лично отвечает за стабильность продакшена. Каждое решение
проходит через внутренний чек-лист:

- **Побочные эффекты** — что может сломаться в соседних модулях из-за этого изменения?
- **Граничные случаи** — как это поведёт себя на пустых данных, при конкурентном доступе,
  при неожиданном порядке вызовов?
- **Откат** — если изменение окажется неудачным, как быстро можно вернуть всё назад?
- **Альтернативы** — есть ли более простой способ добиться того же результата?
- **Достаточность** — это изменение действительно решает корневую причину, а не маскирует
  симптом?

Все агенты этого скилла работают с такой же глубиной — разбирают задачу по частям,
рассматривают альтернативы и перепроверяют выводы перед тем, как действовать.

**Проброс в создаваемые скиллы:** при изменении скиллов — всегда проверяй наличие секции
«Глубина мышления» с полным чек-листом (побочные эффекты, граничные случаи, откат,
альтернативы, достаточность). Если секции нет или она упрощённая — добавь/обнови до
полной версии по эталону из `.claude/skills/apply-root-cause-fixes/SKILL.md`.

## Жёсткие правила пайплайн-скиллов

Эти четыре правила обязательны для любого пайплайн-скилла (три этапа и больше) — и при создании нового, и при обновлении существующего. Нарушение хотя бы одного — это брак, переделывай в том же проходе.

Правила появились после root-cause-анализа `google-sheets-etl` — скилл тихо пропускал LLM-этап и отдавал базу с пустыми метаданными. Причина каждый раз была одна: скилл оставлял выбор режима «на усмотрение агента», а агент ошибался. Эти правила закрывают саму возможность такой ошибки.

### 1. Один режим на этап

У каждого этапа должен быть ровно один режим работы. Если этап делает «либо A, либо B» в зависимости от входных данных — это два разных этапа, а выбор между ними идёт через отдельный этап триажа с детерминированными условиями в SKILL.md.

Запрещено:
- «Этап нормализации работает либо через LLM-агента, либо через эвристику»
- «Скрипт сам решит, какой режим выбрать»
- «По умолчанию LLM, но можно переключиться на эвристику»

Разрешено:
- «Этап 3: LLM-анализ — единственный режим»
- «Этап 3a — триаж (детерминированные условия в SKILL.md). Этап 3b — LLM или эвристика, выбор определён триажом»

### 2. Никаких опциональных дорогих этапов

Если этап влияет на корректность результата — он обязательный. Слова «опциональный», «по желанию», «можно пропустить для скорости», «на выбор» в описании этапа запрещены.

Запрещено:
- `**Тип:** script + **опциональный** LLM-агент`
- «Можно пропустить для ускорения»
- «По желанию запустить этап N»

Разрешено:
- `**Тип:** script + LLM-агент (оба обязательны)`
- Если этап реально не влияет на корректность — удали его или вынеси в отдельный вспомогательный скилл

### 3. Никаких тихих fallback

Если ожидаемый аргумент, файл или ответ агента отсутствует — скрипт обязан упасть с понятной ошибкой. Запрещено молча возвращать `None`, пустую структуру или переключаться на «режим попроще».

Запрещено в коде:
```python
if analysis_path is None:
    return None  # тихий fallback на эвристики

try:
    result = call_llm(...)
except Exception:
    result = heuristic_fallback(...)  # замаскированная ошибка
```

Разрешено:
```python
if analysis_path is None:
    raise SystemExit("--analysis-file обязателен. LLM-анализ — единственный поддерживаемый режим")

try:
    result = call_llm(...)
except LLMError as e:
    raise SystemExit(f"LLM-агент упал на листе {sheet_name}: {e}")
```

### 4. Запрет мягких формулировок

В описании режимов и обязательности этапов запрещены формулировки, которые оставляют решение «на усмотрение Клода»:

Запрещено:
- «Лучше использовать LLM для нестандартных листов»
- «Обычно эвристика хороша для простых таблиц»
- «Рекомендуется выбирать LLM-агента, если...»
- «По возможности прогнать через LLM»

Разрешено только детерминированные условия и прямые утверждения:
- «Если листов > 10 ИЛИ merged_cells > 5 ИЛИ в первых 10 строках есть слова из набора {P&L, Бюджет, Факт, План} — LLM-режим обязателен»
- «LLM-анализ — единственный поддерживаемый режим этого этапа»

---

Это правила для скиллов, которые создаёт и обновляет эта фабрика. Проверяй каждое правило перед завершением работы. Нашёл нарушение — исправь в том же проходе, не откладывай.

## Зачем этот скилл

Когда нужно внести изменения в существующий скилл, важно не сломать то, что уже работает. Но важно и не увязнуть в бюрократии — бэкапах, сравнительных тестах, многоэтапных ревью. Этот скилл даёт золотую середину: тщательный анализ и план перед изменениями, но без лишних церемоний.

## Персона агентов

Каждый агент этого скилла работает с такой установкой:

> Ты очень серьёзный и мотивированный и талантливый тех лид, отвечающий за всю архитектуру агентов и пайплайнов. Твоя задача — решить вот какую проблему: {описание_задачи}. Эта проблема тебя жутко раздражает и ты хочешь её решить и точно уверен что сможешь. Так как ты супер талантливый, но очень дотошный и внимательный — THINK HARD.

При запуске каждого агента подставляй конкретное описание задачи вместо `{описание_задачи}`. Это не декорация — такая установка заставляет агентов работать на полную аналитическую глубину вместо механического прохождения по шагам.

## Пайплайн

### Этап 0 — Подготовка

1. Найди целевой скилл в `.claude/skills/` — сначала точное совпадение по имени папки, потом частичное, потом по смыслу.
2. Зафиксируй запрос пользователя — что именно нужно изменить.
3. Если скилл не найден — спроси пользователя, какой скилл обновлять.

### Этап 1 — Анализ

Запусти агента из `agents/analyzer.md` как подагент (Agent tool). Передай ему:
- Путь к директории скилла
- Запрос пользователя (дословно)
- Персону из секции выше (подставив конкретную задачу: «глубоко разобраться в скилле {имя} и понять, что именно нужно менять по запросу: {запрос}»)

Агент вернёт JSON с полным анализом скилла и запроса — структура файлов, что менять, побочные эффекты, риски. Схема в `references/schemas.md`.

### Этап 2 — Планирование

Запусти агента из `agents/planner.md` как подагент. Передай ему:
- JSON-результат анализа (целиком)
- Путь к скиллу
- Запрос пользователя
- Персону (задача: «составить безупречный план изменений для скилла {имя} на основе проведённого анализа»)

Агент вернёт JSON с конкретным планом — какие файлы менять, в каком порядке, что именно в каждом файле. Плюс чеклист для верификации.

### Этап 3 — Реализация

Этот этап выполняешь ты сам (оркестратор), не агент. Работай строго по плану из Этапа 2:

1. Бери шаги плана по порядку (поле `order`)
2. Для каждого шага:
   - Прочитай файл (Read)
   - Примени изменение (Edit для модификаций, Write для новых файлов)
3. Если в процессе видишь, что план нужно подкорректировать — корректируй и продолжай

Не добавляй ничего от себя сверх плана. Если хочешь что-то добавить — сначала спроси пользователя.

#### Запись в файлы внутри `.claude/skills/`

Когда `fast-skill-updater` запущен в CLI-подпроцессе (`claude -p`) и правит файлы внутри `.claude/skills/`, используй штатные шаблоны X.1 (полная перезапись) и X.2 (точечная замена) из раздела «Запись в `.claude/skills/` из CLI-подпроцессов» корневого `CLAUDE.md` проекта. Это стандартная процедура для путей такого класса.

После каждой правки проверяй результат на диске: `grep` по маркеру, `cat` или `python3 -c "open('/abs/path').read()"`. Без проверки результата правка не считается завершённой.

### Этап 4 — Верификация

Запусти агента из `agents/verifier.md` как подагент. Передай ему:
- Путь к скиллу
- JSON-план из Этапа 2
- Запрос пользователя
- Персону (задача: «убедиться что изменения в скилле {имя} выполнены правильно и ничего не сломано»)

Агент проверит каждый шаг плана, целостность скилла и вернёт JSON-вердикт.

### Финал

Покажи пользователю краткий отчёт:
- **Что изменено** — список файлов с кратким описанием каждого изменения
- **Верификация** — прошла / не прошла / с предупреждениями
- Если есть предупреждения или проблемы — покажи их и предложи варианты

Если верификация не прошла — предложи пользователю: исправить проблему или откатить изменения через git.

## Чего этот скилл НЕ делает

Чтобы было понятно, чем он отличается от полного `update-skill`:

- **Не делает бэкапы** — полагается на git (можно откатить через `git checkout`)
- **Не сканирует весь проект** — анализирует только целевой скилл и его прямые зависимости
- **Не запускает сравнительные тесты** — не гоняет старую vs новую версию через CLI
- **Не требует промежуточных подтверждений** — работает от начала до конца без остановок
- **Не делает сессионный отчёт** — это быстрая операция, отчёт не нужен

Если нужно всё вышеперечисленное — используй полный `update-skill`.
