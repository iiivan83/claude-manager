# Script Creator Pipeline — От идеи до готового скрипта

Скилл объединяет два процесса в один непрерывный пайплайн:

- **Фаза 1 — Спецификация:** исследование кодовой базы → драфт спеки → ревью → исправления
- **Фаза 2 — Реализация:** написание кода → дизайн тестов → прогон тестов → исправления кода

Пользователь описывает идею скрипта → на выходе получает готовый рабочий код с пройденными тестами.

---

## Global Reference Rules

Перед началом работы прочитай `~/.claude/references/writing-style-guide.md` — единый стиль письма для всех документов и отчётов.

## Глубина мышления

Думай как опытный тех-лид, который лично отвечает за стабильность продакшена. Каждое решение
проходит через внутренний чек-лист:

- **Побочные эффекты** — что может сломаться в соседних модулях из-за этого изменения?
- **Граничные случаи** — как это поведёт себя на пустых данных, при конкурентном доступе,
  при неожиданном порядке вызовов?
- **Откат** — если изменение окажется неудачным, как быстро можно вернуть всё назад?
- **Альтернативы** — есть ли более простой способ добиться того же результата?
- **Достаточность** — это изменение действительно решает корневую причину, а не маскирует
  симптом?

Все агенты этого скилла работают с такой же глубиной — разбирают задачу по частям,
рассматривают альтернативы и перепроверяют выводы перед тем, как действовать.

## Общие принципы

- **Каждый этап — отдельный агент** (Agent tool), не в основном контексте
- **Промежуточные данные — JSON**, итоговая спецификация — MD
- **Язык** — русский для коммуникации, код и технические термины — на языке реализации
- **Запись скрипта внутрь `.claude/skills/`** — если целевой путь из спецификации
  (поле `**Размещение:**`) попадает внутрь папки `.claude/skills/`, агенты `coder` и
  `fixer-code` применяют штатные шаблоны X.1 и X.2 из раздела «Запись в `.claude/skills/`
  из CLI-подпроцессов» корневого `CLAUDE.md` проекта. Для путей вне этой папки —
  обычные Write и Edit.

## Размещение файлов

- **Спецификации** → `dev/docs/specs/{DD.MM_HH.MM}-{название-скрипта}-spec.md`
- **Реализованные спецификации** → `dev/docs/specs/realised/{DD.MM_HH.MM}-{название-скрипта}-spec.md`

Где `{DD.MM_HH.MM}` — временна́я метка начала пайплайна (день.месяц_час.минута, например `03.04_14.30`).

- **Логи пайплайна** → `dev/docs/logs/{название-скрипта}/pipeline/`
- **JSON-выходы агентов** → `dev/docs/logs/{название-скрипта}/pipeline/agent-outputs/`
- **Тестовые данные** → `dev/docs/logs/{название-скрипта}/pipeline/tests/`

## Логирование операций

Каждая операция пайплайна обновляет единый JSON-лог оркестратора. Это позволяет в любой момент посмотреть, на каком шаге пайплайн, что было на входе, что на выходе, и где произошёл сбой.

### Лог оркестратора

**Файл:** `dev/docs/logs/{название-скрипта}/pipeline/orchestrator-log.json`

```json
{
  "pipeline": "script-creator-pipeline",
  "script_name": "название-скрипта",
  "created_at": "2026-04-02T14:30:00",
  "steps": [
    {
      "step_number": 1,
      "start_time": "2026-04-02T14:32:00",
      "end_time": "2026-04-02T14:45:00",
      "agent": "codebase-researcher",
      "status": "success",
      "comment": "",
      "output_file": "agent-outputs/01-codebase-researcher.json",
      "next_agent": "drafter"
    }
  ]
}
```

**Паттерн логирования:** запись в orchestrator-log.json делается ДО начала каждого этапа (не после). Это позволяет видеть, на каком этапе пайплайн прервался, если он упал. Также обновляй лог после согласования с пользователем.

### Стандартный формат вывода агентов

Все агенты возвращают результат в едином JSON-формате. Поле `result` содержит данные, специфичные для каждого агента.

```json
{
  "agent": "имя-агента",
  "pipeline": "script-creator-pipeline",
  "script_name": "название-скрипта",
  "timestamp": "2026-04-02T14:32:00",
  "status": "success | partial | failed",
  "input": {
    "description": "Что агент получил на вход",
    "files": ["путь/к/файлу.json"]
  },
  "result": {},
  "created_files": [
    { "path": "путь/к/файлу", "description": "что это за файл" }
  ]
}
```

## Пайплайн: пошаговая инструкция для оркестратора

### Перед запуском

1. Получи от пользователя описание скрипта (может быть одно предложение или подробное описание)
2. Определи краткое имя скрипта для файлов (нижний регистр, через дефис, например: `backup-database`, `sync-translations`)
3. Сгенерируй временну́ю метку `{DD.MM_HH.MM}` (текущие день.месяц_час.минута, например `03.04_14.30`) — она используется в имени файла спецификации
4. Создай папки:
   - `dev/docs/specs/` (если не существует)
   - `dev/docs/logs/{название-скрипта}/pipeline/agent-outputs/`
   - `dev/docs/logs/{название-скрипта}/pipeline/tests/`
5. Инициализируй `orchestrator-log.json` с пустым массивом `steps`

## Фаза 1: Создание спецификации

### Этап 1. Исследователь кодовой базы

**Агент:** прочитай `agents/codebase-researcher.md` и запусти агента через Agent tool.

**Передай агенту:**
- Описание скрипта от пользователя (как есть, без изменений)
- Корневую папку проекта
- Путь для сохранения JSON: `dev/docs/logs/{название}/pipeline/agent-outputs/01-codebase-researcher.json`

**После завершения:**
- Обнови `orchestrator-log.json` (step_number: 1, agent: "codebase-researcher", next_agent: "drafter")

### Этап 2. Драфтер спецификации

**Агент:** прочитай `agents/drafter.md` и запусти агента.

**Передай агенту:**
- Описание скрипта от пользователя (как есть, без изменений)
- Результаты исследования: `agent-outputs/01-codebase-researcher.json`
- Путь для итоговой спецификации: `dev/docs/specs/{DD.MM_HH.MM}-{название-скрипта}-spec.md`
- Путь для JSON-метаданных: `agent-outputs/02-drafter.json`

**После завершения:**
- Обнови `orchestrator-log.json` (step_number: 2, agent: "drafter", next_agent: "reviewer")
- Покажи пользователю краткую сводку: сколько секций создано, какие основные решения приняты

### Этап 3. Ревьюер

**Агент:** прочитай `agents/reviewer.md` и запусти агента.

**Передай агенту:**
- Спецификацию: `dev/docs/specs/{DD.MM_HH.MM}-{название-скрипта}-spec.md`
- Описание скрипта от пользователя (для проверки полноты)
- Результаты исследования: `agent-outputs/01-codebase-researcher.json`
- Путь для JSON-отчёта: `agent-outputs/03-reviewer.json`

**После завершения:**
- Обнови `orchestrator-log.json` (step_number: 3, agent: "reviewer", next_agent: "fixer-spec")
- Покажи пользователю замечания ревьюера (коротко, по пунктам)
- Спроси пользователя: «Применить все замечания? Или есть что-то, что хочешь пропустить?»
- Обнови JSON-файл ревьюера, проставив статусы `"approved"` / `"rejected"` у каждого замечания

### Этап 4. Исправитель спецификации

**Агент:** прочитай `agents/fixer-spec.md` и запусти агента.

**Передай агенту:**
- Спецификацию: `dev/docs/specs/{DD.MM_HH.MM}-{название-скрипта}-spec.md`
- Замечания с одобрениями: `agent-outputs/03-reviewer.json`
- Путь для JSON-отчёта: `agent-outputs/04-fixer-spec.json`

**После завершения:**
- Обнови `orchestrator-log.json` (step_number: 4, agent: "fixer-spec", next_agent: "coder")
- Покажи пользователю итог: сколько замечаний исправлено, что изменилось

## Переход между фазами

После завершения фазы 1 покажи пользователю итог:
- Путь к готовой спецификации
- Краткая сводка: сколько секций, сколько замечаний было и сколько исправлено
- Спроси: «Спецификация готова. Переходим к реализации?»

Если пользователь хочет внести ручные правки — дай ему время, затем продолжи.

## Фаза 2: Реализация

### Этап 5. Кодер

Перед запуском агента прочитай первые 10 строк спецификации и извлеки:
- **Язык** — из поля `**Язык:** bash / python / node`
- **Целевой путь** — из поля `**Размещение:** путь/к/скрипту`

**Агент:** прочитай `agents/coder.md` и запусти агента через Agent tool.

**Передай агенту:**
- Полный путь к спецификации (MD-файл)
- Целевой путь для скрипта (из мета-информации спеки)
- Язык реализации
- Путь для JSON-метаданных: `agent-outputs/05-coder.json`

**После завершения:**
- Обнови `orchestrator-log.json` (step_number: 5, agent: "coder", next_agent: "test-designer")
- Покажи пользователю: что создано, где лежит файл, сколько строк кода

### Этап 6. Тест-дизайнер

Тест-дизайнер работает по принципу «чёрного ящика» — он не видит исходный код, только публичный контракт скрипта из спецификации. Это нужно для объективной проверки: тесты проверяют реальный контракт, а не подстраиваются под реализацию.

**Агент:** прочитай `agents/test-designer.md` и запусти агента через Agent tool.

**Передай агенту строго ограниченный набор информации:**
- Язык реализации (bash / python / node) — только чтобы знать, как запускать
- Путь к скрипту (куда он установлен) — только чтобы знать, что вызывать
- Из спецификации — ТОЛЬКО эти секции (извлеки и передай текстом, НЕ давай путь к полной спеке):
  - **Заголовок** (что делает скрипт — одно предложение)
  - **Входные данные** (CLI-аргументы, переменные окружения, файлы)
  - **Выходные данные** (stdout, файлы, exit codes)
  - **Примеры использования** (если есть в спеке)
  - **Граничные случаи** (если есть в спеке)
- Путь к папке для тестов: `dev/docs/logs/{название}/pipeline/tests/`
- Путь для JSON-метаданных: `agent-outputs/06-test-designer.json`

**После завершения:**
- Обнови `orchestrator-log.json` (step_number: 6, agent: "test-designer", next_agent: "test-runner")
- Покажи пользователю: сколько тест-кейсов создано, какие категории покрыты

### Этап 7. Прогон тестов

Оркестратор сам прогоняет тесты — это не отдельный агент, а работа оркестратора.

1. Прочитай `agent-outputs/06-test-designer.json` — там список тест-кейсов
2. Для каждого тест-кейса:
   - Подготовь входные данные (создай временные файлы, если нужно)
   - Запусти скрипт с указанными аргументами через Bash tool
   - Сохрани: stdout, stderr, exit code, созданные файлы
   - Сравни с ожидаемым результатом из тест-кейса
   - Запиши результат: `pass` или `fail` с деталями
3. Сохрани все результаты в `dev/docs/logs/{название}/pipeline/tests/test-results.json`
4. Обнови `orchestrator-log.json` (step_number: 7, agent: "test-runner (orchestrator)", next_agent зависит от результатов)
5. Покажи пользователю сводку:
   - Сколько тестов пройдено / провалено
   - Для каждого проваленного: что ожидалось vs что получилось
6. Если все тесты пройдены → переходи к завершению
7. Если есть провалы → переходи к этапу 8

### Этап 8. Исправитель кода (если нужен)

Запускается только если тесты выявили проблемы.

**Агент:** прочитай `agents/fixer-code.md` и запусти агента.

**Передай агенту:**
- Путь к реализованному скрипту
- Путь к спецификации (полная — для справки)
- Результаты тестов: `tests/test-results.json`
- Путь для JSON-отчёта: `agent-outputs/07-fixer-code.json`

**После завершения:**
- Обнови `orchestrator-log.json` (step_number: 8, agent: "fixer-code", next_agent: "test-runner")
- Покажи пользователю: что исправлено

**Повторный прогон тестов:** после исправлений запусти этап 7 заново. Максимум 2 цикла (этап 7 → 8 → 7). Если после второго цикла остаются проваленные тесты — покажи их пользователю и спроси, как поступить.

## Завершение

После успешного прохождения всех этапов:

1. Обнови `orchestrator-log.json` финальным статусом
2. Создай папку `dev/docs/specs/realised/`, если она не существует (через `mkdir -p`)
3. Перемести спецификацию: `mv dev/docs/specs/{DD.MM_HH.MM}-{название}-spec.md dev/docs/specs/realised/{DD.MM_HH.MM}-{название}-spec.md`
4. Покажи пользователю итог:
   - Путь к готовому скрипту
   - Результаты тестов (сколько пройдено)
   - Путь к перемещённой спецификации
5. Предложи открыть файл скрипта в VS Code через `code -r`

```
Используй /session-report.

Скрипт: {название-скрипта}
```
