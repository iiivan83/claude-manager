# Создание сценариев пользовательского тестирования

**THINK HARD** на каждом шаге выполнения этого скилла. Используй максимальную глубину мышления при принятии каждого решения — от анализа BRD до формулировки шагов и критериев прохождения. Каждый сценарий будет выполняться вручную или автоматически, поэтому любая неточность или пропуск приведут к тому, что тестировщик не сможет выполнить тест или пропустит баг.

Ты создаёшь подробные сценарии пользовательского тестирования на основе BRD (документа с описанием всех пользовательских путей через Telegram-бота). Сценарии — это пошаговые инструкции, по которым можно проверить, что бот работает именно так, как описано в BRD. Это предпоследний рубеж качества: после E2E тестирования, после ревью кода, перед финальным прогоном.

Твоя задача — не просто пересказать BRD. Ты должен превратить описания сценариев в конкретные, выполнимые шаги с точным текстом сообщений, точными командами и однозначными критериями прохождения. Тестировщик, читающий твой документ, не должен гадать — каждый шаг должен быть настолько конкретным, что его можно выполнить механически.

## Шаг 1: Чтение входных данных

**THINK HARD** — на этом шаге ты формируешь полную картину: что бот должен делать (BRD) и какие проблемы уже были найдены (E2E результаты). Пропуск любого из источников приведёт к неполным сценариям.

Запусти два параллельных агента (Agent tool) для чтения входных данных:

### Агент 1: Чтение BRD

Прочитай `development/docs/brd-user-journeys.md`. Для каждого CJM извлеки:

- Название и номер CJM (например, CJM-02: Отправка текстового сообщения)
- Что делает пользователь — пошаговые действия
- Что происходит внутри — логика работы бота
- Что видит пользователь — ожидаемое поведение
- Что может пойти не так — сценарии ошибок
- Карту состояний пользователя (если описана)
- Механизмы, которые работают во всех сценариях (раздел в конце BRD)

Верни структурированный список всех CJM с извлечёнными данными.

### Агент 2: Чтение результатов E2E тестирования

Найди последний файл результатов E2E тестирования: `development/docs/testing/e2e-test-results_*.md`. Если файлов несколько — бери последний по дате (дата в имени файла в формате ДД-ММ-ЧЧ-ММ).

Если файл найден — извлеки:
- Какие сценарии были протестированы
- Какие прошли (pass), какие упали (fail)
- Описание найденных багов и их исправлений
- Какие граничные случаи проверялись

Если файла нет (E2E тестирование ещё не проводилось) — это нормально. Верни пустой результат и отметь, что E2E результатов пока нет. Сценарии будут создаваться только на основе BRD.

## Шаг 2: Генерация сценариев

**THINK HARD** — это ключевой шаг. Каждый сценарий должен быть настолько конкретным, что его можно выполнить механически, не задавая вопросов. Неоднозначный шаг = невыполнимый тест.

Для каждого CJM из BRD создай три типа сценариев:

### Тип A: Основной сценарий (happy path)

Это «идеальный путь» — когда всё идёт правильно, пользователь делает всё как надо, система работает без сбоев. Один сценарий на CJM.

### Тип B: Сценарии ошибок

Из раздела «Что может пойти не так» в BRD. Каждая описанная ошибка — отдельный сценарий. Если BRD описывает 3 ошибки — будет 3 сценария.

### Тип C: Граничные случаи

Ситуации, которые не описаны в BRD напрямую, но которые реальный пользователь может создать. Стандартный набор для Telegram-бота:

- **Пустое сообщение** — отправить сообщение без текста (только пробелы, или стикер, или голосовое)
- **Очень длинное сообщение** — текст длиннее 4096 символов
- **Спецсимволы в тексте** — HTML-теги (`<b>текст</b>`), амперсанды (`&`), угловые скобки (`<`, `>`), эмодзи, кириллица + латиница в одном сообщении
- **Быстрая отправка** — несколько сообщений подряд без ожидания ответа
- **Одновременные действия** — отправить сообщение во время ожидания ответа от предыдущего
- **Команда во время обработки** — отправить `/stop` или `/new` пока Claude думает

Не все граничные случаи применимы к каждому CJM. Выбирай только те, которые имеют смысл для конкретного сценария.

### Структура каждого сценария

Каждый сценарий записывается по следующему шаблону:

```
### TC-{CJM номер}-{тип}{порядковый номер}: {название}

**Тип:** Happy Path / Ошибка / Граничный случай
**Связь с BRD:** CJM-{номер}: {название CJM}
**Проверяемые требования:** {конкретные пункты из BRD, которые проверяются}

**Предусловия:**
- {что должно быть настроено перед тестом}
- {состояние бота: запущен/остановлен}
- {состояние пользователя: подключён к сессии / не подключён / в режиме /all}

**Шаги:**
1. {конкретное действие} → **Ожидаемый результат:** {что пользователь видит}
2. {конкретное действие} → **Ожидаемый результат:** {что пользователь видит}
3. ...

**Критерии прохождения (pass):**
- [ ] {конкретное условие, которое можно проверить однозначно}
- [ ] {ещё условие}

**Критерии провала (fail):**
- {что означает провал — конкретные признаки}
```

### Правила формулировки шагов

Шаги должны быть максимально конкретными. Это критично — размытый шаг невозможно выполнить одинаково дважды.

**Плохо:**
- «Отправь сообщение боту» (какое сообщение?)
- «Проверь что бот ответил» (что именно проверить?)
- «Подключись к сессии» (какой именно?)

**Хорошо:**
- «Отправь в Telegram сообщение: `Посмотри файл main.py и скажи что он делает`»
- «Убедись что бот ответил сообщением, начинающимся с `#1 ✅`»
- «Отправь команду `/1` для переключения на первую сессию»

### Правила формулировки ожидаемых результатов

Ожидаемый результат должен быть проверяемым. Если результат зависит от содержания ответа Claude (которое непредсказуемо) — описывай формат и структурные признаки ответа, а не его содержание.

**Плохо:**
- «Claude ответит описанием файла» (как проверить полноту описания?)

**Хорошо:**
- «Бот отправит сообщение в формате: `#N ✅` + текст ответа. Ответ содержит текст (не пустой), форматирование HTML корректно (нет "сырых" тегов)»

### Правила формулировки критериев прохождения

Критерий должен быть бинарным — pass или fail, без «скорее да». Каждый критерий — одно конкретное условие.

**Плохо:**
- «Бот работает нормально»
- «Ответ выглядит правильно»

**Хорошо:**
- «Ответ бота начинается с `#N ✅` (где N — номер сессии)»
- «Ответ содержит непустой текст (больше 0 символов после заголовка)»
- «Если ответ длиннее 4096 символов — разбит на несколько сообщений»
- «Индикатор «печатает...» виден в чате во время обработки запроса»

## Шаг 3: Учёт результатов E2E

**THINK HARD** — если E2E тестирование выявило баги, сценарии должны это учитывать. Не просто дублировать E2E-кейсы, а убедиться, что исправленные баги покрыты сценариями.

Если результаты E2E тестирования доступны:

1. Для каждого бага, найденного в E2E — проверь, есть ли сценарий, который его покрывает
2. Если бага нет в существующих сценариях — добавь отдельный сценарий типа «Регрессия» (проверка, что исправленный баг не вернулся)
3. В предусловиях таких сценариев укажи: «Регрессия: проверка исправления бага из E2E тестирования»

## Шаг 4: Сквозные сценарии

**THINK HARD** — сквозные сценарии проверяют переходы между состояниями. Баги часто прячутся именно на стыках: переключился с сессии на сессию, вышел из /all и написал сообщение. Отдельные CJM-сценарии это не ловят.

После создания сценариев по каждому CJM — добавь сквозные сценарии, которые проверяют цепочки действий из нескольких CJM. Они покрывают переходы между состояниями пользователя (карта состояний из BRD).

Примеры сквозных сценариев:

- **Полный жизненный цикл:** `/new` → отправить сообщение → получить ответ → `/new` → отправить сообщение → `/sessions` → переключиться на первую сессию → отправить сообщение
- **Мониторинг и переключение:** `/all` → дождаться сообщения из сессии → нажать на номер сессии → отправить сообщение в эту сессию
- **Работа с файлами в контексте сессий:** создать сессию → отправить фото → получить ответ → переключиться на другую сессию → отправить текст

Сквозные сценарии используют ту же структуру (TC-CROSS-{номер}).

## Шаг 5: Самопроверка

**THINK HARD** — это финальная проверка качества. Каждый пропущенный CJM или размытый шаг — это потенциально пропущенный баг в продакшене.

Перед записью документа проведи самопроверку (минимум 2 итерации):

### Итерация 1: Полнота покрытия

- Каждый CJM из BRD имеет хотя бы один happy path сценарий
- Каждая ошибка из раздела «Что может пойти не так» покрыта сценарием
- Все три состояния пользователя (чистый лист, подключён, радар) протестированы
- Переходы между состояниями покрыты сквозными сценариями
- Механизмы из раздела «работают во всех сценариях» (промежуточные обновления, координация watcher, восстановление после перезапуска) — покрыты

### Итерация 2: Качество формулировок

- Каждый шаг содержит конкретный текст сообщения или конкретную команду
- Каждый ожидаемый результат проверяем (можно однозначно сказать pass или fail)
- Предусловия достаточны для выполнения сценария (ничего не пропущено)
- Критерии прохождения бинарны (нет субъективных оценок)
- Нет дублирования между сценариями (если два сценария проверяют одно и то же — объединить или удалить)

Если какая-либо проверка не прошла — исправь и проведи ещё одну итерацию.

## Шаг 6: Запись документа

Формат имени файла: `user-test-scenarios_ДД-ММ-ЧЧ-ММ.md` — где ДД-ММ — день-месяц, ЧЧ-ММ — часы-минуты текущего времени. Для получения текущего времени выполни команду `date +"%d-%m-%H-%M"`.

Путь: `development/docs/testing/user-test-scenarios_{timestamp}.md`

Если папка `development/docs/testing/` не существует — создай её.

## Шаг 7: Обновление pipeline-state.json

Если файл `pipeline-state.json` существует в корне проекта — обнови его:

1. Прочитай текущее содержимое
2. Найди фазу 8 (Сценарии пользовательского тестирования)
3. Установи статус `completed` и время завершения

Если файла нет — пропусти этот шаг (файл создаётся оркестратором).

## Контекст: зачем нужны эти сценарии

Сценарии пользовательского тестирования — это мост между техническими E2E тестами и реальным опытом пользователя. E2E тесты проверяют, что бот отвечает правильно через API. Пользовательские сценарии проверяют, что взаимодействие через настоящий Telegram-интерфейс работает как ожидается: форматирование, кликабельные ссылки, индикатор «печатает...», разбивка длинных сообщений, визуальное оформление номеров сессий.

Следующий шаг после создания сценариев — их прогон скиллом `run-user-testing`. Поэтому качество сценариев напрямую влияет на качество тестирования: если шаг неоднозначен — тестировщик может выполнить его по-разному и получить разные результаты, что делает тест бесполезным.
