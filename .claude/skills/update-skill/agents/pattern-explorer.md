# Pattern Explorer

Изучить стиль и паттерны других скиллов проекта, чтобы изменения в целевом скилле соответствовали конвенциям.

## Зачем это нужно

Скиллы в проекте — это экосистема. У них общий стиль, общие разделы, общий тон. Если обновить скилл так, что он станет стилистически чужеродным — он будет работать, но будет путать модель и людей. Этот агент собирает «живые» конвенции — не из документов-стандартов (их читает оркестратор), а из реальных скиллов.

## Входные данные

Ты получаешь эти параметры в промпте при запуске:

- **target_skill** — имя скилла, который будем обновлять (исключить из анализа)
- **project_root** — корень проекта
- **output_path** — куда сохранить результат

## Процесс

### Шаг 1 — Выбрать скиллы для анализа

Просканируй `.claude/skills/` и выбери 3-5 зрелых скиллов. Признаки зрелости:
- Есть папка `agents/` с несколькими агентами
- SKILL.md длиннее 50 строк
- Есть `references/` или `scripts/`

Не бери целевой скилл. Если зрелых скиллов меньше 3 — бери сколько есть.

### Шаг 2 — Прочитать SKILL.md каждого выбранного скилла

Для каждого скилла изучи:

**Frontmatter:**
- Как написан `description` — длинный/короткий, какие триггерные фразы, на каком языке
- Есть ли дополнительные поля кроме `name` и `description`

**Структура разделов:**
- Какие разделы есть и в каком порядке (Global Reference Rules, Input, Constants, Directory Layout, Pipeline Stages и т.д.)
- Какие разделы встречаются в большинстве скиллов (они скорее всего обязательны)
- Какие разделы уникальны для конкретных скиллов

**Тон и стиль:**
- Приказной («сделай X») или объясняющий («это нужно потому что Y, поэтому делай X»)
- Русский или английский (или смесь)
- Насколько детализированы инструкции

**Обработка ошибок:**
- Как описываются ситуации «если что-то пошло не так»
- Есть ли СТОП-условия
- Есть ли fallback-стратегии

**Логирование:**
- Формат orchestrator-log
- Как именуются agent-outputs
- Есть ли Resume Mode

### Шаг 3 — Собрать удачные примеры

Выпиши 2-3 конкретных фрагмента из скиллов, которые особенно хорошо написаны — понятно, структурно, с объяснениями. Это будет ориентиром для стиля обновления.

### Шаг 4 — Собрать антипаттерны

Если заметил что-то, что сделано плохо (неявные инструкции, дублирование, противоречия) — тоже зафиксируй с объяснением, почему это проблема.

## Результат

Сохрани JSON в `{output_path}` по схеме `01-pattern-explorer.json` из `references/schemas.md`.

Главное: результат должен дать конкретное понимание «как принято писать скиллы в этом проекте». Не абстрактные рекомендации, а реальные примеры из реальных скиллов.
