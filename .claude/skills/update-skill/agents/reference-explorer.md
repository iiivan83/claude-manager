# Reference Explorer

Найти все файлы проекта, которые ссылаются на целевой скилл, и определить, какие из них нужно обновить при изменении скилла.

## Зачем это нужно

Скиллы не живут в изоляции. Один скилл может вызывать другой, зависеть от его поведения или упоминать его в документации. Если обновить скилл, не зная о таких связях, можно рассинхронизировать систему — вызывающий скилл будет ожидать старого поведения, а обновлённый уже работает по-новому.

## Входные данные

Ты получаешь эти параметры в промпте при запуске:

- **target_skill** — имя скилла, для которого ищем ссылки
- **skill_path** — путь к папке целевого скилла
- **project_root** — корень проекта
- **output_path** — куда сохранить результат

## Что и где искать

### 1. Все остальные скиллы проекта

Прочитай SKILL.md каждого скилла в `.claude/skills/` (кроме целевого). Ищи:
- Прямые вызовы через Skill tool (например, `skill: "target-skill"`)
- Упоминания в description (триггерные фразы, которые отсылают к целевому скиллу)
- Ссылки в теле — «для этого используй target-skill», «передай в target-skill»
- Упоминания в агентах скилла (`agents/*.md`) — иногда агенты вызывают другие скиллы

### 2. Файлы CLAUDE.md

- `~/.claude/CLAUDE.md` — глобальные правила пользователя
- `{project_root}/CLAUDE.md` — правила проекта (если существует)

Ищи правила, которые явно упоминают целевой скилл — например, правила триггеринга или маршрутизации.

### 3. AGENTS.md

`{project_root}/.claude/skills/AGENTS.md` — конвенции скиллов. Могут упоминать конкретные скиллы как примеры или исключения.

### 4. Глобальные справочники

`~/.claude/references/*.md` — проверь все файлы в этой директории. Справочники могут содержать правила, связанные с поведением скилла.

### 5. Саморефлексия

Проверь, ссылается ли целевой скилл на самого себя — например, рекурсивный вызов из агента, или `description` упоминает поведение, которое может измениться.

## Как определить тип связи

- **call** — файл явно вызывает скилл (через `Skill tool`, `claude -p` с упоминанием скилла, или отсылка «запусти update-skill»)
- **dependency** — файл зависит от поведения скилла (описывает workflow, где скилл участвует; или ожидает конкретный выходной формат)
- **mention** — файл просто упоминает скилл по имени (документация, описание, пример)
- **self** — скилл ссылается на самого себя

## Как определить requires_update

Ссылка требует обновления (`requires_update: true`), если:
- Файл вызывает скилл и передаёт ему параметры, которые могут измениться
- Файл описывает поведение скилла, которое будет обновлено
- Файл содержит правило триггеринга, которое может рассинхронизироваться с новым `description`
- Файл содержит границу ответственности («для X используй A, для Y — B»), которая может сдвинуться

Ссылка не требует обновления (`requires_update: false`), если:
- Просто упоминание в списке или примере
- Ссылка на имя скилла, а не на его поведение

## Результат

Сохрани JSON в `{output_path}` по схеме `01-reference-explorer.json` из `references/schemas.md`.

Для каждой найденной ссылки укажи:
- Полный путь к файлу
- Тип связи
- Конкретный фрагмент текста, где найдена ссылка (для контекста)
- Нужно ли обновлять файл и почему

Если ссылок не найдено — это нормальный результат. Сохрани пустой массив `references` и `total_references: 0`.
