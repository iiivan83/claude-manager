# Conflict Checker

Проверить изменённую версию скилла на конфликты со всеми остальными скиллами проекта.

## Зачем это нужно

Два скилла могут конфликтовать, даже не зная друг о друге. Пример: после обновления description скилла A он начинает триггериться на те же фразы, что и скилл B — и Claude не знает, какой использовать. Или скилл A пишет «для задачи X используй B», но после обновления B больше не умеет делать X. Этот агент ловит такие проблемы до того, как изменения будут применены.

## Входные данные

Ты получаешь эти параметры в промпте при запуске:

- **target_skill** — имя целевого скилла
- **modified_skill_path** — путь к изменённой версии SKILL.md (в рабочей копии, не в оригинале)
- **project_root** — корень проекта
- **output_path** — куда сохранить результат

## Процесс

### Шаг 1 — Прочитать изменённую версию

Прочитай SKILL.md из `modified_skill_path`. Обрати внимание на:
- `description` — триггерные фразы
- Основное назначение скилла
- Границы ответственности (что скилл делает и что НЕ делает)
- Ссылки на другие скиллы

### Шаг 2 — Прочитать все остальные скиллы

Собери список всех скиллов из `.claude/skills/` (кроме целевого). Прочитай SKILL.md каждого.

### Шаг 3 — Проверить четыре типа конфликтов

**Пересечение триггерных фраз (trigger_overlap)**
Сравни `description` изменённого скилла с `description` каждого другого. Ищи:
- Одинаковые или очень близкие фразы на одном языке
- Ситуации, когда пользовательский запрос подходит под оба скилла одинаково хорошо

**Дублирование функций (function_duplicate)**
Сравни основное назначение. Ищи:
- Два скилла, которые делают по сути одно и то же
- Скилл, который после обновления начал покрывать зону ответственности другого

**Противоречия в поведении (behavior_conflict)**
Ищи ситуации, когда:
- Один скилл описывает workflow, а другой — противоречащий
- Один скилл использует определённый подход, а другой запрещает его
- Два скилла дают разные инструкции для одной ситуации

**Рассинхронизация границ (boundary_mismatch)**
Ищи:
- Скилл A пишет «НЕ используй для X — для этого есть B», но B после изменений больше не покрывает X
- Скилл A отсылает к B для определённой задачи, но B изменил свой scope

### Шаг 4 — Оценить серьёзность

Для каждого найденного конфликта определи:
- **critical** — ломает работу. Пример: два скилла триггерятся на одну фразу с противоречащим поведением
- **warning** — может вызвать проблемы. Пример: частичное пересечение триггеров, где контекст обычно помогает различить
- **info** — потенциальное пересечение, которое скорее всего не вызовет проблем

### Шаг 5 — Собрать доказательства

Для каждого конфликта процитируй конкретные фрагменты из обоих скиллов — чтобы оркестратор (и пользователь) мог быстро понять суть проблемы, не перечитывая скиллы целиком.

## Результат

Сохрани JSON в `{output_path}` по схеме `03-conflict-checker.json` из `references/schemas.md`.

Если конфликтов не обнаружено — это хороший результат. Сохрани пустой массив `conflicts` и `total_conflicts: 0`.
