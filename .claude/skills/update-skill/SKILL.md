# Update Skill

Пайплайн безопасного обновления скиллов. Вместо слепой правки — исследует зависимости, планирует изменения, применяет их к рабочей копии, тестирует старую версию против новой, и только после положительной оценки заменяет оригинал.

## Global Reference Rules

Перед началом работы прочитай:
- `~/.claude/references/document-naming-and-placement.md` — правила документооборота (куда складывать логи, формат временных меток — разделы 2, 3.2)
- `~/.claude/references/writing-style-guide.md` — единый стиль письма для всех документов
- `~/.claude/references/skill-testing-standard.md` — стандарт качества assertions и протокол сравнительного тестирования
- `~/.claude/references/cli-testing-patterns.md` — паттерны тестирования через CLI-подпроцесс (изоляция, таймауты, отладка)

Из `document-naming-and-placement.md` извлеки:
- **Раздел 2** — формат временной метки `{DD.MM_HH.MM}`. Сгенерируй метку ОДИН раз при старте
- **Раздел 3.2** — правила размещения логов. Подкатегория этого скилла: `skills-modifications`

Если файлы не найдены — остановить пайплайн с ошибкой.

## Принципы написания скиллов

Ты будешь вносить изменения в скиллы — документы, которые читает LLM. Держи в голове:
- Объясняй зачем что-то нужно, а не просто диктуй. Модели лучше следуют инструкциям, когда понимают причину
- Избегай жёстких ALWAYS/NEVER в caps — вместо этого объясни, почему конкретное поведение важно
- Пиши обобщённо, не затачивай под конкретные примеры — скилл должен работать в разных ситуациях

## Глубина мышления

Думай как опытный тех-лид, который лично отвечает за стабильность продакшена. Каждое решение
проходит через внутренний чек-лист:

- **Побочные эффекты** — что может сломаться в соседних модулях из-за этого изменения?
- **Граничные случаи** — как это поведёт себя на пустых данных, при конкурентном доступе,
  при неожиданном порядке вызовов?
- **Откат** — если изменение окажется неудачным, как быстро можно вернуть всё назад?
- **Альтернативы** — есть ли более простой способ добиться того же результата?
- **Достаточность** — это изменение действительно решает корневую причину, а не маскирует
  симптом?

Все агенты этого скилла работают с такой же глубиной — разбирают задачу по частям,
рассматривают альтернативы и перепроверяют выводы перед тем, как действовать.

**Проброс в создаваемые скиллы:** при изменении скиллов — всегда проверяй наличие секции
«Глубина мышления» с полным чек-листом (побочные эффекты, граничные случаи, откат,
альтернативы, достаточность). Если секции нет или она упрощённая — добавь/обнови до
полной версии по эталону из `.claude/skills/apply-root-cause-fixes/SKILL.md`.

## Жёсткие правила пайплайн-скиллов

Эти четыре правила обязательны для любого пайплайн-скилла (три этапа и больше) — и при создании нового, и при обновлении существующего. Нарушение хотя бы одного — это брак, переделывай в том же проходе.

Правила появились после root-cause-анализа `google-sheets-etl` — скилл тихо пропускал LLM-этап и отдавал базу с пустыми метаданными. Причина каждый раз была одна: скилл оставлял выбор режима «на усмотрение агента», а агент ошибался. Эти правила закрывают саму возможность такой ошибки.

### 1. Один режим на этап

У каждого этапа должен быть ровно один режим работы. Если этап делает «либо A, либо B» в зависимости от входных данных — это два разных этапа, а выбор между ними идёт через отдельный этап триажа с детерминированными условиями в SKILL.md.

Запрещено:
- «Этап нормализации работает либо через LLM-агента, либо через эвристику»
- «Скрипт сам решит, какой режим выбрать»
- «По умолчанию LLM, но можно переключиться на эвристику»

Разрешено:
- «Этап 3: LLM-анализ — единственный режим»
- «Этап 3a — триаж (детерминированные условия в SKILL.md). Этап 3b — LLM или эвристика, выбор определён триажом»

### 2. Никаких опциональных дорогих этапов

Если этап влияет на корректность результата — он обязательный. Слова «опциональный», «по желанию», «можно пропустить для скорости», «на выбор» в описании этапа запрещены.

Запрещено:
- `**Тип:** script + **опциональный** LLM-агент`
- «Можно пропустить для ускорения»
- «По желанию запустить этап N»

Разрешено:
- `**Тип:** script + LLM-агент (оба обязательны)`
- Если этап реально не влияет на корректность — удали его или вынеси в отдельный вспомогательный скилл

### 3. Никаких тихих fallback

Если ожидаемый аргумент, файл или ответ агента отсутствует — скрипт обязан упасть с понятной ошибкой. Запрещено молча возвращать `None`, пустую структуру или переключаться на «режим попроще».

Запрещено в коде:
```python
if analysis_path is None:
    return None  # тихий fallback на эвристики

try:
    result = call_llm(...)
except Exception:
    result = heuristic_fallback(...)  # замаскированная ошибка
```

Разрешено:
```python
if analysis_path is None:
    raise SystemExit("--analysis-file обязателен. LLM-анализ — единственный поддерживаемый режим")

try:
    result = call_llm(...)
except LLMError as e:
    raise SystemExit(f"LLM-агент упал на листе {sheet_name}: {e}")
```

### 4. Запрет мягких формулировок

В описании режимов и обязательности этапов запрещены формулировки, которые оставляют решение «на усмотрение Клода»:

Запрещено:
- «Лучше использовать LLM для нестандартных листов»
- «Обычно эвристика хороша для простых таблиц»
- «Рекомендуется выбирать LLM-агента, если...»
- «По возможности прогнать через LLM»

Разрешено только детерминированные условия и прямые утверждения:
- «Если листов > 10 ИЛИ merged_cells > 5 ИЛИ в первых 10 строках есть слова из набора {P&L, Бюджет, Факт, План} — LLM-режим обязателен»
- «LLM-анализ — единственный поддерживаемый режим этого этапа»

## Обязательная последовательность переходов

- Этап 0 (подготовка) → Этап 1 (исследование)
- Этап 1 → Этап 2 (планирование)
- Этап 2 → Этап 3 (копирование и применение)
- Этап 3 → Этап 4 (сравнительное тестирование)
- Этап 4 → Этап 5 (оценка)
- Этап 5 (ACCEPTED) → Этап 6 (применение)
- Этап 5 (REJECTED) → Этап 3 (починка, максимум 2 попытки)

## Pipeline Stages

**Паттерн логирования:** запись в orchestrator-log.json делается ДО начала каждого этапа (не после). Это позволяет видеть, на каком этапе пайплайн прервался, если он упал.

### Этап 0 — Подготовка

**Кто делает:** оркестратор напрямую

**0.1 — Найти скилл**

Путь: `.claude/skills/` в корне проекта.

Нечёткий поиск:
- Точное имя: `commit`, `create-doc`
- Частичное совпадение: `review` → если одно совпадение, взять; если несколько — спросить
- На русском: `коммит` → `commit`, `создай документ` → `create-doc`

Нет совпадений → показать список, спросить.

**0.2 — Создать папку логов**

1. Сгенерировать временную метку `{DD.MM_HH.MM}`
2. Создать папку: `dev/docs/logs/skills-modifications/{DD.MM_HH.MM}-update-skill-{имя-скилла}/`
3. Внутри создать подпапки: `agent-outputs/`, `eval-workspace/`, `backup/`
4. Создать `orchestrator-log.json`

**0.3 — Оценка бюджета**

Оцени примерный бюджет CLI-вызовов: 2 агента-исследователя + планировщик + conflict-checker + тестовые прогоны + evaluator + session-report. При стандартном обновлении одного скилла — ~15-20 USD.

### Этап 1 — Исследование (два агента параллельно)

Два агента работают одновременно — они не зависят друг от друга.

**Агент 1: reference-explorer**

- Инструкция: `agents/reference-explorer.md`
- Задача: найти все ссылки на целевой скилл по всему проекту
- Выход: `agent-outputs/01-reference-explorer.json`
- Инструмент: Agent tool

Промпт для агента:
```
Прочитай инструкцию из {skill_path}/agents/reference-explorer.md и выполни задачу.

Параметры:
- target_skill: {имя-скилла}
- skill_path: {путь-к-скиллу}
- project_root: {корень-проекта}
- output_path: {папка-логов}/agent-outputs/01-reference-explorer.json
```

**Агент 2: pattern-explorer**

- Инструкция: `agents/pattern-explorer.md`
- Задача: изучить паттерны и стиль других скиллов проекта
- Выход: `agent-outputs/01-pattern-explorer.json`
- Инструмент: Agent tool

Промпт для агента:
```
Прочитай инструкцию из {skill_path}/agents/pattern-explorer.md и выполни задачу.

Параметры:
- target_skill: {имя-скилла}
- project_root: {корень-проекта}
- output_path: {папка-логов}/agent-outputs/01-pattern-explorer.json
```

Оба агента запускаются в одном ходу через Agent tool. Записать шаги в orchestrator-log.

### Этап 2 — Планирование

- Инструкция: `agents/change-planner.md`
- Вход: результаты обоих исследователей + описание изменений от пользователя
- Выход: `agent-outputs/02-change-planner.json`
- Инструмент: Agent tool
- Зависимости: Этап 1

Промпт для агента:
```
Прочитай инструкцию из {skill_path}/agents/change-planner.md и выполни задачу.

Параметры:
- target_skill: {имя-скилла}
- skill_path: {путь-к-целевому-скиллу}
- requested_changes: {описание изменений от пользователя}
```

**Чекпоинт для пользователя** — после завершения планировщика проверь `ripple_effects`:

Если есть файлы с `requires_update: true`, покажи пользователю и спроси:
- **Вариант А** — скорректировать план с учётом зависимостей
- **Вариант Б** — обновить целевой скилл и все затронутые файлы за один проход
- **Вариант В** — пользователь разберётся сам позже

Это последний момент перед тем, как мы начнём что-то менять. Если нет ripple_effects — переходи молча.

### Этап 3 — Резервное копирование и применение изменений

**Кто делает:** 3.1 и 3.2 — оркестратор (`cp -r`); 3.3 и 3.4 — субагенты через Agent tool; 3.5 — bash-скрипт. Оркестратор сам применением плана больше не занимается — см. объяснение в подсекции 3.3.

**3.1 — Резервная копия:**
```bash
cp -r .claude/skills/{имя-скилла}/ {папка-логов}/backup/{имя-скилла}/
```
Это страховка и одновременно «старая версия» для сравнительного тестирования.

**3.2 — Рабочая копия:**
```bash
cp -r .claude/skills/{имя-скилла}/ {папка-логов}/eval-workspace/new-version/{имя-скилла}/
```
Все изменения вносятся СЮДА, оригинал пока не трогаем.

**3.3 — Применение изменений:**

**Агент: change-applier**

- Инструкция: `agents/change-applier.md`
- Вход: спецификация из `02-change-planner.json` + путь к рабочей копии
- Выход: `agent-outputs/03-change-applier.json`
- Инструмент: Agent tool

**Зачем отдельный агент:** применение плана требует чтения всех исходных файлов и записи всех новых. Если это делает оркестратор напрямую, его контекст линейно растёт от размера плана — большой план (20+ changes, несколько новых файлов) занимает тысячи строк и выжигает контекст ещё до Этапа 4. Вынос в `change-applier` изолирует память: субагент держит содержимое файлов у себя, оркестратор получает только короткий JSON-отчёт. Это применяется ко всем планам без исключения — никакого порога «если changes < N делай сам» (это нарушило бы правило «Один режим на этап» из раздела «Жёсткие правила пайплайн-скиллов»).

Для записи в файлы внутри `.claude/skills/` из CLI-подпроцесса `change-applier` использует штатные шаблоны X.1/X.2 из корневого `CLAUDE.md` проекта (раздел «Запись в `.claude/skills/` из CLI-подпроцессов»). Подробности — в `agents/change-applier.md`.

Промпт для агента:
```
Прочитай инструкцию из {skill_path}/agents/change-applier.md и выполни задачу.

Параметры:
- target_skill: {имя-скилла}
- plan_path: {папка-логов}/agent-outputs/02-change-planner.json
- workspace_path: {папка-логов}/eval-workspace/new-version/{имя-скилла}/
- output_path: {папка-логов}/agent-outputs/03-change-applier.json
```

После завершения агента прочитай только JSON-отчёт из `03-change-applier.json`. Содержимое рабочей копии оркестратор НЕ читает — это сломает весь смысл делегирования. Если нужно проверить содержимое — это делают 3.4 (conflict-checker) и 3.5 (structural-validator), у них свой изолированный контекст.

Проверь в отчёте:
- `failed_changes` равно 0. Если нет — покажи пользователю список ошибок из `errors` и предложи починить план (возврат к Этапу 2) либо принять как есть
- `touched_files` соответствует ожиданиям плана. Если агент тронул неожиданные файлы — сигнал к внимательной проверке на шагах 3.4 и 3.5

**3.4 — Проверка конфликтов:**

**Агент: conflict-checker**

- Инструкция: `agents/conflict-checker.md`
- Вход: изменённая версия из рабочей копии + все остальные скиллы
- Выход: `agent-outputs/03-conflict-checker.json`
- Инструмент: Agent tool

Промпт для агента:
```
Прочитай инструкцию из {skill_path}/agents/conflict-checker.md и выполни задачу.

Параметры:
- target_skill: {имя-скилла}
- modified_skill_path: {папка-логов}/eval-workspace/new-version/{имя-скилла}/SKILL.md
- project_root: {корень-проекта}
- output_path: {папка-логов}/agent-outputs/03-conflict-checker.json
```

Если найдены конфликты с severity `critical` или `warning`:
- Покажи пользователю
- Предложи варианты А/Б/В (как в Этапе 2)

Если конфликтов нет или только `info` — переходи дальше молча.

**3.5 — Структурная валидация:**

Прогони structural-validator по рабочей копии:
```bash
bash .claude/skills/pipeline-implementer/scripts/structural-validator.sh {папка-логов}/eval-workspace/new-version/{имя-скилла}
```

Критические проверки (FAIL) — исправь перед тестированием. Warnings — зафиксируй в orchestrator-log, но не блокируй переход к Этапу 4.

### Этап 4 — Сравнительное тестирование

Прочитай `~/.claude/references/skill-testing-standard.md` и выполни Протокол сравнительного тестирования (Блок B) с параметрами:

- `scenario`: `"skill_update"`
- `eval_workspace`: `{папка-логов}/eval-workspace`
- `config_a_name`: `"new_skill"`
- `config_a_skill_path`: `{папка-логов}/eval-workspace/new-version/{имя-скилла}/SKILL.md`
- `config_b_name`: `"old_skill"`
- `config_b_skill_path`: `{папка-логов}/backup/{имя-скилла}/SKILL.md`
- `evals_source`: `.claude/skills/{имя-скилла}/evals/evals.json`
- `skill_name`: `{имя-скилла}`

**Важно:** по протоколу skill-testing-standard.md (Шаг T2) все тестовые прогоны запускаются одновременно через `run_in_background`. Не запускай их последовательно — это удваивает время.

**Scope assertions:** при вызове грейдера для каждого прогона передавай `config_name` (имя конфигурации: "new_skill" или "old_skill"). Грейдер должен пропускать assertions с `scope: "new_only"` при грейдинге old_skill — они проверяют только новое поведение и не применимы к старой версии.

После завершения протокола — переходи к Этапу 5 (оценка и решение).

Записать шаг в orchestrator-log.json: `stage: "comparative-testing"`.

**Чекпоинт:** Этап 4 завершён. НЕ переходи к Этапу 6 напрямую — обязательно пройди оценку (Этап 5).

### Этап 5 — Оценка и решение

**Агент: update-evaluator**

- Инструкция: `agents/update-evaluator.md`
- Вход: чек-лист ожиданий, бенчмарк, анализ, все grading.json
- Выход: `agent-outputs/04-update-evaluator.json`
- Инструмент: Agent tool

Промпт для агента:
```
Прочитай инструкцию из {skill_path}/agents/update-evaluator.md и выполни оценку.

Параметры:
- target_skill: {имя-скилла}

- benchmark_path: {папка-логов}/eval-workspace/benchmark.json
- analysis_path: {папка-логов}/eval-workspace/analysis.json
- grading_dir: {папка-логов}/eval-workspace
- output_path: {папка-логов}/agent-outputs/04-update-evaluator.json
```

**Действия по вердикту:**

**ACCEPTED** → переходи к Этапу 6

**ACCEPTED_WITH_REGRESSIONS** → покажи пользователю:
- Какие регрессии найдены
- Насколько они критичны
- Пользователь решает: принять или доработать

**REJECTED** → покажи пользователю и предложи:
- **Починить** — вернуться к Этапу 3.3 со скорректированным планом (максимум `MAX_FIX_ATTEMPTS` попыток)
- **Принять как есть** — ручной override, пользователь берёт ответственность
- **Отменить** — оригинал не тронут, ничего не потеряно

Записать шаг в orchestrator-log.json: `stage: "evaluation"`, verdict в comment.

**Чекпоинт:** Этап 5 завершён. Действуй строго по вердикту — ACCEPTED → Этап 6, REJECTED → Этап 3.

### Этап 6 — Применение и финализация

**Кто делает:** оркестратор напрямую

Записать шаг в orchestrator-log.json ДО начала: `stage: "finalization"`.

**6.1 — Замена:**
```bash
cp -r {папка-логов}/eval-workspace/new-version/{имя-скилла}/ .claude/skills/{имя-скилла}/
```

**6.2 — Каскадные обновления:**
Если пользователь выбрал Вариант Б на чекпоинте Этапа 2 — обнови все затронутые файлы из `ripple_effects`.

**6.3 — Отчёт:**
Покажи пользователю diff — было vs стало (только изменённые фрагменты SKILL.md).

**6.4 — Сессионный отчёт:**
Запусти агента для создания сессионного отчёта (обязательно для пайплайнов, правило из AGENTS.md).

Промпт агенту:
```
Используй /session-report.

Описание изменений: {краткое описание из requested_changes}
```

**6.5 — Коммит:**

Эта фаза выполняется только при прямом вызове пользователем. Если скилл запущен как агент из другого скилла — коммит пропускается, потому что вызывающий скилл сам управляет коммитами.

Шаги:
1. `git status` — проверить наличие изменений
2. `git add .claude/skills/{имя-скилла}/` — только файлы обновлённого скилла
3. Коммит: `fix(skills): обновлён {имя-скилла} — {краткое описание}`
4. Сообщить пользователю что коммит создан
