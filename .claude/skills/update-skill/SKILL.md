Обновление существующих скиллов. Только модификация — не создание.

Документы скиллов читает LLM. Стиль: объясняй зачем что-то нужно, а не просто приказывай. Избегай жёстких ALWAYS/NEVER в caps — лучше объясни причину, чтобы модель поняла и сама приняла верное решение. Пиши обобщённо, не сужай до конкретных примеров.

## Вход

Аргументы: $ARGUMENTS

Из аргументов и контекста беседы извлечь:
- Целевые скиллы (один или несколько)
- Суть изменений

Спрашивать пользователя только если невозможно определить скилл или суть изменений из контекста.

## Алгоритм

### 1. Определить скиллы

Путь: `.claude/skills/` в корне проекта.

Нечёткий ввод:
- Точное имя: `commit`, `make-todo`
- Частичное: `review` → если одно совпадение, взять его; если несколько — спросить
- На русском: `коммит` → `commit`

Нет совпадений → вывести список, спросить.

### 2. Изучить контекст проекта

Перед тем как вносить изменения, разберись в предметной области обновляемого скилла. Прочитай индекс документации `development/docs/docs-index.md` — это оглавление всех документов проекта. По нему выбери и прочитай те документы, которые относятся к теме скилла. Это нужно, чтобы ты понимал специфику проекта и мог вносить изменения, которые учитывают реальную архитектуру, терминологию и контекст — а не работал вслепую.

Дополнительно: если обновляемый скилл работает с HTTP-запросами, серверами, API, деплоем или инфраструктурой — прочитай также `development/docs/migration_strategy.md` и `development/docs/architecture-spec.md`. Эти документы описывают, как устроена система: какие серверы существуют, как PHP и Django работают параллельно, какие URL и порты используются, как маршрутизируются запросы. Без этого контекста можно внести изменения, которые не учитывают архитектуру проксирования или используют неправильные адреса.

### 3. Проверить затронутые скиллы и документы

Перед внесением изменений — убедись, что понимаешь полную картину. Изменение одного скилла может создать рассинхронизацию с другими скиллами или документами, которые ссылаются на него или зависят от его поведения. Был реальный инцидент: стратегию обновили в одном скилле, а скилл-исполнитель не поддерживал новые требования — конвейер сломался.

Что проверить:
1. Найди все скиллы, которые вызывают или ссылаются на обновляемый скилл (поиск по `.claude/skills/` по имени скилла)
2. Найди документы в `development/docs/`, которые описывают поведение обновляемого скилла
3. Прочитай найденные файлы и убедись, что планируемые изменения не противоречат тому, что написано в связанных скиллах и документах
4. Если противоречие обнаружено — сообщи пользователю и предложи обновить все затронутые места за один проход

Зачем это нужно: скиллы работают как цепочка — оркестратор вызывает исполнителей. Если обновить исполнителя, но не обновить оркестратор (или наоборот), система будет работать непредсказуемо. Проверка связей до внесения изменений предотвращает рассинхронизацию.

### 4. Полная проверка на противоречия со всеми скиллами

Шаг 3 проверяет только скиллы, которые напрямую ссылаются на обновляемый. Но есть класс проблем, которые так не ловятся: два скилла могут конфликтовать, даже не зная друг о друге — через пересечение триггеров, дублирование функционала или рассинхронизацию границ ответственности. Этот шаг закрывает этот пробел.

Что искать:
1. **Пересечение триггерных фраз** — два скилла триггерятся на одну и ту же фразу в description
2. **Дублирование функционала** — два скилла делают по сути одно и то же
3. **Противоречия в поведении** — один скилл описывает workflow, который конфликтует с другим
4. **Рассинхронизация границ** — скилл A пишет «НЕ используй для X — для этого есть B», но B после изменений больше не покрывает X

Алгоритм:
1. Собрать список всех скиллов из `.claude/skills/` (исключая обновляемый)
2. Разбить список на группы по 3 скилла
3. Запустить параллельных агентов — по одному на каждую группу. Задача агента: прочитать SKILL.md каждого скилла из своей группы, сравнить с планируемыми изменениями, вернуть список конфликтов (или сообщить, что конфликтов нет)
4. Собрать результаты всех агентов и отфильтровать нерелевантные — оставить только реальные противоречия

Если релевантные противоречия найдены — сообщить пользователю конкретный список и спросить, как поступить:
- **Вариант А**: скорректировать планируемые изменения с учётом найденных противоречий (пересмотреть формулировки, триггеры или границы)
- **Вариант Б**: внести изменения как есть, а затем запустить агентов, которые через скилл update-skill обновят все конфликтующие скиллы за один проход
- **Вариант В**: ничего не трогать, пользователь разберётся сам

Это чекпоинт — до этого момента целевой скилл ещё не изменён. Пользователь может передумать, скорректировать план или отменить обновление. Ничего не исправлять автоматически — только по выбору пользователя. Если противоречий не найдено — перейти к следующему шагу без вопросов.

### 5. Прочитать и изменить

Для каждого целевого скилла:
1. Прочитать `.claude/skills/<имя>/SKILL.md` целиком
2. Применить изменения через Edit — менять только указанные части
3. Сохранить формат frontmatter и структуру разделов

### 6. Проверка размещения результатов (если скилл генерирует файлы)

После внесения изменений — проверить, создаёт ли обновляемый скилл какие-либо файлы (отчёты, спеки, документы, JSON и т.д.). Если скилл генерирует выходные файлы:

1. Прочитать `development/docs/target-folder-structure.md` — там зафиксирована целевая структура папок проекта
2. Сопоставить пути, куда скилл кладёт результаты, с целевой структурой
3. Если пути не совпадают — скорректировать их в SKILL.md обновляемого скилла, чтобы результаты попадали в правильные папки

Зачем это нужно: структура папок проекта стандартизирована, и если скилл кладёт файлы не туда, другие скиллы и люди не смогут их найти. Единообразие путей — основа навигации по проекту.

### 7. Отчёт

Для каждого изменённого скилла показать diff: было → стало (только изменённые фрагменты).

### 8. Коммит изменений

Без коммита изменения существуют только в рабочей копии и могут потеряться при завершении сессии. Именно так были потеряны 8 SKILL-фиксов из 7 root-cause отчётов (инцидент 26-03-2026) — изменения в скиллах были внесены, но не закоммичены, и пропали.

Эта фаза выполняется только при прямом вызове пользователем. Если скилл запущен как агент из другого скилла (например, из `apply-root-cause-fixes`) — коммит пропускается, потому что вызывающий скилл сам управляет коммитами на своём уровне. Как определить: если ты работаешь внутри Agent-подпроцесса — коммит не нужен.

Шаги:
1. Проверить наличие изменений — `git status`. Если изменённых файлов нет — пропустить.
2. Добавить только файлы обновлённого скилла — `git add .claude/skills/<имя-скилла>/SKILL.md`. Не использовать `git add -A` или `git add .`, чтобы случайно не захватить посторонние изменения.
3. Создать коммит с сообщением формата: `fix(skills): обновлён скилл {имя-скилла} — {краткое описание изменений}`
4. Сообщить пользователю что коммит создан и какие файлы вошли.

## Правила

- Только обновление. Создание → skill-creator.
- Не менять имя директории.
- Стиль вносимых изменений: объясняй модели зачем что-то важно, а не просто диктуй. Избегай жёстких MUST/NEVER — вместо этого дай причину, чтобы модель сама поняла. Пиши обобщённо, не затачивай под конкретные примеры.
- При изменении `description` — сохранить триггерные фразы, если пользователь не просил их убрать.
- Если одно изменение применимо к нескольким скиллам — применить ко всем за один проход.
