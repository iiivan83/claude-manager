# Интеграционное тестирование проекта

**THINK HARD** на каждом шаге выполнения этого скилла. Используй максимальную глубину мышления при принятии каждого решения — от выбора тестовых сценариев до анализа ошибок.

Ты пишешь интеграционные тесты для проекта Claude Manager (Telegram-бот, управляющий Claude Code с телефона). Интеграционные тесты проверяют, что модули корректно работают **вместе** — в отличие от юнит-тестов, которые проверяют каждый модуль в изоляции. Это важный рубеж качества: модули могут прекрасно работать по отдельности, но ломаться при взаимодействии из-за несогласованных интерфейсов, неучтённых побочных эффектов или ошибок в порядке вызовов.

Тесты используют моки (фейковые объекты вместо настоящих) для внешних зависимостей (Telegram API, Claude CLI, файловая система), но **реальные связи** между модулями проекта. Это золотая середина: тесты быстрые и детерминированные, но ловят ошибки на стыках модулей.

## Шаг 1: Сбор контекста

**THINK HARD** — без глубокого понимания модулей невозможно написать осмысленные интеграционные тесты. Поверхностное чтение приведёт к тестам, которые проверяют очевидное и пропускают реальные проблемы.

Прочитай все источники параллельно (через Agent tool, если доступен):

### Агент 1: Спецификации и архитектура

Прочитай и запомни ключевые контракты между модулями:

- **Граф зависимостей** — `development/specs/module-dependency-graph.md`. Какие модули от каких зависят, в каких слоях находятся, какие CJM обслуживают. Если файл не существует — прочитай секцию про слои из CLAUDE.md
- **Спецификации реализованных модулей** — все файлы из `development/specs/realized/`. Обрати внимание на публичные API (какие функции модуль экспортирует), типы аргументов и возвращаемых значений, побочные эффекты (запись в файлы, изменение глобального состояния)

### Агент 2: Код модулей

Прочитай реальный код каждого модуля из `src/claude_manager/`. Для интеграционных тестов критически важно понимать:

- Как модули импортируют и вызывают друг друга
- Какие данные передаются между модулями (типы, структура)
- Где используется общее состояние (словари на уровне модуля, файлы JSON)
- Какие ошибки могут возникнуть на границах модулей

### Агент 3: Пользовательские сценарии и существующие тесты

- **BRD** — `development/docs/brd-user-journeys.md`. Каждый CJM (путь пользователя) описывает цепочку взаимодействий через несколько модулей — это готовые сценарии для интеграционных тестов
- **Существующие юнит-тесты** — `tests/test_*.py`. Что уже проверено на уровне отдельных модулей (интеграционные тесты не должны дублировать это). Также прочитай `tests/conftest.py` — какие фикстуры доступны

После чтения — сформируй ментальную карту: какие связки модулей задействуются в каждом пользовательском сценарии, где потенциальные точки отказа.

## Шаг 2: Планирование тестов по 5 группам

**THINK HARD** — каждая группа тестов должна проверять реальные сценарии взаимодействия, а не искусственные. Опирайся на BRD и код, а не на абстрактные представления.

### Группа 1: Жизненный цикл сессии

**Модули:** session_manager + daily_session_registry + process_manager

**Что проверяем** — полный цикл жизни сессии, как описано в CJM-04, CJM-05, CJM-06:

- Создание новой сессии через session_manager с одновременным резервированием дневного номера в daily_session_registry
- Переключение между сессиями — session_manager обновляет привязку chat_id -> session_id, daily_session_registry сохраняет нумерацию
- Мониторинг — session_manager отдаёт текущую сессию, process_manager проверяет состояние процесса
- Остановка — process_manager завершает процесс, session_manager обновляет состояние
- Обновление session_id — когда Claude возвращает реальный ID вместо временного, все словари во всех модулях обновляются согласованно

### Группа 2: Путь сообщения

**Модули:** bot + claude_runner + process_manager + message_splitter

**Что проверяем** — полный путь от приёма сообщения до отправки ответа (CJM-02):

- Пользователь отправляет сообщение -> bot определяет сессию -> claude_runner передаёт в Claude -> ответ возвращается -> message_splitter разбивает при необходимости -> bot отправляет в Telegram
- Длинный ответ (> 4096 символов) корректно разбивается и все части отправляются
- Пустой ответ от Claude — пользователь получает информативное сообщение
- Ошибка от Claude — пользователь получает сообщение об ошибке, retry-механизм работает
- Промежуточные обновления (thinking) доставляются пользователю с правильным форматированием

### Группа 3: Работа с файлами

**Модули:** bot + claude_runner

**Что проверяем** — отправка фотографий и файлов (CJM-03):

- Фото с подписью — bot скачивает файл, сохраняет на диск, формирует правильный промпт для Claude с путём к файлу и текстом подписи
- Фото без подписи — формируется промпт с просьбой описать изображение
- Документ без подписи — формируется промпт с просьбой описать содержимое
- Ошибка скачивания — пользователь получает сообщение об ошибке, Claude не вызывается

### Группа 4: Координация watchers

**Модули:** session_watcher + bot

**Что проверяем** — механизм паузы наблюдателя при активном запросе (чтобы ответ не дублировался):

- Когда bot отправляет сообщение в Claude, watcher для этой сессии ставится на паузу
- После получения ответа от Claude, watcher возобновляется
- Watcher для **других** сессий продолжает работать без паузы во время запроса
- Нет дублирования ответов — один и тот же ответ не приходит пользователю дважды (от watcher и от handle_message)
- Если handle_message завершается с ошибкой, watcher всё равно возобновляется (пауза не зависает навечно)

### Группа 5: Конкурентный доступ

**Модули:** session_manager + daily_session_registry

**Что проверяем** — защита данных при одновременных операциях:

- Два одновременных вызова создания сессии — каждая получает уникальный номер, файл sessions.json не повреждён
- Одновременное чтение и запись в daily_sessions.json — блокировка не допускает повреждения данных
- Атомарная запись — при сбое во время записи файл остаётся в целостном состоянии (через tmp + rename)
- Параллельные переключения сессий — результат предсказуем, последнее переключение побеждает

## Шаг 3: Создание тестов

**THINK HARD** — структура тестов должна быть ясной, фикстуры переиспользуемыми, а каждый тест — проверять одно конкретное взаимодействие.

### 3.1 Структура файлов

Создай папку `tests/integration/` и следующие файлы:

- `tests/integration/__init__.py` — пустой
- `tests/integration/conftest.py` — фикстуры, специфичные для интеграционных тестов. Импортирует и расширяет фикстуры из `tests/conftest.py`
- `tests/integration/test_session_lifecycle.py` — Группа 1
- `tests/integration/test_message_path.py` — Группа 2
- `tests/integration/test_file_handling.py` — Группа 3
- `tests/integration/test_watcher_coordination.py` — Группа 4
- `tests/integration/test_concurrent_access.py` — Группа 5

### 3.2 Правила написания тестов

Каждый тест следует этим принципам:

**Реальные связи, фейковые внешности.** Модули проекта импортируются и вызываются по-настоящему. Мокаются только внешние зависимости: Telegram Bot API, Claude CLI (subprocess), файловая система (через tmp_path). Это ключевое отличие от юнит-тестов, где мокается всё кроме тестируемого модуля.

**Один тест — одно взаимодействие.** Каждый тест проверяет одну конкретную цепочку вызовов между модулями. Не пытайся покрыть весь CJM одним тестом — разбей на отдельные проверки.

**Говорящие имена.** Имя теста должно описывать сценарий: `test_new_session_reserves_daily_number_and_creates_process`, а не `test_session_1` или `test_integration`.

**Async тесты.** Весь код бота асинхронный. Используй `@pytest.mark.asyncio` и `async def test_...`. Помни про `asyncio.to_thread()` для блокирующих операций.

**Изоляция между тестами.** Каждый тест начинает с чистого состояния. Используй tmp_path для файлов. Очищай глобальные словари модуля bot.py в фикстурах (если они используются).

**Фикстуры из conftest.py.** Используй существующие фикстуры: `mock_bot`, `mock_update`, `mock_claude_process`, `allowed_user_id`, `session_dir`. Если их нет или они не подходят — создай в `tests/integration/conftest.py`, но не дублируй существующие.

### 3.3 Написание кода тестов

Напиши тесты для каждой группы. Для каждого теста:

1. **Arrange** — подготовь состояние: создай фикстуры, настрой моки, инициализируй модули
2. **Act** — выполни действие, которое запускает цепочку взаимодействий между модулями
3. **Assert** — проверь результат на уровне каждого вовлечённого модуля

Ориентируйся на BRD при определении ожидаемого поведения. Спецификации модулей из `development/specs/realized/` определяют точные контракты API.

Минимум тестов по группам:

- **Группа 1 (жизненный цикл сессии):** 4-6 тестов
- **Группа 2 (путь сообщения):** 4-6 тестов
- **Группа 3 (работа с файлами):** 3-4 теста
- **Группа 4 (координация watchers):** 3-5 тестов
- **Группа 5 (конкурентный доступ):** 3-4 теста

## Шаг 4: Первый запуск тестов

**THINK HARD** — перед запуском перечитай написанные тесты и убедись, что они корректно импортируют модули и используют правильные API.

Запусти интеграционные тесты:

```bash
python -m pytest tests/integration/ -v
```

Зафиксируй результат: сколько прошло, сколько упало, какие ошибки.

## Шаг 5: Цикл починки

**THINK HARD** — на этом этапе важнее всего правильно определить **источник** ошибки: баг в коде модуля или баг в самом тесте. Неверная диагностика приведёт к «починке» работающего кода.

Если все тесты зелёные — переходи к Шагу 6.

Если есть ошибки — начинай цикл. Максимум 5 итераций.

### Для каждой ошибки:

1. **Прочитай traceback (трейсбек — цепочку вызовов, которая привела к ошибке)**

2. **Определи источник — THINK HARD:**
   - **Баг в тесте** — тест использует устаревший API модуля, неправильный тип данных, или проверяет поведение, которое не соответствует спецификации. Признаки: ImportError, AttributeError на функции/метода теста, assertion проверяет значение которое не описано в спеке
   - **Баг в коде модуля** — модуль нарушает свою спецификацию или несовместим с другими модулями. Признаки: ошибка происходит внутри кода модуля при вызове реального API, данные передаются между модулями в неожиданном формате

3. **Исправь:**
   - Если баг в тесте — исправь тест. Сверься со спецификацией модуля, чтобы убедиться в правильности исправления
   - Если баг в коде модуля — исправь код. Сверься со спецификацией, чтобы убедиться что исправление соответствует ТЗ

4. **Перезапусти ВСЕ тесты (юнит + интеграционные):**

Проверь, существует ли скрипт запуска:
```bash
ls /Users/ivan/Desktop/claude-sandbox/claude_manager/development/scripts/run-all-tests.sh
```

Если существует:
```bash
cd /Users/ivan/Desktop/claude-sandbox/claude_manager && ./development/scripts/run-all-tests.sh
```

Если не существует:
```bash
cd /Users/ivan/Desktop/claude-sandbox/claude_manager && python -m pytest tests/ -v
```

Это ключевое правило: при починке любого бага перегоняются ВСЕ тесты, не только сломанный. Починка интеграционного теста может сломать юнит-тест, починка кода модуля может создать регрессию в другом модуле. Единственный способ это поймать — прогнать всё.

5. **Если появились новые ошибки** — повтори цикл. Каждая итерация начинается с полного прогона всех тестов.

### Если 5 итераций исчерпаны

Если после 5 итераций починки тесты всё ещё не зелёные:

1. Зафиксируй состояние: какие тесты падают, какие ошибки, что было испробовано
2. Обнови `pipeline-state.json` со статусом `failed` и описанием ошибки
3. Сообщи пользователю: какие тесты не удалось починить и почему
