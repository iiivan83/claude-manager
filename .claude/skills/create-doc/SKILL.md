# Создание документа из диалога или контекста

Ты создаёшь структурированные документы, извлекая знания из двух возможных источников:
- **Вставленный диалог** — пользователь скопировал текст беседы и вставил в сообщение
- **Контекст текущей беседы** — пользователь работал в этой ветке и хочет оформить результаты

Оба сценария обрабатываются одинаково — разница только в том, откуда берётся исходная информация.

## Global Reference Rules

Перед началом работы прочитай:
- `~/.claude/references/writing-style-guide.md` — единый стиль письма для всех документов и отчётов
- `~/.claude/references/document-naming-and-placement.md` — правила документооборота: куда складывать файлы, как называть, формат временных меток

Если файлы не найдены — остановись и сообщи пользователю.

## Глубина мышления

Думай как опытный тех-лид, который лично отвечает за стабильность продакшена. Каждое решение
проходит через внутренний чек-лист:

- **Побочные эффекты** — что может сломаться в соседних модулях из-за этого изменения?
- **Граничные случаи** — как это поведёт себя на пустых данных, при конкурентном доступе,
  при неожиданном порядке вызовов?
- **Откат** — если изменение окажется неудачным, как быстро можно вернуть всё назад?
- **Альтернативы** — есть ли более простой способ добиться того же результата?
- **Достаточность** — это изменение действительно решает корневую причину, а не маскирует
  симптом?

Каждое решение в рамках этого скилла проходит через этот чек-лист — разбирай задачу
по частям, рассматривай альтернативы и перепроверяй выводы перед тем, как действовать.

## Главное правило: задавай вопросы

Если при анализе материала что-то непонятно, неоднозначно или вызывает сомнения — **обязательно спроси у пользователя** перед тем как писать документ. Не додумывай и не угадывай.

Примеры ситуаций, когда нужно спросить:
- Непонятно, какая из обсуждённых тем должна стать основной для документа
- В диалоге несколько разных тем — непонятно, один документ или несколько
- Есть противоречивая информация — непонятно, какая версия актуальна
- Непонятен уровень детализации — краткий обзор или подробный разбор
- Неясно назначение документа — для кого он и зачем

Лучше задать лишний вопрос, чем создать документ с неверным содержимым.

## Этап 0: Разведка проекта

Перед началом работы определи, где в этом проекте хранится документация.

### 0.1 Найди директорию документации

1. Прочитай `CLAUDE.md` в корне проекта — найди упоминания папки с документацией (обычно это что-то вроде `docs/`, `documentation/`, `dev/docs/` и т.п.)
2. Если в CLAUDE.md нет явного указания — поищи индексный файл документации: `docs-index.md`, `index.md` или аналог в корне и первых двух уровнях вложенности
3. Если индекс найден — директория документации = директория, где лежит индекс
4. Если ни CLAUDE.md, ни индекс не дали результата — спроси пользователя: «Не нашёл директорию документации проекта. Укажи путь к папке с документами и к индексному файлу (если есть).»

### 0.2 Установи переменные

По результатам разведки зафиксируй две переменные, которые используются во всех дальнейших шагах:
- **DOCS_DIR** — путь к директории документации (например: `dev/docs/`, `docs/`)
- **DOCS_INDEX** — путь к индексному файлу (например: `dev/docs/docs-index.md`). Может отсутствовать — тогда в Шаге 4 пропусти обновление индекса

Выведи результат пользователю:
```
Документация проекта:
- Директория: DOCS_DIR
- Индекс: DOCS_INDEX (или «не найден»)
```

## Шаг 1: Определи источник и извлеки знания

**THINK HARD** — это самый важный шаг. Внимательно проанализируй весь доступный материал.

### Определение источника
- Если в сообщении пользователя есть длинный текст (диалог, лог беседы, копия чата) — это **вставленный диалог**
- Если сообщение короткое (просьба создать документ без вложенного текста) — работай с **контекстом текущей беседы**

### Извлечение знаний

Пройдись по всему материалу и выдели:

1. **Решения** — что было решено? Какие варианты рассматривались и почему выбран именно этот?
2. **Факты** — конкретные числа, имена файлов, структуры, зависимости, технические детали
3. **Выводы** — к каким заключениям пришли в ходе обсуждения?
4. **Действия** — что нужно сделать дальше? Какие шаги запланированы?
5. **Контекст** — почему эта тема вообще обсуждалась? Какая проблема решалась?

Не включай в документ:
- Промежуточные неудачные попытки (только если они важны как «что НЕ работает»)
- Технический шум (ошибки инструментов, повторные запуски)
- Личные реплики, не относящиеся к сути

## Шаг 2: Назови документ

Имя файла должно сразу объяснять содержимое документа без необходимости его открывать.

### Правила именования
- Строчные буквы, snake_case: `module_analysis_courier.md`
- Формат: `{тема}_{тип}.md` или `{тема}_{подтема}.md`
- Имя должно отвечать на вопрос: «Что я найду внутри?»

### Примеры хороших имён
- **`courier_module_analysis.md`** — анализ модуля курьеров
- **`api_architecture_decision.md`** — решение по архитектуре API
- **`auth_upgrade_strategy.md`** — стратегия обновления авторизации
- **`caching_approaches_comparison.md`** — сравнение подходов к кешированию

### Примеры плохих имён
- `doc1.md` — непонятно что внутри
- `notes.md` — слишком общее
- `analysis.md` — анализ чего?

## Шаг 3: Создай документ

### Структура документа

Документ оптимизирован для чтения Claude в будущих беседах. Это техническая документация — никакой воды, только полезная информация.

```markdown
# Заголовок документа (понятный, описательный)

## Контекст

Зачем создан этот документ, какая задача стояла. 1-2 предложения.
Дата: DD-MM-YYYY

## Основное содержание

(Техническое описание — структура зависит от темы)

## Следующие шаги

Если есть запланированные действия — перечисли их.
```

### Принципы форматирования для Claude

Цель — максимальная плотность полезной информации. Claude будет читать этот документ чтобы быстро разобраться в теме и начать работу.

- **Заголовки** — чёткая иерархия H1 → H2 → H3, каждая секция самодостаточна
- **Списки** — маркированные для перечислений, нумерованные для последовательностей
- **Структурированные данные** — буллеты с жирными метками для модулей, зависимостей, статусов (не таблицы)
- **Код** — для имён файлов, команд, фрагментов кода (с указанием языка)
- **Конкретика** — вместо «некоторые модули» пиши «модули Courier, Salary, Contract»
- **Без воды** — никаких вводных фраз, пересказов, повторений. Каждое предложение несёт новую информацию
- **Без дублирования** — каждый факт упоминается один раз в наиболее подходящем месте

### Размещение

По умолчанию документ размещается в DOCS_DIR (определённый в Этапе 0). Если из контекста очевидно, что пользователь просит разместить документ в другой папке, следуй инструкции пользователя

Создай файл с помощью инструмента Write.

## Шаг 4: Обнови индекс документации

Если DOCS_INDEX найден в Этапе 0 — прочитай его и добавь блок для нового документа в таком формате:

```markdown
## DOCS_DIR/имя_файла.md
Читать если:
- Условие 1 (когда этот документ будет полезен)
- Условие 2
- Условие 3
```

Условия должны быть конкретными и помогать быстро решить — нужен ли этот документ для текущей задачи.

Если DOCS_INDEX не найден — пропусти этот шаг и отметь в итоговом отчёте.

## Шаг 5: Верификация — сверка с источником

**THINK HARD** — это критически важный шаг. Не пропускай его.

Перечитай исходный материал (диалог или контекст беседы) и созданный документ. Проверь по пунктам:

1. **Полнота** — все ключевые факты, решения и выводы из источника попали в документ?
2. **Точность** — числа, имена, пути к файлам, технические детали перенесены без искажений?
3. **Нет выдумок** — в документе нет информации, которой не было в источнике?
4. **Структура** — документ логично организован, секции следуют в понятном порядке?
5. **Самодостаточность** — документ понятен без необходимости читать исходный диалог?

Если нашёл расхождения — исправь документ.

Выведи краткий отчёт о верификации:
```
Верификация:
- Полнота: ✅/⚠️ (что пропущено, если есть)
- Точность: ✅/⚠️ (что неточно, если есть)
- Нет выдумок: ✅/⚠️
- Структура: ✅/⚠️
- Самодостаточность: ✅/⚠️
```

## Шаг 6: Проверка на стратегическую важность (обновление CLAUDE.md)

**THINK HARD** — этот шаг выполняется ТОЛЬКО в исключительных случаях. Порог очень высокий.

После создания документа задай себе вопрос: **«Есть ли в этом документе информация, без которой Claude будет работать НЕПРАВИЛЬНО почти в каждой сессии?»**

### Критерии для добавления в CLAUDE.md (должны выполняться ВСЕ три):

1. **Фундаментальность** — это меняет базовые правила работы с проектом (например: сменился фреймворк, изменилась структура БД, появился новый обязательный процесс для всего кода)
2. **Универсальность** — это актуально для 80%+ задач, а не для одного модуля или одной темы
3. **Невыводимость** — это НЕЛЬЗЯ узнать, просто прочитав код или документы в DOCS_DIR — нужно знать заранее, до начала работы

### Примеры того, что НАДО добавить в CLAUDE.md:
- Смена основного фреймворка или языка (например: «Бэкенд переписан с PHP на Python»)
- Принципиальное изменение архитектуры (например: «Все API теперь идут через API Gateway»)
- Новое глобальное правило для всего кода (например: «Все новые модули пишутся только на Python»)

### Примеры того, что НЕ НАДО добавлять в CLAUDE.md:
- Детали конкретного модуля — для этого есть документы в DOCS_DIR
- Решения по конкретной задаче — для этого есть документ задачи
- Прогресс по конкретной области — для этого есть соответствующий документ в DOCS_DIR
- Всё, что можно узнать из кода или существующих документов

### Если информация подходит:

1. Прочитай текущий `CLAUDE.md`
2. Найди подходящий раздел (или создай новый, если нет подходящего)
3. Добавь **1-2 строки** — максимально кратко, только суть
4. Убедись, что не дублируешь то, что уже есть в CLAUDE.md
5. Спроси у пользователя подтверждение: «Эта информация настолько важна, что я хочу добавить её в CLAUDE.md (файл, который читается в каждой сессии): [текст]. Добавить?»

**Важно:** если сомневаешься — НЕ добавляй. Лучше пропустить, чем засорить CLAUDE.md. Этот файл должен оставаться компактным.

## Шаг 7: Итоговый отчёт

```
✅ Документ создан: путь/к/файлу.md
📋 DOCS_INDEX: обновлён / пропущен (индекс не найден)
🔍 Верификация: пройдена
📌 CLAUDE.md: обновлён / не требуется
```
