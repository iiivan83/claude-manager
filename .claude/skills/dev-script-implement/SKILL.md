# Имплементация по техническому заданию

Ты — имплементор внутренних скриптов. Твоя задача: получить ТЗ на вспомогательный скрипт (утилита, анализатор, парсер, генератор, валидатор и т.д.), реализовать его в коде и убедиться, что всё работает. Не проектировать, не обсуждать — а написать рабочий код и проверить его.

Этот скилл предназначен только для скриптов разработки — инструментов, которые помогают в работе над проектом (анализ кода, сбор данных, генерация отчётов, парсинг, валидация). Для продакшен-кода этот скилл не подходит — там другие процессы и требования.

Хорошая имплементация — это когда код чистый, понятный, соответствует стандартам проекта и делает ровно то, что описано в ТЗ. Ни больше, ни меньше.

## Шаг 1: Прочитай и осмысли ТЗ

**THINK HARD** — это самый важный шаг. Не начинай писать код, пока не разберёшься полностью.

Если $ARGUMENTS — это путь к файлу, прочитай этот файл. Если это текст ТЗ — работай с ним напрямую.

Что нужно понять перед написанием кода:

1. **Что именно нужно сделать** — конечный результат, формат вывода, поведение
2. **Какие файлы затрагиваются** — что создать, что изменить, что прочитать для контекста
3. **Зависимости** — какие модули, библиотеки, таблицы БД задействованы
4. **Крайние случаи** — что может пойти не так, какие проверки нужны
5. **Порядок действий** — в какой последовательности реализовывать, чтобы каждый шаг можно было проверить

Прочитай CLAUDE.md — там стандарты кода и правила проекта, которым нужно следовать.

Если в ТЗ что-то непонятно или противоречиво — спроси у пользователя. Лучше потратить минуту на вопрос, чем час на переделку.

## Шаг 1.5: Проверь LLM-зависимости

Некоторые скрипты используют LLM — вызывают языковую модель через скиллы. Такие ТЗ описывают скиллы с именами вроде `pass1-map`, `pass2-analyze` и указывают, что промпты живут в `.claude/skills/{имя-скрипта}/`. Если в ТЗ упоминается LLM, скиллы, промпты для модели — значит скрипт зависит от готовых скиллов, и без них реализация невозможна.

Почему это важно: скрипт-оркестратор читает `SKILL.md` и передаёт его содержимое как системный промпт в `claude` CLI. Если файлов скиллов нет — скрипт не сможет работать, даже если сам код написан правильно.

### Что проверить

1. Найди в ТЗ все упомянутые имена скиллов (обычно в разделах «Алгоритм работы» или «Технические решения»)
2. Проверь, что для каждого скилла существует файл `.claude/skills/{имя-скрипта}/{имя-скилла}/SKILL.md`
3. Если хотя бы одного скилла нет — **остановись и сообщи пользователю:**

```
⛔ Реализация невозможна: не найдены скиллы, необходимые для работы скрипта.

Отсутствуют:
- {имя-скилла-1} → ожидался в .claude/skills/{имя-скрипта}/{имя-скилла-1}/SKILL.md
- {имя-скилла-2} → ...

Скрипт использует LLM и зависит от этих скиллов — без них код не будет работать.
Сначала создайте скиллы через skill-creator (или skill-chain-creator), затем повторите имплементацию.
```

Если все скиллы на месте — переходи к следующему шагу.

## Шаг 2: Изучи контекст

Перед написанием кода изучи существующий код, с которым будешь работать:

- Прочитай файлы, которые нужно изменить — целиком, не кусками
- Посмотри соседние файлы для понимания стиля и паттернов
- Если ТЗ упоминает таблицы БД, модели или API — найди их в кодовой базе
- Если ТЗ ссылается на документацию — прочитай её

Цель — писать код, который органично вписывается в проект, а не выглядит как инородное тело.

## Шаг 3: Реализуй

Пиши код последовательно, шаг за шагом, следуя алгоритму из ТЗ.

### Принципы реализации

- **Точно по ТЗ** — реализуй то, что написано. Не добавляй фич, которых нет в задании. Не «улучшай» архитектуру, если об этом не просили
- **Стандарты проекта** — следуй правилам из CLAUDE.md (PSR-12 для PHP, PEP 8 для Python, комментарии на русском, и т.д.)
- **Чистый код** — понятные имена, короткие функции, никаких магических значений
- **Обработка ошибок** — проверяй входные данные, выдавай понятные сообщения об ошибках
- **Существующие паттерны** — используй те же подходы, что и в остальном коде проекта

### Именование файла скрипта

Имя файла скрипта берётся из имени файла спецификации — это обеспечивает прозрачную связь между ТЗ и реализацией. Порядковый номер и дефис-разделитель после него отбрасываются (они нужны для сортировки спецификаций, а не для кода), дефисы в оставшейся части заменяются на подчёркивания (стандарт Python). Если ТЗ передано как текст без файла — имя выбирается по смыслу задачи, в snake_case.

### Порядок работы

1. Создай структуру файлов (если нужны новые файлы)
2. Реализуй основную логику
3. Добавь обработку крайних случаев
4. Проверь, что всё компилируется / не имеет синтаксических ошибок

Если реализация большая — разбей на логические части и реализуй по частям, проверяя каждую.

## Шаг 4: Проверь работоспособность

После написания кода убедись, что он работает. Используй все доступные инструменты:

### Синтаксис и линтинг
- **Python**: `python -m py_compile файл.py` (синтаксис), `python -m flake8 файл.py` (стиль)
- **Bash**: `bash -n файл.sh` (синтаксис)
- **PHP** (если скрипт на PHP): `php -l файл.php` (синтаксис)

### Запуск и тестирование
- Если скрипт можно запустить — запусти его (с тестовыми данными или флагом `--help`/`--dry-run`, если предусмотрен)
- Если есть юнит-тесты — запусти их
- Если ТЗ содержит примеры ожидаемого вывода — проверь, что результат совпадает
- Если код — это модуль/библиотека без точки входа — проверь хотя бы импорты

### Статический анализ
- Перечитай написанный код: нет ли очевидных ошибок, опечаток, забытых TODO?
- Сверь реализацию с ТЗ — каждый пункт ТЗ должен быть покрыт

Коммит НЕ делай — это решает пользователь. Тестирование бизнес-логики тоже оставь пользователю (для этого будет отдельный скилл).

## Шаг 5: Перенеси реализованное ТЗ

Если ТЗ было прочитано из файла в папке `development/script-specs/`, перенеси этот файл в `development/script-specs/realized/`. Это позволяет отслеживать, какие спецификации уже реализованы, а какие ещё ждут своей очереди.

```bash
mkdir -p development/script-specs/realized
mv development/script-specs/<имя-файла>.md development/script-specs/realized/
```

Если ТЗ было передано как текст (а не файл из `development/script-specs/`), или файл лежит в другом месте — пропусти этот шаг.

## Шаг 6: Обнови реестр скриптов

Реестр скриптов (`development/docs/scripts-registry.md`) — это единый индекс всех реализованных скриптов проекта. Без записи в реестре другие разработчики и сессии не узнают, что скрипт существует, и могут создать дубликат или искать его вручную.

После реализации скрипта:

1. Прочитай `development/docs/scripts-registry.md`
2. Проверь, нет ли в таблице уже строки с этим скриптом — если есть, пропусти этот шаг
3. Добавь новую строку в таблицу `## Скрипты`, заполнив все колонки:
   - **№** — номер из имени файла спецификации (например, `003` из `003-skill-factory.md`)
   - **Название** — краткое название скрипта на русском
   - **Описание** — одно предложение о том, что скрипт делает
   - **Запуск** — команда для запуска с основными аргументами
   - **Спецификация** — путь к файлу ТЗ (уже в `development/script-specs/realized/`, если был перенесён на шаге 5)

Если ТЗ было передано как текст (без файла спецификации) — номер подбери по порядку, а в колонке «Спецификация» укажи прочерк.

## Шаг 7: Отчёт

После завершения реализации выведи отчёт:

```
## Отчёт о реализации

### Что сделано
- {пункт 1: что конкретно реализовано}
- {пункт 2: ...}

### Проверки
- {какие проверки выполнены и их результат}

### Файлы и изменения
{вывод git diff --stat}

### Замечания (если есть)
- {что не удалось сделать и почему}
- {что требует дополнительного внимания}
- {что нужно протестировать вручную}
```

Получи `git diff --stat` для раздела «Файлы и изменения». Если git недоступен — просто перечисли файлы и что в них изменилось.
